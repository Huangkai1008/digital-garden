# 第6章 流与表

我们来到了讨论流（streams）与表（tables）的部分。回顾第 1 章，我们简要讨论了数据的两个重要且正交的维度：基数（cardinality）和构成（constitution）。此前我们严格聚焦基数（有界 vs 无界），忽略了构成（流 vs 表）。现在我们扩展视野，看看构成这一额外维度带来什么。

本章有两个目的：1）描述 Beam Model 与「流与表」理论（由 Martin Kleppmann、Jay Kreps 等人推广，本质上源自数据库世界）的关系；2）用糟糕的物理类比轰炸你。

---

## 流与表基础：流表相对论的特殊理论

流与表的基本思想源自数据库世界。熟悉 SQL 的人可能熟悉表及其核心性质：表包含行和列，每行由某种键（显式或隐式）唯一标识。数据库底层的数据结构通常是**仅追加日志（append-only log）**。事务应用于表时，被记录在日志中，日志内容随后串行应用到表以物化这些更新。在流与表的术语中，该日志 essentially 就是**流**。

由此我们理解如何从流创建表：表就是应用流中更新事务日志的结果。如何从表创建流？Essentially 是逆过程：**流是表的变更日志（changelog）**。表到流转换的典型动机例子是**物化视图（materialized views）**。

综合这两点，得到**流表相对论的特殊理论**：

| 方向 | 含义 |
|------|------|
| **流 → 表** | 随时间对更新流的聚合产生表 |
| **表 → 流** | 随时间对表变化的观察产生流 |

这对概念非常强大，其谨慎应用于流式处理世界是 Apache Kafka 及其生态巨大成功的重要原因。

---

## 广义流表相对论

为将流表理论与 Beam Model 的一切 reconcile，需要解决几个问题：批处理如何融入？流与有界/无界数据集的关系？What、Where、When、How 四个问题如何映射到流表世界？

独立于彼此定义流与表很有启发性：

- **表是静止的数据（data at rest）**：表的快照在任意时刻提供数据集的整体图景。表是数据积累和随时间被观察的概念性「安息地」。
- **流是运动的数据（data in motion）**：流捕获数据随时间的演化。Julian Hyde 喜欢说流像表的导数，表像流的积分。

---

## MapReduce 的流表分析

将传统 MapReduce 作业分解为六个阶段：

| 阶段 | 描述 |
|------|------|
| **MapRead** | 消费输入数据，预处理为标准键值形式 |
| **Map** | 消费键值对，输出零或多条键值对 |
| **MapWrite** | 按键聚类 Map 输出，写入持久存储（group-by-key-and-checkpoint） |
| **ReduceRead** | 消费 shuffle 数据，转换为键/值列表形式 |
| **Reduce** | 消费键及其值列表，输出零或多条记录 |
| **ReduceWrite** | 将 Reduce 输出写入输出数据存储 |

流表视角下的流程：

1. **MapRead**：表 → 流（迭代输入表，将数据放入运动）
2. **Map**：流 → 流（逐元素变换）
3. **MapWrite**：流 → 表（按键分组，将数据安放下来）
4. **ReduceRead**：表 → 流
5. **Reduce**：流 → 流
6. **ReduceWrite**：流 → 表

因此，批处理本质上是一系列**表→流→表**的转换。有界与无界的区别：有界数据对应完整的表；无界数据对应持续演化的表，我们通过流观察其变化。

---

## Beam Model 中的流与表

在 Beam Model 中，What/Where/When/How 映射到流表世界：

- **What**：变换决定流中元素的处理方式
- **Where**：窗口化决定流如何被分组为表（按事件时间边界）
- **When**：触发器决定表的物化视图何时作为流发出
- **How**：累积模式决定多次物化之间的关系

---

## 小结

流与表是理解数据处理的双重透镜。表是静止的数据，流是运动的数据。流聚合产生表，表的变化产生流。批处理和流式处理都可以用这一框架理解；Beam Model 的四个问题自然映射到流表世界。
