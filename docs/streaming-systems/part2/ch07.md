# 第7章 持久化状态的实践

持久化状态（persistent state）literally 就是第 6 章讨论的**表**，附加要求是表必须稳健地存储在相对不易丢失的介质上。本地磁盘可以（只要别问 SRE）；复制到多块磁盘更好；复制到不同物理位置的磁盘更好；只存内存一次肯定不行。

本章目标：

1. 激发管道内持久化状态的需求
2. 审视管道中常见的两种隐式状态形式
3. 考虑一个难以用隐式状态处理的真实用例（广告转化归因），以激发通用显式持久化状态管理的 salient 特征
4. 探索 Apache Beam 中此类状态 API 的具体体现

---

## 动机

### 故障的必然性

处理无界数据的管道 effectively  intended 永久运行。但永久运行是比这些管道通常执行环境所能实现的更苛刻的 SLO。长运行管道不可避免地会遇到机器故障、计划维护、代码变更、偶尔的错误配置等中断。为确保中断后能从断点恢复，长运行管道需要某种对中断前位置的持久记忆。这就是持久化状态的用武之地。

- **有界数据**：系统通常假设失败时可以从头重新处理输入。可能假设故障不频繁，因而为常见情况优化，尽可能少持久化，接受失败时的重算成本。
- **无界数据**：必须假设规模无限。系统从不假设所有数据可重新处理，只有已知子集。为提供至少一次或精确一次语义，不再可重新处理的数据必须计入持久 checkpoint。

持久化状态与批处理或流式无本质绑定；两种情况下都有用。只是处理无界数据时至关重要，因此流式系统通常提供更完善的持久化状态支持。

### 正确性与效率

持久化状态提供两方面的价值：

1. **正确性基础**：面对易失输入时，持久状态允许保留继续处理所需的中间信息，即使输入源已继续前进并遗忘先前给出的记录。
2. **最小化重复工作与持久数据**：通过 checkpoint 管道内的部分进度（计算的中间结果以及 checkpoint 时的输入位置），可大幅减少失败时重复的工作量。通过智能垃圾回收不再需要的持久状态，可将存储量保持在可管理范围。

---

## 隐式状态（Implicit State）

### 原始分组（Raw Grouping）

最直接的实现是**原始分组**：新元素到达时追加到该组的元素列表。在 Beam 中，对 `PCollection` 应用 `GroupByKey` 即得到此行为。输入为 `PCollection<KV<K, V>>`，输出为 `PCollection<KV<K, Iterable<V>>>`。

### 增量组合（Incremental Combining）

原始分组存储所有原始输入，空间效率低。**增量组合**在分组时维护更紧凑的中间表示（如求和时只存总和与计数），可大幅减少存储和重算量。

---

## 显式状态：转化归因案例

广告转化归因（conversion attribution）需要将转化事件与先前的广告展示/点击关联。这需要跨时间、跨事件类型的复杂状态管理，难以用简单的隐式分组表达。

通用显式状态 API 应支持：

- **按键+窗口的状态**：每个键和窗口有独立状态
- **可组合的状态类型**：ValueState、BagState、CombiningState 等
- **定时器（Timers）**：在特定事件时间或处理时间触发回调，用于超时、窗口关闭等

Beam 的 `DoFn` 中的 `@StateId` 和 `@TimerId` 注解提供了此类能力。

---

## 小结

持久化状态是管道中表的持久化存储，为正确性和高效容错提供基础。隐式状态（原始分组、增量组合）适用于许多用例；复杂用例需要显式状态 API 和定时器。Beam 通过 State 和 Timers API 支持这些能力。
