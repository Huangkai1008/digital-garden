# 第8章 流式 SQL

本章探讨如何将流式概念融入 SQL。核心结论（punchline）是：**自然地将流式融入 SQL 的关键，是将关系代数中的核心数据对象——关系——扩展为表示随时间变化的数据集，而非特定时间点的数据集。** 即我们需要**时变关系（time-varying relations, TVR）**，而非时间点关系。

---

## 什么是流式 SQL？

数据库社区可能已理解约 99% 的答案，但尚未有真正清晰、全面、涵盖稳健流式语义全貌的流式 SQL 定义。本章大部分内容在撰写时仍是纯假设性的。Calcite、Flink、Beam 等社区在 2017 年展开协作讨论，本章试图将其中思想提炼为关于将流式概念融入 SQL 的清晰连贯叙述。

---

## 关系代数

SQL 的理论基础是**关系代数（relational algebra）**：用数学方式描述由命名、类型化元组组成的数据之间的关系。关系的核心是**关系本身**，即这些元组的集合。关系代数的一个关键性质是**闭包性**：对任意有效关系应用关系代数的任意算子，总产生另一个关系。

历史上许多支持流式的 SQL 尝试未能满足闭包性：它们将流与经典关系分开对待，提供新算子在两者间转换，并限制可应用于其中之一的操作。要真正将流式 SQL 推向 forefront，需要让流式成为关系代数本身的一等公民，使整个标准关系代数能自然应用于流式和非流式用例。

---

## 时变关系（Time-Varying Relations）

**时变关系**是经典关系随时间的演化。若经典关系是二维表（x 轴为列，y 轴为行），时变关系就像三维表，增加 z 轴捕获二维表随时间的不同版本。

定义时变关系的重要结果是：

1. **经典关系代数的全部算子在应用于时变关系时仍然有效**，且行为与预期完全一致
2. **关系代数的闭包性在应用于时变关系时保持完整**

即，用时变关系替代经典关系并未以任何方式改变游戏规则。一切继续按经典关系世界的方式工作，只是作用于经典关系的序列而非单例。

---

## 流与表的统一视图

在 Beam Model 与流式 SQL 的统一视图中：

- **表**：时变关系在特定时间点的快照。`SELECT TABLE` 返回当前快照。
- **流**：时变关系的变更流。`SELECT STREAM` 返回捕获关系演化的流，包含插入、更新（通过撤回+新值）、删除。

流式 SQL 的挑战在于：如何用声明式 SQL 表达 What、Where、When、How？时变关系使 What 和 Where 自然融入标准 SQL；When 和 How 需要扩展（如窗口化、触发器、累积模式），这些在 Calcite/Flink/Beam 的流式 SQL 提案中有相应设计。

---

## 小结

时变关系是将流式自然融入 SQL 的关键。通过将关系扩展为随时间变化的序列，经典关系代数的全部算子和闭包性得以保留。表是 TVR 的快照，流是 TVR 的变更流。这一框架为稳健的流式 SQL 提供了清晰的理论基础。
