# 第4章 高级窗口

在更深入理解水位线之后，本章探讨与 What、Where、When、How 相关的更多高级主题。

首先讨论**处理时间窗口化**，它是 Where 与 When 的有趣混合，以更好理解其与事件时间窗口化的关系，并了解何时采用它是合适的。然后深入更高级的事件时间窗口化概念，详细讨论会话窗口，最后通过三种自定义窗口类型（非对齐固定窗口、按键固定窗口、有界会话窗口）论证广义自定义窗口化是有用且 surprisingly 直接的概念。

---

## When/Where：处理时间窗口

处理时间窗口化重要的两个原因：

1. **适用场景**：如使用监控（如 Web 服务流量 QPS），希望按观察顺序分析到达数据流时，处理时间窗口化是绝对合适的方法。
2. **不适用场景**：当事件发生时间重要时（如分析用户行为趋势、计费、评分等），处理时间窗口化是绝对错误的方法，识别这些场景至关重要。

在本书采用的、窗口化严格基于事件时间的模型中，实现处理时间窗口化有两种方法：

| 方法 | 描述 |
|------|------|
| **触发器** | 忽略事件时间（使用跨越整个事件时间的全局窗口），用触发器在处理时间轴上提供该窗口的快照。 |
| **入口时间（Ingress time）** | 将数据到达时的入口时间赋为事件时间，然后使用常规事件时间窗口化。Essentially 是 Spark Streaming 1.x 的做法。 |

处理时间窗口化的主要缺点是：**窗口内容会随输入观察顺序的变化而改变**。事件时间窗口化对顺序不敏感（至少在极限下；实际窗格在输入完整前可能不同）；处理时间窗口化则不然。若你关心事件实际发生的时间，必须使用事件时间窗口化，否则结果将无意义。

---

## Where：会话窗口（Session Windows）

会话是一种特殊的动态、数据驱动窗口，捕获由不活动间隔终止的活动期。它们在数据分析中特别有用，可提供特定用户在特定时间段内参与某活动的视图。

从窗口化角度看，会话有两个有趣之处：

- **数据驱动窗口**：窗口的位置和大小直接由输入数据决定，而非基于时间的预定义模式。
- **非对齐窗口**：不统一应用于所有数据，而只应用于特定子集（如每用户）。

在更一般的情况下（会话本身事先未知），必须仅根据数据在时间中的位置构建会话。处理乱序数据时，这变得尤为棘手。

关键洞察：完整的会话窗口本质上是一组较小的、重叠的窗口的组合，每个窗口包含单条记录，序列中相邻记录由不超过预定义超时的不活动间隔分隔。因此，即使以乱序观察会话数据，也可以 simply 通过合并到达的单个数据的重叠窗口来构建最终会话。

```
未合并的原型会话窗口:
  [r1]  [r2]     [r3]  [r4]  [r5]
    \___/          \_____/
合并后的会话:
  [  session 1  ]  [ session 2 ]
```

---

## 广义自定义窗口

除固定、滑动、会话等内置窗口外，Beam Model 支持**广义自定义窗口化**。三种示例类型：

1. **非对齐固定窗口**：不同键的窗口相位偏移，以更均匀地分散窗口完成负载。
2. **按键固定窗口**：每个键有独立的窗口边界。
3. **有界会话窗口**：会话窗口，但限制最大长度或其它约束。

自定义窗口化的核心是**窗口合并（window merging）**：当新数据到达时，若其与现有窗口重叠，则合并；否则创建新窗口。这使会话等动态窗口能够正确处理乱序数据。

---

## 小结

- 处理时间窗口化适用于按观察顺序分析的场景；事件时间窗口化适用于关心事件实际发生时间的场景。
- 会话窗口是数据驱动、非对齐的动态窗口，通过合并重叠的原型窗口构建。
- 广义自定义窗口化允许定义满足特定需求的窗口策略，核心机制是窗口合并。
