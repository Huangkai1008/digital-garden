# 第1章 Streaming 101

流式数据处理（Streaming data processing）如今在大数据领域举足轻重，原因包括：

- 企业渴望更及时地从数据中获得洞察，转向流式处理是降低延迟的有效途径
- 现代业务中日益常见的大规模、无界数据集（unbounded datasets），使用专为此类永无止境数据量设计的系统更容易驾驭
- 数据到达即处理，可将工作负载在时间上更均匀地分布，带来更一致、可预测的资源消耗

尽管业界对流式处理兴趣高涨，流式系统长期以来仍相对批处理系统不够成熟。直到最近，局面才明显扭转。在我更自负的时刻，我希望这在一定程度上得益于我在「Streaming 101」和「Streaming 102」博客文章中提出的有力观点（本书前几章显然基于这些文章）。但现实中，业界也普遍希望流式系统走向成熟，并且有很多聪明且活跃的人在推动这一进程。

尽管在我看来，对流式处理的普遍倡导之战已经基本获胜，我仍将基本沿用「Streaming 101」中的原始论点。其一，这些论点在今天仍然非常适用，即便许多行业已开始响应这一呼声。其二，仍有不少人尚未了解这些观点；本书是一次更深入的尝试，希望把这些要点传达出去。

首先，我会介绍一些重要的背景知识，为后续讨论奠定基础，分为三个部分：

## 术语（Terminology）

精确讨论复杂话题需要精确的术语定义。对于当前使用中含义多样的术语，我会尽量明确其在本书中的含义。

我会讨论流式系统常被诟病的不足，并提出我认为数据处理系统构建者需要采纳的思维方式，以应对现代数据消费者的需求。

我会介绍数据处理中两个主要的时间域，说明它们的关系，并指出它们带来的困难。

## 术语：什么是 Streaming？

在继续之前，我想先澄清一件事：什么是 streaming？如今「streaming」一词被用来指代多种不同的事物（为简洁起见，此前我也在较宽松地使用它），这可能导致对流式处理本质或流式系统实际能力的误解。因此，我倾向于给出一个相对精确的定义。

问题的核心在于，许多本应以其本质来描述的事物（如无界数据处理、近似结果等），却往往以其历史上如何实现（即通过流式执行引擎）以通俗说法描述。这种术语上的不精确模糊了 streaming 的真正含义，有时还会让流式系统背负一种暗示：其能力仅限于历史上被描述为「streaming」的特征，例如近似或推测性结果。

鉴于设计良好的流式系统在产生正确、一致、可重复结果方面与任何现有批处理引擎同样 capable（技术上甚至更强），我倾向于将「streaming」限定为一个非常具体的含义：

::: info 流式系统（Streaming system）
一种数据处理引擎，其设计面向无限数据集。
:::

若要讨论低延迟、近似或推测性结果，我会使用这些具体词汇，而不是笼统地称之为「streaming」。

在讨论可能遇到的不同数据类型时，精确术语同样有用。在我看来，定义数据集形态有两个重要且正交的维度：**基数（cardinality）** 和 **构成（constitution）**。

数据集的基数决定其规模，其中最突出的方面是数据集是有限还是无限。我倾向于用以下两个术语描述数据集的粗粒度基数：

| 术语 | 含义 |
|------|------|
| **有界数据（Bounded data）** | 规模有限的数据集 |
| **无界数据（Unbounded data）** | 规模无限的数据集（至少理论上如此） |

基数之所以重要，是因为无限数据集的无界性会给消费它们的数据处理框架带来额外负担。下一节会进一步讨论。

数据集的构成则决定其物理表现形式，进而定义与数据交互的方式。我们直到第 6 章才会深入探讨构成，但先简要说明：有两个重要的主要构成：

- **表（Table）**：在特定时间点上对数据集的整体视图。SQL 系统传统上以表为主要对象。
- **流（Stream）**：数据集随时间演化的逐元素视图。MapReduce 系数据处理系统传统上以流为主要对象。

我们会在第 6、8、9 章深入探讨流与表的关系，并在第 8 章学习将它们统一起来的底层概念：**时变关系（time-varying relations）**。在此之前，我们主要讨论流，因为这是当今大多数数据处理系统（批处理和流式）中管道开发者直接交互的构成，也是最自然体现流式处理特有挑战的构成。

## 论流式系统的「被夸大」的局限性

接下来讨论流式系统能做什么、不能做什么，重点放在「能」上。本章最想传达的一点是：设计良好的流式系统可以有多强大。

流式系统历史上被局限在一个相对小众的市场：提供低延迟、不准确或推测性结果，往往与更「靠谱」的批处理系统配合，以提供最终正确的结果——即 **Lambda 架构（Lambda Architecture）**。

对不熟悉 Lambda 架构的读者，其基本思路是：同时运行流式系统和批处理系统，两者执行本质上相同的计算。流式系统提供低延迟、不准确的结果（或因使用近似算法，或因流式系统本身不保证正确性），一段时间后批处理系统再提供正确输出。该架构最初由 Twitter 的 Nathan Marz（Storm 的创造者）提出，在当时非常成功，因为流式引擎在正确性上确实不足，批处理引擎又天生笨重，Lambda 让你可以「鱼与熊掌兼得」。但维护 Lambda 系统很麻烦：需要构建、配置和维护两套独立的管道，最后还要以某种方式合并两套管道的结果。

作为多年从事强一致流式引擎的人，我也觉得 Lambda 架构的整个原则有些令人不快。因此，当 Jay Kreps 的「Questioning the Lambda Architecture」文章出现时，我成了它的忠实拥趸。这是最早公开质疑双模式执行必要性的高调声音之一，令人欣喜。Kreps 在可重放系统（如 Kafka）作为流式互连的背景下讨论了可重复性，甚至提出了 **Kappa 架构（Kappa Architecture）**，即用单一管道、配合为任务适当设计的系统来运行。我不确定这个概念是否需要自己的希腊字母命名，但原则上完全支持。

坦率地说，我会更进一步。我认为设计良好的流式系统实际上提供了批处理功能的**严格超集**。除了可能的效率差异外，理论上不应再需要今天意义上的批处理系统。Apache Flink 团队将这一理念付诸实践，构建了底层全流式、即使在「批处理」模式下也始终流式的系统，值得称赞。

### 批处理与流式的效率差异

效率差异并非流式系统的固有局限，而更多是迄今为止大多数流式系统设计选择的结果。批处理与流式之间的效率差距，主要来自批处理系统中更充分的 bundling 和更高效的 shuffle 传输。现代批处理系统在实现复杂优化方面投入大量精力，以相对 modest 的计算资源实现惊人的吞吐量。没有理由认为，使批处理成为今天效率「重量级」的巧妙思路，不能融入面向无界数据的系统，从而让用户灵活选择我们通常所说的高延迟、高效率「批处理」与低延迟、低效率「流式」处理。Google 的 Cloud Dataflow 正是这样做的：在同一统一模型下提供批处理和流式两种 runner。我们使用独立的 runner，是因为恰好有两套各自优化的系统。从工程角度看，长期希望将两者合并为单一系统，融合两者优点并保留选择合适效率水平的灵活性。但今天尚未实现，而且得益于统一的 Dataflow Model，这甚至不是严格必需的，因此可能永远不会发生。

由此得出的推论是：流式系统的广泛成熟，加上面向无界数据处理的稳健框架，终将让 Lambda 架构退居大数据历史的故纸堆。我认为时机已经到来。因为要做到这一点——即在批处理擅长的领域超越批处理——你真正需要的只有两件事：

1. **正确性（Correctness）**：与批处理持平。核心在于一致存储。流式系统需要一种随时间对持久状态进行 checkpoint 的方法（Kreps 在「Why local state is a fundamental primitive in stream processing」中讨论过），且必须设计得足够好，在机器故障下仍保持一致。当 Spark Streaming 几年前首次出现在公众视野时，它是黑暗流式世界中的一致性灯塔。此后情况已大幅改善，但仍有大量流式系统试图在没有强一致性的情况下运行。

::: warning
重申——这一点很重要：强一致性是精确一次处理（exactly-once processing）所必需的，而精确一次又是正确性所必需的，是任何有望达到或超越批处理系统能力的系统的必要条件。除非你真的不在乎结果，否则请远离任何不提供强一致状态的流式系统。批处理系统不需要你事先验证它们能否产生正确结果；不要在无法达到同样标准的流式系统上浪费时间。
:::

2. **用于推理时间的工具（Tools for reasoning about time）**：超越批处理。良好的时间推理工具对于处理具有不同事件时间偏斜的无界、无序数据至关重要。越来越多的现代数据集呈现这些特征，而现有批处理系统（以及许多流式系统）缺乏应对这些困难所需的工具（尽管情况正在迅速改变）。本书大部分内容将围绕这一点的各个方面展开。

我们先建立对时间域这一重要概念的基本理解，然后更深入地探讨「无界、无序、具有不同事件时间偏斜的数据」的含义，最后回顾有界和无界数据处理的常见方法，涵盖批处理和流式系统。

## 事件时间 vs 处理时间

要清晰讨论无界数据处理，必须清楚理解所涉及的时间域。在任何数据处理系统中，通常有两个我们关心的时间域：

| 时间域 | 含义 |
|--------|------|
| **事件时间（Event time）** | 事件实际发生的时间 |
| **处理时间（Processing time）** | 事件在系统中被观察到的时间 |

并非所有用例都关心事件时间（若你的用例不关心，恭喜，你的日子会轻松很多），但许多用例需要。例如：随时间刻画用户行为、大多数计费应用、多种异常检测等。

理想情况下，事件时间和处理时间应始终相等，事件一发生就被处理。现实并非如此仁慈，事件时间与处理时间之间的偏斜不仅非零，而且往往是底层输入源、执行引擎和硬件特征的**高度可变函数**。可能影响偏斜水平的因素包括：

- 共享资源限制，如网络拥塞、网络分区或非专用环境中的共享 CPU
- 软件因素，如分布式系统逻辑、争用等
- 数据本身的特征，如键分布、吞吐量方差或无序性方差（例如，一架飞机上的乘客在全程离线使用手机后，同时解除飞行模式）

因此，若你在任何真实系统中绘制事件时间和处理时间的进展，通常会得到类似图 1-1 中红线的形状。

```
事件时间完整性 (x轴)
    ^
    |    理想线 (斜率=1)
    |   /
    |  /  实际线 (红色)
    | /  ___
    |/__/
    +---------> 处理时间 (y轴)
```

::: info 图 1-1 时间域映射
- x 轴：系统中事件时间的完整性，即事件时间 X，使得所有事件时间小于 X 的数据都已被观察到
- y 轴：处理时间的进展，即数据处理系统执行时观察到的正常时钟时间
:::

在图 1-1 中，斜率为 1 的黑色虚线表示理想情况（处理时间与事件时间完全相等），红线表示现实。此例中，系统在处理时间开始时略有滞后，中间更接近理想，末尾又略有滞后。乍看之下，图中有两种偏斜，分别处于不同时间域：

- **处理时间**：理想线与红线之间的垂直距离是处理时间域中的延迟，表示给定时间的事件发生与被处理之间观察到的延迟（以处理时间计）。这或许是两种偏斜中更自然、更直观的一种。
- **事件时间**：理想线与红线之间的水平距离是管道在该时刻的事件时间偏斜，表示管道当前在事件时间上落后理想线的程度。

实际上，任意时刻的处理时间延迟与事件时间偏斜是相同的，只是看待同一事物的两种方式。关于延迟/偏斜的重要结论是：**由于事件时间与处理时间之间的整体映射不是静态的（即延迟/偏斜可能随时间任意变化），若你关心事件时间，就不能仅在数据被管道观察到的上下文中分析数据。**

不幸的是，许多面向无界数据的系统历史上正是这样运作的。为应对无界数据集的无限性，这些系统通常提供某种窗口化（windowing）概念，即将数据沿时间边界切分为有限片段。若你关心正确性，并希望在事件时间的上下文中分析数据，就不能像许多系统那样用处理时间定义这些时间边界（即处理时间窗口化）；由于处理时间与事件时间之间没有一致的相关性，部分事件时间数据会落入错误的处理时间窗口（由于分布式系统的固有延迟、多种输入源的在线/离线特性等），从而破坏正确性。我们会在后续章节的多个示例以及本书其余部分更详细地探讨这一问题。

使用事件时间窗口化时，情况也并不乐观。在无界数据背景下，无序和可变偏斜会带来**完整性（completeness）问题**：缺乏处理时间与事件时间之间的可预测映射，如何确定何时已观察到给定事件时间 X 的全部数据？对许多真实数据源而言，你根本无法确定。但当今使用的大多数数据处理系统都依赖某种完整性概念，这使它们在应用于无界数据集时处于严重劣势。

我建议：与其试图将无界数据整理成最终会「完整」的有限批次，不如设计允许我们在这些复杂数据集所强加的不确定性世界中生存的工具。新数据会到达，旧数据可能被撤回或更新，我们构建的任何系统都应能自行应对这些事实，而完整性应作为特定、合适用例的便利优化，而非所有用例的语义必需。在深入探讨这种方法的具体形态之前，我们先完成最后一块有用的背景：常见的数据处理模式。

## 数据处理模式

至此，我们已有足够背景，可以开始审视当今有界和无界数据处理中的核心使用模式。我们会同时考察两种处理类型，并在相关处结合我们关心的两类引擎（批处理和流式；在此上下文中，我 essentially 将微批处理归入流式，因为两者在此层面的差异并不重要）。

### 有界数据

处理有界数据在概念上相当直接，可能对每个人都熟悉。在图 1-2 中，我们从左侧充满熵的数据集开始，通过某种数据处理引擎（通常是批处理，但设计良好的流式引擎同样可行，如 MapReduce）运行，最终在右侧得到具有更高内在价值的新结构化数据集。

```
[ 无界熵数据 ] --> [ 数据处理引擎 ] --> [ 结构化数据 ]
```

::: info 图 1-2 使用经典批处理引擎处理有界数据
左侧的有限非结构化数据池通过数据处理引擎运行，在右侧产生相应的结构化数据。
:::

当然，你可以在此方案中计算的内容有无限种变化，但整体模型相当简单。更有趣的是处理无界数据集。下面我们看看无界数据的典型处理方式，从传统批处理引擎的方法开始，到面向无界数据的系统（如大多数流式或微批处理引擎）可采用的方法。

### 无界数据：批处理

批处理引擎虽非显式面向无界数据设计，但自批处理系统诞生以来就一直被用来处理无界数据集。正如你所料，这类方法围绕将无界数据切分为适合批处理的有界数据集集合展开。

**固定窗口（Fixed windows）**

使用批处理引擎的重复运行处理无界数据集的最常见方式，是将输入数据窗口化为固定大小的窗口，然后将每个窗口作为独立的有界数据源处理（有时也称为滚动窗口，tumbling windows），如图 1-3。特别是对于日志等输入源，事件可写入目录和文件层次结构，其名称编码对应的窗口，这类做法乍看相当直接，因为你 essentially 已提前完成基于时间的 shuffle，将数据放入相应的事件时间窗口。

现实中，大多数系统仍面临完整性问题（若部分事件因网络分区在到达日志途中延迟怎么办？若事件在全球收集并需传输到共同位置再处理怎么办？若事件来自移动设备怎么办？），这意味着可能需要某种缓解措施（例如，延迟处理直到确信所有事件已收集，或在数据迟到时重新处理整个窗口的批次）。

```
无界数据 --> [ 固定窗口收集 ] --> [ 批处理引擎 ] --> 结果
```

::: info 图 1-3 使用经典批处理引擎通过临时固定窗口处理无界数据
无界数据集先被收集到有限的固定大小有界数据窗口中，然后通过经典批处理引擎的连续运行进行处理。
:::

**会话（Sessions）**

当你尝试用批处理引擎将无界数据处理成更复杂的窗口策略（如会话）时，这种方法会进一步失效。会话通常定义为由一段不活动间隔终止的活动期（例如，针对特定用户）。使用典型批处理引擎计算会话时，往往会得到跨批次分割的会话，如图 1-4 中的红点所示。可以通过增大批次大小减少分割数量，但代价是延迟增加。另一种选择是添加额外逻辑，将先前运行的会话拼接起来，但代价是复杂度增加。

::: info 图 1-4 使用经典批处理引擎通过临时固定窗口将无界数据处理为会话
无界数据集先被收集到有限的固定大小有界数据窗口中，然后通过经典批处理引擎的连续运行细分为动态会话窗口。
:::

无论如何，使用经典批处理引擎计算会话都不理想。更好的方式是以流式方式构建会话，我们稍后会看到。

### 无界数据：流式

与大多数基于批处理的无界数据处理方法的临时性相反，流式系统是为无界数据而建的。如前所述，对于许多真实世界的分布式输入源，你不仅面对无界数据，还面对以下特征的数据：

- **事件时间高度无序**：若你希望在事件发生的上下文中分析数据，管道中需要某种基于时间的 shuffle
- **事件时间偏斜可变**：不能假设给定事件时间 X 的大部分数据总会在某个常数 ε 的时间 Y 内到达

处理具有这些特征的数据时，有几种方法可选。我大致将这些方法分为四类：**时间无关（time-agnostic）**、**近似（approximation）**、**按处理时间窗口化（windowing by processing time）** 和 **按事件时间窗口化（windowing by event time）**。

下面逐一简要讨论。

#### 时间无关

时间无关处理用于时间 essentially 无关紧要的用例；即所有相关逻辑都由数据驱动。因为这类用例的一切都由更多数据的到达决定，流式引擎除了基本数据传递外，无需支持任何特殊功能。因此，几乎所有现有流式系统都开箱即用地支持时间无关用例（当然，若你关心正确性，系统间的一致性保证会有差异）。批处理系统也适合时间无关地处理无界数据源，只需将无界源切分为任意序列的有界数据集并独立处理即可。本节会看几个具体例子，但鉴于时间无关处理（至少从时间角度）的直截了当，我们不会花太多时间。

**过滤（Filtering）**

时间无关处理的一种非常基本形式是过滤，图 1-5 给出了示例。假设你正在处理 Web 流量日志，希望过滤掉所有非来自特定域名的流量。你会查看每条到达的记录，判断是否属于目标域名，不属于则丢弃。因为这类操作只依赖任意时刻的单个元素，数据源是否无界、无序、具有可变事件时间偏斜都无关紧要。

```
[ 多种类型数据 ] --> [ 过滤 ] --> [ 单一类型数据 ]
```

::: info 图 1-5 过滤无界数据
不同类型的数据集合（从左向右流动）被过滤为包含单一类型的同质集合。
:::

**内连接（Inner joins）**

另一个时间无关的例子是内连接，如图 1-6。连接两个无界数据源时，若你只关心当两个源都有元素到达时的连接结果，逻辑中就没有时间成分。看到一个源的值时，可以简单地在持久状态中缓冲；只有当另一个源的第二个值到达时，才需要发出连接后的记录。（实际上，你可能需要某种针对未发出部分连接的垃圾回收策略，这可能是基于时间的。但对于几乎没有未完成连接的用例，这可能不是问题。）

::: info 图 1-6 对无界数据执行内连接
当观察到两个源中的匹配元素时产生连接。
:::

若将语义切换到某种外连接，就会引入我们讨论过的数据完整性问题：在看到连接的一侧后，如何知道另一侧是否会到达？说实话，你不知道，因此需要引入某种超时概念，这就引入了时间成分。该时间成分 essentially 是一种窗口化，我们稍后会更仔细地看。

#### 近似算法（Approximation algorithms）

第二类主要方法是近似算法，如近似 Top-N、流式 k-means 等。它们接受无界输入源，产生若你眯眼看的话大致符合期望的输出数据，如图 1-7。近似算法的优点是：按设计，它们开销低且面向无界数据。缺点是：算法种类有限，算法本身往往复杂（难以创造新的），其近似性质限制了用途。

::: info 图 1-7 对无界数据计算近似
数据通过复杂算法运行，在另一端产生大致符合期望结果的输出数据。
:::

值得注意的是，这些算法在设计中通常确实有某种时间成分（例如某种内置衰减）。因为它们按到达顺序处理元素，该时间成分通常基于处理时间。对于提供某种可证明近似误差界的算法，这一点尤为重要。若这些误差界以数据按序到达为前提，当你向算法喂入无序、具有可变事件时间偏斜的数据时，它们 essentially 毫无意义。值得牢记。

#### 窗口化（Windowing）

无界数据处理的剩余两种方法都是窗口化的变体。在深入它们的差异之前，应明确我所说的窗口化——上一节只简要提及。**窗口化** simply 是指将数据源（有界或无界）沿时间边界切分为有限块以供处理的概念。图 1-8 展示了三种不同的窗口化模式。

::: info 图 1-8 窗口化策略
每个示例针对三个不同的键展示，突出对齐窗口（应用于所有数据）与非对齐窗口（应用于数据子集）之间的差异。
:::

下面更仔细地看每种策略：

**固定窗口（Fixed windows，又称 tumbling windows）**

前面已讨论。固定窗口将时间切分为具有固定时间长度的片段。通常（如图 1-9），固定窗口的片段均匀应用于整个数据集，这是**对齐窗口（aligned windows）** 的例子。在某些情况下，希望对不同数据子集（例如按键）对窗口进行相位偏移，以更均匀地分散窗口完成负载，这是**非对齐窗口（unaligned windows）** 的例子。

**滑动窗口（Sliding windows，又称 hopping windows）**

固定窗口的泛化，由固定长度和固定周期定义。若周期小于长度，窗口重叠。若周期等于长度，即为固定窗口。若周期大于长度，则是一种奇怪的采样窗口，只随时间查看数据的子集。与固定窗口一样，滑动窗口通常是对齐的，但在某些用例中可作为性能优化而采用非对齐形式。

**会话（Sessions）**

动态窗口的例子，由被大于某超时的不活动间隔终止的事件序列组成。会话常用于分析随时间变化的用户行为，将一系列时间相关事件（例如一次观看的视频序列）组合在一起。会话的有趣之处在于其长度无法先验定义，取决于实际数据。它们也是非对齐窗口的典型例子，因为会话在不同数据子集（例如不同用户）间几乎从不相同。

我们之前讨论的两个时间域（处理时间和事件时间）essentially 就是我们关心的。窗口化在两个域中都有意义，下面分别详细讨论它们的差异。由于按处理时间窗口化历史上更常见，我们从它开始。

**按处理时间窗口化（Windowing by processing time）**

按处理时间窗口化时，系统 essentially 将到达的数据缓冲到窗口中，直到经过一定量的处理时间。例如，对于五分钟固定窗口，系统会缓冲五分钟处理时间的数据，然后将在这五分钟内观察到的所有数据视为一个窗口，发送到下游处理。

::: info 图 1-9 按处理时间窗口化为固定窗口
数据根据到达管道的顺序被收集到窗口中。
:::

按处理时间窗口化有几个不错的性质：

- **简单**：实现极其直接，因为你从不需要担心在时间内的数据 shuffle，只需按到达顺序缓冲，窗口关闭时发送到下游
- **判断窗口完整性直接**：系统对是否已看到窗口的所有输入有完美知识，可以对给定窗口是否完整做出完美决策。这意味着按处理时间窗口化时，无需以任何方式处理「迟到」数据
- **若你想根据观察到的源推断信息**，按处理时间窗口化正是你想要的。许多监控场景属于此类。想象跟踪发送到全球规模 Web 服务的每秒请求数，为检测故障计算这些请求的速率，是按处理时间窗口化的完美用例

尽管有这些优点，按处理时间窗口化有一个很大的缺点：**若所讨论的数据带有事件时间，要使处理时间窗口反映这些事件实际发生的时间，数据必须按事件时间顺序到达。** 不幸的是，按事件时间有序的数据在许多真实世界的分布式输入源中并不常见。

举个简单例子，想象任何收集使用统计以供后续处理的移动应用。当给定移动设备离线任意时间（短暂断网、飞越全国时的飞行模式等），该期间记录的数据要等到设备重新上线才会上传。这意味着数据可能带着分钟、小时、天、周或更长时间的事件时间偏斜到达。对这样的数据集按处理时间窗口化，基本上无法得出任何有用的推断。

再举一例，许多分布式输入源在整体系统健康时，可能看似提供按事件时间有序（或非常接近）的数据。不幸的是，输入源健康时事件时间偏斜低，并不意味着它会一直保持。考虑一个处理在多个大陆收集数据的全球服务。若带宽受限的跨大陆线路出现网络问题（遗憾的是相当常见）进一步降低带宽和/或增加延迟，你的部分输入数据可能突然开始以比以前大得多的偏斜到达。若你按处理时间对这些数据窗口化，窗口将不再代表其中实际发生的数据，而是代表事件到达处理管道的时间窗口，即旧数据与当前数据的任意混合。

在这两种情况下，我们真正想要的是**按事件时间窗口化**，以对事件到达顺序具有鲁棒性的方式处理数据。

**按事件时间窗口化（Windowing by event time）**

当你需要以反映事件实际发生时间的有限块观察数据源时，使用事件时间窗口化。这是窗口化的黄金标准。2016 年之前，大多数使用的数据处理系统缺乏对它的原生支持（尽管任何具有良好一致性模型的系统，如 Hadoop 或 Spark Streaming 1.x，都可以作为构建此类窗口化系统的合理基底）。我很高兴地说，今天的世界已大不相同，从 Flink 到 Spark、Storm、Apex，多个系统都以某种形式原生支持事件时间窗口化。

图 1-10 展示了将无界源窗口化为一小时固定窗口的例子。

::: info 图 1-10 按事件时间窗口化为固定窗口
数据根据发生时间被收集到窗口中。黑色箭头标出与所属事件时间窗口不同的处理时间窗口中到达的示例数据。
:::

图 1-10 中的黑色箭头标出两条特别有趣的数据。每条都到达了与所属事件时间窗口不匹配的处理时间窗口。因此，若这些数据在关心事件时间的用例中被窗口化到处理时间窗口，计算结果将是错误的。正如你所料，使用事件时间窗口的一个好处是事件时间正确性。

在无界数据源上使用事件时间窗口化的另一个好处是，可以创建动态大小的窗口（如会话），而不会出现我们在「无界数据：流式」的会话例子中看到的、在固定窗口上生成会话时的任意分割，如图 1-11 所示。

::: info 图 1-11 按事件时间窗口化为会话窗口
数据被收集到基于对应事件发生时间捕获活动突发的会话窗口中。黑色箭头再次标出将数据放入正确事件时间位置所需的临时 shuffle。
:::

当然，强大的语义很少免费获得，事件时间窗口也不例外。由于窗口通常必须在处理时间上比窗口本身的实际长度存活更久，事件时间窗口有两个显著缺点：

1. **缓冲（Buffering）**：由于窗口生命周期延长，需要更多数据缓冲。值得庆幸的是，持久存储通常是大多数数据处理系统依赖的资源类型中最便宜的（其他主要是 CPU、网络带宽和 RAM）。因此，在使用任何具有强一致持久状态和良好内存缓存层的设计良好的数据处理系统时，这个问题通常远没有你想象的那么严重。此外，许多有用的聚合不需要缓冲整个输入集（例如求和或平均），而是可以增量执行，在持久状态中存储更小的中间聚合。

2. **完整性（Completeness）**：鉴于我们往往无法很好地知道何时已看到给定窗口的所有数据，如何知道窗口的结果何时可以物化？说实话，我们 simply 不知道。对于许多类型的输入，系统可以通过类似 MillWheel、Cloud Dataflow 和 Flink 中的水位线（watermarks）给出窗口完成的相当准确的启发式估计（我们会在第 3、4 章进一步讨论）。但对于绝对正确性至关重要的用例（再次，想想计费），唯一真正的选择是为管道构建者提供一种方式，表达他们希望何时物化窗口结果，以及这些结果应如何随时间细化。

处理窗口完整性（或缺乏完整性）是一个引人入胜的话题，但或许最好在具体示例的上下文中探讨，我们接下来会看到。

## 小结

哇！信息量很大。若你读到这里，值得表扬！但我们才刚刚开始。在深入探讨 Beam Model 方法之前，先简要回顾一下我们学到的内容。本章我们：

- **澄清了术语**，将「streaming」的定义聚焦于面向无界数据构建的系统，而对常被归入「streaming」伞下的不同概念使用更描述性的术语（如近似/推测性结果）。此外，我们强调了大规模数据集的两个重要维度：基数（即有界 vs 无界）和编码（即表 vs 流），后者将占据本书后半部分大量篇幅。
- **评估了设计良好的批处理和流式系统的相对能力**，提出流式实际上是批处理的严格超集，而像 Lambda 架构这样以流式劣于批处理为前提的概念，将随着流式系统的成熟而退出历史舞台。
- **提出了流式系统追赶并最终超越批处理所需的两个高层概念**：正确性和用于推理时间的工具。
- **确立了事件时间与处理时间之间的重要差异**，描述了这些差异在按发生时间分析数据时带来的困难，并建议将方法从完整性概念转向 simply 适应数据随时间的变化。
- **审视了当今有界和无界数据处理的主要方法**，通过批处理和流式引擎，大致将无界方法分为：时间无关、近似、按处理时间窗口化、按事件时间窗口化。

接下来，我们将深入 Beam Model 的细节，从概念上看看我们如何将数据处理的概念分解到四个相关轴：what、where、when、how。我们还将详细审视在多种场景下处理一个简单的具体示例数据集，突出 Beam Model 所支持的多种用例，并用一些具体 API 将我们拉回现实。这些示例将巩固本章介绍的事件时间和处理时间概念，并进一步探索水位线等新概念。
