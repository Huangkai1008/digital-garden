# 第2章 数据处理的 What, Where, When, How

好了，是时候具体起来了！

第 1 章聚焦三个主要领域：术语，精确定义「streaming」等重载术语的含义；批处理 vs 流式，比较两类系统的理论能力，并提出流式系统超越批处理只需两件事：正确性和用于推理时间的工具；以及数据处理模式，审视批处理和流式系统在处理有界和无界数据时采用的概念方法。

本章将进一步聚焦第 1 章的数据处理模式，但更详细，并在具体示例的上下文中展开。结束时，我们将覆盖我认为稳健乱序（out-of-order）数据处理所需的核心原则和概念；这些是真正让你超越经典批处理的推理时间工具。

为让你感受实际效果，我会使用 Apache Beam 代码片段，配合延时图（time-lapse diagrams）提供概念的可视化表示。Apache Beam 是批处理和流式处理的统一编程模型与可移植层，提供多种语言的 SDK（如 Java、Python）。用 Apache Beam 编写的管道可在任何支持的执行引擎（Apache Apex、Apache Flink、Apache Spark、Cloud Dataflow 等）上可移植运行。

我在此使用 Apache Beam 并非因为这是 Beam 书（不是），而是因为它最完整地体现了本书描述的概念。在「Streaming 102」最初撰写时（当时还是 Google Cloud Dataflow 的 Dataflow Model，而非 Apache Beam 的 Beam Model），它 literally 是唯一能提供我们即将覆盖的所有示例所需表达能力的系统。一年半后，我很高兴地说情况已大为改观，大多数主流系统已经或正在向支持与本书描述非常相似的模型迈进。因此可以放心，我们在此覆盖的概念，尽管通过 Beam 视角呈现，同样适用于你将遇到的大多数其他系统。

## 路线图

为本章铺垫，我想列出将支撑其中所有讨论（以及第 I 部分大部分内容）的五个主要概念。其中两个我们已讨论过。

在第 1 章，我首先确立了**事件时间**（事件发生的时间）与**处理时间**（处理过程中被观察到的时间）之间的关键区别。这为本书提出的主要论点之一奠定基础：若你既关心正确性又关心事件实际发生的上下文，就必须相对于其固有事件时间分析数据，而非分析本身遇到数据时的处理时间。

然后我引入了**窗口化**（windowing）概念，即沿时间边界对数据集分区，是应对无界数据源技术上可能永无止境这一事实的常用方法。窗口化策略的简单例子包括固定窗口和滑动窗口，但更复杂的类型如会话（窗口由数据本身特征定义，例如捕获每个用户的活动会话后跟不活动间隔）也被广泛使用。

除这两个概念外，我们还将深入探讨三个新概念：

| 概念 | 含义 |
|------|------|
| **触发器（Triggers）** | 声明相对于某种外部信号，窗口输出应在何时物化的机制。触发器在选择何时发出输出方面提供灵活性。某种意义上，可将其视为规定何时物化结果的流控机制。另一种理解是：触发器像相机的快门，允许你声明何时对正在计算的结果进行时间快照。触发器还使你能多次观察窗口输出的演化，从而为随时间细化结果打开大门，支持在数据到达时提供推测性结果，以及处理上游数据随时间的变化（修订）或迟到数据。 |
| **水位线（Watermarks）** | 相对于事件时间的输入完整性概念。值为时间 X 的水位线表示：「所有事件时间小于 X 的输入数据已被观察到。」因此，水位线在观察无已知终点的无界数据源时充当进度指标。本章会触及水位线基础，Slava 在第 3 章会深入探讨。 |
| **累积（Accumulation）** | 累积模式规定同一窗口多次观察到的结果之间的关系。这些结果可能完全不相交（即表示随时间独立的增量），也可能有重叠。不同累积模式有不同的语义和成本，因而适用于多种用例。 |

此外，为便于理解这些概念之间的关系，我们将在回答四个问题的框架内回顾旧概念、探索新概念。我认为这四个问题对每个无界数据处理问题都至关重要：

1. **What**：计算什么结果？由管道内的变换类型回答。包括求和、构建直方图、训练机器学习模型等。Essentially 也是经典批处理回答的问题。
2. **Where**：在事件时间的何处计算结果？由管道内事件时间窗口化的使用回答。包括第 1 章的常见窗口化例子（固定、滑动、会话）；看似无窗口化概念的用例（如时间无关处理；经典批处理通常也属此类）；以及更复杂的窗口化类型，如限时拍卖。
3. **When**：在处理时间的何时物化结果？由触发器及（可选）水位线的使用回答。有无限种变化，但最常见模式包括：重复更新（即物化视图语义）；利用水位线在相信对应输入完整后才为每个窗口提供单一输出（即按窗口应用的经典批处理语义）；或两者组合。
4. **How**：结果的细化之间有何关系？由累积类型回答：丢弃（结果均独立且不同）、累积（后续结果基于先前结果）、或累积并撤回（发出累积值加上对先前触发值的撤回）。

## 批处理基础：What 与 Where

### What：变换（Transformations）

经典批处理中应用的变换回答「计算什么结果？」。尽管你可能已熟悉经典批处理，我们仍从它开始，因为它是我们添加所有其他概念的基础。

本章（以及本书大部分）我们将看一个单一示例：在由九个值组成的简单数据集上计算按键整数求和。假设我们开发了一款基于团队的移动游戏，希望构建一个管道，通过累加用户手机上报的分数来计算团队得分。若将九个示例分数存入名为 `UserScores` 的 SQL 表，可能如下：

```
| Name  | Team  | Score | EventTime | ProcTime  |
|-------|-------|-------|-----------|-----------|
| Julie | TeamX | 5     | 12:00:26  | 12:05:19  |
| Frank | TeamX | 9     | 12:01:26  | 12:08:19  |
| Ed    | TeamX | 7     | 12:02:26  | 12:05:39  |
| ...   | ...   | ...   | ...       | ...       |
```

按团队分组时，我们主要关心后三列：`Score`（分数）、`EventTime`（事件时间）、`ProcTime`（处理时间）。

**Example 2-1. 求和管道**

```java
PCollection<String> raw = IO.read(...);
PCollection<KV<Team, Integer>> input = raw.apply(new ParseFn());
PCollection<KV<Team, Integer>> totals = input.apply(Sum.integersPerKey());
```

从 I/O 源读取键值数据，Team 为键，Integer 为值。每个键的值被求和，在输出集合中生成按键的和（如团队总分）。

::: info 图 2-3 经典批处理
批处理管道会累积状态直到看到所有输入（顶部绿色虚线表示），此时产生单一输出 48。此例中我们在整个事件时间上计算和，因为未应用任何窗口化变换；故状态和输出矩形覆盖整个 x 轴。
:::

### Where：窗口化（Windowing）

如第 1 章所述，窗口化是沿时间边界切分数据源的过程。常见策略包括固定窗口、滑动窗口、会话窗口。

**Example 2-2. 窗口化求和代码**

```java
PCollection<KV<Team, Integer>> totals = input
  .apply(Window.into(FixedWindows.of(TWO_MINUTES)))
  .apply(Sum.integersPerKey());
```

在批处理引擎上执行时，得到四个输出，对应四个两分钟事件时间窗口。

## 走向流式：When 与 How

### When：触发器（Triggers）

触发器回答「在处理时间的何时物化结果？」。触发器声明窗口输出应在处理时间的何时发生。窗口的每次具体输出称为窗口的一个**窗格（pane）**。

概念上有两种普遍有用的触发器类型：

| 类型 | 描述 |
|------|------|
| **重复更新触发器** | 随窗口内容演化定期生成更新的窗格。可每来一条记录物化，或按处理时间延迟（如每分钟）触发。 |
| **完整性触发器** | 仅在相信窗口输入达到某阈值完整后才物化窗格。最接近批处理：输入完整后才提供结果。 |

**Example 2-3. 每条记录重复触发**

```java
PCollection<KV<Team, Integer>> totals = input
  .apply(Window.into(FixedWindows.of(TWO_MINUTES))
    .triggering(Repeatedly(AfterCount(1))))
  .apply(Sum.integersPerKey());
```

在流式引擎上运行，每个窗口会得到多个输出（窗格），每条对应输入一次。

**Example 2-4/2-5. 对齐与非对齐延迟**

- **对齐延迟**：类似 Spark Streaming 的微批，所有修改的窗口同时更新，导致突发负载。
- **非对齐延迟**：负载在时间上更均匀分布，大规模处理通常更优。

### When：水位线（Watermarks）

水位线是事件时间域中输入完整性的时间概念。值为时间 X 的水位线表示：「所有事件时间小于 X 的输入数据已被观察到。」可将其视为函数 $F(P) \to E$，将处理时间点映射到事件时间点。

| 类型 | 描述 |
|------|------|
| **完美水位线（Perfect watermarks）** | 对输入有完美知识时可构造。无迟到数据。 |
| **启发式水位线（Heuristic watermarks）** | 对多数分布式输入源，完美知识不现实，使用可用信息（分区、分区内顺序、文件增长率等）估计进度。可能出错，导致迟到数据。 |

**Example 2-6. 水位线完整性触发器**

```java
PCollection<KV<Team, Integer>> totals = input
  .apply(Window.into(FixedWindows.of(TWO_MINUTES))
    .triggering(AfterWatermark()))
  .apply(Sum.integersPerKey());
```

水位线的两个缺点：
- **太慢**：正确延迟时，若仅依赖水位线推进来刺激结果，会直接导致输出延迟。
- **太快**：启发式水位线过早推进时，可能产生迟到数据，导致错误输出。

### When：早/准时/晚触发器

将重复更新触发器与水位线触发器结合：

- **零或多个早窗格**：水位线越过窗口末端前的重复更新，含推测性结果。
- **单一准时窗格**：水位线越过窗口末端后触发，断言输入完整。
- **零或多个晚窗格**：迟到数据到达时的重复更新。完美水位线下晚窗格恒为零。

**Example 2-7. 早/准时/晚触发**

```java
PCollection<KV<Team, Integer>> totals = input
  .apply(Window.into(FixedWindows.of(TWO_MINUTES))
    .triggering(AfterWatermark()
      .withEarlyFirings(AlignedDelay(ONE_MINUTE))
      .withLateFirings(AfterCount(1))))
  .apply(Sum.integersPerKey());
```

### When：允许迟到（Allowed Lateness）与垃圾回收

长寿命乱序流处理系统需要垃圾回收。通过定义**允许迟到范围（allowed lateness horizon）**：系统会处理该范围内迟到的记录，超出范围的直接丢弃。窗口状态需保留到水位线超过窗口末端加迟到范围。

**Example 2-8. 带允许迟到的早/准时/晚触发**

```java
PCollection<KV<Team, Integer>> totals = input
  .apply(Window.into(FixedWindows.of(TWO_MINUTES))
    .triggering(AfterWatermark()
      .withEarlyFirings(AlignedDelay(ONE_MINUTE))
      .withLateFirings(AfterCount(1)))
    .withAllowedLateness(ONE_MINUTE))
  .apply(Sum.integersPerKey());
```

### How：累积（Accumulation）

当触发器为同一窗口随时间产生多个窗格时，需回答「结果的细化之间有何关系？」。有三种累积模式：

| 模式 | 描述 |
|------|------|
| **丢弃（Discarding）** | 每次物化窗格时丢弃存储状态。每个窗格独立。适用于下游自行累积（如接收增量并求和的系统）。 |
| **累积（Accumulating）** | 保留状态，后续输入累积到现有状态。每个窗格基于先前窗格。适用于新值可直接覆盖旧值（如 HBase、Bigtable）。 |
| **累积并撤回（Accumulating and retracting）** | 产生新窗格时，同时产生对先前窗格的撤回。适用于下游按不同维度重分组，或动态窗口（如会话）合并时。 |

**表 2-1 累积模式比较（第二窗口 [12:06, 12:08)）**

| 模式 | Pane 1 | Pane 2 | 最终值 | 所有窗格之和 |
|------|--------|--------|--------|--------------|
| 丢弃 | 3 | 9 | 9 | 12 |
| 累积 | 3 | 12 | 12 | 15 |
| 累积并撤回 | 3 | 12, -3 | 12 | 12 |

## 小结

本章涉及的主要概念：

- **事件时间 vs 处理时间**：事件发生时间与系统观察时间的区别
- **窗口化**：沿时间边界切分无界数据的常用方法
- **触发器**：声明何时物化输出的机制
- **水位线**：事件时间进度概念，支持推理完整性
- **累积**：同一窗口多次物化时结果细化之间的关系

四个问题框架：

- **What** = 变换
- **Where** = 窗口化
- **When** = 触发器 + 水位线
- **How** = 累积

同一数据集上，通过少量代码变更即可实现多种输出变体，体现了该流处理模型在正确性、延迟、成本之间权衡的灵活性。
