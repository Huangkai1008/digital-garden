# 第3章 水位线

此前我们从管道作者或数据科学家的视角看待流式处理。第 2 章将水位线（watermarks）作为回答「在事件时间的何处进行处理」和「在处理时间的何时物化结果」的一部分引入。本章从流式处理系统的底层机制视角探讨同样的问题，以帮助理解、应用水位线相关概念。我们将讨论水位线如何在数据入口处创建、如何在管道中传播、如何影响输出时间戳，并展示水位线如何保持回答上述问题所需的保证，同时应对无界数据。

---

## 定义

考虑任何持续摄入数据并输出结果的管道。我们希望解决一个一般性问题：何时可以安全地认为事件时间窗口已关闭（即该窗口不再期望更多数据）。为此，需要刻画管道相对于其无界输入的进度。

一种朴素做法是直接基于当前处理时间定义事件时间窗口。如第 1 章所见，这会很快出问题——数据处理和传输不是瞬时的，处理时间与事件时间几乎从不相等。管道的任何波动都可能导致错误地将消息分配到窗口。另一种直观但最终错误的做法是考虑管道处理消息的速率。速率会随输入、预期结果、可用资源等任意变化，且速率无法回答完整性的根本问题：我们何时已看到某时间区间的所有消息。

我们需要的是一种更稳健的进度度量。为此，对流式数据做一个基本假设：**每条消息都有一个关联的逻辑事件时间戳**。在持续到达的无界数据背景下，这假设是合理的。有了事件时间戳，可以考察管道中这些时间戳的分布。管道中活跃的「在途」消息的事件时间戳将形成一个分布，如图 3-1。

::: info 图 3-1 流式管道中在途与已完成消息的事件时间分布
新消息作为输入到达，保持「在途」直到处理完成。在途分布的最左边缘对应任意时刻最老的未处理元素。
:::

该分布上有一个关键点，位于在途分布的**最左边缘**，对应管道中任意未处理消息的**最老事件时间戳**。我们用它定义水位线：

::: info 水位线定义
水位线是**尚未完成的最老工作**的单调递增时间戳。
:::

该定义提供两个使其有用的基本性质：

| 性质 | 含义 |
|------|------|
| **完整性（Completeness）** | 若水位线已越过某时间戳 T，由其单调性可保证，T 及之前准时（非迟到）事件不会再被处理。因此可以正确发出 T 及之前的任意聚合。即水位线让我们知道何时可以正确关闭窗口。 |
| **可见性（Visibility）** | 若某条消息因任何原因卡在管道中，水位线无法前进。通过检查阻止水位线前进的消息，可以定位问题来源。 |

## 源水位线创建

水位线从何而来？要为数据源建立水位线，必须为从该源进入管道的每条消息分配逻辑事件时间戳。如第 2 章所述，所有水位线创建分为两类：**完美（perfect）** 或 **启发式（heuristic）**。

### 完美水位线创建

完美水位线创建以严格保证「不会再看到事件时间小于水位线的数据」的方式为到达消息分配时间戳。使用完美水位线的管道无需处理迟到数据。但完美水位线需要输入的完美知识，对许多真实分布式输入源不现实。可创建完美水位线的用例包括：

- **入口时间戳（Ingress timestamping）**：将入口时间作为事件时间分配。源水位线 simply 跟踪管道观察到的当前处理时间。2016 年前几乎所有支持窗口化的流式系统都采用此方式。
- **静态时间有序日志集**：静态大小的、按时间排序的日志输入源（如具有静态分区集的 Apache Kafka topic，每个分区内事件时间单调递增），可在其上相对直接地创建完美水位线。源只需跟踪已知静态分区集中未处理数据的最小事件时间。

### 启发式水位线创建

启发式水位线创建产生的水位线仅是「不会再看到事件时间小于水位线的数据」的**估计**。使用启发式水位线的管道可能需要处理一定量的迟到数据。迟到数据是指在水位线已越过其事件时间后才到达的数据。仅启发式水位线可能产生迟到数据。

对许多真实分布式输入源，构造完美水位线在计算或运维上不现实，但可利用输入源的结构特征构建高度准确的启发式水位线。例如：

- **动态时间有序日志集**：动态的结构化日志文件集（每个文件内记录事件时间相对单调递增，但文件间无固定关系），预期完整文件集在运行时未知。常见于由多个独立团队构建和管理的全球规模服务。创建完美水位线不可行，但创建准确的启发式水位线是可行的。
- **Google Cloud Pub/Sub**：Pub/Sub 不保证顺序交付，无法保证完美水位线。Cloud Dataflow 团队利用可用知识构建了相当准确的启发式水位线，本章后文有案例研究。

## 水位线传播

大多数真实管道由多个阶段组成。理解水位线如何在独立阶段间传播，对理解其对管道整体及结果延迟的影响至关重要。

### 管道阶段

每当管道按新维度对数据分组时，通常需要不同阶段。例如：消费原始数据 → 按用户分组并计算每用户聚合 → 按团队分组并计算每团队聚合，会形成三阶段管道。

水位线在输入源创建，然后随数据在系统中流动而概念上流过。可为管道的任意单操作或阶段边界定义水位线：

| 水位线类型 | 含义 |
|------------|------|
| **输入水位线（Input watermark）** | 捕获该阶段上游一切的进度（即该阶段输入的完整程度）。对源，输入水位线是源特定的函数；对非源阶段，输入水位线定义为所有上游源和阶段的所有分片/分区/实例的输出水位线的最小值。 |
| **输出水位线（Output watermark）** | 捕获阶段自身的进度，定义为该阶段输入水位线与阶段内所有非迟到活跃消息的事件时间的最小值。 |

定义输入和输出水位线的一个好处是，可用它们计算阶段引入的事件时间延迟：输出水位线减去输入水位线即为该阶段引入的延迟或滞后。

---

## 小结

水位线是流式处理中推理输入完整性的核心机制。完美水位线需要输入的完美知识；启发式水位线是估计，可能产生迟到数据。水位线在源处创建，沿管道传播，每个阶段的输出水位线由其输入水位线和阶段内活跃数据共同决定。完整性性质使我们可以安全关闭窗口；可见性性质帮助诊断卡住的消息。
