# 第5章 精确一次与副作用

本章从编程模型和 API 转向实现它们的系统。模型和 API 允许用户描述要计算的内容；准确、大规模地实际执行计算则需要系统——通常是分布式系统。

本章聚焦实现系统如何正确实现 Beam Model 以产生准确结果。流式系统常讨论**精确一次处理（exactly-once processing）**，即确保每条记录被处理恰好一次。我们将解释其含义及实现方式。

作为动机示例，本章聚焦 Google Cloud Dataflow 用于高效保证记录精确一次处理的技术。章末也会简要讨论其他流行流式系统的技术。

---

## 为何精确一次重要

对许多用户而言，数据处理管道中任何丢记录或数据丢失的风险都是不可接受的。历史上，许多通用流式系统不对记录处理做任何保证——所有处理都是「尽力而为」。其他系统提供至少一次（at-least-once）保证，确保记录至少被处理一次，但可能重复（导致不准确的聚合）；实践中，许多此类系统在内存中做聚合，机器崩溃时聚合仍可能丢失。

如第 1 章所述，这导致了 **Lambda 架构**：运行流式系统获得快速但不准确的结果，之后（通常日终）批处理系统运行得到正确结果。Lambda 架构存在诸多问题：不准确、不一致、复杂度高、不可预测、延迟等。幸运的是，许多 Beam runner 可以做得更好。

---

## 准确性与完整性

每当 Beam 管道处理一条记录时，我们希望确保该记录既不被丢弃也不被重复。但流式管道的性质决定了记录有时会迟到——在其时间窗口的聚合已被处理之后才到达。Beam SDK 允许用户配置系统应等待迟到数据多久；超过该期限到达的记录会被显式丢弃。该特性贡献的是**完整性**，而非**准确性**：所有及时到达的记录都被准确处理恰好一次，而迟到记录被显式丢弃。

批处理管道也有类似的完整性问题。例如，常见范式是在凌晨 2 点对前一天的所有数据运行作业；若部分昨日数据在凌晨 2 点后才收集到，就不会被该批处理作业处理。

---

## 副作用（Side Effects）

Beam 和 Dataflow 的一个特点是用户注入的自定义代码作为管道图的一部分执行。Dataflow **不保证**该代码每条记录只运行一次（无论流式还是批处理 runner）。它可能对给定记录多次运行用户变换，甚至在同一记录的多次调用中同时运行；这在面对 worker 故障时为保证至少一次处理所必需。只有其中一次调用可以「获胜」并在管道下游产生输出。

因此，**非幂等副作用不保证精确一次执行**。若你编写了具有管道外部副作用的代码（如联系外部服务），这些效果对给定记录可能执行多次。管道最终需要将结果发送到外部世界，此类调用可能非幂等。如本章后文所述，许多 sink 能够添加额外阶段，先将调用重构为幂等操作。

---

## 问题定义

本章聚焦三件事：

1. **Shuffle**：Dataflow 如何保证每条记录被 shuffle 恰好一次
2. **Sources**：Dataflow 如何保证每条源记录被处理恰好一次
3. **Sinks**：Dataflow 如何保证每个 sink 产生准确输出

---

## Shuffle 中的精确一次保证

Dataflow 的流式 shuffle 使用 RPC。为保证记录在 shuffle 中不丢失，Dataflow 采用**上游备份（upstream backup）**：发送方会重试 RPC 直到收到成功确认，即使发送方崩溃也会继续重试。这保证每条记录至少被传递一次。

问题是重试可能产生重复。RPC 返回失败状态通常表示调用可能成功也可能失败。重试一个实际已成功的 RPC 意味着传递记录两次！Dataflow 需要某种方式检测并移除这些重复。

高层算法（图 5-2）：每条发送的消息带有唯一标识符。每个接收方存储已见过并处理的所有标识符目录。每次收到记录时，在其目录中查找该标识符；若找到，则作为重复丢弃。Dataflow 构建在可扩展键值存储之上，用该存储保存去重目录。

### 应对非确定性

Beam Model 允许用户代码产生非确定性输出。同一输入记录上的 ParDo 可能执行两次（因重试）却产生不同输出。期望行为是只有其中一个输出提交到管道；但非确定性使得难以保证两次输出有相同的确定性 ID。

Dataflow 通过 **checkpointing** 使非确定性处理 effectively 确定性：变换的每个输出在交付到下一阶段之前，连同其唯一 ID 被 checkpoint 到稳定存储。shuffle 传递中的任何重试 simply 重放已 checkpoint 的输出——用户的非确定性代码在重试时不会再次运行。用户代码可能运行多次，但只有一次可以「获胜」。

### 性能优化

- **图优化（Graph optimization）**：Dataflow 在执行前对管道图运行一系列优化。**融合（fusion）** 将多个逻辑步骤融合为单一执行阶段，无需为每个步骤存储精确一次数据。**Combiner 提升**在发送到主分组操作之前本地执行部分合并，可大幅减少传递的消息数。
- **Bloom 过滤器**：健康管道中，大多数到达的记录不是重复的。Dataflow 使用 Bloom 过滤器进行快速集合成员检查。Bloom 过滤器可返回假阳性但从不返回假阴性。每个 worker 为每 10 分钟范围维护独立的 Bloom 过滤器，防止过滤器饱和并限制启动时需扫描的数据量。

### 垃圾回收

每个 Dataflow worker 持久存储其见过的唯一记录 ID 目录。需要垃圾回收已确认的记录 ID，否则存储最终会耗尽。发送方为每条记录附带严格递增的序列号，接收方确认后，发送方可垃圾回收早于仍在途中最老序列号的 ID。

---

## 源与 Sink 的精确一次

- **源**：Dataflow 通过可重放源和 checkpoint 保证每条源记录被处理恰好一次。
- **Sink**：对于非幂等 sink，通常通过添加阶段将写入重构为幂等操作（如先写入临时位置，再原子性提交）。

---

## 小结

精确一次处理需要：shuffle 中的去重与 checkpoint、源的可重放性与精确一次读取、sink 的幂等性或原子性提交。Dataflow 通过上游备份、唯一 ID 去重、checkpointing、图优化、Bloom 过滤器等技术实现高效精确一次处理。
