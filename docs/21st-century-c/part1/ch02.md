# 第2章 调试、测试、文档

> Crawling over your window / You think I'm confused, / I'm waiting... / To complete my current ruse.
> —Wire, "I Am the Fly"

本章涵盖调试、测试和文档化你代码的工具——这些是将你的代码从可能有用的一组脚本提升为你和他人可以依赖的东西的必备要素。

因为 C 给了你对内存做蠢事的自由，调试既包括用 GDB 检查逻辑的日常问题，也包括用 Valgrind 检查内存错误分配和泄漏的更技术性问题。在文档方面，本章涵盖接口层面的工具（Doxygen）以及帮助你在程序每一步进行文档化和开发的工具（CWEB）。本章还简要介绍测试框架（test harness），让你能快速为代码编写大量测试，并讨论错误报告和处理输入或用户错误的若干考量。

## 使用调试器

关于调试器的第一条建议简单而简短：

**始终使用调试器。**

你们中的一些人可能觉得这不是什么建议，因为谁会不用调试器呢？在本书第二版中，我可以告诉你，第一版最常见的请求之一就是更全面地介绍调试器，这对许多读者来说完全是新事物。

有些人担心 bug 通常来自理解上的广泛错误，而调试器只提供变量状态和回溯（backtrace）这种低层次信息。确实，在用调试器定位 bug 后，值得花时间考虑你刚发现的潜在问题和理解上的失败，以及它是否在代码其他地方复现。有些死亡证明会深入追问死因：死者因______而死，因______而死，因______而死……在调试器帮你完成这样的追问并更好地理解代码后，你可以将理解封装到更多单元测试中。

关于「始终」：在调试器下运行程序几乎没有成本。调试器也不只是在出问题时才拿出来用的东西。Linus Torvalds 解释说：「我一直在用 gdb……就像可以编程的超级反汇编器。」能够随时暂停、用 `print verbose++` 快速提高冗长级别、通过 `print i=100` 和 continue 强制跳出 `for (int i=0; i<10; i++)` 循环，或通过向函数抛出一系列测试输入来测试它，这些都很有用。交互式语言的粉丝说得对，与代码交互能全程改善开发过程；他们只是从未读到 C 教科书的调试章节，所以从未意识到所有这些交互习惯也适用于 C。

无论你的意图如何，你都需要将人类可读的调试信息（即变量和函数的名称）编译进程序，调试器才能有用。要包含调试符号，在编译器开关（即 CFLAGS 变量）中使用 `-g` 标志。不使用 `-g` 的理由确实很少——它不会减慢程序，给可执行文件增加一千字节对大多数情况无关紧要。通过 `-O0`（oh zero）编译器标志关闭优化后，调试可能更容易，因为优化器可能消除对调试有用的变量并以令人惊讶的方式重排代码。

我主要介绍 GDB，因为在大多数 POSIX 系统上，它是唯一的选择。（顺便说一下，C++ 编译器会进行所谓的代码混淆（mangling）。在 gdb 中会显示出来，我总觉得从 gdb 提示符调试 C++ 代码很痛苦。因为 C 代码编译时没有混淆，我发现 gdb 对 C 更可用，不需要有解混淆名称的 GUI。）LLDB（LLVM/clang 的配套工具）越来越流行，我也会介绍它。

Apple 已停止在 Xcode 套件中附带 GDB，但你可以通过包管理器（如 Macports、Fink 或 Homebrew）安装。在 Mac 上，你可能需要通过 `sudo` 运行调试会话，如 `sudo lldb stddev_bugged`。

你可能在使用 IDE 或其他图形前端，每次点击运行都会在调试器下运行程序。我将展示命令行命令，你应该能轻松将这里的基础知识转化为屏幕上的鼠标点击。根据前端，你可能能使用 .gdbinit 中定义的宏。

直接使用命令行时，你可能需要在另一个窗口或终端用文本编辑器显示代码。简单的调试器/编辑器组合提供了 IDE 的许多便利，可能就是你需要的全部。

### 帧栈（Stack of Frames）

要启动程序，你让系统执行名为 main 的函数。计算机会生成一个帧（frame），将函数的信息放入其中，如输入（对 main 通常命名为 argc 和 argv）以及函数创建的变量。

假设在执行过程中，main 调用了另一个函数 get_agents。然后 main 的执行停止，为 get_agents 生成新帧，保存其各种细节和变量。也许 get_agents 又调用了 agent_address，此时我们有了不断增长的帧栈。最终 agent_address 完成执行，从栈中弹出，get_agents 恢复执行。

如果你的问题只是「我在哪？」，简单的答案就是代码中的行号，有时这就够了。但更常见的是「我是怎么到这里的？」，答案——回溯（backtrace）或调用栈（call stack）——就是帧栈的列表。

示例回溯：

```
#0  0x00413bbe in agent_address (agent_number=312) at addresses.c:100
#1  0x004148b6 in get_agents () at addresses.c:163
#2  0x00404f9b in main (argc=1, argv=0x7fffffffe278) at addresses.c:227
```

栈顶是帧 0，向下到 main，当前是帧 2（但随着栈的增长和收缩会变化）。帧号后的十六进制给出被调函数返回时执行将返回的位置；作为应用程序员，我总把它当作可忽略的视觉噪音。之后是函数名、其输入（argv 的情况又是十六进制地址）以及执行发生的源代码行。

如果你发现 agent_address 中列出的房子明显错误，那可能是 agent_number 输入有问题，此时你要跳到帧 1 问 get_agents 的状态是什么导致了 agent_address 的奇怪状态。审问程序的大部分技巧就在于在栈中跳转，从一个函数的帧追踪到下一个的因果。

### 调试侦探故事

本节将通过与 GDB 或 LLDB 的虚构问答来演示。在本书的代码示例集中，你会找到 stddev_bugged.c，即插入了 bug 的示例 7-4 重写版。改动很小，你可以参考 stddev.c 的列表了解程序。像任何好的侦探故事一样，识别罪魁祸首所需的线索都在你面前。一系列问题将帮助排除嫌疑，直到只剩一个嫌疑，bug 变得明显。

编译程序后（`CFLAGS="-g" make stddev_bugged` 即可），我们启动调试器开始调查：

```bash
gdb stddev_bugged
# 或
lldb stddev_bugged
```

我们现在在调试器命令提示符下，准备提问。

**问：这个程序做什么？**

**答：** run 命令运行程序。这里 GDB 和 LLDB 命令相同；有差异时我会在示例中用 GDB 命令，将 LLDB 命令放在方括号中。像所有 GDB 和 LLDB 命令一样，可以缩写：

```
(gdb) r
mean: 5687.496667 var: 194085710
mean: 0.83 var: 4.1334
[Inferior 1 (process 22734) exited normally]
```

看起来程序产生一些均值和方差。它运行到程序结束没有段错误或其他失败，返回零表示正常执行。

**问：main 中的代码能验证我们从输出得到的内容吗？**

**答：** 查看代码最简单的方式是直接在文本编辑器中打开源代码。即使登录到仅终端的远程机器，也有办法让文本编辑器与调试器并排；见「试试多路复用器」。但 GDB 和 LLDB 也可以通过 list 命令显示代码行……

通过设置断点、单步执行、检查变量，我们最终发现 bug：第 22 行 `ratio = count/(count+1)` 中，若 count 是整数，则 `count/(count+1)` 是整数除法，返回整数（3/4==0），而不是我们小学学的浮点除法（3/4==0.75）。正确做法是将整数常量 1 改为浮点常量 1.0：`ratio = count/(count+1.0)`。

调试器没有提醒我们这个常见错误，但它帮我们找到了代码中第一个出错的地方，在一行上找错误肯定比在 50 行代码块中找容易。一路上，我们检查验证了代码的各种细节，对程序流程和帧栈有了更好理解。

### 常用调试器命令

| 类别 | 命令 | 含义 |
|------|------|------|
| 运行 | `run` | 从头运行程序 |
| 运行 | `run args` | 用给定命令行参数从头运行 |
| 停止 | `b get_rss` | 在某个函数处暂停 |
| 停止 | `b nyt_feeds.c:105` | 在某行代码前暂停 |
| 检查 | `p url` | 打印 url 的值 |
| 检查 | `p *an_array@10` [GDB] | 打印 an_array 前 10 个元素 |

| 帧 | `f 3` | 查看帧 3 |
| 单步 | `s` | 单步一行，可进入子函数 |
| 单步 | `n` | 下一行，不进入子函数 |
| 单步 | `c` | 继续到下一断点或程序结束 |

## GDB 变量

你可以生成便利变量（convenience variables）来节省输入。例如，若要检查结构层次深处的元素：

```
(gdb) set $vd = my_model->dataset->vector->data
p *$vd@10
```

这些不仅是名称；它们是你可以修改的真实变量。在 do-nothing 程序的第 3 或 4 行断点后，试试：

```
(gdb) set $ptr=&x[3]
p *$ptr = 8
p *($ptr++)  # 打印所指值，并前进一步
```

GDB 使用单独的 `$` 作为赋给最后输出的简写变量。所以如果你得到十六进制地址而期望得到该地址的值，下一行放 `p *$` 即可得到值。

### 打印你的结构体

你可以定义简单宏，对显示非平凡数据结构特别有用——这是调试器中的大部分工作。即使是简单的二维数组，显示成一长串数字也会伤眼。在完美世界里，你处理的每个主要结构都会有相关的调试器命令，以对你最有用的方式快速查看。

## 性能分析（Profiling）

程序多快都不够：你总会想要更快。在大多数语言中，第一条建议是用 C 重写一切，但你已经用 C 写了。下一步是找到占用最多时间的函数，从而为更多优化努力提供最大回报。

首先，将 `-pg` 标志添加到 gcc 或 icc 的 CFLAGS。运行程序后，调用 `gprof your_program > profile`，然后在文本编辑器中打开 profile 查看函数、调用以及每个函数占用程序时间百分比的人类可读列表。你可能会对瓶颈所在感到惊讶。

## 用 Valgrind 检查错误

我们调试的大部分时间花在找程序中第一个看起来不对的地方。好的代码和好的系统会为你找到那个点。也就是说，好的系统快速失败（fail fast）。

C 在这方面得分参差不齐。在某些语言中，像 `conut=15` 这样的笔误会生成与你要设置的 count 无关的新变量；在 C 中，它在编译步骤就失败了。另一方面，C 会让你给 9 元素数组的第 10 个元素赋值，然后运行很久才发现你以为的第 10 个元素里是垃圾。

这些内存管理问题是麻烦，所以有工具来应对。其中 Valgrind 是大赢家。它移植到大多数 POSIX 系统（包括 OS X），你可以通过包管理器获取。Windows 用户可能想试试 Dr. Memory。

编译好程序后（当然用 gcc 或 clang 的 `-g` 标志包含调试符号），运行：

```bash
valgrind your_program
```

若有错误，Valgrind 会给你两个看起来很像调试器回溯的回溯。第一个是误用首次被检测到的地方，第二个是 Valgrind 对误用冲突行的最佳猜测，如双重释放块首次释放的位置，或最近 malloc 块分配的位置。

你也可以在第一个错误时启动调试器：

```bash
valgrind --db-attach=yes your_program
```

Valgrind 也做内存泄漏检查：

```bash
valgrind --leak-check=full your_program
```

## 单元测试

当然你在为代码写测试。你为较小组件写单元测试，写集成测试确保组件和睦相处。你甚至可能是先写单元测试再构建程序通过测试的那种人。

现在你有了组织所有这些测试的问题，这就是测试框架（test harness）的用武之地。测试框架是为每个测试设置小环境、运行测试并报告结果是否符合预期的系统。

测试框架有几个优于典型自制测试宏的特性：

- 你需要测试失败。若函数应 abort 或带错误消息退出，你需要能测试程序确实在你预期时退出的设施。
- 每个测试保持独立，不必担心测试 3 影响测试 4 的结果。
- 运行测试前可能需要构建一些数据结构。为测试设置场景有时需要大量工作，所以能在相同设置下运行多个测试会很好。

**示例 2-3. 字典的单元测试（dict_test.c）**

```c
#include <glib.h>
#include "dict.h"
typedef struct {                                         
    dictionary *dd;
} dfixture;
void dict_setup(dfixture *df, gconstpointer test_data){  
    df->dd = dictionary_new();
    dictionary_add(df->dd, "key1", "val1");
    dictionary_add(df->dd, "key2", NULL);
}
void dict_teardown(dfixture *df, gconstpointer test_data){
    dictionary_free(df->dd);
}
void check_keys(dictionary const *d){                    
    char *got_it = dictionary_find(d, "xx");
    g_assert(got_it == dictionary_not_found);
    got_it = dictionary_find(d, "key1");
    g_assert_cmpstr(got_it, ==, "val1");
    got_it = dictionary_find(d, "key2");
    g_assert_cmpstr(got_it, ==, NULL);
}
void test_new(dfixture *df, gconstpointer ignored){
    check_keys(df->dd);
}
void test_copy(dfixture *df, gconstpointer ignored){
    dictionary *cp = dictionary_copy(df->dd);
    check_keys(cp);
    dictionary_free(cp);
}
void test_failure(){                                     
    if (g_test_trap_fork(0, G_TEST_TRAP_SILENCE_STDOUT | 
    G_TEST_TRAP_SILENCE_STDERR)){
        dictionary *dd = dictionary_new();
        dictionary_add(dd, NULL, "blank");
    }
    g_test_trap_assert_failed();
    g_test_trap_assert_stderr("NULL is not a valid key.\n");
}
int main(int argc, char **argv){
    g_test_init(&argc, &argv, NULL);
    g_test_add ("/set1/new test", dfixture, NULL,        
                                  dict_setup, test_new, dict_teardown);
    g_test_add ("/set1/copy test", dfixture, NULL,
                                  dict_setup, test_copy, dict_teardown);
    g_test_add_func ("/set2/fail test", test_failure);   
    return g_test_run();
}
```

### 将程序用作库

函数库和程序之间的唯一区别是程序包含指示执行起点的 main 函数。

通过预处理器条件，可以将测试放在与其余内容相同的文件中。若定义了 Test_operations，则片段是运行测试的程序；若未定义（通常情况），则片段编译时不包含 main，因此是供其他程序使用的库。

在 makefile 中添加：

```makefile
CFLAGS=-DTest_operations
```

## 覆盖率（Coverage）

你的测试覆盖率如何？有没有你写的代码行未被测试触及？gcc 有配套的 gcov，会统计程序触及每行代码的次数。

步骤：在 CFLAGS 中添加 `-fprofile-arcs -ftest-coverage`；运行程序；运行 `gcov yourcode.gcda` 会输出可运行代码行的命中百分比，并生成 yourcode.c.cov。

## 错误检查

关于如何从你写的函数返回错误，需要考虑几个子情况：

- 用户将如何处理错误消息？
- 接收者是人还是另一个函数？
- 如何将错误传达给用户？

**示例 2-5. 处理错误的宏（stopif.h）**

```c
#include <stdio.h>
#include <stdlib.h> //abort
char error_mode;
FILE *error_log;
#define Stopif(assertion, error_action, ...) {                    \
        if (assertion){                                           \
            fprintf(error_log ? error_log : stderr, __VA_ARGS__); \
            fprintf(error_log ? error_log : stderr, "\n");        \
            if (error_mode=='s') abort();                         \
            else                 {error_action;}                  \
        } }
```

示例用法：

```c
Stopif(!inval, return -1, "inval must not be NULL");
Stopif(isnan(calced_val), goto nanval, "Calced_val was NaN. Cleaning up, leaving.");
```

## 交织文档

你需要文档。你知道这一点，也知道代码变化时要保持更新。然而，文档往往是首先被搁置的。说「它能跑；我稍后文档化」太容易了。

所以你需要让写文档尽可能容易。直接含义是让代码的文档与代码在同一文件中，尽可能靠近被文档化的代码，这意味着你需要从代码文件中提取文档的手段。

### Doxygen

Doxygen 是目标简单的简单系统。它最适合为每个函数、结构体或其他块附加描述。这是为永远不会看代码本身的用户文档化接口的情况。描述就在函数、结构体或 whatever 正上方的注释块中，所以容易先写文档注释，再写函数兑现你刚做的承诺。

Doxygen 语法足够简单：

- 若注释块以两个星号开头，`/** 像这样 */`，则 Doxygen 会解析。单星注释 `/* 像这样 */` 被忽略。
- 若要让 Doxygen 解析文件，需要在文件头部有 `/** \file */` 注释。
- 将注释放在函数、结构体等正前方。
- 函数描述可（且应）包含描述输入参数的 `\param` 段和列出预期返回值的 `\return` 行。

运行 `doxygen -g` 生成配置文件，然后运行 doxygen 生成 HTML、PDF、XML 或手册页等输出。

### 用 CWEB 进行文学编程（Literate Programming）

TeX 常被奉为复杂系统做得很好的典范。Knuth 博士通过讨论 TeX 的编写方式解释其高质量：文学编程（literate programming），其中每个过程块前都有该块目的和功能的纯英文解释。CWEB 将英文说明性文本与 C 代码交织。

CWEB 要点：

- 每个 CWEB 特殊代码都有 @ 后跟单字符。
- 每段有注释，然后代码。
- 用 @ 后跟空格开始文本段。
- 用 @c 开始未命名代码块。
- 用标题后跟等号开始命名代码块：`@<an operation@>=`。

## 成为更好的打字员

对键盘不自信的人不愿使用像 `for (i=0; i<10; i++)` 这样符号密集的文本是标准用法的语言。

建议：拿一件轻 T 恤盖在键盘上。把手伸到 T 恤下面，开始打字。目的是阻止我们都会做的偷瞄键位。事实证明键不会乱动，总在你离开时的位置。但那些检查的微停顿让我们把对键盘的信心和熟练度保持在某个安全速度。
