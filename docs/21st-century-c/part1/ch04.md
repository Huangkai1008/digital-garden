# 第4章 版本控制

> Look at the world through your Polaroid glasses / Things'll look a whole lot better for the working classes.
> —Gang of Four, "I Found that Essence Rare"

本章介绍版本控制系统（RCS），它维护项目发展过程中许多不同版本的快照，如书籍、情书或程序的开发阶段。版本控制系统赋予我们重要能力：

- **时间维度**：可以查询 RCS 仓库，查看文件上周的样子以及从那时到现在的变化
- **多版本跟踪**：跟踪项目的多个版本，如我的副本和合著者的副本；实验性分支与稳定版本隔离
- **克隆与分发**：GitHub 有约 21.8 万个自报主要为 C 的项目；克隆仓库是快速获取程序或库的方式
- **合并**：当多人都有同一项目的版本且都能修改时，版本控制让我们能尽可能轻松地合并多条线

本章介绍 Git，它是一个分布式版本控制系统，意味着任何项目副本都可作为项目及其历史的独立仓库工作。

## 通过 diff 进行变更

最基础的版本控制手段是 diff 和 patch，它们是 POSIX 标准的。对两个目录 v1 和 v2 生成统一 diff：

```bash
diff -ur v1 v2 > diff-v1v2
```

应用补丁：

```bash
patch < diff-v1v2
```

反向应用（回退）：

```bash
patch -R < diff-v1v2
```

## Git 的对象

Git 基于一小组对象。关键对象是提交对象（commit object），类似于统一 diff 文件。每个提交对象有（至少）一个父提交对象，链接形成树。

**初始化仓库：**

```bash
git init
```

**克隆仓库：**

```bash
git clone https://github.com/gitster/git.git
```

**添加文件到索引并提交：**

```bash
git add gnomes.c gnomes.h
git commit -a -m "here is an initial commit."
```

`-a` 标志让 Git 在提交前自动运行 `git add -u`，将已跟踪文件的变更加入索引。

## Diff/快照二元性

提交对象有时最好视为某时刻项目的完整快照，有时视为相对其父级的 diff。无论哪种视角，它都包含作者记录、对象名称（通过 -m 附加的消息）、以及（除非是初始提交）指向父提交对象的指针。

对象名称通过 SHA1 哈希分配——40 位十六进制数。只需足够唯一标识提交的哈希部分即可：

```bash
git checkout fe9c4
```

## 分支与合并

创建分支：

```bash
git branch new_feature
git checkout new_feature
# 或一行：git checkout -b new_feature
```

合并分支：

```bash
git checkout master
git merge new_feature
```

变基（rebase）将提交序列重新应用到新基础：

```bash
git rebase master
```

## 远程仓库

添加远程：

```bash
git remote add origin https://github.com/user/repo.git
```

推送与拉取：

```bash
git push origin master
git pull origin master
```
