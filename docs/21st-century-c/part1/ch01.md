# 第1章 编译环境搭建

> Look out honey 'cause I'm using technology.
> —Iggy Pop, "Search and Destroy"

仅靠 C 标准库不足以完成严肃的工作。C 生态系统已扩展到标准之外，这意味着如果你想超越教科书练习，就必须知道如何轻松调用常见但非 ISO 标准的库中的函数。如果你想处理 XML 文件、JPEG 图像或 TIFF 文件，就需要 libxml、libjpeg 或 libtiff——它们都是免费可用的，但不属于标准库。不幸的是，大多数教科书正是在这一点上戛然而止，让你自己去摸索，这就是为什么你能找到 C 语言的批评者说出自相矛盾的话，比如「C 已经 40 岁了，所以你必须在其中从头编写一切」——他们从未搞明白如何链接库。

本章议程如下：

## 搭建必备工具

这比过去在黑暗中摸索每个组件的时代容易多了。你可以在大约 10 到 15 分钟内（加上下载大量好东西所需的时间）搭建一个功能齐全的构建系统。

## 编译 C 程序

是的，你知道怎么做，但我们需要一个能挂接库及其位置的配置；仅仅输入 `cc myfile.c` 已经不够了。Make 几乎是促进程序编译的最简单系统，因此它为讨论提供了很好的模型。我会展示一个尽可能小、但留有足够扩展空间的 makefile。

## 设置变量并添加新库

无论使用什么系统，都将基于一小组类似环境变量的变量，因此我会讨论它们的作用以及如何设置。一旦我们有了所有这些编译机制，添加新库就只是调整已设置变量的问题。

## 搭建编译系统

作为额外收获，我们可以利用到目前为止的一切来搭建一个更简单的编译系统，让我们可以在命令行上剪切粘贴代码。

::: tip IDE 用户特别说明
你可能不是 make 用户，但本节对你仍然相关，因为 make 在编译代码时执行的每条规则，你的 IDE 都有对应的规则。如果你知道 make 在做什么，就能轻松调整你的 IDE。
:::

## 使用包管理器

如果你没有使用包管理器，你就错过了很多。

我提到包管理器有几个原因：首先，你们中的一些人可能还没有安装基础工具。对你们来说，我把这一节放在书的最前面，因为你们需要尽快获得这些工具。一个好的包管理器能让你迅速获得完整的 POSIX 子系统、各种语言的编译器、相当多的游戏、常见的办公生产力工具、数百个 C 库等等。

其次，作为 C 开发者，包管理器是我们获取库并融入工作的关键手段。

第三，如果你正从下载包的人转变为制作包的人，本书会带你走一半的路，展示如何准备你的包以便自动安装，这样当包仓库管理员决定将你的代码纳入仓库时，他或她将能轻松构建最终包。

如果你是 Linux 用户，你会用包管理器设置电脑，已经见识过软件获取过程有多简单。对于 Windows 用户，我将在下面介绍 Cygwin。Mac 用户有多种选择，如 Fink、Homebrew 和 Macports。所有 Mac 选项都依赖 Apple 的 Xcode 包，可通过（取决于你的 Mac 型号）OS 安装光盘、可安装程序目录、Apple App Store 或注册为 Apple 开发者免费获得。

你需要哪些包？以下是 C 开发基础知识的快速概览。由于每个系统有不同的组织方式，其中一些可能被打包不同、默认安装在基础包中或命名奇怪。有疑问时，安装它，因为我们已经过了安装太多东西会导致系统不稳定或变慢的时代。然而，你可能没有带宽（甚至磁盘空间）来安装包管理器提供的每一个包，所以需要一些判断。如果你发现缺少什么，随时可以回头再装。

**一定要安装的包：**

- **编译器**。一定要安装 gcc；clang 可能也可用。
- **GDB**，调试器。
- **Valgrind**，用于测试 C 内存使用错误。
- **gprof**，性能分析器。
- **make**，这样你就不必直接调用编译器。
- **pkg-config**，用于查找库。
- **Doxygen**，用于文档生成。
- **文本编辑器**。有数百种文本编辑器可选。以下是一些主观推荐：
  - Emacs 和 vim 是硬核极客的最爱。Emacs 非常包容（E 代表可扩展）；vim 更极简，对盲打者非常友好。如果你打算花数百小时盯着文本编辑器，值得花时间学习其中一个。
  - Kate 友好美观，提供我们作为程序员期望的便利功能的良好子集，如语法高亮。
  - 作为最后手段，试试 nano，它极其简单，基于文本，因此在 GUI 不工作时也能用。
- **如果你是 IDE 爱好者**，装一个——或几个。同样有很多选择；以下是一些推荐：
  - Anjuta：GNOME 家族。与 Glade（GNOME GUI 构建器）配合良好。
  - KDevelop：KDE 家族。
  - XCode：Apple 的 OS X IDE。
  - Code::blocks：相对简单，在 Windows 上工作。
  - Eclipse：带很多杯架和额外旋钮的豪华车。也跨平台。

后续章节会涉及这些更重量级的工具：

- Autotools：Autoconf、Automake、libtool
- Git
- 替代 shell，如 Z shell

当然，还有能让你省去重复造轮子（或更准确地说，重复造火车头）的 C 库。你可能想要更多，但以下是本书将使用的库：

- libcURL
- libGLib
- libGSL
- libSQLite3
- libXML2

库包的命名方案没有共识，你必须弄清楚你的包管理器如何将单个库拆分为子部分。通常有一个面向用户的包，另一个面向将在自己工作中使用该库的作者，所以务必同时选择基础包和 `-dev` 或 `-devel` 包。有些系统将文档单独放在另一个包中。有些要求你单独下载调试符号，在这种情况下，GDB 会在你第一次在缺少调试符号的东西上运行它时引导你完成步骤。

如果你使用的是 POSIX 系统，安装上述项目后，你将拥有完整的开发系统并准备好编码。对于 Windows 用户，我们将稍作绕道，了解设置如何与主 Windows 系统交互。

## 在 Windows 上编译 C

在大多数系统上，C 是核心 VIP 语言，所有其他工具都为其服务；在 Windows 上，C 却被奇怪地忽视了。

所以我需要花点时间讨论如何为在 C 中编写代码设置 Windows 电脑。如果你现在不是在 Windows 上编写，可以跳过本节，跳到「库在哪里？」。

### Windows 上的 POSIX

因为 C 和 Unix 共同演化，很难只谈一个而不谈另一个。我认为从 POSIX 开始更容易。此外，那些试图在 Windows 电脑上编译在其他地方编写的代码的人会发现这是最自然的路线。

据我所知，有文件系统的事物世界分为两个略有重叠的类别：

- **符合 POSIX 的系统**
- **Windows 系列操作系统**

符合 POSIX 并不意味着系统必须看起来和感觉像 Unix 盒子。例如，典型的 Mac 用户不知道自己正在使用带有美观前端的标准 BSD 系统，但知情者可以去 Utilities 文件夹（在 Applications 文件夹内），然后打开 Terminal 程序，尽情运行 `ls`、`grep` 和 `make`。

此外，我怀疑有多少系统能达到标准 100% 的要求（比如拥有 Fortran '77 编译器）。就我们的目的而言，我们需要一个能像精简 POSIX shell 一样行为的 shell、一些实用程序（sed、grep、make 等）、C99 编译器以及标准 C 库的扩展（如 fork 和 iconv）。这些可以作为主系统的旁注添加。包管理器的底层脚本、Autotools 以及几乎所有其他可移植编码尝试都会在一定程度上依赖这些工具，所以即使你不想整天盯着命令提示符，这些工具在安装时也会派上用场。

在服务器级 OS 和 Windows 7 的完整版上，Microsoft 提供过去称为 INTERIX、现在称为基于 Unix 的应用程序子系统（SUA）的东西，它提供常见的 POSIX 系统调用、Korn shell 和 gcc。该子系统通常不默认提供，但可以作为附加组件安装。但 SUA 不适用于其他当前版本的 Windows，也不会适用于 Windows 8，所以我们不能依赖 Microsoft 为其操作系统提供 POSIX 子系统。

于是，有了 Cygwin。

如果你要从头重建 Cygwin，你的议程将是：

1. 为 Windows 编写一个提供所有 POSIX 函数的 C 库。这必须弥合一些 Windows/POSIX 的不一致，例如 Windows 有 C: 这样的独立驱动器，而 POSIX 有统一的文件系统。在这种情况下，将 C: 别名为 `/cygdrive/c`，D: 别名为 `/cygdrive/d`，依此类推。
2. 既然你可以通过链接到你的库来编译 POSIX 标准程序，就这样做：生成 ls、bash、grep、make、gcc、X、rxvt、libglib、perl、python 等的 Windows 版本。
3. 一旦你构建了数百个程序和库，设置一个允许用户选择要安装的元素的包管理器。

作为 Cygwin 用户，你只需从 Cygwin 网站设置链接下载包管理器并选择包。你肯定需要前面的列表，加上一个像样的终端（试试 mintty，或安装 X 子系统并使用 xterm，因为两者都比 Windows 的 cmd.exe 友好得多），但你会看到，开发系统上熟悉的各种便利几乎都有。

在「路径」一节中，我讨论了影响编译的各种环境变量，包括搜索文件的路径。这不只是 POSIX：Windows 也有环境变量，你可以在控制面板的系统设置部分找到。如果你将 Cygwin 的 bin 目录（可能是 `c:\cygwin\bin`）添加到 Windows 的 PATH，Cygwin 会好用得多。

现在你可以开始编译 C 代码了。

### 使用 POSIX 编译 C

Microsoft 以 Visual Studio 的形式提供 C++ 编译器，它有 C89 兼容模式（通常称为 ANSI C，尽管 C11 才是当前的 ANSI 标准）。这是 Microsoft 目前提供的编译 C 代码的唯一手段。该公司的许多代表已明确表示，除了对少数 C99 特性的支持外，不会有更多（更不用说 C11 支持）。Visual Studio 是唯一仍停留在 C89 的主要编译器，所以我们不得不在别处寻找替代品。

当然，Cygwin 提供 gcc，如果你已经安装了 Cygwin，你就已经有了完整的构建环境。

默认情况下，你在 Cygwin 下编译的程序将依赖其 POSIX 函数库 cygwin1.dll（无论你的代码是否实际包含任何 POSIX 调用）。如果你在安装了 Cygwin 的电脑上运行程序，那就没问题。用户将能够点击可执行文件并按预期运行，因为系统应该能找到 Cygwin DLL。如果你随代码分发 cygwin1.dll，在 Cygwin 下编译的程序可以在未安装 Cygwin 的电脑上运行。在我的机器上，cygwin 的路径是 `/bin/cygwin1.dll`。cygwin1.dll 文件有类似 GPL 的许可证（见「法律旁注」），如果你将 DLL 与 Cygwin 整体分开分发，则必须发布程序的源代码。

如果这是问题，你必须找到一种在不依赖 cygwin1.dll 的情况下重新编译的方法，这意味着从代码中删除任何 POSIX 特定函数（如 fork 或 popen）并使用后面讨论的 MinGW。你可以使用 cygcheck 找出程序依赖哪些 DLL，从而验证你的可执行文件是否链接到 cygwin1.dll。

要查看程序或动态链接库依赖的其他库：

- **Cygwin**：`cygcheck libxx.dll`
- **Linux**：`ldd libxx.so`
- **Mac**：`otool -L libxx.dylib`

### 不使用 POSIX 编译 C

如果你的程序不需要 POSIX 函数，可以使用 MinGW（Minimalist GNU for Windows），它提供标准 C 编译器和一些基本相关工具。MSYS 是 MinGW 的配套工具，提供 shell 和其他实用程序。

MSYS 提供 POSIX shell（你可以找到 mintty 或 RXVT 终端来运行 shell），或者完全离开命令提示符，试试 Code::blocks，一个在 Windows 上使用 MinGW 编译的 IDE。Eclipse 是一个更广泛的 IDE，也可以配置用于 MinGW，尽管需要更多设置。

或者，如果你在 POSIX 命令提示符下更自在，那就设置 Cygwin，获取提供 MinGW 版本 gcc 的包，并用它们编译，而不是使用 Cygwin gcc 的 POSIX 链接默认版本。

如果你还没见过 Autotools，很快就会见到。使用 Autotools 构建的包的标志是其三命令安装：`./configure && make && make install`。MSYS 提供足够的机制，使此类包有很好的机会工作。或者，如果你已经从 Cygwin 命令提示符下载了要构建的包，可以使用以下命令设置包以使用 Cygwin 的 Mingw32 编译器生成无 POSIX 的代码：

```bash
./configure --host=ming32
```

然后照常运行 `make && make install`。

一旦你通过命令行编译或 Autotools 在 MinGW 下编译，你就有了原生 Windows 二进制文件。因为 MinGW 不知道 cygwin1.dll，而你的程序反正不进行 POSIX 调用，你现在有了一个真正的 Windows 程序可执行文件，没人会知道你从 POSIX 环境编译的。

::: warning 注意事项
虽然 MinGW 有一个安装系统基础并提供许多库的包管理器（大多是 MinGW 本身需要的），但这 handful 预编译库与典型包管理器提供的数百个包相比相形见绌。事实上，我 Linux 电脑上的包管理器拥有的 MinGW 编译库比 MinGW 包管理器还多。这是截至撰写时的；到你读到这时，像你这样的用户可能已经向 MinGW 仓库贡献了更多包。然而，MinGW 目前预编译库匮乏。如果你想摆脱 cygwin1.dll，就不能使用随 Cygwin 提供的 libglib.dll 版本。你需要从源代码将 GLib 重新编译为原生 Windows DLL——但 GLib 依赖 GNU 的 gettext 进行国际化，所以你必须先构建该库。现代代码依赖现代库，所以你可能会发现自己在设置其他系统只需包管理器一行调用就能搞定的事情上花很多时间。我们又回到了让人们谈论「C 已经 40 岁了，所以你需要从头编写一切」的那种事情。
:::

所以，注意事项就这些。Microsoft 已经退出对话，留下其他人来实现后 grunge 的 C 编译器和环境。Cygwin 做到了这一点，并提供了带有足够库的完整包管理器来完成部分或全部工作，但它与 POSIX 风格的编写和 Cygwin 的 DLL 相关联。如果那是问题，你需要做更多工作来构建编写体面代码所需的环境和库。

## 库在哪里？

好的，你有了编译器、POSIX 工具链和能轻松安装数百个库的包管理器。现在我们可以继续解决在编译程序时使用它们的问题。

我们必须从编译器命令行开始，它会很快变得一团糟，但我们将以三个（有时三个半）相对简单的步骤结束：

1. 设置一个列出编译器标志的变量。
2. 设置一个列出要链接的库的变量。半步是，有时你只需为编译时链接设置一个变量，有时必须为编译时和运行时链接设置两个。
3. 设置一个将使用这些变量来协调编译的系统。

要使用库，你必须告诉编译器两次你将从库中导入函数：一次用于编译，一次用于链接器。对于标准位置的库，两次声明通过程序文本中的 `#include` 和编译器行上的 `-l` 标志发生。

示例 1-1 展示了一个做有趣数学的快速示例程序（至少对我来说；如果统计学术语对你来说是希腊语，没关系）。C99 标准的误差函数 `erf(x)` 与均值为零、标准差为 √2 的正态分布从零到 x 的积分密切相关。这里我们用 erf 验证统计学家中流行的一个区间（标准大 n 假设检验的 95% 置信区间）。我们将此文件命名为 erf.c。

**示例 1-1. 来自标准库的一行代码（erf.c）**

```c
#include <math.h>  //erf, sqrt
#include <stdio.h> //printf
int main(){
    printf("The integral of a Normal(0, 1) distribution "
           "between -1.96 and 1.96 is: %g\n", erf(1.96*sqrt(1/2.)));
}
```

`#include` 行你应该熟悉。编译器将把 math.h 和 stdio.h 粘贴到这里的代码文件中，从而粘贴 printf、erf 和 sqrt 的声明。math.h 中的声明没有说明 erf 做什么，只说它接受一个 double 并返回一个 double。这对编译器来说足够检查我们用法的一致性并生成一个带有告诉计算机「一旦到达这个注释，去找 erf 函数，并用 erf 的返回值替换这个注释」的注释的目标文件。

链接器的任务是实际找到 erf 来调和该注释，erf 在你硬盘某处的库中。

math.h 中的数学函数被拆分到它们自己的库中，你必须通过添加 `-lm` 标志告诉链接器。这里，`-l` 是表示需要链接库的标志，本例中的库有单字母名称 m。你免费获得 printf，因为有一个隐式的 `-lc` 要求链接器在链接命令末尾链接假定的标准 libc。稍后，我们会看到通过 `-lglib-2.0` 链接 GLib 2.0，通过 `-lgsl` 链接 GNU Scientific Library，等等。

所以如果文件名为 erf.c，使用 gcc 编译器的完整命令行，包括稍后讨论的几个额外标志，将如下所示：

```bash
gcc erf.c -o erf -lm -g -Wall -O3 -std=gnu11
```

所以我们通过程序中的 `#include` 告诉编译器包含数学函数，通过命令行上的 `-lm` 告诉链接器链接数学库。

`-o` 标志给出输出名称；否则，你会得到默认的可执行文件名 a.out。

## 我喜欢的几个标志

你会看到我每次都使用几个编译器标志，我建议你也这样做。

- **-g** 添加调试符号。没有它，你的调试器将无法给你变量或函数名。它们不会减慢程序，我们也不在乎程序是否大一千字节，所以没有理由不使用。适用于 gcc、clang 和 icc（Intel C Compiler）。
- **-std=gnu11** 是 clang 和 gcc 特有的，指定编译器应允许符合 C11 和 POSIX 标准的代码（加上一些 GNU 扩展）。截至撰写时，clang 将默认使用 C99 标准，gcc 使用 C89 标准。如果你的 gcc、clang 或 icc 早于 C11，使用 `-std=gnu99` 让它使用 C99。POSIX 标准规定 c99 应存在于你的系统上，所以上述行用于编译 C99 代码的编译器无关版本将是：

  ```bash
  c99 erf.c -o erf -lm -g -Wall -O3
  ```

  在下面的 makefile 中，我通过设置变量 `CC=c99` 获得此效果。

- **-O3** 表示优化级别三，尝试各种技巧构建更快的代码。如果你在运行调试器时发现太多变量被优化掉而无法跟踪发生了什么，则将其改为 `-O0`。这将是后面 CFLAGS 变量中的常见调整。适用于 gcc、clang 和 icc。
- **-Wall** 添加编译器警告。适用于 gcc、clang 和 icc。对于 icc，你可能更喜欢 `-w1`，它显示编译器的警告，但不显示其备注。

::: tip 始终使用编译器警告
你可能一丝不苟，对 C 标准了如指掌，但你不会比编译器更一丝不苟或更有见识。旧的 C 教科书用大量篇幅告诫你要注意 `=` 和 `==` 的区别，或检查所有变量在使用前是否已初始化。作为更现代的教科书作者，我轻松多了，因为我可以将所有那些告诫总结为一个提示：始终使用编译器警告。

如果编译器建议修改，不要质疑或推迟修复。做一切必要的事情来（1）理解为什么你得到警告以及（2）修复你的代码使其以零警告和零错误编译。编译器消息以晦涩著称，所以如果你在步骤（1）上有困难，将警告消息粘贴到互联网搜索引擎中，看看有多少人在你之前被这个警告困扰。你可能想将 `-Werror` 添加到编译器标志中，这样编译器会将警告视为错误。
:::

## 路径

我硬盘上有超过 70 万个文件，其中一个是 sqrt 和 erf 的声明，另一个是保存编译后函数的目标文件。编译器需要知道在哪些目录中查找正确的头文件和目标文件，当我们使用不属于 C 标准的库时，问题只会变得更复杂。

在典型设置中，库可能安装在至少三个地方：

- 操作系统供应商可能定义一个或两个标准目录，供供应商安装库。
- 可能有一个供本地系统管理员安装不应在下次供应商 OS 升级时被覆盖的包的目录。系统管理员可能有一个应覆盖默认版本的特殊修改版库。
- 用户通常没有写入这些位置的权限，因此应该能够使用其主目录中的库。

OS 标准位置通常不会造成问题，编译器应该知道在这些地方查找标准 C 库以及与其一起安装的任何东西。POSIX 标准将这些目录称为「通常位置」。

但对于其他东西，你必须告诉编译器去哪里找。这会变得复杂：在非标准位置查找库没有标准方法，它在让人们沮丧的 C 事物列表中排名很高。另一方面，你的编译器知道如何在通常位置查找，库分发者倾向于将东西放在通常位置，所以你可能永远不需要手动指定路径。另一方面，有一些工具可以帮助你指定路径。最后，一旦你在系统上找到了非标准位置，你可以在 shell 或 makefile 变量中设置它们，就再也不用想了。

假设你有一个名为 Libuseful 的库安装在你的电脑上，你知道它的各种文件被放在 `/usr/local/` 目录中，这是官方为系统管理员的本地库预留的位置。你已经在代码中放了 `#include <useful.h>`；现在你必须在命令行上放这个：

```bash
gcc -I/usr/local/include use_useful.c -o use_useful -L/usr/local/lib -luseful
```

- `-I` 将给定路径添加到 include 搜索路径，编译器在其中搜索你在代码中 `#included` 的头文件。
- `-L` 添加到库搜索路径。
- **顺序很重要**。如果你有一个名为 specific.o 的文件依赖 Libbroad 库，而 Libbroad 依赖 Libgeneral，那么你需要：

  ```bash
  gcc specific.o -lbroad -lgeneral
  ```

  任何其他顺序，如 `gcc -lbroad -lgeneral specific.o`，可能会失败。你可以认为链接器查看第一项 specific.o，写下未解析的函数、结构和变量名列表。然后它转到下一项 `-lbroad`，在其仍然缺失的列表中搜索项目，同时可能添加新的未解析项目，然后检查 `-lgeneral` 获取列表中仍然缺失的那些项目。如果到列表末尾（包括末尾的隐式 `-lc`）仍有未定位的名称，则链接器停止并将剩余缺失项列表交给用户。

好的，回到位置问题：你想链接的库在哪里？如果它是通过你用来安装操作系统其余部分的同一包管理器安装的，那么它很可能在通常位置，你不用担心。

你可能知道自己的本地库倾向于在哪里，如 `/usr/local` 或 `/sw` 或 `/opt`。你无疑有搜索硬盘的手段，如桌面上的搜索工具或 POSIX：

```bash
find /usr -name 'libuseful*'
```

在 `/usr` 中搜索名称以 libuseful 开头的文件。当你找到 Libuseful 的共享对象文件在 `/some/path/lib` 时，头文件几乎肯定在 `/some/path/include`。

其他人也觉得在硬盘上找库很烦人，pkg-config 通过维护包自我报告为编译所需的标志和位置的仓库来解决这个问题。在命令行输入 pkg-config；如果你得到关于指定包名的错误，那就好，你有 pkg-config 可以用它为你做研究。例如，在我的 PC 上，在命令行输入这两个命令：

```bash
pkg-config --libs gsl libxml-2.0
pkg-config --cflags gsl libxml-2.0
```

给我这两行输出：

```
-lgsl -lgslcblas -lm -lxml2
-I/usr/include/libxml2
```

这些正是使用 GSL 和 LibXML2 编译所需的标志。`-l` 标志揭示 GNU Scientific Library 依赖 Basic Linear Algebra Subprograms（BLAS）库，而 GSL 的 BLAS 库依赖标准数学库。似乎所有库都在通常位置，因为没有 `-L` 标志，但 `-I` 标志表示 LibXML2 头文件的位置。

回到命令行，当你用反引号包围命令时，shell 用命令的输出替换命令。也就是说，当我输入：

```bash
gcc `pkg-config --cflags --libs gsl libxml-2.0` -o specific specific.c
```

编译器看到：

```bash
gcc -I/usr/include/libxml2 -lgsl -lgslcblas -lm -lxml2 -o specific specific.c
```

所以 pkg-config 为我们做了很多工作，但它不够标准，我们不能指望每个人都有它或每个库都在其中注册。如果你没有 pkg-config，你必须自己做这种研究，通过阅读库的手册或像之前那样搜索磁盘。

即使有 pkg-config，我们需要某种东西来为我们组装这一切的需求也越来越明显。每个元素都足够容易理解，但这是一个冗长、机械的繁琐部分列表。

## 运行时链接

静态库由编译器通过有效地将库内容复制到最终可执行文件中来链接。所以程序本身或多或少是一个独立系统。共享库在运行时链接到你的程序，这意味着我们在编译时遇到的找库问题在运行时会再次出现。更糟的是，你程序的用户可能遇到这个问题。

如果库在通常位置之一，生活就很美好，系统在运行时找库不会有问题。如果你的库在非标准路径，那么你需要找到一种方法来修改库的运行时搜索路径。选项：

- 如果你用 Autotools 打包程序，Libtool 知道如何添加正确的标志，你不用担心。
- 当用基于 libpath 中库的 gcc、clang 或 icc 编译程序时，在后续 makefile 中添加：

  ```
  LDADD=-Llibpath -Wl,-Rlibpath
  ```

  `-L` 标志告诉编译器在哪里搜索库以解析符号；`-Wl` 标志将标志从 gcc/clang/icc 传递给链接器，链接器将给定的 `-R` 嵌入到库的运行时搜索路径中。不幸的是，pkg-config 通常不知道运行时路径，所以你可能需要手动输入这些。
- 在运行时，链接器将使用另一个路径来查找不在通常位置且未通过 `-Wl,R....` 在可执行文件中标注的库。此路径可以在 shell 的启动脚本（.bashrc、.zshrc 或任何适用的）中设置。要确保在运行时搜索 libpath 中的共享库，使用：

  ```bash
  export LD_LIBRARY_PATH=libpath:$LD_LIBRARY_PATH    # Linux, Cygwin
  export DYLD_LIBRARY_PATH=libpath:$DYLD_LIBRARY_PATH    # OS X
  ```

有人警告不要过度使用 LD_LIBRARY_PATH（如果有人把恶意冒充库放在路径中，从而在你不知情的情况下替换真实库怎么办？），但如果你的所有库都在一个地方，将一个在你名义控制下的目录添加到路径中并非不合理。

## 使用 Makefile

Makefile 为所有这些无休止的调整提供了解决方案。它基本上是一组有组织的变量和单行 shell 脚本序列。POSIX 标准的 make 程序读取 makefile 获取指令和变量，然后为我们组装冗长繁琐的命令行。在这段之后，几乎没有理由直接调用编译器。

在「Makefile 与 Shell 脚本」中，我会介绍 makefile 的更多细节；这里，我将展示能编译依赖库的基本程序的最小可行 makefile。就是它，全部六行：

```makefile
P=program_name
OBJECTS=
CFLAGS = -g -Wall -O3
LDLIBS=
CC=c99
$(P): $(OBJECTS)
```

**用法：**

- **一次性**：将此（命名为 makefile）保存在与你的 .c 文件相同的目录中。如果你使用 GNU Make，可以选择将名称大写为 Makefile，如果你觉得这样能帮助它从其他文件中脱颖而出。在第一行设置程序名称（使用 progname，不是 progname.c）。
- **每次需要重新编译时**：输入 `make`。

::: tip 动手练习
这是世界著名的 hello.c 程序（Kernighan, 1978），两行：

```c
#include <stdio.h>
int main(){ printf("Hello, world.\n"); }
```

保存它和前面的 makefile 到目录，尝试前面的步骤来编译和运行程序。
:::

### 设置变量

我们很快就会讲到 makefile 的实际功能，但这个 makefile 六行中有五行是关于设置变量的（其中两个当前设置为空白），表明我们应该花点时间更详细地考虑环境变量。

历史上，shell 语法有两条主线：一条主要基于 Bourne shell，另一条主要基于 C shell。C shell 对变量有略微不同的语法，例如 `set CFLAGS="-g -Wall -O3"` 来设置 CFLAGS 的值。但 POSIX 标准是围绕 Bourne 型变量设置语法编写的，所以这是我本书其余部分关注的内容。

Shell 和 make 使用 `$` 表示变量的值，但 shell 使用 `$var`，而 make 希望任何超过一个字符的变量名在括号中：`$(var)`。所以，给定前面的 makefile，`$(P): $(OBJECTS)` 将被求值为 `program_name:`。

有几种方式告诉 make 变量：

- 在调用 make 之前从 shell 设置变量并导出变量，意味着当 shell 生成子进程时，它在其环境变量列表中有该变量。要从 POSIX 标准命令行设置 CFLAGS：

  ```bash
  export CFLAGS='-g -Wall -O3'
  ```

  在家里，我省略这个 makefile 的第一行 `P=program_name`，而是通过 `export P=program_name` 每会话设置一次，这意味着我编辑 makefile 本身的频率更低。
- 你可以将这些 export 命令放在 shell 的启动脚本中，如 .bashrc 或 .zshrc。这保证每次登录或启动新 shell 时，变量都会被设置和导出。如果你确信 CFLAGS 每次都会相同，可以在这里设置，就再也不用想了。
- 你可以通过将赋值放在命令正前来为单个命令导出变量。env 命令列出它知道的环境变量，所以当你运行以下命令时：

  ```bash
  PANTS=kakhi env | grep PANTS
  ```

  你应该看到适当的变量及其值。这就是为什么 shell 不让你在等号周围放空格：空格是它区分赋值和命令的方式。

  使用这种形式只为一行设置和导出给定变量。在命令行上尝试后，再次运行 `env | grep PANTS` 以验证 PANTS 不再是导出的变量。

  随意指定任意多个变量：

  ```bash
  PANTS=kakhi PLANTS="ficus fern" env | grep 'P.*NTS'
  ```

- 如前面的 makefile 中，你可以在 makefile 头部设置变量，使用 `CFLAGS=` 这样的行。在 makefile 中，你可以在等号周围有空格而不会破坏任何东西。
- make 允许你在命令行上设置变量，独立于 shell。因此，这两行几乎等价：

  ```bash
  make CFLAGS="-g -Wall"   # 设置 makefile 变量
  CFLAGS="-g -Wall" make   # 设置 make 及其子进程可见的环境变量
  ```

就你的 makefile 而言，所有这些方式都是等价的，除了 make 调用的子程序会知道新的环境变量但不会知道任何 makefile 变量。

### C 中的环境变量

在你的 C 代码中，用 getenv 获取环境变量。因为 getenv 非常容易使用，它可以用于从命令提示符快速设置选项。

示例 1-2 向屏幕打印用户期望次数的消息。消息通过环境变量 msg 设置，重复次数通过 reps 设置。注意如果 getenv 返回 NULL（通常表示环境变量未设置）时有 10 和 "Hello." 的默认值。

**示例 1-2. 环境变量提供快速调整程序细节的方式（getenv.c）**

```c
#include <stdlib.h> //getenv, atoi
#include <stdio.h>  //printf
int main(){
    char *repstext = getenv("reps");
    int reps = repstext ? atoi(repstext) : 10;
    char *msg = getenv("msg"); 
    if (!msg) msg = "Hello.";
    for (int i=0; i< reps; i++)
        printf("%s\n", msg);
}
```

如前所述，我们可以为单行导出变量，这使得向程序发送变量更加方便。用法：

```bash
reps=10 msg="Ha" ./getenv
msg="Ha" ./getenv
reps=20 msg=" " ./getenv
```

你可能会觉得这很奇怪——程序的输入应该在程序名之后，该死——但奇怪归奇怪，你可以看到程序本身几乎没有设置，我们几乎免费获得了命令行上的命名参数。

当你的程序更进一步时，你可以花时间设置 POSIX 标准的 getopt 或 GNU 标准的 argp_parse 以通常方式处理输入参数。

make 还提供几个内置变量。以下是你阅读以下规则所需的（POSIX 标准）变量：

| 变量 | 含义 |
|------|------|
| `$@` | 完整的目标文件名。目标指需要构建的文件，如从 .c 文件编译的 .o 文件或通过链接 .o 文件制成的程序。 |
| `$*` | 去掉后缀的目标文件。所以如果目标是 prog.o，$* 是 prog，$*.c 会变成 prog.c。 |
| `$<` | 导致此目标被触发和构建的文件名。如果我们在制作 prog.o，可能是因为 prog.c 最近被修改了，所以 $< 是 prog.c。 |

### 规则

现在，让我们关注 makefile 将执行的程序，然后了解变量如何影响它。

撇开变量设置，makefile 的段具有形式：

```
target: dependencies
        script
```

如果通过命令 `make target` 调用目标，则检查依赖项。如果目标是文件，依赖项都是文件，且目标比依赖项新，则文件是最新的，无事可做。否则，目标的处理被搁置，依赖项被运行或生成（可能通过另一个目标），当依赖项脚本都完成后，目标的脚本被执行。

例如，在这成为一本书之前，它是一系列发布到博客的提示。每篇博客文章都有 HTML 和 PDF 版本，都通过 LaTeX 生成。为了简单示例，我省略了很多细节（如 latex2html 的许多选项），但这是人们可以为该过程编写的那种 makefile。

如果你从屏幕或纸上的版本复制任何这些 makefile 片段到名为 makefile 的文件，不要忘记每行开头的空白必须是制表符，不是空格。怪 POSIX。

```makefile
all: html doc publish
doc:
    pdflatex $(f).tex
html:
    latex -interaction batchmode $(f)
    latex2html $(f).tex
publish:
    scp $(f).pdf $(Blogserver)
```

我通过 `export f=tip-make` 这样的命令在命令行上设置 f。当我然后输入 make 时，第一个目标 all 被检查。也就是说，make 本身等价于 make first_target。它依赖 html、doc 和 publish，所以这些目标按顺序被调用。如果我知道还没准备好复制到世界，可以调用 `make html doc` 只做那些步骤。

在之前的简单 makefile 中，我们只有一个目标/依赖/脚本组。例如：

```makefile
P=domath
OBJECTS=addition.o subtraction.o
$(P): $(OBJECTS)
```

这遵循与我的博客 makefile 类似的依赖和脚本序列，但脚本是隐式的。这里，P=domath 是要编译的程序，它依赖目标文件 addition.o 和 subtraction.o。因为 addition.o 没有列为目标，make 使用下面列出的隐式规则从 .c 编译到 .o 文件。然后它对 subtraction.o 和 domath.o 做同样的事（因为 GNU make 隐式假设给定此设置 domath 依赖 domath.o）。一旦所有目标都构建好了，我们没有构建 $(P) 目标的脚本，所以 GNU make 填入其将 .o 文件链接到可执行文件的默认脚本。

POSIX 标准的 make 有从 .c 源代码文件编译 .o 目标文件的这个配方：

```
$(CC) $(CFLAGS) $(LDFLAGS) -o $@ $*.c
```

$(CC) 变量代表你的 C 编译器；POSIX 标准指定默认 CC=c99，但当前版本的 GNU make 设置 CC=cc，通常是 gcc 的链接。在本节开头的 minimal makefile 中，$(CC) 明确设置为 c99，$(CFLAGS) 设置为之前的标志列表，$(LDFLAGS) 未设置因此被替换为空。所以如果 make 确定需要生成 your_program.o，那么给定该 makefile 将运行此命令：

```
c99 -g -Wall -O3 -o your_program.o your_program.c
```

当 GNU make 决定你有从目标文件构建的可执行程序时，它使用这个配方：

```
$(CC) $(LDFLAGS) first.o second.o $(LDLIBS)
```

回想一下链接器中顺序很重要，所以我们需要两个链接器变量。在前面的例子中，我们需要：

```
cc specific.o -lbroad -lgeneral
```

作为链接命令的相关部分。将正确的编译命令与配方比较，我们看到需要设置 `LDLIBS=-lbroad -lgeneral`。

如果你想查看内置到你 make 版本中的默认规则和变量的完整列表，试试：

```bash
make -p > default_rules
```

所以，游戏就是：找到正确的变量并在 makefile 中设置它们。你仍然必须研究正确的标志是什么，但至少你可以把它们写在 makefile 里，就再也不用想了。

::: tip 动手练习
修改你的 makefile 以编译 erf.c。
:::

如果你使用 IDE、CMAKE 或 POSIX 标准 make 的任何其他替代品，你将玩同样的找变量游戏。我将继续讨论前面的 minimal makefile，你应该能轻松在 IDE 中找到对应的变量。

- CFLAGS 变量是根深蒂固的习惯，但你为链接器设置的变量因系统而异。甚至 LDLIBS 也不是 POSIX 标准的，但它是 GNU make 使用的。
- CFLAGS 和 LDLIBS 变量是我们要挂接所有定位和识别库的编译器标志的地方。如果你有 pkg-config，把反引号调用放在这里。例如，我系统上的 makefile，我几乎对所有东西都使用 Apophenia 和 GLib，看起来像：

  ```makefile
  CFLAGS=`pkg-config --cflags apophenia glib-2.0` -g -Wall -std=gnu11 -O3
  LDLIBS=`pkg-config --libs apophenia glib-2.0`
  ```

  或者，手动指定 -I、-L 和 -l 标志，如：

  ```makefile
  CFLAGS=-I/home/b/root/include -g -Wall -O3
  LDLIBS=-L/home/b/root/lib -lweirdlib
  ```

- 在你将库及其位置添加到 LDLIBS 和 CFLAGS 行并且你知道它在你的系统上工作后，几乎没有理由删除它。你真的在乎最终可执行文件可能比如果你为每个程序定制新 makefile 大 10 千字节吗？这意味着你可以写一个总结系统上所有库位置的 makefile，并在项目之间复制而无需任何重写。
- 如果你的程序需要第二个（或更多）C 文件，将 second.o、third.o 等添加到本节开头 makefile 的 OBJECTS 行（没有逗号，名称之间只有空格）。
- 如果你有一个单 .c 文件的程序，你可能根本不需要 makefile。在没有 makefile 和之前 erf.c 的目录中，尝试使用 shell：

  ```bash
  export CFLAGS='-g -Wall -O3 -std=gnu11'
  export LDLIBS='-lm'
  make erf
  ```

  看 make 使用其 C 编译知识完成其余工作。

### 构建共享库的链接器标志是什么？

说实话，我不知道。它在操作系统之间不同，按类型和年份，即使在一个系统上规则也常常很复杂。

相反，第 3 章介绍的工具之一 Libtool 知道每个操作系统上每个共享库生成过程的每个细节。我建议花时间了解 Autotools，从而一劳永逸地解决共享对象编译问题，而不是花时间学习每个目标系统的正确编译器标志和链接过程。

## 从源代码使用库

到目前为止，故事一直是关于使用 make 编译你自己的代码。编译他人提供的代码往往是另一回事。

让我们试试一个示例包。GNU Scientific Library 包含大量数值计算例程。

GSL 通过 Autotools 打包，这是一套工具，将通过测试每个已知的怪癖并实现适当的变通方法，为任何机器准备库。「用 Autotools 打包你的代码」将详细介绍如何用 Autotools 打包你自己的程序和库。但现在，我们可以作为系统的用户开始，享受快速安装有用库的便利。

GSL 通常通过包管理器以预编译形式提供，但为了完成编译步骤，以下是如何获取 GSL 源代码并设置它，假设你在电脑上有 root 权限：

```bash
wget ftp://ftp.gnu.org/gnu/gsl/gsl-1.16.tar.gz 
tar xvzf gsl-*gz                               
cd gsl-1.16
./configure                                    
make
sudo make install                              
```

1. 下载压缩存档。如果你没有 wget，让包管理器安装它，或将此 URL 输入浏览器。
2. 解压存档：x=extract，v=verbose，z=通过 gzip 解压，f=filename。
3. 确定你机器的怪癖。如果 configure 步骤给你关于缺失元素的错误，使用包管理器获取它并再次运行 configure。
4. 安装到正确位置——如果你有权限。

如果你在家尝试，可能就有 root 权限，这会正常工作。如果你在工作使用共享服务器，拥有超级用户权限的几率很低，所以你将无法提供以超级用户执行脚本最后一步所需的密码。在这种情况下，屏住呼吸直到下一节。

安装了吗？示例 1-3 提供了一个使用 GSL 函数尝试找 95% 置信区间的短程序；试试看能否链接和运行：

**示例 1-3. 用 GSL 重做示例 1-1（gsl_erf.c）**

```c
#include <gsl/gsl_cdf.h>
#include <stdio.h>
int main(){
    double bottom_tail = gsl_cdf_gaussian_P(-1.96, 1);
    printf("Area between [-1.96, 1.96]: %g\n", 1-2*bottom_tail);
}
```

要使用你刚安装的库，你需要修改使用库的程序的 makefile 以指定库及其位置。

根据你是否有 pkg-config，你可以做以下之一：

```makefile
LDLIBS=`pkg-config --libs gsl`
# 或
LDLIBS=-lgsl -lgslcblas -lm
```

如果它没有安装在标准位置且 pkg-config 不可用，你需要在这些定义的开头添加路径，如 `CFLAGS=-I/usr/local/include` 和 `LDLIBS=-L/usr/local/lib -Wl,-R/usr/local/lib`。

### 从源代码使用库（即使你的系统管理员不希望你这样做）

如果你在工作使用共享电脑，或在家有一个特别控制欲强的另一半，你可能没有 root 访问权限。然后你必须转入地下，创建自己的私有 root 目录。

第一步是简单地创建目录：

```bash
mkdir ~/root
```

我已经有一个 ~/tech 目录，存放所有技术后勤、手册和代码片段，所以我创建了 ~/tech/root 目录。名称不重要，但我会在这里使用 ~/root 作为虚拟目录。

第二步是将新 root 系统的正确部分添加到所有相关路径。对于程序，那是 .bashrc（或等效）中的 PATH：

```bash
PATH=~/root/bin:$PATH
```

通过将新目录的 bin 子目录放在原始 PATH 之前，它将首先被搜索，你的任何程序副本将首先被找到。因此，你可以用你喜欢的版本替换已在系统标准共享目录中的任何程序。

对于你将融入 C 程序的库，注意前面 makefile 中要搜索的新路径：

```makefile
LDLIBS=-L$(HOME)/root/lib （加上其他标志，如 -lgsl -lm ...）
CFLAGS=-I$(HOME)/root/include （加上 -g -Wall -O3 ...）
```

现在你有了本地 root，你也可以将它用于其他系统，如 Java 的 CLASSPATH。

最后一步是在新 root 中安装程序。如果你有源代码且它使用 Autotools，你只需在正确的地方添加 `--prefix=$HOME/root`：

```bash
./configure --prefix=$HOME/root && make && make install
```

你不需要 sudo 来执行安装步骤，因为一切现在都在你控制的领土上。

因为程序库在你的主目录中，权限不超过你，你的系统管理员不能抱怨它们对他人造成负担。如果你的系统管理员仍然抱怨，那么，尽管可能令人难过，可能是时候分手了。

## 手册

我想曾经有一段时间手册实际上是印刷文档，但在当今，它以 man 命令的形式存在。例如，使用 `man strtok` 阅读 strtok 函数，通常包括要包含的头文件、输入参数和关于其用法的基本说明。手册页倾向于保持简单，有时缺乏示例，并假设读者已经对函数的工作原理有基本了解。如果你需要更基础的教程，你最喜欢的互联网搜索引擎可能提供几个（对于 strtok，见「strtok 赞歌」一节）。GNU C 库手册，也容易在网上找到，非常可读，为初学者编写。

- 如果你记不起要查找的名称，每个手册页都有一行摘要，`man -k searchterm` 将搜索这些摘要。许多系统还有 apropos 命令，类似于 man -k 但添加了一些功能。为了进一步精炼，我经常发现自己将 apropos 的输出通过 grep 管道。
- 手册分为多个部分。第 1 节是命令行命令，第 3 节是库函数。如果你的系统有名为 printf 的命令行程序，那么 `man printf` 将显示其文档，`man 3 printf` 将显示 C 库 printf 命令的文档。
- 有关 man 命令用法的更多信息（如部分的完整列表），试试 `man man`。
- 你的文本编辑器或 IDE 可能有快速调出手册页的手段。例如，vim 用户可以将光标放在单词上并使用 K 打开该单词的手册页。

## 通过 Here Document 编译 C 程序

在这一点上，你已经几次看到编译的模式：

1. 设置表达编译器标志的变量。
2. 设置表达链接器标志的变量，包括你使用的每个库的 -l 标志。
3. 使用 make 或 IDE 的配方将变量转换为完整的编译和链接命令。

本章的其余部分将最后一次做所有这些，使用绝对最小的设置：只有 shell。如果你是通过将代码片段剪切粘贴到解释器来学习脚本语言的动觉学习者，你将能够对在命令提示符上粘贴 C 代码做同样的事。

### 从命令行包含头文件

gcc 和 clang 有一个用于包含头文件的方便标志。例如：

```bash
gcc -include stdio.h
```

等价于在 C 文件开头放：

```c
#include <stdio.h>
```

clang 同样。通过将其添加到我们的编译器调用中，我们终于可以将 hello.c 写成它应该的一行代码：

```c
int main(){ printf("Hello, world.\n"); }
```

通过以下方式编译良好：

```bash
gcc -include stdio.h hello.c -o hi --std=gnu99 -Wall -g -O3
```

或 shell 命令如：

```bash
export CFLAGS='-g -Wall -include stdio.h'
export CC=c99 
make hello
```

关于 -include 的这个提示是编译器特有的，涉及将信息从代码移到编译指令。如果你认为这是不好的形式，好吧，跳过这个提示。

### 统一头文件

请允许我离题几段到头文件的主题。要有用，头文件必须包含代码文件包含头文件时使用的类型、宏和函数的 typedef、宏定义和函数声明。此外，它不应包含代码文件不会使用的 typedef、宏定义和函数声明。

要真正符合这两个条件，你需要为每个代码文件编写单独的头文件，只包含当前代码文件的确切相关部分。没人真的这么做。

曾经有一段时间，编译器需要几秒或几分钟才能编译甚至相对简单的程序，所以减少编译器必须做的工作有人类可察觉的好处。我当前的 stdio.h 和 stdlib.h 副本各有约 1,000 行（试试 `wc -l /usr/include/stdlib.h`），time.h 另有 400 行，意味着这个七行程序：

```c
#include <time.h>
#include <stdio.h>
#include <stdlib.h>
int main(){
    srand(time(NULL));       // Initialize RNG seed.
    printf("%i\n", rand());  // Make one draw.
}
```

实际上是一个约 2,400 行的程序。

你的编译器不再认为 2,400 行是什么大事，这在不到一秒内编译。所以趋势是通过在单个头文件中包含更多元素来节省用户挑选头文件的时间。

你会在后面看到使用 GLib 的示例，顶部有 `#include <glib.h>`。该头文件包含 74 个子头文件，涵盖 GLib 库的所有子部分。这是 GLib 团队良好的用户界面设计，因为我们这些不想花时间挑选库的正好子部分的人可以一行快速完成头文件文书工作，而那些想要详细控制的人可以精确挑选他们需要的头文件。如果 C 标准库有这样一个快速简便的头文件就好了；在 1980 年代不是惯例，但做一个很容易。

::: tip 动手练习
为自己写一个头文件，我们称之为 allheads.h，把你用过的每个头文件都扔进去，所以它看起来像：

```c
#include <math.h>
#include <time.h>
#include <stdio.h>
#include <unistd.h>
#include <stdlib.h>
#include <gsl/gsl_rng.h>
```

我无法确切告诉你它看起来像什么，因为我不知道你日常到底用什么。

现在你有了这个聚合头文件，你可以在你写的每个文件顶部只放一个：

```c
#include <allheads.h>
```

就完成了对头文件的思考。当然，它会展开为也许 10,000 行额外代码，其中大部分与手头的程序无关。但你不会注意到，未使用的声明不会改变最终可执行文件。

如果你正在为其他用户编写公共头文件，那么根据头文件不应包含不必要元素的原则，你的头文件可能不应该有 `#include "allheads.h"` 读入标准库的所有定义和声明——事实上，你的公共头文件可能根本没有任何来自标准库的元素是合理的。这通常是正确的：你的库可能有一个使用 GLib 链表操作的代码段，但这意味着你需要在那个代码文件中 `#include <glib.h>`，而不是在公共库头文件中。
:::

回到在命令行上设置快速编译的想法，统一头文件使编写快速程序更快速。一旦你有了统一头文件，即使是 `#include <allheads.h>` 这样的行，如果你是 gcc 或 clang 用户也是多余的，因为你可以改为将 `-include allheads.h` 添加到 CFLAGS，就再也不用想包含哪些项目外头文件了。

### Here Document

Here document 是 POSIX 标准 shell 的一个特性，你可以用于 C、Python、Perl 或其他任何东西，它们将使本书更有用和有趣。此外，如果你想要多语言脚本，here document 是做到这一点的简单方法。用 Perl 做一些解析，用 C 做数学，然后让 Gnuplot 生成漂亮的图片，全部在一个文本文件中。

这是一个 Python 示例。通常，你会告诉 Python 通过以下方式运行脚本：

```bash
python your_script.py
```

Python 让你给文件名 `-` 来使用 stdin 作为输入文件：

```bash
echo "print 'hi.'" | python -
```

理论上，你可以通过 echo 在命令行上放一些冗长的脚本，但你会很快看到有很多小的、不需要的解析在进行——例如，你可能需要 `\"hi\"` 而不是 `"hi"`。

因此，here document，它根本不做解析。试试这个：

```bash
python - <<"XXXX"
lines=2
print "\nThis script is %i lines long.\n" %(lines,)
XXXX
```

- Here document 是标准 shell 特性，所以它们应该在任何 POSIX 系统上工作。
- "XXXX" 是你喜欢的任何字符串；"EOF" 也很流行，"-----" 只要顶部和底部的破折号数量匹配就好看。当 shell 在单独一行看到你选择的字符串时，它将停止向程序的 stdin 发送脚本。这就是发生的所有解析。
- 还有一个以 `<<-` 开头的变体。该变体删除每行开头的所有制表符，所以你可以将 here document 放在 shell 脚本的缩进部分而不破坏缩进流程。当然，这对 Python here document 将是灾难性的。
- 作为另一个变体，`<<"XXXX"` 和 `<<XXXX` 之间有区别。在第二个版本中，shell 解析某些元素，这意味着你可以让 shell 为你插入 $shell_variables 的值。Shell 严重依赖 $ 表示变量和其他展开；$ 是标准键盘上对 C 没有特殊含义的少数字符之一。就好像写 Unix 的人从头设计它来让编写产生 C 代码的 shell 脚本变得容易……

### 从 stdin 编译

好的，回到 C：我们可以使用 here document 通过 gcc 或 clang 将粘贴到命令行的 C 代码编译，或在多语言脚本中有几行 C。

我们不使用 makefile，所以需要单个编译命令。为了让生活不那么痛苦，让我们给它起个别名。将此粘贴到命令行，或添加到 .bashrc、.zshrc 或任何适用的地方：

```bash
go_libs="-lm"
go_flags="-g -Wall -include allheads.h -O3"
alias go_c="c99 -xc - $go_libs $go_flags"
```

其中 allheads.h 是你之前组装的聚合头文件。使用 -include 标志意味着写 C 代码时少想一件事，我发现当 C 代码中有 # 时 bash 的历史会变得古怪。

在编译行上，你会认出 `-` 表示不是从命名文件读取，而是使用 stdin。`-xc` 将其识别为 C 代码，因为 gcc 代表 GNU Compiler Collection，不是 GNU C Compiler，没有以 .c 结尾的输入文件名来提示它，我们必须明确这不是 Java、Fortran、Objective C、Ada 或 C++（clang  likewise，尽管它的名字意在唤起 C 语言）。

无论你在 makefile 中为自定义 LDLIBS 和 CFLAGS 做了什么，在这里做。

现在我们启航了，可以在命令行上编译 C 代码：

```bash
go_c << '---'
int main(){printf("Hello from the command line.\n");}
---
./a.out
```

我们可以使用 here document 将短 C 程序粘贴到命令行，轻松编写小测试程序。你不仅不需要 makefile，你甚至不需要输入文件。

不要指望这种事成为你的主要工作方式。但在命令行上剪切粘贴代码片段可以很有趣，能够在更长的 shell 脚本中有一个 C 步骤是相当棒的。
