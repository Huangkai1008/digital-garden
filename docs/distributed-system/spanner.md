## 摘要

**Spanner**（扳手）是 Google 的可扩展、多版本、全球分布式、同步复制的数据库。它是首个在全球规模下分布数据并支持**外部一致性**（externally consistent）分布式事务的系统。本文描述 Spanner 的结构、功能集、各种设计决策背后的理由，以及一种暴露时钟不确定性的新型时间 API。该 API 及其实现对于支持外部一致性和多种强大特性至关重要：过去时刻的**非阻塞读取**（non-blocking reads）、**无锁只读事务**（lock-free read-only transactions），以及跨整个 Spanner 的**原子模式变更**（atomic schema changes）。

## 1 引言

Spanner 是 Google 设计、构建和部署的可扩展、全球分布式数据库。在最高抽象层次上，它是一个将数据分片到遍布全球数据中心的许多 **Paxos** [21] 状态机集合上的数据库。复制用于实现全球可用性和地理局部性；客户端在副本之间自动故障转移。Spanner 会随着数据量或服务器数量的变化自动在机器之间重新分片数据，并自动在机器之间（甚至跨数据中心）迁移数据以平衡负载并应对故障。Spanner 设计为可扩展到数百个数据中心、数百万台机器和数万亿行数据库记录。

应用可以使用 Spanner 实现高可用性，即使面对广域自然灾害，也可通过在洲内甚至跨洲复制数据。我们的首个客户是 **F1** [35]，即 Google 广告后端的重写版本。F1 在美国境内分布了五个副本。大多数其他应用可能会在一个地理区域内跨 3 到 5 个数据中心复制数据，但具有相对独立的故障模式。也就是说，只要能够承受 1 或 2 个数据中心故障，大多数应用会选择较低延迟而非更高可用性。

Spanner 的主要关注点是管理跨数据中心复制的数据，但我们也花费了大量时间在分布式系统基础设施之上设计和实现重要的数据库特性。尽管许多项目乐于使用 **Bigtable** [9]，我们也持续收到用户抱怨 Bigtable 对某些类型的应用难以使用：那些具有复杂、不断演进的模式的应用，或那些在广域复制下需要强一致性的应用。（其他作者也提出过类似主张 [37]。）Google 的许多应用选择使用 **Megastore** [5]，因其半关系数据模型和对同步复制的支持，尽管其写吞吐量相对较低。因此，Spanner 已从类似 Bigtable 的版本化键值存储演变为**时序多版本数据库**（temporal multi-version database）。数据存储在模式化的半关系表中；数据被版本化，每个版本自动以其提交时间戳标记；旧版本数据受可配置的垃圾回收策略约束；应用可以读取过去时间戳的数据。Spanner 支持通用事务，并提供基于 **SQL** 的查询语言。

作为全球分布式数据库，Spanner 提供若干有趣特性。首先，数据的复制配置可由应用在细粒度上动态控制。应用可以指定约束来控制哪些数据中心包含哪些数据、数据与用户的距离（以控制读延迟）、副本之间的距离（以控制写延迟），以及维护多少副本（以控制持久性、可用性和读性能）。数据也可以由系统在数据中心之间动态、透明地移动，以平衡数据中心间的资源使用。其次，Spanner 具有两个在分布式数据库中难以实现的特性：它提供**外部一致性**（externally consistent）[16] 的读写，以及数据库在某一时间戳下的**全局一致读取**（globally-consistent reads）。这些特性使 Spanner 能够支持一致备份、一致的 **MapReduce** 执行 [12] 和原子模式更新，全部在全球规模下实现，甚至在存在进行中事务的情况下。

这些特性之所以能够实现，是因为 Spanner 为事务分配具有全局意义的提交时间戳，即使事务可能是分布式的。时间戳反映**串行化顺序**（serialization order）。此外，串行化顺序满足外部一致性（或等价地，**线性一致性**（linearizability）[20]）：若事务 T1 在另一事务 T2 开始之前提交，则 T1 的提交时间戳小于 T2 的。Spanner 是首个在全球规模下提供此类保证的系统。

实现这些性质的关键是新的 **TrueTime** API 及其实现。该 API 直接暴露时钟不确定性，Spanner 时间戳的保证取决于实现提供的边界。若不确定性较大，Spanner 会减速以等待该不确定性过去。Google 的集群管理软件提供 TrueTime API 的实现。该实现通过使用多种现代时钟参考（GPS 和原子钟）将不确定性保持在较小水平（通常小于 10ms）。

第 2 节描述 Spanner 实现的结构、功能集及设计中的工程决策。第 3 节描述我们新的 TrueTime API 并概述其实现。第 4 节描述 Spanner 如何使用 TrueTime 实现外部一致的分布式事务、无锁只读事务和原子模式更新。第 5 节提供 Spanner 性能和 TrueTime 行为的一些基准测试，并讨论 F1 的经验。第 6、7、8 节描述相关工作、未来工作和总结。

## 2 实现

本节描述 Spanner 实现的结构和背后的理由。然后描述**目录**（directory）抽象，用于管理复制和局部性，是数据移动的单位。最后描述我们的数据模型、为何 Spanner 看起来像关系数据库而非键值存储，以及应用如何控制数据局部性。

Spanner 的部署称为 **universe**（宇宙）。鉴于 Spanner 在全球管理数据，运行中的 universe 将只有少数几个。我们目前运行一个测试/沙盒 universe、一个开发/生产 universe 和一个仅生产 universe。

Spanner 组织为一组 **zone**（区域），每个区域大致相当于 Bigtable 服务器 [9] 的一次部署。区域是管理部署的单位。区域集合也是数据可以复制的位置集合。随着新数据中心投入使用和旧数据中心关闭，可以向运行中的系统添加或移除区域。区域也是物理隔离的单位：例如，若不同应用的数据必须分区到同一数据中心的不同服务器集合，则一个数据中心可能有一个或多个区域。

::: tip 图 1：Spanner 服务器组织
:::

```text
                    ┌─────────────────────────────────────────┐
                    │           Universe Master              │
                    │     (状态信息控制台，交互调试)            │
                    └─────────────────────────────────────────┘
                                       │
                    ┌──────────────────┼──────────────────┐
                    │                  │                  │
                    ▼                  ▼                  ▼
            ┌───────────────┐  ┌───────────────┐  ┌───────────────┐
            │  Zone 1       │  │  Zone 2       │  │  Zone 3       │  ...
            │  Zonemaster   │  │  Zonemaster   │  │  Zonemaster   │
            │  Spanservers  │  │  Spanservers  │  │  Spanservers  │
            │  Location     │  │  Location     │  │  Location     │
```

图 1 展示 Spanner universe 中的服务器。一个区域有一个 **zonemaster**（区域主控）和一百到数千个 **spanserver**（跨度服务器）。前者将数据分配给 spanserver；后者向客户端提供数据服务。每区域的 **location proxy**（位置代理）供客户端定位为其数据服务的 spanserver。**universe master**（宇宙主控）和 **placement driver**（放置驱动）目前是单例。universe master 主要是一个控制台，显示所有区域的状态信息以供交互调试。placement driver 处理跨区域的自动化数据移动，时间尺度为分钟级。placement driver 定期与 spanserver 通信，查找需要移动的数据，以满足更新的复制约束或平衡负载。由于篇幅限制，我们仅详细描述 spanserver。

### 2.1 Spanserver 软件栈

本节聚焦 spanserver 实现，以说明复制和分布式事务如何分层到我们基于 Bigtable 的实现之上。软件栈如图 2 所示。在底层，每个 spanserver 负责 100 到 1000 个称为 **tablet**（片）的数据结构实例。tablet 类似于 Bigtable 的 tablet 抽象，它实现以下映射的集合：

```text
(key:string, timestamp:int64) → string
```

与 Bigtable 不同，Spanner 为数据分配时间戳，这是 Spanner 更像多版本数据库而非键值存储的重要方式。

::: tip 图 2：Spanserver 软件栈
:::

```text
┌─────────────────────────────────────────────────────────────┐
│  Lock Table (锁表)          Transaction Manager (事务管理器)   │
│  - 两阶段锁状态               - 分布式事务参与者/协调者          │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────┐
│  Paxos State Machine (Paxos 状态机)                          │
│  - 每个 tablet 一个，长存活 leader，10 秒租约                 │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────┐
│  Tablet (片)                                                 │
│  - (key, timestamp) → value                                  │
│  - B-tree 文件 + WAL，存储在 Colossus 分布式文件系统           │
└─────────────────────────────────────────────────────────────┘
```

tablet 的状态存储在一组类似 B 树的文件和**预写日志**（write-ahead log，WAL）中，全部位于名为 **Colossus**（Google 文件系统 [15] 的继任者）的分布式文件系统上。

为支持复制，每个 spanserver 在每个 tablet 之上实现单个 Paxos 状态机。（早期 Spanner 版本支持每个 tablet 多个 Paxos 状态机，允许更灵活的复制配置。该设计的复杂性使我们放弃了它。）每个状态机将其元数据和日志存储在其对应的 tablet 中。我们的 Paxos 实现支持具有基于时间的 leader 租约的长存活 leader，默认长度为 10 秒。当前 Spanner 实现将每次 Paxos 写记录两次：一次在 tablet 的日志中，一次在 Paxos 日志中。这一选择出于权宜之计，我们可能最终会改进。我们的 Paxos 实现是流水线式的，以提高 Spanner 在广域延迟下的吞吐量；但 Paxos 按顺序应用写（我们将在第 4 节依赖这一事实）。

Paxos 状态机用于实现一致复制的映射集合。每个副本的键值映射状态存储在其对应的 tablet 中。写必须在 leader 处发起 Paxos 协议；读从任何足够新的副本的底层 tablet 直接访问状态。副本集合共同构成一个 **Paxos group**（Paxos 组）。

在每个作为 leader 的副本处，每个 spanserver 实现一个**锁表**（lock table）以实现并发控制。锁表包含**两阶段锁**（two-phase locking）的状态：它将键范围映射到锁状态。（注意，拥有长存活的 Paxos leader 对于高效管理锁表至关重要。）在 Bigtable 和 Spanner 中，我们都为长存活事务（例如报告生成，可能需要数分钟）而设计，这类事务在存在冲突时在**乐观并发控制**（optimistic concurrency control）下表现不佳。需要同步的操作（如事务性读取）在锁表中获取锁；其他操作绕过锁表。

在每个作为 leader 的副本处，每个 spanserver 还实现一个**事务管理器**（transaction manager）以支持分布式事务。事务管理器用于实现**参与者 leader**（participant leader）；组中的其他副本称为**参与者 slave**（participant slave）。若事务仅涉及一个 Paxos 组（大多数事务的情况），它可以绕过事务管理器，因为锁表和 Paxos 共同提供事务性。若事务涉及多个 Paxos 组，这些组的 leader 协调执行**两阶段提交**（two-phase commit）。选择其中一个参与者组作为**协调者**（coordinator）：该组的参与者 leader 称为**协调者 leader**（coordinator leader），该组的 slave 称为**协调者 slave**（coordinator slave）。每个事务管理器的状态存储在底层 Paxos 组中（因此被复制）。

### 2.2 目录与放置

在键值映射集合之上，Spanner 实现支持称为**目录**（directory）的桶抽象，目录是共享共同前缀的一组连续键。（术语 directory 的选择是历史偶然；更好的术语可能是 bucket。）我们将在 2.3 节解释该前缀的来源。支持目录使应用能够通过谨慎选择键来控制数据的局部性。

目录是数据放置的单位。目录中的所有数据具有相同的复制配置。当数据在 Paxos 组之间移动时，按目录逐个移动，如图 3 所示。Spanner 可能移动目录以减轻 Paxos 组的负载；将经常一起访问的目录放入同一组；或将目录移入更靠近其访问者的组。目录可以在客户端操作进行时移动。可以预期，一个 50MB 的目录可以在几秒内移动完成。

::: tip 图 3：目录是 Paxos 组之间数据移动的单位
:::

```text
    Paxos Group A                    Paxos Group B
    ┌─────────────┐                  ┌─────────────┐
    │ Directory 1 │                  │ Directory 4 │
    │ Directory 2 │  ──movedir──►    │ Directory 5 │
    │ Directory 3 │                  │ Directory 6 │
    └─────────────┘                  └─────────────┘
```

Paxos 组可能包含多个目录这一事实意味着 Spanner tablet 与 Bigtable tablet 不同：前者不一定是行空间的单个字典序连续分区。相反，Spanner tablet 是一个容器，可以封装行空间的多个分区。我们做出这一决定是为了能够将经常一起访问的多个目录并置。

**Movedir** 是用于在 Paxos 组之间移动目录的后台任务 [14]。Movedir 也用于向 Paxos 组添加或移除副本 [25]，因为 Spanner 尚不支持 Paxos 内配置变更。Movedir 未实现为单个事务，以避免在大量数据移动时阻塞进行中的读写。相反，movedir 注册其开始移动数据的事实，并在后台移动数据。当它移动了除名义数量外的所有数据后，使用事务原子地移动该名义数量并更新两个 Paxos 组的元数据。

目录也是其**地理复制属性**（或简称 **placement**，放置）可由应用指定的最小单位。我们放置规范语言的设计分离了管理复制配置的职责。管理员控制两个维度：副本的数量和类型，以及这些副本的地理放置。他们在这两个维度上创建命名选项菜单（例如，北美，5 副本加 1 见证）。应用通过用这些选项的组合标记每个数据库和/或单个目录来控制数据的复制方式。例如，应用可能将每个最终用户的数据存储在其自己的目录中，这将使用户 A 的数据在欧洲有三个副本，用户 B 的数据在北美有五个副本。

为表述清晰我们过度简化了。事实上，若目录增长过大，Spanner 会将其分片为多个**片段**（fragment）。片段可能由不同的 Paxos 组（因此不同的服务器）提供。Movedir 实际上在组之间移动的是片段，而非整个目录。

### 2.3 数据模型

Spanner 向应用暴露以下数据特性：基于模式化半关系表的数据模型、查询语言和通用事务。支持这些特性的推动因素众多。对模式化半关系表和同步复制的需求得到 **Megastore** [5] 流行的支持。Google 内部至少有 300 个应用使用 Megastore（尽管其性能相对较低），因其数据模型比 Bigtable 更易管理，且支持跨数据中心同步复制。（Bigtable 仅支持跨数据中心的最终一致复制。）使用 Megastore 的知名 Google 应用包括 Gmail、Picasa、Calendar、Android Market 和 AppEngine。在 Spanner 中支持类似 SQL 的查询语言的需求也很明确，鉴于 **Dremel** [28] 作为交互式数据分析工具的流行。最后，Bigtable 缺乏跨行事务导致频繁抱怨；**Percolator** [32] 部分是为了解决这一缺陷而构建的。一些作者声称通用两阶段提交因性能或可用性问题而过于昂贵 [9, 10, 19]。我们相信，让应用程序员在出现瓶颈时处理因过度使用事务导致的性能问题，比总是围绕缺乏事务进行编码更好。在 Paxos 上运行两阶段提交可以缓解可用性问题。

应用数据模型分层在实现支持的目录桶化键值映射之上。应用在 universe 中创建一个或多个**数据库**（database）。每个数据库可以包含无限数量的模式化**表**（table）。表看起来像关系数据库表，有行、列和版本化值。我们不会详述 Spanner 的查询语言。它看起来像 SQL，并有一些扩展以支持 protocol-buffer 值字段。

Spanner 的数据模型不是纯关系的，因为行必须有名称。更精确地说，每个表必须有一个或多个主键列的有序集合。这一要求是 Spanner 仍看起来像键值存储的地方：主键构成行的名称，每个表定义从主键列到非主键列的映射。仅当为行的键定义了某个值（即使为 NULL）时，行才存在。施加这一结构很有用，因为它让应用通过键的选择来控制数据局部性。

图 4 包含一个按用户、按相册存储照片元数据的 Spanner 模式示例。模式语言类似于 Megastore 的，但额外要求每个 Spanner 数据库必须由客户端分区为一个或多个表层次结构。客户端应用通过 `INTERLEAVE IN` 声明在数据库模式中声明层次结构。层次结构顶部的表是**目录表**（directory table）。目录表中键为 K 的每一行，连同字典序以 K 开头的后代表中的所有行，形成一个目录。`ON DELETE CASCADE` 表示删除目录表中的行会删除任何关联的子行。该图还展示了示例数据库的交错布局：例如，`Albums(2,1)` 表示用户 id 2、相册 id 1 的 Albums 表中的行。这种表交错形成目录的方式很重要，因为它允许客户端描述多表之间存在的局部性关系，这对于分片分布式数据库的良好性能是必要的。没有它，Spanner 将不知道最重要的局部性关系。

::: tip 图 4：照片元数据的 Spanner 模式示例及 INTERLEAVE IN 隐含的交错
:::

```sql
CREATE TABLE Users {
  uid INT64 NOT NULL, email STRING
} PRIMARY KEY (uid), DIRECTORY;

CREATE TABLE Albums {
  uid INT64 NOT NULL, aid INT64 NOT NULL,
  name STRING
} PRIMARY KEY (uid, aid),
INTERLEAVE IN PARENT Users ON DELETE CASCADE;
```

```text
交错布局示意：
Users(1) ─┬─ Albums(1,1)
          ├─ Albums(1,2)
          └─ Albums(1,3)
Users(2) ─┬─ Albums(2,1)
          └─ Albums(2,2)
```

## 3 TrueTime

本节描述 TrueTime API 并概述其实现。我们将大部分细节留给另一篇论文：我们的目标是展示拥有此类 API 的力量。表 1 列出了该 API 的方法。TrueTime 显式地将时间表示为 **TTinterval**，即具有有界时间不确定性的区间（与不向客户端提供不确定性概念的标准时间接口不同）。TTinterval 的端点是 **TTstamp** 类型。`TT.now()` 方法返回保证包含调用 `TT.now()` 期间的绝对时间的 TTinterval。时间纪元类似于带闰秒涂抹的 UNIX 时间。定义瞬时误差界为 ε（区间宽度的一半），平均误差界为 ε̄。`TT.after()` 和 `TT.before()` 方法是 `TT.now()` 的便捷包装。

| 方法 | 返回值 |
|------|--------|
| `TT.now()` | TTinterval: [earliest, latest] |
| `TT.after(t)` | 若 t 肯定已过去则为 true |
| `TT.before(t)` | 若 t 肯定尚未到达则为 true |

*表 1：TrueTime API。参数 t 为 TTstamp 类型。*

用函数 tabs(e) 表示事件 e 的绝对时间。更形式化地说，TrueTime 保证对于调用 tt = TT.now()，有 tt.earliest ≤ tabs(enow) ≤ tt.latest，其中 enow 是调用事件。

TrueTime 使用的基础时间参考是 GPS 和原子钟。TrueTime 使用两种形式的时间参考，因为它们有不同的故障模式。GPS 参考源的脆弱性包括天线和接收器故障、本地无线电干扰、相关故障（例如设计缺陷如错误的闰秒处理和欺骗）以及 GPS 系统中断。原子钟可能以与 GPS 和彼此不相关的方式故障，且随时间推移可能因频率误差而显著漂移。

TrueTime 由每个数据中心的一组 **time master**（时间主控）机器和每台机器的 **timeslave daemon**（时间从属守护进程）实现。大多数 master 配备带专用天线的 GPS 接收器；这些 master 在物理上分离以减少天线故障、无线电干扰和欺骗的影响。其余的 master（我们称为 **Armageddon master**）配备原子钟。原子钟并不昂贵：Armageddon master 的成本与 GPS master 同数量级。所有 master 的时间参考定期相互比较。每个 master 还将其参考推进时间的速率与自己的本地时钟进行交叉检查，若存在显著偏差则驱逐自己。在同步之间，Armageddon master 通告缓慢增加的时间不确定性，该不确定性来自保守应用的最坏情况时钟漂移。GPS master 通告的不确定性通常接近零。

每个守护进程轮询多种 master [29] 以减少对任一 master 错误的脆弱性。有些是来自附近数据中心的 GPS master；其余是来自较远数据中心的 GPS master，以及一些 Armageddon master。守护进程应用 **Marzullo 算法** [27] 的变体来检测和拒绝说谎者，并将本地机器时钟同步到非说谎者。为保护免受损坏的本地时钟影响，表现出大于从组件规格和操作环境推导的最坏情况边界的频率偏移的机器会被驱逐。

在同步之间，守护进程通告缓慢增加的时间不确定性。ε 来自保守应用的最坏情况本地时钟漂移。ε 还取决于 time master 不确定性和与 time master 的通信延迟。在我们的生产环境中，ε 通常是时间的锯齿函数，在每个轮询区间内从约 1ms 变化到 7ms。因此 ε̄ 大多数时间为 4ms。守护进程的轮询区间目前为 30 秒，当前应用的漂移率设为 200 微秒/秒，共同解释了从 0 到 6ms 的锯齿边界。剩余的 1ms 来自与 time master 的通信延迟。在存在故障时可能出现偏离此锯齿的情况。例如，偶尔的 time master 不可用可能导致数据中心范围内 ε 的增加。类似地，过载的机器和网络链路可能导致偶尔的局部 ε 尖峰。

## 4 并发控制

本节描述如何使用 TrueTime 保证并发控制周围的正确性性质，以及如何使用这些性质实现外部一致事务、无锁只读事务和过去时刻的非阻塞读取等特性。这些特性使例如以下保证成为可能：在时间戳 t 进行的全数据库审计读取将精确看到截至 t 已提交的每个事务的效果。

接下来，区分 Paxos 所见的写（我们将在上下文清晰时称为 Paxos 写）与 Spanner 客户端写很重要。例如，两阶段提交为准备阶段生成一个 Paxos 写，但没有对应的 Spanner 客户端写。

### 4.1 时间戳管理

表 2 列出了 Spanner 支持的操作类型。Spanner 实现支持读写事务、只读事务（预声明的快照隔离事务）和快照读取。独立写实现为读写事务；非快照独立读实现为只读事务。两者都在内部重试（客户端无需编写自己的重试循环）。

**只读事务**（read-only transaction）是一种具有**快照隔离**（snapshot isolation）[6] 性能优势的事务。只读事务必须预声明为没有任何写；它不仅仅是没有任何写的读写事务。只读事务中的读在系统选择的时间戳处无锁执行，因此传入的写不会被阻塞。只读事务中的读的执行可以在任何足够新的副本上进行（4.1.3 节）。

**快照读取**（snapshot read）是过去时刻的读取，无锁执行。客户端可以为快照读取指定时间戳，或提供期望时间戳陈旧度的上界并让 Spanner 选择时间戳。无论哪种情况，快照读取的执行在任何足够新的副本上进行。

对于只读事务和快照读取，一旦选择了时间戳，提交就不可避免，除非该时间戳的数据已被垃圾回收。因此，客户端可以避免在重试循环内缓冲结果。当服务器故障时，客户端可以通过重复时间戳和当前读位置在内部在另一台服务器上继续查询。

| 只读事务 | § 4.1.4 | 无锁 | 时间戳用 leader；读用任意，受 § 4.1.3 约束 |

#### 4.1.1 Paxos Leader 租约

Spanner 的 Paxos 实现使用定时租约使 leadership 长存活（默认 10 秒）。潜在 leader 发送定时租约投票请求；在收到法定人数的租约投票后，leader 知道它拥有租约。副本在成功写时隐式延长其租约投票，leader 在租约投票临近过期时请求延长。定义 leader 的**租约区间**（lease interval）从其发现拥有法定人数租约投票时开始，到其不再拥有法定人数租约投票时结束（因为有些已过期）。Spanner 依赖以下**不相交不变式**（disjointness invariant）：对于每个 Paxos 组，每个 Paxos leader 的租约区间与所有其他 leader 的互不相交。附录 A 描述如何强制此不变式。

Spanner 实现允许 Paxos leader 通过释放其 slave 的租约投票而**退位**（abdicate）。为保持不相交不变式，Spanner 约束何时允许退位。定义 smax 为 leader 使用的最大时间戳。后续章节将描述何时推进 smax。在退位之前，leader 必须等待直到 `TT.after(smax)` 为 true。

#### 4.1.2 为读写事务分配时间戳

事务性读写使用两阶段锁。因此，它们可以在所有锁已获取但任何锁尚未释放的任何时刻分配时间戳。对于给定事务，Spanner 为其分配 Paxos 分配给表示事务提交的 Paxos 写的时间戳。

Spanner 依赖以下**单调性不变式**（monotonicity invariant）：在每个 Paxos 组内，Spanner 以单调递增顺序为 Paxos 写分配时间戳，即使跨 leader。单个 leader 副本可以平凡地以单调递增顺序分配时间戳。通过利用不相交不变式在跨 leader 时强制此不变式：leader 必须仅在其 leader 租约区间内分配时间戳。注意，每当分配时间戳 s 时，smax 推进到 s 以保持不相交。

Spanner 还强制以下**外部一致性不变式**（external-consistency invariant）：若事务 T2 的开始发生在事务 T1 的提交之后，则 T2 的提交时间戳必须大于 T1 的提交时间戳。用 estart_i 和 ecommit_i 定义事务 Ti 的开始和提交事件；用 si 定义事务 Ti 的提交时间戳。不变式变为：tabs(ecommit_1) < tabs(estart_2) ⇒ s1 < s2。

执行事务和分配时间戳的协议遵守两条规则，它们共同保证此不变式，如下所示。定义写 Ti 的提交请求到达协调者 leader 的到达事件为 eserver_i。

**Start 规则**：写 Ti 的协调者 leader 分配的提交时间戳 si 不小于在 eserver_i 之后计算的 `TT.now().latest` 的值。注意参与者 leader 在此不重要；4.2.1 节描述它们如何参与下一条规则的实现。

**Commit Wait 规则**：协调者 leader 确保客户端在 `TT.after(si)` 为 true 之前无法看到 Ti 提交的任何数据。提交等待确保 si 小于 Ti 的绝对提交时间，即 si < tabs(ecommit_i)。4.2.1 节描述提交等待的实现。证明：

```text
s1 < tabs(ecommit_1)           (commit wait)
tabs(ecommit_1) < tabs(estart_2)  (假设)
tabs(estart_2) ≤ tabs(eserver_2)  (因果性)
tabs(eserver_2) ≤ s2             (start)
s1 < s2                          (传递性)
```

#### 4.1.3 在时间戳处服务读取

4.1.2 节描述的单调性不变式允许 Spanner 正确判断副本的状态是否足够新以满足读取。每个副本跟踪称为 **safe time**（安全时间）tsafe 的值，即副本已更新的最大时间戳。若 t <= tsafe，副本可以满足在时间戳 t 的读取。

定义 tsafe = min(tPaxos_safe, tTM_safe)，其中每个 Paxos 状态机有一个安全时间 tPaxos_safe，每个事务管理器有一个安全时间 tTM_safe。tPaxos_safe 更简单：它是最高已应用 Paxos 写的时间戳。由于时间戳单调递增且写按顺序应用，就 Paxos 而言，在 tPaxos_safe 或更低处将不再发生写。

若存在零个已准备（但未提交）的事务——即处于两阶段提交两阶段之间的事务，则 tTM_safe 在副本处为 ∞。（对于参与者 slave，tTM_safe 实际上指副本 leader 的事务管理器，其状态 slave 可以通过 Paxos 写传递的元数据推断。）若存在任何此类事务，则受这些事务影响的状态是不确定的：参与者副本尚不知道这些事务是否会提交。如 4.2.1 节所述，提交协议确保每个参与者知道已准备事务时间戳的下界。事务 Ti 的每个参与者 leader（对于组 g）为其准备记录分配准备时间戳 sprepare_i,g。协调者 leader 确保事务的提交时间戳 si >= 所有参与者组 g 的 sprepare_i,g。因此，对于组 g 中的每个副本，在所有在 g 准备的事务 Ti 上，tTM_safe = 在 g 准备的所有事务上的 mini(sprepare_i,g) − 1。

#### 4.1.4 为只读事务分配时间戳

只读事务分两阶段执行：分配时间戳 sread [8]，然后以 sread 处的快照读取执行事务的读。快照读取可以在任何足够新的副本上执行。

在事务开始后的任何时刻，简单地分配 sread = TT.now().latest 通过类似于 4.1.2 节对写提出的论证保持外部一致性。然而，这样的时间戳可能要求 sread 处的数据读执行阻塞，若 tsafe 尚未充分推进。（此外，注意选择 sread 的值也可能推进 smax 以保持不相交。）为减少阻塞概率，Spanner 应分配保持外部一致性的最旧时间戳。4.2.2 节解释如何选择这样的时间戳。

### 4.2 实现细节

本节解释读写事务和只读事务的一些实践细节（之前省略的），以及用于实现原子模式变更的特殊事务类型的实现。然后描述所述基本方案的若干改进。

#### 4.2.1 读写事务

与 Bigtable 一样，事务中发生的写在客户端缓冲直到提交。因此，事务中的读看不到事务写的效果。这一设计在 Spanner 中运作良好，因为读返回任何读取数据的时间戳，未提交的写尚未分配时间戳。

读写事务中的读使用 **wound-wait** [33] 避免死锁。客户端向适当组的 leader 副本发出读，leader 获取读锁然后读取最新数据。当客户端事务保持打开时，它发送 keepalive 消息以防止参与者 leader 使事务超时。当客户端完成所有读并缓冲所有写后，它开始两阶段提交。客户端选择协调者组并向每个参与者的 leader 发送提交消息，包含协调者身份和任何缓冲的写。让客户端驱动两阶段提交避免在广域链路上两次发送数据。

非协调者参与者 leader 首先获取写锁。然后它选择必须大于其已分配给先前事务的任何时间戳的准备时间戳（以保持单调性），并通过 Paxos 记录准备记录。每个参与者然后通知协调者其准备时间戳。

协调者 leader 也首先获取写锁，但跳过准备阶段。它在听到所有其他参与者 leader 后为整个事务选择时间戳。提交时间戳 s 必须大于或等于所有准备时间戳（以满足 4.1.3 节讨论的约束），大于协调者收到其提交消息时 TT.now().latest，且大于 leader 已分配给先前事务的任何时间戳（再次，以保持单调性）。协调者 leader 然后通过 Paxos 记录提交记录（或在等待其他参与者时超时则中止）。

在允许任何协调者副本应用提交记录之前，协调者 leader 等待直到 TT.after(s)，以遵守 4.1.2 节描述的提交等待规则。由于协调者 leader 基于 TT.now().latest 选择 s，现在等待直到该时间戳保证已过去，预期等待至少 2ε。此等待通常与 Paxos 通信重叠。提交等待后，协调者向客户端和所有其他参与者 leader 发送提交时间戳。每个参与者 leader 通过 Paxos 记录事务结果。所有参与者在同一时间戳应用然后释放锁。

#### 4.2.2 只读事务

分配时间戳需要参与读取的所有 Paxos 组之间的协商阶段。因此，Spanner 要求每个只读事务有一个**范围表达式**（scope expression），即总结整个事务将读取的键的表达式。Spanner 自动推断独立查询的范围。

若范围的值由单个 Paxos 组提供，则客户端向该组的 leader 发出只读事务。（当前 Spanner 实现仅在 Paxos leader 处为只读事务选择时间戳。）该 leader 分配 sread 并执行读。对于单站点读，Spanner 通常比 TT.now().latest 做得更好。定义 LastTS() 为 Paxos 组中最后提交写的时间戳。若没有已准备事务，分配 sread = LastTS() 平凡地满足外部一致性：事务将看到最后一次写的结果，因此在其之后排序。

若范围的值由多个 Paxos 组提供，有几种选择。最复杂的选项是与所有组的 leader 进行一轮通信，基于 LastTS() 协商 sread。Spanner 当前实现更简单的选择。客户端避免协商轮，仅让读在 sread = TT.now().latest 处执行（可能等待安全时间推进）。事务中的所有读可以发送到足够新的副本。

#### 4.2.3 模式变更事务

TrueTime 使 Spanner 能够支持原子模式变更。使用标准事务不可行，因为参与者数量（数据库中组的数量）可能达到数百万。Bigtable 在单个数据中心内支持原子模式变更，但其模式变更会阻塞所有操作。

Spanner 模式变更事务是标准事务的通常非阻塞变体。首先，它显式分配未来的时间戳，在准备阶段注册。因此，跨数千台服务器的模式变更可以以对其他并发活动的最小干扰完成。其次，隐式依赖模式的读写在时间 t 与任何注册的模式变更时间戳同步：若其时间戳在 t 之前，它们可以继续；若其时间戳在 t 之后，它们必须阻塞在模式变更事务之后。没有 TrueTime，定义模式变更在 t 发生将是无意义的。

#### 4.2.4 改进

上述定义的 tTM_safe 有一个弱点：单个已准备事务阻止 tsafe 推进。因此，即使读不与事务冲突，也无法在更晚时间戳进行读。可以通过用从键范围到已准备事务时间戳的细粒度映射增强 tTM_safe 来消除此类假冲突。此信息可以存储在锁表中，锁表已映射键范围到锁元数据。当读到达时，只需针对读冲突的键范围的细粒度安全时间进行检查。

上述定义的 LastTS() 有类似弱点：若事务刚提交，非冲突的只读事务仍必须分配 sread 以跟随该事务。因此，读的执行可能被延迟。可以通过用锁表中从键范围到提交时间戳的细粒度映射增强 LastTS() 来类似地弥补此弱点。（我们尚未实现此优化。）当只读事务到达时，可以通过取事务冲突的键范围的 LastTS() 的最大值来分配其时间戳，除非存在冲突的已准备事务（可从细粒度安全时间确定）。

上述定义的 tPaxos_safe 有一个弱点：在缺乏 Paxos 写时无法推进。即，在 t 处的快照读无法在最后写发生在 t 之前的 Paxos 组上执行。Spanner 通过利用 leader 租约区间的不相交性解决此问题。每个 Paxos leader 通过保持未来写时间戳将发生的阈值以上来推进 tPaxos_safe：它维护从 Paxos 序列号 n 到可能分配给 Paxos 序列号 n+1 的最小时间戳的映射 MinNextTS(n)。当副本已应用到 n 时，可以将 tPaxos_safe 推进到 MinNextTS(n) − 1。

单个 leader 可以轻松强制其 MinNextTS() 承诺。由于 MinNextTS() 承诺的时间戳在 leader 的租约内，不相交不变式在跨 leader 时强制 MinNextTS() 承诺。若 leader 希望将 MinNextTS() 推进到其 leader 租约结束之外，它必须首先延长其租约。注意 smax 总是推进到 MinNextTS() 中的最高值以保持不相交。

leader 默认每 8 秒推进 MinNextTS() 值。因此，在缺乏已准备事务时，空闲 Paxos 组中的健康 slave 在最坏情况下可以服务 8 秒前时间戳的读。leader 也可以应 slave 请求按需推进 MinNextTS() 值。

## 5 评估

我们首先测量 Spanner 在复制、事务和可用性方面的性能。然后提供一些 TrueTime 行为数据，以及我们首个客户 F1 的案例研究。

### 5.1 微基准测试

表 3 展示 Spanner 的一些微基准测试。这些测量在分时机器上进行：每个 spanserver 运行在 4GB RAM 和 4 核（AMD Barcelona 2200MHz）的调度单元上。客户端在独立机器上运行。每个区域包含一个 spanserver。客户端和区域放置在网络距离小于 1ms 的一组数据中心中。（这样的布局应该是常见的：大多数应用不需要在全球分布所有数据。）测试数据库用 50 个 Paxos 组和 2500 个目录创建。操作为 4KB 的独立读写。所有读在压缩后从内存提供，因此我们仅测量 Spanner 调用栈的开销。此外，首先进行一轮未测量的读以预热任何位置缓存。

对于延迟实验，客户端发出足够少的操作以避免在服务器处排队。从 1 副本实验来看，提交等待约 5ms，Paxos 延迟约 9ms。随着副本数量增加，延迟大致保持恒定，标准差更小，因为 Paxos 在组的副本上并行执行。随着副本数量增加，达到法定人数的延迟对单个 slave 副本的缓慢更不敏感。

### 5.3 TrueTime

关于 TrueTime 必须回答两个问题：ε 是否真正是时钟不确定性的边界，以及 ε 会变得多糟？对于前者，最严重的问题将是本地时钟漂移大于 200μs/秒：那将破坏 TrueTime 所做的假设。我们的机器统计显示坏 CPU 比坏时钟可能性高 6 倍。即，相对于更严重的硬件问题，时钟问题极其罕见。因此，我们相信 TrueTime 的实现与 Spanner 依赖的任何其他软件一样可信。

### 5.4 F1

Spanner 于 2011 年初开始在生产工作负载下进行实验评估，作为 Google 广告后端重写 **F1** [35] 的一部分。此后端最初基于手动多路分片的 MySQL 数据库。未压缩数据集为数十 TB，与许多 NoSQL 实例相比较小，但足以对分片 MySQL 造成困难。MySQL 分片方案将每个客户及其相关数据分配到固定分片。此布局支持按客户使用索引和复杂查询处理，但需要在应用业务逻辑中了解分片。随着客户数量及其数据的增长，对这种收入关键数据库进行重新分片极其昂贵。上次重新分片花费了两年多的密集努力，涉及跨数十个团队的协调和测试以最小化风险。此操作过于复杂无法定期进行：因此，团队不得不通过将部分数据存储在外部 Bigtable 中来限制 MySQL 数据库的增长，这损害了事务行为和跨所有数据查询的能力。

F1 团队选择使用 Spanner 有几个原因。首先，Spanner 消除了手动重新分片的需要。其次，Spanner 提供同步复制和自动故障转移。使用 MySQL 主从复制，故障转移困难，且存在数据丢失和停机风险。第三，F1 需要强事务语义，这使得使用其他 NoSQL 系统不切实际。应用语义要求跨任意数据的事务和一致读取。F1 团队还需要其数据的二级索引（因为 Spanner 尚不提供二级索引的自动支持），并能够使用 Spanner 事务实现自己的一致全局索引。

所有应用写现在默认通过 F1 发送到 Spanner，而非基于 MySQL 的应用栈。F1 在美国西海岸有 2 个副本，东海岸有 3 个。选择这些副本站点是为了应对潜在重大自然灾害导致的故障，以及其前端站点的选择。轶事上，Spanner 的自动故障转移对他们几乎不可见。尽管过去几个月有非计划集群故障，F1 团队最多需要做的是更新其数据库模式，告诉 Spanner 优先放置 Paxos leader 的位置，以保持它们靠近前端移动的位置。

Spanner 的时间戳语义使 F1 能够高效维护从数据库状态计算的内存数据结构。F1 维护所有变更的逻辑历史日志，作为每个事务的一部分写入 Spanner 本身。F1 在时间戳处获取数据的完整快照以初始化其数据结构，然后读取增量变更以更新它们。

## 6 相关工作

作为存储服务的跨数据中心一致复制已由 **Megastore** [5] 和 **DynamoDB** [3] 提供。DynamoDB 提供键值接口，仅在区域内复制。Spanner 遵循 Megastore 提供半关系数据模型，甚至类似的模式语言。Megastore 未实现高性能。它分层在 Bigtable 之上，施加高通信成本。它也不支持长存活 leader：多个副本可能发起写。来自不同副本的所有写在 Paxos 协议中必然冲突，即使它们逻辑上不冲突：Paxos 组上的吞吐量在每秒几次写时崩溃。Spanner 提供更高性能、通用事务和外部一致性。

Pavlo 等人 [31] 比较了数据库和 MapReduce [12] 的性能。他们指出探索分层在分布式键值存储上的数据库功能的其他努力 [1, 4, 7, 41] 作为两个世界正在融合的证据。我们同意结论，但证明整合多个层次有其优势：例如，将并发控制与复制整合降低了 Spanner 中提交等待的成本。

在复制存储之上分层事务的概念至少可以追溯到 Gifford 的论文 [16]。**Scatter** [17] 是最近基于 DHT 的键值存储，在一致复制之上分层事务。Spanner 专注于提供比 Scatter 更高级的接口。Gray 和 Lamport [18] 描述基于 Paxos 的非阻塞提交协议。他们的协议比两阶段提交产生更多消息成本，这会加剧跨广泛分布组的提交成本。**Walter** [36] 提供快照隔离的变体，在数据中心内工作，但不在跨数据中心。相比之下，我们的只读事务提供更自然的语义，因为我们支持所有操作的外部一致性。

最近有大量关于减少或消除锁开销的工作。**Calvin** [40] 消除并发控制：它预分配时间戳然后按时间戳顺序执行事务。**H-Store** [39] 和 **Granola** [11] 各自支持其自己的事务类型分类，其中一些可以避免锁。这些系统都不提供外部一致性。Spanner 通过提供快照隔离支持来解决争用问题。

**VoltDB** [42] 是支持广域主从复制以进行灾难恢复的分片内存数据库，但不支持更一般的复制配置。它是所谓 **NewSQL** 的例子，即支持可扩展 SQL 的市场推动 [38]。许多商业数据库实现过去时刻的读取，如 **MarkLogic** [26] 和 **Oracle Total Recall** [30]。Lomet 和 Li [24] 描述此类时序数据库的实现策略。

**Farsite** 推导了相对于可信时钟参考的时钟不确定性边界（比 TrueTime 的宽松得多）[13]：Farsite 中的服务器租约以与 Spanner 维护 Paxos 租约相同的方式维护。松散同步的时钟已在先前工作中用于并发控制目的 [2, 23]。我们展示了 TrueTime 让人们能够推理跨 Paxos 状态机集合的全局时间。

## 7 未来工作

我们过去一年大部分时间与 F1 团队合作，将 Google 的广告后端从 MySQL 迁移到 Spanner。我们正在积极改进其监控和支持工具，以及调整其性能。此外，我们一直在改进备份/恢复系统的功能和性能。我们正在实现 Spanner 模式语言、二级索引的自动维护和基于负载的自动重新分片。长期来看，我们计划研究几个特性。乐观地并行执行读可能是值得追求的策略，但初步实验表明正确的实现并非易事。此外，我们计划最终支持 Paxos 配置的直接变更 [22, 34]。

鉴于我们预期许多应用在相对靠近的数据中心之间复制数据，TrueTime ε 可能显著影响性能。我们认为将 ε 降至 1ms 以下没有不可逾越的障碍。可以减少 time master 查询间隔，更好的时钟晶体相对便宜。可以通过改进的网络技术减少 time master 查询延迟，或可能通过替代时间分发技术避免。

最后，有明显的改进领域。尽管 Spanner 在节点数量上可扩展，节点本地数据结构在复杂 SQL 查询上性能相对较差，因为它们为简单键值访问而设计。来自 DB 文献的算法和数据结构可以大大改善单节点性能。其次，响应客户端负载变化在数据中心之间自动移动数据长期以来是我们的目标，但要使该目标有效，我们还需要能够以自动化、协调的方式在数据中心之间移动客户端应用进程。移动进程引发了在数据中心之间管理资源获取和分配的更困难问题。

## 8 结论

综上所述，Spanner 结合并扩展了两个研究社区的思想：来自数据库社区，熟悉、易用的半关系接口、事务和基于 SQL 的查询语言；来自系统社区，可扩展性、自动分片、容错、一致复制、外部一致性和广域分布。自 Spanner 诞生以来，我们花了超过 5 年时间迭代到当前的设计和实现。这一漫长迭代阶段的部分原因是我们逐渐认识到 Spanner 应该做的不仅仅是解决全球复制命名空间的问题，还应该关注 Bigtable 缺失的数据库特性。

我们设计的一个方面脱颖而出：Spanner 功能集的关键是 TrueTime。我们展示了在时间 API 中将时钟不确定性具体化使得构建具有更强时间语义的分布式系统成为可能。此外，随着底层系统对时钟不确定性施加更紧的边界，更强语义的开销会减少。作为社区，我们在设计分布式算法时不应再依赖松散同步的时钟和弱时间 API。

## 附录 A：Paxos Leader 租约管理

确保 Paxos leader 租约区间不相交的最简单方法是，每当 leader 要延长租约时，发出租约区间的同步 Paxos 写。后续 leader 将读取该区间并等待直到该区间已过。

TrueTime 可用于在没有这些额外日志写的情况下确保不相交。第 i 个潜在 leader 保持来自副本 r 的租约投票开始的下界为 vleader_i,r = TT.now().earliest，在 esend_i,r（定义为 leader 发送租约请求时）之前计算。每个副本 r 在租约 egrant_i,r 授予租约，这发生在 ereceive_i,r（副本收到租约请求时）之后；租约在 tend_i,r = TT.now().latest + 10 结束，在 ereceive_i,r 之后计算。副本 r 遵守**单投票规则**（single-vote rule）：在 TT.after(tend_i,r) 为 true 之前，它不会授予另一个租约投票。为在不同 r 化身间强制此规则，Spanner 在授予租约之前在授予副本记录租约投票；此日志写可以搭载在现有 Paxos 协议日志写上。

当第 i 个 leader 收到法定人数投票（事件 equorum_i）时，它计算其租约区间为 lease_i = [TT.now().latest, min_r(vleader_i,r) + 10]。当 TT.before(min_r(vleader_i,r) + 10) 为 false 时，租约在 leader 处被视为已过期。为证明不相交，我们利用第 i 个和第 (i+1) 个 leader 必须在其法定人数中有一个共同副本。称该副本为 r0。证明：

```text
lease_i.end = min_r(vleader_i,r) + 10                    (定义)
min_r(vleader_i,r) + 10 ≤ vleader_i,r0 + 10               (min)
vleader_i,r0 + 10 ≤ tabs(esend_i,r0) + 10               (定义)
tabs(esend_i,r0) + 10 ≤ tabs(ereceive_i,r0) + 10        (因果性)
tabs(ereceive_i,r0) + 10 ≤ tend_i,r0                     (定义)
tend_i,r0 < tabs(egrant_i+1,r0)                         (单投票)
tabs(egrant_i+1,r0) ≤ tabs(equorum_i+1)                 (因果性)
tabs(equorum_i+1) ≤ lease_i+1.start                     (定义)
```
