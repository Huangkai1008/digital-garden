# ZooKeeper：互联网级系统的无等待协调

**作者**：Patrick Hunt, Mahadev Konar, Flavio P. Junqueira, Benjamin Reed  
**机构**：Yahoo! Grid / Yahoo! Research  
**年份**：2010  
**来源**：USENIX Annual Technical Conference (USENIX ATC)

## 摘要

本文描述 **ZooKeeper**（动物园管理员），一种用于协调分布式应用进程的服务。由于 ZooKeeper 是关键基础设施的一部分，其设计目标是提供一个简单且高性能的**内核**（kernel），供客户端构建更复杂的协调原语。它在一个复制的、集中式的服务中融合了**组消息**（group messaging）、**共享寄存器**（shared registers）和**分布式锁服务**（distributed lock services）的元素。ZooKeeper 暴露的接口具有共享寄存器的**无等待**（wait-free）特性，并辅以类似分布式文件系统**缓存失效**（cache invalidation）的**事件驱动机制**（event-driven mechanism），从而提供简单而强大的协调服务。

ZooKeeper 的接口使得高性能服务实现成为可能。除无等待性质外，ZooKeeper 还提供每个客户端的 **FIFO**（先进先出）请求执行保证，以及对所有改变 ZooKeeper 状态的请求的**线性一致性**（linearizability）。这些设计决策使得实现高性能处理流水线成为可能，读请求由本地服务器满足。我们表明，对于目标工作负载（读与写之比为 2:1 到 100:1），ZooKeeper 可处理每秒数万至数十万笔事务。这一性能使 ZooKeeper 能够被客户端应用广泛使用。

## 1 引言

大规模分布式应用需要不同形式的协调。**配置**（configuration）是最基本的协调形式之一。其最简单形式下，配置只是系统进程的操作参数列表，而更复杂的系统则具有动态配置参数。**组成员资格**（group membership）和**领导者选举**（leader election）在分布式系统中也很常见：进程通常需要知道哪些其他进程存活以及这些进程负责什么。**锁**（locks）是一种强大的协调原语，用于实现对关键资源的互斥访问。

协调的一种做法是为每种不同的协调需求开发专门的服务。例如，Amazon Simple Queue Service [3] 专门针对队列。其他服务则专门为领导者选举 [25] 和配置 [27] 而开发。实现更强大原语的服务可用于实现较弱原语。例如，**Chubby** [6] 是一种具有强同步保证的锁服务，锁进而可用于实现领导者选举、组成员资格等。

在设计我们的协调服务时，我们放弃了在服务端实现特定原语的做法，转而选择暴露一个 **API**，使应用开发者能够实现自己的原语。这一选择促成了一个协调内核的实现，该内核能够在不修改服务核心的情况下支持新原语。这种方法使多种形式的协调能够适应应用需求，而不是将开发者限制在一组固定的原语中。

在设计 ZooKeeper 的 API 时，我们放弃了**阻塞原语**（blocking primitives），例如锁。协调服务的阻塞原语可能导致诸多问题，包括慢速或故障客户端对快速客户端性能的负面影响。若处理请求依赖于其他客户端的响应和故障检测，服务本身的实现会变得更加复杂。因此，我们的系统 ZooKeeper 实现了一个 API，用于操纵按文件系统方式分层组织的简单无等待数据对象。事实上，ZooKeeper 的 API 与任何其他文件系统的 API 相似；仅从 API 签名来看，ZooKeeper 似乎是没有 `open` 和 `close` 锁方法的 Chubby。然而，实现无等待数据对象使 ZooKeeper 与基于锁等阻塞原语的系统显著不同。

尽管无等待性质对性能和容错很重要，但仅此不足以实现协调。我们还必须为操作提供顺序保证。我们发现，同时保证所有操作的 FIFO 客户端顺序和线性可化写，既能实现服务的高效实现，也足以实现我们应用所需的协调原语。事实上，我们可以用我们的 API 为任意数量的进程实现**共识**（consensus）；根据 Herlihy 的层次结构，ZooKeeper 实现了一个**通用对象**（universal object）[14]。

ZooKeeper 服务由一组使用**复制**（replication）实现高可用和高性能的服务器组成。其高性能使包含大量进程的应用能够使用这一协调内核管理协调的各个方面。我们能够使用简单的流水线架构实现 ZooKeeper，该架构允许我们在保持低延迟的同时拥有数百或数千个未完成请求。这种流水线自然使来自单个客户端的操作按 FIFO 顺序执行。保证 FIFO 客户端顺序使客户端能够异步提交操作。有了异步操作，客户端可以同时拥有多个未完成操作。例如，当新客户端成为领导者并需要操纵元数据并相应更新时，这一特性非常有用。若没有多个未完成操作的可能性，初始化时间可能达到秒级而非亚秒级。

为保证更新操作满足线性一致性，我们实现了一种基于领导者的**原子广播**（atomic broadcast）协议 [23]，称为 **Zab** [24]。然而，ZooKeeper 应用的典型工作负载以读操作为主，因此扩展读吞吐量变得重要。在 ZooKeeper 中，服务器在本地处理读操作，我们并不使用 Zab 对它们进行全序。在客户端缓存数据是提高读性能的重要技术。例如，进程缓存当前领导者的标识符，而不是每次需要知道领导者时都探测 ZooKeeper，这是有用的。ZooKeeper 使用 **watch**（监视）机制使客户端能够在无需直接管理客户端缓存的情况下缓存数据。通过该机制，客户端可以监视给定数据对象的更新，并在更新时收到通知。Chubby 直接管理客户端缓存，它会阻塞更新以使所有缓存正在被修改数据的客户端的缓存失效。在这种设计下，若这些客户端中有任何一个是慢速或故障的，更新就会被延迟。Chubby 使用**租约**（leases）防止故障客户端无限期阻塞系统。然而，租约只能限制慢速或故障客户端的影响，而 ZooKeeper 的 watch 则完全避免了该问题。

本文讨论 ZooKeeper 的设计与实现。借助 ZooKeeper，我们能够实现应用所需的所有协调原语，尽管只有写操作是线性可化的。为验证我们的方法，我们展示如何用 ZooKeeper 实现一些协调原语。

综上所述，本文的主要贡献如下：

- **协调内核**：我们提出一种具有宽松一致性保证的无等待协调服务，用于分布式系统。我们描述了协调内核的设计与实现，该内核已在许多关键应用中用于实现各种协调技术。
- **协调配方**：我们展示 ZooKeeper 如何用于构建更高级的协调原语，包括分布式应用中常用的阻塞和强一致原语。
- **协调经验**：我们分享 ZooKeeper 的一些使用方式并评估其性能。

## 2 ZooKeeper 服务

客户端通过 **ZooKeeper 客户端库**（ZooKeeper client library）使用客户端 API 向 ZooKeeper 提交请求。除通过客户端 API 暴露 ZooKeeper 服务接口外，客户端库还管理客户端与 ZooKeeper 服务器之间的网络连接。

本节首先提供 ZooKeeper 服务的高层视图，然后讨论客户端用于与 ZooKeeper 交互的 API。

**术语**。本文中，**client**（客户端）指 ZooKeeper 服务的用户，**server**（服务器）指提供 ZooKeeper 服务的进程，**znode**（ZooKeeper 数据节点）指 ZooKeeper 数据中的内存数据节点，数据按称为**数据树**（data tree）的层次命名空间组织。我们还用 **update**（更新）和 **write**（写）指任何修改数据树状态的操作。客户端在连接 ZooKeeper 时建立 **session**（会话），并通过会话句柄发出请求。

### 2.1 服务概述

ZooKeeper 向客户端提供一组**数据节点**（data nodes，znodes）的抽象，这些节点按层次命名空间组织。该层次结构中的 znodes 是客户端通过 ZooKeeper API 操纵的数据对象。层次命名空间常用于文件系统，是一种组织数据对象的理想方式，因为用户熟悉这种抽象，且能更好地组织应用元数据。为引用给定 znode，我们使用文件系统路径的标准 UNIX 表示法。例如，我们用 `/A/B/C` 表示到 znode C 的路径，其中 C 的父节点为 B，B 的父节点为 A。所有 znode 都存储数据；除**临时 znode**（ephemeral znodes）外，所有 znode 都可以有子节点。

::: tip 图 1：ZooKeeper 层次命名空间示意
:::

```text
                    /
                   / \
              /app1   /app2
             /  |  \
    /app1/p_1  p_2  p_3
```

客户端可创建两类 znode：

- **Regular**（常规）：客户端通过显式创建和删除来操纵常规 znode；
- **Ephemeral**（临时）：客户端创建此类 znode，它们要么被显式删除，要么在创建它们的会话终止（有意或由于故障）时由系统自动移除。

此外，在创建新 znode 时，客户端可以设置 **sequential**（顺序）标志。设置了顺序标志创建的节点会在其名称后追加一个单调递增计数器的值。若 n 是新 znode、p 是父 znode，则 n 的序列值永远不小于在 p 下曾创建的任何其他顺序 znode 名称中的值。

ZooKeeper 实现 **watches**（监视）以允许客户端在无需轮询的情况下及时收到变更通知。当客户端发出设置了 watch 标志的读操作时，操作正常完成，但服务器承诺在返回的信息发生变化时通知客户端。Watch 是与会话关联的一次性触发器；一旦触发或会话关闭，它们就会被注销。Watch 表明发生了变更，但不提供变更内容。例如，若客户端在 "/foo" 被修改两次之前发出 `getData("/foo", true)`，客户端将收到一个 watch 事件，告知 "/foo" 的数据已变更。会话事件（如连接丢失事件）也会发送到 watch 回调，以便客户端知道 watch 事件可能被延迟。

**数据模型**。ZooKeeper 的数据模型本质上是一个具有简化 API、仅支持完整数据读写的文件系统，或一个具有层次键的键/值表。层次命名空间有助于为不同应用的命名空间分配子树，并为这些子树设置访问权限。我们还在客户端利用目录概念构建更高级的原语，见第 2.4 节。

与文件系统中的文件不同，znode 并非为通用数据存储而设计。相反，znode 映射到客户端应用的抽象，通常对应于用于协调目的的元数据。例如，图 1 中有两棵子树，一棵用于应用 1（/app1），另一棵用于应用 2（/app2）。应用 1 的子树实现简单的组成员资格协议：每个客户端进程 p_i 在 /app1 下创建 znode p_i，只要进程运行，该 znode 就持续存在。

尽管 znode 并非为通用数据存储而设计，ZooKeeper 确实允许客户端存储一些可用于分布式计算中元数据或配置的信息。例如，在基于领导者的应用中，刚启动的应用服务器了解当前哪台服务器是领导者是有用的。为实现这一目标，我们可以让当前领导者在 znode 空间的已知位置写入该信息。Znode 还具有带时间戳和版本计数器的关联元数据，允许客户端跟踪 znode 的变更并根据 znode 的版本执行条件更新。

**会话**。客户端连接 ZooKeeper 并启动会话。会话有关联的**超时**（timeout）。若 ZooKeeper 在超时时间内未从会话收到任何内容，则认为客户端故障。当客户端显式关闭会话句柄或 ZooKeeper 检测到客户端故障时，会话结束。在会话内，客户端观察到反映其操作执行的一系列状态变更。会话使客户端能够在 ZooKeeper 集群内透明地从一台服务器迁移到另一台，从而在 ZooKeeper 服务器之间持久存在。

### 2.2 客户端 API

下面给出 ZooKeeper API 的相关子集，并讨论每个请求的语义。

| 方法 | 描述 |
|------|------|
| `create(path, data, flags)` | 创建路径名为 path 的 znode，在其中存储 data[]，并返回新 znode 的名称。flags 使客户端可选择 znode 类型：regular、ephemeral，以及设置 sequential 标志 |
| `delete(path, version)` | 若 znode path 处于预期版本，则删除该 znode |
| `exists(path, watch)` | 若路径名为 path 的 znode 存在则返回 true，否则返回 false。watch 标志使客户端能在该 znode 上设置监视 |
| `getData(path, watch)` | 返回与该 znode 关联的数据和元数据（如版本信息）。watch 标志与 exists() 中的工作方式相同，但若 znode 不存在，ZooKeeper 不会设置 watch |
| `setData(path, data, version)` | 若版本号与该 znode 的当前版本匹配，则将 data[] 写入 znode path |
| `getChildren(path, watch)` | 返回 znode 子节点名称的集合 |
| `sync(path)` | 等待操作开始时所有待处理的更新传播到客户端所连接的服务器。path 当前被忽略 |

所有方法都通过 API 提供同步和异步版本。当应用需要执行单个 ZooKeeper 操作且没有并发任务要执行时，使用同步 API，进行必要的 ZooKeeper 调用并阻塞。然而，异步 API 使应用能够同时拥有多个未完成的 ZooKeeper 操作和其他并行执行的任务。ZooKeeper 客户端保证每个操作的相应回调按顺序调用。

注意 ZooKeeper 不使用句柄访问 znode。每个请求包含被操作 znode 的完整路径。这一选择不仅简化了 API（无需 `open()` 或 `close()` 方法），还消除了服务器需要维护的额外状态。

每个更新方法都接受预期版本号，从而实现**条件更新**（conditional updates）。若 znode 的实际版本号与预期版本号不匹配，更新将失败并返回 unexpected version 错误。若版本号为 -1，则不进行版本检查。

### 2.3 ZooKeeper 保证

ZooKeeper 有两个基本顺序保证：

1. **线性可化写**（Linearizable writes）：所有更新 ZooKeeper 状态的请求可串行化且尊重**前驱关系**（precedence）；
2. **FIFO 客户端顺序**（FIFO client order）：来自给定客户端的请求按客户端发送的顺序执行。

注意我们对线性一致性的定义与 Herlihy 最初提出的 [15] 不同，我们称之为 **A-线性一致性**（A-linearizability，异步线性一致性）。在 Herlihy 的原始线性一致性定义中，客户端一次只能有一个未完成操作（客户端即一个线程）。在我们的定义中，我们允许客户端有多个未完成操作，因此我们可以选择不保证同一客户端未完成操作的特定顺序，或保证 FIFO 顺序。我们为我们的性质选择了后者。重要的是，所有对线性可化对象成立的结果对 A-线性可化对象也成立，因为满足 A-线性一致性的系统也满足线性一致性。由于只有更新请求是 A-线性可化的，ZooKeeper 在每个副本本地处理读请求。这使服务能够随服务器数量线性扩展。

为理解这两个保证如何相互作用，考虑以下场景。一个由若干进程组成的系统选举领导者来指挥工作进程。当新领导者接管系统时，它必须更改大量配置参数并在完成后通知其他进程。我们有两个重要需求：

- 随着新领导者开始进行更改，我们不希望其他进程开始使用正在被更改的配置；
- 若新领导者在配置完全更新之前死亡，我们不希望进程使用这一部分配置。

注意，分布式锁（如 Chubby 提供的锁）有助于满足第一个需求，但对第二个需求不足。使用 ZooKeeper，新领导者可以将某路径指定为 **ready znode**（就绪 znode）；其他进程仅在该 znode 存在时使用配置。新领导者通过删除 ready、更新各配置 znode、再创建 ready 来进行配置更改。所有这些更改可以流水线化并异步发出，以快速更新配置状态。尽管单次更改操作的延迟约为 2 毫秒，但若请求逐个发出，必须更新 5000 个不同 znode 的新领导者将需要 10 秒；通过异步发出请求，请求将在不到一秒内完成。由于顺序保证，若进程看到 ready znode，它必然也看到新领导者所做的所有配置更改。若新领导者在创建 ready znode 之前死亡，其他进程知道配置尚未完成，不会使用它。

上述方案仍有一个问题：若进程在新领导者开始更改之前看到 ready 存在，然后在更改进行时开始读取配置，会发生什么？这一问题由通知的顺序保证解决：若客户端正在监视变更，客户端将在看到更改后的新系统状态之前看到通知事件。因此，若读取 ready znode 的进程请求在该 znode 变更时被通知，它将在能够读取任何新配置之前看到通知。

当客户端除 ZooKeeper 外还有自己的通信通道时，可能出现另一个问题。例如，考虑两个客户端 A 和 B，它们在 ZooKeeper 中有共享配置，并通过共享通信通道通信。若 A 在 ZooKeeper 中更改共享配置并通过共享通道告知 B，B 在重新读取配置后期望看到更改。若 B 的 ZooKeeper 副本略落后于 A 的，它可能看不到新配置。使用上述保证，B 可以通过在重新读取配置之前发出一次写来确保看到最新信息。为更高效地处理该场景，ZooKeeper 提供 **sync** 请求：后跟读操作时构成**慢读**（slow read）。sync 使服务器在处理读之前应用所有待处理的写请求，而无需完整写的开销。该原语在思想上与 ISIS 的 flush 原语 [5] 类似。

ZooKeeper 还有以下两个**活性**（liveness）和**持久性**（durability）保证：若多数 ZooKeeper 服务器处于活跃且相互通信，服务将可用；若 ZooKeeper 服务成功响应变更请求，只要多数服务器最终能够恢复，该变更将在任意数量的故障后持久存在。

### 2.4 原语示例

本节展示如何使用 ZooKeeper API 实现更强大的原语。ZooKeeper 服务对这些更强大的原语一无所知，因为它们完全在客户端使用 ZooKeeper 客户端 API 实现。一些常见原语（如组成员资格和配置管理）也是无等待的。对于其他原语（如 **rendezvous**（会合）），客户端需要等待事件。尽管 ZooKeeper 是无等待的，我们仍能用 ZooKeeper 实现高效的阻塞原语。ZooKeeper 的顺序保证允许高效推理系统状态，watch 允许高效等待。

**配置管理**。ZooKeeper 可用于在分布式应用中实现动态配置。其最简单形式下，配置存储在 znode zc 中。进程以 zc 的完整路径名启动。启动的进程通过将 watch 标志设为 true 读取 zc 来获取配置。若 zc 中的配置被更新，进程会收到通知并读取新配置，再次将 watch 标志设为 true。

注意在此方案中，与大多数使用 watch 的方案一样，watch 用于确保进程拥有最新信息。例如，若监视 zc 的进程收到 zc 变更通知，在它能发出对 zc 的读之前 zc 又发生了三次变更，进程不会收到三次额外的通知事件。这不影响进程行为，因为那三次事件本应只是通知进程它已知的事：它对 zc 的信息已过时。

**会合**。有时在分布式系统中，最终系统配置会是什么样子并非总是事先清楚。例如，客户端可能想启动一个主进程和若干工作进程，但启动由调度器完成，因此客户端事先不知道可以给工作进程的地址和端口等信息以连接到主进程。我们用 ZooKeeper 通过**会合 znode** zr 处理该场景，zr 是客户端创建的节点。客户端将 zr 的完整路径名作为主进程和工作进程的启动参数传递。主进程启动时用其使用的地址和端口信息填充 zr。工作进程启动时，将 watch 设为 true 读取 zr。若 zr 尚未填充，工作进程等待 zr 更新时被通知。若 zr 是临时节点，主进程和工作进程可以监视 zr 被删除，并在客户端结束时自行清理。

**组成员资格**。我们利用临时节点实现组成员资格。具体地，我们利用临时节点使我们能看到创建该节点的会话状态这一事实。我们首先指定 znode zg 表示组。当组成员进程启动时，它在 zg 下创建临时子 znode。若每个进程有唯一名称或标识符，则将该名称用作子 znode 的名称；否则，进程使用 SEQUENTIAL 标志创建 znode 以获得唯一名称分配。进程可以将进程信息（如进程使用的地址和端口）放在子 znode 的数据中。

在 zg 下创建子 znode 后，进程正常启动，无需做其他事。若进程失败或结束，表示它的 zg 下的 znode 会自动移除。

进程可以通过简单列出 zg 的子节点获取组信息。若进程想监视组成员资格变更，可将 watch 标志设为 true，并在收到变更通知时刷新组信息（始终将 watch 标志设为 true）。

**简单锁**。尽管 ZooKeeper 不是锁服务，它可用于实现锁。使用 ZooKeeper 的应用通常使用针对其需求定制的同步原语，如上述所示。这里我们展示如何用 ZooKeeper 实现锁，以表明它能实现多种通用同步原语。

最简单的锁实现使用「锁文件」。锁由 znode 表示。要获取锁，客户端尝试使用 EPHEMERAL 标志创建指定的 znode。若创建成功，客户端持有锁。否则，客户端可以设置 watch 标志读取 znode，以便在当前持有者死亡时被通知。客户端在死亡或显式删除 znode 时释放锁。等待锁的其他客户端在观察到 znode 被删除后再次尝试获取锁。

尽管这一简单锁协议有效，它确实有一些问题。首先，它遭受**羊群效应**（herd effect）。若有大量客户端等待获取锁，它们都会在锁释放时争夺锁，尽管只有一个客户端能获得锁。其次，它只实现**排他锁**（exclusive locking）。下面两个原语展示如何克服这两个问题。

**无羊群效应的简单锁**。我们定义锁 znode l 来实现此类锁。直观上，我们将所有请求锁的客户端排队，每个客户端按请求到达顺序获得锁。因此，希望获取锁的客户端执行以下操作：

```text
加锁 (Lock)
1 n = create(l + "/lock-", EPHEMERAL|SEQUENTIAL)
2 C = getChildren(l, false)
3 if n 是 C 中序号最小的 znode, exit
4 p = C 中紧排在 n 之前的 znode
5 if exists(p, true) wait for watch event
6 goto 2

解锁 (Unlock)
1 delete(n)
```

Lock 第 1 行中 SEQUENTIAL 标志的使用使客户端的加锁尝试相对于所有其他尝试有序。若客户端的 znode 在第 3 行有最小序号，客户端持有锁。否则，客户端等待要么持有锁要么将在此客户端 znode 之前获得锁的 znode 被删除。通过仅监视排在客户端 znode 之前的 znode，我们避免了羊群效应，因为锁释放或锁请求被放弃时只唤醒一个进程。一旦客户端监视的 znode 消失，客户端必须检查它是否现在持有锁。（之前的锁请求可能已被放弃，可能仍有序号更低的 znode 在等待或持有锁。）

释放锁只需删除表示锁请求的 znode n。通过在创建时使用 EPHEMERAL 标志，崩溃的进程将自动清理任何锁请求或释放它们可能持有的任何锁。

总之，该锁方案具有以下优点：

1. znode 的移除只导致一个客户端被唤醒，因为每个 znode 恰好被另一个客户端监视，因此没有羊群效应；
2. 无轮询或超时；
3. 由于我们实现锁的方式，我们可以通过浏览 ZooKeeper 数据看到锁争用程度、打破锁并调试锁问题。

**读写锁**。为实现读写锁，我们略微修改锁过程，并分别有读锁和写锁过程。解锁过程与全局锁情况相同。

```text
写锁 (Write Lock)
1 n = create(l + "/write-", EPHEMERAL|SEQUENTIAL)
2 C = getChildren(l, false)
3 if n 是 C 中序号最小的 znode, exit
4 p = C 中紧排在 n 之前的 znode
5 if exists(p, true) wait for event
6 goto 2

读锁 (Read Lock)
1 n = create(l + "/read-", EPHEMERAL|SEQUENTIAL)
2 C = getChildren(l, false)
3 if C 中无序号低于 n 的写 znode, exit
4 p = C 中紧排在 n 之前的写 znode
5 if exists(p, true) wait for event
6 goto 3
```

该锁过程与之前的锁略有不同。写锁仅在命名上不同。由于读锁可共享，第 3 和第 4 行略有不同，因为只有较早的写锁 znode 阻止客户端获得读锁。当有多个客户端等待读锁且序号较低的 "write-" znode 被删除时被通知，可能看起来有「羊群效应」；事实上，这是期望的行为，所有那些读客户端应被释放，因为它们现在可能持有锁。

**双重屏障**。**双重屏障**（double barrier）使客户端能够同步计算的开始和结束。当足够多的进程（由屏障阈值定义）加入屏障后，进程开始计算，完成后离开屏障。我们用 ZooKeeper 中的 znode b 表示屏障。每个进程 p 在进入时通过创建 b 的子 znode 向 b 注册，在准备离开时注销（移除子节点）。当 b 的子 znode 数量超过屏障阈值时，进程可以进入屏障。当所有进程都移除了它们的子节点时，进程可以离开屏障。我们使用 watch 高效等待进入和退出条件满足。要进入，进程监视 b 的 ready 子节点的存在，该子节点将由使子节点数量超过屏障阈值的进程创建。要离开，进程监视特定子节点消失，仅在该 znode 被移除后检查退出条件。

## 4 ZooKeeper 实现

ZooKeeper 通过在组成服务的每台服务器上复制 ZooKeeper 数据实现高可用。我们假设服务器通过崩溃而故障，此类故障服务器可能稍后恢复。图 4 显示 ZooKeeper 服务的高层组件。收到请求后，服务器为其执行做准备（**请求处理器**，request processor）。若该请求需要服务器间协调（写请求），则它们使用**共识协议**（agreement protocol，原子广播的实现），最后服务器将更改提交到在集群所有服务器上完全复制的 ZooKeeper 数据库。对于读请求，服务器只需读取本地数据库的状态并生成对请求的响应。

::: tip 图 4：ZooKeeper 服务的组件
:::

```text
                     ZooKeeper 服务
    ┌─────────────────────────────────────────────────┐
    │  写请求                                           │
    │    ↓                                             │
    │  请求处理器 → 原子广播 → 复制数据库                │
    │    txn         txn         txn                    │
    │                                           ↓      │
    │  读请求 ──────────────────────────────→ 响应     │
    └─────────────────────────────────────────────────┘
```

复制数据库是包含整个数据树的**内存数据库**（in-memory database）。树中每个 znode 默认最多存储 1MB 数据，但该最大值是配置参数，可在特定情况下更改。为可恢复性，我们高效地将更新记录到磁盘，并在应用到内存数据库之前强制写入磁盘介质。事实上，与 Chubby [8] 一样，我们维护已提交操作的**重放日志**（replay log，在我们情况下是**预写日志**，write-ahead log）并生成内存数据库的定期**快照**（snapshots）。

每台 ZooKeeper 服务器都服务客户端。客户端连接恰好一台服务器以提交请求。如前所述，读请求由每台服务器数据库的本地副本服务。改变服务状态的请求（写请求）由共识协议处理。

作为共识协议的一部分，写请求被转发到称为 **leader**（领导者）的单台服务器。其余 ZooKeeper 服务器称为 **followers**（追随者），接收来自领导者的包含状态变更的消息提案，并就状态变更达成一致。

### 4.1 请求处理器

由于消息层是原子的，我们保证本地副本永不分歧，尽管在任何时刻某些服务器可能比其他服务器应用了更多事务。与客户端发送的请求不同，事务是**幂等**（idempotent）的。当领导者收到写请求时，它计算写应用后系统的状态，并将其转换为捕获这一新状态的事务。必须计算未来状态，因为可能有尚未应用到数据库的未完成事务。例如，若客户端执行条件 setData 且请求中的版本号与正在更新的 znode 的未来版本号匹配，服务生成包含新数据、新版本号和更新时间戳的 setDataTXN。若发生错误（如版本号不匹配或要更新的 znode 不存在），则改为生成 errorTXN。

### 4.2 原子广播

所有更新 ZooKeeper 状态的请求被转发到领导者。领导者执行请求并通过 **Zab** [24]（一种原子广播协议）广播 ZooKeeper 状态的变更。收到客户端请求的服务器在交付相应状态变更时响应客户端。Zab 默认使用简单多数**法定人数**（quorum）决定提案，因此 Zab 进而 ZooKeeper 仅在多数服务器正确时才能工作（即，2f+1 台服务器可容忍 f 个故障）。

为实现高吞吐，ZooKeeper 尝试保持请求处理流水线满载。它可能在处理流水线的不同部分有数千个请求。由于状态变更依赖于先前状态变更的应用，Zab 提供比常规原子广播更强的顺序保证。更具体地，Zab 保证领导者广播的变更按发送顺序交付，且来自先前领导者的所有变更在已确立的领导者广播其自己的变更之前交付给它。

有几个实现细节简化了我们的实现并给我们出色的性能。我们使用 TCP 作为传输，因此消息顺序由网络维护，这使我们能简化实现。我们使用 Zab 选择的领导者作为 ZooKeeper 领导者，因此创建事务的同一进程也提出它们。我们使用日志跟踪提案作为内存数据库的预写日志，因此不必两次将消息写入磁盘。

在正常操作期间，Zab 确实按顺序且恰好一次交付所有消息，但由于 Zab 不持久记录每个已交付消息的 id，Zab 可能在恢复期间重新交付消息。由于我们使用幂等事务，只要按顺序交付，多次交付是可接受的。事实上，ZooKeeper 要求 Zab 至少重新交付上次快照开始后交付的所有消息。

### 4.3 复制数据库

每个副本在内存中有 ZooKeeper 状态的副本。当 ZooKeeper 服务器从崩溃恢复时，它需要恢复这一内部状态。重放所有已交付消息以恢复状态在服务器运行一段时间后会耗时过长，因此 ZooKeeper 使用定期快照，仅要求自快照开始以来的消息重新交付。我们称 ZooKeeper 快照为**模糊快照**（fuzzy snapshots），因为我们不锁定 ZooKeeper 状态来拍摄快照；相反，我们对树进行深度优先扫描，原子读取每个 znode 的数据和元数据并将它们写入磁盘。由于生成的模糊快照可能已应用了快照生成期间交付的状态变更的某个子集，结果可能不对应于任何时刻的 ZooKeeper 状态。然而，由于状态变更是幂等的，只要按顺序应用状态变更，我们可以应用它们两次。

例如，假设 ZooKeeper 数据树中两个节点 /foo 和 /goo 在模糊快照开始时分别有值 f1 和 g1，版本均为 1，以下形式为 ⟨transactionType, path, value, new-version⟩ 的状态变更流到达：

```text
⟨SetDataTXN, /foo, f2, 2⟩
⟨SetDataTXN, /goo, g2, 2⟩
⟨SetDataTXN, /foo, f3, 3⟩
```

处理这些状态变更后，/foo 和 /goo 分别有值 f3 和 g2，版本为 3 和 2。然而，模糊快照可能记录了 /foo 和 /goo 分别有值 f3 和 g1，版本为 3 和 1，这不是 ZooKeeper 数据树的有效状态。若服务器使用该快照崩溃并恢复，且 Zab 重新交付状态变更，结果状态对应于崩溃前的服务状态。

### 4.4 客户端-服务器交互

当服务器处理写请求时，它还会发出并清除与该更新对应的任何 watch 的通知。服务器按顺序处理写，且不并发处理其他写或读。这确保了通知的严格顺序。注意服务器在本地处理通知。只有客户端连接的服务器跟踪并触发该客户端的通知。

读请求在每台服务器本地处理。每个读请求被处理并标记与服务器看到的最后事务对应的 **zxid**。该 zxid 定义了读请求相对于写请求的偏序。通过在本地处理读，我们获得出色的读性能，因为这只是本地服务器上的内存操作，没有磁盘活动或共识协议要运行。这一设计选择是实现我们以读为主工作负载下出色性能目标的关键。

使用快速读的一个缺点是不保证读操作的前驱顺序。即，读操作可能返回过时值，尽管对同一 znode 的更新已被提交。并非所有应用都需要前驱顺序，但对于需要的应用，我们实现了 sync。该原语异步执行，由领导者在其本地副本的所有待处理写之后排序。为保证给定读操作返回最新更新值，客户端在读操作之前调用 sync。客户端操作的 FIFO 顺序保证与 sync 的全局保证一起，使读操作的结果能反映 sync 发出之前发生的任何变更。

在我们的实现中，我们不需要原子广播 sync，因为我们使用基于领导者的算法，我们只需将 sync 操作放在领导者与执行 sync 调用的服务器之间请求队列的末尾。为使这有效，追随者必须确信领导者仍是领导者。若有待处理事务提交，则服务器不怀疑领导者。若待处理队列为空，领导者需要发出空事务以提交，并将 sync 排在该事务之后。这有一个很好的性质：当领导者有负载时，不会产生额外的广播流量。在我们的实现中，超时设置使领导者在追随者放弃它们之前意识到自己不是领导者，因此我们不发出空事务。

ZooKeeper 服务器按 FIFO 顺序处理客户端的请求。响应包含响应所对应的 zxid。即使在无活动间隔期间的心跳消息也包含客户端所连接服务器看到的最后 zxid。若客户端连接到新服务器，该新服务器通过将客户端的最后 zxid 与其 lastzxid 比较，确保其对 ZooKeeper 数据的视图至少与客户端的视图一样新。若客户端的视图比服务器新，服务器在追上之前不会与客户端重新建立会话。客户端保证能找到对系统有较新视图的另一台服务器，因为客户端只看到已复制到多数 ZooKeeper 服务器的变更。该行为对保证持久性很重要。

为检测客户端会话故障，ZooKeeper 使用超时。若在会话超时内没有其他服务器从客户端会话收到任何内容，领导者确定发生了故障。若客户端足够频繁地发送请求，则无需发送任何其他消息。否则，客户端在低活动期间发送心跳消息。若客户端无法与服务器通信以发送请求或心跳，它连接到不同的 ZooKeeper 服务器以重新建立会话。为防止会话超时，ZooKeeper 客户端库在会话空闲 s/3 毫秒后发送心跳，若在 2s/3 毫秒内未收到服务器消息则切换到新服务器，其中 s 是会话超时（毫秒）。

## 5 评估

我们在 50 台服务器的集群上进行所有评估。每台服务器有一个 Xeon 双核 2.1GHz 处理器、4GB RAM、千兆以太网和两个 SATA 硬盘。我们将讨论分为两部分：吞吐量和请求延迟。

### 5.1 吞吐量

为评估系统，我们在系统饱和时对吞吐量进行基准测试，以及注入各种故障时吞吐量的变化。我们改变组成 ZooKeeper 服务的服务器数量，但始终保持客户端数量相同。为模拟大量客户端，我们使用 35 台机器模拟 250 个同时客户端。

我们有 ZooKeeper 服务器的 Java 实现，以及 Java 和 C 客户端。对于这些实验，我们使用配置为在一个专用磁盘上记录日志、在另一个上拍摄快照的 Java 服务器。我们的基准客户端使用异步 Java 客户端 API，每个客户端至少有 100 个未完成请求。每个请求由 1K 数据的读或写组成。我们不展示其他操作的基准，因为所有修改状态的操作的性能大致相同，非状态修改操作（除 sync 外）的性能也大致相同。（sync 的性能近似于轻量级写，因为请求必须到达领导者，但不被广播。）客户端每 300ms 发送已完成操作数，我们每 6 秒采样一次。为防止内存溢出，服务器限制系统中的并发请求数。ZooKeeper 使用请求节流防止服务器过载。对于这些实验，我们配置 ZooKeeper 服务器最多处理 2000 个总请求。

ZooKeeper 能够通过将负载分布到组成服务的服务器上实现如此高的吞吐量。我们能够分布负载是因为我们宽松的一致性保证。Chubby 客户端反而将所有请求定向到领导者。图 6 显示若我们不利用这一宽松性并强制客户端仅连接领导者会发生什么。如预期，以读为主的工作负载吞吐量低得多，但即使以写为主的工作负载吞吐量也更低。服务客户端造成的额外 CPU 和网络负载影响领导者协调提案广播的能力，进而对整体写性能产生不利影响。

原子广播协议做了系统的大部分工作，因此比任何其他组件更限制 ZooKeeper 的性能。图 7 显示原子广播组件的吞吐量。为对其性能进行基准测试，我们通过在领导者处直接生成事务来模拟客户端，因此没有客户端连接或客户端请求和回复。在最大吞吐量下，原子广播组件变得受 CPU 限制。理论上，图 7 的性能应与 100% 写的 ZooKeeper 性能匹配。然而，ZooKeeper 客户端通信、ACL 检查和请求到事务的转换都需要 CPU。CPU 争用将 ZooKeeper 吞吐量降低到远低于单独原子广播组件的水平。由于 ZooKeeper 是关键生产组件，迄今为止我们对 ZooKeeper 的开发重点一直是正确性和健壮性。通过消除诸如额外拷贝、同一对象的多次序列化、更高效的内部数据结构等，有大量机会显著提高性能。

### 5.3 屏障性能

在本实验中，我们顺序执行若干屏障以评估用 ZooKeeper 实现的原语的性能。对于给定数量的屏障 b，每个客户端首先进入所有 b 个屏障，然后依次离开所有 b 个屏障。由于我们使用第 2.4 节的双重屏障算法，客户端首先等待所有其他客户端执行 enter() 过程，然后才进行下一次调用（leave() 类似）。

我们在表 3 中报告实验结果。在本实验中，我们有 50、100 和 200 个客户端依次进入数量为 b 的屏障，b ∈ {200, 400, 800, 1600}。尽管应用可以有数千个 ZooKeeper 客户端，但通常只有更小的子集参与每次协调操作，因为客户端通常根据应用的具体情况分组。

该实验的两个有趣观察是：处理所有屏障的时间大致随屏障数线性增加，表明对数据树同一部分的并发访问未产生任何意外延迟；延迟与客户端数量成比例增加。这是未使 ZooKeeper 服务饱和的结果。事实上，我们观察到即使客户端以锁步进行，屏障操作（进入和离开）的吞吐量在所有情况下都在每秒 1,950 到 3,100 次操作之间。在 ZooKeeper 操作中，这对应于每秒 10,700 到 17,000 次操作的吞吐量值。由于我们实现中读写比为 4:1（80% 读操作），我们基准代码使用的吞吐量远低于 ZooKeeper 能达到的原始吞吐量（根据图 5 超过 40,000）。这是由于客户端等待其他客户端。

## 6 相关工作

ZooKeeper 的目标是提供一种服务，缓解分布式应用中协调进程的问题。为实现这一目标，其设计使用了先前协调服务、容错系统、分布式算法和文件系统的思想。

我们并非第一个提出分布式应用协调系统的。一些早期系统为事务应用 [13] 和计算机集群中的信息共享 [19] 提出了分布式锁服务。最近，Chubby 提出了一种为分布式应用管理咨询锁的系统 [6]。Chubby 与 ZooKeeper 共享多个目标。它也有类似文件系统的接口，并使用共识协议保证副本的一致性。然而，ZooKeeper 不是锁服务。客户端可以用它实现锁，但其 API 中没有锁操作。与 Chubby 不同，ZooKeeper 允许客户端连接到任何 ZooKeeper 服务器，而不仅仅是领导者。ZooKeeper 客户端可以使用其本地副本来服务数据和管理 watch，因为其一致性模型比 Chubby 宽松得多。这使 ZooKeeper 能够提供比 Chubby 更高的性能，允许应用更广泛地使用 ZooKeeper。

文献中提出了旨在缓解构建容错分布式应用问题的容错系统。一个早期系统是 ISIS [5]。ISIS 系统将抽象类型规约转换为容错分布式对象，从而使容错机制对用户透明。Horus [30] 和 Ensemble [31] 是从 ISIS 演进的系统。ZooKeeper 采纳了 ISIS 的**虚拟同步**（virtual synchrony）概念。最后，Totem 在利用局域网硬件广播的架构中保证消息交付的全序 [22]。ZooKeeper 适用于各种网络拓扑，这促使我们依赖服务器进程之间的 TCP 连接，而不假设任何特殊拓扑或硬件特性。我们也不暴露 ZooKeeper 内部使用的任何集群通信。

构建容错服务的一项重要技术是**状态机复制**（state-machine replication）[26]，**Paxos** [20] 是一种使异步系统复制状态机的高效实现成为可能的算法。我们使用一种与 Paxos 共享某些特性的算法，但将共识所需的事务日志与数据树恢复所需的预写日志结合，以实现高效实现。已有拜占庭容错复制状态机实用实现的协议提议 [7, 10, 18, 1, 28]。ZooKeeper 不假设服务器可能是拜占庭的，但我们确实采用校验和和健全性检查等机制来捕获非恶意拜占庭故障。Clement 等人讨论了在不修改当前服务器代码库的情况下使 ZooKeeper 完全拜占庭容错的方法 [9]。迄今为止，我们未在生产中观察到使用完全拜占庭容错协议本可预防的故障 [29]。

Boxwood [21] 是使用分布式锁服务器的系统。Boxwood 向应用提供更高级的抽象，并依赖基于 Paxos 的分布式锁服务。与 Boxwood 一样，ZooKeeper 是用于构建分布式系统的组件。然而，ZooKeeper 有高性能要求，并在客户端应用中更广泛使用。ZooKeeper 暴露应用用于实现更高级原语的低级原语。

ZooKeeper 类似于小型文件系统，但它只提供文件系统操作的一小部分子集，并添加了大多数文件系统中没有的功能，如顺序保证和条件写。然而，ZooKeeper 的 watch 在精神上与 AFS [16] 的缓存回调类似。

Sinfonia [2] 引入了**迷你事务**（mini-transactions），一种构建可扩展分布式系统的新范式。Sinfonia 被设计用于存储应用数据，而 ZooKeeper 存储应用元数据。ZooKeeper 将其状态完全复制并保存在内存中以实现高性能和一致延迟。我们对类似文件系统操作和顺序的使用实现了与迷你事务类似的功能。znode 是一个方便的抽象，我们在其上添加了 watch，这是 Sinfonia 中缺少的功能。Dynamo [11] 允许客户端在分布式键值存储中获取和放置相对较小（小于 1M）的数据量。与 ZooKeeper 不同，Dynamo 中的键空间不是层次的。Dynamo 也不为写提供强持久性和一致性保证，而是在读时解决冲突。

DepSpace [4] 使用**元组空间**（tuple space）提供拜占庭容错服务。与 ZooKeeper 一样，DepSpace 使用简单的服务器接口在客户端实现强同步原语。虽然 DepSpace 的性能远低于 ZooKeeper，但它提供更强的容错和保密性保证。

## 7 结论

ZooKeeper 通过向客户端暴露无等待对象，对分布式系统中协调进程的问题采取无等待方法。我们发现 ZooKeeper 对 Yahoo! 内外的多个应用都有用。ZooKeeper 通过使用由本地副本服务的快速读和 watch，对以读为主的工作负载实现了每秒数十万次操作的吞吐量值。尽管我们对读和 watch 的一致性保证看似较弱，我们通过用例表明，这种组合使我们能够在客户端实现高效而复杂的协调协议，尽管读不是前驱有序的且数据对象的实现是无等待的。无等待性质已被证明对高性能至关重要。

尽管我们只描述了少数应用，还有许多其他应用在使用 ZooKeeper。我们相信这种成功归功于其简单的接口和通过该接口可以实现的强大抽象。此外，由于 ZooKeeper 的高吞吐量，应用可以广泛使用它，而不仅仅是粗粒度锁。
