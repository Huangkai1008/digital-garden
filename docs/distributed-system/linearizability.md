# 线性一致性：并发对象的正确性条件

**作者**：Maurice P. Herlihy, Jeannette M. Wing  
**机构**：卡内基梅隆大学  
**年份**：1990  
**来源**：ACM Transactions on Programming Languages and Systems, Vol. 12, No. 3

---

## 摘要

**并发对象**（concurrent object）是由并发进程共享的数据对象。**线性一致性**（linearizability）是一种利用抽象数据类型语义的并发对象正确性条件。它允许高度的并发性，同时允许程序员使用顺序领域中的已知技术来规约和推理并发对象。线性一致性提供了一种假象：并发进程施加的每个操作在其调用（invocation）与响应（response）之间的某个时刻瞬时生效，这意味着并发对象操作的含义可以用前置条件（pre-condition）和后置条件（post-condition）给出。本文定义线性一致性，将其与其他正确性条件进行比较，提出并演示证明实现正确性的方法，并展示在给定线性一致性的前提下如何推理并发对象。

## 1 引言

### 1.1 概述

非形式化地说，**并发系统**（concurrent system）由一组通过共享类型化对象进行通信的顺序进程组成。该模型既涵盖共享对象为消息队列的**消息传递**（message-passing）架构，也涵盖共享对象为内存中数据结构的**共享内存**（shared-memory）架构。每个对象都有一个**类型**（type），它定义了一组可能的值和一组**原语操作**（primitive operations），这些操作是创建和操纵该对象的唯一手段。在顺序系统中，对象的操作由单个进程一次调用一个，操作的含义可以用前置条件和后置条件给出。然而在并发系统中，对象的操作可由并发进程调用，因此有必要为操作调用的可能交错赋予含义。

若一个**并发计算**（concurrent computation）在某种形式化定义的意义上（见第 2 节）与某个合法的**顺序计算**（sequential computation）"等价"，则该并发计算是**线性可化的**（linearizable）。我们将数据类型的（顺序）公理化规约解释为仅允许线性可化的交错。线性一致性不是将数据留作未解释的，而是利用抽象数据类型的语义；它允许高度的并发性，同时允许程序员使用标准验证技术来规约和推理并发对象。与**顺序一致性**（sequential consistency）[31] 或**可串行化**（serializability）[40] 等替代正确性条件不同，线性一致性是一种**局部性质**（local property）：若每个单独的对象是线性可化的，则整个系统是线性可化的。局部性增强了模块化和并发性，因为对象可以独立实现和验证，运行时调度可以完全去中心化。线性一致性也是一种**非阻塞性质**（nonblocking property）：调用**全定义操作**（totally-defined operation）的进程永远不会被迫等待。非阻塞性增强了并发性，并意味着线性一致性是实时响应至关重要的系统的合适条件。线性一致性是一种简单且直观吸引人的正确性条件，它概括并统一了文献中许多隐式和显式的正确性条件。

使用公理化规约和我们的线性一致性概念，我们可以推理两类问题：

1. **实现正确性**：使用将表示不变量（representation invariant）和抽象函数（abstraction function）[18, 25] 的概念推广到并发领域的新技术，推理线性可化对象实现的正确性。
2. **使用推理**：通过将关于并发计算的断言转化为关于其顺序对应物的更简单断言，推理使用线性可化对象的计算。

第 2 节介绍我们的并发系统模型和线性一致性的形式化定义。第 3 节讨论线性一致性的局部性和非阻塞性质，并将其与其他正确性条件进行比较。第 4 节介绍我们用于推理线性可化对象实现的证明技术，并在高度并发队列的两种新颖实现上演示该技术。第 5 节给出关于并发寄存器和队列的推理示例，假设它们是线性可化的。第 6 节综述相关工作并讨论线性一致性作为正确性条件的意义。

### 1.2 动机

在定义并发对象的正确性条件时，有两个要求似乎具有直观意义：第一，每个操作应看似"瞬时生效"；第二，非并发操作的顺序应被保留。这些要求使我们能够直接用可接受的顺序行为来描述可接受的并发行为，这种方法简化了关于并发程序的形式化和非形式化推理。我们在下一节形式化地捕捉这些概念；这里我们非形式化地回顾一些示例，以说明我们认为直观上可接受和不可接受的并发行为。

我们的示例使用**先进先出队列**（FIFO queue），这是一种提供两种操作的简单数据类型：**Enq**（入队）将一项插入队列，**Deq**（出队）返回并移除队列中最旧的项。图 1 展示了当并发进程操纵 FIFO 队列时，队列可能表现的四种不同方式。这里，时间轴从左向右延伸，每个操作与一个区间相关联。重叠的区间表示并发操作。我们用 `E(x) A`（`D(x) A`）表示进程 A 对项 x 的入队（出队）操作。

::: tip 图 1：FIFO 队列历史
时间轴从左向右，重叠区间表示并发操作。E(x) 表示入队 x，D(x) 表示出队 x。
:::

```text
(a) H1（可接受）——x、y 并发入队，出队顺序与入队顺序一致：
    时间 →
A:  |--E(x)--|       |--D(y)--|--E(z)...
B:    |--E(y)--|     |--D(x)--|

(b) H2（不可接受）——x 先入队，但 y 先出队，违反 FIFO：
A:  |--E(x)--|
B:    |--E(y)--|
     x 先入队，但 y 先出队

(c) H3（可接受）——x 在其入队返回前被出队，操作可视为瞬时生效：
A:  |--E(x)--...
B:    |--D(x)--|

(d) H4（不可接受）——y 被出队两次：
A:  |--E(x)--|
B:    |--E(y)--|
C:          |--D(y)--|
            y 被出队两次
```

图 1(a) 中 H1 所示的行为对应于我们对并发 FIFO 队列应如何表现的直观概念。在此场景中，进程 A 和 B 并发地入队 x 和 y。随后，B 出队 x，然后 A 出队 y 并开始入队 z。由于 x 的出队先于 y 的出队，FIFO 性质意味着它们的入队必须以相同顺序生效。事实上，它们的入队是并发的，因此它们确实可以按该顺序生效。A 对 z 的未完成入队说明我们关心进程持续执行操作（可能永远）的行为。

然而，图 1(b) 中 H2 所示的行为在直观上不可接受。这里，对外部观察者而言，x 显然在 y 之前入队，但 y 在 x 未被出队的情况下被出队。为符合我们的非形式化要求，A 本应出队 x。我们认为图 1(c) 中 H3 所示的行为是可接受的，尽管 x 在其入队操作返回之前被出队。直观上，x 的入队在其完成之前就已生效。最后，H4 显然不可接受，因为 y 被出队了两次。

要判断并发历史是否可接受，必须考虑对象的预期语义。例如，对 FIFO 队列可接受的并发行为对栈、集合、目录等可能不可接受。当限于提供读（Read）和写（Write）操作的**寄存器对象**（register objects）时，我们的可接受性直观概念与 Misra 对并发寄存器的公理化 [35] 中使用的概念完全对应。我们的方法可视为将 Misra 的方法推广到具有更丰富操作集的对象。例如，图 2(a) 中的 H5 是可接受的，但 H6 不是（示例取自 [35]）。这两种行为在一点上不同：在 H5 中，B 读到 0；在 H6 中，B 读到 1。后者在直观上不可接受，因为 A 之前读到了 1，意味着 B 对 1 的写必须在 A 的读之前发生。C 随后对 0 的写虽然与 B 对 1 的写并发，但严格在 A 对 1 的读之后。

::: tip 图 2：寄存器历史
W(u) 表示写值 u，R(u) 表示读得 u。
:::

```text
(a) H5（可接受）——B 读到 0：
    W(0)A  R(1)A  W(0)C
       |--W(1)B--|
       |--R(0)B--|

(b) H6（不可接受）——B 读到 1，与 A 的 R(1) 矛盾：
    W(0)A  R(1)A  W(0)C
       |--W(1)B--|
       |--R(1)B--|  ← B 读到 1 不可接受
```

在下一节中，我们通过定义线性一致性概念来形式化这里提出的直观，以涵盖我们论证为直观可接受的那些历史。

## 2 系统模型与线性一致性的定义

### 2.1 历史

非形式化地说，并发系统由一组称为**进程**（processes）的顺序控制线程组成，它们通过称为**对象**（objects）的共享数据结构进行通信。每个对象有唯一的名称和类型。类型定义了一组可能的值和一组原语操作，这些操作是操纵该对象的唯一手段。进程是顺序的：每个进程对对象施加一系列操作，交替发出**调用**（invocation）然后接收相应的**响应**（response）。（动态进程创建可以通过将每个子进程视为在 fork 之前或 join 之后不执行任何操作的额外进程来简单建模。）

形式化地，并发系统的**执行**（execution）由**历史**（history）建模，历史是操作调用和响应事件的有限序列。历史 H 的**子历史**（subhistory）是 H 事件的子序列。操作调用写作 `(x op(args*) A)`，其中 x 是对象名，op 是操作名，args* 表示参数值序列，A 是进程名。对操作调用的响应写作 `(x term(res*) A)`，其中 term 是**终止条件**（termination condition），res* 是结果序列。我们用 "Ok" 表示正常终止。若对象名和进程名一致，则响应与调用**匹配**（match）。若历史中某调用之后没有匹配的响应，则该调用在历史中是**待定的**（pending）。若 H 是历史，则 `complete(H)` 是 H 的极大子序列，仅由调用和匹配的响应组成。

历史 H 是**顺序的**（sequential），当且仅当：

1. H 的第一个事件是调用。
2. 除可能最后一个外，每个调用紧接匹配的响应；每个响应紧接匹配的调用。

非顺序的历史是**并发的**（concurrent）。

历史 H 的**进程子历史**（process subhistory）`H|P`（H at P）是 H 中所有进程名为 P 的事件的子序列。对象 x 的**对象子历史**（object subhistory）`H|x` 类似定义。若对每个进程 P 有 `H|P = H'|P`，则两个历史 H 和 H' **等价**（equivalent）。若 H 的每个进程子历史 `H|P` 是顺序的，则历史 H 是**良构的**（well-formed）。本文考虑的所有历史均假定为良构的。注意，良构历史的进程子历史必然是顺序的，而对象子历史则未必。

历史中的**操作**（operation）e 是由调用 `inv(e)` 和下一个匹配的响应 `res(e)` 组成的对。我们将操作记为 `[q inv/res A]`，其中 q 是对象，A 是进程。若在 H 中 `inv(e₂)` 先于 `inv(e₁)` 且 `res(e₁)` 先于 `res(e₂)`，则操作 e₂ **位于**（lies within）另一操作 e₁ 之内。在不会造成不必要混淆的情况下，我们省略事件的角度括号和操作的方括号；在上下文清楚时省略对象名和进程名。

例如，图 1 中的 H₁ 是 FIFO 队列 q 的以下良构历史：

```text
q Enq(x) A
q Enq(y) B
q Ok( ) B
q Ok( ) A
q Deq( ) B
q Ok(x) B
q Deq( ) A
q Ok(y) A
q Enq(z) A
```

H₁ 中的第一个事件是进程 A 对 Enq 的调用（参数为 x），第四个事件是匹配的响应（终止条件 Ok，无结果）。操作 `[q Enq(y)/Ok( ) B]` 位于操作 `[q Enq(x)/Ok( ) A]` 之内。子历史 `complete(H₁)` 是去掉最后一个（待定）Enq 调用的 H₁。重排前两个事件可得到与 H₁ 等价的众多历史之一。

若每当 H 在 S 中时，H 的每个前缀也在 S 中，则历史集合 S 是**前缀封闭的**（prefix-closed）。**单对象历史**（single-object history）是所有事件与同一对象相关联的历史。对象的**顺序规约**（sequential specification）是该对象的前缀封闭单对象顺序历史集合。若每个对象子历史 `H|x` 属于 x 的顺序规约，则顺序历史 H 是**合法的**（legal）。定义顺序规约的常规技术有很多。本文使用 Larch [19] 的公理化风格，其中对象的顺序历史由**值**（value）概括，该值（非形式化地说）反映历史结束时对象的状态。这些值用于给出对象操作前置条件和后置条件的公理。例如，FIFO 队列 Enq 和 Deq 操作的公理如图 3 所示。Enq 的后置条件规定：终止时，新队列值是旧队列值插入 e 后的结果。Deq 的规约规定：对该操作应用于非空队列时，从队列中移除第一项。若操作像 Enq 一样对每个对象值都有定义，则它是**全的**（total）；否则像对空队列未定义的 Deq 一样是**部分的**（partial）。

### 2.2 线性一致性的定义

历史 H 在操作上诱导一个**非自反偏序**（irreflexive partial order）$<_H$：

$$e_1 <_H e_2 \quad \text{当且仅当} \quad \text{res}(e_1) \text{ 在 } H \text{ 中先于 } \text{inv}(e_2)$$

（在适当处省略偏序的下标。）非形式化地，$<_H$ 捕捉 H 中操作的"实时"前序关系。不被 $<_H$ 关联的操作称为**并发的**（concurrent）。若 H 是顺序的，则 $<_H$ 是全序。

历史 H 是**线性可化的**（linearizable），当且仅当它可以被**扩展**（通过追加零个或多个响应事件）为某个历史 H'，使得：

- **L1**：`complete(H')` 等价于某个合法的顺序历史 S
- **L2**：$<_H \subseteq <_S$（实时前序在线性化中保持）

非形式化地，将 H 扩展为 H' 捕捉了某些待定调用可能已生效、尽管其响应尚未返回给调用者（如图 1 中历史 H₁ 的待定 Enq）的概念。将注意力限于 `complete(H')` 捕捉了其余待定调用尚未生效的概念。L1 规定进程的行为如同它们在完整操作的粒度上交错。L2 规定这种表面上的顺序交错尊重操作的实时前序关系。

我们称 S 为 H 的**线性化**（linearization）。非确定性内在于线性一致性概念中：(1) 对每个 H，可能有多个满足 L1 和 L2 的扩展 H'；(2) 对每个扩展 H'，可能有多个线性化 S。**线性可化对象**（linearizable object）是其并发历史关于某个顺序规约是线性可化的对象。

### 2.3 队列示例回顾

设 "·" 表示事件连接。图 1 中的历史 H₁ 是线性可化的，因为 `H₁ · (q Ok( ) A)` 等价于以下顺序历史：

```text
q Enq(x) A
q Ok( ) A
q Enq(y) B
q Ok( ) B
q Deq( ) B
q Ok(x) B
q Deq( ) A
q Ok(y) A
q Enq(z) A
q Ok( ) A
```

H₂ 不是线性可化的：x 的完整 Enq 操作先于 y 的 Enq，但 y 在 x 之前被出队。

线性一致性不排除 H₃ 这样的历史，其中操作在其返回事件发生之前"生效"：

```text
q Enq(x) A
q Deq( ) B
q Ok(x) B
```

H₃ 可扩展为 H'₃ = H₃ · (q Ok( ) A)，它等价于入队操作在出队之前发生的顺序历史。

最后，H₄ 不是线性可化的，因为 y 入队一次但出队两次，故 H₄ 不等价于任何顺序 FIFO 队列历史。

## 3 线性一致性的性质

本节证明线性一致性是局部和非阻塞性质，并讨论其与其他正确性条件的区别。

### 3.1 局部性

若每当每个单独的对象满足性质 P 时，整个系统就满足性质 P，则称并发系统的性质 P 是**局部的**（local）。线性一致性是局部性质：

::: info 定理 1（局部性）
H 是线性可化的，当且仅当对每个对象 x，H|x 是线性可化的。
:::

**证明**："仅当"部分显然。

对每个 x，选取 H|x 的一个线性化。设 $R_x$ 为构造该线性化时追加到 H|x 的响应集合，$<_x$ 为对应的线性化序。设 H' 为通过将 $R_x$ 中的每个响应追加到 H 而构造的历史。我们将构造 `complete(H')` 操作上的偏序，使得：(1) 对每个 x，$<_x \subseteq <$；(2) $<_H \subseteq <$。设 S 为通过以任意扩展该偏序的全序排列 `complete(H')` 的操作而构造的顺序历史。条件 (1) 蕴含 S 是合法的，故 L1 满足；条件 (2) 蕴含 L2 满足。

设该偏序为所有 $<_x$ 与 $<_H$ 的并的**传递闭包**（transitive closure）。由构造立即可知它满足条件 (1) 和 (2)，但还需证明它是偏序。用反证法。若否，则存在操作集合 $e_1, \ldots, e_n$，使得 $e_1 < e_2 < \cdots < e_n$，$e_n < e_1$，且每对由某个 $<_x$ 或 $<_H$ 直接关联。选取长度最小的环。

假设所有操作与同一对象 x 相关联。由于 $<_x$ 是全序，必存在两个操作 $e_{i-1}$ 和 $e_i$，使得 $e_{i-1} <_H e_i$ 且 $e_i <_x e_{i-1}$，与 x 的线性可化性矛盾。

因此环必须包含至少两个对象的操作。必要时重标，设 $e_1$ 和 $e_2$ 为不同对象的操作。设 z 为与 $e_1$ 关联的对象。我们断言 $e_2, \ldots, e_n$ 中没有一个可以是 x 的操作。由构造，对 $e_2$ 断言成立。设 $e_i$ 为 $e_3, \ldots, e_n$ 中与 x 关联的第一个操作。由于 $e_{i-1}$ 和 $e_i$ 不被 $<_x$ 关联，它们必被 $<_H$ 关联；故 $e_{i-1}$ 的响应先于 $e_i$ 的调用。$e_2$ 的调用先于 $e_{i-1}$ 的响应，否则 $e_{i-1} <_H e_2$，得到更短的环 $e_2, \ldots, e_{i-1}$。最后，$e_1$ 的响应先于 $e_2$ 的调用，因为由构造 $e_1 <_H e_2$。故 $e_1$ 的响应先于 $e_i$ 的调用，即 $e_1 <_H e_i$，得到更短的环 $e_1, e_i, \ldots, e_n$。

由于 $e_n$ 不是 x 的操作，但 $e_n < e_1$，故 $e_n <_H e_1$。但由构造 $e_1 <_H e_2$，且因 $<_H$ 传递，$e_n <_H e_2$，得到更短的环 $e_2, \ldots, e_n$，最终矛盾。∎

因此，我们只需考虑单对象历史。

::: tip 局部性的意义
局部性之所以重要，是因为它允许以模块化方式设计和构建并发系统；线性可化对象可以独立实现、验证和执行。基于非局部正确性性质的并发系统必须要么依赖所有对象的集中式调度器，要么满足施加于对象上的额外约束，以确保它们遵循兼容的调度协议。局部性不应被视为理所当然；如下所述，文献中有许多替代正确性性质的提议并非局部的。
:::
