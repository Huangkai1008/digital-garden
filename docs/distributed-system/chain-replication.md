# 链式复制：支持高吞吐量与高可用性

**作者**：Robbert van Renesse, Fred B. Schneider  
**机构**：FAST Search & Transfer ASA（挪威特罗姆瑟）、康奈尔大学计算机科学系（美国纽约州伊萨卡）  
**年份**：2004  
**来源**：OSDI (Operating Systems Design and Implementation)

---

## 摘要

**链式复制**（Chain Replication）是一种协调**故障停止**（fail-stop）存储服务器集群的新方法。该方法旨在支持大规模存储服务，在保持**强一致性**（strong consistency）保证的同时实现高吞吐量与高可用性。除概述链式复制协议本身外，本文还通过仿真实验探索了原型实现的性能特征，讨论了吞吐量、可用性以及多种对象放置策略（包括基于**分布式哈希表**（Distributed Hash Table, DHT）路由的方案）。

---

## 1 引言

存储系统通常实现若干操作，使客户端能够存储、检索和/或修改数据。文件系统和数据库系统或许是最为人熟知的例子。在文件系统中，操作（读和写）访问单个文件且是**幂等**（idempotent）的；在数据库系统中，操作（事务）可能各自访问多个对象且是**可串行化**（serializable）的。

本文关注介于文件系统与数据库系统之间的存储系统。具体而言，我们关注以下存储系统（下文称为**存储服务**（storage service））：

- 存储对象（性质未指定），
- 支持**查询操作**（query operation）以返回从单个对象派生的值，
- 支持**更新操作**（update operation）以根据某种预编程的、可能非确定性的、涉及该对象先前状态的计算，原子地改变单个对象的状态。

因此，文件系统写是我们存储服务更新的特例，而后者又是数据库事务的特例。

我们越来越多地看到在线供应商（如 Amazon.com）、搜索引擎（如 Google 和 FAST）以及大量其他信息密集型服务，通过将大规模存储系统连接到网络来创造价值。对于此类应用，存储服务是合适的折中方案——数据库系统过于昂贵，而文件系统缺乏足够丰富的语义。

构建大规模存储服务时的一个挑战是：在故障发生以及随之而来的配置变更（故障组件被检测并替换）中，保持高可用性和高吞吐量。一致性保证也可能至关重要。但即使不是，若存储服务能提供强一致性保证，构建面向该存储服务的应用往往也会简化。强一致性保证断言：（i）对单个对象的查询与更新操作以某种顺序执行；（ii）更新操作的效果必然反映在后续查询操作返回的结果中。

强一致性保证常被认为与实现高吞吐量、高可用性相冲突。因此，不愿牺牲系统吞吐量或可用性的系统设计者往往选择不支持强一致性保证。**Google 文件系统**（Google File System, GFS）体现了这种思路 [11]。事实上，大规模存储服务中的强一致性保证与高吞吐量和可用性并非不兼容。本文所讨论的链式复制这一协调故障停止服务器的新方法，同时支持高吞吐量、高可用性和强一致性。

下文安排如下。第 2 节规定通用存储服务的接口。第 3 节解释如何使用链式复制实现查询与更新操作。链式复制可视为**主/备**（primary/backup）方法的一个实例，故第 4 节对二者进行比较。第 5 节总结使用链式复制原型实现和仿真网络分析吞吐量与可用性的实验。部分仿真将链式复制与基于分布式哈希表（DHT）路由的存储系统（如 CFS [7] 和 PAST [19]）进行比较；其他仿真揭示了采用链式复制的系统从服务器故障中恢复时的意外行为。第 6 节将链式复制与可扩展存储系统、以可用性换取一致性、副本放置等相关工作进行比较。第 7 节为结论，随后是尾注。

## 2 存储服务接口

存储服务的客户端发出查询与更新操作的请求。虽然可以保证到达存储服务的每个请求必定被执行，但**端到端论证**（end-to-end argument）[20] 表明这样做意义不大。若存储服务仅对收到的每个请求生成并完成回复，客户端反而更有利，因为这样也能处理丢失的请求和丢失的回复：若在合理时间内未收到回复，客户端可重新发出请求。

- **query(objId, opts)** 的回复由对象 objId 的值派生；选项 opts 表征返回 objId 的哪些部分。objId 的值保持不变。
- **update(objId, newVal, opts)** 的回复取决于选项 opts，在一般情况下可以是值 V，由涉及 objId 当前值和/或 newVal 的某种非确定性预编程方式产生；V 随后成为 objId 的新值。

查询操作是幂等的，但更新操作不必是。因此，重新发出非幂等更新请求的客户端必须采取预防措施，确保更新尚未被执行。例如，客户端可先发出查询以确定对象的当前值是否已反映该更新。

在到达存储服务之前丢失的客户端请求，对客户端而言与被存储服务忽略的请求无法区分。这意味着，当存储服务器出现短暂中断（在此期间客户端请求被忽略）时，客户端不会暴露于新的故障模式。当然，可接受的客户端性能可能取决于限制短暂中断的频率和持续时间。

在链式复制中，每次短暂中断的持续时间远短于移除故障主机或添加新主机所需的时间。因此，在故障、恢复及其他重配置面前，客户端请求处理以最小中断进行。大多数其他副本管理协议要么阻塞部分操作，要么在故障和重配置期间牺牲一致性保证。

我们通过给出客户端对对象状态的视图以及该对象在响应查询与更新请求时的状态转换，来规定存储服务的功能。图 1 用伪代码给出对象 objID 的此类规约。

::: tip 图 1：客户端视角下的对象
:::

```text
状态为：
  HistobjID : 更新请求序列
  PendingobjID : 请求集合

转换规则：
  T1: 客户端请求 r 到达：
      PendingobjID := PendingobjID ∪ {r}

  T2: 客户端请求 r ∈ PendingobjID 被忽略：
      PendingobjID := PendingobjID − {r}

  T3: 客户端请求 r ∈ PendingobjID 被处理：
      PendingobjID := PendingobjID − {r}
      if r = query(objId, opts) then
          根据 opts 基于 HistobjID 生成回复

```

该图用两个变量定义 objID 的状态：已对 objID 执行的更新序列 HistobjID，以及未处理请求集合 PendingobjID。随后列出可能的状态转换。转换 T1 断言到达的客户端请求被加入 PendingobjID。转换 T2 规定部分待处理请求被忽略——该转换推测不会过于频繁发生。转换 T3 给出请求处理的高层视图：请求 r 先从 PendingobjID 中移除；query 导致产生合适的回复，而 update 还将 r（用 · 表示）追加到 HistobjID。

## 3 链式复制协议

假定服务器为故障停止（fail-stop）[21]：

- 每个服务器在故障时停止而非产生错误的状态转换，
- 服务器的停止状态可被环境检测到。

当对象在 t 台服务器上复制时，最多可有 t−1 台服务器故障而不影响对象的可用性。因此，对象的可用性提高到所有托管该对象的服务器均故障的概率；第 5.4 节的仿真探索了典型存储系统中该概率。下文假定最多有 t−1 台复制某对象的服务器同时故障。

在链式复制中，复制给定对象 objID 的服务器按线性顺序排列形成一条**链**（chain）。（见图 2。）链中的第一台服务器称为**头**（head），最后一台称为**尾**（tail），请求处理由服务器大致如下实现：

**回复生成**：每个请求的回复由尾生成并发送。

**查询处理**：每个查询请求被定向到链的尾，并在该处使用尾存储的 objID 副本原子地处理。

**更新处理**：每个更新请求被定向到链的头。请求在该处使用头部的 objID 副本原子地处理，随后状态变更沿可靠 **FIFO** 链路转发到链的下一个元素（在该处被处理并转发），如此直至请求被尾处理。

因此，强一致性成立，因为查询请求与更新请求均在单台服务器（尾）上串行处理。

处理查询请求仅涉及单台服务器，这意味着 query 是相对廉价的操作。但处理更新请求时，t 台服务器中 t−1 台的计算不参与产生回复，可以说存在冗余。不过，这些冗余服务器确实提高了容错能力。

注意，链式复制避免了与 t−1 台服务器相关的部分冗余计算，因为新值由头计算一次后沿链转发，因此每个副本只需执行一次写。这种状态变更的转发也意味着 update 可以是非确定性操作——非确定性选择由头做一次。

### 3.1 协议细节

客户端不直接读写图 1 中的变量 HistobjID 和 PendingobjID，因此我们可以任意方便的方式实现它们。当使用链式复制实现图 1 的规约时：

- HistobjID 定义为 Hist^T_objID，即链尾 T 存储的 HistobjID 值，
- PendingobjID 定义为链中任意服务器收到且尚未被尾处理的客户端请求集合。

链式复制的查询处理与更新处理协议通过证明链中任意服务器所做的每个状态转换等价于空操作或允许的转换 T1、T2 或 T3，来满足图 1 的规约。

根据上述 HistobjID 和 PendingobjID 由链实现的方式（并暂时假定不发生故障），我们观察到影响 HistobjID 和 PendingobjID 的服务器转换仅有：（i）链中服务器从客户端收到请求（影响 PendingobjID），以及（ii）尾处理客户端请求（影响 HistobjID）。由于其他服务器转换等价于空操作，只需证明转换（i）和（ii）与 T1 至 T3 一致即可。

**客户端请求到达链**：客户端将请求发送到头（更新）或尾（查询）。任一收到请求 r 都会将 r 加入某服务器收到但尚未被尾处理的请求集合。因此，任一收到 r 都会将 r 加入 PendingobjID（如上对链的定义），这与 T1 一致。

**尾处理请求**：执行导致请求从任意副本收到且尚未被尾处理的请求集合中移除，因此从 PendingobjID（如上对链的定义）中删除该请求——即 T3 的第一步。此外，尾 T 对该请求的处理使用副本 Hist^T_objID，如上定义，它实现 HistobjID——这正是 T3 其余步骤所规定的。

### 3.2 应对服务器故障

检测到链中某台服务器故障后（根据故障停止假定，所有此类故障均可被检测到），链被重配置以移除故障服务器。为此，我们使用称为**主控**（master）的服务，其职责包括：

- 检测服务器故障，
- 通知链中每台服务器其在删除故障服务器后得到的新链中的新前驱或新后继，
- 通知客户端哪台服务器是链的头、哪台是尾。

下文假定主控是永不故障的单一进程。这简化了表述，但并非现实假设；我们的链式复制原型实现实际上在多个主机上复制主控进程，使用 **Paxos** [16] 协调这些副本，使它们整体表现为一个不故障的单一进程。

主控区分三种情况：（i）头故障，（ii）尾故障，（iii）链中其他服务器故障。但每种情况的处理都依赖于以下关于更新如何在链中传播的洞察。

将链头的服务器标记为 H，下一台为 H+1，依此类推直至尾 T。定义

```text
Hist^i_objID ⊑ Hist^j_objID
```

当且仅当标签为 i 的服务器上的请求序列 Hist^i_objID 是标签为 j 的服务器上序列 Hist^j_objID 的前缀时成立。由于更新在链元素之间通过可靠 FIFO 链路发送，每台服务器收到的更新序列是其后继收到序列的前缀。因此有：

**更新传播不变量**：对于满足 i ≤ j 的服务器 i 和 j（即 i 是链中 j 的前驱），则 Hist^j_objID ⊑ Hist^i_objID。

**头故障**：主控通过从链中移除 H，并将 H 的后继设为新头来处理。若我们假定最多 t−1 台服务器故障，则此类后继必然存在。

通过删除 H 改变链是一个转换，因此必须证明它要么是空操作，要么与图 1 的 T1、T2 和/或 T3 一致。这很容易：改变链中服务器集合可能改变 PendingobjID 的内容——回想，PendingobjID 定义为链中任意服务器收到且尚未被尾处理的请求集合，因此从链中删除服务器 H 的效果是从 PendingobjID 中移除 H 收到但尚未转发给后继的请求。从 PendingobjID 中移除请求与转换 T2 一致，因此从链中删除 H 与图 1 的规约一致。

**尾故障**：主控通过从链中移除尾 T，并将 T 的前驱 T− 设为新尾来处理。如前，若我们假定最多 t−1 台服务器副本故障，则此类前驱必然存在。

对链的这一改变同时改变了 PendingobjID 和 HistobjID 的值，但以与重复 T3 转换一致的方式进行：PendingobjID 的大小减小，因为 Hist^T_objID ⊑ Hist^T−_objID（由更新传播不变量，因 T− < T 成立），因此将尾从 T 改为 T− 可能增加尾完成的请求集合，按定义即减小 PendingobjID 中的请求集合。此外，按 T3 要求，被 T− 完成但未被 T 完成的更新请求现在出现在 HistobjID 中，因为 T− 现在是尾，HistobjID 定义为 Hist^T−_objID。

**其他服务器故障**：链内部服务器 S 的故障通过从链中删除 S 处理。主控先通知 S 的后继 S+ 新链配置，再通知 S 的前驱 S−。然而，除非采取某种手段确保 S 在故障前收到的更新请求仍能沿链转发（因为这些更新请求已出现在 S 的任意前驱 i 的 Hist^i_objID 中），否则这可能使更新传播不变量失效。

执行该转发的明显候选是 S−，但此时需要一些簿记和协调。

设 U 为请求集合，<U 为该集合上请求的全序。定义请求序列 r 与 (U, <U) 一致，若（i）r 中所有请求均出现在 U 中，且（ii）r 中请求按 <U 升序排列。最后，对于与 (U, <U) 一致的请求序列 r 和 r′，定义 r ⊕ r′ 为 r 或 r′ 中出现的所有请求的序列，使得 r ⊕ r′ 与 (U, <U) 一致（因此 r ⊕ r′ 中请求按 <U 排序）。

更新传播不变量通过要求：前驱 S− 连接到新后继 S+ 时所做的第一件事是，使用连接它们的 FIFO 链路将 Hist^S−_objID 中可能尚未到达 S+ 的请求发送给 S+；只有在此之后，S− 才能处理并转发其在承担新链位置后收到的请求。

为此，每台服务器 i 维护一个列表 Sent^i，记录 i 已转发给某后继但可能尚未被尾处理的更新请求。该列表的增删规则很简单：每当服务器 i 将更新请求 r 转发给其后继时，i 也将 r 追加到 Sent^i。尾在完成更新请求 r 的处理时向其前驱发送确认 ack(r)。服务器 i 收到 ack(r) 后，从 Sent^i 中删除 r，并将 ack(r) 转发给其前驱。

尾收到的请求必然已被链中所有前驱收到，因此可得：

**处理中请求不变量**：若 i ≤ j，则 Hist^i_objID = Hist^j_objID ⊕ Sent^i。

::: tip 图 2：一条链
:::

```text
    客户端 ──更新──► HEAD ──────► ... ──────► TAIL ◄──查询── 客户端
                         │                        │
                         └────── 回复 ─────────────┘
```

::: tip 图 3：删除内部副本的时空图
:::

```text
    master          S+          S−
       │             │            │
       │  消息 1      │            │
       │────────────►│            │
       │             │  消息 2    │
       │◄────────────│───────────►│
       │             │            │
       │  消息 3      │            │
       │─────────────────────────►│
       │             │  消息 4     │
       │             │◄────────────│
       │             │            │
```

因此，若 S− 在收到主控关于 S+ 为其新后继的通知后，首先将 Sent^S− 中的请求序列转发给 S+，则更新传播不变量得以保持。此外，S− 无需转发 Sent^S− 中已出现在 Hist^S+_objID 中的前缀。

图 3 描绘执行的协议体现了该方法（包括不发送超过必要前缀的优化）。消息 1 通知 S+ 其新角色；消息 2 确认并告知主控 S+ 已收到的最后一条更新请求的序列号 sn；消息 3 通知 S− 其新角色及 sn，以便 S− 计算要发送给 S+ 的 Sent^S− 后缀；消息 4 携带该后缀。

### 3.3 扩展链

故障服务器从链中移除。但较短的链容忍的故障更少，若服务器故障过多，对象的可用性最终可能受损。解决方案是在链变短时添加新服务器。只要服务器故障率不太高且添加新服务器耗时不太长，链长可保持在期望的 t 台服务器附近（因此需要再发生 t−1 次故障才会影响对象可用性）。

理论上，新服务器可添加到链的任意位置。实践中，将服务器 T+ 添加到链的末端似乎最简单。对于尾 T+，Sent^T+ 的值始终为空列表，因此初始化 Sent^T+ 是平凡的。剩下的只是以满足更新传播不变量的方式初始化本地对象副本 Hist^T+_objID。

Hist^T+_objID 的初始化可通过让链的当前尾 T 将其存储的对象副本 Hist^T_objID 转发给 T+ 完成。该转发（若对象较大可能耗时）可与 T 处理客户端查询请求和处理来自其前驱的更新并发进行，只要每个更新也追加到 Sent^T。由于 Hist^T+_objID ⊑ Hist^T_objID 在整个转发过程中成立，更新传播不变量成立。因此，一旦 Hist^T_objID = Hist^T+_objID ⊕ Sent^T 成立，处理中请求不变量即建立，T+ 可开始作为链的尾服务：

- T 被通知其不再是尾。T 此后可丢弃从客户端收到的查询请求，但更合理的策略是 T 将此类请求转发给新尾 T+。
- Sent^T 中的请求按序发送给 T+。
- 主控被通知 T+ 为新尾。
- 客户端被通知查询请求应定向到 T+。

## 4 主/备协议

链式复制是**主/备**（primary/backup）方法 [3] 的一种形式，后者本身是**状态机**（state machine）方法 [22] 在副本管理中的实例。在主/备方法中，指定的一台服务器（主节点）负责：

- 对客户端请求施加顺序（从而确保强一致性成立），
- 将客户端请求或由此产生的更新按序分发给其他服务器（称为备节点），
- 等待所有非故障备节点的确认，
- 收到这些确认后再向客户端发送回复。

若主节点故障，则某台备节点被提升为主节点。

在链式复制中，主节点对请求排序的角色由两个副本分担。头对更新请求排序；尾通过交织查询请求扩展该序列。这种职责分担不仅划分了排序任务，还使查询请求能够以更低延迟和更低开销处理，因为仅单台服务器（尾）参与处理查询，且该处理从不被链中其他活动延迟。相比之下，在主/备方法中，主节点在响应查询前必须等待备节点对先前更新的确认。

在链式复制和主/备方法中，更新请求都必须传播到复制某对象的所有服务器，否则副本将分歧。链式复制以串行方式完成该传播，导致比主/备方法更高的延迟——后者将请求并行分发给备节点。采用并行传播时，生成回复所需时间与任意非故障备节点延迟的最大值成正比；采用串行传播时，与这些延迟之和成正比。

第 5 节的仿真量化了所有这些性能差异，包括链式复制和主/备方法的变体——其中查询请求可发送到任意服务器（期望以强一致性保证换取更高性能）。

但理解两种方法处理服务器故障方式的差异无需仿真。此处的核心关注是：服务在响应服务器故障而重配置时，客户端经历的短暂中断的持续时间；其次是服务器故障引入的额外延迟。

检测服务器故障的延迟是迄今为止的主导成本，该成本对链式复制和主/备方法相同。因此，下文分析每种方法在假定已检测到服务器故障后的恢复成本；假定消息延迟是协议延迟的主导来源。

对于链式复制，需考虑三种情况：头故障、中间服务器故障、尾故障。

| 故障类型 | 查询处理 | 更新处理 |
|---------|---------|---------|
| **头故障** | 持续不间断 | 不可用约 2 个消息传递延迟（主控向新头及其后继广播消息，再通过广播通知所有客户端新头） |
| **中间服务器故障** | 持续不间断 | 可能延迟但请求不丢失，无短暂中断（只要链的某前缀中已收到请求的某台服务器仍在运行）；可能延迟约 4 个消息传递延迟 |
| **尾故障** | 均不可用约 2 个消息传递延迟 | 均不可用约 2 个消息传递延迟（主控向新尾发送消息，再通过广播通知所有客户端新尾） |

对于主/备方法，需考虑两种情况：主节点故障和备节点故障。查询与更新请求在每种情况下受影响相同。

| 故障类型 | 影响 |
|---------|------|
| **主节点故障** | 经历 5 个消息延迟的短暂中断：主控检测故障并向所有备节点广播，请求各自已处理的更新数并告知暂停处理请求；各备节点回复主控；主控再广播新主节点身份；新主节点为已处理更新数最多的那个，须将备节点缺失的更新转发给它们；最后主控广播通知所有客户端新主节点 |
| **备节点故障** | 若无进行中的更新请求，查询处理持续不间断；若有更新请求进行中，则经历最多 1 个消息延迟的短暂中断（主控向主节点发送消息，表明故障备节点不会发来确认，不应再向该处发送请求） |

因此，链式复制的最坏情况中断（尾故障）从不长于主/备的最坏情况中断（主节点故障）；链式复制的最佳情况（中间服务器故障）短于主/备的最佳情况中断（备节点故障）。尽管如此，若短暂中断持续时间是设计存储服务时的主导考虑，则在链式复制与主/备方法之间选择需要关于请求类型混合以及各类服务器故障概率的信息。

## 5 仿真实验

为更好理解链式复制的吞吐量与可用性，我们在仿真网络中进行了系列实验。这些实验涉及链式复制的原型实现以及部分替代方案。由于我们主要关注链式复制所固有的处理与通信延迟，我们仿真了带宽无限但每条消息延迟为 1 ms 的网络。

### 5.1 单链、无故障

首先考虑仅有一条链、无故障、复制因子 t 为 2、3 和 10 的简单情况。我们比较四种不同副本管理方案的吞吐量：

- **chain**：链式复制。
- **p/b**：主/备。
- **weak-chain**：修改后的链式复制，查询请求可发往任意随机服务器。
- **weak-p/b**：修改后的主/备，查询请求可发往任意随机服务器。

注意，weak-chain 和 weak-p/b 不实现 chain 和 p/b 的强一致性保证。

我们将服务器上的查询延迟固定为 5 ms，更新延迟固定为 50 ms。（这些数字基于查询或更新 Web 搜索索引的实际值。）我们假定每次更新涉及一些涉及磁盘读的初始处理，且转发对象差异进行存储比在每个副本上重复更新处理更廉价；我们预期副本处理对象差异消息的延迟为 20 ms（对应几次磁盘访问和适度计算）。

例如，若链由三台服务器组成，执行一次更新的总延迟为 94 ms：客户端到头 1 ms、头处更新延迟 50 ms、另外两台服务器各 20 ms 处理对象差异消息、外加三次 1 ms 转发延迟。但查询延迟仅为 7 ms。

:::

```text

吞吐量                    吞吐量                    吞吐量
  ^                         ^                         ^
  |    weak                  |    weak                  |    weak
```

图 4 绘制了 t=2、t=3 和 t=10 时，总吞吐量随更新请求百分比变化的曲线。有 25 个客户端，每个执行与给定百分比一致的查询与更新混合请求。每个客户端一次提交一个请求，仅在收到前一请求的响应后延迟足够长时间再发下一个。因此客户端合计最多可有 25 个并发请求未完成。weak-chain 和 weak-p/b 的吞吐量几乎相同，故图 4 仅有一条曲线——标为 weak——而非 weak-chain 和 weak-p/b 的分别曲线。

观察到链式复制（chain）在所有更新百分比和每种研究的复制因子下，性能等于或优于主/备（p/b）。这与我们的预期一致，因为链式复制中头与尾分担了在主/备方法中由主节点单独承担的负载。

链式复制弱变体的曲线或许令人惊讶，因为当更新请求超过 15% 时，这些弱变体表现不如链式复制（及其强一致性）。涉及两个因素：

- 链式复制和主/备的弱变体在查询密集型负载下通过将查询负载分布到所有服务器而优于纯链式复制，该优势随复制因子增加。
- 一旦更新请求百分比增加，普通链式复制优于其弱变体——因为所有更新在头完成。具体而言，在纯链式复制下（i）查询不会在头等待相对耗时的更新请求完成而被延迟，（ii）若查询请求不在头处理，头有更多容量可用于更新请求处理。

由于 weak-chain 和 weak-p/b 不实现强一致性保证，似乎很少有场景会优先选择这些副本管理方案。

最后注意，只要有足够的并发请求使多个请求可流水线化，链式复制和主/备的吞吐量均不受复制因子影响。

### 5.2 多链、无故障

若每个对象由独立链管理且对象较大，则添加新副本可能因将对象状态传输到该新副本所需时间而产生可观延迟。反之，若对象较小，则大规模存储服务将涉及大量对象。系统中每台处理器现在可能托管来自多条链的服务器——复用处理器和通信信道的成本可能变得过高。此外，单台处理器故障现在影响多条链。

对象集合总可分组为单个**卷**（volume），卷本身可被视为链式复制意义上的对象，因此设计者在决定对象大小时有相当大的自由度。

对于下一组实验，我们假定：

- 卷数量恒定，
- 哈希函数将每个对象映射到卷，从而映射到唯一链，
- 每条链由从实现存储服务的处理器中选出的服务器组成。

客户端假定将请求发送给**分发器**（dispatcher），分发器（i）计算哈希以确定存储所关注对象的卷及链，（ii）将请求转发给对应链。（主控向分发器发送每卷的配置信息，避免主控直接与客户端通信。插入分发器为更新和查询增加 1 ms 延迟，但不影响吞吐量。）链产生的回复直接发送给客户端，不经由分发器。

实验中有 25 个客户端，每个随机、均匀地跨链提交查询与更新。客户端尽可能快地发送请求，但每个客户端一次只能有一个未完成请求。

为便于与 GFS 实验 [11] 比较，我们假定 5000 个卷、每个复制三次，并变化服务器数量。我们发现 chain、p/b、weak chain 和 weak p/b 之间差异很小或没有，故图 5 仅展示一种——链式复制——下，每客户端平均请求吞吐量随服务器数量和更新请求百分比的变化。

### 5.3 故障对吞吐量的影响

在链式复制下，每次服务器故障会启动三阶段过程：

1. 主控检测到服务器故障前经过一段时间（实验中保守假定 10 秒）。
2. 故障服务器随后从链中删除。
3. 主控最终向该链添加新服务器并启动数据恢复过程，耗时与（i）故障服务器上存储的数据量、（ii）可用网络带宽成正比。

检测故障或从链中删除故障服务器的延迟可能增加请求处理延迟和短暂中断持续时间。本节实验探索这一点。

我们假定存储服务具有表 1 中的参数；这些值受 GFS [11] 报告启发。关于网络带宽的假定基于在 100 Mbit/秒网络中最多预留一半带宽用于数据恢复；将单台服务器上存储的 150 GB 复制所需时间为 6 小时 40 分钟。

为测量故障对存储服务的影响，我们施加负载。负载的具体细节影响不大。实验使用 11 个客户端。每个客户端重复随机选择对象、执行操作并等待回复；若 3 秒内未收到回复，看门狗计时器使客户端开始下一轮循环。10 个客户端仅提交查询操作；第 11 个客户端仅提交更新操作。

### 5.4 关键数据的大规模复制

随着服务器数量增加，服务器故障的聚合率也会增加。若故障过多，某卷可能变得不可用。该概率取决于卷如何放置在服务器上，尤其是数据恢复期间可实现的并行程度。

我们研究了三种卷放置策略：

- **ring**：卷的副本放置在环上由卷标识符的**一致性哈希**（consistent hash）确定的连续服务器上。这是 CFS [7] 和 PAST [19] 使用的策略。可能的并行数据恢复数量受链长 t 限制。
- **rndpar**：卷的副本随机放置在服务器上。这本质上是 GFS 使用的策略。注意，若有足够服务器，可能的并行数据恢复数量无限制。
- **rndseq**：卷的副本随机放置在服务器上（同 rndpar），但最大并行数据恢复数量受 t 限制（同 ring）。该策略在我们所知的任何系统中均未使用，但可作为量化放置与并行恢复影响的基准。

为理解并行数据恢复的优势，考虑故障的服务器 F 曾参与链 C1, C2, …, Cn。对每条链 Ci，数据恢复需要获取卷数据的源以及将成为链 Ci 新元素的宿主。若有足够处理器且对卷放置无约束，容易确保新元素互不相交。在卷随机放置下，源很可能也不相交。在源与新元素均不相交时，链 C1, C2, …, Cn 的数据恢复可并行进行。C1, C2, …, Cn 数据恢复的较短间隔意味着存在较短的脆弱窗口，在此期间少量并发故障会使某卷不可用。

我们试图量化任意对象的**不可用平均间隔**（Mean Time Between Unavailability, MTBU）随服务器数量和放置策略的变化。假定每台服务器呈现指数分布的故障，**平均故障间隔**（Mean Time Between Failures, MTBF）为 24 小时。随着存储系统中服务器数量增加，卷数量也会增加（否则为何添加服务器）。实验中，卷数量定义为初始服务器数量的 100 倍，每台服务器在时刻 0 存储 100 个卷。

我们假定将一台服务器上的所有数据复制到另一台需要 4 小时，对应在 100 Mbit/秒网络中复制 100 GB，且仅一半带宽可用于数据恢复。与 GFS 实验相同，网络上最大并行数据恢复数量限制为服务器的 40%，最小传输时间设为 10 秒（复制单个 GFS 对象——64 KB——所需时间）。

图 7(a) 显示 ring 策略的 MTBU 随服务器数量呈近似 **Zipf** 分布。因此，为维持特定 MTBU，增加服务器数量时需要增加链长 t。从图中看，链长似乎需要随服务器数量的对数增加。

图 7(b) 显示 rndseq 的 MTBU。当 t > 1 时，rndseq 的 MTBU 低于 ring。与 ring 相比，随机放置更差，因为随机放置下共同存储链副本的 t 台服务器集合更多，因此链因故障丢失的概率更高。

然而，若有足够服务器，随机放置使额外并行恢复机会成为可能。图 7(c) 显示 rndpar 的 MTBU。在服务器较少时，rndpar 与 rndseq 表现相同，但随服务器数量增加的并行恢复机会改善了 MTBU，最终 rndpar 优于 rndseq，更重要的是优于 ring。

## 6 相关工作

**可扩展性**：链式复制是 Jiménez-Peris 和 Patiño-Martínez [14] 所称 **ROWAA**（Read One, Write All Available，读一写全可用）方法的一个例子。他们报告 ROWAA 方法在可用性扩展上优于**法定人数**（quorum）技术，声称 ROWAA 方法的可用性随副本数量指数改善。他们还论证非 ROWAA 复制方法必然更差。由于 ROWAA 方法在除近乎纯写应用外的场景下也表现出比已知最佳法定人数系统更好的吞吐量 [14]，ROWAA 似乎是大多数实际场景下复制的更好选择。

许多文件服务以性能与可扩展性换取一致性。例子包括 Bayou [17]、Ficus [13]、Coda [15] 和 Sprite [5]。通常，这些系统通过提供半自动修复不一致的工具，允许在网络分区时继续运行。我们的链式复制不提供分区操作的优雅处理，而是换取同时支持三者：高性能、可扩展性和强一致性。

大规模**点对点**（peer-to-peer）可靠文件系统是相对较新的研究方向。OceanStore [6]、FARSITE [2] 和 PAST [19] 是例子。其中仅 OceanStore 提供强（事实上是事务性）一致性保证。

Google 文件系统（GFS）[11] 是面向与激发链式复制发明的应用类似应用的大规模集群可靠文件系统。但在 GFS 中，并发覆盖不被串行化，读操作与写操作不同步。因此，不同副本可能处于不同状态，读操作返回的内容可能看似从 GFS 中自发消失。这种弱语义给使用 GFS 的应用程序员带来负担。

**可用性与一致性**：Yu 和 Vahdat [25] 探索一致性与可用性之间的权衡。他们论证即使在不一致模型中，若要在长期保持可用性，尽可能接近强一致性也很重要。另一方面，Gray 等人 [12] 论证强一致性系统在扩展时具有不稳定行为，并提出**试探性更新事务**（tentative update transaction）以规避这些可扩展性问题。

Amza 等人 [4] 提出针对复制优化的**单副本可串行化**（one-copy serializable）事务协议。与链式复制一样，更新发送给所有副本，而查询仅由已知存储所有已完成更新的副本处理。（在链式复制中，尾是已知存储所有已完成更新的那个副本。）[4] 的协议在提供弱一致性的复制协议下表现相当，且在副本数量上扩展良好。未给出面对故障时的行为分析。

**副本放置**：先前关于副本放置的工作集中于实现高吞吐量和/或低延迟，而非支持高可用性。Acharya 和 Zdonik [1] 主张根据未来访问预测（基于过去访问）放置副本。在 Mariposa 项目 [23] 中，一组规则允许用户指定在哪里创建副本、是将数据移到查询还是将查询移到数据、在哪里缓存数据等。一致性是事务性的，但未考虑可用性。Wolfson 等人考虑优化数据库副本放置以优化性能的策略 [24]。OceanStore 项目也考虑副本放置 [10, 6]，但从 **CDN**（Content Distribution Network，内容分发网络，如 Akamai）视角，在支持特定服务质量保证的同时尽可能少地创建副本。还有大量工作（如 [18]）关注网页副本的放置，均从降低延迟和网络负载的角度。

Douceur 和 Wattenhofer 研究如何在 FARSITE [2] 中最大化文件的最坏情况可用性，同时将存储负载均匀分布在所有服务器上 [8, 9]。假定服务器具有不同可用性。他们考虑的算法在能改善文件可用性时反复在机器间交换文件。结果对简单场景具有理论性质；尚不清楚这些算法在现实存储系统中效果如何。

## 7 结论

链式复制支持查询与更新请求的高吞吐量、数据对象的高可用性以及强一致性保证。这在部分上之所以可能，是因为使用链式复制构建的存储服务可以且确实会经历短暂中断，但客户端无法将此类中断与丢失的消息区分开来。因此，链式复制引入的短暂中断不会使客户端暴露于新的故障模式——链式复制在向客户端隐藏的故障与不隐藏的故障之间取得了有趣的平衡。

采用链式复制时，数据对象的高可用性来自精心选择卷副本在服务器上的放置策略。我们的实验表明，采用基于 DHT 的放置策略时，可用性不太可能随服务器数量增加而扩展；但我们也表明，若将卷的随机放置与 GFS 引入的并行数据恢复结合使用，随机放置确实允许可用性随服务器数量扩展。

我们当前的原型主要面向相对同质的 LAN 集群。若将原型部署在异构广域环境中，卷副本的均匀随机放置将不再合理。相反，副本放置将不得不取决于访问模式、网络邻近性和观测到的主机可靠性。为控制负载不均衡，对链元素重新排序的协议可能变得至关重要。

我们的链式复制原型实现由 1500 行 Java 代码组成，另有 2300 行 Java 代码用于 Paxos 库。链式复制协议被构建为向存储服务（或其他应用）进行**上调用**（upcall）的库。本文实验假定仿真网络上的「空服务」。但该库也可在 Java socket 库上运行，因此可用于支持各种类存储服务应用。
