# 第7章 摄取

你已经了解了作为数据工程师可能遇到的各种源系统以及存储数据的方式。现在我们将注意力转向从各种源系统摄取数据的模式与选择。本章讨论数据摄取、摄取阶段的关键工程考量、批处理和流式摄取的主要模式、你将遇到的技术、开发摄取管道时的协作对象，以及暗流在摄取阶段的作用。

## 什么是数据摄取？

**数据摄取（data ingestion）**是将数据从一处移动到另一处的过程。数据摄取意味着数据从源系统移入数据工程生命周期的存储，摄取是中间步骤。

数据摄取与**数据集成（data integration）**不同：摄取是点对点移动，集成将不同来源的数据组合成新数据集。

## 数据管道定义

> 数据管道是使数据贯穿数据工程生命周期各阶段的架构、系统和流程的组合。

数据管道可以是传统 ETL 系统，也可以是云原生管道——从 100 个源拉取数据、组合成 20 张宽表、训练 5 个 ML 模型、部署到生产并监控性能。

## 摄取阶段的关键工程考量

- **有界 versus 无界数据**：所有数据在未被有界之前都是无界的
- **频率**：批处理、微批、实时
- **同步 versus 异步**：同步紧密耦合，异步事件级解耦
- **序列化与反序列化**：确保目标能正确反序列化
- **吞吐量与可扩展性**：设计可扩展和收缩的系统
- **可靠性与持久性**：评估风险，构建适当冗余
- **有效载荷（payload）**：类型、形状、大小、模式
- **推送 versus 拉取 versus 轮询**

## 批处理摄取考量

- 快照或增量提取
- 基于文件的导出与摄取
- ETL versus ELT
- 插入、更新与批大小
- 数据迁移

## 消息与流式摄取考量

- 模式演进
- 迟到数据
- 排序与多次交付
- 重放
- 生存时间（TTL）
- 消息大小
- 错误处理与死信队列

## 摄取数据的方式

- 直接数据库连接
- 变更数据捕获（CDC）
- API
- 消息队列与事件流平台
- 托管数据连接器
- 对象存储传输
- EDI、文件导出、Shell、SSH、SFTP、Webhook、Web 抓取等

## 小结

数据摄取是数据工程生命周期的关键阶段。根据用例选择批处理或流式、同步或异步、推送或拉取。考虑有界/无界、频率、序列化、吞吐量、可靠性和有效载荷特征。暗流——安全、数据管理、DataOps、数据架构、编排、软件工程——贯穿摄取阶段。
