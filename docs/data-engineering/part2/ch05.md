# 第5章 源系统中的数据生成

欢迎来到数据工程生命周期的第一阶段：源系统中的数据生成。如前所述，数据工程师的工作是从源系统获取数据、对其进行处理，并使其有助于服务下游用例。但在获取原始数据之前，你必须了解数据存在何处、如何生成以及其特征和 quirks（特性）。

本章涵盖流行的运营源系统模式、重要类型的源系统，以及数据工程暗流如何应用于生命周期的这一阶段（图 5-1）。

随着数据激增，尤其是数据共享（下文讨论）的兴起，我们预计数据工程师的角色将大幅转向理解数据源与目的地之间的相互作用。数据工程的基本管道任务——将数据从 A 移到 B——将显著简化。另一方面，理解数据在源系统中创建时的本质仍至关重要。

## 数据来源：数据如何创建？

在学习生成数据的系统的各种底层运营模式时，理解数据如何创建至关重要。**数据（data）**是未组织、无上下文的 facts 和 figures 集合。数据可以多种方式创建，包括模拟和数字。

**模拟数据创建**发生在现实世界，如口语、手语、纸上书写或演奏乐器。这种模拟数据往往是短暂的；你有多少次口头对话的内容在对话结束后就消失在空气中？

**数字数据**要么由模拟数据转换为数字形式产生，要么是数字系统的原生产物。模拟转数字的例子是移动短信应用将模拟语音转换为数字文本。数字数据创建的例子是电商平台上的信用卡交易。客户下单，交易从其信用卡扣款，交易信息保存到各种数据库。

本章我们将使用一些常见例子，如与网站或移动应用交互时创建的数据。但事实上，数据无处不在——我们从 IoT 设备、信用卡终端、望远镜传感器、股票交易等捕获数据。

熟悉你的源系统及其如何生成数据。花时间阅读源系统文档，理解其模式和 quirks。如果你的源系统是 RDBMS（关系型数据库管理系统），学习其运作方式（写入、提交、查询等）；学习可能影响你摄取能力的源系统方方面面。

## 源系统：主要概念

源系统以多种方式产生数据。本节讨论你在与源系统工作时经常遇到的主要概念。

### 文件与非结构化数据

**文件（file）**是通常存储在磁盘上的字节序列。应用经常将数据写入文件。文件可能存储本地参数、事件、日志、图像和音频。

此外，文件是数据交换的通用媒介。尽管数据工程师希望以编程方式获取数据，但世界上许多地方仍通过文件收发数据。例如，如果你从政府机构获取数据，很可能需要以 Excel 或 CSV 文件下载，或通过电子邮件接收。

作为数据工程师，你会遇到的主要源文件格式——源自手动或源系统进程输出的文件——包括 Excel、CSV、TXT、JSON 和 XML。这些文件各有 quirks，可以是结构化的（Excel、CSV）、半结构化的（JSON、XML、CSV）或非结构化的（TXT、CSV）。

虽然作为数据工程师你会大量使用某些格式（如 Parquet、ORC、Avro），我们稍后讨论这些，此处聚焦源系统文件。第 6 章涵盖文件的技术细节。

### API

**应用编程接口（API）**是系统间交换数据的标准方式。理论上，API 简化数据工程师的摄取任务。实践中，许多 API 仍暴露大量需工程师管理的数据复杂度。尽管各种服务和框架以及自动化 API 数据摄取的服务兴起，数据工程师仍经常需要投入大量精力维护自定义 API 连接。我们将在本章后面更详细地讨论 API。

### 应用数据库（OLTP 系统）

**应用数据库**存储应用的状态。标准例子是存储银行账户余额的数据库。随着客户交易和支付发生，应用更新银行账户余额。

通常，应用数据库是**在线事务处理（OLTP）**系统——以高速率读写单条数据记录的数据库。OLTP 系统常被称为事务数据库，但这并不一定意味着该系统支持原子事务。

更一般地，OLTP 数据库支持低延迟和高并发。RDBMS 数据库可以在不到一毫秒内选择或更新一行（不计网络延迟），并每秒处理数千次读写。文档数据库集群可以管理更高的文档提交速率，但可能牺牲一致性。某些图数据库也可以处理事务用例。

从根本上说，当数千甚至数百万用户可能同时与应用交互、并发更新和写入数据时，OLTP 数据库作为应用后端表现良好。OLTP 系统不太适合由大规模分析驱动的用例，即单次查询必须扫描大量数据。

**ACID**

对原子事务的支持是一组关键数据库特征的一部分，统称为 **ACID**（你可能记得第 3 章，这代表原子性、一致性、隔离性、持久性）。一致性意味着任何数据库读取将返回检索项的最后写入版本。隔离性意味着如果对同一事物的两次更新并发进行，最终数据库状态将与按提交顺序顺序执行这些更新一致。持久性表示已提交的数据永远不会丢失，即使发生断电。

注意，ACID 特征并非支持应用后端的必要条件，放宽这些约束可以显著提升性能和规模。然而，ACID 特征保证数据库将维护世界的一致视图，大大简化应用开发者的任务。

所有工程师（数据或其他）必须理解在有和没有 ACID 的情况下操作。例如，为提高性能，某些分布式数据库使用放宽的一致性约束，如最终一致性。理解你正在使用的一致性模型有助于你防止灾难。

**原子事务**

**原子事务（atomic transaction）**是一组作为单元提交的多个变更。在图 5-2 的例子中，运行在 RDBMS 上的传统银行应用执行 SQL 语句，检查两个账户余额——账户 A（源）和账户 B（目标）。如果账户 A 有足够资金，则将资金从账户 A 转到账户 B。整个事务应同时更新两个账户余额或失败而不更新任一账户。即，整个操作应作为事务发生。

**OLTP 与分析**

通常，小公司直接在 OLTP 上运行分析。这种模式短期内有效，但最终不可扩展。在某个时刻，在 OLTP 上运行分析查询会遇到性能问题，由于 OLTP 的结构限制或与竞争事务工作负载的资源争用。数据工程师必须理解 OLTP 和应用后端的内部运作，以建立与分析系统的适当集成，而不降低生产应用性能。

随着公司在 SaaS 应用中提供更多分析能力，对混合能力——快速更新与组合分析能力——的需求为数据工程师带来了新挑战。我们将使用**数据应用（data application）**一词来指代融合事务和分析工作负载的应用。

### 在线分析处理系统（OLAP）

与 OLTP 系统相反，**在线分析处理（OLAP）**系统为运行大型分析查询而构建，通常不擅长单条记录查找。例如，现代列式数据库针对扫描大量数据优化，放弃索引以提高可扩展性和扫描性能。任何查询通常涉及扫描最小数据块，通常 100 MB 或更大。尝试在此类系统中每秒查找数千个单独项将使其不堪重负，除非与为此用例设计的缓存层结合使用。

注意，我们使用 OLAP 一词指代任何支持大规模交互式分析查询的数据库系统；我们不仅限于支持 OLAP 立方体（多维数据数组）的系统。OLAP 的「在线」部分意味着系统持续监听传入查询，使 OLAP 系统适合交互式分析。

尽管本章涵盖源系统，OLAP 通常是分析的存储和查询系统。为什么我们在源系统章节讨论它们？在实际用例中，工程师经常需要从 OLAP 系统读取数据。例如，数据仓库可能提供用于训练 ML 模型的数据。或者，OLAP 系统可能服务反向 ETL 工作流，其中分析系统中的派生数据被发送回源系统，如 CRM、SaaS 平台或事务应用。

### 变更数据捕获（CDC）

**变更数据捕获（CDC）**是提取数据库中发生的每个变更事件（插入、更新、删除）的方法。CDC 常用于近实时数据库复制或为下游处理创建事件流。

CDC 根据数据库技术不同而处理方式不同。关系型数据库通常生成直接存储在数据库服务器上的事件日志，可处理以创建流。（见「数据库日志」）。许多云 NoSQL 数据库可以将日志或事件流发送到目标存储位置。

### 日志

**日志（log）**捕获系统中发生事件的信息。例如，日志可能捕获 Web 服务器上的流量和使用模式。你的桌面计算机操作系统（Windows、macOS、Linux）在系统启动以及应用启动或崩溃时记录事件。

日志是丰富的数据源，对下游数据分析、ML 和自动化 potentially 有价值。以下是一些熟悉的日志来源：

- 操作系统
- 应用
- 服务器
- 容器
- 网络
- IoT 设备

所有日志跟踪事件和事件元数据。至少，日志应捕获谁、什么、何时：

- **谁**：与事件相关的人、系统或服务账户（如 Web 浏览器用户代理或用户 ID）
- **发生了什么**：事件及相关元数据
- **何时**：事件的时间戳

**日志编码**

日志以几种方式编码：

- **二进制编码日志**：以自定义紧凑格式编码数据，以提高空间效率和快速 I/O。数据库日志（在「数据库日志」中讨论）是标准例子。
- **半结构化日志**：以对象序列化格式（通常是 JSON）的文本编码。半结构化日志机器可读且可移植。然而，它们比二进制日志效率低得多。尽管名义上机器可读，从它们提取价值通常需要大量自定义代码。
- **纯文本（非结构化）日志**： essentially 存储软件的控制台输出。因此，不存在通用标准。这些日志可以为数据科学家和 ML 工程师提供有用信息，尽管从原始文本数据提取有用信息可能很复杂。

**日志分辨率**

日志以各种分辨率和日志级别创建。日志分辨率指日志中捕获的事件数据量。例如，数据库日志从数据库事件捕获足够信息，以允许在任何时间点重建数据库状态。

另一方面，为大数据系统在日志中捕获所有数据变更通常不切实际。相反，这些日志可能仅记录特定类型的提交事件已发生。日志级别指记录日志条目所需的条件，特别是关于错误和调试。软件通常可配置为记录每个事件或仅记录错误。

**日志延迟：批处理或实时**

批处理日志通常持续写入文件。单个日志条目可以写入消息系统（如 Kafka 或 Pulsar）以用于实时应用。

### 数据库日志

数据库日志足够重要，值得更详细的覆盖。**预写日志（write-ahead log）**——通常以特定数据库原生格式存储的二进制文件——在数据库保证和可恢复性中扮演关键角色。数据库服务器接收对数据库表的写入和更新请求（见图 5-3），在确认请求之前将每个操作存储在日志中。确认带有与日志相关的保证：即使服务器失败，它也可以在重启时通过完成日志中未完成的工作恢复其状态。

数据库日志在数据工程中 extremely 有用，尤其是用于 CDC 从数据库变更生成事件流。

### CRUD 与仅插入

**CRUD**（创建、读取、更新、删除）是编程中常用的事务模式，代表持久存储的四种基本操作。CRUD 是在数据库中存储应用状态的最常见模式。CRUD 的基本原则是数据必须先创建才能使用。数据创建后，可以读取和更新。最后，数据可能需要销毁。CRUD 保证这四种操作将发生在数据上，无论其存储方式如何。

CRUD 是软件应用中广泛使用的模式，你通常会在 API 和数据库中发现 CRUD 用于 RESTful HTTP 请求以及从数据库存储和检索数据。

与任何数据库一样，我们可以使用基于快照的提取从应用应用 CRUD 操作的数据库获取数据。另一方面，使用 CDC 的事件提取为我们提供操作的完整历史，并 potentially 允许近实时分析。

**仅插入模式（insert-only）**

**仅插入模式**直接在包含数据的表中保留历史。不是更新记录，而是插入带时间戳的新记录，指示创建时间（表 5-1）。例如，假设你有一个客户地址表。遵循 CRUD 模式，如果客户更改地址，你只需更新记录。使用仅插入模式，插入具有相同客户 ID 的新地址记录。要按客户 ID 读取当前客户地址，你将查找该 ID 下的最新记录。

| 记录 ID | 值 | 时间戳 |
|---------|-----|--------|
| 1 | 40 | 2021-09-19T00:10:23+00:00 |
| 1 | 51 | 2021-09-30T00:12:00+00:00 |

在某种意义上，仅插入模式直接在表本身中维护数据库日志，如果应用需要访问历史则特别有用。例如，仅插入模式将很好地适用于旨在呈现客户地址历史的银行应用。

单独的仅插入分析模式通常与常规 CRUD 应用表一起使用。在仅插入 ETL 模式中，每当 CRUD 表中发生更新时，数据管道在目标分析表中插入新记录。

仅插入有几个缺点。首先，表可能变得很大，特别是如果数据经常变化，因为每次变更都插入表中。有时根据记录日落日期或保留的最大记录版本数清除记录，以保持表大小合理。第二个缺点是记录查找会产生额外开销，因为查找当前状态涉及运行 `MAX(created_timestamp)`。如果单个 ID 下有数百或数千条记录，此查找操作运行成本高昂。

### 消息与流

与事件驱动架构相关，你经常看到互换使用的两个术语是**消息队列（message queue）**和**流式平台（streaming platform）**，但两者之间存在微妙但重要的区别。定义和对比这些术语是值得的，因为它们涵盖与源系统以及跨越整个数据工程生命周期的实践和技术相关的许多大概念。

**消息（message）**是在两个或多个系统之间传递的原始数据（图 5-4）。例如，我们有系统 1 和系统 2，系统 1 向系统 2 发送消息。这些系统可以是不同的微服务、向无服务器函数发送消息的服务器等。消息通常通过消息队列从发布者发送到消费者，一旦消息被传递，就从队列中移除。

消息是事件驱动系统中的离散单一信号。例如，IoT 设备可能将带有最新温度读数的消息发送到消息队列。然后该消息由确定炉子应打开还是关闭的服务摄取。该服务向采取适当行动的炉子控制器发送消息。一旦消息被接收并采取行动，消息就从消息队列中移除。

相比之下，**流（stream）**是事件记录的仅追加日志。（流被摄取并存储在事件流平台中，我们在「消息与流」中更详细地讨论。）随着事件发生，它们按有序序列累积（图 5-5）；时间戳或 ID 可能对事件排序。（注意，由于分布式系统的微妙之处，事件并不总是按精确顺序传递。）

当你在意许多事件中发生了什么时，你将使用流。由于流的仅追加性质，流中的记录在长保留窗口内持久化——通常数周或数月——允许对记录进行复杂操作，如多记录聚合或能够回退到流中的某个时间点。

值得注意的是，处理流的系统可以处理消息，流式平台经常用于消息传递。当我们想执行消息分析时，我们经常在流中累积消息。在我们的 IoT 例子中，触发炉子打开或关闭的温度读数也可能稍后被分析以确定温度趋势和统计。

## 时间类型

虽然时间对所有数据摄取都是重要考量，但在流式上下文中变得更加关键和微妙，我们将数据视为连续的并期望在其产生后不久消费。让我们看看摄取数据时你会遇到的关键时间类型：事件生成时间、摄取和处理时间以及处理耗时（图 5-6）。

::: info 图 5-6 事件时间、摄取时间、处理时间和处理耗时
:::

**事件时间（event time）**表示事件在源系统中生成的时间，包括原始事件本身的时间戳。事件创建后、在下游摄取和处理之前，将发生不确定的时间滞后。始终为事件经过的每个阶段包含时间戳。在事件发生时以及每个时间阶段——创建、摄取和处理时——记录事件。使用这些时间戳日志准确跟踪数据在数据管道中的移动。

数据创建后，被摄取到某处。**摄取时间（ingestion time）**表示事件从源系统摄取到消息队列、缓存、内存、对象存储、数据库或任何其他数据存储位置的时间（见第 6 章）。摄取后，数据可能立即处理；或在几分钟、几小时或几天内；或 simply 在存储中无限期持久化。

**处理时间（process time）**发生在摄取时间之后，当数据被处理时（通常是转换）。**处理耗时（processing time）**是数据处理所花费的时间，以秒、分钟、小时等衡量。

你希望记录这些各种时间， preferably 以自动化方式。沿数据工作流设置监控，以捕获事件何时发生、何时被摄取和处理以及处理事件花费了多长时间。

## 源系统实践细节

本节讨论与现代源系统交互的实践细节。我们将深入常见数据库、API 和其他方面的细节。这些信息的保质期将比前面讨论的主要概念短；流行的 API 框架、数据库和其他细节将继续快速变化。

尽管如此，这些细节是工作数据工程师的关键知识。我们建议你将这些信息作为基线知识学习，但广泛阅读以跟上持续发展。

### 数据库

在本节中，我们将看看作为数据工程师会遇到常见源系统数据库技术，以及与这些系统工作的高层考量。数据库类型与数据用例一样多。

**理解数据库技术的主要考量**

这里我们介绍跨各种数据库技术发生的主要概念，包括支持软件应用和支持分析用例的：

- **数据库管理系统（DBMS）**：用于存储和服务数据的数据库系统。由存储引擎、查询优化器、灾难恢复和管理数据库系统的其他关键组件组成。
- **查找**：数据库如何查找和检索数据？索引可以帮助加速查找，但并非所有数据库都有索引。了解你的数据库是否使用索引；如果是，设计和维护它们的最佳模式是什么？理解如何利用以实现高效提取。对主要索引类型（包括 B 树和日志结构合并树 LSM）有基本了解也有帮助。
- **查询优化器**：数据库是否使用优化器？其特点是什么？
- **扩展和分布**：数据库是否随需求扩展？它部署什么扩展策略？是水平扩展（更多数据库节点）还是垂直扩展（单机更多资源）？
- **建模模式**：什么建模模式最适合数据库（如数据规范化或宽表）？（见第 8 章关于数据建模的讨论。）
- **CRUD**：数据如何在数据库中查询、创建、更新和删除？每种数据库处理 CRUD 操作的方式不同。
- **一致性**：数据库是完全一致还是支持放宽的一致性模型（如最终一致性）？数据库是否支持可选的读写一致性模式（如强一致读取）？

我们将数据库分为关系型和非关系型类别。事实上，非关系型类别更加多样，但关系型数据库仍在应用后端占据重要空间。

**关系型数据库**

**关系型数据库管理系统（RDBMS）**是最常见的应用后端之一。关系型数据库在 1970 年代由 IBM 开发，1980 年代由 Oracle 普及。互联网的增长见证了 LAMP 栈（Linux、Apache Web 服务器、MySQL、PHP）的兴起以及供应商和开源 RDBMS 选项的激增。即使 NoSQL 数据库（下一节描述）兴起，关系型数据库仍然 extremely 流行。

数据存储在关系（行）表中，每个关系包含多个字段（列）；见图 5-7。注意我们在本书中互换使用列和字段。表中的每个关系具有相同的模式（具有指定静态类型如字符串、整数或浮点的列序列）。行通常作为磁盘上的连续字节序列存储。

表通常由主键索引，主键是表中每行的唯一字段。主键的索引策略与表在磁盘上的布局密切相关。

表还可以有各种外键——与其他表中主键值连接的字段值的字段，促进 join，并允许跨多表分布数据的复杂模式。特别是，可以设计规范化模式。**规范化（normalization）**是确保记录中的数据不在多个地方重复的策略，从而避免需要同时更新多个位置的状态并防止不一致（见第 8 章）。

RDBMS 系统通常符合 ACID。结合规范化模式、ACID 合规性和对高事务速率的支持，使关系型数据库系统成为存储快速变化应用状态的理想选择。数据工程师的挑战是确定如何随时间捕获状态信息。

**非关系型数据库：NoSQL**

虽然关系型数据库对许多用例都很出色，但它们不是万能解决方案。我们经常看到人们从关系型数据库开始，认为它是通用设备，并强行塞入大量用例和工作负载。随着数据和查询需求演变，关系型数据库在其重量下崩溃。那时，你将希望使用适合压力下特定工作负载的数据库。进入非关系型或 **NoSQL** 数据库。NoSQL 代表「不仅仅是 SQL」，指放弃关系范式的一整类数据库。

一方面，放弃关系约束可以提高性能、可扩展性和模式灵活性。但正如架构中总是如此，存在权衡。NoSQL 数据库通常也放弃各种 RDBMS 特征，如强一致性、join 或固定模式。

本书的一大主题是数据创新是持续的。让我们快速看看 NoSQL 的历史，因为它有助于获得关于数据创新如何以及为何影响你作为数据工程师工作的视角。在 2000 年代初，Google 和 Amazon 等科技公司开始超越其关系型数据库，并开创了新的分布式非关系型数据库以扩展其 Web 平台。

虽然 NoSQL 一词最早出现在 1998 年，现代版本由 Eric Evans 在 2000 年代创造。NoSQL 在 2022 年仍然模糊，但已被广泛采用来描述「新派」数据库的宇宙，作为关系型数据库的替代品。

有无数种 NoSQL 数据库设计用于几乎任何可想象的用例。因为 NoSQL 数据库太多无法在本节详尽覆盖，我们考虑以下数据库类型：键值、文档、宽列、图、搜索和时间序列。这些数据库都非常流行并享有广泛采用。数据工程师应理解这些类型的数据库，包括使用考量、它们存储的数据结构以及如何在数据工程生命周期中利用每种。

- **键值存储**：键值数据库是使用唯一标识每条记录的键检索记录的非关系型数据库。这类似于许多编程语言中的哈希映射或字典数据结构，但 potentially 更可扩展。键值存储涵盖几种 NoSQL 数据库类型——例如文档存储和宽列数据库（接下来讨论）。不同类型的键值数据库提供各种性能特征以满足各种应用需求。
- **文档存储**：如前所述，文档存储是专门的键值存储。在此上下文中，文档是嵌套对象；我们通常可以将每个文档视为 JSON 对象用于实际目的。文档存储在集合中并按键检索。集合大致等同于关系型数据库中的表。关系型数据库和文档存储之间的一个关键区别是后者不支持 join。这意味着数据不能轻易规范化，即跨多表拆分。
- **宽列数据库**：宽列数据库针对存储大量数据、高事务速率和极低延迟优化。这些数据库可以扩展到极高的写入速率和大量数据。
- **图数据库**：图数据库显式地以数学图结构（作为节点和边的集合）存储数据。Neo4j 已证明 extremely 流行，而 Amazon、Oracle 和其他供应商提供其图数据库产品。粗略地说，当你想分析元素之间的连接性时，图数据库是很好的选择。
- **搜索数据库**：搜索数据库是用于搜索数据复杂和简单语义及结构特征的非关系型数据库。搜索数据库存在两个突出用例：文本搜索和日志分析。
- **时间序列数据库**：时间序列是按时间组织的一系列值。时间序列数据库针对时间序列数据的检索和统计处理优化。

### API

API 现在是云中、SaaS 平台之间以及公司内部系统之间交换数据的标准和普遍方式。Web 上存在多种类型的 API 接口，但我们主要关注围绕 HTTP 构建的，这是 Web 和云上最流行的类型。

**REST**

我们首先讨论 REST，目前占主导地位的 API 范式。如第 4 章所述，REST 代表表述性状态转移。这套构建 HTTP Web API 的实践和哲学由 Roy Fielding 在 2000 年的博士论文中提出。REST 围绕 HTTP 动词构建，如 GET 和 PUT；实践中，现代 REST 仅使用原始论文中概述的少数动词映射。

REST 的主要思想之一是交互是无状态的。与 Linux 终端会话不同，没有与关联状态变量（如工作目录）的会话概念；每个 REST 调用是独立的。REST 调用可以改变系统状态，但这些变更是全局的，应用于整个系统而不是当前会话。

**GraphQL**

GraphQL 在 Facebook 创建，作为应用数据的查询语言和通用 REST API 的替代品。REST API 通常将你的查询限制在特定数据模型，而 GraphQL 开辟了在单个请求中检索多个数据模型的可能性。这允许比 REST 更灵活和表达性的查询。GraphQL 围绕 JSON 构建，并以类似于 JSON 查询的形状返回数据。

**Webhooks**

**Webhook** 是一种简单的事件驱动数据传输模式。数据源可以是应用后端、网页或移动应用。当源系统中发生指定事件时，这会触发对数据消费者托管的 HTTP 端点的调用。注意连接从源系统到数据接收端，与典型 API 相反。因此，webhook 通常被称为反向 API。

**RPC 和 gRPC**

**远程过程调用（RPC）**在分布式计算中常用。它允许你在远程系统上运行过程。

**gRPC** 是 2015 年在 Google 内部开发并后来作为开放标准发布的远程过程调用库。仅其在 Google 的使用就足以值得纳入我们的讨论。许多 Google 服务，如 Google Ads 和 GCP，提供 gRPC API。gRPC 围绕 Protocol Buffers 开放数据序列化标准构建，也由 Google 开发。

### 数据共享

云数据共享的核心概念是多租户系统支持租户之间共享数据的安全策略。具体而言，任何具有细粒度权限系统的公有云对象存储系统都可以是数据共享平台。流行的云数据仓库平台也支持数据共享能力。当然，数据也可以通过下载或电子邮件交换共享，但多租户系统使过程更容易。

### 消息队列与事件流平台

事件驱动架构在软件应用中普遍存在，并有望进一步增长其流行度。首先，消息队列和事件流平台——事件驱动架构中的关键层——在云环境中更容易设置和管理。其次，**数据应用（data app）**——直接集成实时分析的应用——的兴起正在日益壮大。事件驱动架构在此设置中 ideal，因为事件既可以触发应用中的工作，也可以提供近实时分析。

请注意，流式数据（在此情况下为消息和流）跨越许多数据工程生命周期阶段。与通常直接附加到应用的 RDBMS 不同，流式数据的界限有时不那么清晰。这些系统用作源系统，但由于其短暂性质，它们经常跨越数据工程生命周期。

**消息队列**

消息队列是使用发布-订阅模型在离散系统之间异步发送数据（通常作为小的单独消息，以千字节计）的机制。数据发布到消息队列并传递给一个或多个订阅者（图 5-9）。订阅者确认收到消息，将其从队列中移除。

**事件流平台**

在某些方面，事件流平台是消息队列的延续，因为消息从生产者传递给消费者。如本章前面所讨论的，消息和流之间的重大区别是消息队列主要用于以某些传递保证路由消息。相比之下，事件流平台用于在有序记录日志中摄取和处理数据。在事件流平台中，数据保留一段时间，可以从过去的时间点重放消息。

**主题（topic）**：在事件流平台中，生产者将事件流式传输到主题，即相关事件的集合。主题可以包含欺诈警报、客户订单或 IoT 设备的温度读数等。在大多数事件流平台上，主题可以有零个、一个或多个生产者和消费者。

**流分区（stream partition）**：流分区是流细分为多个流。一个好的类比是多车道高速公路。拥有多条车道允许并行和更高吞吐量。消息按分区键分布在分区之间。具有相同分区键的消息将始终最终在同一分区中。

## 你将与谁协作

访问源系统时，理解你将与之协作的人至关重要。根据我们的经验，与源系统利益相关者的良好外交和关系是成功数据工程中被低估但关键的部分。

这些利益相关者是谁？通常，你将处理两类利益相关者：系统和数据利益相关者（图 5-12）。**系统利益相关者**构建和维护源系统；这些可能是软件工程师、应用开发者和第三方。**数据利益相关者**拥有和控制你所需数据的访问权限，通常由 IT、数据治理组或第三方处理。系统和数据利益相关者通常是不同的人或团队；有时，他们是同一批人。

建议与上游源系统所有者建立**数据合同（data contract）**。数据合同是源系统所有者与从该系统摄取数据用于数据管道的团队之间的书面协议。合同应说明正在提取什么数据、通过什么方法（全量、增量）、多频繁，以及源系统和摄取的联系人（人、团队）是谁。

此外，考虑与上游提供者建立 **SLA（服务级别协议）**。SLA 提供你对所依赖源系统的期望。SLA 的例子可能是「源系统的数据将可靠可用且高质量」。**SLO（服务级别目标）**根据你在 SLA 中约定的内容衡量性能。

## 暗流及其对源系统的影响

与数据工程生命周期的其他部分不同，源系统通常不在数据工程师的控制之下。有一个隐含假设（有人可能称之为希望），即源系统及其产生的数据的利益相关者和所有者正在遵循有关数据管理、DataOps（和 DevOps）、数据架构、编排和软件工程的最佳实践。数据工程师应尽可能获得上游支持，以确保在源系统中生成数据时应用暗流。这样做将使数据工程生命周期其余步骤进行得更顺利。

### 安全

安全至关重要，你最不想的是意外在源系统中创建漏洞点。

### 数据管理

源系统的数据管理对数据工程师具有挑战性。在大多数情况下，你对源系统及其产生的数据只有外围控制——如果有任何控制的话。尽最大可能，你应该理解源系统中数据的管理方式，因为这将直接影响你如何摄取、存储和转换数据。

### DataOps

运营卓越——DevOps、DataOps、MLOps、XOps——应延伸并支持整个栈和数据工程及生命周期。虽然这是理想的，但往往没有完全实现。

### 数据架构

与数据管理类似，除非你参与源系统架构的设计和维护，否则你对上游源系统架构的影响很小。你还应理解上游架构如何设计及其优缺点。

### 编排

在数据工程工作流中编排时，你主要关心的是确保你的编排可以访问源系统，这需要正确的网络访问、身份验证和授权。

### 软件工程

随着数据格局转向简化源系统访问和自动化的工具，你可能需要编写代码。

## 小结

源系统及其数据在数据工程生命周期中至关重要。数据工程师倾向于将源系统视为「别人的问题」——这样做风险自负！滥用源系统的数据工程师在生产宕机时可能需要另谋高就。

有 stick 就有 carrot。与源系统团队更好的协作可以带来更高质量的数据、更成功的成果和更好的数据产品。与这些团队的对口人员建立双向沟通流；设置流程以通知影响分析和 ML 的模式和应用变更。主动沟通你的数据需求以协助应用团队在数据工程过程中。

注意数据工程师与源系统团队之间的集成正在增长。一个例子是反向 ETL，它长期处于阴影中但最近崭露头角。我们还讨论了事件流平台可以在事件驱动架构和分析中发挥作用；源系统也可以是数据工程系统。在合理的地方构建共享系统。

寻找构建面向用户的数据产品的机会。与应用团队讨论他们希望向用户呈现的分析或 ML 可以改善用户体验的地方。让应用团队成为数据工程的利益相关者，并找到分享你成功的方法。
