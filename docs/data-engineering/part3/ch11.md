# 第11章 数据工程的未来

本书源于作者认识到领域的飞速变化为现有数据工程师、有意转入数据工程的人、技术经理以及希望更好理解数据工程如何融入公司的管理者创造了显著知识鸿沟。

当我们开始思考如何组织本书时，朋友们提出了不少反对意见：「你们怎么敢写一个变化如此之快的领域？！」在许多方面，他们是对的。数据工程领域——以及所有与数据相关的事物——确实感觉每天都在变化。在噪音中筛选并找到不太可能变化的信号，是组织和撰写本书最具挑战性的部分之一。

在本书中，我们聚焦于我们认为未来几年仍有价值的大概念——数据工程生命周期及其暗流的连续体。操作顺序、最佳实践和技术的名称可能变化，但生命周期的主要阶段可能在未来多年保持 intact（完整）。我们敏锐地意识到技术继续以令人疲惫的速度变化；在我们这个时代在技术领域工作可能感觉像坐过山车或 perhaps 镜子迷宫。

几年前，数据工程甚至还不存在作为领域或职位。现在你正在阅读一本名为《数据工程基础》的书！你已经了解了数据工程的基础——其生命周期、暗流、技术和最佳实践。你可能会问自己，数据工程的下一步是什么？虽然没有人能预测未来，但我们对过去、现在和当前趋势有良好的视角。我们有幸从前排目睹了数据工程的诞生和演进。这最后一章呈现我们对未来的思考，包括对持续发展的观察和对未来的大胆推测。

## 数据工程生命周期不会消失

虽然数据科学近年来获得了大部分关注，但数据工程正快速成熟为独特且可见的领域。它是技术中增长最快的职业之一，没有失去势头的迹象。随着公司意识到在转向 AI 和 ML 等「更性感」事物之前首先需要建立数据基础，数据工程将继续增长。

这一进步围绕数据工程生命周期展开。

有人质疑日益简单的工具和实践是否会导致数据工程师消失。这种思维是肤浅、懒惰和短视的。随着组织以新方式利用数据，将需要新的基础、系统和工作流来满足这些需求。数据工程师处于设计、架构、构建和维护这些系统的中心。如果工具变得更易用，数据工程师将向上游价值链移动，专注于更高层次的工作。数据工程生命周期不会很快消失。

## 复杂度下降与易用数据工具的兴起

简化、易用的工具持续降低数据工程门槛。这是一件好事，尤其是考虑到我们讨论过的数据工程师短缺。简化趋势将继续。数据工程不依赖特定技术或数据规模。它也不仅适用于大公司。在 2000 年代，部署「大数据」技术需要大团队和雄厚财力。SaaS 托管服务的兴起已在很大程度上消除了理解各种「大数据」系统内部运作的复杂性。数据工程现在是所有公司都能做的事情。

大数据是自身非凡成功的受害者。例如，Google BigQuery（GFS 和 MapReduce 的后代）可以查询 PB 级数据。曾为 Google 内部保留的这项强大技术，现在对任何拥有 GCP 账户的人可用。用户只需按存储和查询的数据付费，而不必构建庞大的基础设施栈。Snowflake、Amazon EMR 和许多其他超可扩展云数据解决方案竞争于此，并提供类似能力。

云对开源工具的使用方式产生了重大转变。即使在 2010 年代初，使用开源通常意味着下载代码并自行配置。如今，许多开源数据工具作为托管云服务提供，直接与专有服务竞争。Linux 在所有主要云上预配置并安装在服务器实例上。AWS Lambda 和 Google Cloud Functions 等无服务器平台允许你在几分钟内部署事件驱动应用，使用 Python、Java、Go 等主流语言在幕后运行于 Linux 之上。希望使用 Apache Airflow 的工程师可以采用 Google 的 Cloud Composer 或 AWS 的托管 Airflow 服务。托管 Kubernetes 允许我们构建高度可扩展的微服务架构。等等。

这从根本上改变了围绕开源代码的讨论。在许多情况下，托管开源与其专有服务竞争对手一样易用。具有高度专业化需求的公司也可以部署托管开源，然后如果需要定制底层代码，稍后迁移到自托管开源。

另一个重要趋势是现成数据连接器的流行度增长（在本书写作时，流行的包括 Fivetran 和 Airbyte）。数据工程师传统上花费大量时间和资源构建和维护连接到外部数据源的管道。新一代托管连接器极具吸引力，即使对高度技术工程师也是如此，因为他们开始认识到为其他项目重新获得时间和心智带宽的价值。API 连接器将成为外包问题，使数据工程师能够专注于驱动业务的独特问题。

数据工具领域激烈竞争与数据工程师数量增长的交叉点意味着数据工具将继续降低复杂性，同时增加更多功能和特性。这种简化只会增长数据工程实践，因为越来越多的公司发现从数据中发现价值的机会。

## 云规模数据 OS 与互操作性提升

让我们简要回顾（单设备）操作系统的一些内部运作，然后将其与数据和云联系起来。无论你使用智能手机、笔记本电脑、应用服务器还是智能恒温器，这些设备都依赖操作系统提供基本服务并编排任务和进程。例如，我可以看到我正在打字的 MacBook Pro 上大约运行着 300 个进程。除其他外，我看到了 WindowServer（负责在图形界面中提供窗口）和 CoreAudio（负责提供底层音频能力）等服务。

当我在这台机器上运行应用时，它不会直接访问声音和图形硬件。相反，它向操作系统服务发送命令来绘制窗口和播放声音。这些命令通过标准 API 发出；规范告诉软件开发人员如何与操作系统服务通信。操作系统编排启动过程以提供这些服务，根据它们之间的依赖关系以正确顺序启动每个服务；它还通过监控服务并在失败时以正确顺序重启它们来维护服务。

现在让我们回到云中的数据。我们在本书中提到的简化数据服务（如 Google Cloud BigQuery、Azure Blob Storage、Snowflake 和 AWS Lambda）类似于操作系统服务，但规模大得多，跨越多个机器而非单个服务器运行。

既然这些简化服务可用，云数据操作系统概念的下一阶段演进将在更高抽象层次发生。Benn Stancil 呼吁出现用于构建数据管道和数据应用的标准化数据 API。我们预测数据工程将逐渐围绕少数数据互操作标准 coalesce（汇聚）。云中的对象存储将作为各种数据服务之间的批处理接口层日益重要。新一代文件格式（如 Parquet 和 Avro）已经在云数据交换方面取代了 CSV 的可怕互操作性和原始 JSON 的糟糕性能。

数据 API 生态系统的另一个关键要素是描述模式和数据层次的元数据目录。目前，这一角色主要由遗留的 Hive Metastore 承担。我们预计将出现新的进入者取代它。元数据将在数据互操作中扮演关键角色，无论是在应用和系统之间还是在云和网络之间，推动自动化和简化。

我们还将看到管理云数据服务的脚手架显著改进。Apache Airflow 已成为第一个真正面向云的数据编排平台，但我们正处于重大增强的边缘。Airflow 将以其巨大的 mindshare 为基础增强能力。Dagster 和 Prefect 等新进入者将通过从头重建编排架构来竞争。

下一代数据编排平台将具有增强的数据集成和数据感知能力。编排平台将集成数据目录和血缘，在此过程中变得更加数据感知。此外，编排平台将构建 IaC（基础设施即代码）能力（类似 Terraform）和代码部署特性（如 GitHub Actions 和 Jenkins）。这将允许工程师编写管道，然后将其传递给编排平台以自动构建、测试、部署和监控。工程师将能够将基础设施规范直接写入管道；缺失的基础设施和服务（如 Snowflake 数据库、Databricks 集群和 Amazon Kinesis 流）将在管道首次运行时部署。

我们还将看到实时数据领域的显著增强——例如流式管道和能够摄取和查询流式数据的数据库。过去，构建流式 DAG 是极其复杂的过程，具有很高的持续运营负担（见第 8 章）。Apache Pulsar 等工具指明了通往未来的道路，其中可以使用相对简单的代码部署具有复杂转换的流式 DAG。我们已经看到托管流式处理器（如 Amazon Kinesis Data Analytics 和 Google Cloud Dataflow）的出现，但我们将看到新一代编排工具来管理这些服务、将它们拼接在一起并监控它们。我们在「实时数据栈」中讨论实时数据。

这种增强的抽象对数据工程师意味着什么？正如我们在本章已经论证的，数据工程师的角色不会消失，但将显著演变。相比之下，更复杂的移动操作系统和框架并没有消除移动应用开发者。相反，移动应用开发者现在可以专注于构建更高质量、更复杂的应用。我们预计随着云规模数据 OS 范式增加各种应用和系统之间的互操作性和简化，数据工程将出现类似发展。

## 「企业级」数据工程

数据工具的日益简化和最佳实践的出现与文档化意味着数据工程将变得更「企业级」。这会让许多读者强烈反感。对某些人来说，「企业」一词会唤起卡夫卡式的噩梦——穿着过度浆洗的蓝衬衫和卡其裤的无面孔委员会、无尽的繁文缛节、以及时间表不断拖延、预算不断膨胀的瀑布式管理开发项目。简而言之，你们中的一些人读到「企业」会想象一个创新走向死亡的无灵魂之地。

幸运的是，这不是我们所说的；我们指的是大公司在数据方面做的一些好事——管理、运营、治理和其他「无聊」内容。我们目前正生活在「企业级」数据管理工具的黄金时代。曾经专属于大型组织的技术和实践正在向下游渗透。大数据和流式数据曾经困难的部分现在已 largely 被抽象掉，焦点转向易用性、互操作性和其他改进。

这允许从事新工具工作的数据工程师在数据管理、DataOps 和数据工程所有其他暗流的抽象中找到机会。数据工程师将变得「企业级」。说到这……

## 职位与职责将演变

虽然数据工程生命周期不会很快消失，但软件工程、数据工程、数据科学和 ML 工程之间的边界日益模糊。事实上，像作者一样，许多数据科学家通过有机过程转变为数据工程师；被 tasked 做「数据科学」但缺乏完成工作的工具，他们承担了设计和构建系统以服务数据工程生命周期的任务。

随着简化向上移动，数据科学家将花更少的时间收集和整理数据。但这种趋势将超越数据科学家。简化也意味着数据工程师将花更少时间在数据工程生命周期的底层任务上（管理服务器、配置等），「企业级」数据工程将变得更加普遍。

随着数据更紧密地嵌入每个企业的流程，新角色将在数据和算法领域涌现。一种可能性是介于 ML 工程和数据工程之间的角色。随着 ML 工具集变得更易用、托管云 ML 服务能力增强，ML 正从临时探索和模型开发转向成为运营学科。

跨越这一鸿沟的新 ML 工程师将了解算法、ML 技术、模型优化、模型监控和数据监控。然而，他们的主要角色将是创建或利用自动训练模型、监控性能并为已充分理解的模型类型运营化完整 ML 流程的系统。他们还将监控数据管道和质量，与当前数据工程领域重叠。ML 工程师将变得更加专业化，从事更接近研究、不太充分理解的模型类型。

职位可能演变的另一个领域是软件工程与数据工程的交叉点。**数据应用（data application）**——融合传统软件应用与分析——将推动这一趋势。软件工程师将需要对数据工程有更深入的理解。他们将发展流式、数据管道、数据建模和数据质量等方面的专长。我们将超越现在普遍的「扔过墙」方法。数据工程师将融入应用开发团队，软件开发人员将获得数据工程技能。应用后端系统与数据工程工具之间的边界也将降低，通过流式和事件驱动架构实现深度集成。

## 超越现代数据栈，迈向实时数据栈

我们坦率地说：**现代数据栈（MDS）**并不那么现代。我们赞扬 MDS 为大众带来了强大的数据工具选择、降低了价格并赋予数据分析师掌控其数据栈的能力。ELT、云数据仓库和 SaaS 数据管道抽象的兴起 certainly 改变了许多公司的游戏规则，为 BI、分析和数据科学开辟了新能力。

话虽如此，MDS 基本上是使用现代云和 SaaS 技术重新包装的旧数据仓库实践；因为 MDS 围绕云数据仓库范式构建，与下一代实时数据应用的潜力相比，它有一些严重限制。从我们的观点来看，世界正超越基于数据仓库的面向内部分析和数据科学，向用下一代实时数据库为整个业务和应用提供实时能力演进。

什么在推动这一演进？在许多情况下，分析（BI 和运营分析）将被自动化取代。目前，大多数仪表板和报告回答关于什么和何时的问题。问问自己：「如果我问什么或何时的问题，下一步采取什么行动？」如果行动是重复的，它就是自动化的候选。当你可以根据事件发生时自动执行行动时，为什么还要看报告来决定是否采取行动？

而且这远远不止于此。为什么使用 TikTok、Uber、Google 或 DoorDash 这样的产品感觉像魔法？虽然对你来说似乎只是点击按钮观看短视频、叫车或订餐、或找到搜索结果，但幕后发生了很多事情。这些产品是真正实时数据应用的例子，在点击按钮时交付你需要的行动，同时在幕后以极小的延迟执行极其复杂的数据处理和 ML。

目前，这种复杂程度被锁定在大型科技公司的定制技术背后，但这种复杂性和能力正在民主化，类似于 MDS 将云规模数据仓库和管道带给大众的方式。数据世界将很快「上线」。

### 实时数据栈

实时技术的这种民主化将引领我们走向 MDS 的继任者：**实时数据栈（live data stack）**将很快可访问和普及。实时数据栈（图 11-1）将使用流式技术将实时分析和 ML 融合到应用中，覆盖从应用源系统到数据处理到 ML 再返回的完整数据生命周期。

正如 MDS 利用云并将本地数据仓库和管道技术带给大众一样，实时数据栈将精英科技公司使用的实时数据应用技术提供给各种规模的公司，作为易用的基于云的产品。这将为创造更好的用户体验和业务价值开辟新的可能性世界。

### 流式管道与实时分析数据库

MDS 将自己限制在将数据视为有界的批处理技术。相比之下，实时数据应用将数据视为无界、连续的流。流式管道和实时分析数据库是促进从 MDS 向实时数据栈迁移的两项核心技术。虽然这些技术已经存在一段时间，但快速成熟的托管云服务将使它们更广泛地部署。

流式技术将在可预见的未来继续经历 extreme 增长。这将与对流式数据业务效用的更清晰关注相结合。迄今为止，流式系统经常被视为昂贵的 novelty 或从 A 到 B 获取数据的 dumb pipe。在未来，流式将从根本上改变组织技术和业务流程；数据架构师和工程师将率先进行这些根本性变革。

**实时分析数据库**支持对此数据的快速摄取和亚秒级查询。这些数据可以与历史数据集丰富或组合。当与流式管道和自动化或能够实时分析的仪表板结合时，将开辟全新的可能性层次。你不再受限于运行缓慢的 ELT 流程、15 分钟更新或其他缓慢移动的部分。数据以连续流移动。随着流式摄取变得更普遍，批处理摄取将越来越少。为什么在数据管道头部创建批处理瓶颈？我们最终将像现在看待拨号调制解调器一样看待批处理摄取。

随着流的兴起，我们预计数据转换将出现「回到未来」时刻。我们将从 ELT（数据库内转换）转向看起来更像 ETL 的东西。我们暂时将其称为**流、转换、加载（STL）**。在流式上下文中，提取是持续、连续的过程。当然，批处理转换不会完全消失。批处理对于模型训练、季度报告等仍然非常有用。但流式转换将成为常态。

虽然数据仓库和数据湖非常适合存储大量数据和执行临时查询，但它们对低延迟数据摄取或快速移动数据的查询优化不佳。实时数据栈将由为流式目的构建的 OLAP 数据库提供动力。今天，Druid、ClickHouse、Rockset 和 Firebolt 等数据库正在引领下一代数据应用后端的动力。我们预计流式技术将继续快速演进，新技术将激增。

### 暗物质数据与……电子表格的兴起？！

我们讨论了快速移动的数据以及随着应用、数据和 ML 更紧密地协作，反馈循环将如何缩小。本节可能看起来奇怪，但我们需要解决当今数据世界中广泛被忽视的问题，尤其是被工程师忽视的。

最广泛使用的数据平台是什么？是 humble 的电子表格。根据你读到的估计，电子表格的用户群在 7 亿到 20 亿人之间。电子表格是数据世界的暗物质。大量数据分析在电子表格中运行，从未进入我们在本书中描述的复杂数据系统。在许多组织中，电子表格处理财务报告、供应链分析，甚至 CRM。

本质上，电子表格是什么？电子表格是支持复杂分析的交互式数据应用。与 pandas（Python 数据分析库）等纯基于代码的工具不同，电子表格可供从只知道如何打开文件和查看报告的人到可以编写复杂过程式数据处理脚本的高级用户等各种用户使用。到目前为止，BI 工具未能为数据库带来可比的交互性。与 UI 交互的用户通常仅限于在特定护栏内切片和切块数据，而不是通用可编程分析。

我们预测将出现一类新工具，将电子表格的交互式分析能力与云 OLAP 系统的后端能力相结合。事实上，一些候选者已经在竞争中。这一产品类别的最终赢家可能继续使用电子表格范式，或可能定义全新的与数据交互的界面 idiom。

## 小结

感谢你与我们同行数据工程之旅！我们穿越了良好架构、数据工程生命周期的阶段和安全最佳实践。我们讨论了在领域继续以非凡速度变化时选择技术的策略。在本章中，我们阐述了对近期和中期未来的大胆推测。

我们预测的某些方面建立在相对稳固的基础上。托管工具的简化和「企业级」数据工程的兴起在我们撰写本书的过程中日复一日地进行。其他预测本质上更具推测性；我们看到了新兴实时数据栈的迹象，但这需要对个人工程师和雇用他们的组织都进行重大范式转变。Perhaps 实时数据的趋势将再次停滞，大多数公司继续专注于基本批处理。当然，存在我们完全未能识别的其他趋势。技术的演进涉及技术和文化的复杂相互作用。两者都不可预测。

数据工程是一个广阔的话题；虽然我们无法在个别领域深入技术细节，但我们希望我们成功创建了一种旅行指南，将帮助当前数据工程师、未来数据工程师以及在该领域附近工作的人在这个处于 flux（变化）中的领域找到方向。我们建议你继续自行探索。当你在本书中发现有趣的话题和想法时，作为社区的一部分继续对话。识别能够帮助你发现流行技术和实践的优缺点和陷阱的领域专家。广泛阅读最新书籍、博客文章和论文。参加 meetup 并聆听演讲。提问并分享你自己的专长。关注供应商公告以跟上最新发展，对所有声称持健康的怀疑态度。

通过这个过程，你可以选择技术。接下来，你需要采用技术并发展专长，perhaps 作为个人贡献者，perhaps 在团队中作为领导，perhaps 跨越整个技术组织。当你这样做时，不要忽视数据工程的更大目标。专注于生命周期、服务你的客户（内部和外部）、你的业务、服务以及你的更大目标。

关于未来，你们中的许多人将在决定下一步中发挥作用。技术趋势不仅由创建底层技术的人定义，也由采用它并善用它的人定义。成功的工具使用与工具创建同样关键。寻找应用实时技术的机会，这些技术将改善用户体验、创造价值并定义全新类型的应用。正是这种实际应用将使实时数据栈成为新的行业标准；perhaps 我们未能识别的其他新技术趋势将赢得胜利。

最后，我们祝愿你拥有激动人心的职业生涯！我们选择在数据工程领域工作、咨询和撰写本书，不仅仅是因为它很流行，而是因为它令人着迷。我们希望我们成功地传达了我们在这一领域工作所发现的一点乐趣。
