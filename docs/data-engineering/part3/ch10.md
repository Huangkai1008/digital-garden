# 第10章 安全与隐私

既然你已了解数据工程生命周期，我们想重申安全的重要性，并分享一些可以纳入日常工作的直接实践。安全对数据工程实践至关重要。这应该是显而易见的，但我们 constantly 惊讶于数据工程师多么 often 将安全视为事后考虑。我们相信安全是数据工程师在工作的每个方面和生命周期的每个阶段首先需要考虑的事情。你每天处理敏感数据、信息和访问。你的组织、客户和业务伙伴期望这些宝贵资产得到最谨慎和关切的处理。一次安全漏洞或数据泄露可能让你的业务陷入绝境；如果是你的过错，你的职业和声誉将毁于一旦。

安全是隐私的关键要素。隐私长期以来对企业在信息技术领域的信任至关重要；工程师直接或间接处理与人们私生活相关的数据。这包括财务信息、私人通信数据（电子邮件、短信、电话）、病史、教育记录和工作经历。泄露或滥用这些信息的公司可能在漏洞曝光时成为众矢之的。

隐私日益成为重大法律问题。例如，《家庭教育权利与隐私法》（FERPA）于 1970 年代在美国生效；《健康保险可携性与责任法》（HIPAA）于 1990 年代跟进；GDPR 于 2010 年代中期在欧洲通过。美国多项隐私法案已通过或即将通过。这只是与隐私相关法规的一小部分（我们相信只是开始）。尽管如此，违反这些法律中任何一项的处罚可能对业务造成重大甚至毁灭性影响。由于数据系统已融入教育、医疗和商业的结构，数据工程师处理与这些法律相关的敏感数据。

数据工程师的确切安全和隐私职责因组织而异。在小型创业公司，数据工程师可能兼任数据安全工程师。大型科技公司将拥有大批安全工程师和安全研究人员。即使在这种情况下，数据工程师通常也能识别自己团队和系统中的安全实践和技术漏洞，可以与专职安全人员协作报告和缓解。

由于安全和隐私对数据工程至关重要（安全是暗流之一），我们希望花更多时间涵盖安全和隐私。在本章中，我们列出数据工程师在安全方面应考虑的一些事项，特别是在人、流程和技术（按此顺序）方面。这不是完整列表，但根据我们的经验列出了我们希望改进的主要事项。

## 人

安全和隐私最薄弱的环节是你。安全 often 在人为层面被破坏，因此假设自己始终是目标。机器人或人类攻击者随时试图渗透你的敏感凭证和信息。这是我们的现实，不会消失。对你在线和离线所做的一切采取防御姿态。运用消极思维的力量，始终保持警惕。

### 消极思维的力量

在一个痴迷于积极思维的世界里，消极思维令人反感。然而，美国外科医生 Atul Gawande 在 2007 年《纽约时报》的专栏文章中 precisely 讨论了这一主题。他的核心论点是，积极思维可能使我们看不到恐怖袭击或医疗紧急情况的可能性，并阻碍准备。消极思维让我们考虑灾难场景并采取预防措施。

数据工程师应积极思考数据利用场景，仅在确有下游需求时收集敏感数据。保护隐私和敏感数据的最佳方式是首先避免摄取这些数据。数据工程师应思考他们使用的任何数据管道或存储系统的攻击和泄露场景。在决定安全策略时，确保你的方法提供真正的安全，而不仅仅是安全的幻觉。

### 始终保持警惕

当有人索要你的凭证时务必谨慎。在怀疑时——被索要凭证时你应极度怀疑——暂停并征求同事和朋友的第二意见。与他人确认请求确实合法。快速聊天或电话比通过电子邮件点击触发的勒索软件攻击便宜。当被索要凭证、敏感数据或机密信息时，不要轻易相信任何人，包括同事。

你也是尊重隐私和道德的第一道防线。你对被要求收集的敏感数据感到不安吗？你对项目中数据处理方式有道德疑问吗？向同事和领导提出你的关切。确保你的工作既合法合规又合乎道德。

## 流程

当人们遵循常规安全流程时，安全成为工作的一部分。使安全成为习惯，定期实践真正安全，践行最小权限原则，理解云中的责任共担模型。

### 安全剧场 versus 安全习惯

在与企业客户合作时，我们看到普遍关注合规（内部规则、法律、标准机构的建议），但对 potentially 糟糕场景的关注不足。不幸的是，这创造了安全的幻觉，但 often 留下明显的漏洞，只需几分钟反思就能发现。

安全需要足够简单有效，才能在整个组织中成为习惯。我们惊讶于有多少公司的安全政策长达数百页却没人阅读，年度安全政策审查人们立即忘记，都是为了勾选安全审计的框。这是**安全剧场（security theater）**，即安全仅满足合规字面要求（SOC-2、ISO 27001 及相关），而无真正承诺。

相反，追求真正、习惯性安全的精神；将安全思维融入文化。安全不需要复杂。例如，在我们公司，我们至少每月运行一次安全培训和政策审查，将其融入团队 DNA，并相互更新我们可以改进的安全实践。安全绝不能成为数据团队的事后考虑。每个人都有责任、都有角色要扮演。它必须是你和与你共事的每个人的优先事项。

### 主动安全

回到消极思维的概念，**主动安全（active security）**意味着在动态变化的世界中思考和研究安全威胁。与其 simply 部署预定的模拟钓鱼攻击，你可以通过研究成功的钓鱼攻击并思考组织安全漏洞来采取主动安全姿态。与其 simply 采用标准合规清单，你可以思考组织特有的内部漏洞以及员工可能泄露或滥用私人信息的动机。

### 最小权限原则

**最小权限原则（principle of least privilege）**意味着人或系统应仅被给予完成当前任务所需的权限和数据，不多不少。Often，我们在云中看到一种反模式：普通用户被给予对所有东西的管理员访问权限，而该人可能只需要少量 IAM 角色来完成工作。给予某人 carte blanche 管理员访问权限是巨大错误，根据最小权限原则绝不应发生。

相反，在用户需要时提供他们（或他们所属组）所需的 IAM 角色。当这些角色不再需要时，收回它们。同样的规则适用于服务账户。对人和机器一视同仁：只给予他们完成工作所需的权限和数据，且仅在需要的时间段内。

当然，最小权限原则对隐私也至关重要。你的用户和客户期望人们仅在必要时查看其敏感数据。确保情况如此。对敏感数据实施列、行和单元格级访问控制；考虑屏蔽 PII 和其他敏感数据，并创建仅包含查看者需要访问的信息的视图。某些数据必须保留但应仅在紧急情况下访问。将这些数据置于「破玻璃」流程之后：用户只有在通过紧急审批流程解决问题、查询关键历史信息等后才能访问。工作完成后立即撤销访问。

### 云中的责任共担

安全在云中是**责任共担（shared responsibility）**。云厂商负责确保其数据中心和硬件的物理安全。同时，你负责在云中构建和维护的应用与系统的安全。大多数云安全漏洞仍由最终用户造成，而非云本身。漏洞因意外错误配置、错误、疏忽和粗心而发生。

### 始终备份数据

数据会消失。有时是坏掉的硬盘或服务器；在其他情况下，有人可能意外删除数据库或对象存储桶。恶意行为者也可以锁定数据。勒索软件攻击如今普遍存在。一些保险公司正在减少攻击时的赔付，让你既要恢复数据又要向劫持数据的恶意行为者付款。你需要定期备份数据，既用于灾难恢复，也用于业务连续性——如果勒索软件攻击中你的数据版本受损。此外，定期测试数据备份的恢复。

数据备份并不严格属于安全和隐私实践；它属于更大的灾难预防范畴，但与安全相邻，尤其是在勒索软件攻击时代。

## 示例安全政策

本节呈现关于凭证、设备和敏感信息的示例安全政策。注意我们不过度复杂化；相反，我们给人们一份可以立即采取的实用行动简短清单。

### 保护凭证

不惜一切代价保护你的凭证。以下是凭证的一些基本规则：

- 对所有东西使用**单点登录（SSO）**。尽可能避免密码，以 SSO 为默认。
- 对 SSO 使用**多因素认证（MFA）**。
- 不要共享密码或凭证。这包括客户密码和凭证。如有疑问，咨询你的汇报对象。如果那人也有疑问，继续深挖直到找到答案。
- 警惕钓鱼和诈骗电话。永远不要给出你的密码。（再次强调，优先使用 SSO。）
- 禁用或删除旧凭证。 Preferably 后者。
- 不要把凭证放在代码中。将密钥作为配置处理，永远不要提交到版本控制。尽可能使用密钥管理器。
- 始终践行最小权限原则。永远不要给予超过完成工作所需的访问权限。这适用于云上和本地的所有凭证和权限。

### 保护设备

- 对所有员工使用的设备使用**设备管理**。如果员工离开公司或你的设备丢失，可以远程擦除设备。
- 对所有设备使用多因素认证。
- 使用公司邮箱凭证登录设备。
- 涵盖凭证和行为的所有政策适用于你的设备。
- 将设备视为自己的延伸。不要让分配的设备离开视线。
- 屏幕共享时，注意你正在共享的确切内容以保护敏感信息和通信。仅共享单个文档、浏览器标签或窗口，避免共享整个桌面。仅共享传达观点所需的内容。
- 视频通话时使用「勿扰」模式；这可以防止消息在通话或录制期间出现。

### 软件更新策略

- 看到更新警报时重启 Web 浏览器。
- 在公司和个人设备上运行次要 OS 更新。
- 公司将识别关键的主要 OS 更新并提供指导。
- 不要使用 OS 的 beta 版本。
- 等待新主要 OS 版本发布一周或两周。

这些是安全如何简单有效的若干基本例子。根据公司的安全概况，你可能需要添加更多要求供人们遵循。再次强调，始终记住人是安全最薄弱的环节。

## 技术

在通过人和流程解决安全之后，是时候看看如何利用技术保护你的系统和数据资产。以下是你应优先考虑的一些重要领域。

### 修补和更新系统

软件会过时，安全漏洞 constantly 被发现。为避免暴露你所用工具旧版本中的安全缺陷，始终在新更新可用时修补和更新操作系统和软件。值得庆幸的是，许多 SaaS 和云托管服务自动执行升级和其他维护，无需你干预。要更新你自己的代码和依赖，要么自动化构建，要么在发布和漏洞上设置警报，以便你可以被提示手动执行更新。

### 加密

加密不是万能药。在人为安全漏洞授予凭证访问权限的情况下，它几乎无法保护你。加密是任何尊重安全和隐私的组织的基线要求。它将保护你免受基本攻击，如网络流量拦截。

让我们分别看看静态加密和传输中加密。

**静态加密（encryption at rest）**

确保数据在静态（存储设备上）时加密。公司笔记本电脑应启用全盘加密，以在设备被盗时保护数据。对存储在服务器、文件系统、数据库和云中对象存储的所有数据实施服务器端加密。所有用于归档目的的数据备份也应加密。最后，在适用处纳入应用级加密。

**传输中加密（encryption over the wire）**

传输中加密现在是当前协议的默认。例如，现代云 API 通常需要 HTTPS。数据工程师应始终注意密钥如何管理；糟糕的密钥处理是数据泄露的重要来源。此外，如果存储桶权限对公众开放，HTTPS 无法保护数据，这是过去十年几起数据丑闻的另一个原因。

工程师还应了解旧协议的安全限制。例如，FTP 在公共网络上 simply 不安全。虽然当数据已经公开时这可能似乎不是问题，但 FTP 容易受到中间人攻击，攻击者拦截下载的数据并在到达客户端之前更改它。最好 simply 避免 FTP。

确保一切在传输中加密，即使使用遗留协议。如有疑问，使用内置加密的稳健技术。

### 日志、监控和告警

黑客和恶意行为者通常不会宣布他们正在渗透你的系统。大多数公司在安全事件发生很久之后才发现。DataOps 的一部分是观察、检测和告警事件。作为数据工程师，你应设置自动化监控、日志和告警，以在系统中发生异常事件时知晓。如有可能，设置自动异常检测。

以下是你应监控的一些领域：

**访问**

谁在何时何地访问什么？授予了什么新访问权限？当前用户是否有奇怪模式可能表明其账户被盗用，如尝试访问他们通常不访问或不应有权限访问的系统？你是否看到新的未识别用户访问你的系统？务必定期梳理访问日志、用户及其角色，确保一切正常。

**资源**

监控磁盘、CPU、内存和 I/O 的异常模式。你的资源是否突然变化？如果是，这可能表明安全漏洞。

**计费**

特别是对于 SaaS 和云托管服务，你需要监督成本。设置预算警报以确保支出在预期范围内。如果计费出现意外激增，这可能表明某人或某物正在将你的资源用于恶意目的。

**过度权限**

越来越多的供应商提供监控用户或服务账户在一段时间内未使用的权限的工具。这些工具通常可以配置为在指定时间后自动提醒管理员或移除权限。

例如，假设某分析师六个月未访问 Redshift。可以移除这些权限，关闭潜在安全漏洞。如果分析师将来需要访问 Redshift，他们可以提交工单恢复权限。

最好在监控中结合这些领域，以获得资源、访问和计费概况的横截面视图。我们建议为数据团队中的每个人设置仪表板以查看监控并在出现异常时接收告警。将此与有效的事件响应计划结合，以在发生安全漏洞时进行管理，并定期演练计划以便做好准备。

### 网络访问

我们 often 看到数据工程师在网络访问方面做相当疯狂的事情。在几个实例中，我们见过公开可用的 Amazon S3 存储桶存放大量敏感数据。我们还目睹过 Amazon EC2 实例的入站 SSH 访问对 0.0.0.0/0（所有 IP）开放，或数据库对公共互联网上的所有入站请求开放访问。这些只是糟糕网络安全实践的少数例子。

原则上，网络安全应留给公司的安全专家。（实践中，在小公司你可能需要承担网络安全的重大责任。）作为数据工程师，你会如此 often 遇到数据库、对象存储和服务器，以至于你至少应了解可以采取的简单措施，以确保符合良好的网络访问实践。了解哪些 IP 和端口开放、对谁开放以及为什么。允许将访问这些端口的系统和用户的入站 IP 地址（即白名单 IP），并避免出于任何原因广泛开放连接。访问云或 SaaS 工具时，使用加密连接。例如，不要从咖啡店使用未加密的网站。

此外，虽然本书几乎完全聚焦于在云中运行工作负载，我们在此简要提及本地托管服务器。回顾第 3 章，我们讨论了强化边界与零信任安全的区别。云通常更接近零信任安全——每个操作都需要身份验证。我们相信云对大多数组织来说是更安全的选择，因为它实施零信任实践并允许公司利用公有云雇用的安全工程师大军。

然而，有时强化边界安全仍然有意义；我们知道核导弹发射井是气隙的（未连接到任何网络），这让我们有些安慰。气隙服务器是强化安全边界的终极例子。只需记住，即使在本地，气隙服务器也容易受到人为安全失败的影响。

### 底层数据工程的安全

对于在数据存储和处理系统内部工作的工程师，考虑每个元素的安全影响至关重要。任何软件库、存储系统或计算节点都是潜在的安全漏洞。晦涩日志库中的缺陷可能允许攻击者绕过访问控制或加密。甚至 CPU 架构和微码都代表潜在漏洞；当敏感数据在内存或 CPU 缓存中静态存储时可能容易受到攻击。链条中的任何环节都不能想当然。

当然，本书主要关于高层数据工程——将工具拼接在一起以处理整个生命周期。因此，我们将深入血腥技术细节留给你。

**内部安全研究**

我们在「流程」中讨论了主动安全的概念。我们还强烈建议对技术采用主动安全方法。具体而言，这意味着每个技术员工都应思考安全问题。为什么这很重要？每个技术贡献者都会发展技术专长领域。即使你的公司雇用大批安全研究人员，数据工程师也会对其职责范围内的特定数据系统和云服务非常熟悉。特定技术的专家处于识别该技术中安全漏洞的有利位置。

鼓励每个数据工程师积极参与安全。当他们识别系统中的潜在安全风险时，他们应思考缓解措施并在部署这些措施方面发挥积极作用。

## 小结

安全需要成为思维和行动的习惯；像对待钱包或智能手机一样对待数据。虽然你可能不会负责公司的安全，但了解基本安全实践并将安全放在首位将有助于减少组织数据安全漏洞的风险。
