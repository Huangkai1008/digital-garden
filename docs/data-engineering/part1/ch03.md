# 第3章 设计良好的数据架构

良好的数据架构在数据生命周期和暗流的每一步提供无缝能力。我们先定义数据架构，然后讨论组件和考量，接着涉及特定批处理模式（数据仓库、数据湖）、流式模式以及统一批处理和流式的模式。贯穿始终，我们强调利用云的能力交付可扩展性、可用性和可靠性。

## 什么是数据架构？

成功的数据工程建立在坚实的数据架构之上。本章旨在回顾几种流行的架构方法和框架，然后提出我们对「良好」数据架构的 opinionated 定义。

**企业架构（enterprise architecture）**包含业务、技术、应用和数据等子集。数据架构是企业架构的子集。

我们对企业架构的定义：

> 企业架构是通过仔细评估权衡做出的灵活且可逆的决策来支持企业变革的系统设计。

**可逆决策**至关重要：世界不断变化，预测未来不可能。可逆决策允许你在世界变化和收集新信息时调整方向。Jeff Bezos 提出单向门和双向门的概念——单向门几乎无法逆转，双向门易于逆转。

我们对**数据架构（data architecture）**的定义：

> 数据架构是通过仔细评估权衡做出的灵活且可逆的决策来支持企业不断演进的数据需求的系统设计。

## 「良好」数据架构

> 永远不要追求最佳架构，而是最不坏的架构。
>
> — Mark Richards 和 Neal Ford

**良好数据架构**以通用、广泛可复用的构建块满足业务需求，同时保持灵活性并做出适当权衡。敏捷是良好数据架构的基础；它承认世界是流动的。良好数据架构灵活且易于维护，随业务变化和可能在未来释放更多价值的新技术实践而演进。

不良数据架构紧密耦合、僵化、过度集中，或使用错误工具，阻碍开发和变更管理。

## 良好数据架构的原则

1. **明智选择通用组件**：选择可在组织内广泛使用的通用组件和实践
2. **为失败做计划**：一切都会失败，始终如此。考虑可用性、可靠性、恢复时间目标（RTO）、恢复点目标（RPO）
3. **为可扩展性架构**：系统应能扩展以处理大量数据，也能缩减以降低成本
4. **架构即领导力**：数据架构师负责技术决策，通过有效领导和培训传播这些选择
5. **持续架构**：架构师不断设计新事物以响应业务和技术变化
6. **构建松耦合系统**：使团队能够在不依赖其他团队的情况下测试、部署和变更系统
7. **做出可逆决策**：降低决策风险，允许快速迭代
8. **优先考虑安全**：安全必须是首要考虑
9. **拥抱 FinOps**：将财务责任和业务价值完全运营化

## 主要架构概念

- **域与服务**：按业务域组织
- **分布式系统与可扩展性**：设计容错
- **紧耦合与松耦合**：单体与微服务
- **用户访问**：单租户与多租户
- **事件驱动架构**：基于事件的系统设计
- **棕地 versus 绿地项目**：现有系统与全新项目

## 数据架构示例与类型

| 类型 | 描述 |
|------|------|
| 数据仓库 | 面向分析的结构化数据存储，Kimball/Inmon 建模 |
| 数据湖 | 原始和半结构化数据的存储，支持多种工作负载 |
| 数据湖屋 | 结合数据湖与数据仓库能力 |
| 现代数据栈 | 现成开源和第三方产品组合 |
| Lambda 架构 | 批处理与流式双路径 |
| Kappa 架构 | 纯流式处理 |
| 数据网格 | 去中心化、域驱动的数据所有权 |

## 小结

良好数据架构是活生生的东西，永不完成。变化和演进是数据架构意义和目的的核心。数据工程生命周期的暗流——安全、数据管理、DataOps、数据架构、编排和软件工程——构成任何数据成熟度阶段公司良好数据架构的基础。
