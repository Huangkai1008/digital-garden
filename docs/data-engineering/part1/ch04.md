# 第4章 跨生命周期的技术选型

当今数据工程面临「富足的尴尬」——我们不缺解决各类数据问题的技术。数据技术以几乎各种方式提供：开源、托管开源、专有软件、专有服务等。然而，人们很容易追逐尖端技术而忽视数据工程的核心目的：设计稳健可靠的系统，使数据贯穿完整生命周期并根据最终用户需求提供服务。

第 3 章讨论了「良好」数据架构及其重要性。我们现在解释如何选择正确的技术来服务这一架构。**架构是战略性的；工具是战术性的**。架构是高层设计、路线图和蓝图；工具用于使架构成为现实。我们强烈建议在确定架构之前不要选择技术。**架构优先，技术其次**。

## 技术选型考量

- **团队规模与能力**：小团队应尽可能使用托管和 SaaS 工具，避免 cargo-cult 工程
- **上市速度**：完美是好的敌人。尽早且频繁交付价值
- **互操作性**：确保技术能与生态系统中的其他技术交互
- **成本优化与业务价值**：TCO、机会成本、FinOps
- **今天 versus 未来**：不变技术与过渡技术
- **位置**：本地、云、混合云、多云
- **自建 versus 购买**：开源、专有、托管
- **单体 versus 模块化**：避免分布式单体
- **无服务器 versus 服务器**：根据用例选择
- **优化、性能与基准战**：谨慎对待营销基准

## 总拥有成本（TCO）与机会成本

**总拥有成本（TCO）**包括直接和间接成本。**资本支出（capex）**需要前期投资；**运营支出（opex）**随时间分摊，按需付费。我们建议数据工程师采用以云和灵活按需技术为中心的 opex 优先方法。

**总机会成本（TOCO）**是选择某技术、架构或流程时产生的机会损失成本。避免「熊夹」——容易进入、极难逃脱的不灵活技术。

## 不变 versus 过渡技术

**不变技术**可能 underpins 云或经受时间考验的语言和范式，如对象存储、SQL、bash。**过渡技术**来来去去，通常轨迹是大量炒作、流行度 meteoric 增长、缓慢衰落。

我们建议每两年评估一次工具，尽可能找到数据工程生命周期中的不变技术，以其为基础，在不变技术周围构建过渡工具。

## 位置：本地、云、混合、多云

- **本地（on premises）**：公司拥有硬件，承担运维责任
- **云（cloud）**：从云提供商租用硬件和托管服务，动态扩展
- **混合云（hybrid cloud）**：本地与云结合
- **多云（multicloud）**：使用多个云提供商

## 自建 versus 购买

开源软件（OSS）提供灵活性和社区支持；专有围墙花园提供集成和托管。根据团队能力、时间和业务价值做决策。

## 小结

选择数据技术时，评估团队能力、上市速度、互操作性、成本、技术生命周期和暗流影响。架构优先，技术其次。每两年重新评估技术选择，围绕不变技术构建。
