# 第4章 微服务通信风格

由于人们往往在没有首先考虑他们可能想要的不同类型的通信的情况下就倾向于选择的技术方法，正确实现微服务之间的通信对许多人来说是有问题的。在本章中，我将尝试梳理不同的通信风格，帮助你理解各自的优缺点，以及哪种方法最适合你的问题空间。

我们将研究同步阻塞和异步非阻塞通信机制，以及比较请求-响应协作与事件驱动协作。

到本章结束时，你应该对可用的不同选项有更好的准备，并拥有当我们开始研究后续章节中更详细的实现关注点时将有所帮助的基础知识。

## 从进程内到进程间

好的，让我们先把简单的东西解决掉——或者至少我希望是简单的东西。即，跨网络的不同进程之间的调用（进程间）与单个进程内的调用（进程内）非常不同。在某种程度上，我们可以忽略这种区别。例如，很容易想到一个对象对另一个对象进行方法调用，然后只是将这种交互映射到两个微服务通过网络通信。撇开微服务不仅仅是对象这一事实，这种思考可能会给我们带来很多麻烦。

让我们看看其中一些差异以及它们可能如何改变你对微服务之间交互的思考。

### 性能

进程内调用的性能与进程间调用的性能 fundamentally 不同。当我进行进程内调用时，底层编译器和运行时可以进行大量优化来减少调用的影响，包括内联调用，就好像根本没有调用一样。进程间调用不可能进行此类优化。必须发送数据包。与进程内调用的开销相比，期望进程间调用的开销是显著的。前者是非常可测量的——在数据中心仅往返单个数据包就以毫秒计——而进行方法调用的开销是你不需要担心的事情。

这往往会导致你想重新思考 API。在进程内有意义的 API 在进程间情况下可能没有意义。我可以在进程内跨 API 边界进行一千次调用而不担心。我想在两个微服务之间进行一千次网络调用吗？也许不想。

当我将参数传递给方法时，我传入的数据结构通常不会移动——更可能的是我传递指向内存位置的指针。将对象或数据结构传递给另一个方法不需要分配更多内存来复制数据。

另一方面，当通过网络在微服务之间进行调用时，数据实际上必须被序列化为某种可以在网络上传输的形式。然后数据需要被发送并在另一端反序列化。因此，我们可能需要对在进程之间发送的有效负载大小更加注意。你上次意识到在进程内传递的数据结构的大小时是什么时候？现实是你可能不需要知道；现在你需要。这可能导致你减少发送或接收的数据量（如果我们考虑信息隐藏，也许不是坏事），选择更高效的序列化机制，或者甚至将数据卸载到文件系统并传递对该文件位置的引用。

这些差异可能不会立即给你带来问题，但你肯定需要意识到它们。我见过很多试图向开发人员隐藏甚至正在进行网络调用的事实的尝试。我们创建抽象来隐藏细节的愿望是我们能够更有效地做更多事情的重要组成部分，但有时我们创建的抽象隐藏了太多。开发人员需要意识到他们正在做会导致网络调用的事情；否则，如果由于编写代码的开发人员不可见的奇怪服务间交互而导致你在下游出现一些令人讨厌的性能瓶颈，你不应该感到惊讶。

### 更改接口

当我们考虑进程内接口的更改时，推出更改的行为是直接的。实现接口的代码和调用接口的代码都打包在同一个进程中。事实上，如果我使用具有重构能力的 IDE 更改方法签名，IDE 本身通常会自动重构对此更改方法的调用。推出这样的更改可以以原子方式进行——接口的两端都打包在单个进程中。

然而，对于微服务之间的通信，暴露接口的微服务和使用该接口的消费微服务是分别可部署的微服务。当对微服务接口进行向后不兼容的更改时，我们要么需要与消费者进行同步部署，确保它们已更新以使用新接口，要么找到某种方式来分阶段推出新的微服务契约。我们将在本章后面更详细地探讨这个概念。

### 错误处理

在进程内，如果我调用一个方法，错误的性质往往相当直接。简单地说，错误要么是预期的且容易处理，要么是灾难性的以至于我们只是将错误传播到调用栈。错误，总的来说，是确定性的。

对于分布式系统，错误的性质可能不同。你容易受到许多你无法控制的错误的影响。网络超时。下游微服务可能暂时不可用。网络断开连接，容器因消耗太多内存而被杀死，在极端情况下，你的数据中心的部分可能会着火。[^1]

Andrew Tanenbaum 和 Maarten Steen 在他们的《分布式系统》[^2] 一书中，分解了你在查看进程间通信时可以看到的五种故障模式。以下是简化版本：

- **崩溃故障（Crash failure）**：在服务器崩溃之前一切正常。重启！
- **遗漏故障（Omission failure）**：你发送了东西，但没有得到响应。还包括你期望下游微服务正在发送消息（可能包括事件）但它只是停止的情况。
- **时序故障（Timing failure）**：某事发生得太晚（你没有及时得到它），或者某事发生得太早！
- **响应故障（Response failure）**：你得到了响应，但它似乎不对。例如，你请求了订单摘要，但响应中缺少需要的部分信息。
- **任意故障（Arbitrary failure）**：也称为拜占庭故障（Byzantine failure），这是当某事出了问题，但参与者无法就故障是否发生（或为什么）达成一致。正如听起来的那样，这到处都是糟糕的时刻。

这些错误中的许多往往是暂时性的——它们是可能会消失的短期问题。考虑我们向微服务发送请求但没有收到回复的情况（一种遗漏故障）。这可能意味着下游微服务根本没有收到请求，所以我们需要再次发送。其他问题不能轻易处理，可能需要人工操作员介入。因此，以允许客户端采取适当行动的方式返回错误的更丰富的语义可能变得重要。

HTTP 是理解这一重要性的协议的一个例子。每个 HTTP 响应都有一个代码，400 和 500 系列代码保留给错误。400 系列错误代码是请求错误——本质上，下游服务告诉客户端原始请求有问题。因此，你可能应该放弃——例如，重试 404 Not Found 有什么意义？500 系列响应代码与下游问题有关，其中一部分向客户端表明问题可能是暂时的。例如，503 Service Unavailable 表明下游服务器无法处理请求，但这可能是暂时状态，在这种情况下，上游客户端可能决定重试请求。另一方面，如果客户端收到 501 Not Implemented 响应，重试可能不会有太大帮助。

无论你是否选择基于 HTTP 的协议进行微服务之间的通信，如果你对错误的性质有一套丰富的语义，你将使客户端更容易执行补偿操作，这反过来应该帮助你构建更健壮的系统。

## 进程间通信的技术：选择太多

在一个我们有太多选择和太少时间的世界里，显而易见的事情就是忽略东西。
—Seth Godin

我们可用于进程间通信的技术范围是巨大的。因此，我们往往被选择压得喘不过气来。我经常发现人们倾向于他们熟悉的技术，或者也许只是从会议上学到的最新热门技术。这样做的问题是，当你采用特定的技术选择时，你往往也采用随之而来的一套想法和约束。这些约束可能不适合你——技术背后的心态可能实际上与你试图解决的问题不一致。

如果你试图构建一个网站，像 Angular 或 React 这样的单页应用技术是一个糟糕的选择。同样，尝试将 Kafka 用于请求-响应真的不是一个好主意，因为它是为更基于事件的交互设计的（我们马上会涉及的主题）。然而，我一次又一次地看到技术在错误的地方被使用。人们选择闪亮的新技术（比如微服务！）而不考虑它是否真正适合他们的问题。

因此，当涉及到我们可用于微服务之间通信的令人眼花缭乱的技术阵列时，我认为首先谈论你想要的通信风格很重要，然后才寻找实现该风格的合适技术。考虑到这一点，让我们看看我多年来一直在使用的模型，以帮助区分微服务到微服务通信的不同方法，这反过来可以帮助你过滤你想要查看的技术选项。

## 微服务通信的风格

在图 4-1 中，我们看到了我用于思考不同通信风格的模型概述。这个模型并不意味着完全详尽（我不是试图在这里呈现进程间通信的大统一理论），但它为考虑微服务架构最广泛使用的不同通信风格提供了很好的高层次概述。

```
┌─────────────────────────────────────────────────────────────────────────┐
│  Synchronous Blocking  │  Asynchronous Nonblocking                       │
│  (HTTP, gRPC)          │  (Message queues, Kafka)                       │
├───────────────────────┼─────────────────────────────────────────────────┤
│  Request-Response     │  Event-Driven          │  Common Data           │
└───────────────────────┴───────────────────────┴─────────────────────────┘
```

**图 4-1. 微服务间通信的不同风格以及实现技术的例子**

我们将在不久后更详细地查看此模型的不同元素，但首先我想简要概述它们：

- **同步阻塞（Synchronous blocking）**：微服务向另一个微服务发出某种调用，并阻塞操作等待响应。
- **异步非阻塞（Asynchronous nonblocking）**：发出调用的微服务能够继续进行任何其他处理，无论调用是否被接收。
- **请求-响应（Request-response）**：微服务向另一个微服务发送请求，要求做某事。它期望收到告知结果的响应。
- **事件驱动（Event-driven）**：微服务发出事件，其他微服务消费并相应做出反应。发出事件的微服务不知道哪些微服务（如果有的话）消费它发出的事件。
- **公共数据（Common data）**：不常被视为通信风格，微服务通过某种共享数据源进行协作。

当使用此模型帮助团队决定正确方法时，我花大量时间理解他们运营的上下文。他们在可靠通信、可接受延迟和通信量方面的需求都将对技术选择发挥作用。但一般来说，我倾向于从决定请求-响应还是事件驱动的协作风格更适合给定情况开始。如果我正在看请求-响应，那么同步和异步实现仍然可用，所以我还有第二个选择要做。然而，如果选择事件驱动的协作风格，我的实现选择将仅限于非阻塞异步选择。

当选择正确的技术时，还有许多其他考虑因素超出了通信风格——例如，对更低延迟通信的需求、安全相关方面或扩展能力。考虑到你的特定问题空间的要求（和约束），你不太可能做出合理的技术选择。当我们在第 5 章查看技术选项时，我们将讨论其中一些问题。

### 混合搭配

值得注意的是，微服务架构作为一个整体可能具有协作风格的混合，这通常是常态。某些交互只是作为请求-响应有意义，而其他作为事件驱动有意义。事实上，单个微服务实现多种形式的协作是常见的。考虑一个 Order 微服务，它暴露一个请求-响应 API，允许下订单或更改订单，然后在做出这些更改时触发事件。

说了这么多，让我们更详细地看看这些不同的通信风格。

## 模式：同步阻塞

对于同步阻塞调用，微服务向下游进程（可能是另一个微服务）发出某种调用，并阻塞直到调用完成，可能直到收到响应。在图 4-2 中，Order Processor 向 Loyalty 微服务发出调用，告知应将一些积分添加到客户的账户。

```
Order Processor ──────[sync call]──────▶ Loyalty
       │ (blocks)                            │
       │◀──────────[response]────────────────┘
```

**图 4-2. Order Processor 向 Loyalty 微服务发送同步调用，阻塞并等待响应**

通常，同步阻塞调用是等待下游进程响应的调用。这可能是因为调用的结果需要用于某些进一步操作，或者只是因为它想确保调用成功并在不成功时进行某种重试。因此，我看到的几乎每个同步阻塞调用也将构成请求-响应调用，我们马上会看到。

### 优点

阻塞的、同步的调用有某种简单和熟悉的东西。我们中的许多人以 fundamentally 同步的风格学习编程——像脚本一样阅读一段代码，每一行依次执行，下一行代码等待轮到它做某事。你可能使用进程间调用的大多数情况可能是以同步、阻塞风格完成的——例如，在数据库上运行 SQL 查询，或向下游 API 发出 HTTP 请求。

当从不太分布式的架构（如单进程单体）迁移时，在有很多其他全新的东西正在进行时坚持那些熟悉的想法可能是有道理的。

### 缺点

同步调用的主要挑战是发生的固有时间耦合，这是我们在第 2 章简要探讨的主题。当 Order Processor 在前面的例子中向 Loyalty 发出调用时，Loyalty 微服务需要可访问才能让调用工作。如果 Loyalty 微服务不可用，则调用将失败，Order Processor 需要弄清楚要执行哪种补偿操作——这可能涉及立即重试、缓冲调用稍后重试，或者也许完全放弃。

这种耦合是双向的。对于这种集成风格，响应通常通过相同的入站网络连接发送回上游微服务。所以如果 Loyalty 微服务想向 Order Processor 发送响应，但上游实例随后已经死亡，响应将丢失。这里的时间耦合不仅仅是两个微服务之间；它是这两个微服务的两个特定实例之间。

由于调用的发送者正在阻塞并等待下游微服务响应，因此如果下游微服务响应缓慢，或者如果网络延迟有问题，则调用的发送者将被阻塞很长时间等待响应。如果 Loyalty 微服务承受重大负载并对请求响应缓慢，这反过来将导致 Order Processor 响应缓慢。

因此，与使用异步调用相比，同步调用的使用可能使系统更容易受到下游中断引起的级联问题的影响。

### 适用场景

对于简单的微服务架构，我对使用同步、阻塞调用没有太大问题。对许多人来说，它们的熟悉性是在掌握分布式系统时的优势。

对我来说，当你开始有更多调用链时，这些类型的调用开始变得有问题——例如，在图 4-3 中，我们有一个来自 MusicCorp 的例子流程，我们正在检查支付是否有潜在的欺诈活动。Order Processor 调用 Payment 服务收款。Payment 服务反过来想与 Fraud Detection 微服务检查是否应该允许。Fraud Detection 微服务反过来需要从 Customer 微服务获取信息。

```
Order Processor ──▶ Payment ──▶ Fraud Detection ──▶ Customer
```

**图 4-3. 作为订单处理流程一部分检查潜在欺诈行为**

如果所有这些调用都是同步和阻塞的，我们可能面临许多问题。四个涉及的微服务中的任何一个，或它们之间的网络调用中的问题，都可能导致整个操作失败。这完全 aside 于这些长链可能导致重大资源争用的事实。在幕后，Order Processor 可能有一个网络连接打开等待 Payment 的回复。Payment 反过来有一个网络连接打开等待 Fraud Detection 的响应，等等。拥有大量需要保持打开的连接可能对运行系统产生影响——你更可能遇到可用连接耗尽或因此遭受增加的网络拥塞的问题。

为了改善这种情况，我们可以首先重新审视微服务之间的交互。例如，也许我们将 Fraud Detection 的使用从主购买流程中移出，如图 4-4 所示，而是让它在后台运行。如果它发现特定客户有问题，他们的记录会相应更新，这可以在支付过程的早期检查。实际上，这意味着我们正在并行地做一些工作。通过减少调用链的长度，我们将看到操作的整体延迟改善，我们将把我们的一个微服务（Fraud Detection）从购买流程的关键路径中移出，为我们关心的关键操作减少一个依赖。

```
Order Processor ──▶ Payment (Fraud check moved to background)
```

**图 4-4. 将 Fraud Detection 移到后台进程可以减少对调用链长度的关注**

当然，我们也可以用某种非阻塞交互替换阻塞调用的使用，而不改变这里的工作流程，这是我们接下来将探讨的方法。

## 模式：异步非阻塞

对于异步通信，通过网络发出调用的行为不会阻塞发出调用的微服务。它能够继续进行任何其他处理，而不必等待响应。非阻塞异步通信有多种形式，但我们将更详细地研究我在微服务架构中最常看到的三种最常见风格。它们是：

- **通过公共数据通信（Communication through common data）**：上游微服务更改一些公共数据，一个或多个微服务后来使用。
- **请求-响应（Request-response）**：微服务向另一个微服务发送请求，要求它做某事。当请求的操作完成时，无论成功与否，上游微服务都会收到响应。具体来说，上游微服务的任何实例都应该能够处理响应。
- **事件驱动交互（Event-driven interaction）**：微服务广播事件，可以将其视为关于已发生某事的 factual 陈述。其他微服务可以监听他们感兴趣的事件并相应做出反应。

### 优点

对于非阻塞异步通信，发出初始调用的微服务和接收调用的微服务（或多个微服务）在时间上解耦。接收调用的微服务不需要在发出调用时同时可访问。这意味着我们避免了我们在第 2 章讨论的时间解耦关注（参见第 42 页的「关于时间耦合的简要说明」）。

如果被调用触发的功能需要很长时间处理，这种通信风格也有益。让我们回到 MusicCorp 的例子，特别是发送包裹的过程。在图 4-5 中，Order Processor 已收款并决定是时候发货包裹了，所以它向 Warehouse 微服务发出调用。找 CD、从货架上取下、打包和取件的过程可能需要很多小时，甚至可能几天，取决于实际发货流程如何工作。因此，Order Processor 向 Warehouse 发出非阻塞异步调用，并让 Warehouse 稍后回调告知 Order Processor 其进度，这是有道理的。这是异步请求-响应通信的一种形式。

```
Order Processor ──[async]──▶ Warehouse (package & ship)
       │                         │
       │ (continues)             │ (hours/days later)
       │                         └──[callback]──▶ Order Processor
```

**图 4-5. Order Processor 启动打包和发货订单的流程，以异步方式完成**

如果我们尝试用同步阻塞调用来做类似的事情，那么我们将不得不重组 Order Processor 和 Warehouse 之间的交互——Order Processor 打开连接、发送请求、阻塞调用线程中的任何进一步操作，并等待可能数小时或数天的响应，这是不可行的。

### 缺点

相对于阻塞同步通信，非阻塞异步通信的主要缺点是复杂性和选择范围。正如我们已经概述的，有不同风格的异步通信可供选择——哪一种适合你？当我们开始深入研究这些不同风格的通信如何实现时，我们可能会看到令人眼花缭乱的技术列表。

如果异步通信不符合你的计算心理模型，采用异步通信风格一开始将具有挑战性。正如我们将在详细研究各种异步通信风格时进一步探讨的，有很多不同的、有趣的方式可以让你陷入很多麻烦。

### Async/Await 与何时异步仍然是阻塞的

与计算的许多领域一样，我们可以在不同上下文中使用相同的术语来表示非常不同的含义。一种似乎特别流行的编程风格是使用 async/await 等结构来处理潜在异步的数据源，但以阻塞、同步的风格。

在示例 4-1 中，我们看到了一个非常简单的 JavaScript 例子。汇率在一天中频繁波动，我们通过消息代理接收这些。我们定义一个 Promise。一般来说，Promise 是将在未来某个时刻解析为状态的东西。在我们的情况下，我们的 `eurToGbp` 最终将解析为下一个欧元到英镑的汇率。

```javascript
async function f() {
  let eurToGbp = new Promise((resolve, reject) => {
    // code to fetch latest exchange rate between EUR and GBP
    // ...
  });
  var latestRate = await eurToGbp;  // 等待获取最新的 EUR 到 GBP 汇率
  process(latestRate);               // 在 Promise 实现之前不会运行
}
```

**示例 4-1. 以阻塞、同步方式处理潜在异步调用的例子**

当我们使用 `await` 引用 `eurToGbp` 时，我们阻塞直到 `latestRate` 的状态被实现——在 `eurToGbp` 的状态被解析之前不会到达 `process`。[^3]

即使我们的汇率是以异步方式接收的，在这种上下文中使用 `await` 意味着我们阻塞直到 `latestRate` 的状态被解析。所以即使我们用来获取汇率的底层技术可以被认为是异步的（例如，等待汇率），从我们代码的角度来看，这本质上是一种同步、阻塞的交互。

### 适用场景

归根结底，在考虑异步通信是否适合你时，你还必须考虑你想选择哪种类型的异步通信，因为每种类型都有自己的权衡。但一般来说，有一些特定的用例会让我选择某种形式的异步通信。长时间运行的流程是一个明显的候选者，正如我们在图 4-5 中探讨的。此外，你有无法轻易重组的长调用链的情况可能是一个好候选者。我们将在研究三种最常见的异步通信形式——请求-响应调用、事件驱动通信和通过公共数据通信——时更深入地探讨这一点。

## 模式：通过公共数据通信

跨越多种实现的通信风格是通过公共数据通信。当微服务将数据放入定义的位置，然后另一个微服务（或可能是多个微服务）使用该数据时，使用此模式。它可以像微服务将文件放在某个位置一样简单，然后在某个时间点后另一个微服务拾取该文件并对其进行处理。这种集成风格本质上是异步的。

这种风格的一个例子可以在图 4-6 中看到，其中 New Product Importer 创建一个文件，然后由下游的 Inventory 和 Catalog 微服务读取。

```
New Product Importer ──[writes file]──▶ Shared Location
                                              │
                    ┌─────────────────────────┼─────────────────────────┐
                    ▼                         ▼                         
             Inventory              Catalog
```

**图 4-6. 一个微服务写出文件，其他微服务使用**

这种模式在某种程度上是你将看到的最常见的通用进程间通信模式，然而我们有时未能将其视为通信模式——我认为主要是因为进程之间的通信往往如此间接以至于难以发现。

### 实现

要实现此模式，你需要某种形式的数据持久存储。在许多情况下，文件系统可能就足够了。我构建了许多系统，只是定期扫描文件系统，注意到新文件的存在，并相应地做出反应。当然，你也可以使用某种健壮的分布式内存存储。值得注意的是，任何将对此数据采取行动的下游微服务都需要自己的机制来识别有新数据可用——轮询是此问题的常见解决方案。

这种模式的两个常见例子是数据湖（Data Lake）和数据仓库（Data Warehouse）。在这两种情况下，这些解决方案通常旨在帮助处理大量数据，但可以说它们在耦合方面处于光谱的两端。对于数据湖，源以它们认为合适的任何格式上传原始数据，此原始数据的下游消费者被期望知道如何处理信息。对于数据仓库，仓库本身是结构化的数据存储。向数据仓库推送数据的微服务需要知道数据仓库的结构——如果结构以向后不兼容的方式更改，这些生产者将需要更新。

对于数据仓库和数据湖，假设信息流是单向的。一个微服务将数据发布到公共数据存储，下游消费者读取该数据并采取适当的行动。这种单向流可以使推理信息流更容易。更有问题的实现是使用共享数据库，其中多个微服务对同一数据存储进行读写，这是我们在第 2 章探讨公共耦合时讨论的例子——图 4-7 显示了 Order Processor 和 Warehouse 都在更新同一条记录。

```
Order Processor ──┐
                  ├──▶ Shared Order Record (both update)
Warehouse ────────┘
```

**图 4-7. 公共耦合的例子，Order Processor 和 Warehouse 都在更新同一订单记录**

### 优点

这种模式可以使用常用且易于理解的技术非常简单地实现。如果你能读写文件或读写数据库，你就可以使用这种模式。使用普遍且易于理解的技术还支持不同类型系统之间的互操作性，包括较旧的大型机应用或可定制的现成（COTS）软件产品。数据量在这里也不太受关注——如果你一次性发送大量数据，这种模式可以很好地工作。

### 缺点

下游消费微服务通常将通过某种轮询机制意识到有新数据要处理，或者也许通过定期触发的定时作业。这意味着这种机制在低延迟情况下可能不太有用。你当然可以将这种模式与某种其他调用结合，通知下游微服务有新数据可用。例如，我可以将文件写入共享文件系统，然后向感兴趣的微服务发送调用，告知有新数据它可能想要。这可以缩小数据发布和数据处理之间的差距。但一般来说，如果你将这种模式用于非常大的数据量，低延迟不太可能在你需求列表的顶部。如果你对发送大量数据并使其更「实时」处理感兴趣，那么使用像 Kafka 这样的流技术将是更好的选择。

另一个大缺点，如果你记得我们回到图 4-7 对公共耦合的探讨，应该是相当明显的，即公共数据存储成为耦合的潜在来源。如果该数据存储以某种方式更改结构，它可能破坏微服务之间的通信。

通信的健壮性也将取决于底层数据存储的健壮性。严格来说这不是缺点，但需要注意。如果你将文件放在文件系统上，你可能想确保文件系统本身不会以有趣的方式失败。

### 适用场景

这种模式真正闪耀的地方是使可能对可以使用什么技术有限制的进程之间的互操作性成为可能。让现有系统与你的微服务的 gRPC 接口对话或订阅其 Kafka 主题，从微服务的角度来看可能更方便，但从消费者的角度来看则不是。较旧的系统可能对它们可以支持的技术有限制，并且可能有很高的变更成本。另一方面，即使是旧的大型机系统也应该能够从文件中读取数据。这当然都取决于使用广泛支持的数据存储技术——我也可以用像 Redis 缓存这样的东西实现这种模式。但你的旧大型机系统能跟 Redis 对话吗？

这种模式的另一个主要优势是共享大量数据。如果你需要向文件系统发送多 GB 的文件或向数据库加载几百万行，那么这种模式就是你要走的路。

## 模式：请求-响应通信

对于请求-响应，微服务向下游服务发送请求，要求它做某事，并期望收到带有请求结果的响应。这种交互可以通过同步阻塞调用进行，也可以以异步非阻塞方式实现。这种交互的一个简单例子如图 4-8 所示，其中 Chart 微服务（整理不同流派最畅销的 CD）向 Inventory 服务发送请求，询问某些 CD 的当前库存水平。

```
Chart ──[request stock levels]──▶ Inventory ──[response]──▶ Chart
```

**图 4-8. Chart 微服务向 Inventory 发送请求询问库存水平**

像这样从其他微服务检索数据是请求-响应调用的常见用例。有时，你只需要确保某事完成了。在图 4-9 中，Warehouse 微服务从 Order Processor 收到请求，要求它预留库存。Order Processor 只需要知道库存已成功预留才能继续收款。如果无法预留库存——也许因为某物品不再可用——那么可以取消支付。在需要按某种顺序完成调用的情况下使用请求-响应调用是常见的。

```
Order Processor ──[reserve stock]──▶ Warehouse
       │                                  │
       │◀────────[stock reserved]─────────┘
       │
       └──[take payment]──▶ Payment
```

**图 4-9. Order Processor 需要确保在收款之前可以预留库存**

### 命令与请求

我听说过有些人谈论发送命令（Commands）而不是请求，特别是在异步请求-响应通信的上下文中。命令一词背后的意图可以说与请求相同——即，上游微服务要求下游微服务做某事。

就个人而言，我更喜欢请求一词。命令意味着必须服从的指令，它可能导致人们觉得命令必须被执行。请求意味着可以被拒绝的东西。微服务根据其自身的内部逻辑审查每个请求并决定是否应该执行该请求是正确的。如果发送的请求违反内部逻辑，微服务应该拒绝它。虽然这是一个微妙的区别，但我不觉得命令一词传达了相同的含义。

我会坚持使用请求而不是命令，但无论你决定使用什么术语，只需记住微服务可以在适当时拒绝请求/命令。

### 实现：同步与异步

像这样的请求-响应调用可以以阻塞同步或非阻塞异步风格实现。对于同步调用，你通常会看到与下游微服务打开网络连接，请求沿此连接发送。在上游微服务等待下游微服务响应时，连接保持打开。在这种情况下，发送响应的微服务不需要真正了解发送请求的微服务——它只是通过入站连接发送东西回去。如果该连接死亡，也许因为上游或下游微服务实例死亡，那么我们可能有问题。

对于异步请求-响应，事情就不那么直接了。让我们重温与预留库存相关的流程。在图 4-10 中，预留库存的请求作为消息通过某种消息代理发送（我们将在本章后面探讨消息代理）。消息不是直接从 Order Processor 到 Inventory 微服务，而是坐在队列中。Inventory 在能够时从该队列消费消息。它读取请求，执行预留库存的相关工作，然后需要将响应发送回 Order Processor 正在读取的队列。Inventory 微服务需要知道将响应路由到哪里。在我们的例子中，它将此响应发送回另一个队列，该队列反过来被 Order Processor 消费。

```
Order Processor ──[request]──▶ Queue ──▶ Inventory
       ▲                              │
       └────────[response]─────────────┘
```

**图 4-10. 使用队列发送库存预留请求**

所以对于非阻塞异步交互，接收请求的微服务需要要么隐式知道将响应路由到哪里，要么被告知响应应该去哪里。使用队列时，我们还有一个额外的好处，即多个请求可以在队列中缓冲等待处理。这可以帮助请求无法足够快地处理的情况。微服务可以在准备好时消费下一个请求，而不是被太多调用压垮。当然，很大程度上取决于队列吸收这些请求的能力。

当微服务以这种方式收到响应时，它可能需要将响应与原始请求关联起来。这可能具有挑战性，因为可能已经过去了很多时间，根据所用协议的性质，响应可能不会返回到发送请求的微服务的同一实例。在我们作为下订单一部分预留库存的例子中，我们需要知道如何将「库存已预留」响应与给定订单关联起来，以便我们可以继续处理该特定订单。处理这个问题的一种简单方法是将与原始请求相关的任何状态存储到数据库中，这样当响应到来时，接收实例可以重新加载任何相关状态并相应采取行动。

最后一点：所有形式的请求-响应交互可能都需要某种形式的超时处理，以避免系统在等待可能永远不会发生的事情时被阻塞的问题。这种超时功能如何实现可能因实现技术而异，但将需要。我们将在第 12 章更详细地了解超时。

### 并行与顺序调用

在使用请求-响应交互时，你经常会遇到需要在继续某些处理之前进行多次调用的情况。

考虑 MusicCorp 需要从三个不同的供应商检查给定物品价格的情况，我们通过发出 API 调用来完成。我们想从所有三个供应商那里得到价格，然后再决定我们想从谁那里订购新库存。我们可以决定按顺序进行三次调用——等待每一次完成再进行下一次。在这种情况下，我们将等待每次调用延迟的总和。如果对每个提供商的 API 调用需要一秒钟返回，我们将等待三秒钟才能决定我们应该从谁那里订购。

更好的选择是并行运行这三个请求；然后操作的整体延迟将基于最慢的 API 调用，而不是每次 API 调用延迟的总和。

响应式扩展和 async/await 等机制在帮助并行运行调用方面非常有用，这可以导致某些操作延迟的显著改善。

### 适用场景

对于任何在进一步处理可以进行之前需要请求结果的情况，请求-响应调用完全有意义。它们也非常适合微服务想知道调用是否没有成功以便它可以执行某种补偿操作（如重试）的情况。如果任一符合你的情况，那么请求-响应是合理的方法；唯一剩下的问题是决定同步与异步实现，具有我们之前讨论的相同权衡。

## 模式：事件驱动通信

与请求-响应调用相比，事件驱动通信看起来相当奇怪。微服务不是要求某个其他微服务做某事，而是发出可能被其他微服务接收或不被接收的事件。它是一种 inherently 异步的交互，因为事件监听器将在它们自己的执行线程上运行。

事件是关于已发生某事的陈述，几乎总是发生在发出事件的微服务世界内部的某事。发出事件的微服务不知道其他微服务使用该事件的意图，事实上它甚至可能不知道其他微服务存在。它在需要时发出事件，这就是其职责的结束。

在图 4-11 中，我们看到 Warehouse 发出与打包订单过程相关的事件。这些事件被两个微服务 Notifications 和 Inventory 接收，它们相应做出反应。Notifications 微服务向客户发送电子邮件以更新订单状态的变化，而 Inventory 微服务可以在物品被打包到客户订单中时更新库存水平。

```
Warehouse ──[events]──▶ Notifications (send email)
       │
       └──────────────▶ Inventory (update stock)
```

**图 4-11. Warehouse 发出一些下游微服务订阅的事件**

Warehouse 只是在广播事件，假设相关方会相应做出反应。它不知道事件的接收者是谁，使事件驱动交互总体上更加松散耦合。当你将其与请求-响应调用进行比较时，可能需要一段时间才能理解责任的倒置。对于请求-响应，我们可能反而期望 Warehouse 在适当时告诉 Notifications 微服务发送电子邮件。在这种模型中，Warehouse 需要知道哪些事件需要客户通知。对于事件驱动交互，我们反而将这种责任推入 Notifications 微服务。

事件背后的意图可以被认为是请求的对立面。事件发出者让接收者决定做什么。对于请求-响应，发送请求的微服务知道应该做什么，并告诉其他微服务它认为接下来需要发生什么。这当然意味着在请求-响应中，请求者必须了解下游接收者能做什么，意味着更大程度的领域耦合。对于事件驱动协作，事件发出者不需要了解任何下游微服务能做什么，事实上它甚至可能不知道它们存在——因此，耦合大大减少。

我们在事件驱动交互中看到的责任分配可以反映我们在组织试图创建更自治团队时看到的责任分配。与其集中持有所有责任，我们希望将其推入团队本身，使它们能够以更自治的方式运作——我们将在第 15 章重新探讨这个概念。在这里，我们将责任从 Warehouse 推入 Notifications 和 Payment——这可以帮助我们减少像 Warehouse 这样的微服务的复杂性，并导致我们系统中「智能」的更均匀分布。我们将在第 6 章比较编排和协调时更详细地探讨这个想法。

### 事件与消息

有时我见过消息（Messages）和事件（Events）这两个术语被混淆。事件是一个事实——关于某事发生的陈述，以及关于究竟发生了什么的一些信息。消息是我们通过异步通信机制（如消息代理）发送的东西。

对于事件驱动协作，我们想要广播该事件，实现该广播机制的典型方法是将事件放入消息中。消息是媒介；事件是有效负载。

同样，我们可能想将请求作为消息的有效负载发送——在这种情况下，我们将实现一种形式的异步请求-响应。

### 实现

我们在这里需要考虑两个主要方面：微服务发出事件的方式，以及消费者发现这些事件已发生的方式。

传统上，像 RabbitMQ 这样的消息代理试图处理这两个问题。生产者使用 API 向代理发布事件。代理处理订阅，允许消费者在事件到达时被告知。这些代理甚至可以处理消费者的状态——例如，通过帮助跟踪他们以前见过哪些消息。这些系统通常被设计为可扩展和健壮的，但这并非没有代价。它可能给开发过程增加复杂性，因为这是你可能需要运行以开发和测试服务的另一个系统。可能还需要额外的机器和专业知识来保持这种基础设施运行。但一旦运行，它可以是一种极其有效的方式来实现松散耦合的、事件驱动的架构。总的来说，我是粉丝。

然而，要警惕中间件世界，消息代理只是其中一小部分。队列本身是完全合理的、有用的东西。然而，供应商往往想将大量软件与它们打包，这可能导致越来越多的智能被推入中间件，如企业服务总线（Enterprise Service Bus）所证明的。确保你知道你得到的是什么：保持你的中间件愚蠢，将智能保留在端点中。

另一种方法是尝试使用 HTTP 作为传播事件的方式。Atom 是一个符合 REST 的规范，定义了（除其他外）发布资源提要的语义。许多客户端库允许我们创建和消费这些提要。所以我们的客户服务可以在客户服务更改时向这样的提要发布事件。我们的消费者只需轮询提要，寻找更改。一方面，我们可以重用现有 Atom 规范和任何相关库的事实是有用的，我们知道 HTTP 非常擅长处理规模。然而，这种 HTTP 的使用在低延迟方面不好（一些消息代理在这方面表现出色），我们仍然必须处理消费者需要跟踪他们见过哪些消息并管理自己的轮询计划的事实。

我见过人们花很长时间实现越来越多的行为，这些行为你使用适当的消息代理开箱即得，以使 Atom 适用于某些用例。例如，竞争消费者模式（Competing Consumer Pattern）描述了一种方法，你启动多个工作实例来竞争消息，这对于扩展工作实例数量以处理独立作业列表效果很好（我们将在下一章回到这一点）。然而，我们希望避免两个或更多工作者看到同一条消息的情况，因为我们最终会做比需要更多次相同任务。使用消息代理，标准队列将处理这个。使用 Atom，我们现在需要管理所有工作者之间的自己的共享状态，以尝试减少重复工作的机会。

如果你已经有一个好的、健壮的消息代理可用，考虑使用它来处理事件的发布和订阅。如果你还没有，看看 Atom，但要意识到沉没成本谬误。如果你发现自己想要越来越多消息代理给你的支持，在某个时候你可能想改变你的方法。

就我们通过这些异步协议实际发送什么而言，与同步通信相同的考虑适用。如果你目前对使用 JSON 编码请求和响应感到满意，坚持使用它。

### 事件中有什么？

在图 4-12 中，我们看到 Customer 微服务正在广播一个事件，告知相关方新客户已在系统中注册。两个下游微服务 Loyalty 和 Notifications 关心此事件。Loyalty 微服务通过为新客户设置账户以便他们可以开始赚取积分来对收到事件做出反应，而 Notifications 微服务向新注册的客户发送电子邮件，欢迎他们来到 MusicCorp 的奇妙乐趣。

```
Customer ──[CustomerRegistered]──▶ Loyalty (setup account)
       │
       └──────────────────────────▶ Notifications (send welcome email)
```

**图 4-12. 当新客户注册时，Notifications 和 Loyalty 微服务收到事件**

对于请求，我们要求微服务做某事并提供执行请求操作所需的信息。对于事件，我们正在广播其他方可能感兴趣的事实，但由于发出事件的微服务不能也不应该知道谁收到事件，我们如何知道其他方可能从事件中需要什么信息？究竟应该 inside 事件中有什么？

**仅 ID**

一种选择是让事件只包含新注册客户的标识符，如图 4-13 所示。Loyalty 微服务只需要此标识符来创建匹配的忠诚度账户，所以它拥有所需的所有信息。然而，虽然 Notifications 微服务知道在收到此类事件时需要发送欢迎电子邮件，但它需要额外信息来完成其工作——至少一个电子邮件地址，可能还有客户姓名以给电子邮件个人触感。由于此信息不在 Notifications 微服务收到的事件中，它别无选择，只能从 Customer 微服务获取此信息，我们在图 4-13 中看到了这一点。

```
Customer ──[event: customerId only]──▶ Notifications
       │                                      │
       │◀──────[request details]──────────────┘
```

**图 4-13. Notifications 微服务需要从 Customer 微服务请求事件中未包含的更多详情**

这种方法有一些缺点。首先，Notifications 微服务现在必须了解 Customer 微服务，增加了额外的领域耦合。虽然正如我们在第 2 章讨论的，领域耦合在耦合光谱上处于较松散的一端，我们仍然希望尽可能避免。如果 Notifications 微服务收到的事件包含它需要的所有信息，那么就不需要此回调。来自接收微服务的回调也可能导致另一个主要缺点——即，在大量接收微服务的情况下，发出事件的微服务可能会因此收到大量请求。想象一下，如果五个不同的微服务都收到了相同的客户创建事件，并且都需要请求额外信息——它们都必须立即向 Customer 微服务发送请求以获取它们需要的。随着对特定事件感兴趣的微服务数量增加，这些调用的影响可能变得显著。

**完全详细的事件**

我喜欢的替代方案是将你 otherwise 乐意通过 API 分享的一切放入事件中。如果你会让 Notifications 微服务询问给定客户的电子邮件地址和姓名，为什么不一开始就把该信息放在事件中呢？在图 4-14 中，我们看到了这种方法——Notifications 现在更加自给自足，能够在不与 Customer 微服务通信的情况下完成其工作。事实上，它可能永远不需要知道 Customer 微服务存在。

```
Customer ──[event: full details]──▶ Notifications (self-sufficient)
```

**图 4-14. 包含更多信息的事件可以允许接收微服务在不需进一步调用事件源的情况下采取行动**

除了包含更多信息的事件可以允许更松散耦合的事实之外，包含更多信息的事件还可以兼作给定实体所发生事情的历史记录。这可能有助于你作为实施审计系统的一部分，或者甚至提供在给定时间点重建实体的能力——意味着这些事件可以作为事件溯源（Event Sourcing）的一部分使用，我们将在稍后简要探讨这个概念。

虽然这种方法 definitely 是我的偏好，但它并非没有缺点。首先，如果与事件相关的数据很大，我们可能对事件的大小有顾虑。现代消息代理（假设你使用一个来实现你的事件广播机制）对消息大小有相当慷慨的限制；Kafka 中消息的默认最大大小是 1 MB，RabbitMQ 最新版本对单条消息有 512 MB 的理论上限（从之前的 2 GB 限制降下来！），尽管可以预期像这样的大消息会有一些有趣的性能问题。但即使是 Kafka 给我们的 1 MB 作为消息的最大大小，也给了我们很大的空间来发送相当多的数据。归根结底，如果你进入一个你开始担心事件大小的空间，那么我建议采用混合方法，其中一些信息在事件中，但其他（较大的）数据可以在需要时查找。

在图 4-14 中，Loyalty 不需要知道客户的电子邮件地址或姓名，但它仍然通过事件收到。如果我们试图限制哪些微服务可以看到什么类型的数据，这可能导致顾虑——例如，我可能想限制哪些微服务可以看到个人可识别信息（PII）、支付卡详情或类似的敏感数据。解决这个问题的一种方法可能是发送两种不同类型的事件——一种包含 PII 可以被某些微服务看到，另一种排除 PII 可以更广泛地广播。这在管理不同事件的可见性和确保两个事件实际被触发方面增加了复杂性。当微服务发送第一种类型的事件但在可以发送第二种事件之前死亡时会发生什么？

另一个考虑是，一旦我们将数据放入事件，它就成为我们与外部世界的契约的一部分。我们必须意识到，如果我们从事件中删除字段，我们可能会破坏外部方。信息隐藏在事件驱动协作中仍然是一个重要的概念——我们在事件中放入的数据越多，外部方对事件的假设就越多。我的一般规则是，如果我乐意通过请求-响应 API 分享相同的数据，我就可以将信息放入事件中。

### 适用场景

事件驱动协作在信息想要被广播的情况下蓬勃发展，以及在你乐意倒置意图的情况下。从告诉其他东西做什么的模型转向让下游微服务自己弄清楚，具有很大的吸引力。

在你关注松散耦合多于其他因素的情况下，事件驱动协作将具有明显的吸引力。

需要注意的是，这种协作风格往往会带来新的复杂性来源，特别是如果你对它的接触有限。如果你对这种通信形式不确定，请记住我们的微服务架构可以（而且可能会）包含不同交互风格的混合。你不必全力以赴采用事件驱动协作；也许从只是一个事件开始，然后从那里开始。

就个人而言，我发现自己几乎默认倾向于事件驱动协作。我的大脑似乎已经以某种方式重新连接，使这些类型的通信对我来说似乎很明显。这并不完全有帮助，因为除了说感觉对之外，很难解释为什么是这样。但这只是我自己的内置偏见——我自然地倾向于根据我自己的经验我所知道的。我对此交互形式的吸引力几乎完全由我先前与过度耦合系统的糟糕经历驱动的可能性很大。我可能只是将军一次又一次地打最后一仗，而没有考虑也许这次真的不同。

我要说的是，抛开我自己的偏见，我看到的团队用事件驱动交互替换请求-响应交互的远多于反向的。

## 谨慎行事

其中一些异步的东西似乎很有趣，对吧？事件驱动架构似乎导致显著更解耦、可扩展的系统。它们可以。但这些通信风格确实导致复杂性的增加。这不仅仅是管理发布和订阅消息所需的复杂性，正如我们刚才讨论的，还有我们可能面临的其他问题中的复杂性。例如，在考虑长时间运行的异步请求-响应时，我们必须考虑当响应回来时该怎么办。它返回到发起请求的同一节点吗？如果是，如果该节点关闭会发生什么？如果不是，我是否需要将信息存储在某个地方以便我相应做出反应？如果你有正确的 API，短期异步可能更容易管理，但即便如此，对于习惯于进程内同步消息调用的程序员来说，这是一种不同的思考方式。

是时候讲一个警示故事了。早在 2006 年，我正在为一家银行构建定价系统。我们会查看市场事件并计算出投资组合中哪些项目需要重新定价。一旦我们确定了要处理的列表，我们将所有这些放到消息队列上。我们使用网格创建定价工作者的池，允许我们按需扩展和缩小定价场。这些工作者使用竞争消费者模式，每个都尽可能快地吞噬消息，直到没有东西可处理。

系统启动并运行了，我们感到相当自满。然而有一天，就在我们推出一个版本之后，我们遇到了一个令人讨厌的问题：我们的工作者不断死亡。死了又死。死了又死。

最终，我们追踪到了问题。一个 bug 悄悄进入，某种类型的定价请求会导致工作者崩溃。我们使用的是事务队列：随着工作者死亡，其对请求的锁超时，定价请求被放回队列——只是让另一个工作者拾取并死亡。这是 Martin Fowler 所说的灾难性故障转移（Catastrophic Failover）的经典例子。

除了 bug 本身之外，我们未能为队列上的作业指定最大重试限制。所以我们修复了 bug，并配置了最大重试。但我们也意识到我们需要一种方式来查看和 potentially 重放这些坏消息。我们最终不得不实施消息医院（或死信队列），消息在失败时被发送到那里。我们还创建了一个 UI 来查看这些消息并在需要时重试它们。如果你只熟悉同步点对点通信，这些类型的问题并不立即明显。

与事件驱动架构和一般异步编程相关的复杂性使我相信，你应该谨慎地开始采用这些想法。确保你有良好的监控，并强烈考虑使用关联 ID，它允许你跨进程边界追踪请求，我们将在第 10 章深入介绍。

我还强烈建议查看 Gregor Hohpe 和 Bobby Woolf 的《企业集成模式》（Enterprise Integration Patterns），[^4] 其中包含你可能想在此空间考虑的更多消息模式的更多细节。

然而，我们也必须对我们可以认为「更简单」的集成风格诚实——与知道事情是否成功相关的问题并不限于异步形式的集成。对于同步、阻塞调用，如果你得到超时，这是因为请求丢失而下游方没有收到吗？还是请求通过了，但响应丢失了？你在那种情况下做什么？如果你重试，但原始请求确实通过了怎么办？（嗯，这就是幂等性（Idempotency）的用武之地，我们将在第 12 章介绍的主题。）

可以说，在故障处理方面，同步阻塞调用在弄清楚事情是否发生（或没有）时可能给我们带来同样多的头痛。只是那些头痛可能对我们来说更熟悉！

## 总结

在本章中，我分解了微服务通信的一些关键风格，并讨论了各种权衡。并不总是有单一的正确选项，但希望我已经详细说明了关于同步和异步调用以及事件驱动和请求-响应通信风格的足够信息，以帮助你对给定上下文做出正确的选择。我自己对异步、事件驱动协作的偏见不仅是我的经验的函数，也是我对一般耦合的厌恶。但这种通信风格带来了不容忽视的显著复杂性，每种情况都是独特的。

在本章中，我简要提到了可用于实现这些交互风格的几种特定技术。我们现在准备好开始本书的第二部分——实现。在下一章中，我们将更深入地探讨实现微服务通信。
