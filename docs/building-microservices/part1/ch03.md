# 第3章 拆分单体

阅读本书的许多人可能没有一块空白的画布来设计你的系统，即使有，从微服务开始可能也不是一个好主意，原因我们在第 1 章探讨过。许多人将已经有一个现有系统，也许是某种形式的单体架构，你正在寻求迁移到微服务架构。

在本章中，我将概述一些初步步骤、模式和一般技巧，以帮助你在向微服务架构的过渡中导航。

## 要有目标

微服务不是目标。拥有微服务并不会「赢」。采用微服务架构应该是一个有意识的决定，基于理性的决策。你应该只有在无法用当前架构找到任何更简单的方式来实现最终目标时，才考虑迁移到微服务架构。

如果没有对你试图实现什么的清晰理解，你可能会陷入将活动与结果混淆的陷阱。我见过团队痴迷于创建微服务，却从未问过为什么。鉴于微服务可能引入的新复杂性来源，这是极其有问题的。

专注于微服务而不是最终目标也意味着你可能停止思考你可能以其他方式带来你所寻求的变革。例如，微服务可以帮助你扩展系统，但通常应该首先考虑许多替代的扩展技术。在负载均衡器后面启动现有单体系统的几个副本，可能比通过复杂而漫长的分解到微服务更有效地帮助你扩展系统。

::: tip
微服务并不容易。先尝试简单的东西。
:::

最后，没有明确的目标，很难知道从哪里开始。你应该首先创建哪个微服务？没有对你试图实现什么的总体理解，你就是在盲目飞行。

所以要清楚你试图实现什么变革，在考虑微服务之前考虑实现该最终目标的更简单方式。如果微服务真的是前进的最佳方式，那么根据该最终目标跟踪你的进展，并在必要时改变方向。

## 渐进式迁移

如果你进行大爆炸式重写，你唯一能保证的就是大爆炸。
—Martin Fowler

如果你到了决定拆分现有单体系统是正确做法的时候，我强烈建议你一点一点地削减单体，一次提取一点。渐进式方法将帮助你在过程中了解微服务，也将限制出错的影响（你会出错的！）。把我们的单体想象成一块大理石。我们可以把整个东西炸掉，但那很少有好结果。一点一点地削减它更有意义。

将大旅程分解为许多小步骤。每个步骤都可以执行并从中学习。如果结果证明是倒退的一步，那只是小一步。无论哪种方式，你都会从中学习，你采取的下一步将受到之前步骤的启发。

将事物分解成更小的部分也允许你识别快速胜利并从中学习。这可以使下一步更容易，并有助于建立势头。通过一次拆分一个微服务，你还可以逐步解锁它们带来的价值，而不必等待某种大爆炸部署。

所有这些导致了对于关注微服务的人，我给出的标准建议：如果你认为微服务是个好主意，从小处开始。选择一两个功能领域，将它们实现为微服务，将它们部署到生产环境，然后反思创建新微服务是否帮助你更接近最终目标。

::: warning
在你运行于生产环境之前，你不会真正体会到微服务架构可能带来的恐怖、痛苦和苦难。
:::

## 单体很少是敌人

虽然我在本书开头已经论证了某种形式的单体架构可以是一个完全有效的选择，但值得重复的是，单体架构并非 inherently 坏的，因此不应被视为敌人。

不要专注于「没有单体」；相反，专注于你期望架构变革带来的好处。

在向微服务转变之后，现有的单体架构通常仍然存在，尽管往往能力有所减弱。例如，提高应用处理更多负载能力的迁移可能通过移除当前瓶颈的 10% 功能来满足，将剩余的 90% 留在单体系统中。

许多人发现单体与微服务共存的现实是「混乱的」——但现实世界运行系统的架构从来不是干净或原始的。如果你想要「干净」的架构，尽管将你可能拥有的理想化系统架构的打印件塑封起来，如果你有完美的预见力和无限的资金的话。真实的系统架构是不断演变的东西，必须随着需求和知识的变化而适应。技巧在于习惯这个想法，我将在第 16 章回到这一点。

通过使你的微服务迁移成为渐进式旅程，你能够逐步削减现有的单体架构，沿途交付改进，同时，重要的是，知道何时停止。

在 surprisingly 罕见的情况下，单体的消亡可能是硬性要求。根据我的经验，这通常仅限于现有单体基于死亡或垂死的技术、绑定到需要退役的基础设施，或者可能是你想抛弃的昂贵的第三方系统的情况。即使在这些情况下，出于我概述的原因，渐进式分解方法也是有保证的。

## 过早分解的危险

当你对领域理解不清时创建微服务存在危险。这个问题可能造成的问题的一个例子来自我之前的公司 Thoughtworks。其产品之一是 Snap CI，一个托管的持续集成和持续交付工具（我们将在第 7 章讨论这些概念）。该团队之前曾在一个类似的工具 GoCD 上工作，这是一个现在开源的持续交付工具，可以本地部署而不是在云中托管。

尽管 Snap CI 和 GoCD 项目之间在很早就有一些代码重用，但最终 Snap CI 变成了一个全新的代码库。尽管如此，团队在 CD 工具领域的先前经验使他们能够更快地识别边界并将系统构建为一组微服务。

然而，几个月后，很明显 Snap CI 的用例有足够微妙的不同，以至于对服务边界的初步看法不太正确。这导致对服务进行了大量更改，以及相关的高变更成本。最终，团队将服务合并回一个单体系统，让团队成员有时间更好地理解边界应该存在的地方。一年后，团队能够将单体系统拆分为微服务，其边界被证明更加稳定。这远不是我见过的这种情况的唯一例子。过早地将系统分解为微服务可能代价高昂，特别是如果你对该领域不熟悉。在许多方面，拥有你想要分解为微服务的现有代码库比从一开始就尝试微服务要容易得多，正是因为这个原因。

## 首先拆分什么？

一旦你牢牢掌握了为什么你认为微服务是个好主意，你可以用这种理解来帮助优先考虑首先创建哪些微服务。想要扩展应用？当前限制系统处理负载能力的功能将排在列表的前面。想要提高上市时间？查看系统的易变性，识别那些最频繁更改的功能部分，看看它们是否可以作为微服务工作。你可以使用像 CodeScene 这样的静态分析工具快速找到代码库中易变的部分。你可以在图 3-1 中看到 CodeScene 中视图的例子，我们看到了开源 Apache Zookeeper 项目中的热点。

但你还必须考虑哪些分解是可行的。某些功能可能如此深入地嵌入现有的单体应用中，以至于无法看出如何将其解缠。或者也许所讨论的功能对应用如此关键，以至于任何更改都被视为高风险。或者，你想要迁移的功能可能已经有些自包含，因此提取似乎非常直接。

```

**图 3-1. CodeScene 中的热点视图，帮助识别代码库中频繁更改的部分**

从根本上说，关于将哪些功能拆分为微服务的决定最终将是这两种力量之间的平衡——提取的容易程度与首先提取微服务的好处。

对于前几个微服务，我的建议是选择更倾向于「容易」光谱一端的东西——一个我们认为在实现端到端目标方面有一定影响的微服务，当然，但我们会认为是低垂果实的东西。对于这样的过渡，特别是可能持续数月或数年的过渡，尽早获得势头感很重要。所以你需要一些快速胜利。

另一方面，如果你尝试提取你认为最容易的微服务却无法使其工作，可能值得重新考虑微服务是否真的适合你和你的组织。

有了几次成功和一些经验教训，你将处于更好的位置来应对更复杂的提取，这些提取可能也在功能的更关键领域运营。

## 按层分解

所以你已经确定了要提取的第一个微服务；接下来呢？嗯，我们可以将该分解进一步分解为更小的步骤。

如果我们考虑基于 Web 的服务堆栈的传统三层，那么我们可以从用户界面、后端应用代码和数据方面来看我们想要提取的功能。

从微服务到用户界面的映射通常不是 1:1（这是我们在第 14 章更深入探讨的主题）。因此，提取与微服务相关的用户界面功能可以被视为单独的步骤。我在这里要提醒不要忽视等式的用户界面部分。我见过太多组织只关注分解后端功能的好处，这往往导致任何架构重组的过度孤岛化方法。有时最大的好处可能来自 UI 的分解，所以忽视这一点会有风险。通常，UI 的分解往往落后于后端向微服务的分解，因为在微服务可用之前，很难看到 UI 分解的可能性；只是确保它不会落后太多。

如果我们然后看后端代码和相关存储，在提取微服务时两者都在范围内至关重要。让我们考虑图 3-2，我们正在寻求提取与管理客户愿望清单相关的功能。有一些应用代码存在于单体中，以及数据库中的一些相关数据存储。那么我们应该先提取哪一部分？

│  └─────────────────┘  └─────────────────┘                   │
└─────────────────────────────────────────────────────────────┘
```

### 代码优先

在图 3-3 中，我们将与愿望清单功能相关的代码提取到了一个新的微服务中。愿望清单的数据在此阶段仍保留在单体数据库中——我们还没有完成分解，直到我们也移出与新 Wishlist 微服务相关的数据。

```
┌──────────────────┐     ┌─────────────────────────────────────┐
│ Wishlist Service │     │ Monolith + Shared DB                 │
│ (new)            │────▶│ (Wishlist data still here)           │
└──────────────────┘     └─────────────────────────────────────┘
```

**图 3-3. 先将愿望清单代码移入新微服务，将数据留在单体数据库中**

根据我的经验，这往往是最常见的第一步。这样做的首要原因是它往往能带来更多的短期好处。如果我们将数据留在单体数据库中，我们就是在为未来储存很多痛苦，所以那也需要解决，但我们从新微服务中获得了很大收益。

提取应用代码往往比从数据库中提取东西更容易。如果我们发现无法干净地提取应用代码，我们可以中止任何进一步的工作，避免解缠数据库的需要。然而，如果应用代码被干净地提取但提取数据被证明是不可能的，我们可能就有麻烦了——因此，即使你决定在数据之前提取应用代码，你也需要查看相关的数据存储，并对提取是否可行以及你将如何进行有一些想法。所以在开始之前，做好应用代码和数据都将被提取的功课。

### 数据优先

在图 3-4 中，我们看到数据在应用代码之前被提取。我较少看到这种方法，但在你不确定数据是否可以干净地分离的情况下可能有用。在这里，你在继续希望更容易的应用代码提取之前证明这是可以做到的。

```
┌──────────────────┐     ┌─────────────────────────────────────┐
│ Wishlist Tables  │     │ Monolith                             │
│ (extracted)      │     │ (Wishlist code still here)            │
└──────────────────┘     └─────────────────────────────────────┘
```

**图 3-4. 与愿望清单功能相关的表首先被提取**

这种方法在短期内的主要好处是降低微服务完整提取的风险。它迫使你提前处理数据库中强制数据完整性丧失或两组数据之间缺乏事务操作等问题。我们将在本章后面简要讨论这两个问题的含义。

## 有用的分解模式

有许多模式可以帮助拆分现有系统。其中许多在我的《Monolith to Microservices》[^1] 一书中有详细探讨；与其在这里全部重复，我将分享其中一些的概述，让你了解什么是可能的。

### 绞杀者无花果模式

在系统重写期间经常使用的一种技术是绞杀者无花果模式（Strangler Fig Pattern），这是 Martin Fowler 创造的一个术语。受一种植物的启发，该模式描述了随时间用新系统包装旧系统的过程，允许新系统逐步接管旧系统的越来越多功能。

图 3-5 中所示的方法很简单。你拦截对现有系统的调用——在我们的情况下是现有的单体应用。如果该功能块的调用在我们的新微服务架构中实现，它被重定向到微服务。如果功能仍由单体提供，调用被允许继续到单体本身。

```
                    ┌─────────────────┐
Request ───────────▶│  Proxy/Router   │
                    │  (Strangler)    │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
       ┌──────────┐   ┌──────────┐   ┌──────────┐
       │ New      │   │ Monolith │   │ Monolith │
       │ Microsvc │   │ (old)    │   │ (old)    │
       └──────────┘   └──────────┘   └──────────┘
```

**图 3-5. 绞杀者无花果模式概述**

这种模式的美妙之处在于，它通常可以在不对底层单体应用进行任何更改的情况下完成。单体甚至不知道它已经被新系统「包装」了。

### 并行运行

当从现有经过验证的应用架构提供的功能切换到基于微服务的花哨新架构时，可能会有些紧张，特别是如果正在迁移的功能对你的组织至关重要。确保新功能运行良好而不冒现有系统行为风险的一种方法是使用并行运行模式（Parallel Run Pattern）：将功能单体实现和新微服务实现并排运行，服务相同的请求，并比较结果。我们将在「并行运行」一节（第 272 页）更详细地探讨这种模式。

### 功能开关

功能开关（Feature Toggle）是一种允许功能打开或关闭，或在某些功能的两种不同实现之间切换的机制。功能开关模式具有良好的通用适用性，但作为微服务迁移的一部分特别有用。

正如我在绞杀者无花果应用中概述的，我们经常在过渡期间将现有功能保留在单体中，我们希望能够切换功能版本——单体中的功能和新微服务中的功能。对于使用 HTTP 代理的绞杀者无花果模式例子，我们可以在代理层实现功能开关，以允许简单的控制在实现之间切换。

有关功能开关的更广泛介绍，我推荐 Pete Hodgson 的文章「Feature Toggles (aka Feature Flags)」。[^2]

## 数据分解关注点

当我们开始拆分数据库时，我们可能会造成许多问题。以下是你可能面临的一些挑战，以及一些帮助技巧。

### 性能

数据库，特别是关系数据库，擅长跨不同表连接数据。非常擅长。事实上如此擅长，以至于我们认为是理所当然的。然而，通常当我们在微服务的名义下拆分数据库时，我们最终不得不将连接操作从数据层移到微服务本身。尽管我们尽力而为，这不太可能同样快。

考虑图 3-6，它说明了我们关于 MusicCorp 所处的情况。我们决定提取我们的目录功能——可以管理和暴露有关艺术家、曲目和专辑的信息的东西。目前，我们单体中的目录相关代码使用 Albums 表来存储我们可能可供销售的 CD 的信息。这些专辑最终在我们的 Ledger 表中被引用，这是我们跟踪所有销售的地方。Ledger 表中的行记录某物售出的日期，以及引用售出物品的标识符；在我们的例子中，该标识符称为 SKU（库存保持单位），这是零售系统中的常见做法。[^3]

每月底，我们需要生成一份概述我们最畅销 CD 的报告。Ledger 表帮助我们了解哪个 SKU 售出了最多副本，但该 SKU 的信息在 Albums 表中。我们想让报告又好读，所以与其说「我们售出了 400 份 SKU 123，赚了 1,596 美元」，我们会在售出内容中添加更多信息，改为说「我们售出了 400 份《Now That's What I Call Death Polka》，赚了 1,596 美元」。为此，由我们的财务代码触发的数据库查询需要将 Ledger 表的信息与 Albums 表连接，如图 3-6 所示。

```
┌─────────────────────────────────────────────────────────────┐
│  Monolithic Database                                         │
│  ┌─────────────┐         ┌─────────────┐                    │
│  │   Ledger    │──JOIN──▶│   Albums    │                    │
│  │  (SKU, qty) │         │ (SKU, name) │                    │
│  └─────────────┘         └─────────────┘                    │
└─────────────────────────────────────────────────────────────┘
```

**图 3-6. 单体数据库中的连接操作**

在我们新的基于微服务的世界中，我们新的 Finance 微服务负责生成畅销报告，但本地没有专辑数据。所以它需要从我们新的 Catalog 微服务获取这些数据，如图 3-7 所示。在生成报告时，Finance 微服务首先查询 Ledger 表，提取上个月最畅销的 SKU 列表。此时，我们本地拥有的唯一信息是 SKU 列表和每个 SKU 售出的副本数量。

```
Finance ──[query Ledger]──▶ Local DB
    │
    └──[fetch album info]──▶ Catalog Microservice ──▶ Albums DB
```

**图 3-7. 用服务调用替换数据库连接操作**

接下来，我们需要调用 Catalog 微服务，请求有关这些 SKU 中每一个的信息。此请求反过来将导致 Catalog 微服务对其自己的数据库进行自己的本地 SELECT。

从逻辑上讲，连接操作仍然在发生，但它现在发生在 Finance 微服务内部而不是数据库中。连接已从数据层转移到应用代码层。不幸的是，此操作的效率将远不如连接保留在数据库中时那样高。我们已经从一个我们有单个 SELECT 语句的世界，到一个新世界，其中我们对 Ledger 表有一个 SELECT 查询，然后调用 Catalog 微服务，这反过来触发对 Albums 表的 SELECT 语句，如图 3-7 所示。

在这种情况下，如果此操作的整体延迟没有增加，我会非常惊讶。这可能在这个特定案例中不是重大问题，因为此报告是每月生成的，因此可以积极缓存（我们将在「缓存」一节（第 435 页）更详细地探讨这个主题）。但如果这是频繁操作，那可能更有问题。我们可以通过允许在 Catalog 微服务中批量查找 SKU，或者甚至通过在本地缓存所需的专辑信息，来减轻这种延迟增加的可能影响。

### 数据完整性

数据库可以用于确保我们数据的完整性。回到图 3-6，Ledger 表和 Album 表在同一个数据库中，我们可以（而且可能会）在 Ledger 表的行和 Album 表之间定义外键关系。这将确保我们始终能够从 Ledger 表中的记录导航回有关售出专辑的信息，因为我们无法在 Ledger 中引用时从 Album 表中删除记录。

随着这些表现在存在于不同的数据库中，我们不再有数据模型完整性的强制。没有什么能阻止我们从 Album 表中删除一行，当我们在尝试准确找出售出的是什么物品时会造成问题。

在某种程度上，你只需要习惯这样一个事实：你不能再依赖数据库来强制实体间关系的完整性。显然，对于保留在单个数据库内的数据，这不是问题。

有许多变通方法，尽管「应对模式」可能是我们处理这个问题的方式的更好术语。我们可以在 Album 表中使用软删除，这样我们实际上不会删除记录，而只是将其标记为已删除。另一个选项可能是在进行销售时将专辑名称复制到 Ledger 表中，但我们必须解决我们想要如何处理专辑名称更改的同步。

### 事务

我们中的许多人已经习惯于依赖从在事务中管理数据获得的保证。基于这种确定性，我们以某种方式构建了应用，知道我们可以依赖数据库为我们处理许多事情。然而，一旦我们开始跨多个数据库拆分数据，我们就失去了我们习惯的 ACID 事务的安全性。（我将在第 6 章更详细地解释 ACID 首字母缩略词并讨论 ACID 事务。）

对于从所有状态更改都可以在单个事务边界内管理的系统迁移的人来说，向分布式系统的转变可能是一个冲击，通常的反应是寻求实现分布式事务以重新获得 ACID 事务在更简单架构中给我们的保证。不幸的是，正如我们将在「数据库事务」一节（第 175 页）深入介绍的，分布式事务不仅实现复杂，即使做得好，它们实际上也不能给我们期望的与更窄范围数据库事务相同的保证。

正如我们在「Saga」一节（第 182 页）继续探讨的，有替代（且更可取）的机制来管理跨多个微服务的状态更改，而不是分布式事务，但它们带来了新的复杂性来源。与数据完整性一样，我们必须接受这样一个事实：出于可能非常好的理由拆分我们的数据库，我们将遇到一组新问题。

### 工具

出于多种原因，更改数据库是困难的，其中之一是可用于让我们轻松进行更改的工具仍然有限。对于代码，我们的 IDE 中有重构工具，我们还有我们正在更改的系统本质上是无状态的额外好处。对于数据库，我们正在更改的东西有状态，我们也缺乏良好的重构类型工具。

有许多工具可以帮助你管理关系数据库模式更改的过程，但大多数遵循相同的模式。每个模式更改在版本控制的增量脚本中定义。然后这些脚本以幂等方式按严格顺序运行。Rails migrations 以这种方式工作，DBDeploy 也是如此，这是我多年前帮助创建的一个工具。

如今，我建议人们使用 Flyway 或 Liquibase 来实现相同的结果，如果他们还没有以这种方式工作的工具。

### 报告数据库

作为从单体应用提取微服务的一部分，我们也拆分了数据库，因为我们希望隐藏对我们内部数据存储的访问。通过隐藏对数据库的直接访问，我们能够更好地创建稳定的接口，使独立可部署性成为可能。不幸的是，当我们确实有合法用例需要从多个微服务访问数据，或者当该数据在数据库中而不是通过 REST API 之类的东西提供更好时，这给我们带来了问题。

使用报告数据库（Reporting Database），我们改为创建一个专为外部访问设计的专用数据库，我们让微服务负责将数据从内部存储推送到可外部访问的报告数据库，如图 3-8 所示。

```
Microservice ──[push data]──▶ Reporting Database ◀──[query]── External Consumers
```

**图 3-8. 报告数据库模式概述**

报告数据库允许我们隐藏内部状态管理，同时仍然以数据库的形式呈现数据——这可能非常有用。例如，你可能希望允许人们运行临时定义的 SQL 查询、进行大规模连接，或使用期望访问 SQL 端点的现有工具链。报告数据库是解决这个问题的好方案。

这里有两个关键点需要强调。首先，我们仍然要实践信息隐藏。所以我们应该在报告数据库中只暴露最少量的数据。这意味着报告数据库中的内容可能只是微服务存储的数据的子集。然而，由于这不是直接映射，它创造了为报告数据库提出完全针对消费者需求量身定制的模式设计的机会——这可能涉及使用 radically 不同的模式，或者甚至完全不同的数据库技术。

第二个关键点是，报告数据库应该像任何其他微服务端点一样对待，微服务维护者的职责是确保即使微服务更改其内部实现细节，此端点的兼容性也得到维护。从内部状态到报告数据库的映射是开发微服务本身的人的责任。

## 总结

所以，总结一下，当着手将功能从单体架构迁移到微服务架构时，你必须对你期望实现什么有清晰的理解。这个目标将塑造你如何进行工作，也将帮助你理解你是否在正确的方向上移动。

迁移应该是渐进的。进行更改，推出该更改，评估它，然后再来。即使拆分一个微服务的行为本身也可以分解为一系列小步骤。

如果你想更详细地探讨本章中的任何概念，我写的另一本书《Monolith to Microservices》（O'Reilly）是对这个主题的深入探讨。

本章的大部分内容在概述方面有些高层次。然而，在下一章中，当我们看看微服务如何相互通信时，我们将开始变得更加技术性。
