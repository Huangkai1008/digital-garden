# 第1章 什么是微服务？

自我撰写本书第一版以来的五年多时间里，微服务（Microservices）已成为越来越受欢迎的架构选择。我不能将随后的人气爆发归功于自己，但急于采用微服务架构意味着，虽然我之前提出的许多想法如今已得到验证，新的想法也同时涌现，而早期的实践则逐渐失宠。因此，是时候再次提炼微服务架构的精髓，并突出使微服务发挥作用的核心概念。

本书整体旨在全面概述微服务对软件交付各个方面的影响。作为开篇，本章将介绍微服务背后的核心思想、将我们带到这里的既有成果，以及这些架构被广泛使用的一些原因。

## 微服务概览

微服务是围绕业务领域建模的、可独立发布的服务。服务封装功能，并通过网络使其可被其他服务访问——你用这些构建块来构建更复杂的系统。一个微服务可能代表库存，另一个代表订单管理，再一个代表物流，但它们共同构成整个电商系统。

微服务是一种架构选择，专注于为你提供多种解决问题的选项。

它们是面向服务架构（Service-Oriented Architecture，SOA）的一种类型，但对服务边界应如何划分持明确立场，且独立可部署性（Independent Deployability）是关键。它们与技术无关，这是其优势之一。

从外部看，单个微服务被视为黑盒。它在一个或多个网络端点（例如队列或 REST API）上承载业务功能，如图 1-1 所示，使用最合适的协议。消费者——无论是其他微服务还是其他类型的程序——通过这些网络端点访问该功能。内部实现细节（如服务使用的技术或数据存储方式）完全对外隐藏。这意味着微服务架构在大多数情况下避免使用共享数据库；相反，每个微服务在需要时封装自己的数据库。

```
┌─────────────────────────────────────────────────────────────┐
│                      Microservice                            │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐ │
│  │   REST API  │  │   Topic     │  │  Internal Logic & DB  │ │
│  └──────┬──────┘  └──────┬──────┘  └─────────────────────┘ │
└─────────┼────────────────┼─────────────────────────────────┘
          │                │
          ▼                ▼
     Consumers         Consumers
```

**图 1-1. 微服务通过 REST API 和主题暴露其功能**

微服务拥抱信息隐藏（Information Hiding）的概念。[^1] 信息隐藏意味着尽可能多地将信息隐藏在组件内部，通过外部接口尽可能少地暴露。这允许在易于更改的内容与难以更改的内容之间进行清晰分离。只要微服务暴露的网络接口不以向后不兼容的方式改变，对外部隐藏的实现可以自由更改。微服务边界内的更改（如图 1-1 所示）不应影响上游消费者，从而实现功能的独立可发布性。这对于允许我们独立地开发微服务并按需发布至关重要。

在讨论隐藏内部实现细节时，不能不提及六边形架构（Hexagonal Architecture）模式，由 Alistair Cockburn 首次详细阐述。[^2] 该模式描述了将内部实现与其外部接口分离的重要性，其思想是你可能希望通过不同类型的接口与相同功能交互。我将微服务画成六边形，部分是为了将它们与「普通」服务区分开来，也是对这一既有成果的致敬。

### 面向服务架构与微服务是一回事吗？

面向服务架构（SOA）是一种设计方法，其中多个服务协作提供某种最终能力集。（这里的服务通常指完全独立的操作系统进程。）这些服务之间的通信通过跨网络调用发生，而不是进程边界内的方法调用。

SOA 作为一种方法来应对大型单体应用的挑战。该方法旨在促进软件的可重用性；例如，两个或多个最终用户应用可以使用相同的服务。SOA 旨在使维护或重写软件更容易，因为理论上我们可以在没有人知道的情况下用一个服务替换另一个服务，只要服务的语义变化不大。

SOA 本质上是一个合理的想法。然而，尽管付出了许多努力，业界在如何做好 SOA 方面缺乏良好共识。在我看来，业界大多未能全面看待问题，也未能提出令人信服的替代方案，以对抗该领域各种供应商所设定的叙事。

许多归咎于 SOA 的问题实际上是通信协议（如 SOAP）、供应商中间件、缺乏关于服务粒度的指导，或关于在何处拆分系统的错误指导等问题。愤世嫉俗者可能会说，供应商利用（在某些情况下推动）SOA 运动来销售更多产品，而这些产品最终破坏了 SOA 的目标。

我见过很多 SOA 的例子，团队努力使服务更小，但他们仍然将所有东西耦合到数据库，并且必须一起部署。面向服务？是的。但这不是微服务。

微服务方法从实际使用中涌现，利用我们对系统和架构的更好理解来做好 SOA。你应该将微服务视为 SOA 的一种具体方法，就像极限编程（Extreme Programming，XP）或 Scrum 是敏捷软件开发的一种具体方法一样。

## 微服务的核心概念

在探索微服务时，必须理解几个核心思想。鉴于某些方面经常被忽视，进一步探索这些概念至关重要，以确保你理解使微服务发挥作用的关键。

### 独立可部署性

独立可部署性是指我们可以对微服务进行更改、部署并将该更改发布给用户，而无需部署任何其他微服务。更重要的是，不仅仅是我们能够做到这一点；而是这实际上是你管理系统中部署的方式。这是你作为默认发布方法采用的纪律。这是一个简单的想法，但在执行中却很复杂。

::: tip
如果你从本书和微服务概念中只记住一件事，那应该是：确保你拥抱微服务的独立可部署性概念。养成将单个微服务的更改部署和发布到生产环境而不必部署任何其他东西的习惯。由此，许多好事将随之而来。
:::

为确保独立可部署性，我们需要确保微服务松散耦合：我们必须能够更改一个服务而不必更改任何其他东西。这意味着我们需要服务之间明确、定义良好且稳定的契约。某些实现选择使这变得困难——例如，共享数据库尤其成问题。

独立可部署性本身显然非常有价值。但要实现独立可部署性，你必须做对许多其他事情，而这些事情又有自己的好处。因此，你也可以将关注独立可部署性视为一种强制函数——通过将此作为结果来关注，你将获得许多附带好处。

### 围绕业务领域建模

像领域驱动设计（Domain-Driven Design，DDD）这样的技术可以让你更好地构建代码以表示软件所运营的真实世界领域。[^3] 对于微服务架构，我们使用相同的想法来定义我们的服务边界。通过围绕业务领域建模服务，我们可以更容易地推出新功能，并以不同方式重新组合微服务以向用户交付新功能。

推出需要更改多个微服务的功能是昂贵的。你需要协调每个服务的工作（可能跨越不同团队），并仔细管理这些服务新版本的部署顺序。这比在单个服务内（或在单体中）进行相同更改需要更多工作。因此，我们希望找到方法使跨服务更改尽可能少。

我经常看到分层架构，如图 1-2 中典型的三层架构所示。这里，架构中的每一层代表不同的服务边界，每个服务边界基于相关的技术功能。如果在此示例中我只需要更改表示层，那将相当高效。然而，经验表明，功能更改通常跨越这些类型架构的多个层——需要更改表示层、应用层和数据层。如果架构比图 1-2 中的简单示例更加分层，这个问题会加剧；通常每一层会进一步拆分为更多层。

通过使我们的服务成为业务功能的端到端切片，我们确保架构的安排使业务功能的更改尽可能高效。可以说，对于微服务，我们做出了优先考虑业务功能高内聚而非技术功能高内聚的决定。

```
┌─────────────────────────────────────────────────────────────┐
│                    Presentation Layer                        │
├─────────────────────────────────────────────────────────────┤
│                   Application Layer                          │
├─────────────────────────────────────────────────────────────┤
│                      Data Layer                              │
└─────────────────────────────────────────────────────────────┘
```

### 拥有自己的状态

我看到人们最难理解的事情之一是微服务应该避免使用共享数据库的想法。如果微服务想要访问另一个微服务持有的数据，它应该向该微服务请求数据。这使微服务能够决定什么是共享的、什么是隐藏的，从而允许我们清晰分离可以自由更改的功能（我们的内部实现）与我们希望不常更改的功能（消费者使用的外部契约）。

如果我们想使独立可部署性成为现实，我们需要确保限制对微服务的向后不兼容更改。如果我们破坏了与上游消费者的兼容性，我们将迫使它们也进行更改。在微服务的内部实现细节与外部契约之间进行清晰划分，有助于减少对向后不兼容更改的需求。

在微服务中隐藏内部状态类似于面向对象（OO）编程中的封装实践。OO 系统中数据的封装是信息隐藏的实例。

::: warning
除非你真的需要，否则不要共享数据库。即使需要，也要尽一切可能避免。在我看来，共享数据库是如果你试图实现独立可部署性所能做的最糟糕的事情之一。
:::

如前一节所述，我们希望将我们的服务视为业务功能的端到端切片，在适当的情况下封装用户界面（UI）、业务逻辑和数据。这是因为我们希望减少更改业务相关功能所需的工作。以这种方式封装数据和行为为我们提供了业务功能的高内聚。通过隐藏支持我们服务的数据库，我们还确保减少耦合。我们将在第 2 章回到耦合和内聚。

### 规模

「微服务应该有多大？」是我最常听到的问题之一。考虑到「微」这个词就在名称中，这并不令人惊讶。然而，当你深入探讨使微服务作为架构类型发挥作用时，规模概念实际上是最不有趣的方面之一。

你如何衡量规模？通过计算代码行数？这对我来说没有多大意义。在 Java 中可能需要 25 行代码的东西可以用 10 行 Clojure 编写。这并不是说 Clojure 比 Java 更好或更差；有些语言只是比其他语言更具表现力。

Thoughtworks 的技术总监 James Lewis 曾说过「微服务应该和我的头一样大」。乍一看，这似乎不太有帮助。毕竟，James 的头到底有多大？这一说法背后的理由是，微服务应保持在易于理解的规模。当然，挑战在于不同人理解事物的能力并不总是相同，因此你需要根据自己的判断来决定什么规模适合你。经验丰富的团队可能能够比另一个团队更好地管理更大的代码库。所以也许最好将 James 的引用理解为「微服务应该和你的头一样大」。

我认为我对于「规模」在微服务方面有任何意义最接近的理解，是《微服务模式》（Microservice Patterns，Manning 出版）作者 Chris Richardson 曾经说过的话——微服务的目标是拥有「尽可能小的接口」。这再次与信息隐藏的概念一致，但它确实代表了对「微服务」一词寻找最初并不存在的意义的尝试。当该术语首次用于定义这些架构时，至少最初的重点并不 specifically 在接口的规模上。

归根结底，规模的概念高度依赖于上下文。与在系统上工作了 15 年的人交谈，他们会觉得拥有 10 万行代码的系统非常容易理解。询问刚加入项目的人的意见，他们会觉得它太大了。同样，询问刚刚开始微服务转型、可能只有 10 个或更少微服务的公司，你会得到与类似规模、微服务多年来已成为常态、现在有数百个微服务的公司不同的答案。

我敦促人们不要担心规模。当你刚开始时，更重要的是关注两件事。首先，你能处理多少个微服务？随着服务增多，系统的复杂性会增加，你需要学习新技能（也许采用新技术）来应对。向微服务的迁移将引入新的复杂性来源，带来所有相关挑战。因此，我强烈主张逐步迁移到微服务架构。其次，你如何定义微服务边界以从中获得最大收益，而不让一切变成可怕的耦合混乱？这些主题在你开始旅程时更加重要。

### 灵活性

James Lewis 的另一句名言是「微服务为你购买选项」。Lewis 用词是刻意的——它们购买选项。它们有成本，你必须决定成本是否值得你想要的选项。在多个维度上——组织、技术、规模、健壮性——由此产生的灵活性可能非常有吸引力。

我们不知道未来会怎样，所以我们希望有一个理论上可以帮助我们解决可能面临的任何问题的架构。在保持选项开放与承担此类架构成本之间找到平衡可能是一门真正的艺术。

将采用微服务视为不像拨动开关，而更像转动旋钮。当你调高旋钮、拥有更多微服务时，你增加了灵活性。但你也可能增加了痛点。这是我又一个强烈主张逐步采用微服务的理由。通过逐渐调高旋钮，你能够更好地评估过程中的影响，并在需要时停止。

### 架构与组织的对齐

MusicCorp 是一家在线销售 CD 的电商公司，使用图 1-2 所示的简单三层架构。我们决定将 MusicCorp 强行带入 21 世纪，作为其中的一部分，我们正在评估现有系统架构。我们有一个基于 Web 的 UI、一个单体后端形式的业务逻辑层，以及传统数据库中的数据存储。这些层通常由不同的团队拥有。我们将在整本书中回到 MusicCorp 的考验和磨难。

我们想对功能进行简单更新：我们想让客户指定他们最喜欢的音乐类型。此更新需要我们更改 UI 以显示类型选择 UI，更改后端服务以允许将类型呈现给 UI 并允许更改值，以及更改数据库以接受此更改。这些更改需要由每个团队管理并按正确顺序部署，如图 1-3 所示。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  UI Team    │────▶│ Backend     │────▶│  DB Team    │
│  Change     │     │ Team Change │     │  Change     │
└─────────────┘     └─────────────┘     └─────────────┘
```

**图 1-3. 跨所有三层进行更改更复杂**

这种架构并不坏。所有架构最终都会围绕一组目标进行优化。三层架构如此普遍的部分原因是它是通用的——每个人都听说过。所以选择你可能在其他地方见过的常见架构的倾向，往往是我们不断看到这种模式的一个原因。但我认为我们一次又一次看到这种架构的最大原因是它基于我们如何组织团队。

著名的康威定律（Conway's Law）指出：

> 设计系统的组织……受到限制，只能产生这些组织沟通结构的副本。
> —Melvin Conway，「委员会如何发明？」

三层架构是这一定律在起作用的一个好例子。过去，IT 组织分组人员的主要方式是根据他们的核心能力：数据库管理员与数据库管理员在一个团队；Java 开发人员与 Java 开发人员在一个团队；前端开发人员（如今了解 JavaScript 和原生移动应用开发等「异国情调」的东西）在另一个团队。我们根据核心能力对人进行分组，所以我们创建可以与这些团队对齐的 IT 资产。

这解释了为什么这种架构如此普遍。它并不坏；它只是围绕一组力量进行了优化——我们传统上如何对人进行分组，围绕熟悉度。但力量已经改变。我们对软件的期望已经改变。我们现在将人组织成多技能团队以减少交接和孤岛。我们想比以往更快地交付软件。这促使我们对组织团队的方式做出不同的选择，所以我们按照拆分系统的方式组织他们。

我们被要求对系统进行的大多数更改与业务功能的更改有关。但在图 1-3 中，我们的业务功能实际上分布在所有三层，增加了功能更改跨越层的可能性。这是一种具有相关技术高内聚但业务功能低内聚的架构。如果我们想使更改更容易，我们需要改变我们分组代码的方式，选择业务功能的内聚而非技术。每个服务可能最终包含这三层的混合，也可能不包含，但这是本地服务实现关注点。

让我们将其与图 1-4 所示的潜在替代架构进行比较。与水平分层架构和组织不同，我们沿着垂直业务线分解组织架构。在这里，我们看到一个专门团队对客户档案方面的更改负有端到端的全部责任，这确保在此示例中更改的范围限于一个团队。

作为实现，这可以通过由档案团队拥有的单个微服务来实现，该微服务暴露 UI 以允许客户更新其信息，客户状态也存储在此微服务中。最喜欢的类型与给定客户相关联，因此此更改更加本地化。在图 1-5 中，我们还显示了从 Catalog 微服务获取可用类型列表，这可能已经存在。我们还看到新的 Recommendation 微服务访问我们最喜欢的类型信息，这可以很容易地在后续版本中跟进。

```
┌─────────────────────────────────────────────────────────────┐
│              Customer Profile Team (Stream-Aligned)          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐   │
│  │  UI Slice   │  │  Backend    │  │  Data               │   │
│  └─────────────┘  └─────────────┘  └─────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**图 1-4. UI 被拆分，由同时管理支持 UI 的服务端功能的团队拥有**

```
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│ Customer Service  │────▶│ Catalog Service  │     │ Recommendation   │
│ (Favorite Genre)  │     │ (Genre List)     │     │ Service           │
└──────────────────┘     └──────────────────┘     └──────────────────┘
```

**图 1-5. 专用的 Customer 微服务可以更容易地记录客户最喜欢的音乐类型**

在这种情况下，我们的 Customer 微服务封装了三层中的薄切片——它有一点 UI、一点应用逻辑和一点数据存储。我们的业务领域成为驱动系统架构的主要力量，希望使更改更容易，也使我们更容易将团队与组织内的业务线对齐。

::: tip
通常，UI 不是直接由微服务提供的，但即使如此，我们仍期望与此功能相关的 UI 部分由 Customer Profile Team 拥有，如图 1-4 所示。团队拥有用户面向功能的端到端切片的概念正在获得关注。《团队拓扑》（Team Topologies）[^4] 一书引入了流对齐团队（Stream-Aligned Team）的概念，体现了这一概念：

> 流对齐团队是与单一、有价值的工作流对齐的团队……团队被授权尽可能快速、安全、独立地构建和交付客户或用户价值，而无需将工作的部分交给其他团队执行。
:::

### 关于「假」公司的说明

## 单体

我们讨论了微服务，但微服务最常被讨论为单体架构的替代架构方法。为了更清晰地区分微服务架构，并帮助你更好地理解微服务是否值得考虑，我还应该讨论我所说的单体到底是什么意思。

当我在整本书中谈论单体时，我主要指的是部署单元。当系统中的所有功能必须一起部署时，我将其视为单体。[^5] 可以说，多种架构符合这一定义，但我将讨论我最常看到的那些：单进程单体、模块化单体和分布式单体。

### 单进程单体

讨论单体时最常见的例子是图 1-6 中所示的所有代码作为单个进程部署的系统。出于健壮性或扩展原因，你可能拥有此进程的多个实例，但基本上所有代码都打包在单个进程中。实际上，这些单进程系统本身可以是简单的分布式系统，因为它们几乎总是最终从数据库读取数据或将数据存储到数据库，或向 Web 或移动应用呈现信息。

```
┌─────────────────────────────────────────────────────────────┐
│                    Single Process                            │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  All Application Code                                │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

**图 1-6. 在单进程单体中，所有代码打包在单个进程中**

虽然这符合大多数人对经典单体的理解，但我遇到的大多数系统比这更复杂。你可能有两个或多个紧密耦合的单体，可能还有一些供应商软件混在其中。

经典的单进程单体部署对许多组织来说是有意义的。Ruby on Rails 的创建者 David Heinemeier Hansson 有效地论证了这种架构对小型组织是有意义的。即使随着组织成长，单体也可能随之成长，这就引出了模块化单体。

### 模块化单体

作为单进程单体的子集，模块化单体是一种变体，其中单进程由独立的模块组成。每个模块可以独立开发，但所有模块仍需要组合在一起进行部署，如图 1-7 所示。将软件分解为模块的概念并不新鲜；模块化软件起源于 1970 年代结构化编程方面的工作，甚至更早。尽管如此，这仍然是一种我不够多组织充分采用的方法。

```
┌─────────────────────────────────────────────────────────────┐
│                    Single Process                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │ Module A │  │ Module B │  │ Module C │  │ Module D │    │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘    │
└─────────────────────────────────────────────────────────────┘
```

**图 1-7. 在模块化单体中，进程内的代码被划分为模块**

对于许多组织来说，模块化单体可以是一个很好的选择。如果模块边界定义良好，它可以允许高度的并行工作，同时通过拥有更简单的部署拓扑来避免更分布式微服务架构的挑战。Shopify 是使用这种技术作为微服务分解替代方案的组织的一个很好的例子，这对该公司似乎效果很好。[^6]

模块化单体的挑战之一是数据库往往缺乏我们在代码层面发现的分解，如果你将来想拆分单体，会导致重大挑战。我见过一些团队试图通过让数据库沿与模块相同的线分解来进一步推动模块化单体的想法，如图 1-8 所示。

### 分布式单体

> 分布式系统是这样一种系统：你甚至不知道存在的计算机的故障会使你自己的计算机无法使用。[^7]
> —Leslie Lamport

分布式单体是由多个服务组成的系统，但无论出于何种原因，整个系统必须一起部署。分布式单体可能符合 SOA 的定义，但往往无法兑现 SOA 的承诺。根据我的经验，分布式单体具有分布式系统的所有缺点和单进程单体的缺点，而没有足够的任一方的好处。在工作中遇到许多分布式单体在很大程度上影响了我对微服务架构的兴趣。

分布式单体通常出现在对信息隐藏和业务功能内聚等概念关注不够的环境中。相反，高度耦合的架构导致更改在服务边界上蔓延，看似局部范围的看似无辜的更改会破坏系统的其他部分。

### 单体与交付争用

随着越来越多的人在同一个地方工作，他们会相互妨碍——例如，不同的开发人员想要更改同一段代码，不同的团队想要在不同时间推送功能上线（或延迟部署），以及关于谁拥有什么、谁做决定的困惑。大量研究已经证明了所有权线混乱的挑战。[^8] 我将此问题称为交付争用（Delivery Contention）。

拥有单体并不意味着你一定会面临交付争用的挑战，就像拥有微服务架构并不意味着你永远不会面临这个问题。但微服务架构确实为你提供了更具体的边界，可以在系统中围绕这些边界划定所有权线，在减少此问题时给你更大的灵活性。

## 单体的优势

一些单体，如单进程或模块化单体，也有许多优势。它们更简单的部署拓扑可以避免与分布式系统相关的许多陷阱。这可以导致更简单的开发工作流，监控、故障排除和端到端测试等活动也可以大大简化。

单体还可以简化单体本身内的代码重用。如果我们想在分布式系统中重用代码，我们需要决定是复制代码、拆分库还是将共享功能推入服务。对于单体，我们的选择要简单得多，许多人喜欢这种简单性——所有代码都在那里；直接使用！

不幸的是，人们已经开始将单体视为要避免的东西——视为 inherently 有问题的东西。我遇到过多个将「单体」一词与「遗留」同义的人。这是一个问题。单体架构是一种选择，而且是有效的选择。我更进一步说，在我看来，它是作为架构风格的明智默认选择。换句话说，我正在寻找被说服使用微服务的理由，而不是寻找不使用它们的理由。

如果我们陷入系统地破坏单体作为交付软件可行选择的陷阱，我们就有风险对自己或软件用户做错事。

## 使能技术

正如我之前提到的，我认为当你刚开始使用微服务时不需要采用大量新技术。事实上，这可能适得其反。相反，随着你扩大微服务架构，你应该不断寻找由日益分布式系统引起的问题，然后寻找可能帮助的技术。

也就是说，技术在微服务作为概念的采用中发挥了重要作用。理解可帮助你充分利用此架构的可用工具，将是任何微服务实施成功的关键部分。事实上，我甚至可以说，微服务需要对支持技术的理解达到这样的程度，以至于逻辑架构和物理架构之间的先前区分可能存在问题——如果你参与帮助塑造微服务架构，你需要对这两个世界有广泛的理解。

我们将在后续章节中详细探讨其中许多技术，但在此之前，让我们简要介绍一些如果你决定使用微服务可能帮助你的使能技术。

### 日志聚合与分布式追踪

随着你管理的进程数量增加，可能难以理解你的系统在生产环境中的行为。这反过来可能使故障排除更加困难。我们将在第 10 章更深入地探讨这些想法，但至少，我强烈主张将日志聚合系统的实施作为采用微服务架构的先决条件。

::: tip
在开始使用微服务时，要谨慎承担过多新技术。也就是说，日志聚合工具如此重要，你应该将其视为采用微服务的先决条件。
:::

这些系统允许你从所有服务收集和聚合日志，为你提供一个中心位置，可以从中分析日志，甚至使其成为主动告警机制的一部分。该领域的许多选项迎合各种情况。我因几个原因非常喜欢 Humio，但主要公有云供应商提供的简单日志服务可能足以让你入门。

你可以通过实施关联 ID（Correlation ID）使这些日志聚合工具更有用，其中单个 ID 用于一组相关的服务调用——例如，可能由于用户交互而触发的调用链。通过将此 ID 作为每个日志条目的一部分记录，隔离与给定调用流相关的日志变得容易得多，这反过来使故障排除容易得多。

随着系统复杂性增长，考虑允许你更好地探索系统正在做什么的工具变得至关重要，提供跨多个服务分析追踪、检测瓶颈和向系统提问的能力，而这些问题你一开始可能不知道自己会想问。开源工具可以提供其中一些功能。Jaeger 就是一个例子，它专注于等式的分布式追踪方面。

但像 Lightstep 和 Honeycomb（如图 1-9 所示）这样的产品将这些想法更进一步。它们代表超越传统监控方法的新一代工具，使探索运行系统的状态容易得多。你可能已经有更传统的工具，但你真的应该看看这些产品提供的能力。它们从零开始构建，以解决微服务架构运营商必须处理的问题。

**图 1-9. Honeycomb 中显示的分布式追踪，允许你识别跨多个微服务的操作时间花在哪里**

### 容器与 Kubernetes

理想情况下，你希望每个微服务实例在隔离中运行。这确保一个微服务中的问题不会影响另一个微服务——例如，通过耗尽所有 CPU。虚拟化是在现有硬件上创建隔离执行环境的一种方式，但当我们考虑微服务的大小时，普通虚拟化技术可能相当沉重。另一方面，容器提供了一种更轻量级的方式来为服务实例提供隔离执行，导致新容器实例的启动时间更快，同时对许多架构来说更具成本效益。

在你开始使用容器后，你还会意识到你需要某种东西来允许你在大量底层机器上管理这些容器。像 Kubernetes 这样的容器编排平台正是这样做的，允许你以提供服务所需的健壮性和吞吐量的方式分发容器实例，同时允许你有效利用底层机器。我们将在第 8 章探讨操作隔离、容器和 Kubernetes 的概念。

::: tip
不要觉得需要急于采用 Kubernetes，甚至容器。它们绝对比更传统的部署技术提供显著优势，但如果你只有少数微服务，它们的采用很难证明是合理的。当管理部署的开销开始成为重大头痛时，开始考虑服务的容器化和 Kubernetes 的使用。但如果你最终这样做，尽你所能确保别人为你运行 Kubernetes 集群，也许通过使用公有云提供商的托管服务。自己运行 Kubernetes 集群可能是大量工作！
:::

### 流式处理

虽然使用微服务我们正在远离单体数据库，但我们仍然需要找到在微服务之间共享数据的方法。与此同时，组织希望从批量报告操作转向更实时的反馈，使他们能够更快地做出反应。因此，允许轻松流式传输和处理通常可能是大量数据的产品在使用微服务架构的人中变得流行。

对于许多人来说，Apache Kafka 已成为微服务环境中流式传输数据的实际选择，而且有充分的理由。消息持久性、压缩以及扩展到处理大量消息的能力等功能可能非常有用。Kafka 已经开始以 KSQLDB 的形式添加流处理能力，但你也可以将其与 Apache Flink 等专用流处理解决方案一起使用。Debezium 是一个开源工具，旨在帮助将现有数据源的数据通过 Kafka 流式传输，帮助确保传统数据源可以成为基于流的架构的一部分。我们将在第 4 章看看流技术如何在微服务集成中发挥作用。

### 公有云与无服务器

公有云提供商，或更具体地说三大提供商——Google Cloud、Microsoft Azure 和 Amazon Web Services（AWS）——为管理你的应用提供大量托管服务和部署选项。随着微服务架构增长，越来越多的工作将推入操作空间。公有云提供商提供大量托管服务，从托管数据库实例或 Kubernetes 集群到消息代理或分布式文件系统。通过使用这些托管服务，你将大量工作外包给 arguably 更能处理这些任务的第三方。

公有云产品中特别令人感兴趣的是无服务器（Serverless）旗下的产品。这些产品隐藏底层机器，允许你在更高的抽象级别工作。无服务器产品的例子包括消息代理、存储解决方案和数据库。函数即服务（Function as a Service，FaaS）平台特别令人感兴趣，因为它们为代码部署提供了很好的抽象。与其担心需要多少服务器来运行你的服务，你只需部署代码并让底层平台按需处理启动代码实例。我们将在第 8 章更详细地了解无服务器。

## 微服务的优势

微服务的优势多种多样。其中许多好处可以归功于任何分布式系统。然而，微服务往往在更大程度上实现这些好处，主要是因为它们在定义服务边界的方式上采取了更明确的立场。通过将信息隐藏和领域驱动设计的概念与分布式系统的力量相结合，微服务可以帮助在分布式架构的其他形式之上实现显著收益。

### 技术异构性

对于由多个协作微服务组成的系统，我们可以决定在每个微服务中使用不同的技术。这使我们能够为每项工作选择正确的工具，而不是必须选择通常最终成为最低公分母的更加标准化、一刀切的方法。

如果我们系统的某部分需要提高性能，我们可能决定使用能够更好地达到所需性能水平的不同技术栈。我们还可能决定，对于系统的不同部分，我们存储数据的方式需要改变。例如，对于社交网络，我们可能将用户的交互存储在面向图的数据库中，以反映社交图的高度互连性质，但也许用户发布的帖子可以存储在面向文档的数据存储中，产生如图 1-10 所示的异构架构。

**图 1-10. 微服务可以让你更容易地采用不同技术**

使用微服务，我们还能够更快地采用技术并理解新进展可能如何帮助我们。尝试和采用新技术的最大障碍之一是与之相关的风险。对于单体应用，如果我想尝试新的编程语言、数据库或框架，任何更改都会影响我的大部分系统。对于由多个服务组成的系统，我有多个新地方可以尝试新技术。我可以选择一个可能风险最低的微服务并在那里使用该技术，知道我可以限制任何潜在的负面影响。许多组织发现这种更快吸收新技术的能力是一个真正的优势。

采用多种技术当然会带来开销。一些组织选择对语言选择施加一些约束。例如，Netflix 和 Twitter 主要使用 Java 虚拟机（JVM）作为平台，因为这些公司对该系统的可靠性和性能有很好的理解。他们还为 JVM 开发库和工具，使大规模操作容易得多，但对 JVM 特定库的依赖使非基于 Java 的服务或客户端更加困难。但 Twitter 和 Netflix 都没有对所有工作只使用一种技术栈。

内部技术实现对消费者隐藏的事实也可以使升级技术更容易。你的整个微服务架构可能基于 Spring Boot，例如，但你可以只为单个微服务更改 JVM 版本或框架版本，使管理升级风险更容易。

### 健壮性

提高应用健壮性的关键概念是隔舱（Bulkhead）。系统的组件可能会失败，但只要该失败不会级联，你就可以隔离问题，系统的其余部分可以继续工作。服务边界成为你明显的隔舱。在单体服务中，如果服务失败，一切都会停止工作。对于单体系统，我们可以在多台机器上运行以减少故障机会，但对于微服务，我们可以构建处理某些组成服务完全故障并相应降低功能的系统。

然而，我们需要小心。为确保我们的微服务系统能够正确拥抱这种改进的健壮性，我们需要理解分布式系统必须应对的新故障来源。网络可以并且将会失败，机器也会。我们需要知道如何处理此类故障以及这些故障对软件最终用户的影响（如果有）。我 certainly 与一些团队合作过，由于没有足够认真地对待这些关注点，他们在迁移到微服务后最终得到了一个不太健壮的系统。

### 扩展

对于大型单体服务，我们需要一起扩展所有东西。也许我们整体系统的一小部分在性能上受到限制，但如果该行为被锁定在巨大的单体应用中，我们需要将一切作为一块来处理扩展。对于较小的服务，我们可以只扩展需要扩展的服务，允许我们在更小、功能较弱的硬件上运行系统的其他部分，如图 1-11 所示。

**图 1-11. 你可以针对只需要扩展的微服务进行扩展**

Gilt 是一家在线时尚零售商，正是出于这个原因采用了微服务。从 2007 年开始使用单体 Rails 应用，到 2009 年 Gilt 的系统无法应对施加在其上的负载。通过拆分其系统的核心部分，Gilt 能够更好地应对流量峰值，如今它拥有 450 多个微服务，每个都在多台独立的机器上运行。

当采用 AWS 等提供的按需供应系统时，我们甚至可以对这些需要扩展的部分按需应用此扩展。这使我们能够更有效地控制成本。架构方法能够与几乎即时的成本节省如此密切相关的并不常见。

归根结底，我们可以以多种方式扩展我们的应用，微服务可以是其中的有效部分。我们将在第 13 章更详细地了解微服务的扩展。

### 部署便利性

对百万行单体应用的单行更改需要部署整个应用才能发布该更改。这可能是一次影响大、风险高的部署。实际上，由于可以理解的恐惧，此类部署最终很少发生。不幸的是，这意味着我们的更改在发布之间继续积累，直到进入生产的新版本应用有大量更改。发布之间的差异越大，我们出错的风险就越高！

对于微服务，我们可以对单个服务进行更改并独立于系统其余部分部署它。这使我们能够更快地部署代码。如果出现问题，可以快速隔离到单个服务，使快速回滚容易实现。这也意味着我们可以更快地将新功能交付给客户。这是 Amazon 和 Netflix 等组织使用这些架构的主要原因之一——确保他们尽可能多地消除交付软件的障碍。

### 组织对齐

我们中的许多人经历过与大型团队和大型代码库相关的问题。当团队分布时，这些问题可能会加剧。我们也知道，在较小代码库上工作的较小团队往往更高效。微服务使我们能够更好地将架构与组织对齐，帮助我们最小化在任何一个代码库上工作的人数以达到团队规模和生产力的最佳点。微服务还允许我们随着组织变化而改变服务的所有权——使我们能够在未来保持架构与组织之间的对齐。

### 可组合性

分布式系统和面向服务架构的关键承诺之一是，我们为功能重用开辟了机会。使用微服务，我们允许以不同方式为不同目的消费我们的功能。当我们考虑消费者如何使用我们的软件时，这可能特别重要。

我们狭隘地考虑桌面网站或移动应用的时代已经过去。现在我们需要考虑我们可能想要以多种方式为 Web、原生应用、移动 Web、平板应用或可穿戴设备编织能力。随着组织从狭隘渠道的思维转向拥抱更全面的客户参与概念，我们需要能够跟上的架构。

对于微服务，想想我们在系统中打开可由外部方寻址的接缝。随着情况变化，我们可以以不同方式构建应用。对于单体应用，我通常有一个可以从外部使用的粗粒度接缝。如果我想将其拆分以获得更有用的东西，我需要一把锤子！

## 微服务的痛点

微服务架构带来许多好处，正如我们已经看到的。但它们也带来许多复杂性。如果你正在考虑采用微服务架构，重要的是你能够比较好坏。实际上，大多数微服务痛点可以归咎于分布式系统，因此很可能在分布式单体中和在微服务架构中同样明显。

我们将在本书其余部分深入探讨其中许多问题——事实上，我认为本书的大部分内容是关于处理拥有微服务架构的痛苦、苦难和恐怖。

### 开发体验

随着你拥有越来越多的服务，开发体验可能开始下降。像 JVM 这样更消耗资源的运行时可能限制可以在单个开发人员机器上运行的微服务数量。我可能可以在我的笔记本电脑上作为独立进程运行四五个基于 JVM 的微服务，但我能运行 10 或 20 个吗？很可能不能。即使使用不太消耗的运行时，你可以本地运行的东西也有限制，这不可避免地会开始关于当你不能在一台机器上运行整个系统时该怎么办的对话。如果你使用的是无法本地运行的云服务，这可能变得更加复杂。

极端解决方案可能涉及「在云中开发」，开发人员不再能够本地开发。我不喜欢这个，因为反馈周期可能受到很大影响。相反，我认为限制开发人员需要工作的系统部分范围可能是更直接的方法。然而，如果你想拥抱更多的「集体所有权」模型，其中任何开发人员都被期望在系统的任何部分工作，这可能会有问题。

### 技术过载

为支持微服务架构采用而涌现的新技术的 sheer 重量可能令人不知所措。我会诚实地说，其中很多技术只是被重新包装为「微服务友好」，但一些进步确实有助于处理这类架构的复杂性。然而，存在一种危险，即这种丰富的新玩具可能导致一种技术崇拜。我见过太多采用微服务架构的公司决定这也是引入大量新的、往往是陌生技术的最佳时机。

微服务可能确实给你每个微服务用不同编程语言编写、在不同运行时运行或使用不同数据库的选项——但这些都是选项，不是要求。你必须仔细平衡你使用的技术的广度和复杂性与多样化技术可能带来的成本。

当你开始采用微服务时，一些基本挑战是不可避免的：你需要花大量时间理解数据一致性、延迟、服务建模等问题。如果你在拥抱大量新技术的同时试图理解这些想法如何改变你对软件开发的思考方式，你会很难。还值得指出的是，试图理解所有这些新技术所占据的带宽将减少你实际向用户交付功能的时间。

当你（逐渐）增加微服务架构的复杂性时，在需要时引入新技术。当你有三个服务时，你不需要 Kubernetes 集群！除了确保你不会被这些新工具的复杂性压垮之外，这种逐渐增加还有一个额外的好处，即允许你获得无疑会随着时间的推移而出现的新的更好的做事方式。

### 成本

至少在短期内，你很可能会看到来自多个因素的成本增加。首先，你可能需要运行更多东西——更多进程、更多计算机、更多网络、更多存储和更多支持软件（这将产生额外的许可费）。

其次，你引入团队或组织的任何更改都会在短期内减慢你的速度。学习新想法并找出如何有效使用它们需要时间。在这段时间里，其他活动会受到影响。这将导致新功能交付的直接放缓或需要增加更多人来抵消这一成本。

根据我的经验，微服务对于主要关注降低成本的组织来说是一个糟糕的选择，因为成本削减心态——将 IT 视为成本中心而非利润中心——将不断阻碍从这种架构中获得最大收益。另一方面，如果你能使用这些架构接触更多客户或并行开发更多功能，微服务可以帮助你赚更多钱。所以微服务是推动更多利润的方式吗？也许。微服务是降低成本的方式吗？不太是。

### 报告

对于单体系统，你通常有一个单体数据库。这意味着想要一起分析所有数据的利益相关者，通常涉及跨数据的大型连接操作，有一个现成的模式可以针对其运行报告。他们可以直接针对单体数据库运行，也许针对只读副本，如图 1-12 所示。

**图 1-12. 直接在单体数据库上进行的报告**

对于微服务架构，我们拆散了这个单体模式。这并不意味着跨所有数据的报告需求已经消失；我们只是使其更加困难，因为现在我们的数据分散在多个逻辑隔离的模式中。

更现代的报告方法，例如使用流式传输允许对大量数据进行实时报告，可以与微服务架构很好地配合，但通常需要采用新想法和相关技术。或者，你可能只需要将数据从微服务发布到中央报告数据库（或也许不太结构化的数据湖）以允许报告用例。

### 监控与故障排除

对于标准单体应用，我们可以采用相当简单的方法进行监控。我们只需要担心少量机器，应用的故障模式有些二元——应用通常要么全部启动要么全部关闭。对于微服务架构，我们是否理解如果只是服务的单个实例关闭会有什么影响？

对于单体系统，如果我们的 CPU 长时间卡在 100%，我们知道这是一个大问题。对于有数十或数百个进程的微服务架构，我们能说同样的话吗？当只有一个进程卡在 100% CPU 时，我们需要在凌晨 3 点叫醒某人吗？

幸运的是，该领域有大量想法可以提供帮助。如果你想更详细地探索这个概念，我推荐 Cindy Sridharan 的《分布式系统可观测性》（Distributed Systems Observability，O'Reilly）作为优秀的起点，尽管我们也会在第 10 章自己看看监控和可观测性。

### 安全

对于单进程单体系统，我们的大部分信息在该进程内流动。现在，更多信息在我们的服务之间通过网络流动。这可能使我们的数据更容易在传输中被观察，也更容易被作为中间人攻击的一部分进行操纵。这意味着你可能需要更加注意保护传输中的数据，并确保你的微服务端点受到保护，以便只有授权方才能使用它们。第 11 章完全致力于探讨这一领域的挑战。

### 测试

对于任何类型的自动化功能测试，你都有一个微妙的平衡。测试执行的功能越多——即测试范围越广——你对应用的信心就越大。另一方面，测试范围越大，设置测试数据和支持夹具就越困难，测试运行时间越长，当测试失败时找出问题就越困难。我将在第 9 章分享许多使测试在这种更具挑战性的环境中工作的方法。

对于任何类型的系统，端到端测试在它们覆盖的功能方面都处于极端，我们习惯于它们比范围较小的单元测试更难编写和维护。通常这是值得的，因为我们想要来自端到端测试以用户可能的方式使用我们系统的信心。

但对于微服务架构，我们端到端测试的范围变得非常大。我们现在需要跨多个进程运行测试，所有这些都需要为测试场景部署和适当配置。我们还需要为当环境问题（如服务实例死亡或失败部署的网络超时）导致我们的测试失败时发生的假阴性做好准备。

这些力量意味着，随着微服务架构增长，你在端到端测试方面将获得递减的投资回报。测试将花费更多，但无法给你过去同样的信心水平。这将推动你走向新形式的测试，如契约驱动测试或生产中的测试，以及探索并行运行或金丝雀发布等渐进式交付技术，我们将在第 8 章看到。

### 延迟

对于微服务架构，以前可能在一个处理器上本地完成的处理现在可能最终被拆分到多个独立的微服务中。以前只在单个进程内流动的信息现在需要在你可能比以往更多地使用的网络上序列化、传输和反序列化。所有这些都可能导致系统延迟恶化。

虽然在设计或编码阶段很难准确衡量对操作延迟的影响，但这是以渐进方式进行任何微服务迁移很重要的另一个原因。进行小更改，然后衡量影响。这假设你有某种方式衡量你关心的操作的端到端延迟——像 Jaeger 这样的分布式追踪工具可以在这里提供帮助。但你还需要了解这些操作可接受的延迟是什么。有时使操作变慢是完全可接受的，只要它仍然足够快！

### 数据一致性

从数据在单个数据库中存储和管理的单体系统，转向多个进程在不同数据库中管理状态的更加分布式的系统，会导致关于数据一致性的潜在挑战。过去你可能依赖数据库事务来管理状态更改，你需要理解在分布式系统中无法轻易提供类似的安全性。在大多数情况下，使用分布式事务来协调状态更改被证明是高度有问题的。

相反，你可能需要开始使用像 Saga（我将在第 6 章详细说明）和最终一致性（Eventual Consistency）这样的概念来管理和推理系统中的状态。这些想法可能需要对你在系统中思考数据的方式进行根本性改变，在迁移现有系统时这可能相当令人望而生畏。再次，这是谨慎对待你分解应用的速度的另一个好理由。采用渐进式分解方法，以便你能够在生产环境中评估架构更改的影响，确实很重要。

## 我应该使用微服务吗？

尽管在某些方面有将微服务架构作为软件默认方法的推动，但鉴于我概述的众多挑战，我认为采用它们仍然需要仔细考虑。你需要评估你自己的问题空间、技能和技术格局，并理解你试图实现什么，然后再决定微服务是否适合你。它们是一种架构方法，不是唯一的架构方法。你自己的背景应该在决定是否走这条路方面发挥巨大作用。

也就是说，我想概述一些通常会让我远离或倾向于选择微服务的情况。

### 可能不适合谁

鉴于定义稳定服务边界的重要性，我认为微服务架构对于全新产品或初创公司来说往往是一个糟糕的选择。在这两种情况下，你正在处理的领域通常在你迭代你试图构建的基础时经历重大变化。领域模型的这种转变反过来将导致对服务边界进行更多更改，而协调跨服务边界的更改是一项昂贵的任务。

一般来说，我觉得在领域模型足够稳定之前等待，然后再寻找定义服务边界更合适。

我确实看到初创公司首先采用微服务的诱惑，理由是「如果我们真的成功，我们将需要扩展！」问题是，你不一定知道是否有人会想使用你的新产品。即使你确实成功到需要高度可扩展的架构，你最终交付给用户的东西可能与一开始构建的非常不同。Uber 最初专注于豪华轿车，Flickr 是从尝试创建多人在线游戏中衍生出来的。寻找产品市场契合的过程意味着你最终可能得到的产品与你开始时认为要构建的非常不同。

初创公司通常也没有足够的人来构建系统，这在微服务方面创造了更多挑战。微服务带来了新的工作和复杂性来源，这可能占用宝贵的带宽。团队越小，这种成本就越明显。当与只有少数开发人员的小团队合作时，我出于这个原因非常犹豫建议微服务。

微服务对初创公司的挑战还因你通常最大的约束是人而加剧。对于小团队，微服务架构可能难以证明是合理的，因为仅仅处理微服务本身的部署和管理就需要工作。有些人将此描述为「微服务税」。当该投资使许多人受益时，更容易证明是合理的。但如果你五人团队中的一个人将时间花在这些问题上，那就是大量宝贵时间没有花在构建产品上。在理解架构中的约束和痛点之后，稍后迁移到微服务要容易得多——然后你可以将精力集中在以最合理的地方使用微服务。

最后，创建将由客户部署和管理的软件的组织可能难以使用微服务。正如我们已经介绍的，微服务架构可以将大量复杂性推入部署和操作领域。如果你自己运行软件，你能够通过采用新技术、开发新技能和改变工作实践来抵消这种新复杂性。你不能期望你的客户这样做。如果他们习惯于以 Windows 安装程序的形式接收你的软件，当你发送下一个版本的软件并说「只需将这些 20 个 Pod 放在你的 Kubernetes 集群上！」时，对他们来说将是一个可怕的震惊。很可能，他们甚至不知道 Pod、Kubernetes 或集群是什么。

### 适合谁

根据我的经验，组织采用微服务可能最大的单一原因是允许更多开发人员在同一个系统上工作而不会相互妨碍。正确设置架构和组织边界，你允许更多人相互独立工作，减少交付争用。五人初创公司可能会发现微服务架构是一种拖累。快速增长的一百人规模公司可能会发现，其增长通过正确围绕产品开发工作对齐的微服务架构更容易适应。

软件即服务（SaaS）应用通常也是微服务架构的良好匹配。这些产品通常被期望 24-7 运行，这在推出更改时创造了挑战。微服务架构的独立可发布性在这一领域是一个巨大的福音。此外，微服务可以根据需要扩展或缩小。这意味着，随着你为系统的负载特征建立合理的基线，你可以更好地控制确保以最具成本效益的方式扩展系统。

微服务的技术无关性确保你可以充分利用云平台。公有云供应商为你的代码提供广泛的服务和部署机制。你可以更容易地将特定服务的要求与最能帮助你实现它们的云服务相匹配。例如，你可能决定将一个服务部署为一组函数，另一个部署为托管虚拟机（VM），另一个部署在托管平台即服务（PaaS）平台上。

虽然值得注意的是，采用广泛的技术往往可能是一个问题，但能够轻松尝试新技术是快速识别可能产生好处的新方法的好方法。FaaS 平台日益流行就是这样一个例子。对于适当的工作负载，FaaS 平台可以大幅减少操作开销，但目前，它不是一种在所有情况下都适合的部署机制。

微服务还为希望通过各种新渠道向客户提供服务的组织带来明显好处。许多数字化转型努力似乎涉及尝试解锁隐藏在现有系统中的功能。愿望是创建新的客户体验，可以通过最有意义的任何交互机制支持用户的需求。

最重要的是，微服务架构是一种可以随着你继续发展系统而给你很大灵活性的架构。当然，这种灵活性有成本，但如果你想保持对未来可能想要进行的更改的选项开放，这可能是一个值得付出的代价。

## 总结

微服务架构可以在选择技术、处理健壮性和扩展、组织团队等方面给你巨大的灵活性。这种灵活性部分是因为许多人正在拥抱微服务架构。但微服务带来了相当程度的复杂性，你需要确保这种复杂性是合理的。对许多人来说，它们已成为默认的系统架构，几乎在所有情况下使用。然而，我仍然认为它们是一种架构选择，其使用必须由你试图解决的问题来证明；通常，更简单的方法可以更容易地交付。

尽管如此，许多组织，尤其是较大的组织，已经展示了微服务可以多么有效。当微服务的核心概念被正确理解和实施时，它们可以帮助创建赋权、高效的架构，使系统能够超越其各部分的总和。

我希望本章能很好地介绍这些主题。接下来，我们将看看如何定义微服务边界，沿途探索结构化编程和领域驱动设计的主题。
