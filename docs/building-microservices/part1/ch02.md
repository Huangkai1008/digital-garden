# 第2章 如何建模微服务

> 我对手的推理让我想起异教徒，当被问及世界站在什么上面时，他回答说：「在乌龟上。」但乌龟站在什么上面？「在另一只乌龟上。」
> —Rev. Joseph Frederick Berg (1854)

所以你知道什么是微服务，我希望你也了解它们的主要好处。你现在可能急于开始创建它们，对吧？但从哪里开始？在本章中，我们将探讨一些基础概念，如信息隐藏、耦合和内聚，并理解它们将如何改变我们关于在微服务周围划定边界的思考。我们还将探讨你可能使用的不同形式的分解，以及更深入地关注领域驱动设计（Domain-Driven Design，DDD）作为这一领域的非常有用的技术。

我们将探讨如何思考微服务的边界，以最大化优势并避免一些潜在的劣势。但首先，我们需要一些可以工作的东西。

## 介绍 MusicCorp

关于想法的书用例子效果更好。在可能的情况下，我会分享来自真实情况的故事，但我发现有一个虚构场景来工作也很有用。在整本书中，我们将回到这个场景，看看微服务的概念如何在这个世界中运作。

所以让我们把注意力转向前沿的在线零售商 MusicCorp。MusicCorp 直到最近还只是一家实体零售商，但在黑胶唱片业务崩溃后，它越来越将精力集中在线上。该公司有一个网站，但觉得现在是时候加倍投入在线世界了。毕竟，那些用于音乐的智能手机只是一时的潮流（Zune 显然更好），音乐爱好者很乐意等待 CD 送到家门口。质量胜过便利，对吧？虽然它可能刚刚了解到 Spotify 实际上是一个数字音乐服务，而不是某种青少年护肤产品，但 MusicCorp 对自己的专注相当满意，并确信所有这些流媒体业务很快就会过去。

尽管有点落后于潮流，MusicCorp 有宏伟的野心。幸运的是，它已经决定占领世界的最佳机会是确保能够尽可能轻松地进行更改。微服务必胜！

## 什么是好的微服务边界？

在 MusicCorp 的团队冲向远方、试图为所有人提供八轨磁带而创建一个又一个服务之前，让我们踩刹车，谈谈我们需要牢记的最重要的基本想法。

我们希望我们的微服务能够以独立的方式被更改、部署并将其功能发布给用户。能够独立于另一个微服务更改一个微服务至关重要。那么，当我们思考如何围绕它们划定边界时，我们需要牢记什么？

本质上，微服务只是模块化分解的另一种形式，尽管模块之间有基于网络的交互以及随之而来的所有相关挑战。幸运的是，这意味着我们可以依赖模块化软件和结构化编程领域的许多既有成果，来帮助指导我们如何定义边界。考虑到这一点，让我们更深入地看看我们在第 1 章简要提到的三个关键概念，它们在弄清楚什么构成好的微服务边界时至关重要——信息隐藏、内聚和耦合。

### 信息隐藏

信息隐藏是 David Parnas 开发的一个概念，用于研究定义模块边界的最有效方式。[^1] 信息隐藏描述了一种愿望，即尽可能多地将细节隐藏在模块（或在我们的情况下，微服务）边界后面。Parnas 研究了模块理论上应该给我们的好处，即：

- **改进开发时间**：通过允许模块独立开发，我们可以允许更多工作并行完成，并减少向项目添加更多开发人员的影响。
- **可理解性**：每个模块可以独立查看和理解。这反过来使理解整个系统的功能更容易。
- **灵活性**：模块可以相互独立地更改，允许对系统功能进行更改而不需要其他模块更改。此外，模块可以以不同方式组合以交付新功能。

这一理想特征列表很好地补充了我们试图用微服务架构实现的目标——事实上，我现在将微服务视为模块化架构的另一种形式。Adrian Colyer 实际上回顾了 David Parnas 这一时期的多篇论文，并针对微服务进行了研究，他的总结非常值得一读。[^2]

正如 Parnas 在他大部分工作中所探讨的，现实是拥有模块并不会导致你实际实现这些结果。很大程度上取决于模块边界是如何形成的。根据他自己的研究，信息隐藏是帮助从模块化架构中获得最大收益的关键技术，用现代眼光看，这同样适用于微服务。

从 Parnas 的另一篇论文中，[^3] 我们有这颗宝石：

> 模块之间的连接是模块对彼此做出的假设。

通过减少一个模块（或微服务）对另一个模块做出的假设数量，我们直接影响它们之间的连接。通过保持假设数量少，更容易确保我们可以更改一个模块而不影响其他模块。如果更改模块的开发人员清楚地了解模块如何被他人使用，开发人员将更容易安全地进行更改，使上游调用者不必也进行更改。

这也适用于微服务，除此之外，我们还有机会部署更改后的微服务而不必部署任何其他东西，可以说放大了 Parnas 描述的改进开发时间、可理解性和灵活性这三个理想特征。

信息隐藏的含义以多种方式体现，我将在整本书中继续这一主题。

### 内聚

我听过描述内聚（Cohesion）最简洁的定义之一是：「一起更改的代码，放在一起。」[^4] 对于我们的目的，这是一个相当好的定义。正如我们已经讨论的，我们正在围绕业务功能更改的便利性优化微服务架构——所以我们希望功能以这样的方式分组，以便我们可以在尽可能少的地方进行更改。我们希望相关的行为放在一起，不相关的行为放在别处。

为什么？嗯，如果我们想更改行为，我们希望能够在一个地方更改它，并尽快发布该更改。如果我们必须在许多不同的地方更改该行为，我们将不得不发布许多不同的服务（可能同时）来交付该更改。在许多不同的地方进行更改更慢，同时部署许多服务有风险——因此我们希望避免两者。

所以我们希望在我们的问题域中找到有助于确保相关行为在一个地方并且尽可能松散地与其他边界通信的边界。如果相关功能分散在系统中，我们说内聚是弱的——而对于我们的微服务架构，我们的目标是强内聚。

### 耦合

当服务松散耦合时，对一个服务的更改不应要求对另一个服务进行更改。微服务的全部意义在于能够对一个服务进行更改并部署它，而不需要更改系统的任何其他部分。这真的非常重要。

什么会导致紧耦合？一个经典错误是选择将一个服务紧密绑定到另一个服务的集成风格，导致服务内部的更改需要消费者也进行更改。

松散耦合的服务对与其协作的服务了解得尽可能少。这也意味着我们可能希望限制从一个服务到另一个服务的不同类型的调用数量，因为除了潜在的性能问题之外，频繁的通信可能导致紧耦合。

然而，耦合有多种形式，我见过许多关于耦合在基于服务的架构中的性质的误解。考虑到这一点，我认为更详细地探讨这个主题很重要，我们很快就会这样做。

### 耦合与内聚的相互作用

正如我们已经提到的，耦合和内聚的概念显然是相关的。从逻辑上讲，如果相关功能分散在我们的系统中，对此功能的更改将在这些边界上蔓延，意味着更紧的耦合。以结构化设计先驱 Larry Constantine 命名的康斯坦丁定律（Constantine's Law）简洁地总结了这一点：[^5]

> 如果内聚强且耦合低，则结构稳定。

这里稳定性的概念对我们很重要。为了让我们的微服务边界兑现独立可部署性的承诺，允许我们并行地开发微服务并减少在这些服务上工作的团队之间的协调量，我们需要边界本身有一定程度的稳定性。如果微服务暴露的契约不断以向后不兼容的方式变化，那么这将导致上游消费者也不得不不断更改。

耦合和内聚密切相关，至少在某种程度上，可以说是一样的，因为两个概念都描述了事物之间的关系。内聚适用于边界内（在我们的上下文中是微服务）事物之间的关系，而耦合描述跨边界事物之间的关系。没有绝对最好的方式来组织我们的代码；耦合和内聚只是表达我们在哪里分组代码以及为什么分组代码所做的各种权衡的一种方式。我们所能努力做的就是在这两个想法之间找到正确的平衡，一个对你的给定上下文和你当前面临的问题最有意义的平衡。

记住，世界不是静态的——随着你的系统需求变化，你可能会发现重新审视你的决定的理由。有时你系统的某些部分可能正在经历如此多的变化，以至于稳定性可能是不可能的。我们将在第 3 章看到这方面的例子，当我分享 Snap CI 产品开发团队的经验时。

## 耦合的类型

你可以从上述概述推断出所有耦合都是坏的。这并不严格正确。归根结底，我们系统中的一些耦合将是不可避免的。我们想要做的是减少我们有多少耦合。

### 领域耦合

领域耦合（Domain Coupling）描述了一种情况，其中一个微服务需要与另一个微服务交互，因为第一个微服务需要使用另一个微服务提供的功能。[^8]

在图 2-2 中，我们看到了 MusicCorp 内部 CD 订单管理的部分方式。在此示例中，Order Processor 调用 Warehouse 微服务来预留库存，并调用 Payment 微服务来收款。因此，Order Processor 依赖并耦合到 Warehouse 和 Payment 微服务以完成此操作。然而，我们在 Warehouse 和 Payment 之间看不到这种耦合，因为它们不交互。

```
Order Processor ──▶ Warehouse (reserve stock)
       │
       └──────────▶ Payment (take payment)
```

**图 2-2. 领域耦合的例子，Order Processor 需要使用其他微服务提供的功能**

在微服务架构中，这种类型的交互在很大程度上是不可避免的。基于微服务的系统依赖多个微服务协作来完成其工作。我们仍然希望将其保持在最低限度；每当你看一个微服务以这种方式依赖多个下游服务时，这可能是一个值得关注的原因——这可能意味着一个微服务做得太多了。

作为一般规则，领域耦合被认为是一种松散的耦合形式，尽管即使在这里我们也可能遇到问题。需要与大量下游微服务通信的微服务可能指向逻辑过度集中化的情况。随着更复杂的数据集在服务之间发送，领域耦合也可能变得有问题——这通常可以指向我们即将探讨的更有问题的耦合形式。

只需记住信息隐藏的重要性。只分享你绝对必须分享的，只发送你需要的绝对最少量的数据。

### 关于时间耦合的简要说明

你可能听说过的另一种耦合形式是时间耦合（Temporal Coupling）。从以代码为中心的耦合观点来看，时间耦合指的是概念纯粹因为它们同时发生而被捆绑在一起的情况。时间耦合在分布式系统的上下文中具有 subtly 不同的含义，它指的是一个微服务需要另一个微服务同时做某事才能完成操作的情况。

两个微服务都需要同时启动并可用才能相互通信以完成操作。所以在图 2-3 中，MusicCorp 的 Order Processor 正在向 Warehouse 服务发出同步 HTTP 调用，Warehouse 需要在调用进行时同时启动并可用。

```
Order Processor ──────[sync HTTP]──────▶ Warehouse
       │ (blocks)                            │
       │◀──────────[response]────────────────┘
```

**图 2-3. 时间耦合的例子，Order Processor 向 Warehouse 微服务发出同步 HTTP 调用**

如果由于某种原因 Order Processor 无法联系 Warehouse，则操作失败，因为我们无法预留要发送的 CD。Order Processor 还必须阻塞并等待 Warehouse 的响应，可能在资源争用方面造成问题。

时间耦合并不总是坏的；这只是需要注意的事情。随着你拥有更多微服务，它们之间有更复杂的交互，时间耦合的挑战可能增加到难以扩展系统和保持其工作的程度。避免时间耦合的一种方法是使用某种形式的异步通信，例如消息代理。

### 传递耦合

「传递耦合」（Pass-Through Coupling）[^9] 描述了一种情况，其中一个微服务将数据传递给另一个微服务，纯粹是因为该数据被更下游的某个其他微服务需要。在许多方面，它是实现耦合中最有问题的形式之一，因为它不仅意味着调用者知道它调用的微服务会调用另一个微服务，而且还可能意味着它需要知道那个一步之遥的微服务如何工作。

作为传递耦合的例子，让我们更仔细地看看 MusicCorp 订单处理的部分工作方式。在图 2-4 中，我们有一个 Order Processor，它正在向 Warehouse 发送请求以准备订单发货。作为请求有效负载的一部分，我们发送 Shipping Manifest。这个 Shipping Manifest 不仅包含客户的地址，还包含运输类型。Warehouse 只是将这个清单传递给下游的 Shipping 微服务。

```
Order Processor ──[Shipping Manifest]──▶ Warehouse ──[Shipping Manifest]──▶ Shipping
```

**图 2-4. 传递耦合，数据传递给微服务纯粹是因为另一个下游服务需要它**

传递耦合的主要问题是，下游所需数据的更改可能导致更显著的上游更改。在我们的示例中，如果 Shipping 现在需要更改数据的格式或内容，那么 Warehouse 和 Order Processor 可能都需要更改。

有几种方法可以解决这个问题。第一种是考虑调用微服务是否应该绕过中间层。在我们的示例中，这可能意味着 Order Processor 直接与 Shipping 通信，如图 2-5 所示。然而，这会导致其他一些头痛。我们的 Order Processor 正在增加其领域耦合，因为 Shipping 是它需要了解的另一个微服务——如果那是唯一的问题，这可能仍然可以，因为领域耦合当然是一种更松散的耦合形式。然而，这里的解决方案变得更复杂，因为我们必须先与 Warehouse 预留库存，然后使用 Shipping 发货包裹，发货完成后我们需要相应地更新库存。这将在 Order Processor 中推入更多复杂性和逻辑，这些以前隐藏在 Warehouse 内部。

```
Order Processor ──▶ Warehouse (reserve)
       │
       └──────────▶ Shipping (direct)
```

**图 2-5. 解决传递耦合的一种方法是直接与下游服务通信**

对于这个具体示例，我可能会考虑一个更简单（尽管更细致）的更改——即完全向 Order Processor 隐藏对 Shipping Manifest 的需求。将管理库存和安排包裹发货的工作委托给我们的 Warehouse 服务的想法是有道理的，但我们不喜欢泄露了一些较低级别的实现——即 Shipping 微服务想要 Shipping Manifest 的事实。隐藏这一细节的一种方法是让 Warehouse 在其契约中接收所需信息，然后让它在本地构建 Shipping Manifest，如图 2-6 所示。

```
Order Processor ──[address, shipping type]──▶ Warehouse ──[builds manifest]──▶ Shipping
```

**图 2-6. 向 Order Processor 隐藏对 Shipping Manifest 的需求**

这意味着如果 Shipping 服务更改其服务契约，只要 Warehouse 收集所需数据，从 Order Processor 的角度来看，这一更改将是不可见的。

虽然这将帮助保护 Warehouse 微服务免受 Shipping 的一些更改，但仍有一些事情需要所有各方都进行更改。让我们考虑我们想要开始国际运输的想法。作为其中的一部分，Shipping 服务需要在 Shipping Manifest 中包含 Customs Declaration。如果这是可选参数，那么我们可以毫无问题地部署 Shipping 微服务的新版本。然而，如果这是必需参数，那么 Warehouse 将需要创建一个。它可能能够用它拥有的（或给定的）现有信息来做到这一点，或者它可能要求 Order Processor 向它传递额外信息。

虽然在这种情况下我们还没有消除需要对所有三个微服务进行更改的需求，但我们获得了更多关于何时以及如何可以进行这些更改的权力。如果我们有初始示例的紧（传递）耦合，添加这个新的 Customs Declaration 可能需要所有三个微服务的同步推出。至少通过隐藏这一细节，我们可以更容易地分阶段部署。

可以帮助减少传递耦合的另一种方法是让 Order Processor 仍然通过 Warehouse 将 Shipping Manifest 发送给 Shipping 微服务，但让 Warehouse 完全不知道 Shipping Manifest 本身的结构。Order Processor 将清单作为订单请求的一部分发送，但 Warehouse 不尝试查看或处理该字段——它只是将其视为数据 blob，不关心内容。相反，它只是将其传递下去。Shipping Manifest 格式的更改仍然需要 Order Processor 和 Shipping 微服务都进行更改，但由于 Warehouse 不关心清单中实际有什么，它不需要更改。

### 公共耦合

当两个或多个微服务使用一组公共数据时，就会发生公共耦合（Common Coupling）。这种耦合形式的一个简单常见例子是多个微服务使用同一个共享数据库，但它也可能表现为使用共享内存或共享文件系统。

公共耦合的主要问题是数据结构的更改可能同时影响多个微服务。考虑图 2-7 中 MusicCorp 一些服务的例子。正如我们之前讨论的，MusicCorp 在全球运营，所以它需要关于其运营国家的各种信息。在这里，多个服务都从共享数据库读取静态参考数据。如果此数据库的模式以向后不兼容的方式更改，将需要对数据库的每个消费者进行更改。实际上，像这样的共享数据往往因此非常难以更改。

```
Service A ──┐
Service B ──┼──▶ Shared DB (Countries reference data)
Service C ──┘
```

**图 2-7. 多个服务从同一数据库访问与国家相关的共享静态参考数据**

图 2-7 中的示例相对来说是相当良性的。这是因为就其本质而言，静态参考数据往往不经常更改，而且因为这些数据是只读的——因此我倾向于对以这种方式共享静态参考数据持宽松态度。然而，如果公共数据的结构更频繁地更改，或者如果多个微服务对相同数据进行读写，公共耦合会变得更有问题。

图 2-8 向我们展示了一种情况，其中 Order Processor 和 Warehouse 服务都从共享的 Order 表读写，以帮助管理向 MusicCorp 客户发送 CD 的过程。两个微服务都在更新 Status 列。Order Processor 可以设置 PLACED、PAID 和 COMPLETED 状态，而 Warehouse 将应用 PICKING 或 SHIPPED 状态。

```
Order Processor ──┐
                 ├──▶ Shared Order Table (both update Status)
Warehouse ───────┘
```

**图 2-8. 公共耦合的例子，Order Processor 和 Warehouse 都在更新同一订单记录**

虽然你可能认为图 2-8 有些人为，但这个 nonetheless 直接的公共耦合例子有助于说明一个核心问题。从概念上讲，我们让 Order Processor 和 Warehouse 微服务都管理订单生命周期的不同方面。在 Order Processor 中进行更改时，我能确定我没有以破坏 Warehouse 对世界的看法的方式更改订单数据吗，反之亦然？

确保以正确方式更改某物状态的一种方法是创建有限状态机。状态机可用于管理某实体从一种状态到另一种状态的转换，确保禁止无效的状态转换。在图 2-9 中，你可以看到 MusicCorp 中订单的允许状态转换。订单可以直接从 PLACED 转到 PAID，但不能直接从 PLACED 转到 PICKING（这个状态机可能不足以满足完整的端到端购买和发货商品所涉及的现实世界业务流程，但我想举一个简单的例子来说明这个想法）。

```
PLACED ──▶ PAID ──▶ PICKING ──▶ SHIPPED ──▶ COMPLETED
```

**图 2-9. MusicCorp 中订单的允许状态转换概述**

这个具体示例中的问题是 Warehouse 和 Order Processor 都共享管理此状态机的责任。我们如何确保它们对允许哪些转换达成一致？有方法可以跨微服务边界管理这样的流程；我们将在讨论第 6 章的 Saga 时回到这个主题。

这里的一个潜在解决方案是确保单个微服务管理订单状态。在图 2-10 中，Warehouse 或 Order Processor 都可以向 Order 服务发送状态更新请求。在这里，Order 微服务是任何给定订单的真实来源。在这种情况下，我们真的重要的是将来自 Warehouse 和 Order Processor 的请求视为仅仅是请求。在这种情况下，Order 服务的职责是管理与订单聚合相关的可接受状态转换。因此，如果 Order 服务收到来自 Order Processor 的将状态直接从 PLACED 移到 COMPLETED 的请求，如果那是无效的更改，它可以自由拒绝该请求。

```
Order Processor ──[request]──▶ Order Service ◀──[request]── Warehouse
```

**图 2-10. Order Processor 和 Warehouse 都可以请求对订单进行更改，但 Order 微服务决定哪些请求是可接受的**

::: tip
确保你将发送给微服务的请求视为下游微服务在无效时可以拒绝的东西。
:::

我在这种情况下看到的另一种方法是将 Order 服务实现为仅仅是数据库 CRUD 操作的包装器，其中请求直接映射到数据库更新。这类似于对象有私有字段但有公共 getter 和 setter——行为已从微服务泄露到上游消费者（减少内聚），我们又回到了在多个不同服务之间管理可接受状态转换的世界。

::: warning
如果你看到一个微服务看起来只是数据库 CRUD 操作的薄包装器，那是你可能具有弱内聚和更紧耦合的迹象，因为应该在该服务中管理数据的逻辑反而分散在你系统的其他地方。
:::

公共耦合的来源也是资源争用的潜在来源。多个微服务使用相同的文件系统或数据库可能会使该共享资源过载，如果共享资源变慢甚至完全不可用，可能会造成严重问题。共享数据库特别容易遇到这个问题，因为多个消费者可以对数据库本身运行任意查询，这反过来可能具有截然不同的性能特征。我见过不止一个数据库被昂贵的 SQL 查询拖垮——我可能甚至曾经是罪魁祸首一两次。[^10]

所以公共耦合有时是可以的，但往往不是。即使它是良性的，也意味着我们对共享数据可以进行的更改受到限制，但它往往说明我们代码中缺乏内聚。它也可能在操作争用方面给我们带来问题。出于这些原因，我们认为公共耦合是最不理想的耦合形式之一——但它可能变得更糟。

### 内容耦合

内容耦合（Content Coupling）描述了一种情况，其中上游服务深入下游服务的内部并更改其内部状态。最常见的形式是外部服务访问另一个微服务的数据库并直接更改它。内容耦合和公共耦合之间的区别是微妙的。在这两种情况下，两个或多个微服务都在读写同一组数据。对于公共耦合，你理解你正在使用共享的外部依赖。你知道它不在你的控制之下。对于内容耦合，所有权线变得不那么清晰，开发人员更改系统变得更加困难。

让我们重温我们之前来自 MusicCorp 的例子。在图 2-11 中，我们有一个 Order 服务，它应该管理系统中有序订单的允许状态更改。Order Processor 正在向 Order 服务发送请求，不仅委托将要进行的确切状态更改，还委托决定哪些状态转换是允许的责任。另一方面，Warehouse 服务正在直接更新存储订单数据的表，绕过 Order 服务中可能检查允许更改的任何功能。我们必须希望 Warehouse 服务有一致的逻辑集来确保只进行有效的更改。充其量，这代表了逻辑的重复。在最坏的情况下，Warehouse 中关于允许更改的检查与 Order 服务中的不同，因此我们可能最终得到处于非常奇怪、令人困惑状态的订单。

```
Order Processor ──[request]──▶ Order Service
                                    │
Warehouse ──────────────────────────┼──▶ Direct DB update (bypasses Order)
```

**图 2-11. 内容耦合的例子，Warehouse 直接访问 Order 服务的内部数据**

在这种情况下，我们还有一个问题，即订单表的内部数据结构暴露给外部方。在更改 Order 服务时，我们现在必须极其小心地对那个特定表进行更改——即使假设我们明显知道该表被外部方直接访问。这里的简单修复是让 Warehouse 向 Order 服务本身发送请求，我们可以在那里审查请求，也可以隐藏内部细节，使对 Order 服务的后续更改容易得多。

::: tip
如果你正在开发微服务，在可以自由更改的内容和不能更改的内容之间进行清晰分离至关重要。明确地说，作为开发人员，你需要知道何时你正在更改作为你的服务向外部世界暴露的契约一部分的功能。你需要确保如果你进行更改，你不会破坏上游消费者。不影响你的微服务暴露的契约的功能可以毫无顾虑地更改。
:::

当然，公共耦合发生的问题也适用于内容耦合，但内容耦合有一些额外的头痛，使其足够有问题，以至于有些人将其称为病态耦合。当你允许外部方直接访问你的数据库时，数据库实际上成为该外部契约的一部分，尽管你无法轻易推理什么可以或不可以更改。你失去了定义什么是共享的（因此不容易更改）和什么是隐藏的能力。信息隐藏已经荡然无存。

简而言之，避免内容耦合。

## 领域驱动设计入门

正如我在第 1 章介绍的，我们用于找到微服务边界的主要机制是围绕领域本身，利用领域驱动设计（DDD）来帮助创建我们领域的模型。现在让我们扩展我们对 DDD 在微服务上下文中如何工作的理解。

让我们的程序更好地表示它们将运营的真实世界的愿望并不新鲜。像 Simula 这样的面向对象编程语言被开发出来，使我们能够对真实领域进行建模。但这个想法真正成形需要的不仅仅是编程语言能力。

Eric Evans 的《领域驱动设计》[^11] 提出了一系列重要的想法，帮助我们更好地在程序中表示问题领域。对这些想法的全面探索超出了本书的范围，但 DDD 有一些值得强调的核心概念，包括：

- **通用语言（Ubiquitous Language）**：定义和采用在代码和描述领域时使用的共同语言，以帮助沟通。
- **聚合（Aggregate）**：作为单个实体管理的对象集合，通常指真实世界的概念。
- **限界上下文（Bounded Context）**：业务领域内的显式边界，向更广泛的系统提供功能，同时也隐藏复杂性。

### 通用语言

通用语言指的是我们应该努力在代码中使用与用户相同的术语的想法。其想法是，在交付团队和实际人员之间拥有共同语言将使对真实领域建模更容易，也应该改善沟通。

作为反例，我回忆起在大型全球银行工作时的情况。我们在企业流动性领域工作，这是一个花哨的术语，基本上指的是在同一企业实体持有的不同账户之间转移现金的能力。产品负责人非常合作，她对各种产品有非常深入的理解，她想将这些产品推向市场。在与她合作时，我们会讨论诸如 haircuts 和 end-of-day sweeps 之类的事情，这些在她的世界中都很有意义，对她的客户也有意义。

另一方面，代码中没有任何这种语言。在之前的某个时候，决定使用数据库的标准数据模型。它被广泛称为「IBM 银行模型」，但我无法确定这是标准的 IBM 产品还是来自 IBM 的顾问的创作。通过定义「安排」（arrangement）的松散概念，理论认为任何银行操作都可以被建模。贷款？那是一种安排。买股票？那也是一种安排！申请信用卡？猜猜看——那也是安排！

数据模型已经污染了代码，以至于代码库被剥夺了对我们正在构建的系统的所有真正理解。我们不是在构建通用的银行应用。我们正在构建一个专门用于管理企业流动性的系统。问题是我们必须将产品负责人丰富的领域语言映射到通用的代码概念——意味着在帮助翻译方面做了大量工作。我们的业务分析师经常只是花时间一遍又一遍地解释相同的概念。

通过将真实世界的语言融入代码，事情变得容易得多。开发人员拿起使用直接来自产品负责人的术语编写的故事，更有可能理解其含义并弄清楚需要做什么。

### 聚合

在 DDD 中，聚合是一个有些令人困惑的概念，有很多不同的定义。它只是对象的任意集合吗？应该从数据库中取出的最小单元？对我一直有效的模型是首先将聚合视为真实领域概念的表示——想想 Order、Invoice、Stock Item 之类的东西。聚合通常有围绕它们的生命周期，这使它们可以被实现为状态机。

作为 MusicCorp 领域中的一个例子，Order 聚合可能包含代表订单中项目的多个行项目。这些行项目仅作为整体 Order 聚合的一部分才有意义。

我们希望将聚合视为自包含的单元；我们希望确保处理聚合状态转换的代码与状态本身一起分组。所以一个聚合应该由一个微服务管理，尽管单个微服务可能拥有对多个聚合的管理。

一般来说，你应该将聚合视为具有状态、身份、将作为系统一部分管理的生命周期的东西。聚合通常指真实世界的概念。

单个微服务将处理一个或多个不同类型的聚合的生命周期和数据存储。如果另一个服务中的功能想要更改这些聚合之一，它需要直接请求对该聚合进行更改，或者让聚合本身对系统中的其他事物做出反应以启动其自己的状态转换，也许通过订阅其他微服务发出的事件。

这里需要理解的关键是，如果外部方请求聚合中的状态转换，聚合可以说 no。你理想地希望以非法状态转换不可能的方式实现你的聚合。

聚合可以与其他聚合有关系。在图 2-12 中，我们有一个与一个或多个 Order 以及一个或多个 Wishlist 相关联的 Customer 聚合。这些聚合可以由同一微服务或不同微服务管理。

```
Customer ──┬──▶ Order (1..*)
           └──▶ Wishlist (1..*)
```

**图 2-12. 一个 Customer 聚合可能与一个或多个 Order 或 Wishlist 聚合相关联**

如果这些聚合之间的关系存在于单个微服务的范围内，如果使用关系数据库，它们可以很容易地使用外键关系之类的东西存储。然而，如果这些聚合之间的关系跨越微服务边界，我们需要某种方式来建模这些关系。

现在，我们可以简单地在本地数据库中直接存储聚合的 ID。例如，考虑一个管理财务分类账的 Finance 微服务，它存储针对客户的交易。在 Finance 微服务的数据库内，我们可以有一个 CustID 列，其中包含该客户的 ID。如果我们想获得有关该客户的更多信息，我们必须使用该 ID 对 Customer 微服务进行查找。

这个概念的问题是它不是很显式——事实上，CustID 列与远程客户之间的关系完全是隐式的。要知道该 ID 是如何被使用的，我们必须查看 Finance 微服务本身的代码。如果我们能够以更明显的方式存储对外部聚合的引用，那就太好了。

在图 2-13 中，我们进行了更改以使关系显式。我们不是使用客户引用的普通 ID，而是存储一个 URI，如果构建基于 REST 的系统，我们可能会使用它。[^12]

**图 2-13. 不同微服务中两个聚合之间关系如何实现的例子**

这种方法的好处是双重的。关系的性质是显式的，在 REST 系统中，我们可以直接解引用此 URI 来查找关联的资源。但如果你不是在构建 REST 系统呢？Phil Calçado 描述了 SoundCloud 使用的这种方法的变体，[^13] 他们开发了用于跨服务引用的伪 URI 方案。例如，`soundcloud:tracks:123` 将是对 ID 为 123 的曲目的引用。这对于查看此标识符的人来说更加显式，但它也是一个足够有用的方案，可以想象创建代码，在需要时简化跨微服务聚合查找。

有很多方法可以将系统分解为聚合，其中一些选择是高度主观的。出于性能原因或便于实现，你可能决定随时间重塑聚合。然而，我认为实现关注点是次要的；我首先让系统用户的心理模型成为我初始设计的指导灯，直到其他因素发挥作用。

### 限界上下文

限界上下文通常代表更大的组织边界。在该边界的范围内，需要执行明确的职责。这有点模糊，所以让我们看另一个具体例子。

在 MusicCorp，我们的仓库是一个活动中心——管理正在发出的订单（和偶尔的退货）、接收新库存、进行叉车比赛等等。 elsewhere，财务部门也许不那么有趣，但在我们组织中仍然有重要职能，处理工资、支付运费等。

限界上下文隐藏实现细节。有内部关注点——例如，使用的叉车类型对仓库以外的人几乎没有兴趣。这些内部关注点应该对外部世界隐藏，外部世界不需要知道，也不应该关心。

从实现的角度来看，限界上下文包含一个或多个聚合。一些聚合可能暴露在限界上下文之外；其他可能内部隐藏。与聚合一样，限界上下文可能与其他限界上下文有关系——当映射到服务时，这些依赖成为服务间依赖。

让我们暂时回到 MusicCorp 业务。我们的领域是我们运营的整个业务。它涵盖从仓库到前台、从财务到订购的一切。我们可能在我们的软件中建模所有这些，也可能不建模，但这是我们运营的领域。让我们想想该领域的部分，看起来像 Eric Evans 所指的限界上下文。

**隐藏的模型**

对于 MusicCorp，我们可以将财务部门和仓库视为两个独立的限界上下文。它们都有明确的外部世界接口（就库存报告、工资单等而言），它们有只有它们需要知道的细节（叉车、计算器）。

财务部门不需要知道仓库的详细内部运作。然而，它确实需要知道一些事情；例如，它需要了解库存水平以保持账户最新。图 2-14 显示了一个上下文图的例子。我们看到对仓库内部的概念，如拣货员（picker）、代表库存位置的货架等。同样，总账中的条目对财务来说是必不可少的，但在这里不对外共享。

**图 2-14. 财务部门和仓库之间的共享模型**

然而，为了能够计算出公司的估值，财务人员需要有关我们持有的库存的信息。然后，库存项目成为两个上下文之间的共享模型。但是，请注意，我们不需要盲目地从仓库上下文中暴露库存项目的一切。在图 2-15 中，我们看到仓库限界上下文内的 Stock Item 如何包含对货架位置的引用，但共享表示只包含计数。所以有仅内部的表示和我们暴露的外部表示。通常，当你有不同的内部和外部表示时，可能有利于对它们进行不同的命名以避免混淆——在这种情况下，一种方法可能是将共享的 Stock Item 称为 Stock Count。

**图 2-15. 共享的模型可以决定隐藏不应对外共享的信息**

**共享模型**

我们也可以有出现在多个限界上下文中的概念。在图 2-14 中，我们看到客户存在于两个位置。这意味着什么？客户被复制了吗？思考方式是，从概念上讲，财务和仓库都需要了解我们的客户。财务需要了解向客户做出的财务支付，而仓库需要了解客户，就其知道已发送哪些包裹以便追踪交付而言。

当你有这样的情况时，像客户这样的共享模型在不同的限界上下文中可以有不同的含义，因此可能被称为不同的东西。我们可能乐于在财务中保留「客户」这个名称，但在仓库中我们可能称他们为「收件人」，因为那是他们在该上下文中扮演的角色。我们在两个位置存储有关客户的信息，但信息不同。财务存储有关客户财务支付（或退款）的信息；仓库存储与发货商品相关的信息。我们可能仍然需要将两个本地概念链接到全局客户，我们可能想要查找有关该客户的常见共享信息，如他们的姓名或电子邮件地址——我们可以使用图 2-13 所示的技术来实现这一点。

### 将聚合和限界上下文映射到微服务

聚合和限界上下文都为我们提供了具有与更广泛系统的明确定义接口的内聚单元。聚合是专注于我们系统中单个领域概念的自包含状态机，限界上下文代表相关聚合的集合，同样具有与更广泛世界的显式接口。

因此，两者都可以很好地作为服务边界。正如我已经提到的，在开始时，你希望减少你使用的服务数量。因此，你可能应该针对包含整个限界上下文的服务。当你站稳脚跟并决定将这些服务拆分为更小的服务时，你需要记住聚合本身不想被拆分——一个微服务可以管理一个或多个聚合，但我们不希望一个聚合由多个微服务管理。

**一路向下的乌龟**

一开始，你可能会识别出一些粗粒度的限界上下文。但这些限界上下文又可以包含进一步的限界上下文。例如，你可以将仓库分解为与订单履行、库存管理或收货相关的能力。在考虑微服务的边界时，首先考虑较大的、粗粒度的上下文，然后在寻找拆分这些接缝的好处时沿这些嵌套上下文进行细分。

这里的一个技巧是，即使你决定稍后将建模整个限界上下文的服务拆分为更小的服务，你仍然可以向外部世界隐藏这一决定——也许通过向消费者呈现更粗粒度的 API。将服务分解为更小部分的决定可以说是一个实现决定，所以如果我们能隐藏它，我们不妨隐藏它。在图 2-16 中，我们看到了这方面的例子。我们将 Warehouse 拆分为 Inventory 和 Shipping。就外部世界而言，仍然只有 Warehouse 微服务。但在内部，我们进一步分解了事物，让 Inventory 管理 Stock Items，让 Shipping 管理 Shipments。记住，我们希望将单个聚合的所有权保持在单个微服务内。

```
External View:     Warehouse (single API)
Internal View:     Inventory + Shipping (decomposed)
```

**图 2-16. Warehouse 服务内部已拆分为 Inventory 和 Shipping 微服务**

这是另一种形式的信息隐藏——我们以这样的方式隐藏了关于内部实现的决定，如果这个实现细节将来再次改变，我们的消费者将不会察觉。

偏好嵌套方法的另一个原因可能是将你的架构分块以简化测试。例如，在测试消费仓库的服务时，我不必 stub 仓库上下文内的每个服务——只需更粗粒度的 API。这也可以给你一个在考虑更大范围测试时的隔离单元。例如，我可能决定进行端到端测试，其中我启动仓库上下文内的所有服务，但对于所有其他协作者，我可能 stub 它们。我们将在第 9 章更多地探讨测试和隔离。

### 事件风暴

事件风暴（Event Storming）是 Alberto Brandolini 开发的一种技术，是一种协作式头脑风暴练习，旨在帮助浮现领域模型。与其让架构师坐在角落里提出自己对领域模型的表示，[^14] 事件风暴将技术和非技术利益相关者聚集在联合练习中。其想法是，通过使领域模型的开发成为联合活动，你最终得到对世界的共享、连贯的看法。

值得一提的是，虽然通过事件风暴定义的领域模型可用于实现事件驱动系统——事实上，映射非常直接——但你也可以使用这样的领域模型来构建更面向请求/响应的系统。

**后勤**

Alberto 对事件风暴应该如何运行有一些非常具体的看法，在一些观点上我非常同意。首先，让每个人都在同一个房间里。这通常是最困难的步骤——让人们的时间表对齐可能是个问题，找到足够大的房间也是如此。这些问题在 COVID 之前的世界都是真的，但当我写这篇文章时正值英国与病毒相关的封锁，我意识到这一步骤在未来可能更加成问题。然而，关键是让所有利益相关者同时在场。你希望有你计划建模的领域所有部分的代表：用户、主题专家、产品负责人——无论谁最能代表领域的每个部分。

一旦每个人都在同一个房间里，Alberto 建议移除所有椅子，以确保每个人都站起来并参与其中。作为背部不好的人，虽然我理解这种策略，但我认识到它可能不适合每个人。我确实同意 Alberto 的一点是需要有一个可以进行建模的大空间。一个常见的解决方案是将大卷牛皮纸钉在房间的墙上，允许使用所有墙壁来捕获信息。

主要的建模工具是便签纸来捕获各种概念，不同颜色的便签代表不同的概念。

**流程**

练习从参与者识别领域事件开始。这些代表系统中发生的事情——它们是你关心的事实。「Order Placed」将是我们在 MusicCorp 上下文中关心的事件，「Payment Received」也是如此。这些用橙色便签捕获。在这一点上，我与 Alberto 的结构有另一个分歧，因为事件是你将捕获的最多的东西，而橙色便签 surprisingly 很难获得。[^15]

接下来，参与者识别导致这些事件发生的命令。命令是人类（软件用户）决定做某事的决定。在这里，你试图理解系统的边界，并识别系统中的关键人类参与者。命令用蓝色便签捕获。

事件风暴会议中的技术人员应该倾听非技术同事在这里提出的内容。这个练习的关键部分是不让任何当前实现扭曲对领域是什么的看法（那稍后会来）。在这个阶段，你想创造一个空间，可以将关键利益相关者头脑中的概念拿出来公开。

随着事件和命令被捕获，接下来是聚合。你在这个阶段拥有的事件不仅有助于分享系统中发生的事情，它们还开始突出潜在的聚合可能是什么。想想前面提到的领域事件「Order Placed」。这里的名词——「Order」——很可能是一个潜在的聚合。而「Placed」描述了可以发生在订单上的事情，所以这可能是聚合生命周期的一部分。聚合用黄色便签表示，与该聚合相关的命令和事件被移动并聚集在聚合周围。这也帮助你理解聚合如何相互关联——一个聚合的事件可能触发另一个聚合中的行为。

随着聚合被识别，它们被分组到限界上下文中。限界上下文最常见的是遵循公司的组织结构，练习的参与者很了解哪些聚合被组织的哪些部分使用。

事件风暴比我刚才描述的更多——这只是作为一个简要概述。有关事件风暴的更详细 look，我建议你阅读 Alberto Brandolini 的（目前正在进行中的）《EventStorming》（Leanpub）。[^16]

## 领域驱动设计对微服务的理由

我们已经探讨了 DDD 如何在微服务的上下文中工作，所以让我们总结一下这种方法对我们如何有用。

首先，使 DDD 如此强大的很大一部分是，对 DDD 如此重要的限界上下文明确地是关于隐藏信息的——向更广泛的系统呈现清晰的边界，同时隐藏能够在不影响系统其他部分的情况下更改的内部复杂性。这意味着当我们遵循 DDD 方法时，无论我们是否意识到，我们也在采用信息隐藏——正如我们所看到的，这对于帮助找到稳定的微服务边界至关重要。

其次，定义共同通用语言的关注在定义微服务端点时大有帮助。它巧妙地为我们提供了一个共享的词汇表，可以在提出 API、事件格式等时借鉴。它还有助于解决在允许限界上下文内的语言更改方面 API 标准化需要走多远的问题——边界内的更改影响该边界本身。

我们对系统实施的更改通常与业务想要对系统行为进行的更改有关。我们正在更改暴露给客户的功能——能力。如果我们的系统沿着代表我们领域的限界上下文进行分解，我们想要进行的任何更改更可能被隔离到单个微服务边界。这减少了我们需要进行更改的地方数量，并允许我们快速部署该更改。

从根本上说，DDD 将业务领域置于我们正在构建的软件的核心。它鼓励我们将业务语言引入我们的代码和服务设计，有助于提高构建软件的人的领域专业知识。这反过来有助于建立对我们软件用户的理解和同理心，并建立技术交付、产品开发和最终用户之间更大的沟通。如果你有兴趣走向流对齐团队，DDD 作为一个机制来帮助将技术架构与更广泛的组织结构对齐，非常契合。在我们日益试图打破 IT 和「业务」之间孤岛的世界中，这并非坏事。

## 业务领域边界的替代方案

正如我所概述的，DDD 在构建微服务架构时非常有用，但如果认为这是你在找到微服务边界时应该考虑的唯一技术，那将是一个错误。事实上，我经常将多种方法与 DDD 结合使用，以帮助识别系统应该如何（以及是否）拆分。让我们看看我们在找到边界时可能考虑的一些其他因素。

### 易变性

我越来越多地听到对领域导向分解的反对，通常是由主张易变性（Volatility）作为分解主要驱动因素的人提出的。基于易变性的分解让你识别系统中经历更频繁更改的部分，然后将该功能提取到它们自己的服务中，在那里可以更有效地进行开发。从概念上讲，我对此没有问题，但将其推广为唯一的方式并不有帮助，特别是当我们考虑可能推动我们走向微服务的不同驱动因素时。例如，如果我最大的问题与扩展应用的需求有关，基于易变性的分解可能不会带来太多好处。

基于易变性分解的心态在双模 IT（Bimodal IT）等方法中也很明显。Gartner 提出的概念，双模 IT 根据不同系统需要多快（或多慢）将世界整齐地分解为朗朗上口的「模式 1」（又称记录系统）和「模式 2」（又称创新系统）类别。我们被告知，模式 1 系统变化不大，不需要太多业务参与。模式 2 是行动所在，有需要快速更改并需要业务密切参与的系统。暂且搁置这种分类方案中固有的 drastic 过度简化，它还暗示了对世界的非常固定的看法，并掩盖了随着公司寻求「数字化」而在行业中显而易见的各种转型。公司系统中过去不需要太多更改的部分突然需要了，以便开辟新的市场机会并以他们以前无法想象的方式向客户提供服务。

让我们回到 MusicCorp。它对我们现在所说的数字的第一次尝试只是有一个网页；在九十年代中期，它提供的只是一个待售物品的列表，但你必须打电话给 MusicCorp 下订单。它只不过是报纸上的广告。然后在线订购成为一件事，整个仓库，直到那时只是用纸处理的，不得不数字化。谁知道——也许 MusicCorp 在某个阶段将不得不考虑以数字方式提供音乐！虽然你可能认为 MusicCorp 落后于时代，但你仍然可以欣赏公司随着理解不断变化的技术和客户行为如何可能要求业务中无法轻易预见的重大更改而经历的动荡程度。

我不喜欢双模 IT 作为一个概念，因为它成为人们将难以更改的东西倾倒到一个整洁的盒子里并说「我们不需要处理那里的问题——那是模式 1」的一种方式。这是公司可以采用的又一个模型，以确保实际上什么都不必改变。它还回避了这样一个事实，即功能更改通常需要「记录系统」（模式 1）的更改才能允许「创新系统」（模式 2）的更改。根据我的经验，采用双模 IT 的组织最终确实有两种速度——慢和更慢。

公平地说，基于易变性分解的支持者中，许多人并不一定推荐像双模 IT 这样简单的模型。事实上，我发现如果主要驱动因素是关于快速上市，这种技术对于帮助确定边界非常有用——提取正在更改或需要频繁更改的功能，在这种情况下完全有意义。但同样，目标决定最合适的机制。

### 数据

你持有和管理的数据的性质可以推动你走向不同形式的分解。例如，你可能希望限制哪些服务处理个人可识别信息（PII），既是为了降低数据泄露的风险，也是为了简化 GDPR 等事情的监督和实施。

对于我最近的一个客户，一家我们称之为 PaymentCo 的支付公司，某些类型数据的使用直接影响我们关于系统分解的决定。PaymentCo 处理信用卡数据，这意味着其系统需要遵守支付卡行业（PCI）标准为如何管理这些数据设定的各种要求。作为此合规的一部分，公司的系统和流程需要进行审计。PaymentCo 需要处理完整的信用卡数据，而且数量意味着其系统必须符合 PCI 1 级，这是最严格级别，需要对与数据管理相关的系统和实践进行季度外部评估。

许多 PCI 要求是常识，但确保整个系统符合这些要求，尤其是系统需要由外部方审计，被证明相当繁重。因此，公司希望拆分处理完整信用卡数据的系统部分——这意味着只有系统子集需要这种额外级别的监督。在图 2-17 中，我们看到了我们提出的设计的简化形式。在绿区（由虚线绿线包围）运营的服务从未看到任何信用卡信息——该数据仅限于红区（由红色虚线包围）的进程（和网络）。网关将调用转移到适当的服务（和适当的区域）；当信用卡信息通过此网关时，它实际上也在红区。

由于信用卡信息从未流入绿区，该区域的所有服务都可以免除完整的 PCI 审计。红区中的服务在此类监督的范围内。在设计时，我们尽一切努力限制必须在此红区中的内容。关键是要确保信用卡信息根本不会流向绿区——如果绿区中的微服务可以请求此信息，或者如果该信息可以由红区中的微服务发送回绿区，那么清晰的分离线就会崩溃。

数据隔离通常由各种隐私和安全关注驱动；我们将在第 11 章回到这个主题和 PaymentCo 的例子。

### 技术

需要使用不同技术的需求也可能是找到边界的因素。你可以在单个运行的微服务中容纳不同的数据库，但如果你想混合不同的运行时模型，你可能面临挑战。如果你确定部分功能需要在像 Rust 这样的运行时中实现，它使你能够挤出额外的性能改进，那最终成为一个主要的强制因素。

### 组织

正如我在第 1 章介绍康威定律时建立的，组织结构与你最终得到的系统架构之间存在固有的相互作用。除了表明这种联系的研究之外，根据我自己的轶事经验，我一次又一次地看到这种情况发生。你如何组织自己最终驱动你的系统架构，无论好坏。当涉及到帮助我们定义服务边界时，我们必须将其作为决策的关键部分来考虑。

定义其所有权将跨越多个不同团队的服务边界不太可能产生我们期望的结果——正如我们将在第 15 章进一步探讨的，微服务的共享所有权是一件令人担忧的事情。因此，在考虑何时何地定义边界时，我们必须考虑现有的组织结构，在某些情况下，我们也许甚至应该考虑改变组织结构以支持我们想要的架构。

当然，我们还必须考虑如果我们的组织结构也发生变化会发生什么。这是否意味着我们现在必须重新架构我们的软件？嗯，在最坏的情况下，它可能使我们检查一个现有的微服务，该微服务现在需要被拆分，因为它包含现在可能由两个独立团队拥有的功能，而以前单个团队负责两个部分。另一方面，通常组织变更只需要现有微服务的所有者变更。考虑这样一种情况：负责仓储操作的团队以前还处理关于应该从供应商订购多少物品的功能。假设我们决定将这一责任移交给一个专门的预测团队，该团队希望从当前销售和计划促销中提取信息，以计算出需要订购什么。如果仓储团队有一个专门的 Supplier Ordering 微服务，这可以只是移交给新的预测团队。另一方面，如果此功能以前集成到仓储拥有的更大范围的系统中，则可能需要拆分出来。

即使我们在现有的组织结构内工作，也存在我们无法将边界放在正确位置的危险。多年前，我和几个同事与加利福尼亚的一个客户合作，帮助公司采用一些更清晰的代码实践并更多地转向自动化测试。我们从一些低垂的果实开始，比如服务分解，这时我们注意到一些更令人担忧的事情。我不能过多地介绍应用做了什么，但它是一个面向公众的应用，拥有庞大的全球客户群。

团队和系统都成长了。最初是一个人的愿景，系统承担了越来越多的功能和越来越多的用户。最终，组织决定通过让巴西的一组新开发人员承担部分工作来增加团队的能力。系统被拆分了，应用的前半部分本质上是无状态的，实现面向公众的网站，如图 2-19 所示。系统的后半部分只是数据存储上的远程过程调用（RPC）接口。本质上，想象你拿代码库中的仓库层并将其作为单独的服务。

两个服务经常需要更改。两个服务都以低级别的、RPC 风格的方法调用进行通信，这些调用过于脆弱（我们将在第 4 章进一步讨论）。服务接口也非常频繁，导致性能问题。这导致需要复杂的 RPC 批处理机制。我称之为「洋葱架构」，因为它有很多层，当我们必须切穿它时让我哭了。

现在从表面上看，沿地理/组织线拆分以前的单体系统的想法完全有道理，正如我们将在第 15 章展开的。然而，在这里，团队没有采用垂直的、以业务为重点的堆栈切片，而是选取了以前进程内的 API 并进行了水平切片。更好的模型是让加利福尼亚的团队有一个端到端的垂直切片，由前端和数据访问功能的相关部分组成，巴西的团队采用另一个切片。

### 内部分层与外部分层

正如我希望你现在看到的，我不是水平分层架构的粉丝。然而，分层可以有其位置。在微服务边界内，在不同层之间进行划分以使代码更容易管理可能完全合理。当这种分层成为绘制微服务和所有权边界的机制时，问题就出现了。

### 混合模型与例外

正如我希望到目前为止清楚的，我在你如何找到这些边界方面并不教条。如果你遵循信息隐藏的指导方针并理解耦合和内聚的相互作用，那么你很可能避免你选择的任何机制的一些最严重的陷阱。我碰巧认为，通过关注这些想法，你更可能最终得到领域导向的架构，但那是另外一回事。

然而，事实是，通常可能有理由混合模型，即使「领域导向」是你决定选择作为定义微服务边界的主要机制。

我们到目前为止概述的不同机制之间也有很多潜在的相互作用。在这里选择过于狭隘会导致你遵循教条而不是做正确的事情。基于易变性的分解如果你的重点是提高交付速度，可能很有意义，但如果这导致你提取一个跨越组织边界的服务，那么由于交付争用，期望你的变更速度会受到影响。

我可能根据我对业务领域的理解定义一个不错的 Warehouse 服务，但如果该系统的一部分需要用 C++ 实现，另一部分用 Kotlin，那么你将不得不沿这些技术线进一步分解。

组织和领域驱动的服务边界是我自己的起点。但那只是我的默认方法。通常，我在这里概述的许多因素会发挥作用，哪些会影响你自己的决定将基于你试图解决的问题。你需要查看你自己的具体情况来确定什么最适合你——希望我给了你一些不同的选项来考虑。只需记住，如果有人说「做这件事的唯一方法是 X！」他们可能只是在向你推销更多教条。你可以做得更好。

说了这么多，让我们通过更详细地探索领域驱动设计来更深入地研究领域建模的主题。

## 总结

在本章中，你学到了一点关于什么构成好的微服务边界，以及如何在我们的问题空间中找到给我们低耦合和强内聚双重好处的接缝。对我们领域的详细理解可以是帮助我们找到这些接缝的重要工具，通过将我们的微服务与这些边界对齐，我们确保由此产生的系统有 every 机会保持这些美德完整。我们还对如何进一步细分我们的微服务有了一点提示。

Eric Evans 的《领域驱动设计》中提出的想法在为我们服务找到合理的边界方面非常有用，我在这里只是浅尝辄止——Eric 的书更详细。如果你想更深入，我可以推荐 Vaughn Vernon 的《实现领域驱动设计》[^17] 来帮助你理解这种方法实用性，而 Vernon 的《领域驱动设计精粹》[^18] 如果你正在寻找更简洁的概述，是一个很好的 condensed 概述。

本章的大部分内容描述了我们将如何找到微服务的边界。但如果你已经有一个单体应用并正在寻求迁移到微服务架构，会发生什么？这是我们将在下一章更详细地探讨的内容。
