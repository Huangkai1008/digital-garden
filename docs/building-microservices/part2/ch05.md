# 第5章 实现微服务通信

如前一章所述，技术选型应主要由你期望的通信风格驱动。在同步阻塞与非同步非阻塞调用之间、请求-响应与事件驱动协作之间做出选择，有助于缩小原本可能很长的技术清单。本章将探讨微服务通信中常用的技术。

## 寻找理想技术

微服务之间如何通信有大量选项。但哪个才是对的——SOAP？XML-RPC？REST？gRPC？新选项层出不穷。在讨论具体技术之前，先思考我们希望所选技术具备什么特性。

### 便于向后兼容

修改微服务时，必须确保不破坏任何消费方的兼容性。因此，我们希望所选技术能方便地做向后兼容的变更。像添加新字段这样的简单操作不应破坏客户端。理想情况下，我们还应能验证变更是否向后兼容，并在部署到生产环境之前获得反馈。

### 接口显式化

微服务对外暴露的接口必须显式、清晰。这意味着：对消费方来说，能清楚知道微服务提供哪些能力；对开发该微服务的开发者来说，能清楚知道哪些功能必须保持稳定以服务外部。我们要避免因微服务变更导致意外破坏兼容性的情况。

显式模式（schema）有助于确保微服务接口的显式性。有些技术强制使用模式，有些则可选。无论哪种，我都强烈建议使用显式模式，并辅以足够文档说明消费方可期望的功能。

### 保持 API 技术无关

IT 行业变化迅速，新工具、框架和语言不断涌现。现在可能是 .NET 技术栈，一年或五年后呢？如果想尝试能提高生产力的替代技术栈怎么办？

我倾向于保持选择开放，这也是我推崇微服务的原因之一。因此，确保微服务间通信所用的 API 保持技术无关非常重要——避免使用会限制技术栈的集成技术。

### 让消费方易于使用

我们希望消费方易于使用我们的微服务。如果消费成本很高，微服务设计得再好也意义有限。理想情况下，我们希望客户端有充分的技术选择自由；另一方面，提供客户端库可以降低使用门槛。但这类库往往与其他目标冲突，例如增加耦合。

### 隐藏内部实现细节

我们不希望消费方与内部实现绑定，否则会导致耦合增加，进而提高变更成本。任何会迫使我们暴露内部表示细节的技术都应避免。

## 技术选型

可选技术很多，这里聚焦最常用和最有代表性的几类：

| 技术 | 描述 |
|------|------|
| **远程过程调用 (RPC)** | 使本地方法调用在远程进程上执行的框架，如 SOAP、gRPC |
| **REST** | 以资源（Customer、Order 等）为中心、用统一动词（GET、POST）访问的架构风格 |
| **GraphQL** | 允许消费方定义自定义查询、从多个下游微服务获取并过滤数据的协议 |
| **消息代理 (Message Brokers)** | 通过队列或主题实现异步通信的中间件 |

### 远程过程调用 (RPC)

远程过程调用（Remote Procedure Call，RPC）指发起本地调用，却在远程服务上执行。多数 RPC 实现需要显式模式，如 SOAP 或 gRPC，模式常称为接口定义语言（IDL）。SOAP 的模式格式称为 WSDL。显式模式便于为不同技术栈生成客户端和服务器桩代码。

::: tip RPC 的挑战
- **技术耦合**：如 Java RMI 与特定平台强绑定
- **本地调用 ≠ 远程调用**：RPC 隐藏了网络复杂性，但两者本质不同
- **脆弱性**：二进制桩生成常导致客户端与服务器必须同步部署
:::

**适用场景**：gRPC 是当前首选。若对客户端和服务器都有较好控制，gRPC 很适合；若需支持多种异构应用，REST over HTTP 更合适。

### REST

表述性状态转移（Representational State Transfer，REST）是一种受 Web 启发的架构风格。核心概念是**资源**：服务所管理的实体（如 Customer）。服务器按请求生成该资源的不同表述，资源的外部呈现与内部存储完全解耦。

HTTP 的动词（GET、POST、PUT 等）与 REST 风格天然契合。REST 还带来丰富的工具生态：缓存代理、负载均衡、监控等。

**超媒体即应用状态引擎（HATEOAS）**：客户端通过链接导航资源，而非硬编码 URI。实践中 HATEOAS 采用较少。

**适用场景**：REST over HTTP 是同步请求-响应接口的稳妥默认选择，尤其需要广泛客户端接入时。

### GraphQL

GraphQL 允许客户端定义查询，一次请求从多个下游微服务获取并过滤所需数据，减少请求次数和数据量，适合移动端等受限场景。

::: warning GraphQL 的挑战
- 复杂查询可能对服务器造成高负载
- 缓存比 REST 更复杂
- 写操作不如读操作自然
:::

**适用场景**：适合系统边界，面向外部客户端（尤其是 GUI、移动端）。本质是调用聚合与过滤机制，不替代通用微服务间通信。

### 消息代理

消息代理是位于进程之间的中间件，用于管理通信。常用能力包括：

- **队列与主题**：队列通常点对点；主题支持多订阅者
- **保证投递**：下游不可用时，代理会暂存消息直至可投递
- **顺序保证**：多数代理支持，但范围可能有限（如 Kafka 仅在分区内保证）

**Kafka** 特点：高吞吐、消息可持久化、支持流处理。适合大规模事件驱动场景。

## 序列化格式

### 文本格式

JSON 已成为主流。Avro 基于 JSON 结构，支持在负载中携带模式，便于多版本消息格式共存。

### 二进制格式

Protocol Buffers 是最常用的二进制序列化格式之一，常与 gRPC 配合使用。其他选项包括 Cap'n Proto、FlatBuffers 等。

## 模式（Schemas）

### 结构性与语义性破坏

- **结构性破坏**：端点结构变更导致消费方不兼容（如删除字段、新增必填字段）
- **语义性破坏**：结构不变但行为变更，导致消费方预期被打破

### 是否使用模式？

使用模式并比较版本可发现结构性破坏；语义性破坏需通过测试发现。显式模式能明确表示契约，并作为安全网。

## 处理微服务间的变更

### 避免破坏性变更

- **扩展式变更**：只添加新内容，不删除旧内容
- **宽容读者（Tolerant Reader）**：消费方只提取所需字段，忽略其余
- **选对技术**：如 Protocol Buffers 的字段编号机制便于扩展
- **显式接口**：用模式明确暴露内容
- **尽早发现意外破坏**：用 Protolock、openapi-diff 等工具在 CI 中检测不兼容变更

### 管理破坏性变更

1. **同步部署（Lockstep Deployment）**：要求接口提供方与所有消费方同时变更——违背独立可部署性
2. **并存不兼容版本**：同时运行新旧版本，旧消费方路由到旧版本——运维和状态管理复杂
3. **模拟旧接口**：在同一服务中同时暴露新旧端点，给消费方迁移时间——推荐做法

## DRY 与微服务中的代码复用

通过库共享代码时需注意：

- 共享代码若跨越服务边界，会引入耦合
- 无法一次性升级所有使用方，需接受多版本并存
- 客户端库应由消费方决定何时升级

## 服务发现

### DNS

简单直接，但动态环境下手动维护 DNS 记录成本高。可将域名解析到负载均衡器，由负载均衡器管理实例。

### 动态服务注册中心

- **Consul**：支持服务发现、配置管理、健康检查，提供 DNS 接口
- **etcd / Kubernetes**：若使用 Kubernetes，平台已内置服务发现

## 服务网格与 API 网关

```
┌─────────────────────────────────────────┐
│  North-South Traffic (API Gateway)       │
│  ┌─────────────────────────────────┐    │
│  │  External Clients → API Gateway │    │
│  └─────────────────────────────────┘    │
│                  ↓                       │
│  East-West Traffic (Service Mesh)        │
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐    │
│  │ A   │←→│ B   │←→│ C   │←→│ D   │    │
│  └─────┘  └─────┘  └─────┘  └─────┘    │
└─────────────────────────────────────────┘
```

- **API 网关**：处理南北流量，管理外部访问、认证、限流等
- **服务网格**：处理东西流量，实现 mTLS、关联 ID、服务发现、负载均衡等

::: tip 避免的用法
- 在 API 网关中做调用聚合或协议重写——应使用 GraphQL 或 BFF 模式
- 用 API 网关作为所有微服务间调用的中介——会增加延迟和复杂度
:::

## 服务文档

- 使用 OpenAPI、AsyncAPI、CloudEvents 等显式模式
- 建立人性化注册中心（Humane Registry），聚合服务元数据、健康状态、文档等

## 小结

- 根据上下文和通信风格选择技术，而非先选技术
- 使用模式以明确契约并尽早发现破坏性变更
- 尽量做向后兼容的变更，保持独立可部署性
- 必须做破坏性变更时，通过模拟旧接口等方式给消费方迁移时间
- 建立人性化注册中心，帮助团队理解系统
