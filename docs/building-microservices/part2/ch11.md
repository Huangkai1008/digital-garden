# 第11章 安全

微服务架构扩大了攻击面，安全必须融入交付流程。本章概述核心原则、数据安全、认证与授权。

## 核心原则

### 最小权限（Least Privilege）

服务与用户仅应拥有完成其职责所需的最小权限。

### 纵深防御（Defense in Depth）

不依赖单一安全控制，多层防护降低单点失效影响。

### 自动化

将安全检查集成到构建、部署流水线，减少人为疏漏。

## 网络安全五大职能

| 职能 | 描述 |
|------|------|
| **识别 (Identify)** | 了解资产、风险、威胁 |
| **保护 (Protect)** | 实施防护措施 |
| **检测 (Detect)** | 发现异常与入侵 |
| **响应 (Respond)** | 处理安全事件 |
| **恢复 (Recover)** | 恢复服务与数据 |

## 应用安全基础

### 凭证（Credentials）

- 避免硬编码密钥
- 使用密钥管理服务（如 Vault）
- 定期轮换凭证

### 补丁（Patching）

及时应用安全补丁，通过自动化流水线部署。

### 备份与重建

定期备份，验证恢复流程，确保可从备份重建系统。

## 隐式信任 vs 零信任

- **隐式信任**：内网默认可信，边界防护为主
- **零信任**：不默认信任任何网络，每次访问都需验证

实际多为两者之间的谱系，根据风险逐步向零信任靠拢。

## 数据安全

### 传输中（Data in Transit）

使用 TLS 等加密传输，避免明文敏感数据。

### 静态（Data at Rest）

对敏感数据加密存储，密钥单独管理。

## 认证与授权

### 服务间认证

- 双向 TLS（mTLS）
- 服务网格可集中管理证书

### 用户认证

- 单点登录（SSO）
- OAuth 2.0、OpenID Connect 等标准

### 细粒度授权

- **混淆代理问题（Confused Deputy）**：确保服务代表用户执行操作时正确校验权限
- **集中 vs 分散**：可在网关集中做粗粒度校验，在服务内做细粒度校验
- **JWT**：可用于在无状态场景传递身份与权限信息

## 小结

- 安全应融入开发与交付流程
- 最小权限、纵深防御、自动化是基础
- 传输与静态数据均需保护
- 认证与授权需区分服务间与用户场景，并考虑零信任演进
