# 第13章 扩展

随着负载增长，系统需要扩展。本章介绍四种扩展维度、缓存策略与自动伸缩。

## 四种扩展维度

### 1. 垂直扩展（Vertical Scaling）

增加单机 CPU、内存等资源。简单但有上限，且成本曲线陡峭。

### 2. 水平复制（Horizontal Duplication）

增加实例数量，通过负载均衡分发流量。微服务架构的典型扩展方式。

### 3. 数据分区（Data Partitioning）

将数据分布到多个节点，按分区键路由。可提高吞吐，但需处理跨分区查询与再平衡。

### 4. 功能分解（Functional Decomposition）

将高负载功能拆分为独立服务，单独扩展。与微服务边界划分密切相关。

## 组合模型

实际系统常组合多种维度：例如对热点服务水平复制，对数据库做读写分离或分片。

## 从小规模开始

避免过早优化。先验证瓶颈，再针对性扩展。

## 缓存

### 用途

- **性能**：减少重复计算或 I/O
- **规模**：降低下游负载
- **鲁棒性**：下游故障时仍可提供部分数据（若可接受陈旧数据）

### 缓存位置

- 客户端
- 反向代理 / CDN
- 服务内部
- 分布式缓存（如 Redis）

### 失效（Invalidation）

缓存失效是难点。策略包括：TTL、主动失效、版本化键等。

::: danger 缓存黄金法则
**缓存失效是计算机科学中两大难题之一。** 设计缓存时，必须明确失效策略与一致性预期。
:::

### 新鲜度 vs 优化

在性能优化与数据新鲜度之间权衡。某些场景可接受短暂陈旧，某些则必须强一致。

### 缓存污染

恶意或异常请求可能导致缓存被“污染”，存储错误或有害数据。需对缓存键与内容做校验与隔离。

## 自动伸缩（Autoscaling）

根据负载自动调整实例数量。云平台与 Kubernetes 均支持基于 CPU、请求数等指标的自动伸缩。

- **scale-up**：负载升高时扩容
- **scale-down**：负载降低时缩容，需注意优雅关闭与连接排空

## 小结

- 四种扩展维度：垂直、水平复制、数据分区、功能分解
- 缓存可提升性能与鲁棒性，但需谨慎设计失效策略
- 自动伸缩有助于应对负载波动，需与部署和监控配合
