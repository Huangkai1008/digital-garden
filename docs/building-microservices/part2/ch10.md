# 第10章 从监控到可观测性

单进程单体时代，问题排查相对直接。微服务架构下，故障面扩大，需要新的思路和工具。本章探讨可观测性的构建块：日志聚合、指标聚合、分布式追踪，以及告警与生产测试。

## 监控 vs 可观测性

- **监控（Monitoring）**：我们做的事——观察系统、设置告警
- **可观测性（Observability）**：系统的属性——通过外部输出理解内部状态的程度

可观测性高的系统，出问题时更容易定位根因。传统监控侧重“预想可能出错的地方并告警”；在分布式系统中，会遇到从未预料的问题。可观测系统提供可被多角度查询的外部输出，支持事后追问。

::: tip 关于“三大支柱”
将可观测性简化为指标、日志、追踪“三大支柱”容易过于狭隘。可观测性是系统属性，实现方式多样。关注**结果**（系统是否可理解），而非堆砌工具。
:::

## 可观测性构建块

### 日志聚合

```
微服务实例 → 本地日志文件 → 采集代理 → 中央存储 → 查询
```

::: tip 首要任务
在构建微服务架构之前，先部署日志聚合工具。将其视为微服务架构的**前提条件**。
:::

**关联 ID（Correlation ID）**：在请求入口生成唯一 ID，沿调用链传递，使同一请求在各服务的日志可关联。尽早引入，后期改造成本高。

**局限**：时钟偏差影响调用顺序判断；日志量大时存储与查询成本高。

### 指标聚合

收集 CPU、内存、响应时间等数值，用于趋势分析、容量规划、告警。需支持按服务、实例等维度聚合与下钻。

**低基数 vs 高基数**：传统指标系统（如 Prometheus）适合低基数；高基数（如按 customer_id、request_id 查询）需要专门工具（如 Honeycomb、Lightstep）。

### 分布式追踪

将跨多个微服务的调用串联为**追踪（Trace）**，每个本地活动记录为**跨度（Span）**。可精确定位延迟瓶颈和调用关系。

实现通常需要：在服务内采集 Span、通过本地代理转发、中央收集与展示。Jaeger、Lightstep、Honeycomb 等支持 OpenTelemetry 等标准。

## 我们做得如何？

### SLA / SLO / SLI

- **SLA（服务等级协议）**：与用户达成的承诺及违约后果
- **SLO（服务等级目标）**：团队承诺达成的目标
- **SLI（服务等级指标）**：用于衡量 SLO 的实际指标

### 错误预算（Error Budget）

若 SLO 为 99.9% 可用性，则每季度允许约 2 小时 11 分钟不可用——即错误预算。预算充足时可尝试风险更高的变更；预算耗尽则应优先提升稳定性。

## 告警

- **优先级**：并非所有问题都需要凌晨 3 点叫醒人
- **告警疲劳**：告警过多会淹没关键信息（如三里岛事故、737 Max）
- **语义监控**：从业务角度定义“健康”，而非仅看 CPU、响应时间

## 小结

- 可观测性是系统属性，监控是活动
- 日志聚合是微服务架构的前提，尽早引入关联 ID
- 指标与分布式追踪补充不同维度的可见性
- 用 SLO 和错误预算管理变更与稳定性
- 告警应聚焦真正需要人工介入的问题
