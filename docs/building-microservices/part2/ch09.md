# 第9章 测试

微服务架构下，测试策略需要相应调整。本章探讨测试类型、范围权衡，以及如何实现服务测试和端到端测试。

## 测试类型

### 测试金字塔（Mike Cohn）

```
        /\
       /  \    端到端测试（少、慢、高信心）
      /----\
     /      \   服务测试
    /--------\
   /          \  单元测试（多、快、低信心）
  /------------\
```

- **越往上**：范围越大、信心越高，但反馈越慢、定位越难
- **越往下**：范围越小、反馈越快，定位更精确，但整体信心较低

### 单元测试

测试单个函数或方法，不启动微服务，不依赖外部文件或网络。应保持大量、极快的单元测试，便于快速反馈和重构。

### 服务测试

绕过用户界面，直接测试单个微服务。需**桩（Stub）掉**所有外部协作方，仅将被测微服务纳入范围。比端到端测试更快、更稳定，定位更清晰。

### 端到端测试

驱动整个系统（通常通过 GUI），覆盖大量生产代码。通过时信心高，但**脆弱、易波动、反馈慢**。

## 实现服务测试

### Mock 与 Stub

- **Stub**：返回预设响应的假服务，不关心被调用次数
- **Mock**：除返回响应外，还验证是否按预期被调用

服务测试中更常用 Stub。可使用 mountebank 等工具创建可编程桩服务。

## 实现端到端测试

### 挑战

1. **使用哪个版本？** 被测服务与协作服务版本如何组合？
2. **扇入模型**：多个流水线扇入到同一端到端测试阶段，任一服务变更触发共享测试
3. **脆弱与波动**：组件多、网络抖动、超时等易导致非确定性失败
4. **所有权**：端到端测试由谁编写和维护？跨团队时易成“无主之地”
5. **运行时长**：端到端套件可能极慢，影响反馈周期
6. **堆积效应**：测试失败期间继续提交会加剧修复难度
7. **元版本（Metaversion）**：为“整套系统”定义版本会鼓励多服务同步部署，削弱独立可部署性

### 契约测试与消费者驱动契约 (CDC)

契约测试验证服务接口是否符合消费方预期。**消费者驱动契约（Consumer-Driven Contract）** 由消费方定义期望，提供方实现。Pact 是常用工具。

::: tip 端到端测试建议
- 尽量用更小范围的测试替代端到端测试
- 若必须保留，明确所有权，控制规模，持续清理冗余
- 波动测试应追踪并修复或移除，避免“正常化偏差”
:::

## 生产环境测试

从预生产验证延伸到生产环境验证：

- **冒烟测试**：部署后快速验证基本功能
- **语义监控**：从业务角度判断系统是否正常
- **金丝雀发布**：配合监控，观察新版本对部分用户的影响

## 小结

- 测试金字塔：单元测试为主，服务测试次之，端到端测试尽量少
- 服务测试需桩掉下游，保持隔离
- 端到端测试成本高，应谨慎使用并积极替代
- 契约测试有助于在接口变更时及早发现问题
