# 第6章 工作流

前两章讨论了微服务之间如何通信。但当需要多个微服务协作实现业务流程时呢？在分布式系统中正确建模和实现这类工作流颇具挑战。本章将探讨使用分布式事务的陷阱，以及 **Saga** 这一更合适的建模方式。

## 数据库事务

事务通常指我们希望作为单一单元执行的一组操作。在数据库中，事务用于确保一个或多个状态变更要么全部成功，要么全部失败。

### ACID 事务

ACID 代表原子性、一致性、隔离性和持久性：

| 属性 | 含义 |
|------|------|
| **原子性 (Atomicity)** | 事务内操作要么全部完成，要么全部失败 |
| **一致性 (Consistency)** | 变更后数据库处于有效、一致状态 |
| **隔离性 (Isolation)** | 并发事务互不干扰，中间状态对其他事务不可见 |
| **持久性 (Durability)** | 提交后数据不会因系统故障丢失 |

::: tip 微服务中的 ACID
微服务仍可在**单个服务内部**使用 ACID 事务。问题在于：当操作跨越多个微服务、涉及多个数据库时，我们失去了**跨服务的原子性**。
:::

### 仍为 ACID，但缺乏原子性？

单数据库内两个表的更新可以放在同一 ACID 事务中。但当每个变更发生在不同微服务的数据库中时，就变成两个独立事务，无法保证整体原子性。

## 分布式事务——两阶段提交 (2PC)

两阶段提交试图在分布式系统中实现跨进程的原子变更：

1. **投票阶段**：协调者询问所有参与者是否能执行变更；参与者锁定资源并投票
2. **提交阶段**：若全部同意，协调者发送提交指令，参与者执行变更并释放锁

::: danger 为何避免 2PC
- 协调分布式锁复杂，易产生死锁
- 参与者越多、延迟越高，问题越多
- 无法保证提交同时发生，会失去隔离性
- 长时间锁定资源影响系统吞吐
:::

## 分布式事务——坚决不用

若数据必须保持真正的原子性和一致性，且无法在不使用 ACID 事务的情况下实现，则**不要拆分**——将相关状态保留在单一数据库中。

若必须拆分，又不想承受分布式事务的代价，可考虑 **Saga**。

## Saga

Saga 是一种**无需长时间锁定资源**即可协调多个状态变更的算法。它将流程拆分为可独立执行的离散步骤，每个步骤可在本地 ACID 事务中完成。

::: warning Saga 不提供 ACID 级原子性
Saga 在整体层面**不保证原子性**。每个步骤内部可以有 ACID 事务，但 Saga 本身只提供足够信息让我们推理当前状态，并自行处理失败影响。
:::

### Saga 失败模式

- **向后恢复（Backward Recovery）**：回滚，执行补偿事务（Compensating Transaction）
- **向前恢复（Forward Recovery）**：从失败点重试，继续执行

Saga 处理的是**业务失败**（如余额不足），而非技术失败（如超时、500 错误）。技术失败需单独处理。

### 补偿事务

补偿事务用于“撤销”已提交的事务。与数据库回滚不同，补偿事务是**语义层面的回滚**——无法让原事务“从未发生”，只能通过新操作尽量恢复业务语义。

例如：已发送的邮件无法“撤回”，补偿操作可以是再发一封说明订单取消的邮件。

### 调整步骤顺序以减少回滚

将易失败的步骤提前、将难以补偿的步骤延后，可减少需要执行的补偿事务数量。例如：在订单实际发货后再发放积分，可避免发货失败时撤销积分。

## 实现 Saga

### 编排式 Saga (Orchestrated)

```
┌──────────────────────────────────────────────────────────┐
│  Order Processor (Orchestrator)                           │
│  ┌─────┐    ┌─────┐    ┌─────┐    ┌─────┐    ┌─────┐   │
│  │ 1.  │───→│ 2.  │───→│ 3.  │───→│ 4.  │───→│ 5.  │   │
│  │Reserve│   │Pay  │   │Package│   │Ship │   │Points│   │
│  └─────┘    └─────┘    └─────┘    └─────┘    └─────┘   │
│      ↓           ↓           ↓           ↓           ↓   │
│  Warehouse   Payment    Warehouse   Warehouse   Loyalty   │
└──────────────────────────────────────────────────────────┘
```

- 中央协调者（Orchestrator）定义执行顺序并触发补偿
- 强依赖请求-响应式调用
- 流程显式、可观测，但协调者与所有参与服务耦合

### 编排式 Saga (Choreographed)

```
Order Placed ──→ Warehouse (reserve) ──→ Stock Reserved
                      │
Payment Taken ──┬─────┴─────┬──→ Warehouse (dispatch)
                │           └──→ Loyalty (award points)
```

- 无中央协调者，各服务通过事件协作
- 耦合低，但流程分散，难以追踪整体状态
- 需使用**关联 ID（Correlation ID）** 串联事件，便于追踪和恢复

### 编排式 vs 编排式

| 维度 | 编排式 (Orchestrated，有协调者) | 编排式 (Choreographed，无协调者) |
|------|----------------------|------------------------|
| 耦合 | 高（协调者知晓所有服务） | 低 |
| 可观测性 | 高 | 低，需通过事件投影 |
| 适用场景 | 单团队拥有整个流程 | 多团队协作 |

## Saga 与分布式事务

分布式系统先驱 Pat Helland 指出：

> 在大多数分布式事务系统中，单个节点失败会导致事务提交停滞，进而使应用卡死。系统越大，宕机概率越高。就像一架需要所有引擎都工作的飞机，增加引擎反而降低了可用性。

显式将业务流程建模为 Saga，可避免分布式事务的诸多问题，同时使隐式流程变得显式、可维护。

## 小结

- 跨微服务的工作流应显式建模为 Saga，而非依赖分布式事务
- Saga 不提供 ACID 级原子性，需通过补偿事务处理失败
- 编排式适合单团队、强控制；编排式适合多团队、松耦合
- 编排式需重视关联 ID 和事件投影，以追踪 Saga 状态
