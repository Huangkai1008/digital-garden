# 第8章 部署

部署单进程单体应用相对直接。微服务则因依赖关系和技术选型多样而复杂得多。Kubernetes 的兴起和 FaaS 平台的出现带来了更多部署方式，但核心原则仍然重要。本章将阐述部署的核心原则，并说明不同工具如何帮助或阻碍这些原则的落地。

## 从逻辑到物理

此前讨论微服务时多采用逻辑视图。实际部署时需考虑物理拓扑：多实例、数据库、环境差异等。

### 多实例

为应对负载和故障，通常会有多个微服务实例。通过负载均衡将请求分发到不同实例。实例应分布在多台机器、多个可用区（Availability Zone），以应对单点故障。

### 数据库

- 同一逻辑微服务的多个实例**共享**同一数据库（逻辑上仍为“不共享数据库”，因仅同一服务访问）
- 可使用主从复制：写走主节点，读走副本
- 同一物理数据库基础设施可托管多个逻辑隔离的数据库（如 Invoice、Order 各一个）

### 环境

软件在不同环境中运行：开发机、CI、预生产、生产等。各环境用途不同，拓扑和配置也不同。同一制品应能部署到不同环境，仅通过外部配置区分。

## 微服务部署原则

### 隔离执行

每个微服务实例应有独立的计算资源，避免相互影响。避免多微服务混布在同一主机。容器在隔离与成本之间提供了良好折中。

### 聚焦自动化

微服务数量增加会带来更多进程、配置和监控对象。必须**持续投入自动化**，选择支持自动化的工具，并将自动化作为文化的一部分。

### 基础设施即代码 (IaC)

用机器可读的代码定义基础设施（如 Terraform、Pulumi、Chef、Puppet）。配置可版本控制、可测试、可重复执行。

### 零停机部署

上游消费方不应感知到发布。异步通信场景下较易实现；同步场景下可借助滚动升级、蓝绿部署等。

### 期望状态管理

声明所需实例数、资源等，由平台自动维持。实例宕机时平台自动拉起新实例。Kubernetes、云厂商的自动伸缩组等均支持此能力。

## 部署选项

| 选项 | 描述 |
|------|------|
| **物理机** | 直接部署到物理机，无虚拟化——现已少见 |
| **虚拟机 (VM)** | 部署到 VM，隔离好但开销较大 |
| **容器** | 部署到容器，轻量、启动快，与 Kubernetes 配合成为主流 |
| **应用容器** | 如 Tomcat、IIS，多应用共享运行时——不推荐 |
| **PaaS** | 如 Heroku、App Engine，高度抽象——适合匹配约束的场景 |
| **FaaS** | 如 AWS Lambda、Azure Functions，按需执行、按量计费 |

### 容器

- 与 VM 共享内核，无 Hypervisor 开销，启动更快
- Docker 普及了镜像概念和工具链
- 容器化本身不提供期望状态管理，需 Kubernetes 等编排平台

### FaaS

- 按触发执行，按使用付费
- 冷启动、执行时长限制、无状态等需考虑
- 与微服务映射：可一微服务一函数，或按聚合（Aggregate）拆分为多函数

### 选择建议

1. **能用就别动**：现有方案若有效，不必跟风
2. **尽量让出控制权**：若 PaaS 或 FaaS 能满足需求，优先使用
3. **容器化是稳妥折中**：成本与隔离平衡好，本地开发友好，预期 Kubernetes 会进入技术栈

## Kubernetes 与容器编排

Kubernetes 已成为容器编排的事实标准。核心概念：

- **集群 (Cluster)**：节点（Node）集合 + 控制平面（Control Plane）
- **Pod**：一个或多个容器的部署单元
- **Service**：稳定的路由端点，将流量导向 Pod
- **Deployment**：管理 Pod 和 ReplicaSet，支持滚动升级、回滚、扩缩容

### 渐进式交付

将**部署**与**发布**分离：

- **部署**：将某版本软件安装到某环境
- **发布**：使功能对用户可用

技术手段包括：

- **特性开关 (Feature Toggles)**：隐藏未完成或需灰度功能
- **金丝雀发布 (Canary Release)**：先对少量用户发布，观察无问题再全量
- **并行运行 (Parallel Run)**：新旧版本并行，对比结果

## 小结

- 微服务部署需考虑隔离、自动化、IaC、零停机、期望状态管理
- 容器 + Kubernetes 是当前主流选择；FaaS 适合特定场景
- 将部署与发布分离，采用渐进式交付降低发布风险
