# 第7章 构建

前面主要讨论了微服务的设计，接下来需要关注开发流程如何适应这种架构。在部署和测试之前，先看开发者提交代码后会发生什么。本章将回顾持续集成（CI）和持续交付（CD），以及源代码与构建的映射方式。

## 持续集成简介

持续集成（Continuous Integration，CI）的核心目标是让团队保持同步，通过频繁集成和验证新提交的代码实现。CI 服务器检测提交、检出代码，并执行编译、测试等验证。最低要求是每天集成一次，实践中许多团队会每天多次集成。

### 你真的在做 CI 吗？

使用 CI 工具不等于真正实践 CI。Jez Humble 提出三个自检问题：

1. **是否每天向主干提交一次？** 频繁集成才能避免未来的集成地狱
2. **是否有测试套件验证变更？** 没有测试，我们只知道语法上集成成功，无法确认行为是否正确
3. **构建失败时，修复是否是团队第一优先级？** 红灯意味着上次变更可能未正确集成，应停止其他提交直至修复

### 分支模型

- **功能分支（Feature Branching）**：每个功能一个分支，表面看便于隔离开发，实则**延迟集成**，合并时冲突更复杂
- **基于主干开发（Trunk-Based Development）**：所有人向同一主干提交，用特性开关（Feature Flags）隐藏未完成功能——**推荐**

::: tip 关于分支
尽早集成、频繁集成。避免使用长期功能分支，考虑基于主干开发。若必须用分支，保持短命。
:::

## 构建流水线与持续交付

### 构建流水线

将构建分为多个阶段，例如：快速测试 → 慢速测试 → 性能测试 → 生产部署。同一制品（Artifact）贯穿整个流水线，逐步增强信心。

### 持续交付 (CD)

持续交付（Continuous Delivery）将每次提交视为发布候选，对每个候选评估生产就绪度，并建模从提交到生产的所有阶段（包括人工阶段）。

### 持续交付 vs 持续部署

- **持续交付**：每次提交是发布候选，通过验证后可部署，但部署决策可有人工参与
- **持续部署**：通过验证的提交**自动**部署，无需人工干预

持续部署是持续交付的延伸。没有持续交付就无法持续部署，但可以只做持续交付而不做持续部署。

## 制品创建

::: tip 制品创建原则
1. **构建一次**：为微服务创建部署制品一次，在所有环境中复用
2. **验证的即部署的**：测试的制品与部署的制品必须相同
3. **环境无关**：制品本身不含环境特定配置，配置单独管理
:::

## 源代码与构建的映射

### 单一巨型仓库 + 单一构建

所有微服务代码在一个仓库，一个构建产出多个制品。问题：任意小改动都会触发全量构建和测试；难以判断哪些服务真正变更；多团队共享时职责不清。**不推荐**（除非项目极早期、单团队短期使用）。

### 模式：每微服务一仓库（Multirepo）

每个微服务独立仓库，变更与 CI 构建一一对应。优点：职责清晰、所有权明确。挑战：跨仓库修改需多次提交；无法原子提交。

::: tip 跨仓库修改
若经常需要跨微服务修改，可能是**服务边界划分不当**的信号。可考虑合并微服务。
:::

### 模式：Monorepo

多个微服务（或项目）代码在同一仓库。优点：可原子提交跨项目变更；代码复用更细粒度。挑战：需要将目录映射到不同构建；大规模时需 Bazel 等图式构建工具；需细粒度所有权控制（如 CODEOWNERS）。

**每团队 Monorepo**：每个团队拥有自己的 Monorepo，包含该团队所有微服务。在保持集体所有权的同时，避免大规模 Monorepo 的复杂性。

## 我的选择

综合考虑，我更倾向于**每微服务一仓库（Multirepo）**。Monorepo 的细粒度复用和原子提交优势，难以抵消规模化后的挑战：鼓励跨服务修改、所有权模糊、需要新工具链。从集体所有权 Monorepo 迁移到其他模型往往成本很高。

## 小结

- CI/CD 是微服务成功的基础
- 优先基于主干开发，避免长期功能分支
- 制品构建一次，全流水线复用
- Multirepo 与 Monorepo 各有适用场景；规模化时 Multirepo 更易管理
