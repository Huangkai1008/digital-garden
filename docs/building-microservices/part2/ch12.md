# 第12章 弹性

分布式系统中故障无处不在。弹性（Resiliency）关注系统在故障下的表现与恢复能力。本章介绍稳定性模式、CAP 与混沌工程。

## 什么是弹性？

- **鲁棒性（Robustness）**：承受冲击的能力
- **回弹（Rebound）**：从故障中恢复的能力
- **优雅扩展（Graceful Extensibility）**：在压力下适度降级而非崩溃
- **持续适应（Sustained Adaptability）**：长期应对变化的能力

## 故障无处不在

微服务架构中，网络、节点、依赖服务都可能失败。需在设计时假设故障会发生，并考虑降级策略。

## 稳定性模式

### 超时（Time-outs）

避免无限等待下游响应，设置合理超时并处理超时后的补偿逻辑。

### 重试（Retries）

对瞬时故障进行有限次重试，需注意**幂等性**——重复请求不应产生重复副作用。

### 舱壁（Bulkheads）

隔离资源，防止单个依赖故障耗尽线程池或连接池，拖垮整个服务。

### 熔断器（Circuit Breaker）

当下游持续失败时，熔断器“打开”，快速失败并避免继续调用。一段时间后尝试“半开”，逐步恢复。

### 隔离（Isolation）

将不可靠依赖与核心路径隔离，减少故障传播。

### 冗余（Redundancy）

多实例、多可用区部署，提高可用性。

## 幂等性

确保重复请求不会导致重复业务效果。对支付、订单创建等操作尤为重要，重试与消息重复投递都依赖幂等设计。

## CAP 定理

在分区（Partition）发生时，系统无法同时保证：

- **C**onsistency（一致性）
- **A**vailability（可用性）
- **P**artition tolerance（分区容错）

通常需在 CP 与 AP 之间权衡，且不同子系统可有不同选择。

## 混沌工程

通过主动注入故障，验证系统在真实故障下的表现。

- **演练日（Game Days）**：计划内、受控的故障演练
- **生产实验**：在生产中做小范围、可逆的故障注入（如 Netflix Chaos Monkey）

::: tip 混沌工程原则
- 先建立稳态假设（如错误率、延迟）
- 引入最小可行故障
- 在生产中实验时控制爆炸半径
- 持续自动化并融入交付流程
:::

## 小结

- 假设故障会发生，设计时考虑降级与恢复
- 超时、重试、熔断器、舱壁等模式可提升稳定性
- 幂等性是重试与消息重复投递的前提
- CAP 提醒我们一致性、可用性、分区容错无法兼得
- 混沌工程帮助验证弹性设计
