# 第16章 演进式架构师

正如我们到目前为止所看到的，微服务给了我们很多选择，因此也有很多决策要做。例如，我们应该使用多少种不同的技术，我们应该让不同的团队使用不同的编程习惯吗，我们应该拆分还是合并微服务？我们如何进行这些决策？随着这些架构允许的更快的变更节奏和更流动的环境，架构师的角色也必须改变。在本章中，我将对架构师的角色采取相当有主见的观点，并希望对象牙塔发起最后一次冲击。

## 名称的含义（What's in a Name?）

> 你一直在用那个词。我不认为它的意思是你想的意思。
> —Inigo Montoya，《公主新娘》

架构师有一份重要的工作。他们负责确保系统有一个连贯的技术愿景，一个应该有助于交付客户需要的软件的愿景。在某些地方，他们可能只需要与一个团队合作，在这种情况下，架构师的角色和技术负责人的角色通常是一体的。在其他地方，他们可能正在为整个工作项目定义愿景，与全球多个团队协调，或者甚至是整个组织。无论架构师在什么层面运作，他们的角色都很难界定，尽管它通常是企业组织中开发者明显的职业发展路径，它也是我们领域中获得比几乎任何其他角色更多批评的角色。比任何其他角色都多，架构师可以直接影响所构建系统的质量、同事的工作条件以及组织响应变更的能力，然而他们的角色似乎被理解得非常差。为什么会这样？

## 什么是软件架构？（What Is Software Architecture?）

我们的行业很年轻。我们有时似乎忘记了，我们创建在今天我们认可的计算机上运行的程序只有大约 75 年。我们的职业不适合社会整体理解的整齐盒子。我们不像电工、水管工、医生或工程师。有多少次你在聚会上告诉别人你做什么，结果对话就停止了？整个世界都在努力理解软件开发——正如我在本书中多次概述的，我们似乎经常自己也不理解它。所以我们从其他职业借用。我们称自己为软件「工程师」或「架构师」。但我们不是社会理解的那些职业意义上的架构师或工程师。架构师和工程师有我们只能梦想的严谨和纪律，他们在社会中的重要性被很好地理解。我记得在我朋友成为合格架构师的前一天和他交谈。「明天，」他说，「如果我在酒吧给你关于如何建造某物的建议而它是错的，我要承担责任。我可能会被起诉，因为在法律眼中我现在是合格的架构师，如果我搞错了，我应该承担责任。」这些工作对社会的重要性意味着人们必须满足所需的资格。例如，在英国，在被称架构师之前需要至少七年的学习。但这些工作也基于可以追溯到数千年的知识体系。而软件架构师呢？不太一样。这 partly 是我为什么认为许多形式的 IT 认证毫无价值，因为我们对「好」是什么样子知之甚少。

我这样说不是为了贬低软件工程这个术语（由 Margaret Hamilton 在 1960 年代创造），它既是关于当前现实的，也是 aspirational 的。这个术语的出现是为了呼吁提高正在创建的软件的质量，并认识到软件项目经常失败， yet 越来越多地被用于关键任务和安全关键领域。自那时以来已经做了大量工作来改善情况，但我在行业 20 年后的看法是，我们仍然有很多关于做好（或至少做得更好）工作要学的。我们中的一部分人想要认可，所以我们从已经有我们渴望的认可的其他职业借用名称。但这可能被证明是有问题的，如果我们从那些职业借用工作实践而不理解其背后的心态或考虑软件开发与土木工程等的不同。这一切都不应被视为我们不应该旨在工作中更有严谨性的论据——只是我们不能简单地从其他地方借用想法并假设它们对我们有效。我们的行业非常年轻，挑战是我们在行业层面同意的绝对事物要少得多。

也许架构师这个术语，或者至少对架构师做什么的普遍理解，在这方面造成了最大的伤害：有人绘制详细计划供他人解释并期望他们被执行的想法；部分艺术家、部分工程师的平衡，监督往往是单一愿景的创作，所有其他观点都服从，除了结构工程师偶尔对物理定律的反对。在我们行业，这种架构师的观点导致了一些可怕的实践，架构师创建一张又一张的图表、一页又一页的文档，旨在告知完美系统的构建，而未能考虑根本不可知的未来，完全缺乏对其计划实施有多难或它们是否实际有效的任何理解，更不用说在我们学到更多时进行变更的能力。但建筑环境的架构师在不同于软件架构师的领域运作。他们的约束不同，最终产品不同。建筑中的变更成本比软件开发中高得多。你不能倒掉混凝土，但你可以更改代码，甚至我们运行代码的基础设施也比以前更具可塑性，这要归功于虚拟化。建筑物一旦建成就相当固定——它们可以被更改、扩建或拆除，但相关成本非常高。但我们期望我们的软件不断变更以满足我们的需求。所以如果软件架构与建筑环境的架构不同，也许我们应该更清楚地说明软件架构实际上是什么。

软件架构最著名的定义之一来自 Ralph Johnson 的电子邮件：「架构是关于重要的东西。无论那是什么。」所以这是否意味着任何重要的东西都是由架构师完成的？这是否意味着正在完成的所有其他工作都不重要？这个经常被引用的陈述的问题在于，它经常被孤立使用，对 Ralph 分享它的更广泛回应没有任何理解。首先，很明显他是从软件开发者的角度谈论的。他继续说：

> 所以，更好的定义是「在大多数成功的软件项目中，在该项目上工作的专家开发者对系统设计有共同的理解。这种共同的理解被称为『架构』。这种理解包括系统如何划分为组件以及组件如何通过接口交互。这些组件通常由更小的组件组成，但架构只包括所有开发者理解的组件和接口。」

这将是一个更好的定义，因为它明确表示架构是一种社会建构（嗯，软件也是，但架构更是如此），因为它不仅取决于软件，还取决于群体共识认为软件的哪部分重要。这里 Ralph 在最一般的意义上使用组件这个术语。在本书的背景下，我们可以将组件视为我们的微服务，也许是这些微服务内部的模块。软件架构是系统的形状。架构发生，无论是有意还是意外。我们做出一系列临时决策，我们最终得到结果——不按架构思考事情，我们仍然最终得到架构。架构有时可以是我们忙于制定其他计划时发生的事情。专职架构师是应该看到和理解整个系统、理解作用于其上的力量的人。他们需要确保有一个适合目的并被清楚理解的架构愿景——一个满足系统及其用户需求以及系统上工作的人的需求的架构愿景。只关注一个方面——例如逻辑而非物理、形状而非开发者体验——限制了架构师的有效性。如果你接受架构是关于理解系统，那么限制你关心的范围就限制了你的推理和进行变更的能力。架构对与之共存的人来说可能是不可见的。它可以如此轻微以至于并不真正存在。它可以是指引和帮助实现正确结果的东西。它可以令人窒息和专横。它可以在你甚至没有意识到它存在的情况下令人愉悦，并在没有任何恶意的情况下压垮你的精神。所以无论架构是否「关于重要的东西」，它肯定很重要。

另一个经常用来定义软件架构的简洁引用来自 Martin 分享 Ralph 观点的同一篇文章：「所以你可能会最终将架构定义为人们认为难以变更的东西。」Martin 关于架构是难以变更的东西的想法在某种程度上是有道理的，并将我们带回到建筑环境中架构的概念。在事物更难变更的地方，它们需要更多的前期思考来真正确保我们朝正确方向前进。但将复杂想法的简单定义作为工作定义运行有一个问题——如果这个陈述完全是你对软件架构的思考方式，你会错过很多。是的，很多软件架构是关于思考难以变更的事情，但也是关于创造允许设计中变更的空间。

## 让变更成为可能（Making Change Possible）

回到建筑物而非软件系统的世界：建筑师 Mies van der Rohe 可以说比任何其他建筑师都更多地开创了我们现在认为的现代摩天大楼——他著名的西格拉姆大厦成为随后许多建筑的蓝图。西格拉姆大厦与之前的许多建筑不同。建筑物的外墙是非结构性的——它们包裹着钢制外框架。主要建筑服务——电梯、楼梯、空调、水和废物以及电气系统——穿过中央混凝土核心。今天观看现代高层建筑的建设，首先建造的是这个中央混凝土核心，经常看到巨型起重机栖息在顶部。西格拉姆大厦的每一层都没有内部结构墙——这意味着你在空间使用方面有完全的灵活性。你可以根据需要重新配置空间，通过吊顶和地板本身的管道将电线和空调路由到每层的不同部分。有趣的是，西格拉姆大厦是使用建筑进行时设计演变的过程开发的。现在我们以前在哪里见过这个想法？这个设计背后的想法是交付 Mies van der Rohe 所说的「通用空间」——一个大的、单跨的体积，可以重新配置以满足不同的需求。建筑物的使用会改变，所以想法是交付在如何使用方面尽可能灵活的空间。这样，Mies van der Rohe 不仅必须专注于建筑的基本美学，为核心服务找到如果以后难以（如果不是不可能）变更的空间，还必须确保建筑物可以以最初设想以外的不同方式使用。 shortly，我们将看看我们如何在微服务架构的空间中允许变更。

## 架构师的演进愿景（An Evolutionary Vision for the Architect）

我们作为软件架构师的需求比设计和建造建筑物的人变化得更快——我们可用的工具和技术也是如此。我们创造的东西不是时间上的固定点。一旦投入生产，我们的软件将随着使用方式的改变而继续演变。对于我们创造的大多数东西，我们必须接受，一旦软件交到用户手中，我们将不得不反应和适应，而不是期望一个永不改变的制品。因此，软件架构师需要将思维从创造完美的最终产品转变，转而专注于帮助创造一个框架，在这个框架中正确的系统可以涌现并随着我们学到更多而继续成长。尽管我在本章中花了很多时间 discouraging 与其他职业的比较，但有一个类比我喜欢，当谈到 IT 架构师的角色时，我认为很好地概括了角色的这一方面。Thoughtworks 的 Erik Doernenburg 首先与我分享了我们应该更多地将架构师的角色视为城市规划师而非建筑环境建筑师的想法。城市规划师的角色对于玩过《模拟城市》或《城市：天际线》的任何人来说应该很熟悉。城市规划师的角色是查看大量信息来源，然后尝试优化城市的布局以最适合当今公民的需求，同时考虑未来的使用。然而，他们影响城市演变的方式很有趣。他们不说「在那里建这座具体的建筑」；相反，他们定义允许在特定约束内进行局部决策的区域。所以，就像在《模拟城市》中一样，你可能将城市的一部分指定为工业区，另一部分为住宅区。然后由其他人决定创建什么建筑，但有限制：如果你想建工厂，它需要在工业区。城市规划师不会太担心一个区域内发生什么，而是会花更多时间弄清楚人和公用事业如何从一个区域移动到另一个区域。不止一个人将城市比作生物。城市随时间变化。它随着居住者以不同方式使用它或外部力量塑造它而转变和演变。城市规划师尽最大努力预测这些变化，但接受试图对所有方面施加直接控制是徒劳的。所以我们的架构师作为城市规划师需要以 broad strokes 设定方向，只在有限的情况下对实现细节高度具体。他们需要确保系统现在适合目的，但也是未来的平台。与软件的类比应该是显而易见的。随着用户使用我们的软件，我们需要反应和改变。我们无法预见将要发生的一切，所以与其为每一种可能的情况做计划，我们应该计划通过避免过度规定每一件事的冲动来允许变更。我们的城市——系统——需要成为每个使用它的人的好地方、快乐的地方。

## 定义系统边界（Defining System Boundaries）

继续架构师作为城市规划师的比喻一会儿，我们的区域是什么？这些是我们的微服务边界，也许是微服务的粗粒度组。作为架构师，我们需要少担心区域内发生什么，多担心区域之间发生什么。这意味着我们需要花时间思考我们的微服务如何相互通信，并确保我们能够正确监控系统的整体健康状况。从架构空间来看，这是我们创造自己的通用空间的方式——通过定义一些特定的边界，我们向构建系统的同事突出显示那些可以更自由地进行变更而不会破坏我们架构某些基本方面的区域。看一个非常简单的例子，在图 16-1 中我们看到 Recommendations 微服务从 Promotions 和 Sales 微服务访问信息。正如我们已经详细讨论的，我们可以自由地更改隐藏在这三个微服务内部的功能，而不必担心破坏整个系统——只要我继续维护 Recommendations 对这些下游微服务如何交互的期望，我可以在 Sales 或 Promotions 中更改任何我想要的东西。

```
+------------------+     +------------------+
| Recommendations  | --> | Promotions       |
|                  | --> | Sales            |
+------------------+     +------------------+
```

**图 16-1** 只要微服务之间的交互不变，微服务边界内的变更很容易进行

我们也可以在更大范围的层面上为变更创造空间。在图 16-2 中，我们看到图 16-1 中的微服务实际上存在于映射到特定团队责任的营销区域中。我们定义了营销功能如何与更大系统交互的预期行为。在营销区域内，我们可以进行任何我们喜欢的变更，只要保持与更大系统的兼容性。回到理解难以变更的事情的想法，组织结构通常属于这一类，因此现有的团队结构可以帮助你定义这些区域。在团队拥有的微服务内协调变更比更改暴露给其他团队的交互更容易。

```
+------------------------------------------+
| Marketing Zone (Team-owned)              |
|  +------------------+  +------------------+
|  | Recommendations  |  | Promotions        |
|  | Sales            |  | ...              |
|  +------------------+  +------------------+
+------------------------------------------+
```

**图 16-2** 区域内变更比区域间变更更容易进行

这与我们在「小团队，大组织」中讨论的团队 API 概念很好地契合。架构师可以帮助促进团队 API 的创建，确保团队的微服务和工作实践与更广泛的组织契合。通过定义可以在不损害整个系统的情况下进行这些变更的空间，我们使开发者的生活更轻松，也将我们的注意力集中在更难变更的系统部分。还记得我们在第 2 章探讨的信息隐藏概念吗？正如我们在那里探讨的，将信息隐藏在微服务边界内使为消费者创建稳定接口变得容易得多。当我们对微服务进行变更时，更容易确保我们没有破坏与外部消费者的兼容性。在这里，我们可以定义架构以在团队层面提供信息隐藏，而不仅仅是在微服务层面。这给了我们另一层信息隐藏，并创造了一个更大的安全空间，团队可以在其中进行局部变更而不会破坏更广泛的系统。在每个微服务或更大的区域内，你可能对拥有该区域的团队选择不同的技术栈或数据存储感到满意。当然，其他关注点可能会在这里介入。你让团队为工作选择正确工具的倾向可能会因为如果你有 10 种不同的技术栈要支持，招聘人或让他们在团队之间移动变得更难而有所节制。同样，如果每个团队选择完全不同的数据存储，你可能会发现自己缺乏足够的经验来大规模运行它们中的任何一个。例如，Netflix 大多标准化了 Cassandra 作为数据存储技术。尽管它可能不是所有情况的最佳选择，Netflix 认为围绕 Cassandra 构建工具和专业知识的价值比不得不大规模支持和运营可能更适合某些任务的其他多个平台更重要。Netflix 是一个极端例子，规模可能是最强的压倒性因素，但你明白这个意思。然而，微服务之间可能会变得混乱。如果一个微服务决定暴露 REST over HTTP，另一个使用 gRPC，第三个使用 Java RMI，那么集成可能成为噩梦，因为消费微服务必须理解和支持多种交换风格。这就是为什么我尽量坚持我们应该「担心盒子之间发生什么，对盒子内部发生什么保持宽松」的准则。所以成功的架构既关乎允许变更以满足我们用户的需求。但人们经常忘记的一件事是，我们的系统不仅容纳用户；它还容纳实际构建软件的人。成功的架构还有助于创造一个做我们工作的好环境。

## 社会建构（A Social Construct）

> 没有计划能在与敌人接触后幸存。
> —Helmuth von Moltke（ heavily 意译）

所以你已经考虑了愿景、约束以及你需要完成什么。你认为你理解什么会难以变更，以及你需要让变更成为可能的空间。现在呢？嗯，架构是发生的事情，而不是你认为应该发生的事情——这是愿景与现实之间的区别。建筑环境的架构师需要与建造建筑物的人合作，帮助他们理解愿景是什么，但当现实挑战该愿景时也要改变计划。你认为可能的事情从根本上可能不可能。如果架构师在某种程度上没有与创建系统的人嵌入，那么他们将无法帮助向做工作的人传达愿景，架构师也不会理解该愿景在哪里不再适合目的。施工人员可能在地面上遇到没有预料到的事情，或者也许供应短缺可能导致设计的重新考虑。架构是发生的事情，而不是计划的事情。如果作为架构师你将自己从将这一愿景付诸实践的过程中移除，那么你不是架构师——你是梦想家。将涌现的架构可能与你想的没有任何关系。它会发生，有你或没有你。实现架构需要许多人的工作和大量大大小小、有意无意的设计决策。正如 Grady Booch 所说：

> 起初，软件密集型系统的架构是愿景的陈述。最后，每个这样的系统的[a]rchitecture 是沿途做出的数十亿大大小小、有意无意的设计决策的反映。

这意味着即使你有一个最终对架构负责的专职个人，也有许多人负责将这一愿景付诸实践。实现成功的架构将是团队努力。回到 Ralph Johnson 之前的引用，「架构是一种社会建构。」一个很好的例子来自 Comcast，它分享了如何通过使用架构公会（architecture guild）去中心化决策的经验。鉴于其规模，Comcast 决定借鉴行业指导小组的经验，其中集体决策是关键：

> 在 Comcast，我们意识到这个问题看起来与开放标准机构的工作方式非常相似：让多个自主团体就技术方法达成一致。我们设计了一个内部架构公会，明确模仿一个非常成功的标准机构，即定义许多重要互联网协议的互联网工程任务组（IETF）。
> —Comcast Cable 首席软件架构师 Jon Moore

Comcast 的方法有一定程度的正式性，一些组织可能会觉得繁重，但鉴于其规模和分布，它似乎对该公司运作良好。

## 宜居性（Habitability）

另一个来自建筑环境并在软件开发领域产生共鸣的概念是宜居性（habitability）。我首先从 Frank Buschmann 那里了解到这个术语——他解释说架构师有责任确保他们创造的环境适合工作。如果架构是描述难以变更的事物如何组合在一起的系统框架，那么有时可能也需要施加约束。搞错了，在系统中工作可能变得痛苦和容易出错。正如《Patterns of Software》的作者 Richard Gabriel 所解释的：

> 宜居性是源代码的特征，使后来接触代码的程序员能够理解其结构和意图，并舒适自信地更改它。

然而，现代软件开发生态系统不仅仅包括代码——它延伸到我们使用的技术和我们采用的工作实践。我经常看到开发者诅咒他们被告知使用的技术——通常是由从未不得不使用它的人选择的技术。你越是让架构的演变和你使用的工具和技术的选择成为协作过程，就越容易确保最终结果是一个宜居的环境，构建系统的人在其中感到快乐和高效。如果我们要确保我们创造的系统对我们的开发者是宜居的，那么我们的架构师和其他决策者需要理解其决策的影响。至少，这意味着花时间与团队在一起，理想情况下实际上花时间与团队一起编码。对于你们中实践结对编程的人，架构师作为结对的一方加入团队一小段时间成为简单的事情。参与合奏编程练习也可能产生显著好处，尽管参与此类小组活动的架构师需要意识到他们的存在可能会改变合奏的动态。理想情况下，你应该在正常任务上工作，以真正理解「正常」工作是什么样子。我无法强调架构师实际花时间与构建系统的团队在一起有多重要！这比打电话或只是看他们的代码要有效得多。至于你应该多久做一次，这在很大程度上取决于你合作的团队规模。但关键是这应该是常规活动。例如，如果你与四个团队合作，也许确保每四周与每个团队度过半天，与他们一起完成交付任务，以确保你与合作的团队建立意识和改善沟通。

## 有原则的方法（A Principled Approach）

> 规则是为傻瓜遵守、为智者指导的。
> —通常归因于 Douglas Bader

系统设计中的决策都是关于权衡的，微服务架构给了我们很多权衡要做！选择数据存储时，我们选择我们经验较少但能更好扩展的平台吗？我们的系统中有两种不同的技术栈可以吗？三种呢？有些决策可以完全根据我们掌握的信息当场做出，这些是最容易的。但那些可能需要在信息不完整的情况下做出的决策呢？框架可以在这里提供帮助，帮助框架我们决策的好方法是定义一组基于我们试图实现的目标指导它的原则和实践。让我们依次看看框架的这些方面。

**战略目标（Strategic Goals）**

架构师的角色已经足够令人生畏了，所以幸运的是我们通常不必也定义战略目标！战略目标应该说明你的公司要去哪里以及它如何看待自己最好地让客户满意。这些将是高层次的目标，可能根本不包含技术。它们可以在公司层面或部门层面定义。它们可能是「扩展到东南亚以解锁新市场」或「让客户使用自助服务尽可能多地实现」之类的事情。关键是它们定义了你的组织的方向，所以你需要确保技术与之一致。如果你是定义公司技术愿景的人，这可能意味着你需要花更多时间与组织的非技术部分（或通常所说的「业务」）在一起。业务的驱动愿景是什么？它是如何变化的？

**原则（Principles）**

原则是你为了将你所做的与某个更大的目标对齐而制定的规则，它们有时会改变。例如，如果你作为组织的战略目标之一是减少新功能的上市时间，你可以定义一个原则，说交付团队在其软件的整个生命周期中拥有完全控制权，可以在准备好时独立于任何其他团队发布。如果另一个目标是你的组织正在积极扩大其在其他国家的产品，你可以决定实施一个原则，即整个系统必须是可移植的，以便可以本地部署以尊重数据主权。你可能不想要很多这些。少于 10 条原则是一个好数字——足够少让人们记住它们，或适合小海报。你有的原则越多，它们重叠或相互矛盾的可能性就越大。Heroku 的十二要素（Twelve Factors）是一组围绕帮助你在 Heroku 平台上创建运行良好的应用程序的目标构建的设计原则。这些原则在其他背景下可能也有意义。其中一些实际上是基于你的应用程序需要在 Heroku 上工作所需表现的行为的约束。约束真的是非常难（或几乎不可能）改变的东西，而原则是我们决定选择的东西。你可以决定明确指出哪些是原则、哪些是约束，以帮助突出你真正无法改变的事情。就个人而言，我认为将它们保留在同一列表中可能有一些价值，以鼓励不时挑战约束，看看它们是否真的不可动摇！

**实践（Practices）**

我们的实践是我们如何确保我们的原则得到执行。它们是一组详细的、实用的任务执行指南。它们通常是技术特定的，应该足够低级，任何开发者都能理解。实践可以包括编码指南、所有日志数据需要集中捕获的事实，或 HTTP/REST 是标准集成风格的事实。由于其技术性质，实践通常比原则更频繁地改变。与原则一样，有时实践反映你组织内的约束。例如，如果你决定选择 Azure 作为你的云平台，这需要反映在你的实践中。实践应该支撑你的原则。一个说交付团队控制其系统全生命周期的原则可能意味着你有一个实践，说所有微服务都部署到隔离的 AWS 账户中，提供资源的自助管理和与其他团队的隔离。

**结合原则和实践**

一个人的原则是另一个人的实践。例如，你可能决定将 HTTP/REST 的使用称为原则而非实践。那没问题。关键是有价值的是拥有指导系统演变的 overarching 想法，以及有足够的细节让人们知道如何实现这些想法。对于足够小的群体，也许单一团队，结合原则和实践可能没问题。然而，对于更大的组织，技术和工作实践可能因地点而异，你可能希望在不同地方有不同的实践集，只要它们都映射到共同的原则集。例如，.NET 团队可能有一套实践，Java 团队有另一套。但原则对两者可能是相同的。

## 引导演进式架构（Guiding an Evolutionary Architecture）

所以如果我们的架构不是静态的而是不断变化和演进的，我们如何确保它以我们想要的方式成长和改变，而不是只是变异成某种难以管理的巨大痛苦、苦难和相互指责的 blob？在《Building Evolutionary Architectures》中，作者概述了适应性函数（fitness functions）来帮助收集关于架构相对「适应性」的信息，以帮助架构师决定是否需要采取行动。书中写道：

> 进化计算包括许多机制，允许解决方案通过软件每一代的小变化逐渐涌现。在每一代解决方案中，工程师评估当前状态：它是更接近还是远离最终目标？例如，当使用遗传算法优化机翼设计时，适应性函数评估风阻、重量、气流和其他对良好机翼设计 desirable 的特征。架构师定义适应性函数来解释什么是更好的，并帮助衡量何时达到目标。在软件中，适应性函数检查开发者是否保留了重要的架构特征。

适应性函数的想法是它用于理解某些重要属性的当前状态，这样如果该属性在某个允许范围之外改变，则需要调查该变更。通常，适应性函数将用于确保架构正在按照已制定的原则和约束构建。借用《Building Evolutionary Architectures》的例子，考虑给定服务的响应必须在 100 毫秒或更短时间内收到的要求。你可以实现一个适应性函数来从该服务收集性能数据，也许在性能测试环境中或从真实运行系统中，以确保系统的实际行为满足要求。《Building Evolutionary Architectures》在这个话题上有很多更多细节，如果你想进一步探讨这个概念，我强烈推荐。架构的适应性函数可以有许多形状和形式。然而，基本概念是你收集真实世界数据来理解你的架构是否正在实现针对该标准的「适应性」。这可能与系统性能、代码耦合、周期时间或许多其他方面相关。这些适应性函数充当另一个信息来源，帮助架构师理解他们可能需要参与的地方。但请注意，对我来说，适应性函数在与构建系统的人密切合作时效果最好。适应性函数应该是帮助你理解架构是否朝正确方向发展的有用方式，但它们不能替代实际与一线人员交谈的需要。事实上，我建议定义正确的适应性函数将需要密切合作。

## 流对齐组织中的架构（Architecture in a Stream-Aligned Organization）

在第 15 章中，我们看了现代软件交付组织如何正在转向更流对齐的模型，其中自主的独立团队专注于功能的端到端交付，其优先级由产品驱动。我们还谈到了横切团队——支持流对齐团队的赋能团队。架构师在这个世界中适合哪里？嗯，有时流对齐团队的范围足够复杂，需要专职架构师（这里我们再次看到传统技术负责人和架构师角色之间的模糊）。但在许多情况下，架构师被要求跨多个团队工作。架构师的许多职责可以被视为赋能职责——清晰传达技术愿景、理解出现的挑战并帮助相应调整技术愿景。架构师帮助连接人们，关注大局并帮助团队理解他们正在做的事情如何融入更大的整体。这很好地契合了架构师作为赋能团队一部分的想法，如图 16-4 所示。这样的赋能团队可以由混合的人组成——也许是全职 dedicated 于团队的人，以及其他不时帮忙的人。

我非常喜欢的一个模型是在这个团队中有少量专职架构师（在许多情况下可能只有一两个人），但随着时间的推移用来自每个交付团队的技术人员来增强这个团队——至少每个团队的技术负责人。架构师负责确保小组运作。这分配了工作并确保有更高程度的认同。它还确保信息从团队自由流入小组，因此决策更加明智和知情。有时小组可能做出架构师不同意的决定。此时，架构师该怎么办？以前处于这个位置，我可以告诉你这是面临的最具挑战性的情况之一。我经常采取我应该遵循小组决定的方法。我持这样的观点：我已经尽力说服人们，但最终我不够有说服力。小组往往比个人更明智，我不止一次被证明是错的！想象一下，给一个小组空间来做出决定，然后最终被忽视，这有多令人沮丧。但有时我否决了小组。但为什么，什么时候？你如何划清界限？想想教孩子骑自行车。你不能替他们骑。你看着他们摇晃，但如果你每次看起来他们可能会摔倒时都介入，那么他们永远不会学会，而且无论如何他们摔倒的次数比你想象的少得多！但如果你看到他们即将转向交通或附近的鸭塘，那么你必须介入。当然，我在这种情况下经常被证明是错的——我让团队去做我觉得是错误的事情，他们做的有效！同样，作为架构师，你需要牢牢掌握什么时候，比喻地说，你的团队正在转向鸭塘。你还需要意识到，即使你知道你是对的并否决了团队，这也可能破坏你的地位，也让团队觉得他们没有发言权。有时正确的事情是同意你不同意的决定。知道什么时候这样做、什么时候不这样做很难，但有时至关重要。当架构师还必须参与治理活动时，事情变得有趣，我们 shortly 会讨论。这可能导致对任何横切架构团队角色的困惑。当一个团队偏离技术战略时会发生什么？可以吗？也许这是一个明智的例外，但也可能导致更根本的问题。以权宜之计名义做出的短期决定可能会损害正在尝试进行的更大变更。想象架构小组正试图帮助组织摆脱集中式数据的使用，因为它造成的耦合和运维问题，但其中一个团队决定只是将一些新数据扔进共享数据库，因为它承受着快速交付的压力。那时会发生什么？根据我的经验，这一切都归结为良好、清晰的沟通和对职责的理解。如果我看到产品负责人做出我觉得会破坏我正在进行的某种横切活动的决定，我会去和他们聊聊。也许答案是短期决定是正确的（可以说这最终成为我们有意识地承担的技术债务）。在其他情况下，也许产品负责人能够改变他们计划的内容以帮助配合整体战略。在最坏的情况下，问题可能需要升级。在我在前几章谈到的在线房地产公司 REA，产品负责人偶尔会做出以导致技术债务积累的方式优先考虑工作的决定，导致后续问题。问题是产品负责人主要对其交付功能和让客户满意的能力负责——而技术债务的问题往往落在技术负责人的肩上。进行了转变，使产品负责人也对软件的技术方面负责——这意味着他们必须在理解系统的更多技术方面（如安全或性能）方面发挥更积极的作用，并在优先考虑要完成的工作方面与技术专家更协作。让非技术产品负责人对技术活动的优先考虑更负责并非易事，但根据我的经验绝对值得。

## 组建团队（Building a Team）

成为系统技术愿景的主要负责人并确保你正在执行这一愿景，不仅仅是确保做出正确的技术决策。与你一起工作的人将完成工作。任何技术负责人的角色都是帮助这些人成长——帮助他们成为创造该愿景的一部分——并确保他们也能积极参与塑造和实施该愿景。帮助你周围的人进行职业成长可以采取多种形式，其中大多数超出了本书的范围。不过，有一个方面微服务架构特别相关。有了更大的单体系统，人们挺身而出「拥有」某物的机会更少。有了微服务，另一方面，我们有多个自主的代码库，它们将有自己的独立生命周期。通过让人们在接受更多责任之前拥有单个微服务来帮助他们挺身而出，可以是帮助他们实现自己职业目标的好方法，同时也减轻了负责人的负担！我坚信伟大的软件来自伟大的人。如果你只担心等式的技术方面，你错过了超过一半的画面。

## 所需标准（The Required Standard）

当你在实践和思考需要做出的权衡时，要找到的最重要的平衡之一是在系统中允许多少可变性。识别微服务之间应该保持不变的关键方法之一是定义行为良好、优秀的微服务在你系统中的样子。在你系统中什么是「好公民」微服务？它需要有什么能力来确保你的系统是可管理的，一个坏的微服务不会拖垮整个系统？与人一样，在一种背景下「好公民」微服务的样子并不反映它在其他地方的样子。尽管如此，我认为有一些行为良好的微服务的共同特征相当重要。这些是允许太多分歧可能导致相当可怕时间的少数关键领域。正如 Facebook 的 Ben Christensen 所说，当你考虑更大的画面时，「它需要是一个由许多具有自主生命周期但都组合在一起的小部分组成的 cohesive 系统。」所以你需要找到一个平衡，在其中你优化单个微服务的自主权，同时不忽视更大的画面。定义每个微服务应该具有的清晰属性是明确该平衡所在的一种方式。让我们谈谈其中一些属性。

**监控（Monitoring）**

我们能够绘制系统健康的连贯、跨服务视图是必不可少的。这必须是系统范围的视图，而不是微服务特定的视图。正如我们在第 10 章讨论的，了解单个微服务的健康状况是有用的，但通常只有在试图诊断更广泛的问题或理解更大趋势时。为了让这尽可能容易，我建议确保所有微服务以相同的方式发出与健康相关的和一般监控相关的指标。你可以选择采用推送机制，每个微服务需要将这些数据推送到中央位置。无论你选择什么，尽量保持标准化。让盒子内的技术不透明，不要要求你的监控系统改变以支持它。日志属于这里的同一类别：我们需要它在一个地方。

**接口（Interfaces）**

选择少量定义的接口技术有助于集成新消费者。有一个标准是好的。两个也不错。有二十种不同的集成风格不好。这不仅仅是选择技术和协议。例如，如果你选择 HTTP/REST，你会使用动词还是名词？你将如何处理资源的分页？你将如何处理端点的版本控制？

**架构安全（Architectural Safety）**

我们负担不起一个行为不当的微服务毁了每个人的派对。我们必须确保我们的微服务相应地屏蔽自己不健康的、下游的调用。我们拥有的不正确处理下游调用潜在故障的微服务越多，我们的系统就越脆弱。这可能意味着，例如，你想强制要求围绕服务间通信的某些实践，例如要求使用断路器（我们在「稳定性模式」中探讨过的话题）。遵守规则在响应代码方面也很重要。如果你的断路器依赖 HTTP 代码，而一个微服务决定为错误发送回 2XX 代码或将 4XX 代码与 5XX 代码混淆，那么这些安全措施可能会崩溃。即使你不使用 HTTP，类似的关注也适用；我们需要知道请求正常并正确处理、请求错误因此微服务无法对其做任何事、以及请求可能正常但我们无法判断因为服务器宕机之间的区别。知道这一点对于确保我们能够快速失败并追踪问题至关重要。如果我们的微服务对这些规则敷衍了事，我们最终会得到一个更脆弱的系统。

## 治理与铺好的路（Governance and the Paved Road）

架构师需要处理的一部分是治理。我所说的治理是什么意思？事实证明 COBIT（信息技术的控制目标）框架有一个相当好的定义：

> 治理通过评估利益相关者需求、条件和选项；通过优先排序和决策设定方向；以及监控绩效、合规性和与商定方向和目标的进展，确保实现企业目标。

简而言之，我们可以将治理视为商定事情应该如何完成，确保人们知道事情应该如何完成，并确保事情以那种方式完成。在某些环境中，治理只是非正式地发生，作为正常软件开发活动的一部分。在其他环境中，特别是在更大的组织内，这可能需要是一个更具体的职能。治理可以适用于 IT 论坛中的多个事物。我们想关注技术治理的方面，我觉得这是架构师的工作。如果架构师的工作之一是确保有技术愿景，那么治理就是确保我们正在构建的与这一愿景匹配，并在需要时演变愿景。从根本上说，治理应该是小组活动。一个正常运作的治理小组可以一起工作来分担工作和塑造愿景。对于足够小的团队，这可能是一次非正式的聊天，或者对于更大范围，可能是具有正式小组成员资格的更有结构的定期会议。我认为这是我们之前介绍的原则应该被讨论和根据需要改变的地方。如果需要正式小组，该小组需要主要由正在执行被治理工作的人组成。该小组还应该负责跟踪和管理技术风险。聚在一起商定事情如何完成是个好主意。但花时间确保人们遵循这些指南就不那么有趣了，给开发者施加负担来实现你期望每个微服务做的所有这些标准事情也是如此。我坚信让做正确的事变得容易——正如我们在第 15 章讨论的，铺好的路在这里是一个非常有用的概念。架构师有责任清晰表达愿景——你要去哪里——并让到达那里变得容易。因此，他们应该参与帮助塑造你构建的任何铺好的路的要求。对许多人来说，平台将是最大的例子——架构师最终成为平台团队的重要利益相关者。我们已经深入研究了平台的角色，所以让我们看看我们可以用来让人们尽可能容易地做正确的事的另外几个技术。

**范例（Exemplars）**

书面文档是好的和有用的。我 clearly 看到它的价值；毕竟我写了这本书。但开发者也喜欢代码——他们可以运行和探索的代码。如果你有一套你希望鼓励的标准或最佳实践，那么拥有可以指向人们的范例是有用的。想法是人们通过模仿你系统的一些更好的部分不会走得太远。理想情况下，这些应该是你系统中正确做事的真实微服务，而不是仅仅作为「完美例子」实现的隔离微服务。通过确保你的范例实际上正在被使用，你确保你拥有的所有原则实际上都有意义。

**定制微服务模板（Tailored Microservice Template）**

如果你能让所有开发者以很少的工作遵循你拥有的大多数指南，那不是很好吗？如果开箱即用，开发者已经有了实现每个微服务需要的核心属性的大部分代码呢？许多针对不同编程语言的框架试图为你自己的微服务模板提供构建块。Spring Boot 可能是 JVM 上这类框架最成功的例子。核心 Spring Boot 框架相当轻量，但然后你可以决定拉取一组库来提供检查健康、提供 HTTP 或暴露指标等功能。所以开箱即用，你有一个可以从命令行启动的简单「Hello World」微服务。然后许多人将这些框架标准化为他们的公司设置。例如，在启动新微服务时，他们可能会编写脚本，以便他们获得已经接入其组织使用的核心库的 Spring Boot 模板；它可能已经拉入处理断路器的库并配置为处理入站调用的 JWT 认证。通常，这样的自动化模板创建也会创建匹配的构建管道。

**需要谨慎**

这些定制微服务模板的选择和配置通常是平台团队的任务。例如，他们可能为每种支持的语言提供模板，确保使用模板时产生的微服务与平台本身良好配合。然而，这可能带来挑战。我见过许多团队的士气和生产力被强加给他们的强制框架摧毁。在推动改善代码复用的过程中，越来越多的工作被放入集中式框架，直到它成为压倒性的怪物。如果你决定使用定制微服务模板，非常仔细地考虑它的工作是什么。理想情况下，它的使用应该是纯粹可选的，但如果你要更强制地采用它，你需要理解开发者易用性必须是主要的指导力量。允许使用模板的开发者推荐甚至贡献对框架的变更，也许作为内部开源模型的一部分，可以在这里有很大帮助。正如我们在「DRY 与微服务世界中代码复用的危险」中讨论的，我们必须意识到共享代码的危险。在我们创建可复用代码的愿望中，我们可以在微服务之间引入耦合来源。至少我交谈过的一个组织非常担心这一点，以至于它实际上手动将微服务模板代码复制到每个微服务中。这意味着对核心微服务模板的升级需要更长时间才能应用到整个系统，但这比耦合的危险更不令人担忧。我交谈过的其他团队只是将微服务模板视为共享二进制依赖，尽管他们必须非常 diligent 地不让 DRY（不要重复自己）的倾向导致过度耦合的系统！

**大规模铺好的路**

内部微服务模板和框架的使用通常出现在拥有大量微服务的组织中。Netflix 和 Monzo 是两个这样的组织。每个都决定在一定程度上标准化其技术栈（Netflix 的情况是 JVM，Monzo 的情况是 Go），允许它通过使用通用工具集加快创建具有标准、预期行为的新微服务。有了更多 divergent 技术栈，拥有适合自己需求的标准微服务模板变得更加困难。如果你要 embrace 多种不同的技术栈，你需要为每种匹配的微服务模板。不过，这可能是你 subtly 限制团队语言选择的一种方式。如果内部微服务模板只支持 JVM，那么如果人们必须自己做更多工作，可能会 discourage 选择替代栈。例如，Netflix 特别关注容错等方面，以确保其系统一部分的中断不能使一切瘫痪。为了处理这个问题，已经做了大量工作，确保 JVM 上有客户端库，为团队提供保持其微服务行为良好所需的工具。引入新技术栈将意味着不得不复制所有这些努力。Netflix 的主要关注点 less 是关于重复的努力，more 是关于如此容易搞错这一事实。如果新实现的容错可能影响系统的更多部分，微服务搞错的风险很高。Netflix 通过使用与使用适当库的 JVM 本地通信的「边车服务」（sidecar services）来缓解这一问题。服务网格（service meshes）给了我们另一种卸载通用行为的潜在方式。一些通常被视为内部微服务责任的功能现在可以推送到微服务网格。这可以确保用不同编程语言编写的微服务之间更一致的行为，也可以减少这些微服务模板的职责。

## 技术债务（Technical Debt）

我们经常处于无法完全按照我们的技术愿景执行的情况。通常，我们需要做出选择，削减一些角落以推出一些紧急功能。这只是我们将发现自己不得不做出的又一个权衡。我们的技术愿景存在是有原因的。如果我们偏离这个原因，可能短期有益但长期成本。帮助我们理解这种权衡的概念是技术债务（technical debt）。当我们积累技术债务时，就像现实世界的债务一样，它有持续的成本，是我们想要偿还的东西。有时技术债务不仅仅是我们通过走捷径造成的。如果我们对系统的愿景改变了，但不是我们所有的系统都匹配怎么办？在这种情况下，我们也创造了新的技术债务来源。架构师的工作是关注更大的画面并理解这种平衡。对债务水平以及在哪里参与有一些看法很重要。根据你的组织，你可能能够提供温和的指导，但让团队自己决定如何跟踪和偿还债务。对于其他组织，你可能需要更有结构，也许维护一个定期审查的债务日志。

## 异常处理（Exception Handling）

所以我们的原则和实践指导我们的系统应该如何构建。但当我们的系统偏离时会发生什么？有时我们做出的决定只是规则的例外。在这些情况下，可能值得在某个地方的日志中捕获这样的决定以供将来参考。如果发现足够的例外，最终可能值得改变适用的原则或实践以反映对世界的新理解。例如，我们可能有一个实践说我们将始终使用 MySQL 进行数据存储。但然后我们看到使用 Cassandra 进行高度可扩展存储的令人信服的理由，此时我们改变我们的实践说「对大多数存储需求使用 MySQL，除非你预期大量增长，在这种情况下使用 Cassandra。」不过，值得重申的是，每个组织都不同。我与一些公司合作过，其中开发团队有高度的信任和自主权，原则是轻量级的（对明显异常处理的需求大大减少，如果不是消除的话）。在开发者自由较少的更有结构的组织中，跟踪例外可能对于确保到位的规定正确反映人们面临的挑战至关重要。话虽如此，我是微服务的粉丝，作为优化团队自主权的一种方式，给他们尽可能多的自由来解决手头的问题。如果你在一个对开发者如何工作施加很多限制的组织中工作，那么微服务可能不适合你。

## 小结

总结本章，以下是我认为演进式架构师的核心职责：

**愿景（Vision）**
确保系统有一个清晰传达的技术愿景，将帮助它满足你的客户和组织的需求。

**同理心（Empathy）**
理解你的决策对客户和同事的影响。

**协作（Collaboration）**
与构建系统的人密切合作，确保愿景被理解并可以付诸实践。

**适应性（Adaptability）**
愿意随着我们学到更多而演变愿景和原则。

**治理（Governance）**
确保我们正在构建的与愿景匹配，并在需要时演变愿景。

**赋能（Empowerment）**
创造空间让人们做出自己的决定，同时提供指导和支持。
