# 第14章 用户界面

到目前为止，我们尚未真正触及用户界面（User Interface）的世界。你们中有些人可能只是向客户提供冰冷、生硬的 API，但我们许多人希望创建美观、实用的用户界面，让客户满意。毕竟，用户界面正是我们将所有这些微服务（microservices）整合成对客户有意义的地方。

我刚开始接触计算时，我们主要讨论的是运行在桌面上的大型、臃肿的客户端。我花了很多时间使用 Motif 和 Swing，试图让软件尽可能好用。这些系统通常只是用于创建和操作本地文件，但许多也有服务端组件。我在 Thoughtworks 的第一份工作就是创建一个基于 Swing 的电子销售点系统，它只是众多组件之一，其中大部分在服务器端。

然后是 Web。我们开始将 UI 视为「瘦」客户端，更多逻辑在服务端。起初，服务端程序渲染整个页面并发送给客户端浏览器，浏览器几乎不做任何事。任何交互都由服务端通过用户点击链接或填写表单触发的 GET 和 POST 处理。随着时间的推移，JavaScript 成为为基于浏览器的 UI 添加动态行为的更流行选择，某些应用可以说和旧的桌面客户端一样「胖」。随后出现了移动应用，如今我们有了多样化的图形用户界面交付方式——不同平台，以及针对这些平台的不同技术。这些技术为我们提供了多种选择，让我们能够创建由微服务支撑的有效用户界面。本章将探讨这些内容。

## 走向数字化（Toward Digital）

过去几年，组织开始不再将 Web 或移动端区别对待，而是更整体地思考数字化。客户使用我们提供的服务的最佳方式是什么？这对我们的系统架构有什么影响？我们无法准确预测客户最终如何与产品交互，这一认识推动了更细粒度 API 的采用，例如微服务提供的那些。通过以不同方式组合微服务暴露的能力，我们可以为客户在桌面应用、移动设备、可穿戴设备上策划不同的体验，甚至在他们访问实体店时以物理形式呈现。

因此，将用户界面视为我们编织各种能力以提供给用户的场所。考虑到这一点，我们如何将这些能力整合在一起？我们需要从两个方面看待这个问题：谁（who）和如何（how）。首先，我们将考虑组织层面——在交付用户界面时，谁负责什么？其次，我们将看一组可用于实现这些界面的模式。

## 所有权模型（Ownership Models）

正如我们在第 1 章讨论过的，传统的分层架构（layered architecture）在有效交付软件时可能带来问题。在图 14-1 中，我们看到一个例子：用户界面层的责任由单一前端团队拥有，后端服务工作由另一个团队完成。在这个例子中，添加一个简单的控件需要三个不同团队参与。这类分层组织结构会因需要不断协调变更和团队间交接工作而显著影响交付速度。

```
+------------------+     +------------------+     +------------------+
|   Frontend Team  | --> |  Backend Team    | --> |  Database Team   |
|   (UI Layer)     |     |  (Services)      |     |  (Data)          |
+------------------+     +------------------+     +------------------+
        添加一个控件需要三个团队协调
```

**图 14-1** 跨三层做变更更复杂

我更偏好且认为更符合独立可部署性（independent deployability）目标的模型是：将 UI 拆分并由同时管理服务端组件的团队管理，如图 14-2 所示。在这里，单一团队最终负责添加新控件所需的所有变更。

拥有端到端功能完整所有权的团队能够更快地做出变更。完整所有权鼓励每个团队与软件最终用户建立直接联系。对于后端团队来说，很容易失去对最终用户是谁的追踪。

尽管有这些缺点，我（遗憾地）仍然看到专用前端团队是使用微服务的公司中更常见的组织模式。为什么会这样？

## 专用前端团队的驱动力（Drivers for Dedicated Frontend Teams）

对专用前端团队的需求似乎归结为三个关键因素：专家稀缺、追求一致性、技术挑战。

首先，交付用户界面需要一定程度的专业技能。有交互和图形设计方面，还有交付出色 Web 或原生应用体验所需的技术知识。具备这些技能的专家可能很难找到，由于这些人如此稀缺，诱惑是把他们全部集中在一起，以确保他们只专注于自己的专业领域。

第二个驱动力是一致性。如果有一个团队负责交付面向客户的用户界面，你可以确保 UI 具有一致的外观和感觉。你使用一致的控件集来解决类似问题，使用户界面看起来像一个单一、 cohesive 的整体。

第三，某些用户界面技术以非单体方式工作可能具有挑战性。这里我特别想到单页应用（Single-Page Applications, SPAs），至少从历史上看，它们不容易拆分。传统上，Web 用户界面由多个网页组成，你从一个页面导航到另一个。而 SPAs 中，整个应用在单个网页内提供。Angular、React 和 Vue 等框架理论上允许创建比「老式」网站更复杂的用户界面。我们将在本章后面看一组模式，为你提供如何分解用户界面的不同选择，就 SPAs 而言，我将展示微前端（micro frontends）概念如何让你在使用 SPA 框架的同时仍避免单体用户界面。

## 走向流对齐团队（Toward Stream-Aligned Teams）

我认为，如果你试图优化高吞吐量，拥有专用前端团队通常是一个错误——它会在组织中创造新的交接点，拖慢进度。理想情况下，我们的团队围绕端到端功能切片对齐，让每个团队能够向客户交付新功能，同时减少所需的协调量。我偏好的模型是团队拥有域中特定部分的端到端功能交付。这与 Matthew Skelton 和 Manuel Pais 在《Team Topologies》中描述的流对齐团队（stream-aligned teams）一致。正如他们所描述的：

> 流对齐团队是对齐于单一、有价值工作流的团队……团队被授权尽可能快速、安全、独立地构建和交付客户或用户价值，无需将工作交接给其他团队来完成部分工作。

从某种意义上说，我们讨论的是全栈团队（full stack teams）而非全栈开发者。对用户面功能交付负有端到端责任的团队，也将与最终用户建立更明显、更直接的联系。我经常看到「后端」团队对软件做什么或用户需要什么有模糊的概念，这在实现新功能时可能导致各种误解。另一方面，端到端团队会发现更容易与使用他们创建的软件的人建立直接联系——他们可以更专注于确保所服务的人得到他们需要的东西。

举一个具体例子，我曾与 FinanceCo 合作过一段时间，这是一家总部位于欧洲的成功且不断增长的金融科技公司。在 FinanceCo，几乎所有团队都从事直接影响客户体验的软件，并拥有面向客户的关键绩效指标（Key Performance Indicators, KPIs）——给定团队的成功较少取决于它交付了多少功能，而更多取决于帮助改善使用软件的人的体验。变更如何影响客户变得非常清晰。这只有在大多数团队对其交付的软件具有直接的、面向客户的职责时才有可能。当你离最终用户更远时，就更难理解你的贡献是否成功，你最终可能专注于与使用你软件的人关心的事情相去甚远的目标。

让我们重新审视专用前端团队存在的原因——专家、一致性、技术挑战——现在看看如何解决这些问题。

## 共享专家（Sharing Specialists）

优秀的开发者可能很难找到，当你需要具有某种专业技能的开发者时，情况就更复杂了。在用户界面领域，例如，如果你同时提供原生移动端和 Web 界面，你可能需要既有 iOS 和 Android 经验又有现代 Web 开发经验的人。这还不包括你可能需要的专用交互设计师、平面设计师、无障碍专家等。在这些更「窄」领域具有足够深度技能的人可能很少，他们可能总是有比时间更多的工作要做。

正如我们之前提到的，传统的组织结构方法会让你把所有拥有相同技能的人放在同一个团队，让你可以严格控制他们做什么。但正如我们也讨论过的，这会导致孤岛组织。

将具有专业技能的人放在自己的专用团队中，也剥夺了其他开发者学习这些抢手技能的机会。你不需要每个开发者都成为 iOS 专家，但让一些开发者在该领域学习足够的技能来帮助处理简单的事情，让你的专家自由处理真正困难的任务，仍然是有用的。技能共享也可以通过建立实践社区（communities of practice）来帮助——你可以考虑有一个跨团队的 UI 社区，让人们与同行分享想法和挑战。

我记得以前所有数据库变更都必须由中央数据库管理员（Database Administrators, DBAs）池完成。结果开发者对数据库如何工作几乎没有意识，更频繁地创建使用数据库不当的软件。此外，经验丰富的 DBA 被要求做的大部分工作由琐碎的变更组成。随着更多数据库工作被拉入交付团队，开发者总体上对数据库有了更好的理解，可以开始自己做琐碎的工作，让宝贵的 DBA 专注于更复杂的数据库问题，更好地利用他们更深的技能和经验。类似的转变也发生在运维和测试领域，更多这类工作被拉入团队。

所以，相反，将专家移出专用团队不会抑制专家做好工作的能力；事实上，这可能会增加他们专注于真正需要他们关注的困难问题的带宽。诀窍是找到更有效的方式部署你的专家。理想情况下，他们会被嵌入团队。但有时，可能没有足够的工作来证明他们在给定团队的全职存在，在这种情况下，他们可能会在多个团队之间分配时间。另一种模型是拥有一个具有这些技能的专用团队，其明确职责是赋能其他团队。在《Team Topologies》中，Skelton 和 Pais 将这些团队描述为赋能团队（enabling teams）。他们的工作是走出去帮助专注于交付新功能的团队完成工作。你可以把这些团队想象成有点像内部咨询——他们可以进来与流对齐团队度过有针对性的时间，帮助其在特定领域变得更自给自足，或提供一些专用时间来帮助推出特别困难的工作。

所以无论你的专家是全职嵌入给定团队，还是致力于赋能他人做同样的工作，你都可以消除组织孤岛，同时帮助提升同事的技能。

## 确保一致性（Ensuring Consistency）

经常被引用作为专用前端团队理由的第二个问题是一致性。通过让单一团队负责用户界面，你确保 UI 具有一致的外观和感觉。这可以从简单的事情延伸到以相同方式解决相同的界面问题——使用一致的设计和交互语言，帮助用户与系统交互。这种一致性不仅有助于传达产品本身的某种精致感，还确保用户在交付新功能时更容易使用。

不过，有方法可以帮助确保跨团队的一致性。如果你使用赋能团队模型，专家在多个团队中度过时间，他们可以帮助确保每个团队完成的工作是一致的。创建共享资源，如活的 CSS 风格指南或共享 UI 组件，也可以帮助。

作为赋能团队用于帮助一致性的具体例子，英国《金融时报》的 Origami 团队与设计团队合作构建 Web 组件，封装品牌身份——确保跨流对齐团队的一致外观和感觉。这类赋能团队提供两种形式的帮助——首先，它分享其交付已构建组件的专业知识；其次，它帮助确保 UI 提供一致的用户体验。

然而，值得注意的是，对一致性的驱动力不应被视为普遍正确。一些组织似乎有意识地决定不要求其用户界面的一致性，因为他们认为允许团队更多自主权更可取。亚马逊就是这样一家组织。其主购物网站的早期版本有大量不一致，小部件使用相当不同的控件风格。当你看亚马逊云服务（Amazon Web Services, AWS）的 Web 控制面板时，这种表现更加明显。AWS 中的不同产品有 massively 不同的交互模型，使用户界面相当令人困惑。然而，这似乎是亚马逊减少团队间内部协调的合乎逻辑的延伸。AWS 中产品团队的自主权增加似乎也以其他方式体现，不仅仅是在往往脱节的用户体验方面。经常有多种不同方式来完成同一任务（例如运行容器工作负载），AWS 内部的不同产品团队经常在相似但不兼容的解决方案上重叠。你可能会批评最终结果，但 AWS 已经表明，通过拥有这些高度自主的、面向产品的团队，它创建了一家在市场上具有明显领先地位的公司。至少就 AWS 而言，交付速度胜过用户体验的一致性。

## 克服技术挑战（Working Through Technical Challenges）

在用户界面开发方面，我们经历了一些有趣的演变——从基于绿色屏幕终端的文本用户界面到丰富的桌面应用、Web，再到原生移动体验。在许多方面，我们兜了一圈又多了——我们的客户端应用现在以如此复杂和精密的方式构建，它们绝对可以与 21 世纪头十年用户界面开发主流的丰富桌面应用相媲美。

在某种程度上，变化越多，保持不变的就越多。我们经常仍在使用与 20 年前相同的 UI 控件——按钮、复选框、表单、组合框等。我们在这个领域增加了一些组件，但远比你想象的少。改变的是我们首先用来创建这些图形用户界面的技术。

这一领域的一些较新技术，特别是单页应用，在分解用户界面时给我们带来问题。此外，我们期望同一用户界面在更广泛设备上交付，这带来了需要解决的其他问题。

从根本上说，我们的用户希望以尽可能无缝的方式与我们的软件互动。无论是通过桌面上的浏览器，还是通过原生或 Web 移动应用，结果都是一样的——用户通过单一界面与我们的软件交互。他们不应该关心用户界面是以模块化还是单体方式构建的。因此，我们必须寻找方法来分解我们的用户界面功能并将其重新整合，同时解决单页应用、移动设备等带来的挑战。这些问题将在本章剩余部分占据我们。

## 模式：单体前端（Pattern: Monolithic Frontend）

单体前端模式描述了一种架构，其中所有 UI 状态和行为都在 UI 本身中定义，对支撑微服务进行调用以获取所需数据或执行所需操作。图 14-3 显示了这样的例子。我们的屏幕想要显示关于专辑及其曲目列表的信息，因此 UI 发出请求从 Album 微服务拉取这些数据。我们还通过向 Promotions 微服务请求信息来显示最近的特价信息。在这个例子中，我们的微服务返回 JSON，UI 用它来更新显示的信息。

```
+------------------+     +------------------+
|   Album Screen   | --> |  Album Service   |
|                  | --> |  Promotions Svc  |
+------------------+     +------------------+
```

**图 14-3** 专辑详情屏幕从下游微服务拉取信息以渲染 UI

这种模型是构建单体单页应用的人最常用的，通常有专用前端团队。我们对微服务的要求相当简单——它们只需要以 UI 易于解释的形式共享信息。对于基于 Web 的 UI，这意味着我们的微服务可能需要以文本格式提供数据，JSON 是最可能的选择。UI 然后需要创建组成界面的各种组件，处理与后端的状态同步等。对服务间通信使用二进制协议对基于 Web 的客户端会更困难，但对原生移动设备或「厚」桌面应用可能没问题。

### 何时使用

这种方法有一些缺点。首先，由于其单体实体的性质，它可能成为专用前端团队的驱动力（或由其引起）。让多个团队共享这个单体前端的责任可能具有挑战性，因为有多个争用来源。其次，我们几乎没有能力为不同类型的设备定制响应。如果使用 Web 技术，我们可以改变屏幕布局以适应不同的设备约束，但这不一定延伸到改变对支撑微服务的调用。我的移动客户端可能只能显示订单的 10 个字段，但如果微服务拉回订单的全部一百个字段，我们最终会检索不必要的数据。这种方法的一个解决方案是让用户界面在发出请求时指定要拉取哪些字段，但这假设每个支撑微服务都支持这种形式的交互。在「GraphQL」一节中，我们将看看后端为前端（Backend for Frontend, BFF）模式和 GraphQL 的使用如何在这种情况下提供帮助。

实际上，当你想让 UI 的所有实现和行为都在一个可部署单元中时，这种模式效果最好。对于同时开发前端和所有支撑微服务的单一团队，这可能没问题。就个人而言，如果你有多个团队在开发你的软件，我认为你应该抵制这种冲动，因为它可能导致你滑入分层架构及相关的组织孤岛。但是，如果你无法避免分层架构和匹配的组织结构，这可能是你最终会使用的模式。

## 模式：微前端（Pattern: Micro Frontends）

微前端方法是一种组织模式，在这种模式下，前端的不同部分可以独立开发和部署。引用 Cam Jackson 关于该主题的一篇强烈推荐文章，我们可以将微前端定义为：「一种架构风格，其中独立可交付的前端应用被组合成更大的整体。」

对于希望拥有后端微服务和支撑 UI 交付的流对齐团队来说，这成为一个基本模式。微服务为后端功能提供独立可部署性，微前端为前端提供独立可部署性。

微前端概念由于单体、JavaScript 密集的 Web UI（以单页应用为典型）带来的挑战而流行起来。使用微前端，不同团队可以独立于其他团队对前端的不同部分进行开发和更改。回到图 14-2，库存团队、购买流程团队和客户档案团队各自能够独立于其他团队更改与其工作流相关的前端功能。

### 实现

对于基于 Web 的前端，我们可以考虑两种关键的分解技术来帮助实现微前端模式。基于小部件（widget-based）的分解涉及将前端的不同部分拼接成单个屏幕。另一方面，基于页面（page-based）的分解将前端拆分为独立的网页。两种方法都值得进一步探索，我们马上会讲到。

### 何时使用

如果你想采用端到端、流对齐的团队，并试图摆脱分层架构，微前端模式是必不可少的。我也可以想象在你想保留分层架构的情况下它会有用，但前端的现在功能如此之大，需要多个专用前端团队。

这种方法有一个关键问题，我不确定能否解决。有时微服务提供的能力不能整齐地放入小部件或页面。当然，我可能想在我们的网站上的一页上的一个框中展示推荐，但如果我想在其他地方编织动态推荐呢？例如，当我搜索时，我希望输入提示自动触发新的推荐。交互形式越横切，这种模型就越不适合，我们越有可能退回到只是进行 API 调用。

## 自包含系统（Self-Contained Systems）

自包含系统（Self-Contained System, SCS）是一种架构风格，可以说源于微服务早期对 UI 关注的缺乏。一个 SCS 可以由多个组件（可能是微服务）组成，当放在一起时构成单个 SCS。

根据定义，自包含系统必须符合一些特定标准，我们可以看到与微服务试图实现的许多相同目标重叠。你可以在非常清晰的 SCS 网站上找到更多关于自包含系统的信息，以下是一些要点：

- 每个 SCS 是一个自主的 Web 应用，没有共享 UI
- 每个 SCS 由一个团队拥有
- 应尽可能使用异步通信
- SCS 之间不能共享业务代码

SCS 方法没有像微服务那样普及，尽管我同意它概述的许多原则，但这不是我经常遇到的概念。我特别喜欢指出自包含系统应该由一个团队拥有。我确实想知道，这种缺乏更广泛使用是否解释了为什么 SCS 方法的某些方面似乎过于狭隘和规定性。例如，坚持每个 SCS 是「自主的 Web 应用」意味着许多类型的用户界面永远不能被考虑为 SCS。这是否意味着我构建的使用 gRPC 的原生 iOS 应用可以是 SCS 的一部分？

那么 SCS 方法与微服务冲突吗？不完全是。我曾在许多微服务上工作，当单独考虑时，它们本身就符合 SCS 的定义。SCS 方法中有一些我同意的有趣想法，其中许多我们已经在本书中介绍过。我只是觉得这种方法过于规定性，以至于对 SCS 感兴趣的人可能会发现采用这种方法极具挑战性，因为它可能需要对软件交付的许多方面进行全面更改。

我确实担心像 SCS 概念这样的宣言会引导我们走上过于关注活动而非原则和结果的道路。你可以遵循每一个 SCS 特征，仍然可能错过重点。反思后，我觉得 SCS 方法是一种以技术为中心的方法来推广组织概念。因此，我宁愿关注流对齐团队和减少协调的重要性，让技术和架构由此流动。

## 模式：基于页面的分解（Pattern: Page-Based Decomposition）

在基于页面的分解中，我们的 UI 被分解为多个网页。不同的页面集可以由不同的微服务提供。在图 14-4 中，我们看到 MusicCorp 的这种模式的例子。对 `/albums/` 中页面的请求直接路由到 Albums 微服务，由它处理提供这些页面，我们对 `/artists/` 做类似的事情。使用通用导航来帮助将这些页面缝合在一起。这些微服务反过来可能会获取构建这些页面所需的信息——例如，从 Inventory 微服务获取库存水平以在 UI 上显示哪些商品有货。

```
/albums/*  --> Albums Microservice
/artists/* --> Artists Microservice
    通用导航
```

**图 14-4** 用户界面由多个页面组成，不同页面组由不同微服务提供

使用这种模型，拥有 Albums 微服务的团队将能够端到端渲染完整的 UI，使团队更容易理解其变更如何影响用户。

### Web

在单页应用之前，我们有 Web。我们与 Web 的交互基于访问 URL 和点击导致新页面加载到我们浏览器的链接。我们的浏览器被构建为允许在这些页面周围导航，带有书签来标记感兴趣的页面以及后退和前进控件来重新访问以前访问的页面。你们可能都在翻白眼想，「我当然知道 Web 是如何工作的！」；然而，这是一种似乎已经失宠的用户界面风格。当我看到我们当前基于 Web 的用户界面实现时，它的简单性是我怀念的——我们通过自动假设基于 Web 的 UI 意味着单页应用而失去了很多。

就处理不同类型的客户端而言，没有什么能阻止页面根据请求页面的设备性质调整其显示的内容。渐进增强（progressive enhancement）或优雅降级（graceful degradation）的概念现在应该被很好地理解。

从技术实现角度来看，基于页面分解的简单性在这里是真正的吸引力。你不需要在浏览器中运行任何花哨的 JavaScript，也不需要 use 有问题的 iFrame。用户点击链接，就请求新页面。

### 适用场景

对于单体前端或微前端方法都很有用，如果我的用户界面是网站，基于页面的分解将是我对用户界面分解的默认选择。网页作为分解单元是 Web 整体的核心概念，它成为分解大型基于 Web 的用户界面的简单而明显的技术。

我认为问题在于，在急于使用单页应用技术的过程中，这些用户界面变得越来越罕见，以至于在我看来更适合网站实现的用户体验最终被硬塞进单页应用。你当然可以将基于页面的分解与我们介绍的其他模式结合使用。例如，我可以有一个包含小部件的页面——我们接下来会看这个。

## 模式：基于小部件的分解（Pattern: Widget-Based Decomposition）

使用基于小部件的分解，图形界面中的屏幕包含可以独立更改的小部件。在图 14-5 中，我们看到 MusicCorp 前端的例子，有两个小部件为购物篮和推荐提供 UI 功能。

```
+------------------+  +------------------+
|  Basket Widget   |  | Recommendations  |
|                  |  |    Widget        |
+------------------+  +------------------+
```

**图 14-5** MusicCorp 上使用的购物篮和推荐小部件

MusicCorp 的推荐小部件拉回可以循环和过滤的推荐轮播。如图 14-6 所示，当用户与推荐小部件交互时——例如循环到下一组推荐，或将商品添加到愿望清单——可能会导致对支撑微服务的调用，在这种情况下可能是 Recommendations 和 Wishlist 微服务。这可能与同时拥有这些支撑微服务和小部件本身的团队很好地对齐。

```
Recommendation Widget --> Recommendations Microservice
                       --> Wishlist Microservice
```

**图 14-6** 推荐微前端与支撑微服务之间的交互

一般来说，你需要一个「容器」应用来定义界面的核心导航以及需要包含哪些小部件。如果我们从端到端流导向团队的角度思考，我们可以想象单一团队提供推荐小部件并负责支撑的 Recommendations 微服务。

这种模式在现实世界中很常见。例如，Spotify 用户界面大量使用这种模式。一个小部件可能持有播放列表，另一个可能持有关于艺术家的信息，第三个可能持有关于你关注的艺术家和其他 Spotify 用户的信息。这些小部件在不同情况下以不同方式组合。

你仍然需要某种组装层来将这些部分拉在一起。不过，这可能简单到使用服务端或客户端模板。

### 实现

如何将小部件拼接进你的 UI 很大程度上取决于你的 UI 是如何创建的。对于简单的网站，使用客户端或服务端模板将小部件作为 HTML 片段包含可以相当简单，尽管如果小部件有更复杂的行为，你可能会遇到问题。例如，如果我们的推荐小部件包含大量 JavaScript 功能，我们如何确保这与加载到网页其余部分的行为不会冲突？理想情况下，整个小部件可以以不会破坏 UI 其他方面的方式打包。

如何将自包含功能交付到 UI 而不破坏其他功能的问题，历史上对于单页应用尤其成问题，部分原因是模块化的概念似乎不是创建支撑 SPA 框架时的首要关注点。这些挑战值得更深入探索。

**依赖**

尽管 iFrame 在过去是一种 heavily 使用的技术，我们倾向于避免使用它们将不同小部件拼接成单个网页。iFrame 在尺寸方面有一系列挑战，并且在使前端不同部分之间难以通信方面也有挑战。相反，小部件通常要么使用服务端模板拼接进 UI，要么在客户端动态插入浏览器。在这两种情况下，挑战在于小部件与前端其他部分在同一浏览器页面中运行，这意味着你需要小心不同小部件彼此不冲突。

例如，我们的推荐小部件可能使用 React v16，而购物篮小部件仍在使用 React v15。这当然可能是一种祝福，因为它可以帮助我们尝试不同的技术（我们可以为不同小部件使用不同的 SPA 框架），但在更新正在使用的框架版本时也有帮助。我与许多团队交谈过，他们在 Angular 或 React 版本之间迁移时遇到挑战，主要是因为新框架版本中使用的约定差异。升级整个单体 UI 可能令人生畏，但如果你可以逐步进行，逐块更新前端的部分，你可以分解工作并减轻升级引入新问题的风险。

缺点是你最终可能在依赖项之间有很多重复，这反过来可能导致页面加载大小的大量膨胀。例如，我最终可能包含 React 框架及其相关传递依赖的多个不同版本。许多网站现在的页面加载大小是某些操作系统的几倍，这并不令人惊讶。作为一项快速非科学研究，我在写作时检查了 CNN 网站的页面加载，它是 7.9 MB，比 Alpine Linux 的 5 MB 大得多。7.9 MB 实际上是我看到的一些页面加载大小的小端。

**页面内小部件之间的通信**

尽管我们的小部件可以独立构建和部署，我们仍然希望它们能够相互交互。以 MusicCorp 为例，当用户在畅销榜中选择一张专辑时，我们希望 UI 的其他部分根据选择进行更新，如图 14-7 所示。

```
Charts Widget --[Album Selected event]--> Recommendation Widget
                  |                      --> Album Details Widget
```

**图 14-7** 图表小部件可以发出其他 UI 部分监听的事件

我们实现这一点的方式是让图表小部件发出自定义事件。浏览器已经支持许多我们可以用来触发行为的标准事件。这些事件允许我们对按钮被按下、鼠标滚动等做出反应，如果你花过任何时间构建 JavaScript 前端，你可能已经大量使用了这样的事件处理。创建你自己的自定义事件是一个简单的步骤。

所以在我们的情况下，当在图表中选择一个项目时，该小部件会发出 `Album Selected` 自定义事件。推荐和专辑详情小部件都订阅该事件并相应做出反应，推荐根据选择更新，专辑详情被加载。这种交互当然应该已经为我们所熟悉，因为它模仿了我们在「模式：事件驱动通信」中讨论的微服务之间的事件驱动交互。唯一的真正区别是这些事件交互发生在浏览器内部。

**Web 组件（Web Components）**

乍一看，Web 组件标准应该是实现这些小部件的明显方式。Web 组件标准描述了如何创建可以沙箱化其 HTML、CSS 和 JavaScript 方面的 UI 组件。不幸的是，Web 组件标准似乎花了很长时间才稳定下来，而浏览器正确支持的时间更长。围绕它们的许多初始工作似乎已经停滞，这显然影响了采用。例如，我还没有遇到过使用从微服务提供的 Web 组织的组织。

鉴于 Web 组件标准现在得到了相当好的支持，我们可能会看到它们作为实现沙箱化小部件或更大微前端的常见方式在未来出现，但在等待多年之后，我不抱太大希望。

### 何时使用

这种模式使多个流对齐团队能够轻松地为同一 UI 做出贡献。它比基于页面的分解提供更多灵活性，因为不同团队交付的小部件可以同时共存于 UI 中。它还创造了让赋能团队提供可被流对齐团队使用的可重用小部件的机会——我之前在提到英国《金融时报》Origami 团队的角色时分享了一个例子。

如果你正在构建丰富的基于 Web 的用户界面，小部件分解模式非常有用，我会强烈建议在任何使用 SPA 框架并希望分解前端职责、走向微前端方法的情况下使用小部件。围绕这个概念的技术和支持技术在过去几年中显著改善，以至于在构建基于 SPA 的 Web 界面时，将我的 UI 分解为微前端将是我的默认方法。

我在 SPAs 背景下对小部件分解的主要担忧与设置组件单独打包所需的工作以及有效负载大小的问题有关。前者可能是一次性成本，只是弄清楚什么风格的打包最适合你现有的工具链。后者更有问题。一个小部件依赖项中的简单小变更可能导致大量新依赖项被包含在应用中， drastically 膨胀页面大小。如果你正在构建页面重量是一个关注点的用户界面，我建议设置一些自动检查，在页面重量超过某个可接受的阈值时提醒你。

另一方面，如果小部件本质上更简单且主要是静态组件，使用像客户端或服务端模板这样简单的东西包含它们相比之下非常直接。

## 约束（Constraints）

在继续讨论下一个模式之前，我想谈谈约束这个话题。越来越多地，我们软件的用户从各种不同设备与之交互。每个设备都对我们的软件必须适应的约束施加了不同的限制。例如，对于桌面 Web 应用，我们考虑访客使用的浏览器或屏幕分辨率等约束。视力受损的人可能通过屏幕阅读器使用我们的软件，行动受限的人可能更可能使用键盘式输入来导航屏幕。

所以尽管我们的核心服务——我们的核心产品——可能相同，我们需要一种方法来适应每种界面类型存在的不同约束，以及我们用户的不同需求。如果你愿意，这可以纯粹从财务角度驱动——更多满意的客户意味着更多钱。但也有一种人性、伦理的考虑浮现：当我们忽视有特定需求的客户时，我们剥夺了他们使用我们服务的机会。在某些情况下，由于设计决策而使人们无法导航 UI 已导致法律诉讼和罚款——例如，英国与许多其他国家一样，正确地制定了立法以确保残疾人对网站的访问。

移动设备带来了一系列新的约束。我们的移动应用与服务器通信的方式可能会产生影响。这不仅仅是纯带宽问题，移动网络的限制可能起作用。不同类型的交互可能耗尽电池寿命，导致一些愤怒的客户。交互的性质也根据设备而变化。我在平板电脑上不容易右键点击。在手机上，我可能想为 mostly 单手使用设计我的界面，大多数操作由拇指控制。在其他地方，我可能允许人们在带宽紧张的地方通过 SMS 与服务交互——例如，SMS 作为界面在全球南方的使用量很大。

关于用户界面可访问性的更广泛讨论超出了本书的范围，但我们至少可以探索不同类型客户端（如移动设备）带来的具体挑战。处理客户端设备不同需求的常见解决方案是在客户端执行某种过滤和调用聚合。不需要的数据可以被剥离，不需要发送到设备，多个调用可以合并为单个调用。

接下来，我们将看两个在这种空间中有用的模式——中央聚合网关（central aggregating gateway）和后端为前端（Backend for Frontend, BFF）模式。我们还将看看 GraphQL 如何被用来帮助为不同类型的界面定制响应。

## 模式：中央聚合网关（Pattern: Central Aggregating Gateway）

中央用途的聚合网关位于外部用户界面和下游微服务之间，为所有用户界面执行调用过滤和聚合。没有聚合，用户界面可能需要进行多次调用来获取所需信息，经常丢弃检索到但不需要的数据。

在图 14-8 中，我们看到这种情况。我们想显示一个包含客户最近订单信息的屏幕。屏幕需要显示一些关于客户的一般信息，然后按日期顺序列出他们的许多订单，以及摘要信息，显示每个订单的日期、状态和价格。

```
UI --> Customer Microservice (拉取完整客户信息)
UI --> Order Microservice (拉取订单详情)
```

**图 14-8** 为单个屏幕进行多次调用以获取信息

我们直接调用 Customer 微服务，拉取关于客户的完整信息，尽管我们只需要几个字段。然后我们从 Order 微服务获取订单详情。我们可以稍微改善这种情况，也许通过改变 Customer 或 Order 微服务以返回更符合我们在此特定情况下要求的数据，但这仍然需要两次调用。

使用聚合网关，我们可以改为从用户界面向网关发出一次调用。聚合网关然后执行所有需要的调用，将结果合并为单个响应，并丢弃用户界面不需要的任何数据（图 14-9）。

```
UI --> Gateway --> Customer Microservice
              --> Order Microservice
     (合并、过滤、返回)
```

**图 14-9** 服务端中央网关处理对下游微服务的调用过滤和聚合

这样的网关也可以帮助批量调用。例如，与其需要通过单独调用查找 10 个订单 ID，我可以向聚合网关发送一个批量请求，它可以处理其余部分。

从根本上说，拥有某种聚合网关可以减少外部客户端需要进行的调用次数，并减少需要发送回的数据量。这可以在减少带宽使用和改善应用延迟方面带来显著好处。

**所有权**

随着更多用户界面使用中央网关，随着更多微服务需要为这些用户界面进行调用聚合和过滤逻辑，网关成为潜在的争用来源。谁拥有网关？是由创建用户界面的人拥有，还是由拥有微服务的人拥有？我经常发现中央聚合网关做得太多，最终由专用团队拥有。你好，孤岛分层架构！

从根本上说，调用聚合和过滤的性质在很大程度上由外部用户界面的需求驱动。因此，网关由创建 UI 的团队拥有是合理的。不幸的是，特别是在拥有专用前端团队的组织中，该团队可能没有构建如此重要的后端组件的技能。

无论谁最终拥有中央网关，它都有可能成为交付的瓶颈。如果多个团队需要对网关进行更改，其开发将需要这些团队之间的协调，拖慢进度。如果一个团队拥有它，该团队在交付方面可能成为瓶颈。我们很快会看到后端为前端模式如何帮助解决这些问题。

**不同类型的用户界面**

如果所有权方面的挑战可以得到管理，中央聚合网关可能仍然运作良好，直到我们考虑不同设备及其不同需求的问题。正如我们讨论过的，移动设备的 affordance 非常不同。我们的屏幕空间更少，这意味着我们可以显示更少的数据。打开许多服务器端资源的连接可能耗尽电池寿命和有限的数据计划。此外，我们想在移动设备上提供的交互性质可能大不相同。想想典型的实体零售商。在桌面应用上，我可能允许你查看待售商品并在线订购或在商店预订。但在移动设备上，我可能想允许你扫描条形码进行价格比较，或在店内给你基于位置的优惠。随着我们构建越来越多的移动应用，我们意识到人们以非常不同的方式使用它们，因此我们需要暴露的功能也会不同。

所以在实践中，我们的移动设备会想要进行不同且更少的调用，并会想要显示与其桌面对应物不同（可能更少）的数据。这意味着我们需要向 API 后端添加额外功能以支持不同类型的用户界面。在图 14-10 中，我们看到 MusicCorp 的 Web 界面和移动界面都使用同一网关获取客户摘要屏幕，但每个客户端想要不同的信息集。Web 界面想要更多关于客户的信息，还有每个订单中商品的简要摘要。这导致我们在后端网关中实现两条不同的聚合和过滤调用。

```
Web UI  --> Gateway (聚合 A)
Mobile  --> Gateway (聚合 B)
```

**图 14-10** 为不同设备支持不同的聚合调用

这可能导致网关中的大量膨胀，特别是如果我们考虑不同的原生移动应用、面向客户的网站、内部管理界面等。我们还有问题，当然，虽然这些不同的 UI 可能由不同的团队拥有，但网关是单一单元——我们有多团队必须在同一部署单元上工作的老问题。我们的单一聚合后端可能成为瓶颈，因为如此多的变更试图对同一可部署工件进行。

**多个关注点**

在处理 API 调用时，服务端可能有一系列需要解决的关注点。除了调用聚合和过滤之外，我们可以考虑更通用的关注点，如 API 密钥管理、用户认证或调用路由。通常这些通用关注点可以由 API 网关产品处理，这些产品有多种大小和价格点（其中一些高得令人咋舌！）。根据你需要的复杂程度，购买产品（或许可服务）来处理其中一些关注点可能很有意义。你真的想自己管理 API 密钥签发、跟踪、速率限制等吗？尽可以看看这个领域的产品来解决这些通用关注点，但要小心不要也试图用这些产品来做你的调用聚合和过滤，即使它们声称可以。

当定制别人构建的产品时，你经常不得不在他们的世界中工作。你的工具链受到限制，因为你可能无法使用你的编程语言和开发实践。与其编写 Java 代码，你正在用某种奇怪的产品特定 DSL（可能使用 JSON）配置路由规则。这可能是一种令人沮丧的体验，你正在将系统的一些智能烘焙到第三方产品中。这可能会降低你以后移动此行为的能力。常见的是意识到调用聚合的模式实际上与某些领域功能相关，可能证明微服务本身是合理的（我们稍后在讨论 BFF 时会探索更多）。如果这种行为在供应商特定的配置中，移动此功能可能更有问题，因为你可能必须重新发明它。

如果聚合网关变得足够复杂以至于需要专用团队来拥有和管理，情况可能会变得更糟。在最坏的情况下，采用更多横向团队所有权可能导致一种情况：要推出一些新功能，你必须让前端团队进行更改，聚合网关团队进行更改，拥有微服务的团队也要进行更改。突然一切都开始变得更慢了。

所以如果你想使用专用 API 网关，请继续——但强烈考虑将你的过滤和聚合逻辑放在其他地方。

### 何时使用

对于由单一团队拥有的解决方案，其中一个团队开发用户界面和后端微服务，我会对拥有单一中央聚合网关感到满意。也就是说，这个团队听起来做了很多工作——在这种情况下，我倾向于看到用户界面之间有很大的一致性，这通常首先消除了对这些聚合点的需求。

如果你确实决定采用单一中央聚合网关，请小心限制你在其中放入的功能。例如，我会非常警惕将这种功能推入更通用的 API 网关产品，出于之前概述的原因。

不过，在优化用户对我们用户界面的体验方面，在服务端进行某种形式的调用过滤和聚合的概念可能非常重要。问题是在拥有多个团队的交付组织中，中央网关可能导致这些团队之间需要大量协调。

所以如果我们仍然想在服务端进行聚合和过滤，但想消除与中央网关所有权模型相关的问题，我们能做什么？这就是后端为前端模式发挥作用的地方。

## 模式：后端为前端（Pattern: Backend for Frontend, BFF）

BFF 与中央聚合网关之间的主要区别在于 BFF 本质上是单一用途的——它是为特定用户界面开发的。这种模式在帮助处理用户界面的不同关注点方面已被证明非常成功，我在包括 SoundCloud 和 REA 在内的多个组织中看到它运作良好。如图 14-11 所示，我们重新审视 MusicCorp，Web 和移动购物界面现在各自有自己的聚合后端。

```
Web UI  --> Web BFF  --> Microservices
Mobile  --> Mobile BFF --> Microservices
```

**图 14-11** 每个用户界面都有自己的 BFF

由于其特定性质，BFF 避开了一些中央聚合网关的担忧。由于我们不是试图为所有人做所有事，BFF 避免了成为开发的瓶颈，多个团队都试图共享所有权。我们也不太担心与用户界面的耦合，因为耦合是更可接受的。给定的 BFF 是用于特定用户界面的——假设它们由同一团队拥有，那么固有的耦合就更容易管理。

我经常将 BFF 与用户界面的使用描述为就像 UI 实际上被分成两部分。一部分坐在客户端设备上（Web 界面或原生移动应用），第二部分 BFF 嵌入在服务端。BFF 与特定用户体验紧密耦合，通常由与用户界面相同的团队维护，从而更容易根据 UI 需要定义和调整 API，同时简化客户端和服务端组件发布的协调过程。

**需要多少个 BFF？**

在在不同平台上交付相同（或相似）用户体验时，我见过两种不同的方法。我偏好的模型（也是我最常看到的模型）是严格为每种不同类型的客户端提供单一 BFF——这是我在 REA 看到的模型，如图 14-12 所示。Android 和 iOS 应用虽然覆盖相似的功能，但各自有自己的 BFF。

```
iOS App  --> iOS BFF
Android  --> Android BFF
```

**图 14-12** REA 的 iOS 和 Android 应用有不同的 BFF

一种变体是寻找机会让同一类型的用户界面的多种客户端使用同一 BFF。SoundCloud 的 Listener 应用允许人们在 Android 或 iOS 设备上收听内容。SoundCloud 对 Android 和 iOS 使用单一 BFF，如图 14-13 所示。

```
iOS App  --> Mobile BFF
Android  --> Mobile BFF
```

**图 14-13** SoundCloud 在 iOS 和 Android 应用之间共享 BFF

我对这种第二种模型的主要担忧是，你让使用单一 BFF 的客户端类型越多，BFF 越有可能因处理多个关注点而变得臃肿。但这里要理解的关键是，即使客户端共享 BFF，也是为了同一类用户界面——所以虽然 SoundCloud 的 iOS 和 Android 原生 Listener 应用使用同一 BFF，其他原生应用会使用不同的 BFF。如果同一团队同时拥有 Android 和 iOS 应用并拥有 BFF，我对使用这种模型更放松——如果这些应用由不同团队维护，我更倾向于推荐更严格的模型。所以你可以将你的组织结构视为决定哪种模型最有意义的主要驱动力之一（康威定律再次获胜）。

REA 的 Stewart Gleadow 建议的准则是「一种体验，一个 BFF」。所以如果 iOS 和 Android 体验非常相似，那么证明拥有单一 BFF 更容易。然而，如果它们差异很大，那么拥有不同的 BFF 更有意义。就 REA 而言，虽然两种体验之间有重叠，但不同团队拥有它们并以不同方式推出类似功能。有时同一功能可能在不同移动设备上以不同方式部署——Android 应用的原生体验可能需要在 iOS 上重新设计才能感觉原生。

REA 故事的另一个教训（我们已经多次覆盖）是，软件往往在围绕团队边界对齐时效果最好，BFF 也不例外。SoundCloud 拥有单一移动团队使得拥有单一 BFF 乍一看似乎合理，REA 为两个独立团队拥有两个不同的 BFF 也是如此。值得注意的是，与我交谈的 SoundCloud 工程师建议，为 Android 和 iOS listener 应用拥有一个 BFF 是他们可能会重新考虑的事情——他们有一个移动团队，但实际上他们是 Android 和 iOS 专家的混合，他们发现自己 mostly 在其中一个或另一个应用上工作，这意味着他们实际上是两个团队。

**复用与 BFF**

每个用户界面一个 BFF 的担忧之一是你最终可能在 BFF 之间有很多重复。例如，它们可能最终执行相同类型的聚合，对与下游服务交互有相同或相似的代码等。如果你正在寻找提取通用功能，那么挑战之一往往是找到它。这种重复可能发生在 BFF 本身，但也可能最终被烘焙到不同的客户端中。由于这些客户端使用非常不同的技术栈，识别这种重复正在发生可能很困难。由于组织倾向于对服务端组件有共同的技术栈，拥有多个有重复的 BFF 可能更容易发现和提取。

有些人对这种反应是想要将 BFF 合并回一起，他们最终得到通用聚合网关。我对退回到单一聚合网关的担忧是，我们最终可能失去的比得到的更多，特别是因为可能有其他方法来解决这种重复。

正如我之前说过的，我对微服务之间的重复代码相当放松。也就是说，虽然在单个微服务边界内我通常会尽我所能将重复重构到合适的抽象中，但当面对微服务之间的重复时，我没有同样的反应。这主要是因为我还经常更担心提取共享代码可能导致服务之间紧密耦合的潜在（我们在「DRY 与微服务世界中代码复用的危险」中探讨过这个话题）。也就是说，当然有这种情况是合理的。

当确实需要提取通用代码以在 BFF 之间实现复用时，有两个明显的选择。第一个，通常更便宜但更 fraught，是提取某种共享库。这可能成问题的原因是共享库是耦合的主要来源，特别是当用于生成调用下游服务的客户端库时。尽管如此，有这种感觉正确的情况——特别是当被抽象的代码纯粹是服务内部的关注点时。

另一个选择是将共享功能提取到新的微服务中。如果被提取的功能代表业务领域功能，这可能效果很好。这种方法的一种变体可能是将聚合职责推给更下游的微服务。让我们考虑一种情况，我们想显示客户愿望清单中的商品列表，以及关于这些商品是否有货和当前价格的信息，如表 14-1 所示。

| 专辑 | 库存状态 | 价格 |
|------|----------|------|
| The Brakes, Give Blood | In Stock! | $5.99 |
| Blue Juice, Retrospectable | Out of Stock | $7.50 |
| Hot Chip, Why Make Sense? | Going Fast! (2 left) | $9.99 |

Customer 微服务存储关于愿望清单和每个商品 ID 的信息。Catalog 微服务存储每个商品的名称和价格，库存水平存储在我们的 Inventory 微服务中。要在 iOS 和 Android 应用上显示相同的控件，每个 BFF 都需要对支撑微服务进行相同的三次调用，如图 14-14 所示。

减少此处功能重复的一种方法是将这种通用行为提取到新的微服务中。在图 14-15 中，我们看到我们新的专用 Wishlist 微服务，我们的 Android 和 iOS 应用都可以使用。我不得不说，同样的代码在两个地方使用不会自动让我想以这种方式提取服务，但如果创建新服务的交易成本足够低，或者我在超过两个地方使用该代码，我肯定会考虑——在这种特定情况下，如果我们也在 Web 界面上显示愿望清单，例如，专用微服务会开始看起来更有吸引力。我认为在第三次实现某件事时创建抽象的古老格言在服务级别仍然是一个好的经验法则。

**桌面 Web 及更广的 BFF**

你可以认为 BFF 只在解决移动设备约束方面有用。桌面 Web 体验通常是在更强大的设备上交付的，具有更好的连接性，进行多次下游调用的成本是可管理的。这可以让你的 Web 应用直接对下游服务进行多次调用，无需 BFF。

不过，我见过在 Web 上也使用 BFF 可能有用的情况。当你在服务端生成更大比例的 Web UI 时（例如，使用服务端模板），BFF 是完成此操作的明显位置。这种方法还可以简化一些缓存，因为你可以在 BFF 前面放置反向代理，允许你缓存聚合调用的结果。

我至少见过一个组织为需要拨打电话的其他外部方使用 BFF。回到我 perennial 的例子 MusicCorp，我可能暴露一个 BFF 以允许第三方提取版税支付信息，或允许向一系列机顶盒设备流式传输，如图 14-16 所示。这些不再是真正的 BFF，因为外部方不是呈现「用户界面」，但这是同一模式在不同场景中的使用例子，所以我认为值得分享。

这种方法可能特别有效，因为第三方通常没有或很少有能力（或意愿）使用或更改他们进行的 API 调用。使用中央 API 后端，你可能不得不保留旧版本的 API 只是为了满足无法进行更改的一小部分外部方；使用 BFF，这个问题大大减少。它还限制了破坏性变更的影响。你可以以会破坏与其他方兼容的方式更改 Facebook 的 API，但由于他们使用不同的 BFF，他们不受此变更的影响。

### 何时使用

对于仅提供 Web UI 的应用，我怀疑 BFF 只有在服务端需要大量聚合时才有意义。否则，我认为我们已经介绍的其他 UI 组合技术可能同样有效，不需要额外的服务端组件。

一旦你需要为移动 UI 或第三方提供特定功能，我会强烈考虑从一开始就为每个客户端使用 BFF。如果部署额外服务的成本很高，我可能会重新考虑，但 BFF 带来的关注点分离在大多数情况下是一个相当令人信服的提议。如果构建 UI 和下游服务的人之间有显著分离，我会更倾向于使用 BFF，出于我概述的原因。

然后我们来到如何实现 BFF 的问题——让我们看看 GraphQL 及其可以发挥的作用。

## GraphQL

GraphQL 是一种查询语言，允许客户端发出查询以访问或变更数据。与 SQL 一样，GraphQL 允许这些查询动态更改，允许客户端精确定义它想要返回什么信息。例如，使用标准的 REST over HTTP 调用，当对 Order 资源发出 GET 请求时，你会得到该订单的所有字段。但如果在那种特定情况下，你只想要订单的总金额呢？你当然可以忽略其他字段，或者提供包含你所需信息的替代资源（也许是 Order Summary）。使用 GraphQL，你可以发出请求只询问你需要的字段，如示例 14-1 所示。

```graphql
{
    order(id: 123) {
        date
        total
        status
        delivery {
            company
            driver
            duedate
        }
    }
}
```

**示例 14-1** 用于获取订单信息的 GraphQL 查询示例

在这个查询中，我们请求了订单 123，我们请求了总价和订单状态。我们更进一步请求了关于此订单交付的信息，这样我们可以获取将交付我们包裹的司机姓名、他们工作的公司以及包裹预计到达时间。使用普通的 REST API，除非交付信息包含在 Order 资源中，我们可能需要进行额外的调用来获取此信息。所以 GraphQL 不仅帮助我们请求我们想要的字段，还可以减少往返。像这样的查询要求我们定义我们正在访问的各种数据类型——显式定义类型是 GraphQL 的关键部分。

要实现 GraphQL，我们需要一个解析器（resolver）来处理查询。GraphQL 解析器位于服务端，将 GraphQL 查询映射到实际获取信息的调用。所以在微服务架构的情况下，我们需要一个能够将 ID 为 123 的订单请求映射到对微服务的等效调用的解析器。

这样，我们可以使用 GraphQL 实现聚合网关，甚至 BFF。GraphQL 的好处是，我们可以简单地通过从客户端更改查询来轻松更改我们想要的聚合和过滤；只要 GraphQL 类型支持我们想要进行的查询，GraphQL 服务端不需要任何更改。如果我们不再想在示例查询中看到司机姓名，我们只需从查询本身中省略它，它就不会被发送。另一方面，如果我们想看到我们为此订单获得的积分数量，假设此信息在订单类型中可用，我们只需将其添加到查询中，信息就会返回。这与需要 BFF 本身也应用聚合逻辑变更的 BFF 实现相比是一个显著优势。

GraphQL 让客户端设备动态更改正在进行的查询而无需服务端更改的灵活性，意味着你的 GraphQL 服务器成为共享、争用资源的可能性较小，正如我们与通用聚合网关讨论的那样。也就是说，如果你需要暴露新类型或向现有类型添加字段，仍然需要服务端更改。因此，你可能仍然希望多个 GraphQL 服务器后端沿团队边界对齐——所以 GraphQL 成为实现 BFF 的一种方式。

我对 GraphQL 有担忧，我在第 5 章中详细概述了。也就是说，它是允许动态查询以适应不同类型用户界面需求的简洁解决方案。

## 混合方法（A Hybrid Approach）

上述许多选项不需要一刀切。我可以看到组织采用基于小部件的分解方法来创建网站，但在移动应用方面使用后端为前端方法。关键是我们需要保留我们提供给用户的基础能力的 cohesion。我们需要确保与订购音乐或更改客户详情相关的逻辑存在于处理这些操作的服务中，而不是散布在整个系统中。避免将太多行为放入任何中间层的陷阱是一个棘手的平衡行为。

## 小结

正如我希望展示的，功能分解不必止于服务端，拥有专用前端团队并非不可避免。我分享了多种构建用户界面的不同方式，这些方式可以利用支撑微服务，同时实现专注的端到端交付。
