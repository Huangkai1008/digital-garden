# 第9章 并发回顾

同时获得良好的并行性能、并发下的正确性和可理解的代码是内核设计中的一大挑战。直接使用锁是通往正确性的最佳路径，但并不总是可能的。本章重点介绍 xv6 被迫以复杂方式使用锁的例子，以及 xv6 使用类似锁的技术但不使用锁的例子。

## 9.1 锁定模式

缓存项通常是锁定的挑战。例如，文件系统的块缓存（`kernel/bio.c:26`）存储最多 NBUF 个磁盘块的副本。给定磁盘块在缓存中最多有一个副本至关重要；否则，不同的进程可能会对应该是同一块的不同副本进行冲突的更改。每个缓存的块存储在 `struct buf`（`kernel/buf.h:1`）中。`struct buf` 有一个 `lock` 字段，有助于确保一次只有一个进程使用给定的磁盘块。然而，该锁不够：如果块根本不在缓存中，而两个进程想同时使用它怎么办？没有 `struct buf`（因为块尚未缓存），因此没有可锁定的东西。xv6 通过将额外的锁（`bcache.lock`）与缓存块的身份集关联来处理这种情况。需要检查块是否被缓存（例如，`bget`（`kernel/bio.c:59`））或更改缓存块集的代码必须持有 `bcache.lock`；在该代码找到所需的块和 `struct buf` 后，它可以释放 `bcache.lock` 并仅锁定特定块。这是一种常见的模式：一个锁用于项集，加上每个项一个锁。

通常，获取锁的同一函数将释放它。但更精确的看待方式是，锁在必须看起来原子的序列开始时获取，在该序列结束时释放。如果序列在不同函数、不同线程或不同 CPU 中开始和结束，则锁获取和释放必须做同样的事情。锁的功能是强制其他使用等待，而不是将数据固定到特定代理。一个例子是 `yield`（`kernel/proc.c:503`）中的 `acquire`，它在调度器线程中释放而非在获取进程中。另一个例子是 `ilock`（`kernel/fs.c:293`）中的 `acquiresleep`；此代码在读取磁盘时经常睡眠；它可能在不同的 CPU 上醒来，这意味着锁可能在不同的 CPU 上获取和释放。

## 9.2 类似锁的模式

在许多地方，xv6 使用引用计数或标志以类似锁的方式指示对象已分配且不应被释放或重用。进程的 `p->state` 以这种方式行事，文件、inode 和 buf 结构中的引用计数也是如此。虽然在每种情况下锁保护标志或引用计数，但后者防止对象被过早释放。

文件系统使用 `struct inode` 引用计数作为一种可以由多个进程持有的共享锁，以避免如果代码使用普通锁会发生的死锁。例如，`namex`（`kernel/fs.c:652`）中的循环依次锁定路径名的每个组件命名的目录。然而，`namex` 必须在循环结束时释放每个锁，因为如果它持有多个锁，如果路径名包含点（例如，`a/./b`），它可能会与自己死锁。它也可能与涉及目录和 `..` 的并发查找死锁。如第 8 章所解释的，解决方案是让循环将目录 inode 带到下一次迭代，其引用计数增加但未锁定。

## 9.3 完全没有锁

xv6 有几个地方在没有锁的情况下共享可变数据。一个是在自旋锁的实现中，尽管人们可以将 RISC-V 原子指令视为依赖硬件实现的锁。另一个是 `main.c`（`kernel/main.c:7`）中的 `started` 变量，用于防止其他 CPU 在 CPU 零完成初始化 xv6 之前运行；`volatile` 确保编译器实际生成加载和存储指令。

xv6 包含一个 CPU 或线程写入某些数据而另一个 CPU 或线程读取数据但没有专门保护该数据的锁的情况。例如，在 `fork` 中，父进程写入子进程的用户内存页，子进程（不同的线程，可能在不同的 CPU 上）读取这些页；没有锁显式保护这些页。这严格来说不是锁定问题，因为子进程在父进程完成写入之前不会开始执行。这是一个潜在的内存排序问题（见第 6 章），因为如果没有内存屏障，没有理由期望一个 CPU 看到另一个 CPU 的写入。然而，由于父进程释放锁，子进程在启动时获取锁，`acquire` 和 `release` 中的内存屏障确保子进程的 CPU 看到父进程的写入。

## 9.4 并行性

锁定主要是关于为了正确性而抑制并行性。因为性能也很重要，内核设计者通常必须考虑如何以既实现正确性又允许并行性的方式使用锁。虽然 xv6 不是为高性能系统设计的，但仍然值得考虑哪些 xv6 操作可以并行执行，哪些可能在锁上冲突。

xv6 中的管道是相当好的并行性的例子。每个管道有自己的锁，因此不同的进程可以在不同的 CPU 上并行读写不同的管道。然而，对于给定的管道，写入者和读取者必须等待彼此释放锁；它们不能同时读写同一管道。从空管道读取（或向满管道写入）必须阻塞也是事实，但这不是由于锁定方案。

上下文切换是更复杂的例子。两个内核线程，每个在自己的 CPU 上执行，可以同时调用 `yield`、`sched` 和 `swtch`，调用将并行执行。线程各自持有锁，但它们是不同的锁，因此它们不必等待彼此。然而，一旦进入 `scheduler`，两个 CPU 可能在搜索 RUNNABLE 进程表时在锁上冲突。也就是说，xv6 可能在上下文切换期间从多个 CPU 获得性能优势，但可能不如它可能获得的多。

另一个例子是来自不同 CPU 上不同进程的并发 `fork` 调用。调用可能必须为 `pid_lock` 和 `kmem.lock` 以及搜索进程表以查找 UNUSED 进程所需的每进程锁彼此等待。另一方面，两个 fork 进程可以完全并行地复制用户内存页和格式化页表页。

上述每个例子中的锁定方案在某些情况下牺牲了并行性能。在每种情况下，使用更复杂的设计可以获得更多并行性。是否值得取决于细节：相关操作被调用的频率、代码在持有争用锁时花费多长时间、可能同时运行冲突操作的 CPU 数量、代码的其他部分是否是更严格的瓶颈。很难猜测给定的锁定方案是否可能导致性能问题，或者新设计是否明显更好，因此通常需要在现实工作负载上进行测量。

## 9.5 练习

1. 修改 xv6 的管道实现，允许对同一管道的读取和写入在不同核心上并行进行。
2. 修改 xv6 的 `scheduler()` 以减少不同核心同时查找可运行进程时的锁争用。
3. 消除 xv6 的 `fork()` 中的一些序列化。
