# 第7章 调度

任何操作系统都可能运行的进程数多于计算机的 CPU 数，因此需要计划在进程之间对 CPU 进行分时。理想情况下，共享对用户进程是透明的。常见方法是通过将进程**多路复用**（multiplexing）到硬件 CPU 上，为每个进程提供拥有自己虚拟 CPU 的错觉。本章解释 xv6 如何实现这种多路复用。

## 7.1 多路复用

xv6 在两种情况下通过将每个 CPU 从一个进程切换到另一个进程来多路复用。首先，xv6 的 `sleep` 和 `wakeup` 机制在进程等待设备或管道 I/O 完成、等待子进程退出或在 `sleep` 系统调用中等待时切换。其次，xv6 定期强制切换以应对长时间计算而不睡眠的进程。这种多路复用创造了每个进程拥有自己 CPU 的错觉，就像 xv6 使用内存分配器和硬件页表创造每个进程拥有自己内存的错觉一样。

实现多路复用带来了一些挑战。首先，如何从一个进程切换到另一个？尽管**上下文切换**（context switching）的想法很简单，但实现是 xv6 中最不透明的代码之一。其次，如何以对用户进程透明的方式强制切换？xv6 使用硬件定时器中断驱动上下文切换的标准技术。第三，所有 CPU 在相同的共享进程集之间切换，需要锁定计划以避免竞争。第四，进程的内存和其他资源必须在进程退出时释放，但它不能自己做所有这些，因为（例如）它不能在仍在使用时释放自己的内核栈。第五，多核机器的每个核心必须记住它正在执行哪个进程，以便系统调用影响正确进程的内核状态。最后，`sleep` 和 `wakeup` 允许进程放弃 CPU 并等待被另一个进程或中断唤醒。需要小心避免导致唤醒通知丢失的竞争。xv6 尽可能简单地解决这些问题，但由此产生的代码仍然很棘手。

## 7.2 代码：上下文切换

**图 7.1** 概述了从一个用户进程切换到另一个所涉及的步骤：用户-内核转换（系统调用或中断）到旧进程的内核线程，上下文切换到当前 CPU 的调度器线程，上下文切换到新进程的内核线程，以及陷阱返回到用户级进程。xv6 调度器为每个 CPU 有专用线程（保存的寄存器和栈），因为在旧进程的内核栈上执行调度器是不安全的：其他核心可能会唤醒该进程并运行它，在两个不同的核心上使用相同的栈将是灾难性的。在本节中，我们将检查内核线程和调度器线程之间切换的机制。

从一个线程切换到另一个涉及保存旧线程的 CPU 寄存器并恢复新线程先前保存的寄存器；栈指针和程序计数器被保存和恢复的事实意味着 CPU 将切换栈和正在执行的代码。

函数 `swtch` 执行内核线程切换的保存和恢复。`swtch` 不直接了解线程；它只是保存和恢复称为**上下文**（contexts）的 32 个 RISC-V 寄存器集。当进程放弃 CPU 的时候，进程的内核线程调用 `swtch` 保存自己的上下文并返回到调度器上下文。每个上下文包含在 `struct context`（`kernel/proc.h:2`）中，本身包含在进程的 `struct proc` 或 CPU 的 `struct cpu` 中。`swtch` 接受两个参数：`struct context *old` 和 `struct context *new`。它将当前寄存器保存到 old，从 new 加载寄存器，并返回。

让我们跟踪一个进程通过 `swtch` 进入调度器。我们在第 4 章看到，中断结束时的一种可能是 `usertrap` 调用 `yield`。`yield` 依次调用 `sched`，后者调用 `swtch` 将当前上下文保存到 `p->context` 并切换到先前保存到 `cpu->context` 的调度器上下文（`kernel/proc.c:497`）。

`swtch`（`kernel/swtch.S:3`）只保存被调用者保存的寄存器；C 编译器在调用者保存的寄存器中生成代码。`swtch` 保存每个被调用者保存的寄存器。它保存 sp 和 ra，因为它们分别保存栈指针和返回地址。`swtch` 现在返回到 `scheduler` 的栈上，因为 `scheduler` 之前调用了 `swtch` 切换到 `p->context`。此时，我们正在 `scheduler` 的线程中，而不是进程的内核线程中，并且 `sched` 的栈帧不再有效。

## 7.3 代码：调度

`scheduler`（`kernel/proc.c:513`）为每个 CPU 运行一个循环。该循环必须持有 `p->lock`，因此它首先获取该锁（`kernel/proc.c:521`）。然后它扫描进程表，寻找 `state == RUNNABLE` 的进程。它找到这样的进程后，将该进程的每 CPU 状态设置为当前 CPU，将进程状态设置为 RUNNING，调用 `kvminithart` 切换页表，调用 `swtch` 切换到进程的内核线程，并最终在进程的内核线程中返回。然后进程继续执行，直到（例如）它让出或定时器中断强制切换。如果 `scheduler` 找不到 RUNNABLE 进程，它会调用 `intr_on` 并自旋，等待中断；最终某个进程会变为 RUNNABLE（例如，当进程退出或睡眠时），然后当前 CPU 的中断会唤醒 `scheduler` 循环。

## 7.4 代码：mycpu 和 myproc

xv6 经常需要指向当前进程的 `struct proc` 的指针。在单处理器上，可以有一个全局变量指向当前进程。这在多处理器上不起作用，因为每个 CPU 运行不同的进程。解决此问题的方法是利用每个 CPU 都有自己的一组寄存器的事实；我们可以使用其中一个来帮助查找每 CPU 状态。

RISC-V 为每个 CPU 分配一个**硬件线程 ID**（hartid），存储在 `tp` 寄存器中。`mycpu`（`kernel/proc.c:83`）返回当前 CPU 的 `struct cpu`。它使用 `tp` 索引 `cpus` 数组。确保 `tp` 正确设置是 `usertrapret`、`uservec` 和 `kvminithart` 的责任；它们设置 `tp` 指向当前 CPU 的 `struct cpu`。

`myproc`（`kernel/proc.c:88`）返回当前 CPU 上运行的进程的 `struct proc` 指针。`myproc` 禁用中断，调用 `mycpu`，从 `struct cpu` 获取当前进程指针（`c->proc`），然后重新启用中断。即使启用了中断，`myproc` 的返回值在禁用中断时也是有效的：定时器中断可能将当前进程强制切换到调度器，但不会更改 `c->proc` 指向的 `struct proc`。

## 7.5 Sleep 与 Wakeup

调度和锁有助于向进程隐藏 CPU 的存在，但到目前为止我们还没有帮助进程等待 I/O 或等待另一个进程的抽象。例如，管道读取端等待有人写入管道；父进程的 `wait` 调用等待子进程退出。xv6 的解决方案是 `sleep` 和 `wakeup` 原语。

`sleep` 让调用进程进入睡眠，释放 CPU 供其他工作使用。`wakeup` 唤醒在给定「通道」上睡眠的所有进程。`sleep` 和 `wakeup` 通常被称为**序列协调**（sequence coordination）或**条件同步**（conditional synchronization）机制。

考虑 `wait` 和 `exit` 之间的信号量式同步。`exit` 将子进程置于 ZOMBIE 状态，唤醒可能在 `wait` 中睡眠的父进程，让出 CPU，并让调度器运行其他进程。父进程的 `wait` 最终会注意到子进程已退出，释放子进程的 `struct proc`，并返回子进程的 PID。如果父进程在子进程之前到达 `wait`，父进程必须 sleep 直到子进程退出；子进程的 `exit` 调用 `wakeup` 唤醒父进程。

`sleep` 和 `wakeup` 的挑战是避免**丢失唤醒**（lost wake-up）。假设 `wait` 发现子进程的 `state != ZOMBIE`。在 `wait` 调用 `sleep` 之前，子进程退出并在另一个 CPU 上调用 `exit`。`exit` 调用 `wakeup`，但没有进程在睡眠，所以唤醒丢失。父进程的 `wait` 最终会调用 `sleep`，但可能永远不会被唤醒。避免这种竞争是 `sleep` 接口接受锁参数的原因。调用者必须在检查条件之前持有关联的锁，并且 `sleep` 在将进程置于睡眠并安排唤醒后释放锁。锁强制 `exit` 中的 `wakeup` 在父进程的 `wait` 检查子进程状态和调用 `sleep` 之间等待，因此 `wakeup` 不会丢失。

## 7.6 代码：Sleep 与 Wakeup

让我们看看 `sleep`（`kernel/proc.c:548`）和 `wakeup`（`kernel/proc.c:582`）的实现。基本思路是 `sleep` 将当前进程标记为 SLEEPING 并调用 `sched` 释放 CPU；`wakeup` 查找在给定通道上睡眠的进程并将其标记为 RUNNABLE。

`sleep` 获取 `p->lock`（`kernel/proc.c:559`）。现在持有 `sleep` 的调用者的锁和 `p->lock`。调用者（例如 `wait` 中的代码）必须持有锁；它将在检查条件之前获取锁，以便 `wakeup` 不会错过检查和 `sleep` 之间的窗口。现在 `sleep` 持有 `p->lock`，释放调用者的锁是安全的：其他进程可能会调用 `wakeup`，但 `wakeup` 会等待获取 `p->lock`，因此会等待 `sleep` 完成将进程置于睡眠，从而避免 `wakeup` 错过 `sleep` 的竞争。

有一个复杂情况：如果调用者的锁是 `p->lock`，那么 `sleep` 会死锁，因为它在持有 `p->lock` 时会尝试获取它。在这种情况下，`sleep` 不获取和释放锁（`kernel/proc.c:567-568`）。例如，`wait` 持有 `p->lock` 并调用 `sleep`。`wakeup` 在将进程标记为 RUNNABLE 之前持有 `wait_lock`，因此 `wait` 和 `wakeup` 不会错过彼此。

## 7.7 代码：管道

使用 `sleep` 和 `wakeup` 同步生产者和消费者的更复杂例子是 xv6 的管道实现。我们在第 1 章看到了管道的接口：写入管道一端的数据被复制到内核缓冲区，然后可以从管道的另一端读取。后续章节将研究围绕管道的文件描述符支持，但现在让我们看看 `pipewrite` 和 `piperead` 的实现。

每个管道由一个 `struct pipe` 表示，其中包含锁和数据缓冲区。字段 `nread` 和 `nwrite` 计算从缓冲区读取和写入的总字节数。缓冲区环绕：在 `buf[PIPESIZE-1]` 之后写入的下一个字节是 `buf[0]`。计数不环绕。此约定让实现区分满缓冲区（`nwrite == nread+PIPESIZE`）和空缓冲区（`nwrite == nread`），但意味着索引到缓冲区必须使用 `buf[nread % PIPESIZE]` 而不是 `buf[nread]`（对 nwrite 类似）。

假设对 `piperead` 和 `pipewrite` 的调用在两个不同的 CPU 上同时发生。`pipewrite`（`kernel/pipe.c:77`）首先获取管道的锁，该锁保护计数、数据及其相关不变量。`piperead`（`kernel/pipe.c:106`）然后尝试获取锁，但不能。它在 `acquire`（`kernel/spinlock.c:22`）中自旋等待锁。当 `piperead` 等待时，`pipewrite` 循环遍历正在写入的字节（`addr[0..n-1]`），依次将每个添加到管道（`kernel/pipe.c:95`）。在此循环期间，可能发生缓冲区满（`kernel/pipe.c:88`）。在这种情况下，`pipewrite` 调用 `wakeup` 提醒任何睡眠的读取者缓冲区中有数据等待，然后在 `&pi->nwrite` 上睡眠，等待读取者从缓冲区取出一些字节。`sleep` 作为将 `pipewrite` 的进程置于睡眠的一部分释放 `pi->lock`。

现在 `pi->lock` 可用，`piperead` 设法获取它并进入其临界区：它发现 `pi->nread != pi->nwrite`（`kernel/pipe.c:113`）（`pipewrite` 因为 `pi->nwrite == pi->nread+PIPESIZE`（`kernel/pipe.c:88`）而睡眠），因此它进入 for 循环，从管道复制数据（`kernel/pipe.c:120`），并将 `nread` 增加复制的字节数。那么多字节现在可用于写入，因此 `piperead` 在返回之前调用 `wakeup`（`kernel/pipe.c:127`）唤醒任何睡眠的写入者。`wakeup` 找到在 `&pi->nwrite` 上睡眠的进程，即运行 `pipewrite` 但在缓冲区满时停止的进程。它将该进程标记为 RUNNABLE。

## 7.8 代码：Wait、Exit 与 Kill

`sleep` 和 `wakeup` 可用于多种等待。第 1 章介绍的一个有趣的例子是子进程的 `exit` 与其父进程的 `wait` 之间的交互。在子进程死亡时，父进程可能已经在 `wait` 中睡眠，或者可能在做其他事情；在后一种情况下，后续对 `wait` 的调用必须观察到子进程的死亡，可能在其调用 `exit` 很久之后。xv6 记录子进程死亡直到 `wait` 观察到的方式是让 `exit` 将调用者置于 ZOMBIE 状态，它保持在那里直到父进程的 `wait` 注意到它，将子进程的状态更改为 UNUSED，复制子进程的退出状态，并将子进程的进程 ID 返回给父进程。如果父进程在子进程之前退出，父进程将子进程交给 init 进程，后者永久调用 `wait`；因此每个子进程都有父进程来清理。一个挑战是避免同时的父进程和子进程 `wait` 和 `exit` 之间的竞争和死锁，以及同时的 `exit` 和 `exit`。

`wait` 首先获取 `wait_lock`（`kernel/proc.c:391`）。原因是 `wait_lock` 充当条件锁，有助于确保父进程不会错过来自退出子进程的唤醒。然后 `wait` 扫描进程表。如果它找到 ZOMBIE 状态的子进程，它释放该子进程的资源和 proc 结构，将子进程的退出状态复制到提供给 `wait` 的地址（如果不是 0），并返回子进程的进程 ID。如果 `wait` 找到子进程但没有一个已退出，它调用 `sleep` 等待其中任何一个退出（`kernel/proc.c:433`），然后再次扫描。`wait` 通常持有两个锁，`wait_lock` 和某个进程的 `pp->lock`；避免死锁的顺序是先 `wait_lock` 然后 `pp->lock`。

`exit`（`kernel/proc.c:347`）记录退出状态，释放一些资源，调用 `reparent` 将子进程交给 init 进程，在父进程可能在 `wait` 中时唤醒父进程，将调用者标记为僵尸，并永久让出 CPU。`exit` 在此序列期间持有 `wait_lock` 和 `p->lock`。它持有 `wait_lock` 因为它是 `wakeup(p->parent)` 的条件锁，防止 `wait` 中的父进程丢失唤醒。`exit` 也必须为此序列持有 `p->lock`，以防止 `wait` 中的父进程在子进程最终调用 `swtch` 之前看到子进程处于 ZOMBIE 状态。`exit` 以与 `wait` 相同的顺序获取这些锁以避免死锁。

## 7.9 进程锁

与每个进程关联的锁（`p->lock`）是 xv6 中最复杂的锁。考虑 `p->lock` 的简单方式是，在读取或写入以下 `struct proc` 字段时必须持有它：`p->state`、`p->chan`、`p->killed`、`p->xstate` 和 `p->pid`。这些字段可以被其他进程或其他核心上的调度器线程使用，因此它们必须由锁保护是自然的。

然而，`p->lock` 的大多数用途是保护 xv6 进程数据结构和算法的更高级方面。以下是 `p->lock` 所做的完整集合：

- 与 `p->state` 一起，防止为新进程分配 `proc[]` 槽的竞争。
- 在创建或销毁时隐藏进程。
- 防止父进程的 `wait` 收集已将其状态设置为 ZOMBIE 但尚未让出 CPU 的进程。
- 防止另一个核心的调度器在让出进程设置其状态为 RUNNABLE 之后但在完成 `swtch` 之前决定运行它。
- 确保只有一个核心的调度器决定运行 RUNNABLE 进程。
- 防止定时器中断在进程在 `swtch` 中时导致进程让出。
- 与条件锁一起，有助于防止 `wakeup` 忽略正在调用 `sleep` 但尚未完成让出 CPU 的进程。
- 防止 `kill` 的受害者进程在 `kill` 检查 `p->pid` 和设置 `p->killed` 之间退出并可能被重新分配。
- 使 `kill` 对 `p->state` 的检查和写入原子化。

`p->parent` 字段由全局锁 `wait_lock` 保护而非 `p->lock`。只有进程的父进程修改 `p->parent`，尽管该字段由进程本身和搜索其子进程的其他进程读取。`wait_lock` 的目的是在 `wait` 睡眠等待任何子进程退出时充当条件锁。退出的子进程持有 `wait_lock` 或 `p->lock`，直到它设置其状态为 ZOMBIE、唤醒其父进程并让出 CPU。`wait_lock` 还序列化父进程和子进程的并发退出，以便继承子进程的 init 进程保证从 `wait` 中唤醒。`wait_lock` 是全局锁而非每个父进程的每进程锁，因为直到进程获取它，它无法知道其父进程是谁。

## 7.10 现实世界

xv6 调度器实现简单的调度策略，依次运行每个进程。此策略称为**轮转**（round robin）。真实操作系统实现更复杂的策略，例如允许进程有优先级。想法是，可运行的高优先级进程将比可运行的低优先级进程更受调度器青睐。这些策略可能很快变得复杂，因为通常有相互竞争的目标：例如，操作系统可能还想保证公平性和高吞吐量。此外，复杂的策略可能导致意外的交互，如**优先级反转**（priority inversion）和** convoy**。当低优先级和高优先级进程都使用特定锁时，当低优先级进程获取时可能阻止高优先级进程取得进展，就会发生优先级反转。当许多高优先级进程等待获取共享锁的低优先级进程时，会形成等待进程的长 convoy；一旦 convoy 形成，它可以持续很长时间。为避免此类问题，复杂调度器需要额外的机制。

## 7.11 练习

1. Sleep 必须检查 `lk != &p->lock` 以避免死锁。假设通过将 `if(lk != &p->lock){ acquire(&p->lock); release(lk); }` 替换为 `release(lk); acquire(&p->lock);` 来消除特殊情况。这样做会破坏 sleep。如何？
2. 在 xv6 中实现信号量，不使用 sleep 和 wakeup（但可以使用自旋锁）。用信号量替换 xv6 中 sleep 和 wakeup 的使用。评估结果。
3. 修复上述 kill 和 sleep 之间的竞争，以便在受害者的 sleep 循环检查 `p->killed` 之后但在调用 sleep 之前发生的 kill 导致受害者放弃当前系统调用。
4. 设计一个计划，使每个 sleep 循环都检查 `p->killed`，以便例如，如果进程在 virtio 驱动中被另一个进程杀死，它可以快速从 while 循环返回。
5. 修改 xv6，使其在从一个进程的内核线程切换到另一个时只进行一次上下文切换，而不是通过调度器线程切换。让出的线程需要自己选择下一个线程并调用 `swtch`。挑战将是防止多个核心意外执行同一线程；正确获取锁；以及避免死锁。
6. 修改 xv6 的调度器以在无可运行进程时使用 RISC-V WFI（等待中断）指令。尝试确保，任何时候有可运行进程等待运行时，没有核心在 WFI 中暂停。
