# 第6章 锁

大多数内核，包括 xv6，交错执行多项活动。交错的一个来源是**多处理器**（multiprocessor）硬件：具有多个独立执行的 CPU 的计算机，例如 xv6 的 RISC-V。这些多个 CPU 共享物理 RAM，xv6 利用这种共享来维护所有 CPU 读写的数据结构。这种共享带来了一个 CPU 在另一个 CPU 正在更新时读取数据结构的可能性，甚至多个 CPU 同时更新相同的数据；没有仔细设计，这种并行访问可能会产生不正确的结果或损坏的数据结构。即使在单处理器上，内核也可能在多个线程之间切换 CPU，导致它们的执行交错。最后，修改与某些可中断代码相同数据的设备中断处理程序，如果中断在错误的时间发生，可能会损坏数据。**并发**（concurrency）一词指的是由于多处理器并行、线程切换或中断而导致多个指令流交错的情况。

内核充满了并发访问的数据。例如，两个 CPU 可能同时调用 `kalloc`，从而并发地从空闲链表头部弹出。内核设计者喜欢允许大量并发，因为它可以通过并行性产生更高的性能，以及更高的响应性。然而，因此内核设计者必须说服自己在存在这种并发的情况下正确性。有许多方法可以得出正确的代码，有些比其他更容易推理。旨在实现并发下正确性的策略以及支持它们的抽象称为**并发控制**（concurrency control）技术。

xv6 根据情况使用多种并发控制技术；还有更多可能。本章侧重于广泛使用的技术：**锁**（lock）。锁提供**互斥**（mutual exclusion），确保一次只有一个 CPU 可以持有锁。如果程序员将锁与每个共享数据项关联，并且代码在使用项时始终持有关联的锁，则该项将一次只被一个 CPU 使用。在这种情况下，我们说锁**保护**（protects）数据项。虽然锁是易于理解的并发控制机制，但锁的缺点在于它们可能限制性能，因为它们**序列化**（serialize）并发操作。本章其余部分解释为什么 xv6 需要锁、xv6 如何实现它们以及如何使用它们。

## 6.1 竞争

作为为什么需要锁的例子，考虑两个有已退出子进程的进程在两个不同的 CPU 上调用 `wait`。`wait` 释放子进程的内存。因此，在每个 CPU 上，内核将调用 `kfree` 释放子进程的内存页。内核分配器维护一个链表：`kalloc()`（`kernel/kalloc.c:69`）从空闲页链表弹出一页内存，`kfree()`（`kernel/kalloc.c:47`）将一页推入空闲链表。为了最佳性能，我们可能希望两个父进程的 `kfree` 会并行执行而不必等待另一个，但鉴于 xv6 的 `kfree` 实现，这将是不正确的。

**图 6.1** 更详细地说明了设置：空闲页的链表在由两个 CPU 共享的内存中，它们使用加载和存储指令操作链表。（实际上，处理器有缓存，但概念上多处理器系统的行为就好像有单个共享内存。）如果没有并发请求，你可能会按如下方式实现链表推操作：

```c
struct element {
  int data;
  struct element *next;
};

struct element *list = 0;

void push(int data)
{
  struct element *l;

  l = malloc(sizeof *l);
  l->data = data;
  l->next = list;
  list = l;
}
```

如果单独执行，此实现是正确的。然而，如果多个副本并发执行，代码就不正确。如果两个 CPU 同时执行 `push`，两者可能如**图 6.2** 所示在执行第 16 行之前都执行第 15 行，这会导致如图所示的不正确结果。将会有两个链表元素，其 next 设置为 list 的先前值。当第 16 行对 list 的两次赋值发生时，第二次将覆盖第一次；第一次赋值涉及的元素将丢失。

第 16 行的丢失更新是**竞争**（race）的例子。竞争是内存位置被并发访问且至少一次访问是写的情况。竞争通常是错误的迹象，要么是丢失更新（如果访问是写），要么是对未完全更新的数据结构的读取。竞争的结果取决于编译器生成的机器代码、所涉及的两个 CPU 的时序以及内存系统如何排序它们的内存操作，这会使竞争引起的错误难以重现和调试。例如，在调试 `push` 时添加 print 语句可能会改变执行的时序，足以使竞争消失。

避免竞争的通常方法是使用锁。锁确保互斥，因此一次只有一个 CPU 可以执行 `push` 的敏感行；这使上述场景不可能发生。上述代码的正确加锁版本只添加几行（以黄色突出显示）：

```c
struct element *list = 0;
struct lock listlock;

void push(int data)
{
  struct element *l;
  l = malloc(sizeof *l);
  l->data = data;
  acquire(&listlock);
  l->next = list;
  list = l;
  release(&listlock);
}
```

`acquire` 和 `release` 之间的指令序列通常称为**临界区**（critical section）。锁通常被称为保护 list。

当我们说锁保护数据时，我们真正意思是锁保护适用于数据的一些**不变量**（invariants）集合。不变量是在操作之间维护的数据结构的属性。通常，操作的正确行为取决于操作开始时不变量为真。操作可能暂时违反不变量，但必须在完成前重新建立它们。例如，在链表情况下，不变量是 list 指向链表中的第一个元素，每个元素的 next 字段指向下一个元素。`push` 的实现暂时违反此不变量：在第 17 行，l 指向下一个链表元素，但 list 还不指向 l（在第 18 行重新建立）。我们上面检查的竞争发生是因为第二个 CPU 在不变量（暂时）被违反时执行了依赖链表不变量的代码。正确使用锁确保一次只有一个 CPU 可以在临界区中操作数据结构，因此没有 CPU 会在数据结构的不变量不成立时执行数据结构操作。

你可以将锁视为**序列化**并发临界区，使它们一次运行一个，从而保留不变量（假设临界区在单独时是正确的）。你也可以将由同一锁保护的临界区视为彼此**原子**（atomic），因此每个只看到早期临界区的完整更改集，永远不会看到部分完成的更新。

虽然对正确性有用，但锁本质上限制性能。例如，如果两个进程并发调用 `kfree`，锁将序列化两个临界区，因此在不同 CPU 上运行它们没有好处。我们说多个进程**冲突**（conflict）如果它们同时想要同一个锁，或者说锁经历**争用**（contention）。内核设计中的主要挑战是在追求并行性时避免锁争用。xv6 在这方面做得很少，但复杂的内核专门组织数据结构和算法以避免锁争用。

## 6.2 代码：锁

xv6 有两种类型的锁：**自旋锁**（spinlocks）和**睡眠锁**（sleep-locks）。我们从自旋锁开始。xv6 将自旋锁表示为 `struct spinlock`（`kernel/spinlock.h:2`）。结构中的重要字段是 `locked`，当锁可用时为零、持有时为非零的字。逻辑上，xv6 应该通过执行如下代码获取锁：

```c
void acquire(struct spinlock *lk) // 不工作！
{
  for(;;) {
    if(lk->locked == 0) {
      lk->locked = 1;
      break;
    }
  }
}
```

不幸的是，此实现在多处理器上不能保证互斥。可能发生两个 CPU 同时到达第 25 行，看到 `lk->locked` 为零，然后都通过执行第 26 行获取锁。此时，两个不同的 CPU 持有锁，这违反了互斥属性。我们需要的是使第 25 和 26 行作为**原子**（即不可分割）步骤执行的方法。

因为锁被广泛使用，多核处理器通常提供实现第 25 和 26 行原子版本的指令。在 RISC-V 上，该指令是 `amoswap r, a`。`amoswap` 读取内存地址 a 处的值，将寄存器 r 的内容写入该地址，并将读取的值放入 r。也就是说，它交换寄存器和内存地址的内容。它使用特殊硬件原子地执行此序列，防止任何其他 CPU 在读取和写入之间使用内存地址。

xv6 的 `acquire`（`kernel/spinlock.c:22`）使用可移植的 C 库调用 `__sync_lock_test_and_set`，它归结为 `amoswap` 指令；返回值是 `lk->locked` 的旧（交换的）内容。`acquire` 函数将交换包装在循环中，重试（自旋）直到它获取锁。每次迭代将 1 交换到 `lk->locked` 并检查先前的值；如果先前的值为零，则我们已获取锁，交换将把 `lk->locked` 设置为 1。如果先前的值为 1，则其他 CPU 持有锁，我们原子地将 1 交换到 `lk->locked` 的事实没有改变其值。

一旦获取锁，`acquire` 为调试记录获取锁的 CPU。`lk->cpu` 字段由锁保护，必须仅在持有时更改。函数 `release`（`kernel/spinlock.c:47`）与 `acquire` 相反：它清除 `lk->cpu` 字段，然后释放锁。概念上，`release` 只需要将零赋给 `lk->locked`。C 标准允许编译器用多个存储指令实现赋值，因此 C 赋值相对于并发代码可能是非原子的。相反，`release` 使用执行原子赋值的 C 库函数 `__sync_lock_release`。此函数也归结为 RISC-V `amoswap` 指令。

## 6.3 代码：使用锁

xv6 在许多地方使用锁以避免竞争。如上所述，`kalloc`（`kernel/kalloc.c:69`）和 `kfree`（`kernel/kalloc.c:47`）是一个很好的例子。尝试练习 1 和 2 看看如果这些函数省略锁会发生什么。你可能会发现很难触发不正确行为，这表明很难可靠地测试代码是否没有锁错误和竞争。xv6 很可能有尚未发现的竞争。

使用锁的一个困难部分是决定使用多少锁以及每个锁应该保护哪些数据和不变式。有几个基本原则。首先，每当一个变量可以被一个 CPU 写入而另一个 CPU 可以同时读取或写入时，应使用锁来防止两个操作重叠。其次，记住锁保护不变量：如果不变量涉及多个内存位置，通常它们都需要由单个锁保护以确保不变量得到维护。

上述规则说明何时需要锁，但没有说明何时不需要锁，对于效率来说重要的是不要锁太多，因为锁会减少并行性。如果并行性不重要，那么可以安排只有一个线程而不必担心锁。简单内核可以通过在进入内核时获取并在退出内核时释放的单个锁在多处理器上做到这一点（尽管诸如管道读取或 wait 之类的阻塞系统调用会带来问题）。许多单处理器操作系统已使用这种方法（有时称为「大内核锁」）转换为在多处理器上运行，但该方法牺牲了并行性：一次只有一个 CPU 可以在内核中执行。如果内核进行任何繁重的计算，使用更大的一组更细粒度的锁会更有效，以便内核可以同时在多个 CPU 上执行。

作为粗粒度锁定的例子，xv6 的 `kalloc.c` 分配器有一个由单个锁保护的单个空闲链表。如果不同 CPU 上的多个进程同时尝试分配页，每个都将不得不通过在 `acquire` 中自旋等待轮到它。自旋浪费 CPU 时间，因为它不是有用的工作。如果对锁的争用浪费了 CPU 时间的很大一部分，也许可以通过将分配器设计更改为有多个空闲链表（每个都有自己的锁）来改善性能，以允许真正的并行分配。

作为细粒度锁定的例子，xv6 为每个文件有单独的锁，因此操作不同文件的进程通常可以继续而不必等待彼此的锁。如果希望允许进程同时写入同一文件的不同区域，文件锁定方案可以更加细粒度。最终，锁粒度决策需要由性能测量以及复杂性考虑来驱动。

## 6.4 死锁与锁顺序

如果通过内核的代码路径必须同时持有多个锁，重要的是所有代码路径以相同顺序获取这些锁。如果不这样做，就有死锁的风险。假设 xv6 中的两条代码路径需要锁 A 和 B，但代码路径 1 以 A 然后 B 的顺序获取锁，另一条路径以 B 然后 A 的顺序获取锁。假设线程 T1 执行代码路径 1 并获取锁 A，线程 T2 执行代码路径 2 并获取锁 B。接下来 T1 将尝试获取锁 B，T2 将尝试获取锁 A。两个获取都将无限期阻塞，因为在两种情况下另一个线程持有所需的锁，并且不会释放它直到其获取返回。为避免此类死锁，所有代码路径必须以相同顺序获取锁。对全局锁获取顺序的需求意味着锁实际上是每个函数规范的一部分：调用者必须以导致锁以约定顺序获取的方式调用函数。

## 6.5 可重入锁

使用**可重入锁**（re-entrant locks，也称为递归锁）可能看起来可以避免一些死锁和锁顺序挑战。想法是，如果锁被进程持有，且该进程再次尝试获取锁，则内核可以允许这样做（因为进程已经持有锁），而不是像 xv6 内核那样调用 panic。然而，可重入锁使推理并发变得更难：可重入锁打破了锁导致临界区相对于其他临界区是原子的直觉。因此，xv6 使用更易于理解的不可重入锁。只要程序员牢记锁定规则，两种方法都可以工作。

## 6.6 锁与中断处理程序

一些 xv6 自旋锁保护线程和中断处理程序都使用的数据。例如，`clockintr` 定时器中断处理程序可能同时递增 `ticks`（`kernel/trap.c:164`），而内核线程在 `sys_sleep`（`kernel/sysproc.c:59`）中读取。锁 `tickslock` 序列化两次访问。自旋锁和中断的交互带来了潜在危险。假设 `sys_sleep` 持有 `tickslock`，其 CPU 被定时器中断中断。`clockintr` 会尝试获取 `tickslock`，看到它被持有，并等待释放。在这种情况下，`tickslock` 永远不会被释放：只有 `sys_sleep` 可以释放它，但 `sys_sleep` 在 `clockintr` 返回之前不会继续运行。因此 CPU 会死锁。为避免这种情况，如果自旋锁被中断处理程序使用，CPU 绝不能在中断启用时持有该锁。xv6 更保守：当 CPU 获取任何锁时，xv6 总是禁用该 CPU 上的中断。`acquire` 调用 `push_off`，`release` 调用 `pop_off` 来跟踪当前 CPU 上锁的嵌套级别。

## 6.7 指令与内存排序

很自然地认为程序按源代码语句出现的顺序执行。这对于单线程代码是合理的心理模型，但当多个线程通过共享内存交互时是不正确的。一个原因是编译器以与源代码暗示的顺序不同的顺序发出加载和存储指令，并可能完全省略它们（例如，通过在寄存器中缓存数据）。另一个原因是 CPU 可能乱序执行指令以提高性能。为避免此类问题，xv6 在 `acquire` 和 `release` 中都使用 `__sync_synchronize()`。`__sync_synchronize()` 是**内存屏障**（memory barrier）：它告诉编译器和 CPU 不要在屏障上重新排序加载或存储。

## 6.8 睡眠锁

有时 xv6 需要长时间持有锁。例如，文件系统（第 8 章）在磁盘上读写其内容时保持文件锁定，这些磁盘操作可能需要数十毫秒。持有自旋锁那么长时间会导致浪费，如果另一个进程想要获取它，因为获取进程会在自旋时长时间浪费 CPU。自旋锁的另一个缺点是进程不能在保留自旋锁时让出 CPU；我们希望这样做，以便持有锁的进程等待磁盘时其他进程可以使用 CPU。在持有自旋锁时让出是非法的，因为如果第二个线程然后尝试获取自旋锁，可能导致死锁；由于 `acquire` 不让出 CPU，第二个线程的自旋可能会阻止第一个线程运行和释放锁。在持有锁时让出也会违反自旋锁持有时必须关闭中断的要求。因此，我们希望一种在等待获取时让出 CPU 的锁类型，并允许在持有时让出（和中断）。

xv6 以睡眠锁的形式提供此类锁。`acquiresleep`（`kernel/sleeplock.c:22`）在等待时让出 CPU，使用第 7 章将解释的技术。在高层次上，睡眠锁有一个由自旋锁保护的 `locked` 字段，`acquiresleep` 对 `sleep` 的调用原子地让出 CPU 并释放自旋锁。结果是其他线程可以在 `acquiresleep` 等待时执行。

因为睡眠锁保持中断启用，它们不能用于中断处理程序。因为 `acquiresleep` 可能让出 CPU，睡眠锁不能用于自旋锁临界区内（尽管自旋锁可以用于睡眠锁临界区内）。

自旋锁最适合短临界区，因为等待它们浪费 CPU 时间；睡眠锁适用于冗长的操作。

## 6.9 现实世界

尽管多年来对并发原语和并行性的研究，使用锁编程仍然具有挑战性。通常最好将锁隐藏在更高级的构造中，如同步队列，尽管 xv6 不这样做。如果你使用锁编程，明智的做法是使用试图识别竞争的工具，因为很容易遗漏需要锁的不变量。大多数操作系统支持 POSIX 线程（Pthreads），允许用户进程在多个 CPU 上同时运行多个线程。Pthreads 支持用户级锁、屏障等。Pthreads 还允许程序员可选地指定锁应该是可重入的。没有原子指令也可以实现锁，但很昂贵，大多数操作系统使用原子指令。

## 6.10 练习

1. 注释掉 `kalloc`（`kernel/kalloc.c:69`）中对 `acquire` 和 `release` 的调用。这似乎应该对调用 `kalloc` 的内核代码造成问题；你期望看到什么症状？运行 xv6 时，你看到这些症状了吗？运行 `usertests` 时呢？如果你没有看到问题，为什么？看看能否通过在 `kalloc` 的临界区插入虚拟循环来引发问题。
2. 假设你改为注释掉 `kfree` 中的锁定（在恢复 `kalloc` 中的锁定之后）。现在可能出什么问题？`kfree` 中缺少锁是否比 `kalloc` 中危害更小？
3. 如果两个 CPU 同时调用 `kalloc`，一个将不得不等待另一个，这对性能不利。修改 `kalloc.c` 以获得更多并行性，以便来自不同 CPU 的 `kalloc` 的并发调用可以继续而不必等待彼此。
4. 使用 POSIX 线程编写并行程序，大多数操作系统都支持。例如，实现并行哈希表并测量 put/get 的数量是否随核心数增加而扩展。
5. 在 xv6 中实现 Pthread 的子集。也就是说，实现用户级线程库，以便用户进程可以有多个线程，并安排这些线程可以在不同的 CPU 上并行运行。提出一个设计，正确处理线程进行阻塞系统调用和更改其共享地址空间。
