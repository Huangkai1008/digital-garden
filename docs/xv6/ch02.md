# 第2章 操作系统组织

操作系统的关键要求之一是同时支持多项活动。例如，使用第 1 章描述的系统调用接口，进程可以用 `fork` 启动新进程。操作系统必须在这些进程之间对计算机资源进行**分时**（time-share）。例如，即使进程数多于硬件 CPU 数，操作系统也必须确保所有进程都有机会执行。操作系统还必须安排进程之间的**隔离**（isolation）。也就是说，如果一个进程有错误并发生故障，它不应影响不依赖该故障进程的进程。然而，完全隔离过于严格，因为进程应该能够有意交互；管道就是一个例子。因此，操作系统必须满足三个要求：**多路复用**（multiplexing）、**隔离**（isolation）和**交互**（interaction）。

本章概述操作系统如何组织以实现这三个要求。事实证明有很多方法可以做到，但本文侧重于围绕**单体内核**（monolithic kernel）的主流设计，许多 Unix 操作系统都使用这种设计。本章还概述 xv6 进程（xv6 中的隔离单位）以及 xv6 启动时第一个进程的创建。

xv6 在**多核**（multi-core）RISC-V 微处理器上运行，其许多低级功能（例如进程实现）是 RISC-V 特有的。RISC-V 是 64 位 CPU，xv6 用「LP64」C 编写，这意味着 C 语言中的 long（L）和指针（P）是 64 位，但 int 是 32 位。本书假设读者在某种架构上做过一些机器级编程，并将在出现时介绍 RISC-V 特有的概念。RISC-V 的有用参考是「The RISC-V Reader: An Open Architecture Atlas」。用户级 ISA 和特权架构是官方规范。

完整计算机中的 CPU 被支持硬件包围，其中大部分以 I/O 接口的形式存在。xv6 为 qemu 的「-machine virt」选项模拟的支持硬件编写。这包括 RAM、包含引导代码的 ROM、连接到用户键盘/屏幕的串行连接以及用于存储的磁盘。

## 2.1 抽象物理资源

遇到操作系统时人们可能首先会问：为什么要有它？也就是说，可以将图 1.2 中的系统调用实现为库，应用程序与之链接。在此计划中，每个应用程序甚至可以有自己的库以满足其需求。应用程序可以直接与硬件资源交互并以最适合应用程序的方式使用这些资源（例如，实现高或可预测的性能）。一些嵌入式设备或实时系统的操作系统就是以这种方式组织的。

这种库方法的缺点在于，如果有多个应用程序运行，应用程序必须表现良好。例如，每个应用程序必须定期放弃 CPU 以便其他应用程序可以运行。如果所有应用程序相互信任且没有错误，这种协作式分时方案可能可行。更典型的是应用程序不相互信任且有错误，因此人们通常需要比协作方案提供的更强的隔离。

要实现强隔离，禁止应用程序直接访问敏感硬件资源是有帮助的，而是将资源抽象为服务。例如，Unix 应用程序仅通过文件系统的 `open`、`read`、`write` 和 `close` 系统调用与存储交互，而不是直接读写磁盘。这为应用程序提供了路径名的便利，并允许操作系统（作为接口的实现者）管理磁盘。即使隔离不是问题，有意交互的进程（或只是希望彼此不相干扰）也可能发现文件系统比直接使用磁盘更方便的抽象。

同样，Unix 透明地在进程之间切换硬件 CPU，根据需要保存和恢复寄存器状态，因此应用程序不必意识到分时。这种透明性允许操作系统共享 CPU，即使某些应用程序处于无限循环中。

作为另一个例子，Unix 进程使用 `exec` 构建其内存映像，而不是直接与物理内存交互。这允许操作系统决定将进程放在内存中的何处；如果内存紧张，操作系统甚至可能将进程的一些数据存储在磁盘上。`exec` 还为用户提供文件系统存储可执行程序映像的便利。

许多 Unix 进程之间的交互形式通过文件描述符发生。文件描述符不仅抽象了许多细节（例如管道或文件中数据的存储位置），它们还以简化交互的方式定义。例如，如果管道中的一个应用程序失败，内核会为管道中的下一个进程生成文件结束信号。

图 1.2 中的系统调用接口经过精心设计，既提供程序员便利，又提供强隔离的可能性。Unix 接口不是抽象资源的唯一方式，但已被证明是非常好的方式。

## 2.2 用户模式、监督模式与系统调用

强隔离要求应用程序与操作系统之间有一条硬边界。如果应用程序出错，我们不希望操作系统失败或其他应用程序失败。相反，操作系统应该能够清理失败的应用程序并继续运行其他应用程序。要实现强隔离，操作系统必须安排应用程序无法修改（甚至读取）操作系统的数据结构和指令，并且应用程序无法访问其他进程的内存。

CPU 提供硬件支持以实现强隔离。例如，RISC-V 有三种 CPU 可以执行指令的模式：**机器模式**（machine mode）、**监督模式**（supervisor mode）和**用户模式**（user mode）。在机器模式下执行的指令具有完全特权；CPU 在机器模式下启动。机器模式主要用于配置计算机。xv6 在机器模式下执行几行后切换到监督模式。

在监督模式下，CPU 允许执行**特权指令**（privileged instructions）：例如，启用和禁用中断、读写保存页表地址的寄存器等。如果用户模式下的应用程序尝试执行特权指令，CPU 不会执行该指令，而是切换到监督模式，以便监督模式代码可以终止应用程序，因为它做了不该做的事。第 1 章的图 1.1 说明了这种组织。应用程序只能执行用户模式指令（例如，加法等），据说在**用户空间**（user space）运行，而监督模式下的软件还可以执行特权指令，据说在**内核空间**（kernel space）运行。在内核空间（或监督模式）运行的软件称为**内核**（kernel）。

想要调用内核函数（例如 xv6 中的 `read` 系统调用）的应用程序必须转换到内核；应用程序不能直接调用内核函数。CPU 提供一条特殊指令，将 CPU 从用户模式切换到监督模式并进入内核指定的入口点。（RISC-V 为此提供 `ecall` 指令。）一旦 CPU 切换到监督模式，内核可以验证系统调用的参数（例如，检查传给系统调用的地址是否是应用程序内存的一部分），决定是否允许应用程序执行请求的操作（例如，检查应用程序是否允许写入指定文件），然后拒绝或执行。重要的是内核控制转换到监督模式的入口点；如果应用程序可以决定内核入口点，恶意应用程序可能，例如，在内核中跳过参数验证的点进入。

## 2.3 内核组织

一个关键设计问题是操作系统的哪部分应该在监督模式下运行。一种可能是整个操作系统驻留在内核中，因此所有系统调用的实现都在监督模式下运行。这种组织称为**单体内核**（monolithic kernel）。

在这种组织中，整个操作系统以完全硬件特权运行。这种组织很方便，因为操作系统设计者不必决定操作系统的哪部分不需要完全硬件特权。此外，操作系统的不同部分更容易协作。例如，操作系统可能有一个缓冲区缓存，可以同时被文件系统和虚拟内存系统共享。

单体组织的缺点在于操作系统不同部分之间的接口通常很复杂（我们将在本文其余部分看到），因此操作系统开发人员很容易犯错。在单体内核中，错误是致命的，因为监督模式下的错误通常会导致内核失败。如果内核失败，计算机停止工作，因此所有应用程序也会失败。计算机必须重启才能再次启动。

为了减少内核中错误的风险，操作系统设计者可以最小化在监督模式下运行的操作系统代码量，并在用户模式下执行大部分操作系统。这种内核组织称为**微内核**（microkernel）。

在微内核中，内核接口由用于启动应用程序、发送消息、访问设备硬件等的几个低级函数组成。这种组织允许内核相对简单，因为大部分操作系统驻留在用户级服务器中。

在现实世界中，单体内核和微内核都很流行。许多 Unix 内核是单体的。例如，Linux 有单体内核，尽管一些操作系统功能作为用户级服务器运行（例如窗口系统）。Linux 为操作系统密集型应用程序提供高性能，部分原因是内核的子系统可以紧密集成。

Minix、L4 和 QNX 等操作系统组织为带服务器的微内核，已在嵌入式环境中广泛部署。L4 的变体 seL4 足够小，已被验证具有内存安全和其他安全属性。

操作系统开发人员之间关于哪种组织更好的争论很多，没有 conclusive 证据。此外，这在很大程度上取决于「更好」的含义：更快的性能、更小的代码量、内核的可靠性、完整操作系统（包括用户级服务）的可靠性等。

还有一些实际考虑可能比组织问题更重要。一些操作系统有微内核，但出于性能原因在内核空间运行一些用户级服务。一些操作系统有单体内核，因为那是它们开始的方式，并且几乎没有动力转向纯微内核组织，因为新功能可能比重写现有操作系统以适应微内核设计更重要。

从本书的角度来看，微内核和单体操作系统共享许多关键思想。它们实现系统调用，使用页表，处理中断，支持进程，使用锁进行并发控制，实现文件系统等。本书侧重于这些核心思想。

xv6 与大多数 Unix 操作系统一样实现为单体内核。因此，xv6 内核接口对应于操作系统接口，内核实现完整的操作系统。由于 xv6 不提供许多服务，其内核比一些微内核小，但概念上 xv6 是单体的。

## 2.5 进程概述

xv6（与其他 Unix 操作系统一样）中的隔离单位是**进程**（process）。进程抽象防止一个进程破坏或窥探另一个进程的内存、CPU、文件描述符等。它还防止进程破坏内核本身，因此进程无法颠覆内核的隔离机制。内核必须谨慎实现进程抽象，因为有错误或恶意的应用程序可能诱使内核或硬件做坏事（例如，绕过隔离）。内核用于实现进程的机制包括用户/监督模式标志、地址空间和线程的时间片。

为了帮助执行隔离，进程抽象向程序提供它拥有自己私有机器的错觉。进程向程序提供看似私有的内存系统或**地址空间**（address space），其他进程无法读写。进程还向程序提供看似自己的 CPU 来执行程序的指令。

xv6 使用**页表**（由硬件实现）为每个进程提供自己的地址空间。RISC-V 页表将**虚拟地址**（RISC-V 指令操作的地址）转换（或「映射」）为**物理地址**（CPU 芯片发送到主存的地址）。

xv6 为每个进程维护单独的页表，定义该进程的地址空间。如**图 2.3** 所示，地址空间包括从虚拟地址零开始的进程**用户内存**。指令在前，然后是全局变量，接着是栈，最后是进程可以按需扩展的「堆」区域（用于 malloc）。有许多因素限制进程地址空间的最大大小：RISC-V 上的指针是 64 位宽；硬件在页表中查找虚拟地址时只使用低 39 位；xv6 只使用这 39 位中的 38 位。因此，最大地址是 $2^{38}-1 = 0x3fffffffff$，即 MAXVA（`kernel/riscv.h:363`）。在地址空间顶部，xv6 为 **trampoline** 保留一页，并为进程的 **trapframe** 映射一页。xv6 使用这两页转换到内核和返回；trampoline 页包含转换进出内核的代码，映射 trapframe 对于保存/恢复用户进程的状态是必要的，我们将在第 4 章解释。

xv6 内核为每个进程维护许多状态，它收集到 `struct proc`（`kernel/proc.h:85`）中。进程最重要的内核状态是其页表、内核栈和运行状态。我们将使用 `p->xxx` 表示法引用 proc 结构的元素；例如，`p->pagetable` 是指向进程页表的指针。

每个进程都有一个**执行线程**（或简称线程），执行进程的指令。线程可以挂起并稍后恢复。为了在进程之间透明地切换，内核挂起当前运行的线程并恢复另一个进程的线程。线程的大部分状态（局部变量、函数调用返回地址）存储在线程的栈上。每个进程有两个栈：用户栈和内核栈（`p->kstack`）。当进程执行用户指令时，只有其用户栈在使用，其内核栈为空。当进程进入内核（系统调用或中断）时，内核代码在进程的内核栈上执行；当进程在内核中时，其用户栈仍包含保存的数据，但不主动使用。进程的线程在主动使用其用户栈和内核栈之间交替。内核栈是独立的（并受用户代码保护），因此即使进程破坏了其用户栈，内核也能执行。

进程可以通过执行 RISC-V `ecall` 指令进行系统调用。该指令提升硬件特权级别并将程序计数器更改为内核定义的入口点。入口点的代码切换到内核栈并执行实现系统调用的内核指令。当系统调用完成时，内核切换回用户栈并通过调用 `sret` 指令返回到用户空间，该指令降低硬件特权级别并恢复在系统调用指令之后执行用户指令。进程的线程可以在内核中「阻塞」等待 I/O，并在 I/O 完成后从离开处恢复。

`p->state` 指示进程是否已分配、准备运行、正在运行、等待 I/O 或正在退出。

`p->pagetable` 保存进程的页表，采用 RISC-V 硬件期望的格式。xv6 在用户空间执行该进程时使分页硬件使用进程的 `p->pagetable`。进程的页表也作为分配给存储进程内存的物理页地址的记录。

总之，进程捆绑了两个设计思想：一个地址空间，给进程自己内存的错觉；一个线程，给进程自己 CPU 的错觉。在 xv6 中，进程由一个地址空间和一个线程组成。在真实操作系统中，进程可能有多个线程以利用多个 CPU。

## 2.6 代码：启动 xv6、第一个进程与系统调用

为了使 xv6 更具体，我们将概述内核如何启动和运行第一个进程。后续章节将更详细地描述在此概述中出现的机制。

当 RISC-V 计算机上电时，它初始化自身并运行存储在只读存储器中的引导加载程序。引导加载程序将 xv6 内核加载到内存中。然后，在机器模式下，CPU 从 `_entry`（`kernel/entry.S:7`）开始执行 xv6。RISC-V 启动时禁用分页硬件：虚拟地址直接映射到物理地址。

加载程序将 xv6 内核加载到物理地址 `0x80000000`。将其放在 `0x80000000` 而不是 `0x0` 的原因是因为地址范围 `0x0:0x80000000` 包含 I/O 设备。

`_entry` 处的指令设置栈，以便 xv6 可以运行 C 代码。xv6 在 `start.c`（`kernel/start.c:11`）中声明初始栈 `stack0` 的空间。`_entry` 处的代码将栈指针寄存器 sp 加载为 `stack0+4096`（栈顶），因为 RISC-V 上的栈向下增长。现在内核有了栈，`_entry` 调用 `start`（`kernel/start.c:21`）处的 C 代码。

函数 `start` 执行一些仅在机器模式下允许的配置，然后切换到监督模式。要进入监督模式，RISC-V 提供 `mret` 指令。该指令最常用于从监督模式到机器模式的先前调用返回。`start` 不是从这样的调用返回，而是设置事情就好像有过一样：它将寄存器 mstatus 中的先前特权模式设置为监督，通过将 main 的地址写入寄存器 mepc 将返回地址设置为 main，通过将 0 写入页表寄存器 satp 在监督模式下禁用虚拟地址转换，并将所有中断和异常委托给监督模式。

在跳转到监督模式之前，`start` 执行另一项任务：它编程时钟芯片以生成定时器中断。完成这些后，`start` 通过调用 `mret`「返回」到监督模式。这导致程序计数器更改为 `main`（`kernel/main.c:11`）。

`main`（`kernel/main.c:11`）初始化几个设备和子系统后，通过调用 `userinit`（`kernel/proc.c:233`）创建第一个进程。第一个进程执行用 RISC-V 汇编编写的小程序，该程序进行 xv6 中的第一次系统调用。`initcode.S`（`user/initcode.S:3`）将 exec 系统调用的编号 `SYS_EXEC`（`kernel/syscall.h:8`）加载到寄存器 a7，然后调用 `ecall` 重新进入内核。

内核在 `syscall`（`kernel/syscall.c:132`）中使用寄存器 a7 中的编号调用所需的系统调用。系统调用表（`kernel/syscall.c:107`）将 SYS_EXEC 映射到 `sys_exec`，内核调用它。如第 1 章所见，`exec` 用新程序（本例中为 `/init`）替换当前进程的内存和寄存器。

一旦内核完成 `exec`，它返回到 `/init` 进程中的用户空间。`init`（`user/init.c:15`）在需要时创建新的控制台设备文件，然后将其作为文件描述符 0、1 和 2 打开。然后它在控制台上启动 Shell。系统就绪。

## 2.7 安全模型

你可能想知道操作系统如何处理有错误或恶意的代码。因为应对恶意比处理意外错误严格更难，所以将此主题视为与安全相关是合理的。以下是操作系统设计中典型安全假设和目标的高级视图。

操作系统必须假设进程的用户级代码会尽其所能破坏内核或其他进程。用户代码可能尝试解引用其允许地址空间之外的指针；它可能尝试执行任何 RISC-V 指令，甚至那些不打算用于用户代码的指令；它可能尝试读写任何 RISC-V 控制寄存器；它可能尝试直接访问设备硬件；它可能向系统调用传递巧妙的值以试图诱使内核崩溃或做坏事。内核的目标是限制每个用户进程，使其只能读写/执行自己的用户内存，使用 32 个通用 RISC-V 寄存器，并以系统调用允许的方式影响内核和其他进程。内核必须阻止任何其他操作。这通常是内核设计中的绝对要求。

对内核自身代码的期望完全不同。内核代码假定由善意且谨慎的程序员编写。内核代码期望无错误，当然不包含任何恶意。这一假设影响我们如何分析内核代码。例如，有许多内部内核函数（例如自旋锁）如果内核代码错误使用会导致严重问题。在检查任何特定内核代码时，我们想说服自己它行为正确。然而，我们假设内核代码总体上是正确编写的，并遵循有关使用内核自身函数和数据结构的所有规则。在硬件层面，RISC-V CPU、RAM、磁盘等假定按文档所述运行，没有硬件错误。

当然，在现实生活中事情并非如此简单。很难防止聪明的用户代码通过消耗内核保护的资源（磁盘空间、CPU 时间、进程表槽等）使系统无法使用（或导致崩溃）。编写无错误代码或设计无错误硬件通常是不可能的；如果恶意用户代码的编写者知道内核或硬件错误，他们会利用它们。即使在成熟、广泛使用的内核（如 Linux）中，人们也不断发现新的漏洞。值得在内核中设计针对可能错误的保护措施：断言、类型检查、栈保护页等。最后，用户代码和内核代码之间的区别有时是模糊的：一些特权用户级进程可能提供基本服务并有效地成为操作系统的一部分，在某些操作系统中，特权用户代码可以插入新代码到内核（如 Linux 的可加载内核模块）。

## 2.8 现实世界

大多数操作系统都采用了进程概念，大多数进程看起来与 xv6 的相似。然而，现代操作系统在进程内支持多个线程，以允许单个进程利用多个 CPU。在进程内支持多个线程涉及相当多的机制，xv6 没有，包括潜在的接口更改（例如 Linux 的 `clone`，`fork` 的变体）以控制进程线程共享哪些方面。
