# 第4章 陷阱与系统调用

有三种事件会导致 CPU 搁置指令的普通执行并强制将控制权转移到处理该事件的特殊代码。一种情况是**系统调用**（system call），当用户程序执行 `ecall` 指令请求内核为其做某事时。另一种情况是**异常**（exception）：一条指令（例如除以零或非法虚拟地址）做了一些非法操作。第三种情况是**设备中断**（device interrupt），当设备（例如磁盘硬件）需要 CPU 的注意时。

本书使用**陷阱**（trap）作为这些情况的通用术语。通常，陷阱发生时正在执行的代码之后会恢复执行，不需要知道发生了什么。也就是说，我们通常希望陷阱是透明的；这对中断尤其重要，因为被中断的代码通常没有预料到会发生陷阱。

xv6 的陷阱处理分为四个阶段：RISC-V CPU 的硬件操作、为内核 C 代码准备的汇编「向量」、决定如何处理陷阱的 C 陷阱处理程序，以及系统调用或设备驱动服务例程。虽然三种陷阱类型之间的共性表明可以使用单一代码路径，但事实证明，为三种不同的情况（用户陷阱、内核陷阱和定时器中断）分别设置不同的汇编向量和 C 陷阱处理程序是最清晰的。

## 4.1 RISC-V 陷阱机制

每个 RISC-V CPU 都有一组控制寄存器，内核写入它们以告诉 CPU 如何处理陷阱，内核可以读取它们以了解已发生的陷阱。RISC-V 文档包含完整说明。`riscv.h`（`kernel/riscv.h:1`）包含 xv6 使用的定义。以下是最重要寄存器的概述：

- **stvec**：内核在此写入陷阱处理程序的地址；RISC-V 跳转到此处处理陷阱。
- **sepc**：陷阱发生时，RISC-V 在此保存程序计数器（因为 `pc` 会被 `stvec` 覆盖）。`sret`（从陷阱返回）指令将 `sepc` 复制到 `pc`。内核可以写入 `sepc` 以控制 `sret` 的去向。
- **scause**：RISC-V 在此放置一个描述陷阱原因的数字。
- **sstatus**：`sstatus` 中的 SIE 位启用/禁用设备中断。内核清除 SIE 以在陷阱处理期间禁用设备中断。SPP 位指示陷阱是来自用户模式还是监督模式，并控制 `sret` 返回的模式。

上述寄存器与监督模式下的陷阱相关，xv6 处理所有陷阱。在机器模式下处理陷阱的类似寄存器也存在；xv6 仅在引导序列的早期使用它们。

当需要强制陷阱时，RISC-V 硬件对所有陷阱类型执行以下操作：

1. 如果陷阱是设备中断，且 sstatus 的 SIE 位被清除，则不执行以下任何操作。
2. 清除 SIE 以禁用中断。
3. 将 pc 复制到 sepc。
4. 将当前模式（用户或监督）保存到 sstatus 的 SPP 位。
5. 将 scause 设置为描述陷阱原因的数字。
6. 将模式设置为监督模式。
7. 将 stvec 复制到 pc。
8. 在 pc 处开始执行。

注意，CPU 不会切换到内核页表，不会切换到内核中的栈，也不会保存除 pc 之外的任何寄存器。内核软件必须执行这些任务。CPU 在陷阱期间做最少工作的一个原因是为软件提供灵活性；例如，某些操作系统在某些情况下省略页表切换以增加陷阱处理速度。

## 4.2 用户空间的陷阱

xv6 对用户空间陷阱的设置发生在 `trampoline.S` 和 `trap.c` 中。

当用户空间发生陷阱时，`satp` 指向用户页表，但 `stvec` 指向内核陷阱处理程序。由于用户页表没有映射内核，`stvec` 不能直接指向内核代码。相反，xv6 使用包含 `uservec` 的 trampoline 页，该页在用户页表和内核页表中都有映射。

当 `uservec` 开始时，所有 32 个通用寄存器都包含被中断代码的值。但 `uservec` 需要能够修改这些寄存器以设置 `satp` 并生成保存寄存器的地址。RISC-V 以 `sscratch` 寄存器的形式提供帮助。`uservec` 开头的 `csrw` 指令将 a0 保存到 sscratch。现在 `uservec` 有一个寄存器（a0）可以使用。

`uservec` 的下一个任务是将 32 个用户寄存器保存起来。内核为每个进程分配一页内存用于 trapframe 结构，该结构（除其他外）有空间保存 32 个用户寄存器。`uservec` 从 `sscratch` 中检索 trapframe 的地址（内核在进入用户空间之前将 trapframe 的地址放在那里），然后将所有用户寄存器保存到 trapframe 中。

然后 `uservec` 从 trapframe 中检索内核栈指针、当前 CPU 的 hartid、内核页表的地址，并切换到内核页表。它调用 `usertrap` 进行 C 陷阱处理。

`usertrap` 的工作是确定陷阱的原因、处理它并返回。它首先更改 `stvec`，以便内核中的陷阱由 `kernelvec` 处理，而不是 `uservec`。它保存 `sepc`（保存的用户程序计数器），因为 `usertrap` 可能调用 `yield` 切换到另一个进程的内核线程，该进程可能会覆盖 `sepc`。如果陷阱是系统调用，`syscall` 处理它；如果是设备中断，`devintr` 处理它；否则是异常，内核杀死故障进程。系统调用路径在保存的用户 pc 上加 4，因为 RISC-V 在 `ecall` 之后留下指向 `ecall` 指令的 pc。在退出时，`usertrap` 检查进程是否被杀死或是否应该让出 CPU（如果定时器中断触发了）。

返回用户空间的第一步是调用 `usertrapret`。此函数设置 RISC-V 控制寄存器，为来自用户空间的未来陷阱做准备。这涉及更改 `stvec` 以引用 `uservec`，准备 `uservec` 依赖的 trapframe 字段，并将 `sepc` 设置为先前保存的用户程序计数器。最后，`usertrapret` 在用户页表和内核页表中映射的 trampoline 页上调用 `userret`，因为 `userret` 中的指令会切换页表。

`userret` 将 trapframe 的地址从 a0 复制到 sscratch，切换到用户页表，从 trapframe 恢复用户寄存器，交换 a0 和 sscratch 以恢复用户 a0 并保存 trapframe 的地址，并执行 `sret` 返回到用户空间。

## 4.3 代码：调用系统调用

第 2 章以 `initcode.S` 调用 exec 系统调用结束（`user/initcode.S:11`）。让我们看看用户调用如何到达内核中 exec 系统调用的实现。

`initcode.S` 将 exec 的参数放在寄存器 a0 和 a1 中，并将系统调用编号放在 a7 中。系统调用编号与 `syscalls` 数组中的条目匹配，该数组是函数指针表。`syscall` 从 trapframe 中保存的 a7 检索系统调用编号，并调用相应的系统调用处理函数。如果系统调用编号无效，`syscall` 打印错误并返回 -1。

## 4.4 代码：系统调用参数

内核中的系统调用处理函数需要找到用户传递的参数。由于用户代码调用包装函数，参数根据 RISC-V C 调用约定放在寄存器中。内核陷阱代码将用户寄存器保存到当前进程的 trapframe 中，内核代码可以在那里找到它们。函数 `argint`、`argaddr` 和 `argfd` 从 trapframe 检索第 n 个系统调用参数，作为整数、指针或文件描述符。它们都调用 `argraw` 来检索适当保存的用户寄存器。

某些系统调用传递指针作为参数；内核必须使用这些指针来读写用户内存。例如，`exec` 系统调用向内核传递指向用户空间中字符串参数的指针数组。这些指针带来两个挑战。首先，用户程序可能有错误或恶意，可能传递无效指针或旨在欺骗内核访问内核内存而非用户内存的指针。其次，xv6 使用直接映射的内核页表；用户虚拟地址与内核虚拟地址不同。内核不能直接使用用户指针；它必须首先将用户指针指向的数据复制到内核内存。

内核使用 `copyinstr`、`copyin` 和 `copyout` 等函数安全地将数据传入和传出用户提供的地址。`copyin` 从用户虚拟地址复制数据到内核虚拟地址。它通过将 `walk` 转换为用户页表以查找数据的物理地址（pa），然后将数据从 pa 复制到内核的 `dst` 来实现。用户页表可以映射任意物理内存，因此内核必须验证用户提供的虚拟地址确实是用户地址空间的一部分。否则，恶意用户程序可以让内核读取其他内存。类似的函数 `copyout` 将数据从内核复制到用户提供的地址。

## 4.5 内核空间的陷阱

xv6 根据执行的是用户代码还是内核代码，对 CPU 陷阱寄存器进行不同的配置。当内核在 CPU 上执行时，内核将 `stvec` 指向 `kernelvec` 的汇编代码。由于 xv6 已经在内核中，`kernelvec` 可以依赖 `satp` 已经设置为内核页表，并且栈指针指向有效的内核栈。`kernelvec` 将所有寄存器推送到内核栈上，这样它们可以被中断的代码恢复。`kernelvec` 在保存寄存器后跳转到 `kerneltrap`。

`kerneltrap` 处理两种类型的陷阱：设备中断和异常。它调用 `devintr` 来检查并处理前者。如果陷阱不是设备中断，则它是异常；内核中的异常是致命错误；内核调用 `panic` 并停止执行。

如果 `kerneltrap` 因定时器中断而调用，且进程的内核线程正在运行（而非调度器线程），`kerneltrap` 调用 `yield` 给其他进程机会运行。在某个时刻，其中一个进程会返回用户空间，`kerneltrap` 会返回到该进程。

当 `kerneltrap` 完成时，它需要返回到被中断的代码。它从栈上弹出保存的寄存器并执行 `sret`，这会复制 `sepc` 到 `pc` 并恢复被中断的内核代码。

值得思考的是，如果 `kerneltrap` 因定时器中断而调用 `yield`，陷阱返回是如何发生的。

## 4.6 页错误异常

xv6 对异常的响应相当无聊：如果用户空间发生异常，内核会杀死故障进程。如果内核发生异常，内核会 panic。许多操作系统对页错误（page-fault exceptions）的响应更有趣。

例如，许多内核使用页错误实现**写时复制**（copy-on-write，COW）fork。要解释写时复制 fork，考虑第 3 章描述的 xv6 的 `fork`。`fork` 导致子进程的初始内存内容与 fork 时父进程的相同。xv6 使用 `uvmcopy`（`kernel/vm.c:306`）实现 fork，它为子进程分配物理内存并复制父进程的内存。如果子进程和父进程可以共享父进程的物理页，则写时复制可以省略复制。

**惰性分配**（lazy allocation）是另一个例子。`sbrk` 系统调用分配进程的堆内存。内核为大型分配分配和零化内存是昂贵的。此外，应用程序通常请求的内存比它们需要的多。因此，使用惰性分配，`sbrk` 不分配物理内存，只是记住分配了哪些用户地址。当进程第一次使用任一给定页时，它会产生页错误；然后内核分配该页并映射它。许多操作系统对应用程序透明地使用惰性分配。

由于应用程序通常请求的内存比它们需要的多，惰性分配是一种胜利：对于应用程序从未使用的页，内核根本不需要做任何工作。此外，如果应用程序请求大量增长地址空间，则没有惰性分配的 `sbrk` 是昂贵的：如果应用程序请求千兆字节的内存，内核必须分配并清零许多页。

## 4.7 现实世界

由于系统调用接口的稳定性，陷阱机制已经成功使用了数十年。许多操作系统使用与 xv6 相同的思路。许多内核使用页错误实现写时复制 fork、惰性分配、按需分页、内存映射文件等。写时复制 fork 和惰性分配的结合可以带来更高效的 fork，因为不需要复制或分配内存。惰性分配与按需分页的结合可以带来更高效的 `sbrk`。惰性分配与按需分页的结合可以带来更高效的 `exec`，因为不需要将整个二进制加载到内存中。许多操作系统使用页错误实现内存映射文件。现代操作系统对陷阱的响应通常更有趣。
