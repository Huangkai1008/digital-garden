# 第1章 操作系统接口

操作系统的职责是在多个程序之间共享计算机，并提供比硬件单独支持更有用的服务集合。操作系统管理并抽象底层硬件，例如，文字处理器无需关心正在使用哪种磁盘硬件。操作系统在多个程序之间共享硬件，使它们同时（或看似同时）运行。最后，操作系统为程序提供受控的交互方式，使它们能够共享数据或协同工作。

操作系统通过接口向用户程序提供服务。设计良好的接口并非易事。一方面，我们希望接口简单且狭窄，因为这样更容易正确实现。另一方面，我们可能倾向于向应用程序提供许多复杂功能。解决这一张力的诀窍是设计依赖少数机制、可组合以提供广泛通用性的接口。

本书以单一操作系统为具体示例来说明操作系统概念。该操作系统 xv6 提供 Ken Thompson 和 Dennis Ritchie 的 Unix 操作系统引入的基本接口，并模仿 Unix 的内部设计。Unix 提供机制组合良好的狭窄接口，具有令人惊讶的通用性。该接口非常成功，现代操作系统——BSD、Linux、macOS、Solaris，甚至在较小程度上 Microsoft Windows——都具有类 Unix 接口。理解 xv6 是理解这些系统及许多其他系统的良好开端。

如 **图 1.1** 所示，xv6 采用传统的**内核**（kernel）形式，即向运行中的程序提供服务的特殊程序。每个运行中的程序称为**进程**（process），其内存包含指令、数据和栈。指令实现程序的计算。数据是计算所作用的变量。栈组织程序的过程调用。一台计算机通常有许多进程，但只有一个内核。

当进程需要调用内核服务时，它会调用**系统调用**（system call），即操作系统接口中的调用之一。系统调用进入内核；内核执行服务并返回。因此，进程在用户空间和内核空间之间交替执行。

如后续章节所述，内核使用 CPU 提供的硬件保护机制，确保在用户空间执行的每个进程只能访问自己的内存。内核以实现这些保护所需的硬件特权执行；用户程序在没有这些特权的情况下执行。当用户程序调用系统调用时，硬件会提升特权级别并开始执行内核中预先安排好的函数。

内核提供的系统调用集合是用户程序所见的接口。xv6 内核提供 Unix 内核传统提供的服务和系统调用的子集。**图 1.2** 列出了 xv6 的所有系统调用。

本章其余部分概述 xv6 的服务——进程、内存、文件描述符、管道和文件系统——并通过代码片段和 shell（Unix 的命令行用户界面）如何使用它们的讨论来说明。Shell 对系统调用的使用说明了它们是如何精心设计的。

Shell 是读取用户命令并执行它们的普通程序。Shell 是用户程序而非内核的一部分，这说明了系统调用接口的力量：Shell 没有什么特别的。这也意味着 Shell 易于替换；因此，现代 Unix 系统有多种 Shell 可供选择，每种都有各自的用户界面和脚本功能。xv6 Shell 是 Unix Bourne Shell 精髓的简单实现。其实现位于 `user/sh.c:1`。

## 1.1 进程与内存

xv6 进程由用户空间内存（指令、数据和栈）以及内核私有的每进程状态组成。xv6 对进程进行**分时**（time-share）：它在等待执行的进程集合之间透明地切换可用 CPU。当进程未执行时，xv6 保存进程的 CPU 寄存器，在下次运行该进程时恢复。内核为每个进程关联一个**进程标识符**（process identifier，PID）。

进程可以使用 `fork` 系统调用创建新进程。`fork` 为新进程提供调用进程内存的精确副本，包括指令和数据。`fork` 在原始进程和新进程中都会返回。在原始进程中，`fork` 返回新进程的 PID。在新进程中，`fork` 返回零。原始进程和新进程通常称为**父进程**（parent）和**子进程**（child）。

| 系统调用 | 描述 |
|---------|------|
| `int fork()` | 创建进程，返回子进程 PID |
| `int exit(int status)` | 终止当前进程；状态报告给 `wait()`。不返回 |
| `int wait(int *status)` | 等待子进程退出；退出状态在 `*status`；返回子进程 PID |
| `int kill(int pid)` | 终止进程 PID。成功返回 0，错误返回 -1 |
| `int getpid()` | 返回当前进程 PID |
| `int sleep(int n)` | 暂停 n 个时钟周期 |
| `int exec(char *file, char *argv[])` | 加载文件并以参数执行；仅错误时返回 |
| `char *sbrk(int n)` | 将进程内存增长 n 字节。返回新内存起始地址 |
| `int open(char *file, int flags)` | 打开文件；flags 表示读/写；返回 fd（文件描述符）|
| `int write(int fd, char *buf, int n)` | 从 buf 向文件描述符 fd 写入 n 字节；返回 n |
| `int read(int fd, char *buf, int n)` | 将 n 字节读入 buf；返回读取数；文件结束时返回 0 |
| `int close(int fd)` | 释放打开的文件 fd |
| `int dup(int fd)` | 返回引用与 fd 相同文件的新文件描述符 |
| `int pipe(int p[])` | 创建管道，将读/写文件描述符放入 p[0] 和 p[1] |
| `int chdir(char *dir)` | 更改当前目录 |
| `int mkdir(char *dir)` | 创建新目录 |
| `int mknod(char *file, int, int)` | 创建设备文件 |
| `int fstat(int fd, struct stat *st)` | 将打开文件信息放入 `*st` |

| `int link(char *file1, char *file2)` | 为 file1 创建另一名称 (file2) |
| `int unlink(char *file)` | 删除文件 |

**图 1.2**：xv6 系统调用。如无特别说明，这些调用成功返回 0，错误返回 -1。

例如，考虑以下用 C 语言编写的程序片段：

```c
int pid = fork();
if(pid > 0){
  printf("parent: child=%d\n", pid);
  pid = wait((int *) 0);
  printf("child %d is done\n", pid);
} else if(pid == 0){
  printf("child: exiting\n");
  exit(0);
} else {
  printf("fork error\n");
}
```

`exit` 系统调用导致调用进程停止执行并释放内存和打开的文件等资源。`exit` 接受整数状态参数，通常 0 表示成功，1 表示失败。`wait` 系统调用返回当前进程已退出（或被杀死）的子进程的 PID，并将子进程的退出状态复制到传给 `wait` 的地址；如果调用者的子进程都未退出，`wait` 会等待其中一个退出。如果调用者没有子进程，`wait` 立即返回 -1。如果父进程不关心子进程的退出状态，可以向 `wait` 传递 0 地址。

在该示例中，输出行

```
parent: child=1234
child: exiting
```

可能以任意顺序出现（甚至交错），取决于父进程或子进程谁先到达 `printf` 调用。子进程退出后，父进程的 `wait` 返回，导致父进程打印

```
parent: child 1234 is done
```

尽管子进程最初与父进程具有相同的内存内容，但父进程和子进程使用独立的内存和寄存器执行：在一个进程中修改变量不会影响另一个。例如，当 `wait` 的返回值存储到父进程的 `pid` 时，不会改变子进程中的变量 `pid`。子进程中 `pid` 的值仍为零。

`exec` 系统调用用从文件系统存储的文件加载的新内存映像替换调用进程的内存。该文件必须具有特定格式，指定文件的哪部分是指令、哪部分是数据、从哪条指令开始等。xv6 使用 ELF 格式，第 3 章将更详细讨论。通常该文件是编译程序源代码的结果。当 `exec` 成功时，它不会返回到调用程序；相反，从文件加载的指令在 ELF 头中声明的入口点开始执行。`exec` 接受两个参数：包含可执行文件的文件名和字符串参数数组。例如：

```c
char *argv[3];
argv[0] = "echo";
argv[1] = "hello";
argv[2] = 0;
exec("/bin/echo", argv);
printf("exec error\n");
```

该片段用运行参数列表 `echo hello` 的程序 `/bin/echo` 的实例替换调用程序。大多数程序忽略参数数组的第一个元素，按惯例它是程序名。

xv6 Shell 使用上述调用来代表用户运行程序。Shell 的主要结构很简单；参见 `main`（`user/sh.c:146`）。主循环使用 `getcmd` 从用户读取一行输入。然后调用 `fork`，创建 Shell 进程的副本。父进程调用 `wait`，而子进程运行命令。例如，如果用户向 Shell 输入了 "echo hello"，`runcmd` 将以 "echo hello" 作为参数被调用。`runcmd`（`user/sh.c:55`）运行实际命令。对于 "echo hello"，它将调用 `exec`（`user/sh.c:79`）。如果 `exec` 成功，子进程将执行来自 echo 的指令而非 runcmd。在某个时刻 echo 将调用 `exit`，导致父进程在 `main`（`user/sh.c:146`）中从 `wait` 返回。

你可能会好奇为什么 `fork` 和 `exec` 没有合并为单个调用；我们稍后将看到 Shell 在 I/O 重定向实现中利用这种分离。为避免创建重复进程然后立即替换（用 `exec`）的浪费，操作系统内核使用虚拟内存技术（如写时复制，见第 4.6 节）针对此用例优化 `fork` 的实现。

xv6 隐式分配大多数用户空间内存：`fork` 分配子进程复制父进程内存所需的内存，`exec` 分配足够容纳可执行文件的内存。运行时需要更多内存的进程（可能用于 `malloc`）可以调用 `sbrk(n)` 将其数据内存增长 n 字节；`sbrk` 返回新内存的位置。

## 1.2 I/O 与文件描述符

**文件描述符**（file descriptor）是表示进程可读写的内核管理对象的小整数。进程可以通过打开文件、目录或设备，或创建管道，或复制现有描述符来获得文件描述符。为简化起见，我们通常将文件描述符引用的对象称为「文件」；文件描述符接口抽象了文件、管道和设备之间的差异，使它们看起来都像字节流。我们将输入和输出称为 I/O。

在内部，xv6 内核将文件描述符用作每进程表的索引，因此每个进程都有从零开始的私有文件描述符空间。按惯例，进程从文件描述符 0（标准输入）读取，向文件描述符 1（标准输出）写入输出，向文件描述符 2（标准错误）写入错误消息。我们将看到，Shell 利用该惯例实现 I/O 重定向和管道。Shell 确保始终打开三个文件描述符（`user/sh.c:152`），默认情况下它们是控制台的文件描述符。

`read` 和 `write` 系统调用从文件描述符命名的打开文件读取和写入字节。调用 `read(fd, buf, n)` 从文件描述符 fd 读取最多 n 字节，将它们复制到 buf，并返回读取的字节数。引用文件的每个文件描述符都有与之关联的偏移量。`read` 从当前文件偏移量读取数据，然后将该偏移量前进读取的字节数：后续 `read` 将返回第一次 `read` 返回的字节之后的字节。当没有更多字节可读时，`read` 返回零表示文件结束。

调用 `write(fd, buf, n)` 从 buf 向文件描述符 fd 写入 n 字节并返回写入的字节数。只有在发生错误时才会写入少于 n 字节。与 `read` 一样，`write` 在当前文件偏移量处写入数据，然后将该偏移量前进写入的字节数：每次 `write` 从前一次结束处继续。

以下程序片段（构成程序 cat 的精髓）将数据从其标准输入复制到标准输出。如果发生错误，它向标准错误写入消息。

```c
char buf[512];
int n;
for(;;){
  n = read(0, buf, sizeof buf);
  if(n == 0)
    break;
  if(n < 0){
    fprintf(2, "read error\n");
    exit(1);
  }
  if(write(1, buf, n) != n){
    fprintf(2, "write error\n");
    exit(1);
  }
}
```

该代码片段中需要注意的重要一点是，cat 不知道它是从文件、控制台还是管道读取。同样，cat 不知道它是向控制台、文件还是其他什么打印。文件描述符的使用以及文件描述符 0 是输入、文件描述符 1 是输出的惯例允许 cat 的简单实现。

`close` 系统调用释放文件描述符，使其可供未来的 `open`、`pipe` 或 `dup` 系统调用重用（见下文）。新分配的文件描述符始终是当前进程最低编号的未使用描述符。

文件描述符与 `fork` 的交互使 I/O 重定向易于实现。`fork` 复制父进程的文件描述符表及其内存，因此子进程以与父进程完全相同的打开文件开始。`exec` 系统调用替换调用进程的内存但保留其文件表。此行为允许 Shell 通过 fork、在子进程中重新打开选定的文件描述符，然后调用 `exec` 运行新程序来实现 I/O 重定向。以下是 Shell 为命令 `cat < input.txt` 运行的代码的简化版本：

```c
char *argv[2];
argv[0] = "cat";
argv[1] = 0;
if(fork() == 0) {
  close(0);
  open("input.txt", O_RDONLY);
  exec("cat", argv);
}
```

子进程关闭文件描述符 0 后，`open` 保证为新打开的 input.txt 使用该文件描述符：0 将是最小的可用文件描述符。然后 cat 以文件描述符 0（标准输入）引用 input.txt 执行。父进程的文件描述符不会因此序列而改变，因为它只修改子进程的描述符。

xv6 Shell 中的 I/O 重定向代码正是以这种方式工作的（`user/sh.c:83`）。回想一下，在代码的这一点，Shell 已经 fork 了子 Shell，`runcmd` 将调用 `exec` 加载新程序。

`open` 的第二个参数由一组以位表示的标志组成，控制 `open` 的行为。可能的值在文件控制（fcntl）头文件（`kernel/fcntl.h:1-5`）中定义：`O_RDONLY`、`O_WRONLY`、`O_RDWR`、`O_CREATE` 和 `O_TRUNC`，它们指示 `open` 以只读、只写或读写方式打开文件，在文件不存在时创建，以及将文件截断为零长度。

现在应该清楚为什么 `fork` 和 `exec` 分开调用是有帮助的：在两者之间，Shell 有机会重定向子进程的 I/O，而不干扰主 Shell 的 I/O 设置。人们可以想象一个假设的合并 `forkexec` 系统调用，但用这种调用做 I/O 重定向的选项似乎很笨拙。Shell 可以在调用 `forkexec` 之前修改自己的 I/O 设置（然后撤销这些修改）；或者 `forkexec` 可以将 I/O 重定向指令作为参数；或者（最不吸引人）每个像 cat 这样的程序都可以被教会自己做 I/O 重定向。

尽管 `fork` 复制文件描述符表，但每个底层文件偏移量在父进程和子进程之间共享。考虑此示例：

```c
if(fork() == 0) {
  write(1, "hello ", 6);
  exit(0);
} else {
  wait(0);
  write(1, "world\n", 6);
}
```

在此片段结束时，附加到文件描述符 1 的文件将包含数据 `hello world`。父进程的 `write`（由于 `wait`，仅在子进程完成后运行）从子进程的 `write` 结束处继续。此行为有助于从 Shell 命令序列产生顺序输出，如 `(echo hello; echo world) >output.txt`。

`dup` 系统调用复制现有文件描述符，返回引用相同底层 I/O 对象的新描述符。两个文件描述符共享偏移量，就像 `fork` 复制的文件描述符一样。这是另一种将 hello world 写入文件的方式：

```c
fd = dup(1);
write(1, "hello ", 6);
write(fd, "world\n", 6);
```

如果两个文件描述符是通过 `fork` 和 `dup` 调用序列从同一原始文件描述符派生的，则它们共享偏移量。否则，即使它们来自对同一文件的 `open` 调用，文件描述符也不共享偏移量。`dup` 允许 Shell 实现诸如 `ls existing-file non-existing-file > tmp1 2>&1` 这样的命令。`2>&1` 告诉 Shell 给命令一个文件描述符 2，它是描述符 1 的副本。现有文件的名称和不存在的文件的错误消息都将显示在文件 tmp1 中。xv6 Shell 不支持错误文件描述符的 I/O 重定向，但现在你知道如何实现它了。

文件描述符是强大的抽象，因为它们隐藏了所连接对象的细节：向文件描述符 1 写入的进程可能正在向文件、控制台等设备或管道写入。

## 1.3 管道

**管道**（pipe）是作为一对文件描述符（一个用于读取，一个用于写入）暴露给进程的小型内核缓冲区。向管道一端写入数据使该数据可从管道另一端读取。管道为进程提供通信方式。

以下示例代码运行程序 wc，其标准输入连接到管道的读取端。

```c
int p[2];
char *argv[2];
argv[0] = "wc";
argv[1] = 0;
pipe(p);
if(fork() == 0) {
  close(0);
  dup(p[0]);
  close(p[0]);
  close(p[1]);
  exec("/bin/wc", argv);
} else {
  close(p[0]);
  write(p[1], "hello world\n", 12);
  close(p[1]);
}
```

程序调用 `pipe`，创建新管道并在数组 p 中记录读写文件描述符。`fork` 后，父进程和子进程都有引用管道的文件描述符。子进程调用 `close` 和 `dup` 使文件描述符零引用管道的读取端，关闭 p 中的文件描述符，并调用 `exec` 运行 wc。当 wc 从其标准输入读取时，它从管道读取。父进程关闭管道的读取端，向管道写入，然后关闭写入端。

如果没有可用数据，对管道的 `read` 会等待数据被写入或引用写入端的所有文件描述符被关闭；在后一种情况下，`read` 将返回 0，就像到达数据文件末尾一样。`read` 会阻塞直到不可能有新数据到达，这是子进程在执行上述 wc 之前关闭管道写入端的重要原因之一：如果 wc 的文件描述符之一引用管道的写入端，wc 将永远看不到文件结束。

xv6 Shell 以与上述代码类似的方式实现管道，如 `grep fork sh.c | wc -l`（`user/sh.c:101`）。子进程创建管道连接管道的左端和右端。然后它为管道左端调用 `fork` 和 `runcmd`，为右端调用 `fork` 和 `runcmd`，并等待两者完成。管道的右端可能是本身包含管道的命令（例如 `a | b | c`），它本身会 fork 两个新的子进程（一个用于 b，一个用于 c）。因此，Shell 可能创建进程树。该树的叶子是命令，内部节点是等待左右子进程完成的进程。

管道似乎不比临时文件更强大：管道

```
echo hello world | wc
```

可以不用管道实现为

```
echo hello world >/tmp/xyz; wc </tmp/xyz
```

管道在此情况下至少有三个优于临时文件的优点。首先，管道自动清理自己；使用文件重定向时，Shell 必须小心在完成后删除 `/tmp/xyz`。其次，管道可以传递任意长的数据流，而文件重定向需要磁盘上有足够的可用空间来存储所有数据。第三，管道允许管道阶段的并行执行，而文件方法要求第一个程序在第二个开始之前完成。

## 1.4 文件系统

xv6 文件系统提供**数据文件**（包含未解释的字节数组）和**目录**（包含对数据文件和其他目录的命名引用）。目录形成树，从称为**根**（root）的特殊目录开始。像 `/a/b/c` 这样的路径引用根目录 `/` 中名为 a 的目录内名为 b 的目录内名为 c 的文件或目录。不以 `/` 开头的路径相对于调用进程的**当前目录**（current directory）求值，可以用 `chdir` 系统调用更改。以下两个代码片段打开同一文件（假设所涉及的所有目录都存在）：

```c
chdir("/a");
chdir("b");
open("c", O_RDONLY);
open("/a/b/c", O_RDONLY);
```

第一个片段将进程的当前目录更改为 `/a/b`；第二个既不引用也不更改进程的当前目录。

有用于创建新文件和目录的系统调用：`mkdir` 创建新目录，带 `O_CREATE` 标志的 `open` 创建新数据文件，`mknod` 创建新设备文件。此示例说明全部三种：

```c
mkdir("/dir");
fd = open("/dir/file", O_CREATE|O_WRONLY);
close(fd);
mknod("/console", 1, 1);
```

`mknod` 创建引用设备的特殊文件。与设备文件关联的是主设备号和次设备号（`mknod` 的两个参数），它们唯一标识内核设备。当进程稍后打开设备文件时，内核将 `read` 和 `write` 系统调用重定向到内核设备实现，而不是传递给文件系统。

文件名与文件本身不同；同一底层文件（称为 **inode**）可以有多个名称（称为**链接**，links）。每个链接由目录中的条目组成；该条目包含文件名和对 inode 的引用。inode 保存有关文件的元数据，包括其类型（文件或目录或设备）、长度、文件内容在磁盘上的位置以及指向文件的链接数。

`fstat` 系统调用从文件描述符引用的 inode 检索信息。它填充 `struct stat`，在 `stat.h`（`kernel/stat.h`）中定义为：

```c
#define T_DIR  1  // Directory
#define T_FILE 2  // File
#define T_DEVICE 3  // Device
struct stat {
  int dev;     // File system's disk device
  uint ino;    // Inode number
  short type;  // Type of file
  short nlink; // Number of links to file
  uint64 size; // Size of file in bytes
};
```

`link` 系统调用创建引用与现有文件相同 inode 的另一个文件系统名称。此片段创建名为 a 和 b 的新文件：

```c
open("a", O_CREATE|O_WRONLY);
link("a", "b");
```

读取或写入 a 与读取或写入 b 相同。每个 inode 由唯一的 inode 号标识。在上述代码序列之后，可以通过检查 `fstat` 的结果确定 a 和 b 引用相同的底层内容：两者将返回相同的 inode 号（ino），nlink 计数将设置为 2。

`unlink` 系统调用从文件系统中删除名称。文件的 inode 和保存其内容的磁盘空间只有在文件的链接计数为零且没有文件描述符引用它时才会被释放。因此，将

```c
unlink("a");
```

添加到上一个代码序列会使 inode 和文件内容仍可作为 b 访问。此外，

```c
fd = open("/tmp/xyz", O_CREATE|O_RDWR);
unlink("/tmp/xyz");
```

是创建无名称临时 inode 的惯用方式，当进程关闭 fd 或退出时将清理。

Unix 提供可从 Shell 作为用户级程序调用的文件实用程序，例如 `mkdir`、`ln` 和 `rm`。这种设计允许任何人通过添加新的用户级程序来扩展命令行界面。事后看来，这个计划似乎显而易见，但在 Unix 时代设计的其他系统通常将此类命令内置到 Shell（并将 Shell 内置到内核）中。

一个例外是 `cd`，它内置在 Shell 中（`user/sh.c:161`）。`cd` 必须更改 Shell 本身的当前工作目录。如果 `cd` 作为常规命令运行，Shell 会 fork 子进程，子进程将运行 `cd`，`cd` 将更改子进程的工作目录。父进程（即 Shell 的）工作目录不会改变。

## 1.5 现实世界

Unix 的「标准」文件描述符、管道和对其操作的便捷 Shell 语法的组合是编写通用可重用程序的重大进步。这一想法催生了「软件工具」文化，这是 Unix 力量和流行的重要原因，Shell 是第一个所谓的「脚本语言」。Unix 系统调用接口今天在 BSD、Linux 和 macOS 等系统中仍然存在。

Unix 系统调用接口已通过可移植操作系统接口（POSIX）标准标准化。xv6 不符合 POSIX：它缺少许多系统调用（包括 `lseek` 等基本调用），并且它提供的许多系统调用与标准不同。我们对 xv6 的主要目标是简单性和清晰性，同时提供简单的类 UNIX 系统调用接口。一些人已用更多系统调用和简单 C 库扩展 xv6 以运行基本 Unix 程序。然而，现代内核提供的系统调用和内核服务种类比 xv6 多得多。例如，它们支持网络、窗口系统、用户级线程、许多设备的驱动程序等。现代内核持续快速演进，提供许多超出 POSIX 的功能。

Unix 用单一的文件名和文件描述符接口集统一了对多种资源（文件、目录和设备）的访问。这一想法可以扩展到更多种类的资源；Plan 9 是一个很好的例子，它将「资源即文件」概念应用于网络、图形等。然而，大多数 Unix 派生的操作系统没有遵循这条路线。

文件系统和文件描述符一直是强大的抽象。即便如此，操作系统接口还有其他模型。Unix 的前身 Multics 以使其看起来像内存的方式抽象文件存储，产生了非常不同的接口风格。Multics 设计的复杂性对 Unix 的设计者产生了直接影响，他们旨在构建更简单的东西。

xv6 不提供用户概念或保护一个用户免受另一个用户侵害；用 Unix 术语来说，所有 xv6 进程都以 root 身份运行。

本书研究 xv6 如何实现其类 Unix 接口，但这些想法和概念不仅适用于 Unix。任何操作系统都必须将进程多路复用到底层硬件上，将进程彼此隔离，并提供受控的进程间通信机制。学习 xv6 后，你应该能够查看其他更复杂的操作系统，并在这些系统中看到 xv6 背后的概念。
