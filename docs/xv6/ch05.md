# 第5章 中断与设备驱动

驱动是管理设备的代码，配置硬件、告诉设备执行操作、处理产生的中断，以及管理与设备交互的进程。驱动代码可能很棘手，因为驱动与它管理的设备并发执行。此外，驱动必须理解设备的硬件接口，这些接口可能很复杂且文档不完善。

需要操作系统关注的设备通常可以被配置为产生中断，这是陷阱的一种。内核陷阱处理代码识别何时设备引发中断并调用驱动的中断处理程序；在 xv6 中，这种分派发生在 `devintr` 中。

许多设备驱动在两种上下文中执行代码。上半部分在进程的内核线程中运行，通过 `read` 和 `write` 等系统调用调用。这部分代码可能要求硬件执行操作（例如，请求磁盘读取），然后等待操作完成。设备执行操作是异步的，所以硬件完成操作后会引发中断。驱动的下半部分（中断处理程序）告诉硬件该做什么，然后唤醒等待操作的进程。

## 5.1 代码：控制台输入

控制台驱动（`console.c`）是驱动结构的简单示例。控制台驱动通过 RISC-V 上的 UART 硬件接收用户键入的字符。控制台驱动一次累积一行输入，处理特殊字符如退格和 Ctrl-u。用户进程使用 `read` 系统调用从控制台获取输入行。当用户键入时，UART 硬件向 CPU 引发接收中断。

`uartgetc` 从 UART 硬件读取一个字符。每次 UART 接收一个字符时，它都会引发中断，`uartintr` 调用 `consoleintr`；它获取字符并将其交给 `cons.buf`。`consoleintr` 积累输入直到整行到达，然后唤醒等待的 `consoleread`。`consoleread` 在 `cons.buf` 中积累整行，然后将其复制到用户空间。

## 5.2 代码：控制台输出

在连接到控制台的终端上，`write` 系统调用最终到达 `uartputc`。驱动维护一个输出缓冲区（`uart_tx_buf`），这样写进程不需要等待 UART 完成发送；相反，`uartputc` 将每个字符追加到缓冲区，调用 `uartstart` 启动发送器（如果尚未运行），然后返回。仅当缓冲区满时，`uartputc` 才会等待。每次 UART 完成发送一个字节时，它都会引发中断。`uartintr` 调用 `uartstart`，检查设备是否真的完成发送，并将下一个缓冲的字符交给设备。因此，如果一个进程向控制台写入大量字节，第一个进程将用第一批填充 `uart_tx_buf`，然后 `uartputc` 在 `uart_tx_buf` 中睡眠等待。当 UART 完成发送一个字节时，`uartintr` 将唤醒该进程，该进程将尝试发送更多字节。

## 5.3 驱动中的并发

你可能已经注意到对 `consoleread` 和 `consoleintr` 的调用可能同时发生。`cons.lock` 序列化这些调用，防止竞争。例如，如果没有锁，`consoleintr` 可能在 `consoleread` 读取 `cons.buf` 时向其写入。更一般地，锁保护驱动与设备之间的并发，以及一个进程在 `consoleread` 中等待输入时另一个进程可能正在调用 `read` 的事实。

并发的一个更微妙的方面是，一个进程可能正在等待来自设备的输入，而中断到达时，等待的进程可能是不同的进程。因此，中断处理程序不能唤醒任何特定进程；它们必须只唤醒等待 I/O 的进程。`consoleintr` 调用 `wakeup` 唤醒在 `cons.r` 上睡眠的进程；任何等待输入的进程都会醒来。

## 5.4 定时器中断

xv6 使用定时器中断来切换进程；`yield` 在 `usertrap` 和 `kerneltrap` 中从定时器中断调用。时钟硬件为每个 CPU 引发中断。`clockintr` 累积这种中断并请求定时器中断；如果进程已运行了时间片，`clockintr` 调用 `yield` 让出 CPU。

## 5.5 现实世界

UART 驱动通过轮询与设备通信，也称为**编程 I/O**（programmed I/O）。程序 I/O 简单，但 CPU 在等待设备时效率低下。另一种方法是**直接内存访问**（direct memory access，DMA），设备直接将数据写入物理内存，然后引发中断。DMA 设备驱动在等待 I/O 时更高效，可以更好地利用 CPU，但 DMA 的编程更复杂。例如，必须仔细协调，以便设备不会覆盖进程仍在使用的内存。
