# 第8章 文件系统

文件系统的目的是组织和存储数据。文件系统通常支持用户和应用程序之间的数据共享，以及**持久性**（persistence），以便数据在重启后仍然可用。

xv6 文件系统提供类 Unix 的文件、目录和路径名（见第 1 章），并将其数据存储在 virtio 磁盘上以实现持久性。文件系统解决了几个挑战：

- 文件系统需要磁盘上的数据结构来表示命名目录和文件的树、记录保存每个文件内容的块的身份，以及记录磁盘的哪些区域是空闲的。
- 文件系统必须支持**崩溃恢复**（crash recovery）。也就是说，如果发生崩溃（例如断电），文件系统必须在重启后仍然正确工作。风险是崩溃可能中断一系列更新并留下不一致的磁盘上数据结构（例如，既在文件中使用又标记为空闲的块）。
- 不同的进程可能同时操作文件系统，因此文件系统代码必须协调以维护不变量。
- 访问磁盘比访问内存慢几个数量级，因此文件系统必须维护流行块的**内存缓存**（in-memory cache）。

本章其余部分解释 xv6 如何解决这些挑战。

## 8.1 概述

xv6 文件系统实现组织为七个层次，如**图 8.1** 所示。磁盘层在 virtio 硬盘上读写块。**缓冲区缓存**（buffer cache）层缓存磁盘块并同步对它们的访问，确保一次只有一个内核进程可以修改存储在任何特定块中的数据。**日志**（logging）层允许更高层将多个块的更新包装在**事务**（transaction）中，并确保在崩溃面前块被原子更新（即全部更新或全部不更新）。**inode 层**（inode layer）提供单个文件，每个文件表示为具有唯一 i-number 和一些保存文件数据的块的 inode。**目录层**（directory layer）将每个目录实现为一种特殊的 inode，其内容是一系列目录项，每个包含文件名和 i-number。**路径名层**（pathname layer）提供分层路径名，如 `/usr/rtm/xv6/fs.c`，并通过递归查找解析它们。**文件描述符层**（file descriptor layer）使用文件系统接口抽象许多 Unix 资源（例如管道、设备、文件等），简化应用程序程序员的生活。

```
┌─────────────────────────────────────────┐
│ 文件描述符 (File descriptor)              │
├─────────────────────────────────────────┤
│ 路径名 (Pathname)                        │
├─────────────────────────────────────────┤
│ 目录 (Directory)                         │
├─────────────────────────────────────────┤
│ inode                                   │
├─────────────────────────────────────────┤
│ 日志 (Logging)                           │
├─────────────────────────────────────────┤
│ 缓冲区缓存 (Buffer cache)                 │
├─────────────────────────────────────────┤
│ 磁盘 (Disk)                              │
└─────────────────────────────────────────┘
```

**图 8.1**：xv6 文件系统的层次。

磁盘硬件传统上将磁盘上的数据呈现为编号的 512 字节**块**（blocks）（也称为扇区）序列：扇区 0 是前 512 字节，扇区 1 是下一个，依此类推。操作系统用于其文件系统的块大小可能与磁盘使用的扇区大小不同，但通常块大小是扇区大小的倍数。xv6 在 `struct buf`（`kernel/buf.h:1`）类型的对象中保存它已读入内存的块的副本。存储在此结构中的数据有时与磁盘不同步：它可能尚未从磁盘读入（磁盘正在处理但尚未返回扇区内容），或者可能已被软件更新但尚未写入磁盘。

文件系统必须有计划在磁盘上存储 inode 和内容块的位置。为此，xv6 将磁盘分为几个部分，如**图 8.2** 所示。文件系统不使用块 0（它保存引导扇区）。块 1 称为**超级块**（superblock）；它包含有关文件系统的元数据（文件系统的块大小、数据块数、inode 数和日志中的块数）。从块 2 开始的块保存日志。日志之后是 inode，每块多个 inode。之后是跟踪哪些数据块正在使用的位图块。其余块是数据块；每个要么在位图块中标记为空闲，要么保存文件或目录的内容。超级块由单独的程序 `mkfs` 填充，该程序构建初始文件系统。

## 8.2 缓冲区缓存层

缓冲区缓存有两个工作：（1）同步对磁盘块的访问，确保内存中只有一块的副本，并且一次只有一个内核线程使用该副本；（2）缓存流行块，以便它们不需要从慢速磁盘重新读取。代码在 `bio.c` 中。

缓冲区缓存导出的主要接口由 `bread` 和 `bwrite` 组成；前者获取包含块副本的 buf，可以在内存中读取或修改，后者将修改的缓冲区写入磁盘上的适当块。内核线程在使用完缓冲区后必须通过调用 `brelse` 释放它。缓冲区缓存使用每缓冲区的睡眠锁确保一次只有一个线程使用每个缓冲区（因此每个磁盘块）；`bread` 返回锁定的缓冲区，`brelse` 释放锁。

## 8.3 代码：缓冲区缓存

缓冲区缓存是缓冲区的双向链表。函数 `binit` 由 `main`（`kernel/main.c:27`）调用，使用静态数组 `buf`（`kernel/bio.c:43-52`）中的 NBUF 个缓冲区初始化链表。对缓冲区缓存的所有其他访问通过 `bcache.head` 引用链表，而非 buf 数组。

缓冲区有两个与之关联的状态字段。字段 `valid` 表示缓冲区包含块的副本。字段 `disk` 表示缓冲区内容已交给磁盘，磁盘可能会更改缓冲区（例如，将数据从磁盘写入 data）。

`bread`（`kernel/bio.c:93`）调用 `bget` 获取给定扇区的缓冲区（`kernel/bio.c:97`）。如果缓冲区需要从磁盘读取，`bread` 在返回缓冲区之前调用 `virtio_disk_rw` 执行此操作。

`bget`（`kernel/bio.c:59`）扫描缓冲区链表以查找具有给定设备和扇区号的缓冲区（`kernel/bio.c:65-73`）。如果有这样的缓冲区，`bget` 获取缓冲区的睡眠锁。然后 `bget` 返回锁定的缓冲区。

如果没有给定扇区的缓存缓冲区，`bget` 必须创建一个，可能重用保存不同扇区的缓冲区。它第二次扫描缓冲区链表，查找未使用的缓冲区（`b->refcnt = 0`）；任何这样的缓冲区都可以使用。`bget` 编辑缓冲区元数据以记录新的设备和扇区号，并获取其睡眠锁。注意，赋值 `b->valid = 0` 确保 `bread` 将从磁盘读取块数据，而不是错误地使用缓冲区的先前内容。

重要的是每个磁盘扇区最多有一个缓存缓冲区，以确保读取者看到写入，并且因为文件系统使用缓冲区上的锁进行同步。`bget` 通过从第一个循环检查块是否被缓存到第二个循环声明块现在被缓存（通过设置 dev、blockno 和 refcnt）持续持有 `bcache.lock` 来确保此不变量。这导致块存在的检查和（如果不存在）指定保存块的缓冲区的声明是原子的。

## 8.4 日志层

文件系统设计中最有趣的问题之一是崩溃恢复。问题出现是因为许多文件系统操作涉及对磁盘的多次写入，而部分写入后的崩溃可能使磁盘上的文件系统处于不一致状态。例如，假设在文件截断（将文件长度设置为零并释放其内容块）期间发生崩溃。根据磁盘写入的顺序，崩溃可能留下引用标记为空闲的内容块的 inode，或者留下已分配但未引用的内容块。

后者相对良性，但引用已释放块的 inode 可能在重启后导致严重问题。重启后，内核可能将该块分配给另一个文件，现在我们有两个不同的文件无意中指向同一块。如果 xv6 支持多个用户，这种情况可能是安全问题，因为旧文件的所有者将能够读写属于不同用户的新文件中的块。

xv6 使用简单形式的**日志**（logging）解决文件系统操作期间的崩溃问题。xv6 系统调用不直接写入磁盘上的文件系统数据结构。相反，它将希望进行的所有磁盘写入的描述放在磁盘上的日志中。一旦系统调用记录了所有写入，它向磁盘写入特殊的**提交**（commit）记录，指示日志包含完整操作。此时，系统调用将写入复制到磁盘上的文件系统数据结构。在这些写入完成后，系统调用擦除磁盘上的日志。

如果系统崩溃并重启，文件系统代码在运行任何进程之前按如下方式从崩溃中恢复。如果日志被标记为包含完整操作，则恢复代码将写入复制到磁盘上文件系统中它们所属的位置。如果日志未标记为包含完整操作，恢复代码忽略日志。恢复代码通过擦除日志完成。

为什么 xv6 的日志解决文件系统操作期间的崩溃问题？如果崩溃发生在操作提交之前，则磁盘上的日志将不会被标记为完整，恢复代码将忽略它，磁盘状态将如同操作甚至没有开始。如果崩溃发生在操作提交之后，则恢复将重放操作的所有写入，如果操作已开始将它们写入磁盘上的数据结构，可能会重复它们。无论哪种情况，日志使操作相对于崩溃是原子的：恢复后，操作的所有写入要么都出现在磁盘上，要么都不出现。

## 8.5 日志设计

日志位于超级块中指定的已知固定位置。它由头块后跟一系列更新的块副本（「已记录块」）组成。头块包含扇区号数组，每个已记录块一个，以及日志块计数。磁盘上头块的计数要么为零（表示日志中没有事务），要么为非零（表示日志包含具有指定数量已记录块的完整已提交事务）。xv6 在事务提交时写入头块，但不在之前，并在将已记录块复制到文件系统后将计数设置为零。因此，事务中途的崩溃将导致日志头块中的计数为零；提交后的崩溃将导致非零计数。

每个系统调用的代码指示必须相对于崩溃原子化的写入序列的开始和结束。为了允许不同进程并发执行文件系统操作，日志系统可以将多个系统调用的写入累积到一个事务中。因此，单个提交可能涉及多个完整系统调用的写入。为避免将系统调用拆分到多个事务中，日志系统仅在没有任何文件系统系统调用正在进行时提交。将多个事务一起提交的想法称为**组提交**（group commit）。

## 8.6 代码：日志

系统调用中日志的典型用法如下：

```c
begin_op();
...
bp = bread(...);
bp->data[...] = ...;
log_write(bp);
...
end_op();
```

`begin_op` 等待直到日志系统当前未在提交，并且有足够的未保留日志空间来容纳此调用的写入。`log_write` 充当 `bwrite` 的代理。它记录块的扇区号，在磁盘上的日志中为其保留槽位，并固定缓冲区缓存中的块以防止块缓存驱逐它。`end_op` 首先减少未完成系统调用的计数。如果计数现在为零，它通过调用 `commit()` 提交当前事务。

## 8.7 代码：块分配器

文件和目录内容存储在磁盘块中，必须从空闲池分配。xv6 的块分配器在磁盘上维护空闲位图，每块一位。零位表示相应的块空闲；一位表示正在使用。程序 `mkfs` 设置对应于引导扇区、超级块、日志块、inode 块和位图块的位。块分配器提供两个函数：`balloc` 分配新的磁盘块，`bfree` 释放块。

## 8.8 Inode 层

术语 inode 可以有两种相关含义之一。它可能指包含文件大小和数据块号列表的磁盘上数据结构。或者「inode」可能指内存中的 inode，它包含磁盘上 inode 的副本以及内核内需要的额外信息。

磁盘上的 inode 打包在称为 inode 块的磁盘连续区域中。每个 inode 大小相同，因此给定数字 n，很容易找到磁盘上的第 n 个 inode。事实上，这个数字 n，称为 **inode 号**（inode number）或 **i-number**，是实现中如何识别 inode 的。

磁盘上的 inode 由 `struct dinode`（`kernel/fs.h:32`）定义。`type` 字段区分文件、目录和特殊文件（设备）。零类型表示磁盘上的 inode 是空闲的。`nlink` 字段计算引用此 inode 的目录项数，以识别何时应释放磁盘上的 inode 及其数据块。`size` 字段记录文件中内容的字节数。`addrs` 数组记录保存文件内容的磁盘块的块号。

内核在称为 `itable` 的表中将活动 inode 集保存在内存中；`struct inode`（`kernel/file.h:17`）是磁盘上 `struct dinode` 的内存副本。仅当有 C 指针引用该 inode 时，内核才在内存中存储 inode。`ref` 字段计算引用内存中 inode 的 C 指针数，如果引用计数降至零，内核从内存中丢弃 inode。`iget` 和 `iput` 函数获取和释放 inode 的指针，修改引用计数。inode 的指针可以来自文件描述符、当前工作目录和瞬态内核代码（如 exec）。

## 8.9 代码：Inodes

要分配新的 inode（例如，创建文件时），xv6 调用 `ialloc`（`kernel/fs.c:199`）。`ialloc` 类似于 `balloc`：它循环遍历磁盘上的 inode 结构，一次一块，查找标记为空闲的。当找到时，它通过将新类型写入磁盘来声明它，然后通过尾调用 `iget` 返回 inode 表中的条目。`ialloc` 的正确操作取决于一次只有一个进程可以持有对 `bp` 的引用这一事实。`iget` 在 inode 表中查找具有所需设备和 inode 号的活动条目。如果找到，它返回对该 inode 的新引用。代码必须使用 `ilock` 在读取或写入其元数据或内容之前锁定 inode。`iput` 通过减少引用计数释放对 inode 的 C 指针。

## 8.10 代码：Inode 内容

磁盘上的 inode 结构 `struct dinode` 包含 size 和块号数组。inode 数据在 dinode 的 `addrs` 数组中列出的块中找到。前 NDIRECT 个数据块在数组的前 NDIRECT 个条目中列出；这些块称为**直接块**（direct blocks）。接下来的 NINDIRECT 个数据块列在 inode 中，而是在称为**间接块**（indirect block）的数据块中。`addrs` 数组中的最后一个条目给出间接块的地址。函数 `bmap` 管理此表示，以便 `readi` 和 `writei` 等更高级例程不需要管理这种复杂性。`bmap` 返回 inode `ip` 的第 bn 个数据块的磁盘块号。如果 `ip` 还没有这样的块，`bmap` 会分配一个。

## 8.11 代码：目录层

目录在内部实现得与文件非常相似。其 inode 的类型为 T_DIR，其数据是一系列目录项。每个条目是 `struct dirent`（`kernel/fs.h:56`），包含名称和 inode 号。函数 `dirlookup` 在目录中搜索具有给定名称的条目。如果找到，它返回指向相应 inode 的指针（未锁定），并设置 `*poff` 为目录内条目的字节偏移量。函数 `dirlink` 将具有给定名称和 inode 号的新目录项写入目录 `dp`。

## 8.12 代码：路径名

路径名查找涉及对 `dirlookup` 的一系列调用，路径的每个组件一次。`namei`（`kernel/fs.c:687`）求值路径并返回相应的 inode。函数 `nameiparent` 是变体：它在最后一个元素之前停止，返回父目录的 inode 并将最终元素复制到 name。两者都调用通用函数 `namex` 来完成实际工作。`namex` 首先决定路径求值从哪里开始。如果路径以斜杠开头，求值从根开始；否则从当前目录开始。然后它使用 `skipelem` 依次考虑路径的每个元素。

## 8.13 文件描述符层

Unix 接口的一个很酷的方面是 Unix 中的大多数资源都表示为文件，包括控制台、管道等设备，当然还有真实文件。文件描述符层是实现这种统一性的层。

xv6 为每个进程提供自己的打开文件表或文件描述符，如我们在第 1 章所见。每个打开的文件由 `struct file`（`kernel/file.h:1`）表示，它是 inode 或管道的包装器，加上 I/O 偏移量。每次调用 `open` 都会创建新的打开文件（新的 `struct file`）：如果多个进程独立打开同一文件，不同的实例将有不同的 I/O 偏移量。另一方面，单个打开的文件（相同的 `struct file`）可以出现在一个进程的文件表中多次，也可以出现在多个进程的文件表中。如果一个进程使用 `open` 打开文件，然后使用 `dup` 创建别名或使用 `fork` 与子进程共享，就会发生这种情况。引用计数跟踪对特定打开文件的引用数。文件可以打开以供读取或写入或两者。`readable` 和 `writable` 字段跟踪这一点。

## 8.14 代码：系统调用

有了较低层提供的函数，大多数系统调用的实现是微不足道的（见 `kernel/sysfile.c`）。有几个调用值得仔细看看。`sys_link` 和 `sys_unlink` 编辑目录，创建或删除对 inode 的引用。它们是使用事务力量的另一个好例子。`sys_link` 获取其参数（两个字符串 old 和 new），假设 old 存在且不是目录，增加其 `ip->nlink` 计数，然后调用 `nameiparent` 找到 new 的父目录和最终路径元素，并创建指向 old 的 inode 的新目录项。函数 `create` 为新 inode 创建新名称。它是三个文件创建系统调用的概括：带 O_CREATE 标志的 `open` 创建新的普通文件，`mkdir` 创建新目录，`mkdev` 创建新设备文件。

## 8.15 现实世界

真实世界操作系统中的缓冲区缓存比 xv6 的复杂得多，但它服务于相同的两个目的：缓存和同步对磁盘的访问。xv6 的缓冲区缓存像 V6 一样使用简单的最近最少使用（LRU）驱逐策略；有许多更复杂的策略可以实现，每种对某些工作负载好，对其他的不那么好。xv6 的日志系统效率低下。提交不能与文件系统系统调用并发发生。它记录整个块，即使块中只有几个字节被更改。它执行同步日志写入，一次一块。真实日志系统解决所有这些问题。xv6 使用与早期 UNIX 相同的基本磁盘上 inode 和目录布局；这种方案多年来一直非常持久。
