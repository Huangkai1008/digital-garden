# 第3章 页表

页表是操作系统为每个进程提供私有地址空间和内存的最流行机制。页表决定内存地址的含义，以及可以访问物理内存的哪些部分。它们允许 xv6 隔离不同进程的地址空间并将它们多路复用到单个物理内存上。页表是流行的设计，因为它们提供间接级别，允许操作系统执行许多技巧。xv6 执行一些技巧：在多个地址空间中映射相同的内存（trampoline 页），以及用未映射页保护内核和用户栈。本章其余部分解释 RISC-V 硬件提供的页表以及 xv6 如何使用它们。

## 3.1 分页硬件

提醒一下，RISC-V 指令（用户和内核）操作**虚拟地址**（virtual addresses）。机器的 RAM 或**物理内存**（physical memory）用**物理地址**（physical addresses）索引。RISC-V 页表硬件通过将每个虚拟地址映射到物理地址来连接这两种地址。

xv6 在 Sv39 RISC-V 上运行，这意味着 64 位虚拟地址中只有低 39 位被使用；高 25 位未使用。在此 Sv39 配置中，RISC-V 页表逻辑上是 $2^{27}$（134,217,728）个**页表项**（page table entries，PTE）的数组。每个 PTE 包含 44 位**物理页号**（physical page number，PPN）和一些标志。分页硬件通过使用 39 位中的高 27 位索引到页表以找到 PTE，并生成 56 位物理地址，其高 44 位来自 PTE 中的 PPN，低 12 位从原始虚拟地址复制。**图 3.1** 以页表作为 PTE 简单数组的逻辑视图显示此过程（完整故事见**图 3.2**）。页表使操作系统能够以 4096（$2^{12}$）字节对齐块的粒度控制虚拟到物理地址的转换。这样的块称为**页**（page）。

在 Sv39 RISC-V 中，虚拟地址的高 25 位不用于转换。物理地址也有增长空间：PTE 格式中有空间让物理页号再增长 10 位。RISC-V 的设计者基于技术预测选择了这些数字。$2^{39}$ 字节是 512 GB，对于在 RISC-V 计算机上运行的应用程序应该足够的地址空间。

如**图 3.2** 所示，RISC-V CPU 分三步将虚拟地址转换为物理地址。页表在物理内存中存储为三级树。树的根是包含 512 个 PTE 的 4096 字节页表页，这些 PTE 包含树下一级页表页的物理地址。这些页中的每一页都包含树最后一级的 512 个 PTE。分页硬件使用 27 位的高 9 位选择根页表页中的 PTE，中间 9 位选择树下一级页表页中的 PTE，低 9 位选择最终 PTE。（在 Sv48 RISC-V 中，页表有四级，虚拟地址的 39 到 47 位索引到顶级。）

如果转换地址所需的三个 PTE 中的任何一个不存在，分页硬件会引发**页错误异常**（page-fault exception），由内核处理该异常（见第 4 章）。图 3.2 的三级结构允许与图 3.1 的单级设计相比，以内存高效的方式记录 PTE。在虚拟地址大范围没有映射的常见情况下，三级结构可以省略整个页目录。例如，如果应用程序只使用从地址零开始的几页，则顶级页目录的条目 1 到 511 无效，内核不必为那 511 个中间页目录分配页。此外，内核也不必为那 511 个中间页目录的底层页目录分配页。因此，在此示例中，三级设计节省了 511 页用于中间页目录和 $511 \times 512$ 页用于底层页目录。

尽管 CPU 在硬件中遍历三级结构作为执行加载或存储指令的一部分，但三级的潜在缺点在于 CPU 必须从内存加载三个 PTE 才能将加载/存储指令中的虚拟地址转换为物理地址。为避免从物理内存加载 PTE 的成本，RISC-V CPU 在**转换后备缓冲器**（Translation Look-aside Buffer，TLB）中缓存页表项。

每个 PTE 包含告诉分页硬件如何允许使用关联虚拟地址的标志位。PTE_V 指示 PTE 是否存在：如果未设置，对页的引用会导致异常（即不允许）。PTE_R 控制是否允许指令读取页。PTE_W 控制是否允许指令写入页。PTE_X 控制 CPU 是否可以将页内容解释为指令并执行。PTE_U 控制用户模式下的指令是否允许访问页；如果未设置 PTE_U，PTE 只能在监督模式下使用。图 3.2 显示了这一切如何工作。标志和所有其他页硬件相关结构在 `kernel/riscv.h` 中定义。

要告诉 CPU 使用页表，内核必须将根页表页的物理地址写入 satp 寄存器。CPU 将使用其自己的 satp 指向的页表转换后续指令生成的所有地址。每个 CPU 有自己的 satp，因此不同的 CPU 可以运行不同的进程，每个进程都有由其自己的页表描述的私有地址空间。

通常内核在其页表中映射所有物理内存，以便它可以使用加载/存储指令读写物理内存中的任何位置。由于页目录在物理内存中，内核可以通过使用标准存储指令写入 PTE 的虚拟地址来编程页目录中 PTE 的内容。

关于术语的几点说明。物理内存指 DRAM 中的存储单元。物理内存的一个字节有一个地址，称为物理地址。指令只使用虚拟地址，分页硬件将其转换为物理地址，然后发送到 DRAM 硬件进行读写。与物理内存和虚拟地址不同，**虚拟内存**（virtual memory）不是物理对象，而是指内核提供的用于管理物理内存和虚拟地址的抽象和机制的集合。

## 3.2 内核地址空间

xv6 为每个进程维护一个页表，描述每个进程的用户地址空间，加上描述内核地址空间的单个页表。内核配置其地址空间的布局，以便在可预测的虚拟地址访问物理内存和各种硬件资源。**图 3.3** 显示此布局如何将内核虚拟地址映射到物理地址。文件 `kernel/memlayout.h` 声明 xv6 内核内存布局的常量。

QEMU 模拟的计算机包括从物理地址 `0x80000000` 开始并至少延续到 `0x88000000` 的 RAM（物理内存），xv6 称之为 PHYSTOP。QEMU 模拟还包括磁盘接口等 I/O 设备。QEMU 将设备接口作为位于物理地址空间 `0x80000000` 以下的**内存映射**（memory-mapped）控制寄存器暴露给软件。内核可以通过读写这些特殊物理地址与设备交互；此类读写与设备硬件通信而非与 RAM。第 4 章解释 xv6 如何与设备交互。

内核使用「直接映射」访问 RAM 和内存映射设备寄存器；也就是说，在等于物理地址的虚拟地址映射资源。例如，内核本身位于虚拟地址空间和物理内存中的 KERNBASE=0x80000000。直接映射简化了读写物理内存的内核代码。例如，当 `fork` 为子进程分配用户内存时，分配器返回该内存的物理地址；`fork` 在将父进程的用户内存复制到子进程时直接将该地址用作虚拟地址。

有几个内核虚拟地址不是直接映射的：

- **Trampoline 页**。它映射在虚拟地址空间顶部；用户页表具有相同的映射。第 4 章讨论 trampoline 页的作用，但我们在这里看到页表的一个有趣用例；物理页（保存 trampoline 代码）在内核的虚拟地址空间中映射两次：一次在虚拟地址空间顶部，一次使用直接映射。
- **内核栈页**。每个进程有自己的内核栈，映射在高处，以便 xv6 可以在其下方留下未映射的**保护页**（guard page）。保护页的 PTE 无效（即未设置 PTE_V），因此如果内核溢出内核栈，可能会引发异常，内核将 panic。没有保护页，溢出的栈会覆盖其他内核内存，导致错误操作。panic 崩溃更可取。

虽然内核通过高内存映射使用其栈，但它们也可以通过直接映射地址访问。另一种设计可能只有直接映射，并在直接映射地址使用栈。然而，在这种安排中，提供保护页将涉及取消映射否则会引用物理内存的虚拟地址，然后这些地址将难以使用。

内核使用 PTE_R 和 PTE_X 权限映射 trampoline 和内核文本的页。内核从这些页读取和执行指令。内核使用 PTE_R 和 PTE_W 权限映射其他页，以便它可以读写这些页中的内存。保护页的映射无效。

## 3.3 代码：创建地址空间

xv6 中操作地址空间和页表的大部分代码位于 `vm.c`（`kernel/vm.c:1`）。核心数据结构是 `pagetable_t`，它实际上是指向 RISC-V 根页表页的指针；`pagetable_t` 可以是内核页表或每进程页表之一。核心函数是 `walk`（查找虚拟地址的 PTE）和 `mappages`（为新映射安装 PTE）。以 `kvm` 开头的函数操作内核页表；以 `uvm` 开头的函数操作用户页表；其他函数用于两者。`copyout` 和 `copyin` 在作为系统调用参数提供的用户虚拟地址之间复制数据；它们在 vm.c 中，因为它们需要显式转换这些地址以找到相应的物理内存。

在引导序列早期，`main` 调用 `kvminit`（`kernel/vm.c:54`）使用 `kvmmake`（`kernel/vm.c:20`）创建内核的页表。此调用发生在 xv6 在 RISC-V 上启用分页之前，因此地址直接引用物理内存。`kvmmake` 首先分配一页物理内存来保存根页表页。然后它调用 `kvmmap` 安装内核需要的转换。转换包括内核的指令和数据、到 PHYSTOP 的物理内存以及实际上是设备的内存范围。`proc_mapstacks`（`kernel/proc.c:33`）为每个进程分配内核栈。它调用 `kvmmap` 在 KSTACK 生成的虚拟地址映射每个栈，为无效的栈保护页留出空间。

`kvmmap`（`kernel/vm.c:132`）调用 `mappages`（`kernel/vm.c:143`），为虚拟地址范围到相应物理地址范围的页表安装映射。它对范围内的每个虚拟地址分别执行，按页间隔。对于要映射的每个虚拟地址，`mappages` 调用 `walk` 找到该地址的 PTE 地址。然后它初始化 PTE 以保存相关物理页号、所需权限（PTE_W、PTE_X 和/或 PTE_R）以及 PTE_V 以将 PTE 标记为有效（`kernel/vm.c:158`）。

`walk`（`kernel/vm.c:86`）模拟 RISC-V 分页硬件，因为它查找虚拟地址的 PTE（见图 3.2）。`walk` 每次 9 位地下降三级页表。它使用每级的 9 位虚拟地址找到下一级页表或最终页的 PTE（`kernel/vm.c:92`）。如果 PTE 无效，则所需的页尚未分配；如果设置了 alloc 参数，`walk` 分配新的页表页并将其物理地址放入 PTE。它返回树最底层的 PTE 地址（`kernel/vm.c:102`）。

上述代码依赖于物理内存被直接映射到内核虚拟地址空间。例如，当 `walk` 下降页表级别时，它从 PTE（`kernel/vm.c:94`）中提取下一级页表页的（物理）地址，然后使用该地址作为虚拟地址获取下一级的 PTE（`kernel/vm.c:92`）。

`main` 调用 `kvminithart`（`kernel/vm.c:62`）安装内核页表。它将根页表页的物理地址写入 satp 寄存器。此后，CPU 将使用内核页表转换地址。由于内核使用恒等映射，下一条指令的现在虚拟地址将映射到正确的物理内存地址。

每个 RISC-V CPU 在 TLB 中缓存页表项，当 xv6 更改页表时，它必须告诉 CPU 使相应的缓存 TLB 项无效。如果不这样做，TLB 可能在稍后某个时刻使用旧的缓存映射，指向在此期间已分配给另一个进程的物理页，因此进程可能能够涂写其他进程的内存。RISC-V 有一条 `sfence.vma` 指令可以刷新当前 CPU 的 TLB。xv6 在 `kvminithart` 中重新加载 satp 寄存器后执行 `sfence.vma`，并在返回到用户空间之前切换到用户页表的 trampoline 代码中执行（`kernel/trampoline.S:89`）。

在更改 satp 之前也有必要发出 `sfence.vma`，以便等待所有未完成的加载和存储完成。此等待确保对页表的前述更新已完成，并确保前述加载和存储使用旧页表而非新页表。

为避免刷新完整 TLB，RISC-V CPU 可能支持地址空间标识符（ASID）。内核然后可以只刷新特定地址空间的 TLB 项。xv6 不使用此功能。

## 3.4 物理内存分配

内核必须为页表、用户内存、内核栈和管道缓冲区在运行时分配和释放物理内存。

xv6 使用内核末尾和 PHYSTOP 之间的物理内存进行运行时分配。它一次分配和释放整个 4096 字节的页。它通过页本身穿线链表来跟踪哪些页是空闲的。分配包括从链表中移除一页；释放包括将释放的页添加到链表。

## 3.5 代码：物理内存分配器

分配器位于 `kalloc.c`（`kernel/kalloc.c:1`）。分配器的数据结构是可用于分配的物理内存页的空闲链表。每个空闲页的链表元素是 `struct run`（`kernel/kalloc.c:17`）。分配器从哪里获得保存该数据结构的内存？它将每个空闲页的 `run` 结构存储在空闲页本身中，因为那里没有存储其他东西。空闲链表由自旋锁保护（`kernel/kalloc.c:21-24`）。链表和锁包装在结构中以明确锁保护结构中的字段。现在忽略锁和对 `acquire` 和 `release` 的调用；第 6 章将详细研究锁。

函数 `main` 调用 `kinit` 初始化分配器（`kernel/kalloc.c:27`）。`kinit` 将空闲链表初始化为保存内核末尾和 PHYSTOP 之间的每一页。xv6 应该通过解析硬件提供的配置信息来确定有多少物理内存可用。相反，xv6 假设机器有 128 兆字节的 RAM。`kinit` 调用 `freerange` 通过每页调用 `kfree` 将内存添加到空闲链表。PTE 只能引用在 4096 字节边界上对齐的物理地址（是 4096 的倍数），因此 `freerange` 使用 PGROUNDUP 确保它只释放对齐的物理地址。分配器开始时没有内存；这些对 `kfree` 的调用给它一些来管理。

分配器有时将地址视为整数以对它们执行算术（例如，在 `freerange` 中遍历所有页），有时使用地址作为指针读写内存（例如，操作存储在每个页中的 run 结构）；地址的这种双重使用是分配器代码充满 C 类型转换的主要原因。另一个原因是释放和分配本质上会改变内存的类型。

函数 `kfree`（`kernel/kalloc.c:47`）首先将正在释放的内存中的每个字节设置为值 1。这将导致在释放后使用内存的代码（使用「悬空引用」）读取垃圾而非旧的有效内容；希望这会使此类代码更快地崩溃。然后 `kfree` 将页前置到空闲链表：它将 pa 转换为 `struct run` 的指针，在 `r->next` 中记录空闲链表的旧开头，并将空闲链表设置为 r。`kalloc` 移除并返回空闲链表中的第一个元素。

## 3.6 进程地址空间

每个进程有单独的页表，当 xv6 在进程之间切换时，它也会更改页表。**图 3.4** 比图 2.3 更详细地显示进程的地址空间。进程的用户内存从虚拟地址零开始，可以增长到 MAXVA（`kernel/riscv.h:360`），允许进程原则上寻址 256 千兆字节的内存。

进程的地址空间由包含程序文本的页（xv6 使用 PTE_R、PTE_X 和 PTE_U 权限映射）、包含程序预初始化数据的页、栈页和堆页组成。xv6 使用 PTE_R、PTE_W 和 PTE_U 权限映射数据、栈和堆。

在用户地址空间内使用权限是强化用户进程的常用技术。如果文本用 PTE_W 映射，则进程可能意外修改自己的程序；例如，编程错误可能导致程序写入空指针，修改地址 0 处的指令，然后继续运行，可能造成更多破坏。为了立即检测此类错误，xv6 在没有 PTE_W 的情况下映射文本；如果程序意外尝试存储到地址 0，硬件将拒绝执行存储并引发页错误（见第 4.6 节）。然后内核杀死进程并打印信息性消息，以便开发人员可以追踪问题。

同样，通过在没有 PTE_X 的情况下映射数据，用户程序不能意外跳转到程序数据中的地址并在该地址开始执行。

在现实世界中，通过仔细设置权限强化进程也有助于防御安全攻击。对手可能向程序（例如 Web 服务器）提供精心构造的输入，触发程序中的错误，希望将该错误转化为漏洞利用。仔细设置权限和其他技术，例如随机化用户地址空间的布局，使此类攻击更难。

栈是单页，显示为由 `exec` 创建的初始内容。包含命令行参数的字符串以及指向它们的指针数组在栈的最顶部。就在其下方是允许程序在 `main` 启动的值，就好像函数 `main(argc, argv)` 刚刚被调用一样。

为了检测用户栈溢出分配的栈内存，xv6 通过清除 PTE_U 标志在栈正下方放置不可访问的保护页。如果用户栈溢出且进程尝试使用栈下方的地址，硬件将生成页错误异常，因为保护页对在用户模式下运行的程序不可访问。真实世界的操作系统可能会在栈溢出时自动为用户栈分配更多内存。

当进程向 xv6 请求更多用户内存时，xv6 增长进程的堆。xv6 首先使用 `kalloc` 分配物理页。然后它向进程的页表添加指向新物理页的 PTE。xv6 在这些 PTE 中设置 PTE_W、PTE_R、PTE_U 和 PTE_V 标志。大多数进程不使用整个用户地址空间；xv6 在未使用的 PTE 中保持 PTE_V 清除。

我们在这里看到页表使用的几个很好的例子。首先，不同进程的页表将用户地址转换到物理内存的不同页，因此每个进程都有私有的用户内存。其次，每个进程将其内存视为从零开始的连续虚拟地址，而进程的物理内存可以是非连续的。第三，内核在用户地址空间顶部映射包含 trampoline 代码的页（没有 PTE_U），因此单页物理内存在所有地址空间中显示，但只能由内核使用。

## 3.7 代码：sbrk

`sbrk` 是进程缩小或增长其内存的系统调用。系统调用由函数 `growproc`（`kernel/proc.c:260`）实现。`growproc` 根据 n 是正还是负调用 `uvmalloc` 或 `uvmdealloc`。`uvmalloc`（`kernel/vm.c:226`）使用 `kalloc` 分配物理内存，并使用 `mappages` 向用户页表添加 PTE。`uvmdealloc` 调用 `uvmunmap`（`kernel/vm.c:171`），它使用 `walk` 查找 PTE 并使用 `kfree` 释放它们引用的物理内存。

xv6 使用进程的页表不仅告诉硬件如何映射用户虚拟地址，还作为分配给该进程的物理内存页的唯一记录。这就是为什么释放用户内存（在 `uvmunmap` 中）需要检查用户页表的原因。

## 3.8 代码：exec

`exec` 是用从文件读取的数据替换进程用户地址空间的系统调用，该文件称为二进制或可执行文件。二进制通常是编译器和链接器的输出，保存机器指令和程序数据。`exec`（`kernel/exec.c:23`）使用 `namei`（`kernel/exec.c:36`）打开命名的二进制路径，第 8 章将解释。然后，它读取 ELF 头。xv6 二进制采用广泛使用的 ELF 格式，在 `kernel/elf.h` 中定义。ELF 二进制由 ELF 头 `struct elfhdr`（`kernel/elf.h:6`）后跟程序段头序列 `struct proghdr`（`kernel/elf.h:25`）组成。每个 proghdr 描述必须加载到内存的应用程序部分；xv6 程序有两个程序段头：一个用于指令，一个用于数据。

第一步是快速检查文件可能包含 ELF 二进制。ELF 二进制以四字节「魔数」0x7F、'E'、'L'、'F' 或 ELF_MAGIC（`kernel/elf.h:3`）开头。如果 ELF 头有正确的魔数，`exec` 假设二进制格式正确。

`exec` 使用 `proc_pagetable`（`kernel/exec.c:49`）分配没有用户映射的新页表，使用 `uvmalloc`（`kernel/exec.c:65`）为每个 ELF 段分配内存，并使用 `loadseg`（`kernel/exec.c:10`）将每个段加载到内存中。`loadseg` 使用 `walkaddr` 找到要写入 ELF 段每页的分配内存的物理地址，并使用 `readi` 从文件读取。

`exec` 分配并初始化用户栈。它只分配一个栈页。`exec` 一次一个地将参数字符串复制到栈顶，在 `ustack` 中记录指向它们的指针。它在将传给 `main` 的 argv 列表末尾放置空指针。`ustack` 中的前三个条目是假的返回程序计数器、argc 和 argv 指针。

`exec` 在栈页正下方放置不可访问的页，以便尝试使用超过一个栈页的程序会收到页错误。这种安排允许 xv6 在程序使用过多栈时报告错误。

## 3.9 现实世界

许多操作系统设计利用页表将任意硬件物理内存布局转换为可预测的内核虚拟地址布局。RISC-V 支持物理地址级别的保护，但 xv6 不使用该功能。在内存较多的机器上，使用 RISC-V 对「超级页」的支持可能是有意义的。当物理内存较小时，小页是有意义的，以允许分配和换出。

## 3.10 练习

1. 编写一个程序，测量 `sbrk` 分配一页需要多长时间。
2. 编写一个程序，测量 `fork` 后第一次写入页需要多长时间。
3. 修改 xv6 以使用大页。
4. 修改 xv6 以在用户栈溢出时自动扩展用户栈。
