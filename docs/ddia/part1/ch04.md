# 第4章 编码与演化

> 一切都在变化，没有什么静止不动。
>
> — 以弗所的赫拉克利特，柏拉图在《克拉底鲁篇》中引用（公元前 360 年）

应用不可避免地随时间变化。随着新产品的推出、用户需求得到更好的理解或业务环境的变化，功能被添加或修改。在第 1 章中，我们介绍了可演化性的概念：我们应该致力于构建使适应变化变得容易的系统（见第 21 页「可演化性：让变更容易」）。

在大多数情况下，应用功能的更改也需要更改它存储的数据：也许需要捕获新字段或记录类型，也许现有数据需要以新方式呈现。

我们在第 2 章讨论的数据模型有不同的方式应对这种变化。关系数据库通常假设数据库中的所有数据符合一个模式：尽管该模式可以更改（通过模式迁移；即 ALTER 语句），但在任何时间点只有一个模式生效。相比之下，读时模式（「无模式」）数据库不强制执行模式，因此数据库可以包含在不同时间写入的旧格式和新格式数据的混合（见第 39 页「文档模型中的模式灵活性」）。

当数据格式或模式发生变化时，应用代码通常需要发生相应的更改（例如，你向记录添加新字段，应用代码开始读取和写入该字段）。然而，在大型应用中，代码更改通常无法瞬时发生：

- **对于服务端应用**，你可能希望执行**滚动升级**（rolling upgrade，也称为分阶段推出），一次将新版本部署到几个节点，检查新版本是否运行顺利，并逐渐完成所有节点。这允许在不服务停机的情况下部署新版本，从而鼓励更频繁的发布和更好的可演化性。
- **对于客户端应用**，你受用户支配，用户可能一段时间内不会安装更新。

这意味着旧版本和新版本的代码，以及旧格式和新格式的数据，可能同时共存于系统中。为了使系统继续平稳运行，我们需要在两个方向上保持兼容性：

**向后兼容**（Backward compatibility）：较新的代码可以读取由较旧代码写入的数据。

**向前兼容**（Forward compatibility）：较旧的代码可以读取由较新代码写入的数据。

向后兼容通常不难实现：作为较新代码的作者，你知道较旧代码写入的数据的格式，因此你可以显式处理它（如有必要，只需保留旧代码来读取旧数据）。向前兼容可能更棘手，因为它要求较旧的代码忽略较新版本代码所做的添加。

在本章中，我们将研究几种数据编码格式，包括 JSON、XML、Protocol Buffers、Thrift 和 Avro。特别是，我们将研究它们如何处理模式变化以及如何支持旧数据和新数据及代码需要共存的系统。然后我们将讨论这些格式如何用于数据存储和通信：在 Web 服务、表述性状态传递（REST）和远程过程调用（RPC）中，以及参与者（actors）和消息队列等消息传递系统。

## 数据编码格式

程序通常以（至少）两种不同的表示形式处理数据：

1. **在内存中**，数据保存在对象、结构体、列表、数组、哈希表、树等中。这些数据结构针对 CPU 的高效访问和操作进行了优化（通常使用指针）。
2. **当你想将数据写入文件或通过网络发送时**，你必须将其编码为某种自包含的字节序列（例如，JSON 文档）。由于指针对任何其他进程都没有意义，这种字节序列表示看起来与通常用于内存的数据结构大不相同。

因此，我们需要两种表示之间的某种转换。从内存表示到字节序列的转换称为**编码**（encoding，也称为序列化或编组），反向称为**解码**（decoding，解析、反序列化、解组）。

::: info 术语冲突
不幸的是，序列化也在事务的上下文中使用（见第 7 章），含义完全不同。为了避免过度使用该词，我们将在本书中坚持使用编码，尽管序列化可能是更常见的术语。
:::

由于这是一个如此常见的问题，有无数不同的库和编码格式可供选择。让我们做一个简要概述。

### 语言特定格式

许多编程语言带有将内存对象编码为字节序列的内置支持。例如，Java 有 `java.io.Serializable` [1]，Ruby 有 Marshal [2]，Python 有 pickle [3]，等等。许多第三方库也存在，如 Java 的 Kryo [4]。

这些编码库非常方便，因为它们允许以最少的额外代码保存和恢复内存对象。然而，它们也有一些深层问题：

- 编码通常与特定编程语言绑定，用另一种语言读取数据非常困难。如果你以这种编码存储或传输数据，你正在承诺自己可能很长时间使用当前编程语言，并排除了与使用不同语言的其他组织的系统集成。
- 为了在相同的对象类型中恢复数据，解码过程需要能够实例化任意类。这经常是安全问题的来源 [5]：如果攻击者可以让你的应用解码任意字节序列，他们可以实例化任意类，这反过来通常允许他们做可怕的事情，如远程执行任意代码 [6, 7]。
- 这些库中的版本控制通常是事后才想到的：由于它们旨在快速轻松地编码数据，它们经常忽视向前和向后兼容的不便问题。
- 效率（编码或解码所花费的 CPU 时间，以及编码结构的大小）也经常是事后才想到的。例如，Java 的内置序列化以其糟糕的性能和臃肿的编码而臭名昭著 [8]。

由于这些原因，除了非常短暂的目的之外，使用你语言的内置编码通常是个坏主意。

### JSON、XML 与二进制变体

转向可以由许多编程语言编写和读取的标准化编码，JSON 和 XML 是明显的竞争者。它们广为人知、广泛支持，几乎同样广泛不受欢迎。XML 经常因过于冗长和不必要地复杂而受到批评 [9]。JSON 的流行主要是由于其在 Web 浏览器中的内置支持（因为它是 JavaScript 的子集）以及相对于 XML 的简单性。CSV 是另一种流行的语言无关格式，尽管功能较弱。

JSON、XML 和 CSV 是文本格式，因此某种程度上是人类可读的（尽管语法是流行的辩论话题）。除了表面的语法问题之外，它们还有一些微妙的问题：

- **数字编码存在很多歧义**。在 XML 和 CSV 中，你无法区分数字和恰好由数字组成的字符串（除非引用外部模式）。JSON 区分字符串和数字，但不区分整数和浮点数，也不指定精度。这在处理大数字时是个问题；例如，大于 $2^{53}$ 的整数无法在 IEEE 754 双精度浮点数中精确表示，因此当在用浮点数的语言（如 JavaScript）中解析时，此类数字会变得不准确。Twitter 使用 64 位数字标识每条推文，大于 $2^{53}$ 的数字的例子发生在 Twitter 上。Twitter API 返回的 JSON 包含推文 ID 两次，一次作为 JSON 数字，一次作为十进制字符串，以解决 JavaScript 应用无法正确解析数字的问题 [10]。
- **JSON 和 XML 对 Unicode 字符串**（即人类可读的文本）有良好的支持，但它们不支持**二进制字符串**（没有字符编码的字节序列）。二进制字符串是一个有用的特性，因此人们通过使用 Base64 将二进制数据编码为文本来解决此限制。然后模式用于指示该值应被解释为 Base64 编码的。这有效，但有点 hack，并使数据大小增加 33%。
- **XML [11] 和 JSON [12] 都有可选的模式支持**。这些模式语言相当强大，因此相当复杂难以学习和实现。XML 模式的使用相当广泛，但许多基于 JSON 的工具不使用模式。由于数据（如数字和二进制字符串）的正确解释取决于模式中的信息，不使用 XML/JSON 模式的应用可能需要硬编码适当的编码/解码逻辑。
- **CSV 没有任何模式**，因此由应用定义每行和每列的含义。如果应用更改添加新行或列，你必须手动处理该更改。CSV 也是一种相当模糊的格式（如果值包含逗号或换行符会发生什么？）。尽管其转义规则已被正式指定 [13]，但并非所有解析器都正确实现它们。

尽管有这些缺陷，JSON、XML 和 CSV 对许多目的来说已经足够好。它们可能会保持流行，特别是作为数据交换格式（即，用于将数据从一个组织发送到另一个组织）。在这些情况下，只要人们就格式是什么达成一致，格式有多漂亮或高效通常并不重要。让不同组织就任何事情达成一致的困难超过了大多数其他考虑。

**二进制编码**

对于仅在你组织内部使用的数据，使用最低公分母编码格式的压力较小。例如，你可以选择更紧凑或解析更快的格式。对于小数据集，收益可以忽略不计，但一旦你进入太字节，数据格式的选择可能会产生很大影响。

JSON 比 XML 更简洁，但两者与二进制格式相比仍然使用大量空间。这一观察导致了大量 JSON 二进制编码（MessagePack、BSON、BJSON、UBJSON、BISON 和 Smile，仅举几例）和 XML（WBXML 和 Fast Infoset，例如）的开发。这些格式已在各种利基市场采用，但没有一个像 JSON 和 XML 的文本版本那样广泛采用。

**示例 4-1. 我们将在本章用几种二进制格式编码的示例记录**

```json
{
    "userName": "Martin",
    "favoriteNumber": 1337,
    "interests": ["daydreaming", "hacking"]
}
```

### Thrift 与 Protocol Buffers

Apache Thrift [15] 和 Protocol Buffers（protobuf）[16] 是基于相同原理的二进制编码库。Protocol Buffers 最初在 Google 开发，Thrift 最初在 Facebook 开发，两者都在 2007–08 年开源 [17]。

Thrift 和 Protocol Buffers 都需要对任何编码的数据进行模式定义。要用 Thrift 编码示例 4-1 中的数据，你会在 Thrift 接口定义语言（IDL）中这样描述模式：

```thrift
struct Person {
  1: required string       userName,
  2: optional i64          favoriteNumber,
  3: optional list<string> interests
}
```

Protocol Buffers 的等效模式定义看起来非常相似：

```protobuf
message Person {
    required string user_name       = 1;
    optional int64  favorite_number = 2;
    repeated string interests       = 3;
}
```

Thrift 和 Protocol Buffers 各自带有代码生成工具，接受像这里显示的模式定义，并生成在各种编程语言中实现该模式的类。你的应用代码可以调用此生成的代码来编码或解码模式的记录。

**字段标签与模式演化**

我们之前说过模式不可避免地需要随时间变化。我们称之为**模式演化**（schema evolution）。Thrift 和 Protocol Buffers 如何在保持向后和向前兼容的同时处理模式变化？

如示例所示，编码的记录只是其编码字段的串联。每个字段由其标签号（示例模式中的数字 1、2、3）标识，并用数据类型（例如，string 或 integer）注释。如果未设置字段值，则简单地从编码记录中省略。由此你可以看到字段标签对编码数据的含义至关重要。你可以在模式中更改字段的名称，因为编码的数据从不引用字段名称，但你不能更改字段的标签，因为那会使所有现有编码数据无效。

你可以向模式添加新字段，前提是你给每个字段一个新的标签号。如果旧代码（不知道你添加的新标签号）尝试读取由新代码写入的数据，包括它不认识的标签号的新字段，它可以简单地忽略该字段。数据类型注释允许解析器确定需要跳过多少字节。这保持了向前兼容：旧代码可以读取由新代码写入的记录。

向后兼容呢？只要每个字段有唯一的标签号，新代码总是可以读取旧数据，因为标签号仍然具有相同的含义。唯一的细节是，如果你添加新字段，你不能使其成为 required。如果你要添加字段并使其成为 required，当新代码读取由旧代码写入的数据时，该检查会失败，因为旧代码不会写入你添加的新字段。因此，为了保持向后兼容，你在模式初始部署后添加的每个字段必须是 optional 或有默认值。

删除字段就像添加字段一样，向后和向前兼容的考虑相反。这意味着你只能删除 optional 字段（required 字段永远不能删除），并且你永远不能再次使用相同的标签号。

### Avro

Apache Avro [20] 是另一种二进制编码格式，与 Protocol Buffers 和 Thrift 有趣地不同。它于 2009 年作为 Hadoop 的子项目启动，因为 Thrift 不适合 Hadoop 的用例 [21]。

Avro 也使用模式来指定正在编码的数据的结构。它有两种模式语言：一种（Avro IDL）用于人类编辑，一种（基于 JSON）更容易机器读取。

我们的示例模式，用 Avro IDL 编写，可能看起来像这样：

```avro
record Person {
    string               userName;
    union { null, long } favoriteNumber = null;
    array<string>        interests;
}
```

该模式的等效 JSON 表示如下：

```json
{
    "type": "record",
    "name": "Person",
    "fields": [
        {"name": "userName",       "type": "string"},
        {"name": "favoriteNumber", "type": ["null", "long"], "default": null},
        {"name": "interests",      "type": {"type": "array", "items": "string"}}
    ]
}
```

首先，注意模式中没有标签号。如果我们使用此模式编码示例记录（示例 4-1），Avro 二进制编码只有 32 字节长——我们见过的所有编码中最紧凑的。

要解析二进制数据，你按模式中出现的顺序遍历字段，并使用模式告诉你每个字段的数据类型。这意味着只有当读取数据的代码使用与写入数据的代码**完全相同的模式**时，才能正确解码二进制数据。读取器和写入器之间模式的任何不匹配都会意味着错误解码的数据。

那么，Avro 如何支持模式演化？

**写入器模式与读取器模式**

使用 Avro，当应用想要编码一些数据时（将其写入文件或数据库、通过网络发送等），它使用它知道的任何版本的模式编码数据——例如，该模式可能被编译到应用中。这被称为**写入器模式**（writer's schema）。

当应用想要解码一些数据时（从文件或数据库读取、从网络接收等），它期望数据在某个模式中，这被称为**读取器模式**（reader's schema）。那是应用代码依赖的模式——代码可能在该应用的构建过程中从该模式生成。

Avro 的关键思想是写入器模式和读取器模式不必相同——它们只需要兼容。当数据被解码（读取）时，Avro 库通过并排查看写入器模式和读取器模式，将数据从写入器模式转换为读取器模式来解决差异。Avro 规范 [20] 精确定义了此解析如何工作。

**模式演化的好处**

正如我们所看到的，Protocol Buffers、Thrift 和 Avro 都使用模式来描述二进制编码格式。它们的模式语言比 XML Schema 或 JSON Schema 简单得多，后者支持更详细的验证规则（例如，「此字段的字符串值必须匹配此正则表达式」或「此字段的整数值必须在 0 到 100 之间」）。由于 Protocol Buffers、Thrift 和 Avro 更简单实现和更简单使用，它们已经发展到支持相当广泛的编程语言。

基于模式的二进制编码有许多不错的属性：

- 它们可以比各种「二进制 JSON」变体紧凑得多，因为它们可以从编码数据中省略字段名称。
- 模式是有价值的文档形式，并且由于解码需要模式，你可以确保它是最新的（而手动维护的文档可能容易与现实脱节）。
- 维护模式数据库允许你在部署任何东西之前检查模式变化的向前和向后兼容性。
- 对于静态类型编程语言的用户，从模式生成代码的能力很有用，因为它支持编译时类型检查。

## 数据流模式

在本章开头我们说，每当你想将一些数据发送给与你共享内存的另一个进程时——例如，每当你想通过网络发送数据或将其写入文件时——你需要将其编码为字节序列。然后我们讨论了各种不同的编码方式。我们谈到了向前和向后兼容，这对可演化性很重要（通过允许你独立升级系统的不同部分，而不必一次性更改所有内容，使变更变得容易）。兼容性是编码数据的进程与解码它的另一个进程之间的关系。

这是一个相当抽象的想法——数据从进程流向另一个进程的方式有很多种。谁编码数据，谁解码它？在本章其余部分，我们将探讨数据在进程之间流动的一些最常见方式：

- **通过数据库**（见第 129 页「通过数据库的数据流」）
- **通过服务调用**（见第 131 页「通过服务的数据流：REST 和 RPC」）
- **通过异步消息传递**（见第 136 页「消息传递数据流」）

### 通过数据库的数据流

在数据库中，写入数据库的进程编码数据，从数据库读取的进程解码它。可能只有一个进程访问数据库，在这种情况下，读取者只是同一进程的较新版本——在这种情况下，你可以将存储在数据库中视为向未来的自己发送消息。

这里显然需要向后兼容；否则你未来的自己将无法解码你之前写入的内容。

一般来说，几个不同的进程同时访问数据库很常见。这些进程可能是几个不同的应用或服务，或者可能只是同一服务的几个实例（为可扩展性或容错并行运行）。无论哪种方式，在应用正在变化的环境中，访问数据库的一些进程可能运行较新的代码，一些可能运行较旧的代码——例如，因为新版本当前正在滚动升级中部署，所以一些实例已更新而其他还没有。

这意味着数据库中的值可能由较新版本的代码写入，随后由仍在运行的较旧版本的代码读取。因此，数据库通常也需要向前兼容。

然而，还有一个额外的障碍。假设你向记录模式添加一个字段，较新的代码将新字段的值写入数据库。随后，较旧版本的代码（还不知道新字段）读取记录，更新它，并写回。在这种情况下，期望的行为通常是旧代码保持新字段完整，即使它无法解释。

### 通过服务的数据流：REST 和 RPC

当你有需要通过网络通信的进程时，有几种不同的安排方式。最常见的安排是有两个角色：客户端和服务器。服务器通过网络暴露 API，客户端可以连接到服务器向该 API 发出请求。服务器暴露的 API 被称为**服务**（service）。

Web 以这种方式工作：客户端（Web 浏览器）向 Web 服务器发出请求，发出 GET 请求下载 HTML、CSS、JavaScript、图像等，并发出 POST 请求向服务器提交数据。API 由标准化的一组协议和数据格式（HTTP、URL、SSL/TLS、HTML 等）组成。

Web 服务有两种流行的方法：REST 和 SOAP。它们在哲学上几乎完全对立，通常是各自支持者之间激烈辩论的主题。

**REST** 不是协议，而是建立在 HTTP 原则之上的设计哲学 [34, 35]。它强调简单的数据格式，使用 URL 标识资源，使用 HTTP 特性进行缓存控制、身份验证和内容类型协商。与 SOAP 相比，REST 至少在跨组织服务集成的背景下一直在普及 [36]，通常与微服务相关联 [31]。根据 REST 原则设计的 API 称为 RESTful。

相比之下，**SOAP** 是用于发出网络 API 请求的基于 XML 的协议。尽管它最常用于 HTTP，但它旨在独立于 HTTP 并避免使用大多数 HTTP 特性。相反，它带有大量复杂的相关标准（称为 WS-* 的 Web 服务框架），添加各种功能 [37]。

**远程过程调用（RPC）的问题**

Web 服务只是通过网络发出 API 请求的漫长技术线中的最新体现，其中许多获得了大量炒作但存在严重问题。Enterprise JavaBeans（EJB）和 Java 的远程方法调用（RMI）仅限于 Java。分布式组件对象模型（DCOM）仅限于 Microsoft 平台。公共对象请求代理架构（CORBA）过于复杂，不提供向后或向前兼容 [41]。

这些都基于**远程过程调用**（RPC）的想法，自 1970 年代就存在 [42]。RPC 模型试图使对远程网络服务的请求看起来与在同一进程内调用编程语言中的函数或方法相同（这种抽象称为**位置透明性**，location transparency）。

尽管 RPC 乍一看似乎方便，但该方法存在根本缺陷 [43, 44]。网络请求与本地函数调用非常不同：

- **本地函数调用**是可预测的，要么成功要么失败，仅取决于你控制下的参数。**网络请求**是不可预测的：由于网络问题，请求或响应可能丢失，或者远程机器可能很慢或不可用，此类问题完全超出你的控制。网络问题很常见，因此你必须预见它们，例如通过重试失败的请求。
- **本地函数调用**要么返回结果，要么抛出异常，要么永不返回（因为它进入无限循环或进程崩溃）。**网络请求**有另一种可能的结果：由于超时，它可能在没有结果的情况下返回。在这种情况下，你根本不知道发生了什么：如果你没有从远程服务得到响应，你无法知道请求是否通过。（我们在第 8 章更详细地讨论这个问题。）
- **如果你重试失败的网络请求**，可能发生请求实际上正在通过，只有响应正在丢失。在这种情况下，重试将导致操作被执行多次，除非你在协议中构建去重（**幂等性**，idempotence）机制。本地函数调用没有这个问题。（我们在第 11 章更详细地讨论幂等性。）
- **每次调用本地函数时**，它通常需要大约相同的时间执行。**网络请求**比函数调用慢得多，其延迟也 wildly 变化：在好的时候它可能在不到一毫秒内完成，但当网络拥塞或远程服务过载时，做完全相同的事情可能需要许多秒。
- **当你调用本地函数时**，你可以高效地传递对本地内存中对象的引用（指针）。**当你发出网络请求时**，所有这些参数都需要编码成可以通过网络发送的字节序列。如果参数是数字或字符串等基本类型，那没问题，但对于更大的对象很快就变得有问题。
- **客户端和服务**可能用不同的编程语言实现，因此 RPC 框架必须将数据类型从一种语言转换为另一种语言。这可能最终变得丑陋，因为并非所有语言都有相同的类型——例如，回想一下 JavaScript 对大于 $2^{53}$ 的数字的问题（见第 114 页「JSON、XML 与二进制变体」）。在单一进程中用单一语言编写时不存在这个问题。

所有这些因素意味着，试图使远程服务看起来太像编程语言中的本地对象没有意义，因为它 fundamentally 是不同的东西。REST 的部分吸引力在于它不试图隐藏它是网络协议的事实（尽管这似乎并不能阻止人们在 REST 之上构建 RPC 库）。

### 消息传递数据流

我们一直在研究编码数据从进程流向另一个进程的不同方式。到目前为止，我们讨论了 REST 和 RPC（一个进程通过网络向另一个进程发送请求并期望尽快得到响应），以及数据库（一个进程写入编码数据，另一个进程在未来的某个时间再次读取它）。

在本最后部分，我们将简要研究**异步消息传递**系统，它们介于 RPC 和数据库之间。它们与 RPC 相似，因为客户端的请求（通常称为消息）以低延迟传递给另一个进程。它们与数据库相似，因为消息不是通过直接网络连接发送，而是通过称为**消息代理**（message broker，也称为消息队列或面向消息的中间件）的中介，临时存储消息。

与直接 RPC 相比，使用消息代理有几个优势：

- 如果接收者不可用或过载，它可以充当缓冲区，从而提高系统可靠性
- 它可以自动将消息重新传递给已崩溃的进程，从而防止消息丢失
- 它避免发送者需要知道接收者的 IP 地址和端口号（这在虚拟机经常来来去去的云部署中特别有用）
- 它允许一条消息发送给多个接收者
- 它在逻辑上将发送者与接收者解耦（发送者只是发布消息，不关心谁消费它们）

然而，与 RPC 相比，一个区别是消息传递通信通常是单向的：发送者通常不期望收到其消息的回复。进程发送响应是可能的，但这通常会在单独的通道上完成。这种通信模式是异步的：发送者不等待消息被传递，而是简单地发送它然后忘记它。

**消息代理**

过去，消息代理的格局由 TIBCO、IBM WebSphere 和 webMethods 等公司的商业企业软件主导。最近，RabbitMQ、ActiveMQ、HornetQ、NATS 和 Apache Kafka 等开源实现变得流行。我们将在第 11 章更详细地比较它们。

**分布式参与者框架**

参与者模型是单进程中并发的编程模型。与其直接处理线程（以及竞争条件、锁定和死锁的相关问题），逻辑被封装在参与者中。每个参与者通常代表一个客户端或实体，它可能有一些本地状态（不与任何其他参与者共享），它通过发送和接收异步消息与其他参与者通信。消息传递不保证：在某些错误场景中，消息会丢失。由于每个参与者一次只处理一条消息，它不需要担心线程，每个参与者可以由框架独立调度。

在分布式参与者框架中，此编程模型用于在多个节点上扩展应用。使用相同的消息传递机制，无论发送者和接收者是在同一节点还是不同节点上。如果它们在不同节点上，消息被透明地编码为字节序列，通过网络发送，并在另一端解码。

## 小结

在本章中，我们研究了几种将数据结构转换为网络上或磁盘上的字节的方式。我们看到了这些编码的细节如何不仅影响其效率，而且更重要的是影响应用的架构和你部署它们的选项。

特别是，许多服务需要支持滚动升级，其中服务的新版本逐渐一次部署到几个节点，而不是同时部署到所有节点。滚动升级允许在不停机的情况下发布服务的新版本（从而鼓励频繁的小发布而不是罕见的大发布），并使部署风险更低（允许在故障发布影响大量用户之前检测和回滚它们）。这些属性对可演化性——对应用进行更改的容易程度——非常有益。

在滚动升级期间，或出于各种其他原因，我们必须假设不同的节点正在运行我们应用代码的不同版本。因此，重要的是系统中流动的所有数据都以提供向后兼容（新代码可以读取旧数据）和向前兼容（旧代码可以读取新数据）的方式编码。

我们讨论了几种数据编码格式及其兼容性属性：

- **编程语言特定的编码**仅限于单一编程语言，通常无法提供向前和向后兼容。
- **像 JSON、XML 和 CSV 这样的文本格式**广泛使用，其兼容性取决于你如何使用它们。它们有可选的模式语言，有时有帮助有时是障碍。这些格式在数据类型方面有些模糊，因此你必须小心处理数字和二进制字符串等事情。
- **像 Thrift、Protocol Buffers 和 Avro 这样的二进制模式驱动格式**允许紧凑、高效的编码，具有明确定义的向前和向后兼容语义。模式对于静态类型语言中的文档和代码生成很有用。然而，它们的缺点是数据需要解码才能人类可读。

我们还讨论了几种数据流模式，说明了数据编码重要的不同场景：

- **数据库**，其中写入数据库的进程编码数据，从数据库读取的进程解码它
- **RPC 和 REST API**，其中客户端编码请求，服务器解码请求并编码响应，客户端最终解码响应
- **异步消息传递**（使用消息代理或参与者），其中节点通过相互发送由发送者编码并由接收者解码的消息进行通信

我们可以得出结论，只要稍加注意，向后/向前兼容和滚动升级是完全可以实现的。愿你的应用演化迅速，部署频繁。
