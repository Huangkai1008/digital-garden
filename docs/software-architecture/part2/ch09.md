# 第9章 基础

架构风格（architecture style），有时也称为架构模式（architecture pattern），描述了一组命名化的组件关系，涵盖多种架构特性。架构风格名称类似于设计模式，为经验丰富的架构师之间创造了一个简短的共同语言。例如，当架构师谈论分层单体（layered monolith）时，对话对象会理解其结构方面、哪些架构特性表现良好（以及哪些可能引发问题）、典型的部署模型、数据策略等诸多信息。因此，架构师应熟悉基础通用架构风格的基本名称。

每个名称都承载了大量约定俗成的细节，这也是设计模式的目的之一。架构风格描述了拓扑结构、假定与默认的架构特性，既有有益的也有不利的。本书第二部分余下内容将涵盖许多常见的现代架构模式。然而，架构师还应熟悉嵌入在更大模式中的若干基础模式。

## 基础模式

若干基础模式在软件架构史上反复出现，因为它们为组织代码、部署或架构的其他方面提供了有用的视角。例如，架构中的分层概念——基于功能分离不同关注点——与软件本身一样古老。然而，分层模式继续以不同形式呈现，包括第 10 章讨论的现代变体。

### 大泥球（Big Ball of Mud）

架构师将缺乏任何可辨识架构结构的情况称为**大泥球（Big Ball of Mud）**，得名于 Brian Foote 和 Joseph Yoder 在 1997 年发表论文中定义的同名反模式：

> 大泥球是一种杂乱无章、蔓延无序、粗制滥造、胶带铁丝、意大利面代码的丛林。这些系统展现出不受约束的增长和反复权宜修补的明显迹象。信息在系统遥远元素之间随意共享，往往到了几乎所有重要信息都变成全局或重复的地步。
>
> 系统的整体结构可能从未被良好定义。即便曾经定义过，也可能已经侵蚀到面目全非。但凡有一点架构意识的程序员都会避开这些泥潭。只有那些不关心架构、或许安于日复一日修补这些溃堤漏洞的人，才会满足于在这样的系统上工作。
>
> —Brian Foote 和 Joseph Yoder

用现代术语来说，大泥球可能描述的是事件处理器直接连到数据库调用、没有真正内部结构的简单脚本应用。许多琐碎应用一开始如此，随着持续增长变得难以驾驭。

一般而言，架构师应不惜代价避免这类架构。缺乏结构会使变更越来越困难。这类架构在部署、可测试性、可扩展性和性能方面也存在问题。

不幸的是，这种架构反模式在现实中相当常见。很少有架构师有意为之，但许多项目因缺乏对代码质量和结构的治理而不经意间制造了混乱。例如，Neal 曾与一个客户项目合作，其结构如图 9-1 所示。该客户（出于显而易见的原因隐去其名）在数年内尽可能快地创建了一个基于 Java 的 Web 应用。技术可视化展示了他们的架构耦合：圆周长上的每个点代表一个类，每条线代表类之间的连接，线条越粗表示连接越强。在此代码库中，对任何类的修改都难以预测对其他类的连锁副作用，使变更成为令人胆寒的事。

### 单一架构（Unitary Architecture）

软件诞生之初，只有计算机，软件在其上运行。随着硬件和软件演进的各个时代，二者起初是单一实体，随后因更复杂能力的需求而分离。例如，大型机最初是单一系统，后来逐渐将数据分离到独立的系统。同样，个人计算机刚出现时，大量商业开发聚焦于单机。随着联网 PC 普及，分布式系统（如客户端/服务器）应运而生。

除嵌入式系统和其他高度受限环境外，单一架构已很少见。一般而言，软件系统会随时间增长功能，需要关注点分离以维持性能、规模等运行架构特性。

### 客户端/服务器（Client/Server）

随着时间推移，各种力量要求从单一系统分离；如何做到这一点构成了许多架构风格的基础。许多架构风格涉及如何有效分离系统的各个部分。

架构中的一种基础风格将技术功能分离于前端与后端之间，称为**两层架构**或**客户端/服务器（client/server）架构**。根据时代和计算能力，存在多种不同形态。

**桌面 + 数据库服务器**

早期的个人计算机架构鼓励开发者在 Windows 等用户界面中编写富桌面应用，将数据分离到独立的数据库服务器。这种架构与可通过标准网络协议连接的独立数据库服务器的出现相吻合。它允许展示逻辑驻留在桌面，而计算量更大（无论从量还是复杂度）的操作发生在更强大的数据库服务器上。

**浏览器 + Web 服务器**

现代 Web 开发到来后，常见的分离变为 Web 浏览器连接到 Web 服务器（后者又连接到数据库服务器）。职责分离与桌面变体类似，但客户端更薄，作为浏览器，允许在防火墙内外更广泛地分发。尽管数据库与 Web 服务器分离，架构师通常仍将其视为两层架构，因为 Web 和数据库服务器在运维中心的一类机器上运行，而用户界面在用户的浏览器上运行。

**三层架构（Three-tier）**

1990 年代末期相当流行的架构是**三层架构**，提供了更多层次的分离。随着 Java 和 .NET 中应用服务器等工具流行，公司开始在拓扑中构建更多层：使用工业级数据库服务器的数据库层、由应用服务器管理的应用层、以生成 HTML 编码的前端，以及随着能力扩展越来越多的 JavaScript。

三层架构与**公共对象请求代理架构（CORBA）**和**分布式组件对象模型（DCOM）**等网络级协议相对应，促进了分布式架构的构建。

正如今天的开发者不必担心 TCP/IP 等网络协议如何工作（它们就是能工作），大多数架构师也不必担心分布式架构中的这种底层管道。那个时代此类工具提供的能力，今天以工具（如消息队列）或架构模式（如第 14 章涵盖的事件驱动架构）的形式存在。

### 三层架构、语言设计与长期影响

在 Java 语言设计的时代，三层计算风靡一时。因此，人们假定未来所有系统都将是三层架构。C++ 等现有语言的常见痛点之一，是以一致方式在系统之间通过网络移动对象非常繁琐。因此，Java 的设计者决定使用称为**序列化（serialization）**的机制将这一能力内置到语言核心。Java 中的每个 Object 都实现一个要求支持序列化的接口。设计者认为既然三层架构将永远是架构风格，将其融入语言将带来极大便利。当然，那种架构风格来了又去，但遗留物至今仍出现在 Java 中，令想要添加现代特性、却因向后兼容必须支持几乎没人再用的序列化的语言设计者大为沮丧。

理解设计决策的长期影响一直困扰着我们，在软件中如此，在其他工程学科中亦然。 favoring 简单设计的永恒建议，在许多方面是对未来后果的防御。

## 单体与分布式架构（Monolithic Versus Distributed Architectures）

架构风格可分为两大类型：**单体（monolithic）**（所有代码的单一部署单元）和**分布式（distributed）**（通过远程访问协议连接的多个部署单元）。虽然没有任何分类方案是完美的，但分布式架构都共享一组单体架构风格所没有的共同挑战和问题，使这一分类成为各种架构风格之间的良好划分。本书将详细描述以下架构风格：

**单体**
- 分层架构（第 10 章）
- 管道架构（第 11 章）
- 微内核架构（第 12 章）

**分布式**
- 基于服务的架构（第 13 章）
- 事件驱动架构（第 14 章）
- 基于空间的架构（第 15 章）
- 服务导向架构（第 16 章）
- 微服务架构（第 17 章）

分布式架构风格在性能、可扩展性和可用性方面比单体架构风格强大得多，但为此付出了重大权衡。所有分布式架构面临的第一组问题由**分布式计算的谬误（fallacies of distributed computing）**描述，由 L. Peter Deutsch 和 Sun Microsystems 的其他同事于 1994 年首次提出。谬误是指被认为或假定为真但实际并非如此的事物。分布式计算的全部八条谬误至今仍适用于分布式架构。以下各节描述每条谬误。

### 谬误 #1：网络是可靠的（The Network Is Reliable）

开发者和架构师都假定网络是可靠的，但它不是。虽然网络随时间变得更加可靠，但事实是网络总体上仍然不可靠。这对所有分布式架构都至关重要，因为所有分布式架构风格都依赖网络进行与服务的通信以及服务之间的通信。如图 9-2 所示，服务 B 可能完全健康，但服务 A 因网络问题无法到达它；更糟的是，服务 A 向服务 B 发出处理某些数据的请求，却因网络问题未收到响应。这就是服务之间存在超时和熔断器的原因。系统越依赖网络（如微服务架构），潜在可靠性就越低。

### 谬误 #2：延迟为零（Latency Is Zero）

如图 9-3 所示，当通过方法或函数调用向另一组件发起本地调用时，该时间（t_local）以纳秒或微秒计。然而，当通过 REST、消息或 RPC 等远程访问协议发起相同调用时，访问该服务的时间（t_remote）以毫秒计。因此，t_remote 始终大于 t_local。

任何分布式架构中的延迟都不是零，但大多数架构师忽视这一谬误，坚称他们的网络很快。问自己这个问题：你知道生产环境中 RESTful 调用的平均往返延迟是多少吗？是 60 毫秒？还是 500 毫秒？

使用任何分布式架构时，架构师必须了解这一平均延迟。这是判断分布式架构是否可行的唯一途径，尤其考虑微服务（见第 17 章）时，因其服务的细粒度性质及服务间的通信量。假设每次请求平均延迟 100 毫秒，将 10 次服务调用串联以执行特定业务功能，就会为请求增加 1,000 毫秒！

了解平均延迟很重要，但更重要的是了解第 95 到第 99 百分位。平均延迟可能只有 60 毫秒（这很好），但第 95 百分位可能是 400 毫秒！通常是这种「长尾」延迟会扼杀分布式架构的性能。在大多数情况下，架构师可以从网络管理员处获取延迟值（见第 129 页「谬误 #6：只有一个管理员」）。

### 谬误 #3：带宽是无限的（Bandwidth Is Infinite）

在单体架构中，带宽通常不是问题，因为一旦处理进入单体，处理该业务请求几乎不需要带宽。然而，如图 9-4 所示，一旦系统在微服务等分布式架构中被拆分为更小的部署单元（服务），与这些服务的通信以及它们之间的通信会显著消耗带宽，导致网络变慢，从而影响延迟（谬误 #2）和可靠性（谬误 #1）。

为说明这一谬误的重要性，考虑图 9-4 中的两个服务。假设左侧服务管理网站的愿望清单项，右侧服务管理客户档案。每当愿望清单请求进入左侧服务时，它必须向右侧客户档案服务发起服务间调用以获取客户姓名，因为该数据是愿望清单响应契约所需的，但左侧的愿望清单服务没有该姓名。客户档案服务向愿望清单服务返回 45 个属性共 500 KB，而后者只需要姓名（200 字节）。这是一种称为**戳记耦合（stamp coupling）**的耦合形式。这可能听起来不显著，但愿望清单项的请求每秒约 2,000 次。这意味着从愿望清单服务到客户档案服务的这一服务间调用每秒发生 2,000 次。每次请求 500 KB，仅这一服务间调用（在每秒数百次调用中）使用的带宽就是 1 GB！

分布式架构中的戳记耦合消耗大量带宽。如果客户档案服务只传回愿望清单服务所需的数据（本例中 200 字节），传输数据的总带宽仅为 400 KB。戳记耦合可通过以下方式解决：

- 创建私有 RESTful API 端点
- 在契约中使用字段选择器
- 使用 GraphQL 解耦契约
- 使用消费者驱动契约（CDC）的价值驱动契约
- 使用内部消息端点

无论使用何种技术，确保在分布式架构中服务或系统之间传递最少量的数据是应对这一谬误的最佳方式。

### 谬误 #4：网络是安全的（The Network Is Secure）

大多数架构师和开发者在使用虚拟专用网络（VPN）、可信网络和防火墙时过于舒适，往往忘记分布式计算的这一谬误：**网络不安全**。在分布式架构中，安全变得更具挑战性。如图 9-5 所示，每个分布式部署单元的每个端点都必须得到保护，以使未知或恶意请求无法到达该服务。从单体转向分布式架构时，威胁和攻击的表面积成倍增加。即使在进行服务间通信时也必须保护每个端点，是同步、高度分布式架构（如微服务或基于服务的架构）性能往往较慢的另一个原因。

### 谬误 #5：拓扑永不改变（The Topology Never Changes）

这一谬误指整体网络拓扑，包括整体网络中使用的所有路由器、集线器、交换机、防火墙、网络和设备。架构师假定拓扑是固定的、永不改变。当然它会变。它一直在变。这一谬误的意义是什么？

假设架构师周一早上上班，所有人忙得团团转，因为生产环境中的服务不断超时。架构师与团队合作，拼命想找出原因。周末没有部署新服务。会是什么？几小时后，架构师发现凌晨 2 点进行了一次小型网络升级。这次所谓的「小型」网络升级使所有延迟假设失效，触发了超时和熔断器。

架构师必须与运维和网络管理员保持持续沟通，了解什么在变、何时变，以便相应调整，减少前述类型的意外。这看似显而易见且容易，实则不然。事实上，这一谬误直接引向下一条谬误。

### 谬误 #6：只有一个管理员（There Is Only One Administrator）

架构师经常陷入这一谬误，假定只需与一位管理员协作和沟通。如图 9-7 所示，典型大公司有数十位网络管理员。关于延迟（第 125 页「谬误 #2：延迟为零」）或拓扑变更（第 128 页「谬误 #5：拓扑永不改变」），架构师应该与谁沟通？这一谬误指向分布式架构的复杂性以及为正确运行一切所需的协调量。单体应用由于这些架构风格的单一部署单元特性，不需要这种级别的沟通和协作。

### 谬误 #7：传输成本为零（Transport Cost Is Zero）

许多软件架构师将这一谬误与延迟（第 125 页「谬误 #2：延迟为零」）混淆。此处的传输成本不是指延迟，而是与发起「简单 RESTful 调用」相关的实际金钱成本。架构师（错误地）假定必要的基础设施已就位且足以进行简单 RESTful 调用或拆分单体应用。通常并非如此。分布式架构的成本显著高于单体架构，主要由于对额外硬件、服务器、网关、防火墙、新子网、代理等的需求增加。

每当着手分布式架构时，我们鼓励架构师就容量、带宽、延迟和安全区域分析当前服务器和网络拓扑，以免陷入这一谬误的意外陷阱。

### 谬误 #8：网络是同质的（The Network Is Homogeneous）

大多数架构师和开发者假定网络是同质的——仅由一家网络硬件供应商组成。没有比这更偏离事实的了。大多数公司的基础设施中有多家网络硬件供应商，甚至更多。

那又怎样？这一谬误的意义在于，并非所有异构硬件供应商都能良好协作。大多数能工作，但 Juniper 硬件能与 Cisco 硬件无缝集成吗？网络标准多年来已演进，使这不再那么成问题，但事实是并非所有情况、负载和环境都经过充分测试，因此网络数据包偶尔会丢失。这进而影响网络可靠性（第 124 页「谬误 #1：网络是可靠的」）、延迟假设和断言（第 125 页「谬误 #2：延迟为零」）以及关于带宽的假设（第 126 页「谬误 #3：带宽是无限的」）。换句话说，这一谬误与所有其他谬误相连，在处理网络（使用分布式架构时必不可少）时形成无尽的困惑和沮丧循环。

## 其他分布式考量（Other Distributed Considerations）

除前述八条分布式计算谬误外，分布式架构还面临单体架构所没有的其他问题和挑战。虽然这些其他问题的细节超出本书范围，我们在以下各节中列出并总结它们。

### 分布式日志（Distributed Logging）

在分布式架构中，由于应用和系统日志的分布，执行根因分析以确定特定订单为何丢失非常困难且耗时。在单体应用中通常只有一个日志，更容易追踪请求并确定问题。然而，分布式架构包含数十到数百个不同的日志，位于不同位置且格式各异，难以追踪问题。

Splunk 等日志整合工具有助于将来自各种来源和系统的信息整合到一个统一的日志和控制台，但这些工具仅触及分布式日志所涉复杂性的表面。分布式日志的详细解决方案和模式超出本书范围。

### 分布式事务（Distributed Transactions）

架构师和开发者在单体架构世界中视事务为理所当然，因为它们非常直接且易于管理。由持久化框架执行的标准提交和回滚利用 **ACID**（原子性、一致性、隔离性、持久性）事务来保证数据以正确方式更新，确保高数据一致性和完整性。分布式架构则不然。

分布式架构依赖所谓的**最终一致性（eventual consistency）**，确保由独立部署单元处理的数据在某个未指定时间点全部同步到一致状态。这是分布式架构的权衡之一：以数据一致性和数据完整性为代价，获得高可扩展性、性能和可用性。

**事务性 Saga** 是管理分布式事务的一种方式。Saga 使用事件溯源进行补偿或有限状态机来管理事务状态。除 Saga 外，还使用 **BASE** 事务。BASE 代表 (B)asic 可用性、(S)oft 状态和 (E)ventual 一致性。BASE 事务不是软件，而是一种技术。BASE 中的软状态指数据从源到目标的传输，以及数据源之间的不一致。基于所涉系统或服务的基本可用性，系统将通过架构模式和消息传递最终变得一致。

### 契约维护与版本管理（Contract Maintenance and Versioning）

分布式架构中另一个特别困难的挑战是契约的创建、维护和版本管理。**契约（contract）** 是客户端和服务双方约定的行为和数据。契约维护在分布式架构中尤其困难，主要由于由不同团队和部门拥有的解耦服务和系统。版本弃用所需的通信模型更加复杂。
