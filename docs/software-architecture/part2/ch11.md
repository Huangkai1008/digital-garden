# 第11章 管道架构风格

管道架构（pipeline architecture）（也称为**管道与过滤器架构（pipes and filters architecture）**）是软件架构中反复出现的基础风格之一。一旦开发者和架构师决定将功能拆分为离散部分，这一模式就随之而来。大多数开发者知道这种架构是 Bash 和 Zsh 等 Unix 终端 shell 语言背后的基本原理。

许多函数式编程语言的开发者会看到语言构造与此架构元素之间的相似之处。事实上，许多使用 MapReduce 编程模型的工具都遵循这一基本拓扑。虽然这些示例展示了管道架构风格的低级实现，它也可用于更高级别的业务应用。

## 拓扑（Topology）

管道架构的拓扑由管道（pipes）和过滤器（filters）组成，如图 11-1 所示。

```
  [Producer] ──pipe──> [Transformer] ──pipe──> [Tester] ──pipe──> [Consumer]
     Figure 11-1. 管道架构的基本拓扑
```

管道和过滤器以特定方式协调，管道在过滤器之间形成单向通信，通常以点对点方式进行。

### 管道（Pipes）

此架构中的管道形成过滤器之间的通信通道。出于性能原因，每个管道通常是单向和点对点的（而非广播），从一个源接受输入并始终将输出导向另一个。管道上承载的有效载荷可以是任何数据格式，但架构师倾向于较小的数据量以实现高性能。

### 过滤器（Filters）

过滤器是自包含的、独立于其他过滤器的，通常是无状态的。过滤器应只执行一项任务。复合任务应由一系列过滤器而非单个过滤器处理。

此架构风格中存在四种类型的过滤器：

**生产者（Producer）**
流程的起点，仅出站，有时称为源（source）。

**转换器（Transformer）**
接受输入，可选地对部分或全部数据执行转换，然后转发到出站管道。函数式倡导者会将其识别为 map。

**测试器（Tester）**
接受输入，测试一个或多个条件，然后根据测试可选地产生输出。函数式程序员会将其识别为类似于 reduce。

**消费者（Consumer）**
管道流的终止点。消费者有时将管道过程的最终结果持久化到数据库，或者可能在用户界面屏幕上显示最终结果。

管道和过滤器各自的单向性质和简单性鼓励组合复用。许多开发者使用 shell 发现了这种能力。「More Shell, Less Egg」博客中的著名故事说明了这些抽象有多么强大。Donald Knuth 被要求编写程序解决此文本处理问题：读取文本文件，确定 n 个最常用的词，并打印这些词及其频率的排序列表。他写了一个超过 10 页 Pascal 的程序，沿途设计（并记录）了新算法。然后，Doug McIlroy 演示了一个可以轻松放入一条 Twitter 帖子的 shell 脚本，更简单、优雅、易懂地解决了问题（如果你理解 shell 命令）：

```bash
tr -cs A-Za-z '\n' |
tr A-Z a-z |
sort |
uniq -c |
sort -rn |
sed ${1}q
```

即使是 Unix shell 的设计者也常常对开发者用他们简单但强大的组合抽象创造的创造性用法感到惊讶。

## 示例（Example）

管道架构模式出现在各种应用中，尤其是促进简单单向处理的任务。例如，许多**电子数据交换（EDI）**工具使用此模式，使用管道和过滤器构建从一种文档类型到另一种的转换。**ETL 工具**（提取、转换、加载）也利用管道架构实现从一个数据库或数据源到另一个的数据流和修改。Apache Camel 等编排器和中介器利用管道架构将信息从业务流程的一个步骤传递到另一个。

为说明管道架构的使用方式，考虑以下示例，如图 11-2 所示，其中各种服务遥测信息通过流式传输从服务发送到 Apache Kafka。

```
  Kafka Topic
       │
       ▼
  [Service Info Capture] ──> [Duration Filter] ──> [Duration Calculator]
       │                            │
       │                            └──> [Uptime Filter] ──> [Uptime Calculator]
       │                                                              │
       └──────────────────────────────────────────────────────────────┴──> [Database Output] ──> MongoDB
     Figure 11-2. 管道架构示例
```

注意图 11-2 中使用管道架构风格处理流向 Kafka 的不同类型数据流。**Service Info Capture** 过滤器（生产者过滤器）订阅 Kafka 主题并接收服务信息。然后它将捕获的数据发送到名为 **Duration Filter** 的测试器过滤器，以确定从 Kafka 捕获的数据是否与服务请求的持续时间（毫秒）相关。注意过滤器之间的关注点分离；Service Metrics Capture 过滤器只关心如何连接到 Kafka 主题并接收流数据，而 Duration Filter 只关心限定数据并可选地将其路由到下一个管道。如果数据与服务请求的持续时间（毫秒）相关，则 Duration Filter 将数据传递给 **Duration Calculator** 转换器过滤器。否则，它将其传递给 **Uptime Filter** 测试器过滤器以检查数据是否与正常运行时间指标相关。如果不是，则管道结束——该数据对此特定处理流不感兴趣。否则，如果是正常运行时间指标，则将其传递给 Uptime Calculator 以计算服务的正常运行时间指标。这些转换器然后将修改后的数据传递给 **Database Output** 消费者，后者将数据持久化到 MongoDB 数据库。

此示例展示了管道架构的可扩展性。例如，在图 11-2 中，可以轻松在 Uptime Filter 之后添加新的测试器过滤器，将数据传递给另一个新收集的指标，如数据库连接等待时间。

## 架构特性评分（Architecture Characteristics Ratings）

图 11-3 特性评分表中的一星评分表示该特定架构特性在架构中未得到良好支持，而五星评分表示该架构特性是架构风格中最强的特性之一。记分卡中识别的每个特性的定义可在第 4 章找到。

由于应用逻辑按过滤器类型（生产者、测试器、转换器、消费者）分区，管道架构风格是一种技术分区架构。此外，由于管道架构通常作为单体部署实现，架构量子始终为一。

| 特性 | 评分 |
|------|------|
| 总体成本 | ★★★★★ |
| 简单性 | ★★★★★ |
| 模块化 | ★★★★ |
| 可部署性 | ★★ |
| 可测试性 | ★★★ |
| 可靠性 | ★★★ |
| 弹性 | ★ |
| 可扩展性 | ★ |
| 容错 | ★ |
| 可用性 | ★★ |

总体成本和简单性结合模块化是管道架构风格的主要优势。由于本质上是单体，管道架构没有与分布式架构风格相关的复杂性，简单易懂，构建和维护成本相对较低。通过各种过滤器类型和转换器之间的关注点分离实现架构模块化。这些过滤器中的任何一个都可以修改或替换而不影响其他过滤器。例如，在图 11-2 所示的 Kafka 示例中，可以修改 Duration Calculator 以更改持续时间计算，而不影响任何其他过滤器。

可部署性和可测试性虽然仅为中等，但由于通过过滤器实现的模块化水平，评分略高于分层架构。也就是说，此架构风格仍然是单体，因此，繁琐、风险、部署频率和测试完整性仍然影响管道架构。

与分层架构一样，由于大多数分布式架构中缺乏网络流量、带宽和延迟，整体可靠性在此架构风格中为中等（三星）。我们只给它可靠性三星，是因为此架构风格的单体部署性质，加上可测试性和可部署性问题（如必须测试整个单体并为任何给定变更部署整个单体）。

由于单体部署，管道架构的弹性和可扩展性非常低（一星）。虽然可以使单体内的某些功能比其他功能扩展更多，但这通常需要非常复杂的设计技术，如多线程、内部消息传递和其他并行处理实践，此架构风格并不适合这些技术。然而，由于管道架构由于单体用户界面、后端处理和单体数据库始终是单一系统量子，应用只能基于单一架构量子扩展到一定程度。

由于单体部署和缺乏架构模块化，管道架构不支持容错。如果管道架构的一小部分导致内存不足情况发生，整个应用单元都会受到影响并崩溃。此外，由于大多数单体应用通常经历的高平均恢复时间（MTTR），整体可用性受到影响，启动时间从较小应用的 2 分钟到大多数大型应用的 15 分钟或更长不等。
