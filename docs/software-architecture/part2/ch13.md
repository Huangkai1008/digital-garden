# 第13章 基于服务的架构风格

基于服务的架构（service-based architecture）是微服务架构风格的混合体，被认为是最务实的架构风格之一，主要因其架构灵活性。虽然基于服务的架构是分布式架构，但它没有微服务或事件驱动架构等其他分布式架构的复杂度和成本，使其成为许多业务相关应用的非常流行的选择。

## 拓扑（Topology）

基于服务架构的基本拓扑遵循分布式宏观分层结构，由独立部署的用户界面、独立部署的远程粗粒度服务和单体数据库组成。此基本拓扑如图 13-1 所示。

```
┌─────────────────────────────────────────────────────────┐
│              User Interface (单体或联邦)                  │
└─────────────────────────────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ Domain Svc 1 │   │ Domain Svc 2 │   │ Domain Svc N │  ...
└──────────────┘   └──────────────┘   └──────────────┘
        └───────────────────┼───────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│              Monolithic Database                         │
└─────────────────────────────────────────────────────────┘
     Figure 13-1. 基于服务架构风格的基本拓扑
```

此架构风格中的服务通常是粗粒度的「应用的一部分」（通常称为**领域服务（domain services）**），独立且单独部署。服务通常以任何单体应用相同的方式部署（如 EAR 文件、WAR 文件或程序集），因此不需要容器化（尽管你可以在 Docker 等容器中部署领域服务）。由于服务通常共享单一单体数据库，应用上下文内的服务数量一般在 4 到 12 个之间，平均约 7 个服务。

在大多数情况下，基于服务的架构中每个领域服务只有一个实例。然而，根据可扩展性、容错和吞吐量需求，领域服务的多个实例当然可以存在。服务的多个实例通常需要在用户界面和领域服务之间某种负载均衡能力，以便用户界面可以定向到健康可用的服务实例。

使用远程访问协议从用户界面远程访问服务。虽然通常使用 REST 从用户界面访问服务，但也可以使用消息传递、**远程过程调用（RPC）**甚至 SOAP。虽然可以使用由代理或网关组成的 API 层从用户界面（或其他外部请求）访问服务，但在大多数情况下，用户界面使用嵌入在用户界面、API 网关或代理中的**服务定位器模式（service locator pattern）**直接访问服务。

基于服务架构的一个重要方面是它通常使用中央共享数据库。这允许服务以与传统单体分层架构相同的方式利用 SQL 查询和连接。由于服务数量少（4 到 12 个），数据库连接在基于服务的架构中通常不是问题。然而，数据库变更可能是个问题。第 169 页的「数据库分区」部分描述了在基于服务的架构中解决和管理数据库变更的技术。

## 拓扑变体（Topology Variants）

基于服务的架构风格中存在许多拓扑变体，使其可能成为最灵活的架构风格之一。例如，图 13-1 所示的单一单体用户界面可以拆分为用户界面领域，甚至达到与每个领域服务匹配的级别。这些用户界面变体如图 13-2 所示。

同样，可能存在将单一单体数据库拆分为独立数据库的机会，甚至达到与每个领域服务匹配的领域范围数据库（类似于微服务）。在这些情况下，重要的是确保每个独立数据库中的数据不被另一个领域服务需要。这避免了领域服务之间的服务间通信（基于服务的架构中务必避免的）以及数据库之间的数据重复。这些数据库变体如图 13-3 所示。

最后，也可以在用户界面和服务之间添加由反向代理或网关组成的 API 层，如图 13-4 所示。当向外部系统暴露领域服务功能时，或当整合共享横切关注点并将其移出用户界面时（如指标、安全、审计要求和服务发现），这是良好实践。

## 服务设计与粒度（Service Design and Granularity）

由于基于服务架构中的领域服务通常是粗粒度的，每个领域服务通常使用由 API 门面层、业务层和持久层组成的分层架构风格设计。另一种流行的设计方法是使用与模块化单体架构风格类似的子领域对每个领域服务进行领域分区。这些设计方法如图 13-5 所示。

无论服务设计如何，领域服务必须包含某种用户界面与之交互以执行某种业务功能的 API 访问门面。API 访问门面通常承担编排来自用户界面的业务请求的责任。例如，考虑来自用户界面的下订单（也称为目录结账）的业务请求。此单一请求由 OrderService 领域服务内的 API 访问门面接收，在内部编排单一业务请求：下订单、生成订单 ID、应用支付、更新所订每个产品的产品库存。在微服务架构风格中，这可能涉及编排许多独立部署的远程单一用途服务来完成请求。内部类级编排与外部服务编排之间的这种差异指向基于服务的架构与微服务在粒度方面的许多显著差异之一。

由于领域服务是粗粒度的，使用涉及数据库提交和回滚的常规 **ACID**（原子性、一致性、隔离性、持久性）数据库事务来确保单一领域服务内的数据库完整性。另一方面，像微服务这样的高度分布式架构通常有细粒度服务，使用称为 **BASE 事务**（基本可用性、软状态、最终一致性）的分布式事务技术，依赖最终一致性，因此不支持与基于服务架构中 ACID 事务相同级别的数据库完整性。

为说明这一点，考虑基于服务架构中的目录结账流程示例。假设客户下订单，用于支付的信用卡已过期。由于这是同一服务内的原子事务，添加到数据库的所有内容都可以使用回滚移除，并向客户发送通知，说明无法应用支付。现在考虑具有较小细粒度服务的微服务架构中的相同流程。首先，OrderPlacement 服务将接受请求、创建订单、生成订单 ID 并将订单插入订单表。完成后，订单服务将向 PaymentService 发出远程调用，后者将尝试应用支付。如果由于信用卡过期无法应用支付，则无法下订单，数据处于不一致状态（订单信息已插入但未批准）。在这种情况下，该订单的库存怎么办？应该标记为已订购并减少吗？如果库存低而另一个客户想购买该商品怎么办？应该允许该新客户购买，还是应该为试图用过期信用卡下订单的客户保留预留库存？这些只是在使用多个细粒度服务编排业务流程时需要解决的几个问题。

领域服务是粗粒度的，允许更好的数据完整性和一致性，但有权衡。使用基于服务的架构，对 OrderService 中订单放置功能的变更需要测试整个粗粒度服务（包括支付处理），而使用微服务，相同变更只会影响小型 OrderPlacement 服务（不需要对 PaymentService 进行变更）。此外，由于部署的代码更多，基于服务的架构有更多可能破坏某些东西（包括支付处理）的风险，而微服务中每个服务有单一职责，因此变更时破坏其他功能的机会更少。

## 数据库分区（Database Partitioning）

虽然不要求，但由于给定应用上下文中服务数量少（4 到 12 个），基于服务架构中的服务通常共享单一单体数据库。这种数据库耦合可能在数据库表模式变更方面带来问题。如果做得不当，表模式变更可能影响每个服务，使数据库变更在工作和协调方面成为非常昂贵的任务。

在基于服务的架构中，表示数据库表模式的共享类文件（通常称为实体对象）驻留在所有领域服务使用的自定义共享库中（如 JAR 文件或 DLL）。共享库可能还包含 SQL 代码。创建单一实体对象共享库的做法是实现基于服务架构的最低效方式。对数据库表结构的任何变更还需要对包含所有相应实体对象的单一共享库进行变更，因此需要对每个服务进行变更和重新部署，无论服务是否实际访问已变更的表。共享库版本管理可以帮助解决此问题，但尽管如此，使用单一共享库很难知道哪些服务实际受到表变更的影响，而无需手动详细分析。此单一共享库场景如图 13-6 所示。

减轻数据库变更影响和风险的一种方法是对数据库进行逻辑分区，并通过联邦共享库体现逻辑分区。注意图 13-7 中数据库被逻辑分区为五个独立领域（common、customer、invoicing、order 和 tracking）。还要注意领域服务使用的五个相应共享库与数据库中的逻辑分区匹配。使用此技术，对特定逻辑领域（本例中为 invoicing）内表的变更与包含实体对象（可能还有 SQL）的相应共享库匹配，仅影响使用该共享库的服务，本例中为开票服务。没有其他服务受到此变更的影响。

注意图 13-7 中 common 领域的使用以及所有服务使用的相应 common_entities_lib 共享库。这是相对常见的情况。这些表对所有服务是共同的，因此，对这些表的变更需要协调访问共享数据库的所有服务。减轻对这些表（及相应实体对象）变更的一种方法是在版本控制系统中锁定 common 实体对象，并限制只有数据库团队可以变更。这有助于控制变更并强调对所有服务使用的共同表变更的重要性。

::: tip 建议
在基于服务的架构中，尽可能使数据库中的逻辑分区细粒度，同时保持良好定义的数据领域，以更好地控制数据库变更。
:::

## 示例架构（Example Architecture）

为说明基于服务架构风格的灵活性和能力，考虑用于回收旧电子设备（如 iPhone 或 Galaxy 手机）的电子回收系统的真实示例。回收旧电子设备的处理流程如下：首先，客户通过网站或自助服务亭询问公司他们可以为旧电子设备获得多少钱（称为报价）。如果满意，客户将电子设备寄给回收公司，回收公司随后接收实体设备（称为接收）。收到后，回收公司将评估设备以确定设备是否处于良好工作状态（称为评估）。如果设备处于良好工作状态，公司将向客户寄送承诺的设备款项（称为核算）。在此过程中，客户可以随时访问网站查看物品状态（称为物品状态）。根据评估，设备然后通过安全销毁或转售进行回收（称为回收）。最后，公司根据回收活动定期运行临时和计划的财务及运营报告（称为报告）。

图 13-8 使用基于服务的架构展示了此系统。注意前述描述中识别的每个领域区域如何实现为独立部署的独立领域服务。可以通过仅扩展需要更高吞吐量的服务（本例中为面向客户的 Quoting 服务和 ItemStatus 服务）来实现可扩展性。其他服务不需要扩展，因此只需要单一服务实例。

还要注意用户界面应用如何联邦到各自的领域：面向客户、接收、回收和核算。这种联邦允许用户界面的容错、可扩展性和安全性（外部客户没有到内部功能的网络路径）。最后，注意此示例中有两个独立的物理数据库：一个用于外部面向客户的操作，一个用于内部操作。这允许内部数据和操作驻留在与外部操作分开的网络区域（由垂直线表示），提供更好的安全访问限制和数据保护。通过防火墙的单向访问允许内部服务访问和更新面向客户的信息，但反之则不行。或者，根据使用的数据库，也可以使用内部表镜像和表同步。

## 架构特性评分（Architecture Characteristics Ratings）

图 13-9 特性评分表中的一星评分表示该特定架构特性在架构中未得到良好支持，而五星评分表示该架构特性是架构风格中最强的特性之一。记分卡中识别的每个特性的定义可在第 4 章找到。

基于服务的架构是**领域分区架构（domain-partitioned architecture）**，意味着结构由领域驱动而非技术考虑（如展示逻辑或持久逻辑）。考虑前述电子回收应用示例。每个服务作为独立部署的软件单元，范围限定于特定领域（如物品评估）。在此领域内所做的变更仅影响特定服务、相应的用户界面和相应的数据库。不需要修改其他任何东西来支持特定的评估变更。

作为分布式架构，量子数量可以大于或等于一。尽管可能有 4 到 12 个独立部署的服务，如果这些服务都共享相同的数据库或用户界面，整个系统将只是单一量子。然而，如第 165 页「拓扑变体」所示，用户界面和数据库都可以联邦，导致整个系统内有多个量子。在电子回收示例中，系统包含两个量子，如图 13-10 所示：一个用于应用的面向客户部分，包含独立的客户用户界面、数据库和服务集（Quoting 和 Item Status）；一个用于接收、评估和回收电子设备的内部操作。注意，尽管内部操作量子包含独立部署的服务和两个独立的用户界面，它们都共享相同的数据库，使应用的内部操作部分成为单一量子。

虽然基于服务的架构不包含任何五星评分，但它在许多重要和关键领域仍然评分很高（四星）。使用此架构风格将应用拆分为独立部署的领域服务，允许更快的变更（敏捷性）、由于领域范围有限而更好的测试覆盖率（可测试性），以及比大型单体更频繁、风险更低的部署能力（可部署性）。这三个特性带来更好的上市时间，允许组织以相对高的速率交付新功能和错误修复。

基于服务的架构的容错和整体应用可用性也评分很高。尽管领域服务往往是粗粒度的，四星评分来自这样一个事实：使用此架构风格，服务通常是自包含的，由于数据库共享和代码共享而不利用服务间通信。因此，如果一个领域服务宕机（如电子回收应用示例中的 Receiving 服务），它不会影响其他六个服务中的任何一个。

由于服务的粗粒度性质，可扩展性只评三星，相应地，弹性只评两星。虽然使用此架构风格当然可以实现程序化可扩展性和弹性，但比细粒度服务（如微服务）复制了更多功能，因此在机器资源方面效率不高，成本效益也不高。通常基于服务的架构只有单一服务实例，除非需要更好的吞吐量或故障转移。电子回收应用示例是一个很好的例子——只有 Quoting 和 Item Status 服务需要扩展以支持高客户量，但其他运营服务只需要单一实例，使支持单一内存缓存和数据库连接池等更容易。

简单性和总体成本是使此架构风格与其他更昂贵和复杂的分布式架构（如微服务、事件驱动架构甚至基于空间的架构）区分开来的另外两个驱动因素。这使基于服务的架构成为最容易实现且最具成本效益的分布式架构之一。虽然这是一个有吸引力的主张，但在所有包含四星评分的特性中，这种成本节约和简单性都有权衡。成本和复杂度越高，这些评分就越好。

由于领域服务的粗粒度性质，基于服务的架构往往比其他分布式架构更可靠。更大的服务意味着与服务的通信以及服务之间的网络流量更少、分布式事务更少、使用的带宽更少，因此提高了相对于网络的整体可靠性。

## 何时使用此架构风格（When to Use This Architecture Style）

此架构风格的灵活性（见第 165 页「拓扑变体」）结合三星和四星架构特性评分的数量，使基于服务的架构成为可用的最务实的架构风格之一。虽然肯定有其他更强大的分布式架构风格，但一些公司发现那种能力代价太高，而另一些公司发现他们根本不需要那么多能力。就像拥有法拉利的动力、速度和敏捷性，却只用于在高峰时段以每小时 50 公里的速度上下班——当然看起来很酷，但多么浪费资源和金钱！

基于服务的架构在进行领域驱动设计时也是天然契合。由于服务是粗粒度和领域范围的，每个领域都很好地契合到独立部署的领域服务中。基于服务架构中的每个服务涵盖特定领域（如电子回收应用中的回收），因此将该功能划分到单一软件单元中，使对该领域应用变更更容易。

维护和协调数据库事务始终是分布式架构的问题，因为它们通常依赖最终一致性而非传统的 ACID（原子性、一致性、隔离性和持久性）事务。然而，由于领域服务的粗粒度性质，基于服务的架构比任何其他分布式架构更好地保留 ACID 事务。在某些情况下，用户界面或 API 网关可能编排两个或多个领域服务，在这些情况下，事务将需要依赖 saga 和 BASE 事务。然而，在大多数情况下，事务范围限定于特定领域服务，允许大多数单体应用中发现的传统提交和回滚事务功能。

最后，基于服务的架构是在不陷入粒度复杂性和陷阱的情况下实现良好架构模块化水平的良好选择。随着服务变得更加细粒度，围绕编排和编排的问题开始出现。当必须协调多个服务以完成特定业务事务时，编排和编排都是必需的。**编排（orchestration）**是通过使用控制和管理事务工作流的独立中介服务来协调多个服务（如管弦乐队中的指挥）。**编排（choreography）**则是多个服务的协调，每个服务相互通信而不使用中央中介（如舞蹈中的舞者）。随着服务变得更加细粒度，编排和编排都是将服务联系在一起以完成业务事务所必需的。然而，由于基于服务架构中的服务往往更粗粒度，它们不需要像其他分布式架构那样多的协调。
