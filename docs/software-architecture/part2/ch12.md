# 第12章 微内核架构风格

微内核架构风格（microkernel architecture style）（也称为**插件架构（plug-in architecture）**）在几十年前提出，至今仍被广泛使用。这种架构风格天然适合基于产品的应用（打包并可供下载安装为单一单体部署，通常作为第三方产品安装在客户现场），但也广泛用于许多非产品的定制业务应用。

## 拓扑（Topology）

微内核架构风格是一种相对简单的单体架构，由两个架构组件组成：**核心系统（core system）**和**插件组件（plug-in components）**。应用逻辑在独立插件组件与基本核心系统之间划分，提供应用特性和自定义处理逻辑的可扩展性、适应性和隔离。图 12-1 展示了微内核架构风格的基本拓扑。

```
┌─────────────────────────────────────────────────────┐
│                    Core System                       │  核心系统
│  ┌─────────┐  ┌─────────┐  ┌─────────┐            │
│  │Plugin A │  │Plugin B │  │Plugin C │  ...        │  插件组件
│  └─────────┘  └─────────┘  └─────────┘            │
└─────────────────────────────────────────────────────┘
     Figure 12-1. 微内核架构风格的基本组件
```

### 核心系统（Core System）

核心系统正式定义为运行系统所需的最小功能。Eclipse IDE 是一个很好的例子。Eclipse 的核心系统只是一个基本文本编辑器：打开文件、修改文本、保存文件。直到添加插件，Eclipse 才开始成为可用的产品。然而，核心系统的另一种定义是应用中的**快乐路径（happy path）**（通用处理流），几乎没有或没有自定义处理。将核心系统的圈复杂度移除并放入独立的插件组件，可实现更好的可扩展性和可维护性，以及更高的可测试性。例如，假设电子设备回收应用必须对收到的每台电子设备执行特定的自定义评估规则。这类处理的 Java 代码可能如下所示：

```java
public void assessDevice(String deviceID) {
   if (deviceID.equals("iPhone6s")) {
      assessiPhone6s();
   } else if (deviceID.equals("iPad1"))
      assessiPad1();
   } else if (deviceID.equals("Galaxy5"))
      assessGalaxy5();
   } else ...
      ...
   }
}
```

与其将所有这种客户特定的自定义放在具有大量圈复杂度的核心系统中，不如为每台被评估的电子设备创建独立的插件组件。不仅特定的客户端插件组件将独立设备逻辑与其余处理流隔离，它们还允许可扩展性。添加要评估的新电子设备只需添加新插件组件并更新注册表。使用微内核架构风格，评估电子设备只需核心系统定位并调用相应的设备插件，如以下修订后的源代码所示：

```java
public void assessDevice(String deviceID) {
 String plugin = pluginRegistry.get(deviceID);
 Class<?> theClass = Class.forName(plugin);
 Constructor<?> constructor = theClass.getConstructor();
 DevicePlugin devicePlugin =
  (DevicePlugin)constructor.newInstance();
 DevicePlugin.assess();
}
```

在此示例中，评估特定电子设备的所有复杂规则和指令都自包含在独立的插件组件中，可从核心系统通用执行。

根据规模和复杂度，核心系统可以实现为分层架构或模块化单体（如图 12-2 所示）。在某些情况下，核心系统可以拆分为独立部署的领域服务，每个领域服务包含特定于该领域的插件组件。例如，假设支付处理是代表核心系统的领域服务。每种支付方式（信用卡、PayPal、店铺积分、礼品卡和采购订单）将是特定于支付领域的独立插件组件。在所有情况下，整个单体应用通常共享单一数据库。

核心系统的展示层可以嵌入核心系统内，或实现为独立的用户界面，核心系统提供后端服务。事实上，独立的用户界面也可以实现为微内核架构风格。图 12-3 展示了这些与核心系统相关的展示层变体。

### 插件组件（Plug-In Components）

插件组件是独立的组件，包含专业化处理、附加特性和旨在增强或扩展核心系统的自定义代码。此外，它们可用于隔离高度易变的代码，在应用内创造更好的可维护性和可测试性。理想情况下，插件组件应彼此独立，它们之间没有依赖关系。

插件组件与核心系统之间的通信通常是点对点的，意味着连接插件与核心系统的「管道」通常是对插件组件入口点类的方法调用或函数调用。此外，插件组件可以是**基于编译的（compile-based）**或**基于运行时的（runtime-based）**。运行时插件组件可以在运行时添加或移除，无需重新部署核心系统或其他插件，通常通过 **OSGi**（Java）、Penrose（Java）、Jigsaw（Java）或 Prism（.NET）等框架管理。基于编译的插件组件管理更简单，但修改、添加或移除时需要重新部署整个单体应用。

点对点插件组件可以实现为共享库（如 JAR、DLL 或 Gem）、Java 中的包名或 C# 中的命名空间。继续电子回收评估应用示例，每个电子设备插件可以编写并实现为 JAR、DLL 或 Ruby Gem（或任何其他共享库），设备名称与独立共享库的名称匹配，如图 12-4 所示。

或者，图 12-5 所示的更简单方法是在同一代码库或 IDE 项目内将每个插件组件实现为独立的命名空间或包名。创建命名空间时，我们建议以下语义：`app.plug-in.<domain>.<context>`。例如，考虑命名空间 `app.plugin.assessment.iphone6s`。第二个节点（plugin）明确表示此组件是插件，因此应严格遵守有关插件组件的基本规则（即它们是自包含的且与其他插件分离）。第三个节点描述领域（本例中为 assessment），从而允许插件组件按共同目的组织和分组。第四个节点（iphone6s）描述插件的特定上下文，便于定位特定设备插件进行修改或测试。

插件组件不必始终与核心系统进行点对点通信。存在其他替代方案，包括使用 REST 或消息传递作为调用插件功能的手段，每个插件是独立服务（甚至可能是使用容器实现的微服务）。尽管这听起来像是提高整体可扩展性的好方法，但请注意，由于单体核心系统，此拓扑（如图 12-6 所示）仍然只是单一架构量子。每个请求必须首先通过核心系统才能到达插件服务。

远程访问方式访问实现为独立服务的插件组件的好处是，它提供更好的整体组件解耦、允许更好的可扩展性和吞吐量，并允许运行时变更而无需 OSGi、Jigsaw 或 Prism 等特殊框架。它还允许与插件的异步通信，根据场景，这可能显著提高整体用户响应性。使用电子回收示例，核心系统可以发出异步请求以启动对特定设备的评估，而不必等待电子设备评估运行。当评估完成时，插件可以通过另一个异步消息通道通知核心系统，进而通知用户评估已完成。

有这些好处就有权衡。远程插件访问将微内核架构转变为分布式架构而非单体，使大多数第三方本地部署产品难以实现和部署。此外，它增加了整体复杂性和成本，并使整体部署拓扑复杂化。如果插件无响应或未运行，特别是使用 REST 时，请求无法完成。单体部署则不会如此。是否使从核心系统到插件组件的通信为点对点或远程，应基于特定需求，因此需要仔细权衡分析该方法的利弊。

插件组件直接连接到中央共享数据库不是常见做法。相反，核心系统承担此责任，将所需数据传递给每个插件。这种做法的主要原因是解耦。数据库变更应只影响核心系统，而非插件组件。也就是说，插件可以有自己的独立数据存储，仅该插件可访问。例如，电子回收系统示例中的每个电子设备评估插件可以有自己的简单数据库或规则引擎，包含每个产品的所有特定评估规则。插件组件拥有的数据存储可以是外部的（如图 12-7 所示），也可以作为插件组件或单体部署的一部分嵌入（如内存或嵌入式数据库的情况）。

### 注册表（Registry）

核心系统需要知道哪些插件模块可用以及如何访问它们。实现此目的的一种常见方式是通过**插件注册表（plug-in registry）**。此注册表包含有关每个插件模块的信息，包括其名称、数据契约和远程访问协议细节（取决于插件如何连接到核心系统）。例如，标记高风险税务审计项的税务软件插件可能有一个注册表条目，包含服务名称（AuditChecker）、数据契约（输入数据和输出数据）和契约格式（XML）。

注册表可以简单到核心系统拥有的内部映射结构，包含键和插件组件引用，也可以复杂到嵌入核心系统内或外部部署的注册和发现工具（如 Apache ZooKeeper 或 Consul）。使用电子回收示例，以下 Java 代码在核心系统内实现简单注册表，展示点对点条目、消息条目和 RESTful 条目的示例，用于评估 iPhone 6S 设备：

```java
Map<String, String> registry = new HashMap<String, String>();
static {
  // 点对点访问示例
  registry.put("iPhone6s", "Iphone6sPlugin");
  // 消息示例
  registry.put("iPhone6s", "iphone6s.queue");
  // RESTful 示例
  registry.put("iPhone6s", "https://atlas:443/assess/iphone6s");
}
```

### 契约（Contracts）

插件组件与核心系统之间的契约通常在插件组件领域内是标准的，包括行为、输入数据和插件组件返回的输出数据。自定义契约通常出现在插件组件由第三方开发、你无法控制插件使用的契约的情况。在这种情况下，常见做法是在插件契约与你的标准契约之间创建适配器，以便核心系统不需要为每个插件编写专用代码。

插件契约可以用 XML、JSON 实现，甚至可以用在插件与核心系统之间来回传递的对象实现。遵循电子回收应用，以下契约（实现为名为 AssessmentPlugin 的标准 Java 接口）定义了整体行为（assess()、register() 和 deregister()），以及插件组件预期的相应输出数据（AssessmentOutput）：

```java
public interface AssessmentPlugin {
 public AssessmentOutput assess();
 public String register();
 public String deregister();
}
public class AssessmentOutput {
 public String assessmentReport;
 public Boolean resell;
 public Double value;
 public Double resellPrice;
}
```

在此契约示例中，设备评估插件预期返回格式化的评估报告字符串；resell 标志（true 或 false）表示该设备是否可以在第三方市场上转售或安全处置；最后，如果可以转售（另一种回收形式），项目的计算价值是多少以及建议的转售价格应为多少。

注意此示例中核心系统与插件组件之间的角色和职责模型，特别是 assessmentReport 字段。核心系统的职责不是格式化和理解评估报告的细节，而只是打印或向用户显示。

## 示例与用例（Examples and Use Cases）

用于开发和发布软件的大多数工具都使用微内核架构实现。一些示例包括 Eclipse IDE、PMD、Jira 和 Jenkins 等。Chrome 和 Firefox 等互联网 Web 浏览器是使用微内核架构的另一个常见产品示例：查看器和其他插件添加了基本浏览器（代表核心系统）中原本没有的附加能力。基于产品的软件示例不胜枚举，但大型业务应用呢？微内核架构也适用于这些情况。为说明这一点，考虑涉及保险理赔处理的保险公司示例。

理赔处理是一个非常复杂的过程。每个司法管辖区对保险理赔中允许和不允许的内容有不同的规则和法规。例如，某些司法管辖区（如州）允许如果你的挡风玻璃被石头损坏则免费更换挡风玻璃，而其他州则不允许。这为标准理赔流程创造了几乎无限的条件集。

大多数保险理赔应用利用大型复杂的规则引擎来处理大部分这种复杂性。然而，这些规则引擎可能演变成复杂的大泥球，修改一条规则会影响其他规则，或者进行简单的规则变更需要大量分析师、开发者和测试人员来确保简单变更不会破坏任何东西。使用微内核架构模式可以解决许多这些问题。

每个司法管辖区的理赔规则可以包含在独立的插件组件中（实现为源代码或由插件组件访问的特定规则引擎实例）。这样，可以为特定司法管辖区添加、移除或更改规则，而不影响系统的任何其他部分。此外，可以添加和移除新司法管辖区，而不影响系统的其他部分。此示例中的核心系统将是提交和处理理赔的标准流程，这是不常变更的东西。

可以利用微内核架构的另一个大型复杂业务应用示例是税务准备软件。例如，美国有一个基本的两页税务表格，称为 1040 表格，包含计算个人纳税义务所需的所有信息的摘要。1040 税务表格中的每一行都有一个数字，需要许多其他表格和工作表才能得出该数字（如总收入）。这些附加表格和工作表中的每一个都可以实现为插件组件，1040 摘要税务表格是核心系统（驱动者）。这样，税法的变更可以隔离到独立的插件组件，使变更更容易、风险更低。

## 架构特性评分（Architecture Characteristics Ratings）

图 12-8 特性评分中的一星评分表示该特定架构特性在架构中未得到良好支持，而五星评分表示该架构特性是架构风格中最强的特性之一。记分卡中识别的每个特性的定义可在第 4 章找到。

与分层架构风格类似，简单性和总体成本是微内核架构风格的主要优势，可扩展性、容错和可扩展性是其主要弱点。这些弱点是由于微内核架构通常的单体部署。此外，与分层架构风格一样，量子数量始终为单一（一个），因为所有请求必须通过核心系统才能到达独立插件组件。相似之处到此为止。

微内核架构风格的独特之处在于，它是唯一可以同时进行领域分区和技术分区的架构风格。虽然大多数微内核架构是技术分区的，但领域分区方面主要来自强烈的领域与架构同构。例如，需要为每个位置或客户进行不同配置的问题与此架构风格极其匹配。另一个例子是强烈强调用户自定义和功能可扩展性的产品或应用（如 Jira 或 Eclipse 等 IDE）。

可测试性、可部署性和可靠性略高于平均水平（三星），主要是因为功能可以隔离到独立插件组件。如果做得正确，这会减少变更的整体测试范围，也会减少部署的整体风险，特别是如果插件组件以运行时方式部署时。

模块化和可扩展性也略高于平均水平（三星）。使用微内核架构风格，可以通过独立、自包含的插件组件添加、移除和更改附加功能，从而使使用此架构风格创建的应用相对容易扩展和增强，并允许团队更快地响应变更。考虑上一节的税务准备软件示例。如果美国税法变更（它一直在变），需要新的税务表格，该新税务表格可以作为插件组件创建并添加到应用中，无需太多努力。同样，如果不再需要税务表格或工作表，该插件可以简单地从应用中移除。

性能始终是微内核架构风格的一个有趣特性。我们给它三星（略高于平均水平），主要是因为微内核应用通常较小，不会像大多数分层架构那样增长得那么大。此外，它们不会像第 10 章讨论的架构天坑反模式那样受到那么多影响。最后，微内核架构可以通过拔掉不需要的功能来精简，从而使应用运行更快。Wildfly（以前的 JBoss 应用服务器）是一个很好的例子。通过拔掉集群、缓存和消息传递等不必要的功能，应用服务器比启用这些功能时运行得快得多。
