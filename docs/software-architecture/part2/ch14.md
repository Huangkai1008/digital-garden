# 第14章 事件驱动架构风格

事件驱动架构（event-driven architecture）风格是一种流行的分布式异步架构风格，用于构建高可扩展、高性能的应用。它高度适应性强，可用于小型应用以及大型复杂应用。事件驱动架构由解耦的事件处理组件组成，异步接收和处理事件。它可以作为独立架构风格使用，也可嵌入其他架构风格（如事件驱动微服务架构）。

大多数应用遵循所谓的**基于请求的模型（request-based model）**（见图 14-1）。在此模型中，向系统请求执行某种操作的请求发送给请求编排器。请求编排器通常是用户界面，但也可以通过 API 层或企业服务总线实现。请求编排器的角色是确定性地、同步地将请求定向到各种请求处理器。请求处理器处理请求，在数据库中检索或更新信息。

基于请求模型的一个好例子是客户请求检索过去六个月的订单历史。检索订单历史信息是对系统在特定上下文中获取数据的、数据驱动的、确定性请求，而非系统必须响应的事件。

另一方面，**基于事件的模型（event-based model）**对特定情况做出反应，并根据该事件采取行动。基于事件模型的例子是在线拍卖中为特定物品提交出价。提交出价不是对系统的请求，而是在当前要价公布后发生的事件。系统必须通过比较此出价与同时收到的其他出价来确定当前最高出价者，从而响应此事件。

## 拓扑（Topology）

事件驱动架构有两种主要拓扑：**中介者拓扑（mediator topology）**和**代理拓扑（broker topology）**。当你需要控制事件流程的工作流时，通常使用中介者拓扑；当你需要高度响应性和对事件处理的动态控制时，使用代理拓扑。由于这两种拓扑的架构特性和实现策略不同，理解每种拓扑以确定哪种最适合特定情况很重要。

### 代理拓扑（Broker Topology）

代理拓扑与中介者拓扑不同，没有中央事件中介者。相反，消息流以链式广播方式通过轻量级消息代理（如 RabbitMQ、ActiveMQ、HornetQ 等）分布在事件处理器组件之间。当你的事件处理流程相对简单且不需要中央事件编排和协调时，此拓扑很有用。

代理拓扑中有四个主要架构组件：**发起事件（initiating event）**、**事件代理（event broker）**、**事件处理器（event processor）**和**处理事件（processing event）**。发起事件是启动整个事件流的初始事件。发起事件发送到事件代理中的事件通道进行处理。由于代理拓扑中没有管理控制事件的中介者组件，单个事件处理器从事件代理接受发起事件并开始处理。接受发起事件的事件处理器执行与该事件处理相关的特定任务，然后通过创建称为处理事件的内容异步向系统其余部分广播其所作所为。处理事件然后异步发送到事件代理以供进一步处理（如需要）。其他事件处理器监听处理事件，通过执行某些操作来响应该事件，然后通过新的处理事件广播其所作所为。此过程持续到没有人对最终事件处理器所做的感兴趣为止。

事件代理组件通常是联邦的（意味着多个基于领域的集群实例），每个联邦代理包含该特定领域事件流中使用的所有事件通道。由于代理拓扑的解耦异步即发即弃广播性质，通常使用**发布-订阅（publish-and-subscribe）**消息模型中的主题（topic）。

在代理拓扑中，每个事件处理器向系统其余部分广播其所作所为始终是良好实践，无论是否有其他事件处理器关心该操作。如果该事件处理需要附加功能，这种做法提供架构可扩展性。

**代理拓扑的权衡**：高解耦、高可扩展性、高响应性、高性能、高容错 vs 工作流控制、错误处理、可恢复性、重启能力、数据一致性。

### 中介者拓扑（Mediator Topology）

事件驱动架构的中介者拓扑解决了上一节所述代理拓扑的一些不足。此拓扑的核心是**事件中介者（event mediator）**，它管理和控制需要多个事件处理器协调的发起事件的工作流。组成中介者拓扑的架构组件是发起事件、事件队列、事件中介者、事件通道和事件处理器。

与代理拓扑不同，中介者拓扑中的事件处理器不向系统其余部分广播其所作所为。中介者组件了解并控制工作流，可以维护事件状态并管理错误处理、可恢复性和重启能力。

事件中介者可以用多种方式实现，取决于其处理事件的性质和复杂度。对于需要简单错误处理和编排的事件，Apache Camel、Mule ESB 或 Spring Integration 等中介者通常足够。对于需要大量条件处理和多个动态路径及复杂错误处理指令的事件工作流，Apache ODE 或 Oracle BPEL Process Manager 等基于 **BPEL**（业务流程执行语言）的中介者是更好的选择。对于涉及整个事件过程中人工干预的长时间运行事务，需要 jBPM 等 **BPM**（业务流程管理）引擎。

**中介者拓扑的权衡**：工作流控制、错误处理、可恢复性、重启能力、更好的数据一致性 vs 事件处理器耦合更高、可扩展性较低、性能较低、容错较低、复杂工作流建模困难。

### 异步能力（Asynchronous Capabilities）

事件驱动架构风格相比其他架构风格提供独特特性：它完全依赖异步通信，既用于即发即弃处理（无需响应）也用于请求/回复处理（需要事件消费者响应）。异步通信可以是提高系统整体响应性的强大技术。

响应性与性能不同。当用户不需要任何信息返回（除确认或感谢消息外）时，为何让用户等待？响应性是关于通知用户操作已被接受并将立即处理，而性能是关于使端到端流程更快。

### 错误处理（Error Handling）

**工作流事件模式（workflow event pattern）**是响应式架构中解决异步工作流错误处理问题的一种方式。该模式利用**工作流委托（workflow delegate）**通过委托、遏制和修复来处理错误。事件消费者在处理数据时遇到错误，立即将错误委托给工作流处理器并继续处理队列中的下一条消息，从而不影响整体响应性。

### 防止数据丢失（Preventing Data Loss）

处理异步通信时，数据丢失始终是主要关注点。事件驱动架构中有许多可能发生数据丢失的地方。可通过以下基本技术缓解：**持久消息队列**与**同步发送**、**客户端确认模式（client acknowledge mode）**、**ACID 事务**与**最后参与者支持（LPS）**。

### 广播能力（Broadcast Capabilities）

事件驱动架构的另一个独特特性是能够广播事件而无需知道谁（如果有）在接收消息以及他们如何处理。广播可能是事件处理器之间最高级别的解耦。

### 请求-回复（Request-Reply）

当需要订单 ID 或确认号时，通过**请求-回复消息传递（request-reply messaging）**实现同步通信。实现方式包括**关联 ID（correlation ID）**和**临时队列（temporary queue）**。

### 在基于请求与基于事件之间选择（Choosing Between Request-Based and Event-Based）

我们建议对结构良好的、数据驱动的请求（如检索客户档案数据）选择基于请求的模型，当需要工作流的确定性和控制时。我们建议对需要高响应性和规模的灵活、基于操作的事件选择基于事件的模型，具有复杂和动态的用户处理。

### 混合事件驱动架构（Hybrid Event-Driven Architectures）

在许多情况下，事件驱动架构与其他架构风格结合使用，形成**混合架构（hybrid architecture）**。微服务和基于空间的架构都利用事件驱动架构作为其一部分。向任何架构风格添加事件驱动架构有助于消除瓶颈、在事件请求积压时提供背压点，并提供其他架构风格所没有的用户响应性水平。

## 架构特性评分（Architecture Characteristics Ratings）

事件驱动架构主要是技术分区架构，任何特定领域分布在多个事件处理器中，通过中介者、队列和主题联系在一起。量子数量可以从一到多，通常基于每个事件处理器内的数据库交互和请求-回复处理。

事件驱动架构在性能、可扩展性和容错方面获得五星，是此架构风格的主要优势。整体简单性和可测试性评分相对较低，主要由于此架构风格中典型的非确定性和动态事件流。事件驱动架构具有高度演进性，因此获得五星评分。
