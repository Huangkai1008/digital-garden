# 第10章 分层架构风格

分层架构（layered architecture），也称为 **n 层架构风格（n-tiered architecture style）**，是最常见的架构风格之一。这种架构风格是大多数应用的事实标准，主要因其简单性、熟悉度和低成本。由于康威定律（Conway's Law）——设计系统的组织受限于产生这些组织通信结构副本的设计——它也是开发应用的一种非常自然的方式。在大多数组织中，有用户界面（UI）开发者、后端开发者、规则开发者和数据库专家（DBA）。这些组织层次与传统的分层架构的层级很好地契合，使其成为许多业务应用的自然选择。分层架构风格也落入若干架构反模式，包括**隐含架构反模式（architecture by implication anti-pattern）**和**偶然架构反模式（accidental architecture anti-pattern）**。如果开发者或架构师不确定正在使用哪种架构风格，或者敏捷开发团队「直接开始编码」，很可能他们正在实现的就是分层架构风格。

## 拓扑（Topology）

分层架构风格中的组件被组织成逻辑水平层，每层在应用中执行特定角色（如展示逻辑或业务逻辑）。尽管在必须存在的层数和类型方面没有具体限制，但大多数分层架构由四个标准层组成：展示层、业务层、持久层和数据库层，如图 10-1 所示。在某些情况下，业务层和持久层会合并为单一业务层，特别是当持久逻辑（如 SQL 或 HSQL）嵌入在业务层组件中时。因此，较小的应用可能只有三层，而较大、更复杂的业务应用可能包含五层或更多。

```
┌─────────────────────────────────────┐
│          Presentation Layer          │  展示层
├─────────────────────────────────────┤
│           Business Layer            │  业务层
├─────────────────────────────────────┤
│         Persistence Layer           │  持久层
├─────────────────────────────────────┤
│          Database Layer             │  数据库层
└─────────────────────────────────────┘
     Figure 10-1. 分层架构中的标准逻辑层
```

图 10-2 从物理分层（部署）角度说明了各种拓扑变体。第一种变体将展示层、业务层和持久层合并为单一部署单元，数据库层通常表示为独立的外部物理数据库（或文件系统）。第二种变体将展示层物理分离到自己的部署单元，业务层和持久层合并到第二个部署单元。同样，在此变体中，数据库层通常通过外部数据库或文件系统物理分离。第三种变体将所有四个标准层合并为单一部署，包括数据库层。此变体可能适用于使用内部嵌入式数据库或内存数据库的较小应用。许多本地部署（on-premises）产品使用这第三种变体构建并交付给客户。

分层架构风格中的每一层在架构中都有特定的角色和职责。例如，展示层负责处理所有用户界面和浏览器通信逻辑，而业务层负责执行与请求相关的特定业务规则。架构中的每一层围绕满足特定业务请求所需的工作形成抽象。例如，展示层不需要知道或关心如何获取客户数据；它只需要以特定格式在屏幕上显示该信息。同样，业务层不需要关心如何格式化客户数据以在屏幕上显示，甚至客户数据来自哪里；它只需要从持久层获取数据，对数据执行业务逻辑（如计算值或聚合数据），并将该信息传递给展示层。

分层架构风格中的这种关注点分离概念使得在架构内构建有效的角色和职责模型变得容易。特定层内的组件范围有限，只处理与该层相关的逻辑。例如，展示层中的组件只处理展示逻辑，而驻留在业务层中的组件只处理业务逻辑。这使开发者能够利用其特定技术专长，专注于领域的技术方面（如展示逻辑或持久逻辑）。然而，这一好处的权衡是缺乏整体敏捷性（快速响应变更的能力）。

分层架构是一种**技术分区架构（technically partitioned architecture）**（与领域分区架构相对）。组件组不是按领域（如客户）分组，而是按其在架构中的技术角色（如展示或业务）分组。因此，任何特定业务领域都分布在架构的所有层中。例如，「客户」领域包含在展示层、业务层、规则层、服务层和数据库层中，使得对该领域应用变更变得困难。因此，领域驱动设计（DDD）方法与分层架构风格的配合效果不佳。

## 隔离层（Layers of Isolation）

分层架构风格中的每一层可以是**封闭的（closed）**或**开放的（open）**。封闭层意味着当请求自上而下从一层移动到另一层时，请求不能跳过任何层，而必须通过紧邻其下的层才能到达下一层（见图 10-3）。例如，在封闭分层架构中，源自展示层的请求必须首先经过业务层，然后到持久层，最后才能到达数据库层。

```
  Presentation ──→ Business ──→ Persistence ──→ Database
  (封闭：必须逐层通过)
     Figure 10-3. 分层架构中的封闭层
```

注意在图 10-3 中，对于简单的检索请求，展示层直接访问数据库会更快、更容易，绕过不必要的层（在 2000 年代初曾被称为快速通道读取者模式）。要实现这一点，业务层和持久层必须开放，允许请求绕过其他层。开放层好还是封闭层好？这一问题的答案在于一个关键概念：**隔离层（layers of isolation）**。

隔离层概念意味着在架构某一层所做的变更通常不会影响其他层中的组件，前提是这些层之间的契约保持不变。每一层独立于其他层，因此对架构中其他层的内部运作知之甚少或一无所知。然而，要支持隔离层，涉及请求主要流程的层必然必须是封闭的。如果展示层可以直接访问持久层，那么对持久层的变更将同时影响业务层和展示层，产生组件之间层相互依赖的非常紧密耦合的应用。这种架构会变得非常脆弱，且难以且昂贵地变更。

隔离层概念还允许架构中的任何层被替换而不影响任何其他层（同样，假设有良好定义的契约和使用业务委托模式）。例如，你可以在分层架构风格中利用隔离层概念，用 React.js 替换较旧的 JavaServer Faces（JSF）展示层，而不影响应用中的任何其他层。

## 添加层（Adding Layers）

虽然封闭层促进隔离层并因此有助于在架构内隔离变更，但有时某些层开放是有意义的。例如，假设业务层内有包含业务组件通用功能的共享对象（如日期和字符串工具类、审计类、日志类等）。假设有一项架构决策规定展示层禁止使用这些共享业务对象。此约束如图 10-4 所示，虚线从展示组件指向业务层中的共享业务对象。这种情况难以治理和控制，因为从架构上讲，展示层可以访问业务层，因此可以访问该层内的共享对象。

在架构上强制此限制的一种方法是在架构中添加一个包含所有共享业务对象的新服务层。添加此新层现在从架构上限制展示层访问共享业务对象，因为业务层是封闭的（见图 10-5）。然而，新服务层必须标记为开放；否则业务层将被迫通过服务层访问持久层。将服务层标记为开放允许业务层要么访问该层（如实线箭头所示），要么绕过该层到达下一层（如图 10-5 中的虚线箭头所示）。

利用开放和封闭层的概念有助于定义架构层与请求流之间的关系。它还为开发者提供必要的信息和指导，以理解架构内的各种层访问限制。未能记录或正确传达架构中哪些层是开放和封闭的（以及原因），通常会导致紧密耦合和脆弱的架构，非常难以测试、维护和部署。

## 其他考量（Other Considerations）

当尚不确定最终将使用哪种架构风格时，分层架构是大多数应用的良好起点。这是许多微服务工作的常见做法，当架构师仍在确定微服务是否是正确的架构选择时，但开发必须开始。然而，使用此技术时，请确保将复用保持在最低限度，并保持对象层次结构（继承树深度）相当浅，以保持良好的模块化水平。这将有助于日后迁移到另一种架构风格。

分层架构需要注意的一点是**架构天坑反模式（architecture sinkhole anti-pattern）**。当请求在层与层之间移动时仅进行简单的直通处理，每层内没有执行业务逻辑，就会发生此反模式。例如，假设展示层响应用户检索基本客户数据（如姓名和地址）的简单请求。展示层将请求传递给业务层，业务层除了将请求传递给规则层外什么都不做，规则层又除了将请求传递给持久层外什么都不做，持久层然后对数据库层进行简单的 SQL 调用以检索客户数据。然后数据一路传回堆栈，没有额外的处理或逻辑来聚合、计算、应用规则或转换数据。这导致不必要的对象实例化和处理，影响内存消耗和性能。

每个分层架构至少会有一些落入架构天坑反模式的场景。判断架构天坑反模式是否在起作用的关键是分析落入此类别的请求百分比。80-20 规则通常是遵循的良好实践。例如，如果只有 20% 的请求是天坑，是可以接受的。然而，如果 80% 的请求是天坑，则是分层架构不是问题领域正确架构风格的良好指标。解决架构天坑反模式的另一种方法是将架构中的所有层设为开放，当然要意识到权衡是增加架构内管理变更的难度。

## 为何使用此架构风格（Why Use This Architecture Style）

分层架构风格是小型、简单应用或网站的良好选择。对于预算和时间非常紧张的情况，它也是一个良好的架构选择，特别是作为起点。由于开发者与架构师之间的简单性和熟悉度，分层架构可能是成本最低的架构风格之一，促进较小应用的开发便利性。当架构师仍在分析业务需求和需求，不确定哪种架构风格最佳时，分层架构风格也是良好的选择。

随着使用分层架构风格的应用增长，可维护性、敏捷性、可测试性和可部署性等特性会受到不利影响。因此，使用分层架构的大型应用和系统可能更适合其他更模块化的架构风格。

## 架构特性评分（Architecture Characteristics Ratings）

特性评分表（如图 10-6 所示）中的一星评分表示该特定架构特性在架构中未得到良好支持，而五星评分表示该架构特性是架构风格中最强的特性之一。记分卡中识别的每个特性的定义可在第 4 章找到。

| 特性 | 评分 |
|------|------|
| 总体成本 | ★★★★★ |
| 简单性 | ★★★★★ |
| 可部署性 | ★ |
| 可测试性 | ★★ |
| 可靠性 | ★★★ |
| 弹性 | ★ |
| 可扩展性 | ★ |
| 性能 | ★★ |
| 容错 | ★ |
| 可用性 | ★★ |

总体成本和简单性是分层架构风格的主要优势。由于本质上是单体，分层架构没有与分布式架构风格相关的复杂性，简单易懂，构建和维护成本相对较低。然而，作为警示，随着单体分层架构变大并因此更复杂，这些评分会迅速下降。

可部署性和可测试性对此架构风格的评分非常低。可部署性低是由于部署的繁琐（部署工作量）、高风险和缺乏频繁部署。分层架构风格中对类文件的三行简单变更需要重新部署整个部署单元，可能包括数据库变更、配置变更或与原始变更一起混入的其他编码变更。此外，这三行简单变更通常与数十个其他变更捆绑在一起，从而进一步增加部署风险（以及部署频率）。低可测试性评分也反映了这一场景；对于三行简单变更，大多数开发者不会花数小时执行整个回归测试套件（即使首先存在这样的东西），特别是与对单体应用同时进行的数十个其他变更一起。我们给可测试性两星（而非一星）是因为能够模拟或存根组件（甚至整个层），这减轻了整体测试工作量。

由于大多数分布式架构中缺乏网络流量、带宽和延迟，整体可靠性在此架构风格中为中等（三星）。我们只给分层架构可靠性三星，是因为单体部署的性质，加上可测试性（测试完整性）和部署风险的低评分。

由于单体部署和缺乏架构模块化，分层架构的弹性和可扩展性非常低（一星）。虽然可以使单体内的某些功能比其他功能扩展更多，但这通常需要非常复杂的设计技术，如多线程、内部消息传递和其他并行处理实践，此架构风格并不适合这些技术。然而，由于分层架构由于单体用户界面、后端处理和单体数据库始终是单一系统量子，应用只能基于单一量子扩展到一定程度。

性能始终是分层架构的一个有趣特性。我们只给它两星，因为该架构风格由于缺乏并行处理、封闭分层和天坑架构反模式，根本不适合高性能系统。与可扩展性一样，可以通过缓存、多线程等解决性能问题，但这不是此架构风格的自然特性；架构师和开发者必须努力才能实现这一切。

由于单体部署和缺乏架构模块化，分层架构不支持容错。如果分层架构的一小部分导致内存不足情况发生，整个应用单元都会受到影响并崩溃。此外，由于大多数单体应用通常经历的高平均恢复时间（MTTR），整体可用性受到影响，启动时间从较小应用的 2 分钟到大多数大型应用的 15 分钟或更长不等。
