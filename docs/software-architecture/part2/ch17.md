# 第17章 微服务架构

微服务（microservices）是近年来获得显著势头的极其流行的架构风格。本章概述使此架构在拓扑和哲学上与众不同的重要特性。

## 历史（History）

微服务由 Martin Fowler 和 James Lewis 于 2014 年 3 月发表的著名博客文章「Microservices」较早命名并推广。微服务深受**领域驱动设计（DDD）**思想启发，特别是**限界上下文（bounded context）**概念。限界上下文代表一种解耦风格——有界上下文内的内部部分（如代码和数据模式）耦合在一起以产生工作，但从不与限界上下文外的任何东西耦合。微服务的主要目标是高解耦，在物理上建模限界上下文的逻辑概念。

## 拓扑（Topology）

由于单一用途性质，微服务中的服务规模比其他分布式架构（如编排式服务导向架构）小得多。架构师期望每个服务包含独立运行所需的所有必要部分，包括数据库和其他依赖组件。

### 分布式（Distributed）

微服务形成分布式架构：每个服务在自己的进程中运行。将服务解耦到这种程度可以简单解决在大量采用多租户基础设施托管应用的架构中的常见问题。性能通常是微服务分布式性质的负面副作用。

### 限界上下文（Bounded Context）

微服务的驱动哲学是限界上下文概念：每个服务建模一个领域或工作流。因此，每个服务包括在应用内运行所需的一切，包括类、其他子组件和数据库模式。微服务将领域分区架构的概念推向极致。

### 粒度（Granularity）

「微服务」是一个标签，不是描述。——Martin Fowler

架构师努力为微服务找到正确的服务粒度，经常犯将服务做得太小的错误，这需要他们在服务之间构建通信链接才能完成有用工作。服务边界的目的是捕获领域或工作流。一些指导原则：目的（领域）、事务（事务边界）、编排（通信开销）。

### 数据隔离（Data Isolation）

由限界上下文概念驱动的微服务的另一要求是数据隔离。微服务试图避免各种耦合，包括作为集成点使用的共享模式和数据库。架构师必须警惕实体陷阱，不要简单地将服务建模为类似于数据库中的单一实体。

### API 层（API Layer）

微服务的大多数图示在系统消费者（用户界面或其他系统的调用）之间包含 API 层，但它是可选的。如果架构师想忠实于此架构的基本哲学，不应将 API 层用作中介者或编排工具：所有有趣的逻辑应发生在限界上下文内。

### 运维复用（Operational Reuse）

鉴于微服务偏好重复而非耦合，架构师如何处理真正受益于耦合的架构部分，如监控、日志和熔断器等运维关注点？**边车模式（sidecar pattern）**提供了解决方案。一旦团队知道每个服务包含公共边车，他们可以构建**服务网格（service mesh）**，允许跨架构统一控制日志和监控等关注点。

### 前端（Frontends）

两种常见的用户界面风格：单体前端（单一用户界面通过 API 层调用）和**微前端（microfrontends）**（每个服务发出该服务的用户界面，前端与其他发出的用户界面组件协调）。

### 通信（Communication）

架构师必须决定同步或异步通信。微服务架构通常利用**协议感知的异构互操作性（protocol-aware heterogeneous interoperability）**。

**编排（Choreography）**：利用与代理事件驱动架构相同的通信风格，无中央协调器。**编排（Orchestration）**：创建局部中介者来协调跨多个服务的调用。

### 事务与 Saga（Transactions and Sagas）

跨服务边界构建事务违反微服务架构的核心解耦原则。最佳建议：不要这样做！应修复粒度组件。

对于需要跨服务事务协调的例外情况，**Saga 模式**是流行的分布式事务模式。**补偿事务框架（compensating transaction framework）**：如果事务的一部分失败，中介者必须向所有成功的部分发送请求，告诉它们撤销先前的请求。

::: tip 建议
跨服务的少量事务有时是必要的；如果它是架构的主导特性，那就犯了错误！
:::

## 架构特性评分（Architecture Characteristics Ratings）

微服务架构风格在我们的标准评分尺度上提供若干极端。值得注意的是对自动化部署、可测试性等现代工程实践的高度支持。微服务没有 DevOps 革命就无法存在。此架构的亮点是可扩展性、弹性和演进性。性能通常是微服务中的问题。微服务明确是以领域为中心的架构，具有任何现代架构中最 distinct 的量子。
