# 第15章 基于空间的架构风格

大多数基于 Web 的业务应用遵循相同的通用请求流：来自浏览器的请求命中 Web 服务器，然后应用服务器，最后数据库服务器。虽然这种模式对少量用户效果很好，但随着用户负载增加，瓶颈开始出现——首先在 Web 服务器层，然后在应用服务器层，最后在数据库服务器层。基于用户负载增加对瓶颈的通常响应是横向扩展 Web 服务器。这相对容易且成本低，有时能解决瓶颈问题。然而，在大多数高用户负载情况下，扩展 Web 服务器层只是将瓶颈下移到应用服务器。扩展应用服务器比 Web 服务器更复杂、更昂贵，通常只是将瓶颈下移到数据库服务器，后者更难、更昂贵地扩展。最终会得到三角形拓扑，三角形最宽部分是 Web 服务器（最容易扩展），最小部分是数据库（最难扩展）。

在任何具有大量并发用户负载的高容量应用中，数据库通常是可并发处理事务数量的最终限制因素。基于空间的架构风格专门设计用于解决涉及高可扩展性、弹性和高并发问题的问题。它也是具有可变和不可预测并发用户量的应用的有用架构风格。

## 通用拓扑（General Topology）

基于空间的架构得名于**元组空间（tuple space）**概念——使用通过共享内存通信的多个并行处理器的技术。通过将中央数据库作为系统中的同步约束移除，转而利用复制的内存数据网格，实现高可扩展性、高弹性和高性能。应用数据保存在内存中并在所有活动处理单元之间复制。当处理单元更新数据时，它异步将该数据发送到数据库，通常通过具有持久队列的消息传递。处理单元随着用户负载增减而动态启动和关闭，从而解决可变可扩展性。由于应用的标准事务处理不涉及中央数据库，数据库瓶颈被移除，从而在应用内提供近乎无限的可扩展性。

组成基于空间的架构的架构组件包括：包含应用代码的**处理单元（processing unit）**、用于管理和协调处理单元的**虚拟化中间件（virtualized middleware）**、用于异步将更新数据发送到数据库的**数据泵（data pumps）**、执行数据泵更新的**数据写入器（data writers）**，以及从数据库读取数据并在启动时将其交付给处理单元的**数据读取器（data readers）**。

### 处理单元（Processing Unit）

处理单元包含应用逻辑（或应用逻辑的部分），通常包括基于 Web 的组件以及后端业务逻辑。处理单元的内容根据应用类型而异。较小的基于 Web 的应用可能部署到单一处理单元，而较大的应用可能根据应用的功能区域将应用功能拆分为多个处理单元。处理单元还可以包含小型、单一用途的服务（如微服务）。除应用逻辑外，处理单元还包含通常通过 Hazelcast、Apache Ignite 和 Oracle Coherence 等产品实现的**内存数据网格（in-memory data grid）**和复制引擎。

### 虚拟化中间件（Virtualized Middleware）

虚拟化中间件处理架构内控制数据同步和请求处理各个方面的基础设施关注点。组成虚拟化中间件的组件包括**消息网格（messaging grid）**、**数据网格（data grid）**、**处理网格（processing grid）**和**部署管理器（deployment manager）**。

### 数据泵、数据写入器与数据读取器（Data Pumps, Data Writers, Data Readers）

数据泵是将数据发送到另一个处理器、后者然后更新数据库中数据的方式。数据写入器接受来自数据泵的消息，并用数据泵消息中的信息更新数据库。数据读取器负责从数据库读取数据并通过反向数据泵将其发送到处理单元。数据读取器仅在三种情况之一下被调用：同一命名缓存的所有处理单元实例崩溃、同一命名缓存内所有处理单元重新部署，或检索复制缓存中不包含的归档数据。

### 数据冲突（Data Collisions）

在使用主动/主动状态的复制缓存时，由于复制延迟可能存在数据冲突。冲突率公式：`CollisionRate = N * UR² / (S * RL)`，其中 N 为实例数，UR 为更新率，S 为缓存大小，RL 为复制延迟。

### 云与本地部署（Cloud Versus On-Premises）

基于空间的架构可以部署在云环境中，同时将物理数据库和相应数据保留在本地，支持非常有效的基于云的数据同步。

### 复制缓存与分布式缓存（Replicated Versus Distributed Caching）

**复制缓存**：每个处理单元包含自己的内存数据网格，在使用同一命名缓存的所有处理单元之间同步。极快，支持高容错，无单点故障。

**分布式缓存**：需要专门保存集中缓存的外部服务器或服务。支持更高级别的数据一致性，但性能较低，容错是问题。

### 近缓存考量（Near-Cache Considerations）

近缓存是桥接内存数据网格与分布式缓存的混合缓存模型。我们不建议将近缓存模型用于基于空间的架构。

## 实现示例（Implementation Examples）

**音乐会售票系统**：具有独特的问题领域，在热门音乐会宣布之前并发用户量相对较低。一旦音乐会门票开售，用户量通常从数百并发用户激增到数千（可能数万）。基于空间的架构由于此类应用所需的高弹性要求而非常适合。

**在线拍卖系统**：与音乐会售票系统具有相同特性——都需要高水平的性能和弹性，都有不可预测的用户和请求负载峰值。基于空间的架构非常适合此类问题领域。

## 架构特性评分（Architecture Characteristics Ratings）

基于空间的架构在弹性、可扩展性和性能方面最大化（均为五星）。这些是此架构风格的驱动属性和主要优势。通过利用内存数据缓存和移除数据库作为约束，实现这三项架构特性的高水平。整体简单性和可测试性评分较低。成本是选择此架构风格的另一个因素。基于空间的架构难以识别分区类型，我们将其识别为既领域分区又技术分区。
