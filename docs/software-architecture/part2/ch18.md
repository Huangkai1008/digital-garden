# 第18章 选择合适的架构风格

**看情况！** 在所有可用选择（以及几乎每天出现的新选择）中，我们想告诉你该用哪一种——但我们不能。没有什么比组织内的诸多因素及其构建的软件更具情境性。选择架构风格代表了对架构特性权衡、领域考量、战略目标等诸多方面分析和思考的 culmination。

然而，无论决策多么情境化，围绕选择合适架构风格存在一些通用建议。

## 架构中的「时尚」变迁（Shifting "Fashion" in Architecture）

偏好的架构风格随时间变化，由多种因素驱动：

- **过去的观察**：新架构风格通常源于过去经验的观察和痛点
- **生态系统的变化**：持续变化是软件开发生态系统的可靠特征
- **新能力**：当新能力出现时，架构可能不仅用一种工具替换另一种，而是转向 entirely 新范式
- **加速**：不仅生态系统不断变化，变化速率也在持续上升
- **领域变化**：开发者为之编写软件的领域不断变化
- **技术变化**：组织试图至少跟上部分技术变化
- **外部因素**：许多与软件开发仅 peripherally 相关的外部因素可能驱动组织内的变化

## 决策标准（Decision Criteria）

选择架构风格时，架构师必须考虑促成领域设计结构的所有各种因素。从根本上说，架构师设计两件事：无论指定的领域是什么，以及使系统成功所需的所有其他结构元素。

架构师在设计决策时应熟悉以下方面：

- **领域**：架构师应理解领域的许多重要方面
- **影响结构的架构特性**：架构师必须发现并阐明支持领域和外部因素所需的架构特性
- **数据架构**：架构师和 DBA 必须在数据库、模式和其他数据相关关注点上协作
- **组织因素**：许多外部因素可能影响设计
- **流程、团队和运维关注点的知识**：许多特定项目因素影响架构师的设计
- **领域/架构同构（domain/architecture isomorphism）**：某些问题领域与架构的拓扑匹配

考虑所有这些因素后，架构师必须做出若干决定：

1. **单体与分布式**：使用量子概念，架构师必须确定单一架构特性集是否足以满足设计，还是系统的不同部分需要不同的架构特性？
2. **数据应驻留何处？** 在分布式架构中，架构师必须决定哪些服务应持久化数据
3. **服务之间的通信风格——同步还是异步？** 默认使用同步，必要时使用异步

::: tip 建议
默认使用同步，必要时使用异步。
:::

## 单体案例研究：Silicon Sandwiches（Monolith Case Study）

在 Silicon Sandwiches 架构 kata 中，我们确定单一量子足以实现此系统。我们创建了两种不同的组件设计：领域分区和技术分区。

**模块化单体（Modular Monolith）**：构建以领域为中心的组件，单一数据库，作为单一量子部署。架构师设计 Override 端点，开发者可以上传各个自定义。每个领域组件引用 Override 组件以获取每个可自定义特性。

**微内核（Microkernel）**：核心系统由领域组件和单一关系数据库组成。每个自定义出现在插件中，通用插件在一组插件中，本地插件各有自己的数据。利用**前端后端（BFF）**模式，使 API 层成为薄的微内核适配器。

## 分布式案例研究：Going, Going, Gone（Distributed Case Study）

GGG kata 呈现更有趣的架构挑战。基于组件分析，此架构的不同部分需要不同的架构特性。在候选分布式架构中，低级事件驱动或微服务最匹配大多数架构特性。两者中，微服务更好地支持不同的运维架构特性。

GGG 的实现包括：BidCapture、BidStreamer、BidTracker、Auctioneer Capture、Auction Session、Payment、Video Capture、Video Streamer 等服务。架构师仔细识别了此架构中的同步和异步通信风格。最终设计解析为五个量子：Payment、Auctioneer、Bidder、Bidder Streams 和 Bid Tracker。

注意，这不是 GGG 的「正确」设计，当然也不是唯一的设计。我们甚至不暗示它是最好的可能设计，但它似乎具有 least worst 的权衡集。选择微服务，然后智能地使用事件和消息，允许架构从通用架构模式中充分利用，同时仍为未来开发和扩展奠定基础。
