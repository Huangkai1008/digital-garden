# 第22章 打造高效团队

除了创建技术架构和做出架构决策外，软件架构师还负责指导开发团队实施架构。善于此道的软件架构师会创建高效的开发团队，密切协作解决问题并创造制胜解决方案。虽然这听起来显而易见，但我们多次看到架构师忽视开发团队，在孤岛环境中工作以创建架构。然后该架构被移交给开发团队，开发团队随后难以正确实施架构。能够使团队富有成效是有效且成功的软件架构师与其他软件架构师区分开来的方式之一。本章介绍架构师可以利用的一些使开发团队有效的基本技术。

## 团队边界（Team Boundaries）

根据我们的经验，软件架构师可以显著影响开发团队的成功或失败。感到被排除在循环之外或与软件架构师（以及架构）疏远的团队，往往对系统的各种约束没有正确的指导水平和知识水平，因此无法正确实施架构。

软件架构师的职责之一是创建和传达约束，即开发者可以实施架构的**盒子（box）**。架构师可以创建过紧、过松或恰到好处的边界。这些边界如图 22-1 所示。边界过紧或过松会直接影响团队成功实施架构的能力。

```
过紧边界 ──► 团队受限于工具和库
恰到好处 ──► 团队有正确指导和工具
过松边界 ──► 团队承担架构师角色
```

**图 22-1. 软件架构师创建的边界类型**

创建过多约束的架构师在开发团队周围形成紧盒子，阻止访问有效实施系统所需的许多工具、库和实践。这会在团队内造成挫败感，通常导致开发者离开项目去寻找更快乐、更健康的环境。

相反的情况也可能发生。软件架构师可以创建过松的约束（或根本没有约束），将所有重要的架构决策留给开发团队。在这种情况下，与紧约束一样糟糕，团队 essentially 承担了软件架构师的角色，进行概念验证并就设计决策争论，而没有适当的指导水平，导致低效、困惑和挫败。

有效的软件架构师努力提供适当的指导水平和约束，使团队拥有正确的工具和库来有效实施架构。本章其余部分专门讨论如何创建这些有效边界。

## 架构师人格（Architect Personalities）

架构师有三种基本人格类型：**控制狂架构师（control freak architect）**（图 22-2）、**扶手椅架构师（armchair architect）**（图 22-3）和**有效架构师（effective architect）**（图 22-5）。每个人格与前一节讨论的团队边界类型相匹配：控制狂架构师产生紧边界，扶手椅架构师产生松边界，有效架构师产生恰到好处的边界。

### 控制狂（Control Freak）

**图 22-2. 控制狂架构师**

控制狂架构师试图控制软件开发过程的每个细节。控制狂架构师做出的每个决策通常过于细粒度和低层次，导致对开发团队施加过多约束。

控制狂架构师产生前一节讨论的紧边界。控制狂架构师可能限制开发团队下载任何有用的开源或第三方库，反而坚持团队使用语言 API 从头编写一切。控制狂架构师还可能对命名约定、类设计、方法长度等施加严格限制。他们甚至可能为开发团队编写伪代码。Essentially，控制狂架构师剥夺了开发者的编程艺术，导致挫败感和对架构师缺乏尊重。

很容易成为控制狂架构师，尤其是从开发者过渡到架构师时。架构师的职责是创建应用的构建块（组件）并确定这些组件之间的交互。开发者在此工作中的职责是然后获取这些组件并使用类图和设计模式确定如何实施。然而，在从开发者到架构师的过渡中，诱惑太大，想要也创建类图和设计模式，因为那是新晋架构师之前的角色。

例如，假设架构师创建一个组件（架构的构建块）来管理系统中的参考数据。参考数据包括网站上使用的静态名值对数据，以及产品代码和仓库代码等（整个系统使用的静态数据）。架构师的职责是识别组件（本例中为 Reference Manager）、确定该组件的核心操作集（例如 GetData、SetData、ReloadCache、NotifyOnUpdate 等）以及哪些组件需要与 Reference Manager 交互。控制狂架构师可能认为实施此组件的最佳方式是通过利用内部缓存的并行加载器模式，使用该缓存的特定数据结构。虽然这可能是有效的设计，但不是唯一的设计。更重要的是，提出 Reference Manager 的这种内部设计不再是架构师的职责——这是开发者的职责。

正如我们将在第 331 页的「控制多少？」中讨论的，有时架构师需要扮演控制狂的角色，取决于项目的复杂性和团队的技能水平。然而，在大多数情况下，控制狂架构师会 disrupt 开发团队，不提供适当的指导水平，碍事，在领导团队实施架构方面无效。

### 扶手椅架构师（Armchair Architect）

**图 22-3. 扶手椅架构师**

扶手椅架构师是很久没有编码（如果有的话）的架构师类型，在创建架构时不考虑实施细节。他们通常与开发团队脱节，从不在场，或者一旦初始架构图完成就从一个项目转到另一个项目。

在某些情况下，扶手椅架构师在技术或业务领域方面 simply 力不从心，因此无法从技术或业务问题角度领导或指导团队。例如，开发者做什么？当然是写代码。编写程序代码真的很难造假；开发者要么写软件代码，要么不写。然而，架构师做什么？没人知道！大多数架构师画很多线和方框——但架构师在这些图中应该多详细？关于架构有一个肮脏的小秘密——作为架构师真的很容易造假！

假设扶手椅架构师力不从心或没有时间为股票交易系统架构适当的解决方案。在这种情况下，架构图可能看起来像图 22-4 所示的那样。这个架构没有什么问题——只是太高层了，对任何人都没有用。

```
┌─────────────────────────────────────────────────────────┐
│  交易系统架构（扶手椅架构师创建）                          │
│                                                          │
│  [用户] ──► [系统] ──► [数据库]                            │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**图 22-4. 扶手椅架构师创建的交易系统架构**

扶手椅架构师在开发团队周围创建松边界，如前一节所讨论的。在这种情况下，开发团队最终承担架构师的角色，essentially 做架构师应该做的工作。团队速度和生产力因此受到影响，团队对系统应该如何工作感到困惑。

像控制狂架构师一样，很容易成为扶手椅架构师。架构师可能陷入扶手椅架构师人格的最大指标是没有足够的时间与实施架构的开发团队在一起（或选择不花时间与开发团队在一起）。开发团队需要架构师的支持和指导，他们需要架构师在出现技术或业务相关问题时随时可用。扶手椅架构师的其他指标包括：

- 未完全理解业务领域、业务问题或使用的技术
- 软件开发实践不足
- 未考虑与架构解决方案实施相关的含义

在某些情况下，架构师并非有意成为扶手椅架构师，而是 simply 在项目或开发团队之间 spread 太薄而「发生」，失去对技术或业务领域的接触。架构师可以通过更多地参与项目使用的技术并理解业务问题和业务领域来避免这种人格。

### 有效架构师（Effective Architect）

**图 22-5. 有效软件架构师**

有效的软件架构师对团队产生适当的约束和边界，确保团队成员良好协作并拥有团队上的适当指导水平。有效架构师还确保团队拥有正确和适当的工具和技术。此外，他们消除可能阻碍开发团队实现目标的任何障碍。

虽然这听起来显而易见且容易，但并非如此。成为开发团队上的有效领导者是一门艺术。成为有效的软件架构师需要与团队密切协作，并赢得团队的尊重。我们将在本书的后续章节中探讨成为有效软件架构师的其他方式。但现在，我们介绍一些关于有效架构师应对开发团队施加多少控制的指南。

## 控制多少？（How Much Control?）

成为有效的软件架构师就是知道对给定开发团队施加多少控制。这个概念被称为**弹性领导力（Elastic Leadership）**，由作者和顾问 Roy Osherove 广泛推广。我们将在此领域与 Osherove 的工作略有偏离，专注于软件架构的具体因素。

知道有效软件架构师应该在多大程度上是控制狂、在多大程度上是扶手椅架构师涉及五个主要因素。这些因素还决定了软件架构师可以同时管理多少团队（或项目）：

| 因素 | 说明 |
|------|------|
| **团队熟悉度** | 团队成员彼此了解程度如何？他们以前在项目上合作过吗？通常，团队成员越了解彼此，需要的控制越少，因为团队成员开始自我组织。相反，团队成员越新，需要的控制越多，以帮助促进团队成员之间的协作并减少团队内的小团体。 |
| **团队规模** | 团队有多大？（我们认为同一团队超过 12 名开发者是大团队，4 人或更少是小团队。）团队越大，需要的控制越多。团队越小，需要的控制越少。这在第 335 页的「团队警告信号」中更详细讨论。 |
| **整体经验** | 有多少团队成员是高级成员？有多少是初级成员？是初级和高级开发者的混合团队吗？他们对技术和业务领域的了解程度如何？初级开发者较多的团队需要更多控制和指导，而高级开发者较多的团队需要更少控制。在后一种情况下，架构师从指导者角色转变为促进者角色。 |
| **项目复杂度** | 项目是高度复杂还是简单的网站？高度复杂的项目需要架构师更频繁地与团队在一起并协助出现的问题，因此团队需要更多控制。相对简单的项目 straightforward，因此不需要太多控制。 |
| **项目持续时间** | 项目是短（两个月）、长（两年）还是平均持续时间（六个月）？持续时间越短，需要的控制越少；相反，项目越长，需要的控制越多。 |

虽然大多数因素在更多或更少控制方面是有道理的，但项目持续时间因素可能看起来没有道理。如列表所示，项目持续时间越短，需要的控制越少；项目越长，需要的控制越多。直觉上这可能似乎 reversed，但事实并非如此。考虑一个快速的两个月项目。两个月没有太多时间进行需求确认、实验、开发代码、测试每个场景并发布到生产。在这种情况下，架构师应更多地扮演扶手椅架构师的角色，因为开发团队已经有强烈的紧迫感。控制狂架构师只会碍事并可能延迟项目。相反，考虑两年的项目持续时间。在这种情况下，开发者较为放松，没有紧迫感，可能正在计划休假和长时间午餐。架构师需要更多控制，以确保项目及时推进并首先完成复杂任务。

在大多数项目中，这些因素通常在项目开始时用于确定控制水平；但随着系统继续演进，控制水平会变化。因此，我们建议在整个项目生命周期中持续分析这些因素，以确定对开发团队施加多少控制。

为说明如何使用这些因素中的每一个来确定架构师对团队的控制水平，假设每个因素有 20 分的固定刻度。负值更倾向于扶手椅架构师（更少控制和参与），而正值更倾向于控制狂架构师（更多控制和参与）。此刻度如图 22-6 所示。

## 团队警告信号（Team Warning Signs）

如前一节所述，团队规模是影响架构师应对开发团队施加多少控制的因素之一。团队越大，需要的控制越多；团队越小，需要的控制越少。考虑最有效的开发团队规模时，三个因素起作用：

- **过程损失（Process loss）**
- **多元无知（Pluralistic ignorance）**
- **责任扩散（Diffusion of responsibility）**

过程损失，也称为**布鲁克斯定律（Brook's law）**，最初由 Fred Brooks 在其著作《人月神话》（The Mythical Man Month，Addison-Wesley）中提出。过程损失的基本思想是，你向项目添加的人越多，项目花费的时间就越多。如图 22-9 所示，团队潜力由团队上每个人的集体努力定义。然而，对于任何团队，实际生产力始终低于团队潜力，差异就是团队的过程损失。

**图 22-9. 团队规模影响实际生产力（布鲁克斯定律）**

有效的软件架构师会观察开发团队并寻找过程损失。过程损失是确定特定项目或工作的正确团队规模的良好因素。过程损失的一个迹象是向仓库推送代码时频繁的合并冲突。这表明团队成员可能相互踩脚并在同一代码上工作。在团队内寻找并行区域并让团队成员在独立服务或应用区域上工作是避免过程损失的一种方式。每当新团队成员加入项目时，如果没有创建并行工作流的区域，有效架构师会质疑为什么向团队添加新成员，并向项目经理展示该额外人员可能对团队产生的负面影响。

随着团队规模变得过大，也会发生多元无知。多元无知是指每个人都同意（但私下拒绝）一个规范，因为他们认为他们遗漏了显而易见的东西。例如，假设在一个大团队中，大多数人同意在两个远程服务之间使用消息传递是最好的解决方案。然而，团队中的一个人认为这是愚蠢的想法，因为两个服务之间有安全防火墙。然而，该人没有发言，也同意使用消息传递（但私下拒绝该想法），因为他们害怕要么遗漏了显而易见的东西，要么害怕如果发言可能被视为傻瓜。在这种情况下，拒绝该规范的人是正确的——由于两个远程服务之间的安全防火墙，消息传递将不起作用。如果他们发言（并且团队规模更小），原始解决方案将被挑战，并将使用另一种协议（如 REST），在这种情况下会是更好的解决方案。

多元无知的概念因丹麦儿童故事《皇帝的新装》（Hans Christian Andersen）而闻名。在故事中，国王相信他的新衣服对任何不值得看到的人都是 invisible 的。他赤身裸体地四处走动，问他所有臣民他们有多喜欢他的新衣服。所有臣民，害怕被当作愚蠢或不值得，回应国王说他的新衣服是有史以来最好的。这种 folly 继续下去，直到一个孩子终于喊道国王根本没有穿任何衣服。

有效的软件架构师应在任何类型的协作会议或讨论中持续观察面部表情和肢体语言，如果感觉到多元无知的发生，应充当促进者。在这种情况下，有效架构师可能会打断并询问该人对提议的解决方案的看法，并在他们发言时站在他们一边并支持他们。

表明适当团队规模的第三个因素称为责任扩散。责任扩散基于这样一个事实：随着团队规模增加，它对沟通产生负面影响。对团队中谁负责什么感到困惑以及事情被遗漏是团队过大的良好迹象。

看图 22-10 中的图片。你观察到什么？

**图 22-10. 责任扩散**

这张图片显示有人站在小型乡村道路旁的一辆抛锚汽车旁。在这种情况下，有多少人可能会停下来询问驾车者是否一切正常？因为这是一条小社区的小路，可能是每个经过的人。然而，有多少次驾车者被困在大城市繁忙高速公路旁，数千辆车 simply 驶过，没有人停下来询问是否一切正常？一直如此。这是责任扩散的好例子。随着城市变得越繁忙、越拥挤，人们假设驾车者已经打电话或救援正在路上，因为许多人目睹了该事件。然而，在大多数情况下，救援并没有在路上，驾车者被困在没电或被遗忘的手机旁，无法呼救。

有效的架构师不仅帮助指导开发团队实施架构，还确保团队健康、快乐并共同实现共同目标。寻找这三个警告信号并 consequently 帮助纠正它们有助于确保有效的开发团队。

## 利用清单（Leveraging Checklists）

飞行员在每次飞行中都使用清单——即使是最有经验、最老练的老手。飞行员有起飞、降落和数千种其他情况的清单，既有常见的也有不寻常的边缘情况。他们使用清单是因为一个遗漏的飞机设置（如将襟翼设置为 10 度）或程序（如获得进入终端管制区的许可）可能意味着安全飞行与灾难性飞行之间的差异。

Atul Gawande 博士写了一本出色的书《清单宣言》（The Checklist Manifesto，Picador），其中描述了外科手术清单的力量。对医院中葡萄球菌感染的高发生率感到震惊，Gawande 博士创建了外科清单以尝试降低这一比率。在书中，他证明使用清单的医院中葡萄球菌感染率降至接近零，而未使用清单的对照医院中葡萄球菌感染率继续上升。

清单有效。它们为确保一切都被覆盖和解决提供了绝佳载体。如果清单如此有效，为什么软件开发行业不利用它们？我们通过个人经验坚信，清单对开发团队的有效性有很大影响。然而，这一说法有注意事项。首先，大多数软件开发者不是在驾驶客机或进行心脏直视手术。换句话说，软件开发者不需要对所有事情都使用清单。使团队有效的关键是知道何时利用清单、何时不利用。

考虑图 22-11 所示的创建新数据库表的清单。

**图 22-11. 不良清单示例**

这不是清单，而是一组程序步骤，因此不应在清单中。例如，如果表单尚未提交，则无法验证数据库表！任何具有依赖任务程序流程的流程都不应在清单中。经常执行且无错误的简单、众所周知的流程也不需要清单。

适合清单的流程是那些没有任何程序顺序或依赖任务的流程，以及那些容易出错或经常遗漏或跳过的步骤的流程。使清单有效的关键是不要过度使一切都成为清单。架构师发现清单确实使开发团队更有效，因此开始使一切都成为清单，引发所谓的**收益递减定律（law of diminishing returns）**。架构师创建的清单越多，开发者使用它们的可能性就越小。创建清单时的另一个关键成功因素是使它们尽可能小，同时仍捕获流程中的所有必要步骤。开发者通常不会遵循太大的清单。寻求可以通过自动化执行的项，并将它们从清单中移除。

::: tip 建议
不要担心在清单中陈述显而易见的事情。通常被跳过或遗漏的就是显而易见的事情。
:::

我们发现最有效的三个关键清单是开发者代码完成清单、单元和功能测试清单以及软件发布清单。以下各节讨论每个清单。

### 霍桑效应（The Hawthorne Effect）

向开发团队引入清单时相关的问题之一是让开发者真正使用它们。有些开发者时间不够就 simply 将特定清单中的所有项标记为已完成而没有实际执行任务，这太常见了。

解决此问题的方法之一是与团队讨论使用清单的重要性以及清单如何对团队产生影响。让团队成员阅读 Atul Gawande 的《清单宣言》以充分理解清单的力量，并确保每个团队成员理解每个清单背后的推理以及为何使用它。让开发者协作决定什么应该和不应该在清单上也有帮助。

当其他方法都失败时，架构师可以援引所谓的**霍桑效应（Hawthorne effect）**。霍桑效应 essentially 意味着如果人们知道他们正在被观察或监控，他们的行为会改变，通常他们会做正确的事。例子包括建筑物内外实际上不工作或没有真正录制任何内容的高度可见的摄像头（这非常常见！）和网站监控软件（这些报告中有多少实际上被查看？）。

霍桑效应也可用于治理清单的使用。架构师可以让团队知道使用清单对团队的有效性至关重要，因此，所有清单都将被验证以确保任务实际执行，而事实上架构师只是偶尔抽查清单的正确性。通过利用霍桑效应，开发者将不太可能跳过项或将未完成的任务标记为已完成。

### 开发者代码完成清单（Developer Code Completion Checklist）

开发者代码完成清单是使用的有效工具，特别是当软件开发者声称他们「完成」了代码时。它对于定义所谓的「完成定义」也很有用。如果清单中的一切都完成了，那么开发者可以声称他们实际上完成了代码。

以下是开发者代码完成清单中应包含的一些内容：

- 未包含在自动化工具中的编码和格式标准
- 经常被忽视的项（如吸收的异常）
- 项目特定标准
- 特殊团队指示或程序

图 22-12 说明了开发者代码完成清单的示例。

**图 22-12. 开发者代码完成清单示例**

注意清单中显而易见的任务「运行代码清理和代码格式化」和「确保没有吸收的异常」。开发者有多少次在一天结束时或迭代结束时匆忙而忘记从 IDE 运行代码清理和格式化？很多次。在《清单宣言》中，Gawande 发现外科手术方面也有同样的现象——显而易见的往往是通常被遗漏的。

还要注意第 2、3、6 和 7 项中的项目特定任务。虽然这些是清单中的好项，但架构师应始终审查清单，看看是否有任何项可以自动化或编写为代码验证检查器的插件。例如，虽然「在服务 API 类上包含 @ServiceEntrypoint」可能无法进行自动检查，但「验证只有公共方法调用 setFailure()」当然可以（这是任何类型的代码爬虫工具的 straightforward 自动检查）。检查自动化区域有助于减少清单的大小和噪音，使其更有效。

### 单元和功能测试清单（Unit and Functional Testing Checklist）

Perhaps 最有效的清单之一是单元和功能测试清单。此清单包含软件开发者倾向于忘记测试的一些更不寻常和边缘情况的测试。每当 QA 的某人基于特定测试用例发现代码问题时，该测试用例应添加到此清单中。此特定清单通常是由于可以对代码运行的所有类型的测试而最大的清单之一。此清单的目的是确保尽可能完整的编码，以便当开发者完成清单时，代码 essentially 已准备好投入生产。

以下是典型单元和功能测试清单中的一些项：

- 文本和数字字段中的特殊字符
- 最小和最大值范围
- 不寻常和极端测试用例
- 缺失字段

像开发者代码完成清单一样，任何可以编写为自动化测试的项都应从清单中移除。例如，假设股票交易应用清单中有一项用于测试负股数（如对苹果 [AAPL] 的 -1,000 股买入）。如果此检查通过测试套件中的单元或功能测试自动化，则该项应从清单中移除。

开发者在编写单元测试时有时不知道从哪里开始或要编写多少单元测试。此清单提供了一种确保在开发软件过程中包含一般或特定测试场景的方式。此清单在由独立团队执行这些活动的环境中，在弥合开发者和测试者之间的差距方面也很有效。开发团队执行的完整测试越多，测试团队的工作就越容易，使测试团队能够专注于清单中未涵盖的某些业务场景。

### 软件发布清单（Software Release Checklist）

将软件发布到生产环境 Perhaps 是软件开发生命周期中最容易出错的方面之一，因此非常适合清单。此清单有助于避免构建失败和部署失败，并显著降低与发布软件相关的风险量。

软件发布清单通常是清单中最易变的，因为它不断变化以解决每次部署失败或有问题时的新错误和情况。

以下是软件发布清单中通常包含的一些项：

- 服务器或外部配置服务器中的配置更改
- 添加到项目的第三方库（JAR、DLL 等）
- 数据库更新和相应的数据库迁移脚本

每当构建或部署失败时，架构师应分析失败的根本原因并将相应条目添加到软件发布清单中。这样，下次构建或部署时将验证该项，防止该问题再次发生。

## 提供指导（Providing Guidance）

软件架构师还可以通过使用设计原则提供指导来使团队有效。这也有助于形成本章第一节描述的盒子（约束），开发者可以在其中工作以实施架构。有效传达这些设计原则是创建成功团队的关键之一。

为说明这一点，考虑向开发团队提供关于通常称为**分层栈（layered stack）**——组成应用的第三方库集合（如 JAR 文件和 DLL）——使用的指导。开发团队通常对分层栈有很多问题，包括他们是否可以就各种库做出自己的决定、哪些可以、哪些不可以。

使用此示例，有效的软件架构师可以通过首先让开发者回答以下问题来向开发团队提供指导：

1. 提议的库与系统内现有功能之间是否有任何重叠？
2. 提议的库的理由是什么？

第一个问题指导开发者查看现有库，看看新库提供的功能是否可以通过现有库或现有功能满足。根据我们的经验，开发者有时会忽略此活动，创建大量重复功能，特别是在有大型团队的大型项目中。

第二个问题促使开发者质疑为什么真正需要新库或功能。在这里，有效的软件架构师会要求技术论证和业务论证，说明为什么需要额外的库。这可以成为在开发团队内创建业务论证需求意识的强大技术。

### 业务论证的影响（The Impact of Business Justifications）

本书作者之一（Mark）曾是一个特别复杂的基于 Java 的项目的大型开发团队的首席架构师。团队成员之一特别痴迷于 Scala 编程语言， desperately 想在项目上使用它。对使用 Scala 的渴望最终变得如此具有破坏性，以至于几位关键团队成员告知 Mark 他们计划离开项目并转到其他「毒性较小」的环境。Mark 说服两位关键团队成员暂时搁置决定，并与 Scala 爱好者进行了讨论。Mark 告诉 Scala 爱好者，他支持在项目中使用 Scala，但 Scala 爱好者必须提供使用 Scala 的业务论证，因为涉及培训成本和重写工作。Scala 爱好者欣喜若狂，说他会立即着手，他离开会议时喊道：「谢谢——你是最棒的！」

第二天，Scala 爱好者完全变了样地来到办公室。他立即接近 Mark 并要求与他交谈。他们俩进了会议室，Scala 爱好者立即（谦虚地）说：「谢谢。」Scala 爱好者向 Mark 解释，他可以想出世界上所有使用 Scala 的技术理由，但那些技术优势在所需的架构特性（「-ilities」）方面没有任何业务价值：成本、预算和时间表。事实上，Scala 爱好者意识到成本、预算和时间表的增加不会带来任何好处。

意识到自己造成的破坏，Scala 爱好者迅速将自己转变为团队中最优秀、最有帮助的成员之一，这一切都是因为被要求为他想要在项目上的东西提供业务论证。这种论证意识的提高不仅使他成为更好的软件开发者，还造就了更强大、更健康的团队。

作为后记，两位关键开发者一直留在项目直到最后。

继续分层栈治理的例子，传达设计原则的另一种有效技术是通过图形解释开发团队可以做出决定的和不能做出决定的。图 22-13 中的插图是此图形（以及指导）在控制分层栈时可能看起来的示例。

**图 22-13. 为分层栈提供指导**

在图 22-13 中，架构师将提供每个第三方库类别将包含的示例，然后是关于开发者可以做什么和不能做什么的指导（设计原则）（本章第一节描述的盒子）。例如，以下是为任何第三方库定义的三个类别：

- **特殊用途（Special purpose）**：用于 PDF 渲染、条形码扫描等情况的特定库，不值得编写自定义软件。
- **通用用途（General purpose）**：语言 API 之上的包装器，包括 Java 的 Apache Commons 和 Guava 等。
- **框架（Framework）**：用于持久化（如 Hibernate）和控制反转（如 Spring）等。换句话说，这些库组成应用的整个层或结构，具有高度侵入性。

一旦分类（前述类别仅是示例——可以定义更多），架构师然后围绕此设计原则创建盒子。注意图 22-13 所示的示例中，对于此特定应用或项目，架构师已指定对于特殊用途库，开发者可以做出决定，不需要就该库咨询架构师。然而，注意对于通用用途，架构师已表明开发者可以进行重叠分析和论证以提出建议，但该类别库需要架构师批准。最后，对于框架库，那是架构师的决定——换句话说，开发团队甚至不应该对这些类型的库进行分析；架构师已决定承担这些类型库的责任。

## 总结（Summary）

使开发团队有效是艰苦的工作。它需要大量经验和实践，以及强大的人际技能（我们将在本书后续章节中讨论）。也就是说，本章描述的关于弹性领导力、利用清单和通过有效传达设计原则提供指导的简单技术确实有效，并已证明在使开发团队更聪明、更有效地工作方面有效。

有人可能会质疑架构师在此类活动中的角色，反而将使团队有效的努力分配给开发经理或项目经理。我们强烈反对这一前提。软件架构师不仅向团队提供技术指导，还领导团队实施架构。软件架构师与开发团队之间的密切协作关系使架构师能够观察团队动态，从而促进使团队更有效的变革。这正是技术架构师与有效软件架构师的区别所在。
