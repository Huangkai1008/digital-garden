# 第20章 分析架构风险

每种架构都伴随风险，无论是可用性、可扩展性还是数据完整性方面的风险。分析架构风险是架构的关键活动之一。通过持续分析风险，架构师可以解决架构中的缺陷并采取纠正措施以降低风险。本章介绍一些用于评估风险、创建风险评估以及通过名为**风险风暴（risk storming）** 的活动识别风险的关键技术和实践。

## 风险矩阵（Risk Matrix）

评估架构风险时出现的第一个问题是确定风险应归类为低、中还是高。这种分类通常带有太多主观性，造成对架构哪些部分真正属于高风险与中风险的混淆。幸运的是，架构师可以利用风险矩阵来帮助降低主观性水平，并评估架构特定区域的相关风险。

架构风险矩阵（如图 20-1 所示）使用两个维度来评估风险：风险的**整体影响**和**该风险发生的可能性**。每个维度有低（1）、中（2）、高（3）的评级。这些数字在矩阵的每个格子内相乘，得出代表该风险的客观数值。数字 1 和 2 视为低风险（绿色），3 和 4 视为中风险（黄色），6 到 9 视为高风险（红色）。

```
              影响（Impact）
              低(1)    中(2)    高(3)
可能性   高(3)   3        6        9
(Likelihood) 中(2)   2        4        6
            低(1)   1        2        3
```

**图 20-1. 确定架构风险的矩阵**

要了解如何使用风险矩阵，假设对应用中使用的主中央数据库的可用性存在担忧。首先考虑影响维度——如果数据库宕机或不可用，整体影响是什么？架构师可能认为这是高风险，使该风险为 3（中）、6（高）或 9（高）。然而，在应用第二维度（风险发生可能性）后，架构师意识到数据库在集群配置的高可用服务器上，因此数据库不可用的可能性很低。因此，高影响与低可能性的交叉点得出整体风险评级为 3（中风险）。

::: tip 建议
利用风险矩阵评估风险时，先考虑影响维度，再考虑可能性维度。
:::

## 风险评估（Risk Assessments）

前一节描述的风险矩阵可用于构建所谓的**风险评估**。风险评估是关于架构在某种有意义的评估标准方面整体风险的汇总报告。

风险评估可能差异很大，但通常包含基于应用服务或领域区域的某些评估标准的风险（由风险矩阵评估）。此基本风险评估报告格式如图 20-2 所示，其中浅灰（1-2）为低风险，中灰（3-4）为中风险，深灰（6-9）为高风险。通常这些用绿色（低）、黄色（中）和红色（高）进行颜色编码，但灰度在黑白渲染和色盲情况下也有用。

| 领域/服务 | 可用性 | 性能 | 可扩展性 | 数据完整性 |
|-----------|--------|------|----------|------------|
| 客户注册 | 6 | 4 | 3 | 9 |
| 目录结账 | 2 | 3 | 6 | 2 |
| 订单履行 | 2 | 2 | 2 | 6 |

**图 20-2. 标准风险评估示例**

风险矩阵的量化风险可以按风险标准以及按服务或领域区域累积。例如，注意图 20-2 中数据完整性的累积风险是最高风险区域，总计 17，而可用性的累积风险仅为 10（风险最少）。每个领域区域的相对风险也可以由示例风险评估确定。这里，客户注册承载最高区域风险，而订单履行承载最低风险。这些相对数字可以随后被跟踪，以展示特定风险类别或领域区域内风险的改善或恶化。

虽然图 20-2 的风险评估示例包含所有风险分析结果，但很少以这种方式呈现。过滤对于在给定背景下直观地传达特定信息至关重要。例如，假设架构师在会议中目的是呈现系统的高风险区域。与其按图 20-2 所示呈现风险评估，可以使用过滤只显示高风险区域（如图 20-3 所示），提高整体信噪比并呈现系统状态的清晰图景（好或坏）。

**图 20-3. 过滤风险评估仅显示高风险**

图 20-2 的另一个问题是，该评估报告只显示时间快照；它不显示事情是在改善还是恶化。换句话说，图 20-2 不显示风险的方向。呈现风险方向有些棘手。如果使用上下箭头表示方向，上箭头意味着什么？事情是变好还是变坏？我们多年来询问人们上箭头意味着事情是逐渐变坏还是变好，几乎 50% 的人说上箭头意味着事情逐渐变坏，而几乎 50% 的人说上箭头表示事情变好。左右箭头也是如此。因此，使用箭头表示方向时，必须使用图例。然而，我们也发现这也不起作用。一旦用户滚动超过图例，混淆会再次发生。

我们通常使用风险评级旁边的通用方向符号加号（+）和减号（-）来表示方向，如图 20-4 所示。注意图 20-4 中，虽然客户注册的性能为中（4），但方向是减号（红色），表示正在逐渐恶化并走向高风险。另一方面，注意目录结账的可扩展性为高（6）带加号（绿色），显示正在改善。没有加号或减号的风险评级表示风险稳定，既未变好也未变坏。

**图 20-4. 用加减号显示风险方向**

有时，即使加减号也会让某些人困惑。表示方向的另一种技术是利用箭头以及风险评级数字所趋向的数字。如图 20-5 所示，该技术不需要图例，因为方向很清楚。此外，使用颜色（红色箭头表示更差，绿色箭头表示更好）使风险走向更加清晰。

**图 20-5. 用箭头和数字显示风险方向**

风险方向可以通过本书前面描述的适应度函数进行持续测量来确定。通过客观分析每个风险标准，可以观察到趋势，提供每个风险标准的方向。

## 风险风暴（Risk Storming）

没有架构师能独自确定系统的整体风险。原因有二。首先，单个架构师可能遗漏或忽视风险区域，很少有架构师对系统的每个部分都有全面了解。这就是风险风暴可以发挥作用的地方。

风险风暴是一种协作练习，用于确定特定维度内的架构风险。常见维度（风险区域）包括：未经验证的技术、性能、可扩展性、可用性（包括传递依赖）、数据丢失、单点故障和安全。虽然大多数风险风暴工作涉及多位架构师，但明智的做法是也包括高级开发者和技术负责人。他们不仅会从实施角度提供架构风险视角，而且让开发者参与有助于他们更好地理解架构。

风险风暴工作包括个人部分和协作部分。在个人部分，所有参与者单独（无协作）使用前一节描述的风险矩阵为架构区域分配风险。风险风暴的这一非协作部分至关重要，以便参与者不会影响或将注意力从架构的特定区域转移。在风险风暴的协作部分，所有参与者共同工作以就风险区域达成共识、讨论风险并形成降低风险的解决方案。

架构图用于风险风暴工作的两个部分。对于整体风险评估，通常使用综合架构图，而应用特定区域内的风险风暴会使用上下文架构图。进行风险风暴工作的架构师有责任确保这些图是最新的并可供所有参与者使用。

风险风暴分为三个主要活动：

1. **识别（Identification）**
2. **共识（Consensus）**
3. **缓解（Mitigation）**

识别始终是个人、非协作活动，而共识和缓解始终是协作的，涉及所有参与者在同一房间（至少虚拟地）共同工作。以下各节详细讨论这些主要活动。

### 识别（Identification）

风险风暴的识别活动涉及每个参与者单独识别架构内的风险区域。以下步骤描述风险风暴工作的识别部分：

1. 进行风险风暴的架构师在协作部分前一到两天向所有参与者发出邀请。邀请包含架构图（或查找位置）、风险风暴维度（该特定风险风暴工作正在分析的风险区域）、风险风暴协作部分的日期和地点。
2. 使用本章第一节描述的风险矩阵，参与者单独分析架构并将风险分类为低（1-2）、中（3-4）或高（6-9）。
3. 参与者准备带有相应颜色（绿、黄、红）的小便利贴，并写下相应的风险数字（在风险矩阵上找到）。

大多数风险风暴工作只涉及分析一个特定维度（如性能），但由于人员可用性或时间问题，有时可能在一次风险风暴工作中分析多个维度（如性能、可扩展性和数据丢失）。当在一次风险风暴工作中分析多个维度时，参与者在便利贴上在风险数字旁边写下维度，以便每个人都知道具体维度。例如，假设三位参与者在中央数据库发现了风险。三人都将风险识别为高（6），但一位参与者发现的是可用性方面的风险，而两位参与者发现的是性能方面的风险。这两个维度将分别讨论。

::: tip 建议
尽可能将风险风暴工作限制在单一维度。这允许参与者将注意力集中在该特定维度上，并避免对架构同一区域识别多个风险区域的混淆。
:::

### 共识（Consensus）

风险风暴工作中的共识活动高度协作，目标是在所有参与者之间就架构内的风险达成共识。当有大尺寸打印的架构图可用并贴在墙上时，该活动最有效。如果没有大尺寸打印版本，可以在大屏幕上显示电子版本。

到达风险风暴会议时，参与者开始将他们的便利贴放在他们单独发现风险的架构图区域。如果使用电子版本，进行风险风暴会议的架构师会询问每位参与者并在识别风险的架构区域电子放置风险（见图 20-7）。

**图 20-7. 风险区域的初步识别**

一旦所有便利贴就位，风险风暴的协作部分就可以开始。风险风暴此活动的目标是作为团队分析风险区域并就风险评估达成共识。注意图 20-7 中识别的几个风险区域：

1. 两位参与者单独将弹性负载均衡器识别为中风险（3），而一位参与者识别为高风险（6）。
2. 一位参与者单独将推送扩展服务器识别为高风险（9）。
3. 三位参与者单独将 MySQL 数据库识别为中风险（3）。
4. 一位参与者单独将 Redis 缓存识别为高风险（9）。
5. 三位参与者将 MongoDB 日志识别为低风险（2）。
6. 架构的所有其他区域被认为不承载任何风险，因此架构的其他区域没有便利贴。

列表中第 3 和第 5 项在此活动中不需要进一步讨论，因为所有参与者就风险水平和评估达成一致。然而，注意第 1 项存在意见分歧，第 2 和第 4 项只有一位参与者识别了风险。这些项需要在此活动中讨论。

第 1 项显示两位参与者单独将弹性负载均衡器识别为中风险（3），而一位参与者识别为高风险（6）。在这种情况下，另外两位参与者询问第三位参与者为何将风险识别为高。假设第三位参与者说他们将风险评为高是因为如果弹性负载均衡器宕机，整个系统将无法访问。虽然这是事实，确实将整体影响评级提高到高，但另外两位参与者说服第三位参与者这种情况发生的可能性很低。经过大量讨论，第三位参与者同意，将该风险水平降至中（3）。然而，第一和第二位参与者可能没有看到第三位参与者看到的弹性负载均衡器的特定风险方面，因此需要风险风暴此活动中的协作。

举例说明，考虑列表中第 2 项，一位参与者单独将推送扩展服务器识别为高风险（9），而其他参与者没有将其识别为任何风险。在这种情况下，所有其他参与者询问识别该风险的参与者为何将其评为高。该参与者然后说他们在高负载下推送扩展服务器持续宕机方面有过糟糕经历，这正是此架构所具有的。此示例展示了风险风暴的价值——没有该参与者的参与，没有人会看到高风险（当然直到深入生产！）。

列表中第 4 项是个有趣的案例。一位参与者将 Redis 缓存识别为高风险（9），而其他参与者没有看到该缓存在架构中有任何风险。其他参与者询问该区域高风险的 rationale 是什么，该参与者回答说：「什么是 Redis 缓存？」在这种情况下，Redis 对该参与者来说是未知的，因此该区域是高风险。

::: tip 建议
对于未经验证或未知的技术，始终分配最高风险评级（9），因为风险矩阵无法用于此维度。
:::

列表中第 4 项的例子说明了为何将开发者带入风险风暴会议是明智的（且重要的）。开发者不仅可以更多地了解架构，而且一位参与者（本例中是团队中的开发者）不知道给定技术这一事实为架构师提供了关于整体风险的宝贵信息。

此过程持续到所有参与者就识别的风险区域达成一致。一旦所有便利贴合并，此活动结束，下一活动可以开始。此活动的最终结果如图 20-8 所示。

### 缓解（Mitigation）

一旦所有参与者就架构风险区域的评估达成一致，最后也是最重要的活动发生——**风险缓解**。缓解架构内的风险通常涉及对否则可能被认为完美的某些区域进行更改或增强。

此活动（通常也是协作的）寻求减少或消除第一活动中识别的风险的方法。可能有些情况下，基于风险识别，原始架构需要完全更改，而其他情况可能是直接的架构重构，例如添加队列以进行背压来减少吞吐量瓶颈问题。

无论架构需要什么更改，此活动通常会产生额外成本。因此，关键利益相关者通常决定成本是否超过风险。例如，假设通过风险风暴会议，中央数据库在整体系统可用性方面被识别为中风险（4）。在这种情况下，参与者同意对数据库进行集群，并将单一数据库拆分为独立的物理数据库，将缓解该风险。然而，虽然风险将显著降低，但该解决方案将花费 20,000 美元。架构师然后将与关键业务利益相关者举行会议讨论此权衡。在此谈判期间，业务负责人决定价格太高，成本不超过风险。架构师没有放弃，然后建议不同的方法——跳过集群，只将数据库拆分为两部分怎么样？在这种情况下，成本降至 8,000 美元，同时仍缓解大部分风险。在这种情况下，利益相关者同意该解决方案。

前一场景展示了风险风暴不仅可以对整体架构产生影响，还可以对架构师与业务利益相关者之间的谈判产生影响。风险风暴与本章开头描述的风险评估相结合，为识别和跟踪风险、改善架构以及处理关键利益相关者之间的谈判提供了绝佳载体。

## 敏捷故事风险分析（Agile Story Risk Analysis）

风险风暴除了架构之外，还可用于软件开发的其他方面。例如，我们在故事梳理期间利用风险风暴来确定给定敏捷迭代内用户故事完成的整体风险（以及 consequently 该迭代的整体风险评估）。使用风险矩阵，用户故事风险可以通过第一维度（如果故事在迭代内未完成的整体影响）和第二维度（故事未完成的可能性）来识别。通过利用相同的架构风险矩阵处理故事，团队可以识别高风险故事、仔细跟踪它们并确定优先级。

## 风险风暴示例（Risk Storming Examples）

为说明风险风暴的力量以及它如何改善系统的整体架构，考虑支持护士就各种健康状况向患者提供建议的呼叫中心系统示例。此类系统的需求如下：

- 系统将使用第三方诊断引擎，为护士或患者提供问题并指导其医疗问题。
- 患者可以通过呼叫中心致电与护士交谈，或选择使用直接访问诊断引擎的自助网站，绕过护士。
- 系统必须支持全国 250 名并发护士和全国数十万并发自助患者。
- 护士可以通过医疗记录交换访问患者医疗记录，但患者无法访问自己的医疗记录。
- 系统在医疗记录方面必须符合 HIPAA。这意味着除了护士之外，任何人都不能访问医疗记录至关重要。
- 需要在系统中解决疫情和感冒流感季节的高流量。
- 呼叫路由到护士基于护士档案（如双语需求）。
- 第三方诊断引擎每秒可处理约 500 个请求。

系统架构师创建了图 20-9 所示的高层架构。在此架构中有三个独立的基于 Web 的用户界面：一个用于自助服务，一个用于接听电话的护士，一个用于管理人员添加和维护护士档案及配置设置。系统的呼叫中心部分由接收呼叫的呼叫接受器和根据护士档案将呼叫路由到下一个可用护士的呼叫路由器组成（注意呼叫路由器如何访问中央数据库获取护士档案信息）。此架构的核心是诊断系统 API 网关，它执行安全检查并将请求定向到适当的后端服务。

**图 20-9. 护士诊断系统示例的高层架构**

系统中有四个主要服务：案例管理服务、护士档案管理服务、医疗记录交换接口和外部第三方诊断引擎。除外部系统和呼叫中心服务的专有协议外，所有通信均使用 REST。

架构师已多次审查此架构并认为已准备好实施。作为自我评估，研究需求和图 20-9 的架构图，尝试确定此架构在可用性、弹性和安全方面的风险水平。确定风险水平后，确定架构需要什么更改来缓解该风险。以下各节包含可用于比较的场景。

### 可用性（Availability）

在第一次风险风暴练习中，架构师选择首先关注可用性，因为系统可用性对此系统的成功至关重要。在风险风暴识别和协作活动后，参与者使用风险矩阵得出以下风险区域（如图 20-10 所示）：

- 中央数据库的使用被识别为高风险（6），因为高影响（3）和中可能性（2）。
- 诊断引擎可用性被识别为高风险（9），因为高影响（3）和未知可能性（3）。
- 医疗记录交换可用性被识别为低风险（2），因为它不是系统运行所需的组件。
- 由于每个服务的多个实例和 API 网关的集群，系统的其他部分不被认为有可用性风险。

**图 20-10. 可用性风险区域**

在风险风暴工作中，所有参与者同意，虽然如果数据库宕机护士可以手动写下案例笔记，但呼叫路由器在数据库不可用时无法运行。为缓解数据库风险，参与者选择将单一物理数据库拆分为两个独立数据库：一个包含护士档案信息的集群数据库，一个用于案例笔记的单实例数据库。此架构更改不仅解决了对数据库可用性的担忧，还帮助保护案例笔记免受管理员访问。缓解此风险的另一个选择是在呼叫路由器中缓存护士档案信息。然而，由于呼叫路由器的实现未知，可能是第三方产品，参与者选择了数据库方法。

缓解外部系统（诊断引擎和医疗记录交换）可用性风险更难管理，因为缺乏对这些系统的控制。缓解此类可用性风险的一种方法是研究这些系统是否有公布的服务级别协议（SLA）或服务级别目标（SLO）。SLA 通常是合同协议并具有法律约束力，而 SLO 通常不是。根据研究，架构师发现诊断引擎的 SLA 保证 99.99% 可用（即每年 52.60 分钟停机时间），医疗记录交换保证 99.9% 可用性（即每年 8.77 小时停机时间）。根据相对风险，此信息足以消除识别的风险。

此风险风暴会议后架构的相应更改如图 20-11 所示。注意现在使用两个数据库，SLA 也发布在架构图上。

**图 20-11. 解决可用性风险的架构修改**

### 弹性（Elasticity）

在第二次风险风暴练习中，架构师选择关注弹性——用户负载的峰值（也称为可变可扩展性）。虽然只有 250 名护士（为大多数服务提供了自动调节器），但系统的自助服务部分也可以像护士一样访问诊断引擎，显著增加对诊断接口的请求数量。参与者担心疫情和流感季节，届时系统的预期负载将显著增加。

在风险风暴会议期间，参与者都将诊断引擎接口识别为高风险（9）。每秒只有 500 个请求，参与者计算出诊断引擎接口无法跟上预期吞吐量，特别是当前架构使用 REST 作为接口协议。

缓解此风险的一种方法是在 API 网关和诊断引擎接口之间利用异步队列（消息）在诊断引擎调用积压时提供背压点。虽然这是好做法，但仍无法缓解风险，因为护士（以及自助患者）会等待诊断引擎的响应太久，这些请求可能会超时。利用所谓的**救护车模式（Ambulance Pattern）** 会给护士比自助服务更高的优先级。因此需要两个消息通道。虽然此技术有助于缓解风险，但仍无法解决等待时间。参与者决定，除了提供背压的队列技术外，缓存与疫情相关的特定诊断问题将消除疫情和流感呼叫 ever 到达诊断引擎接口的需求。

相应的架构更改如图 20-12 所示。注意除了两个队列通道（一个用于护士，一个用于自助患者）外，还有一个名为诊断疫情缓存服务器的新服务，处理与特定疫情或流感相关问题的所有请求。有了此架构，限制因素被移除（对诊断引擎的调用），允许数万并发请求。没有风险风暴工作，此风险可能直到疫情或流感季节发生才会被识别。

**图 20-12. 解决弹性风险的架构修改**

### 安全（Security）

在前两次风险风暴工作的结果和成功鼓舞下，架构师决定就另一个必须支持以确保系统成功的重要架构特性——安全——举行最后一次风险风暴会议。由于 HIPAA 监管要求，通过医疗记录交换接口访问医疗记录必须安全，仅允许护士在需要时访问医疗记录。架构师认为这不是问题，因为 API 网关中有安全检查（认证和授权），但好奇参与者是否发现任何其他安全风险元素。

在风险风暴期间，参与者都将诊断系统 API 网关识别为高安全风险（6）。此高评级的理由是管理员或自助患者访问医疗记录的高影响（3）加上中可能性（2）。风险发生可能性未被评为高是因为每个 API 调用的安全检查，但仍评为中因为所有调用（自助服务、管理员和护士）都通过同一 API 网关。架构师最初只将风险评为低（2），在风险风暴共识活动中被说服风险实际上是高的，需要缓解。

参与者都同意，为每种用户类型（管理员、自助服务/诊断和护士）设置独立的 API 网关将防止来自管理员 Web 用户界面或自助服务 Web 用户界面的调用 ever 到达医疗记录交换接口。架构师同意，创建了最终架构，如图 20-13 所示。

**图 20-13. 解决安全风险的最终架构修改**

前一场景说明了风险风暴的力量。通过与其他架构师、开发者和关键利益相关者在对系统成功至关重要的风险维度上协作，识别出了否则会被忽视的风险区域。比较图 20-9 和图 20-13，注意风险风暴前后架构的显著差异。这些显著更改解决了架构内的可用性、弹性和安全担忧。

风险风暴不是一次性过程。相反，它是任何系统生命周期内的持续过程，以在生产中发生之前捕获和缓解风险区域。风险风暴工作的频率取决于许多因素，包括变更频率、架构重构工作和架构的增量开发。在添加主要功能后或每次迭代结束时，对某个特定维度进行风险风暴工作是典型的。
