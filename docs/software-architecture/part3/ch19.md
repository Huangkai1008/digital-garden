# 第19章 架构决策

架构师的核心职责之一是做出架构决策。架构决策通常涉及应用程序或系统的结构，但也可能涉及技术决策，尤其是当这些技术决策影响架构特性时。无论何种情境，好的架构决策都能帮助开发团队做出正确的技术选择。做出架构决策需要收集足够的相关信息、论证决策、记录决策，并有效地将决策传达给相关利益方。

## 架构决策反模式（Architecture Decision Anti-Patterns）

做出架构决策是一门艺术。不出所料，架构师在决策时会出现若干架构反模式。程序员 Andrew Koenig 将反模式定义为：开始时看似好主意，但会把你引入困境的东西。反模式的另一种定义是：产生负面结果的可重复过程。在做出架构决策时，可能出现（且通常会出现）的三大架构反模式是：**明哲保身（Covering Your Assets）** 反模式、**土拨鼠日（Groundhog Day）** 反模式和**邮件驱动架构（Email-Driven Architecture）** 反模式。这三种反模式通常呈递进关系：克服明哲保身反模式会导致土拨鼠日反模式，克服后者又会导致邮件驱动架构反模式。要做出有效且准确的架构决策，架构师必须克服这三种反模式。

### 明哲保身反模式（Covering Your Assets Anti-Pattern）

在尝试做出架构决策时，首先出现的是明哲保身反模式。当架构师因害怕选错而回避或推迟架构决策时，就会出现这种反模式。

克服此反模式有两种方式。第一种是等到**最后责任时刻（last responsible moment）** 再做出重要架构决策。最后责任时刻意味着等到你有足够信息来论证和验证决策，但不要等太久，以免阻碍开发团队或陷入**分析瘫痪（Analysis Paralysis）** 反模式。第二种避免此反模式的方式是持续与开发团队协作，确保你做出的决策能按预期实施。这一点至关重要，因为架构师不可能了解某项技术的所有细节及所有相关问题。通过与开发团队密切协作，架构师可以在出现问题时迅速调整架构决策。

举例说明：假设架构师决定所有与产品相关的参考数据（产品描述、重量、尺寸）都缓存在所有需要该信息的服务实例中，使用只读复制缓存，主副本由目录服务拥有。复制缓存意味着如果产品信息有任何变更（或添加新产品），目录服务会更新其缓存，然后通过复制（内存）缓存产品复制到所有其他需要该数据的服务。这一决策的合理理由是减少服务间耦合，并有效共享数据而无需进行服务间调用。然而，实施该架构决策的开发团队发现，由于某些服务的可扩展性要求，该决策需要比可用内存更多的进程内内存。通过与开发团队密切协作，架构师可以迅速了解问题并调整架构决策以适应这些情况。

### 土拨鼠日反模式（Groundhog Day Anti-Pattern）

一旦架构师克服明哲保身反模式并开始做决策，就会出现第二种反模式：土拨鼠日反模式。土拨鼠日反模式发生在人们不知道决策为何做出时，因此会一遍又一遍地反复讨论。该反模式得名于 Bill Murray 的电影《土拨鼠日》，片中每天都是 2 月 2 日。

土拨鼠日反模式之所以发生，是因为架构师做出架构决策后，未能提供决策的论证（或完整论证）。在论证架构决策时，同时提供技术和业务论证很重要。例如，架构师可能决定将单体应用拆分为独立服务，以解耦应用的功能方面，使应用的各部分使用更少的虚拟机资源，并能独立维护和部署。虽然这是技术论证的好例子，但缺少的是业务论证——换句话说，业务为什么要为这次架构重构买单？该决策的良好业务论证可能是更快交付新业务功能，从而缩短上市时间。另一个可能是降低与新功能开发和发布相关的成本。

在论证决策时提供业务价值对任何架构决策都至关重要。这也是判断是否应该首先做出该架构决策的良好试金石。如果某项架构决策不提供任何业务价值，那或许不是好决策，应重新考虑。

四种最常见的业务论证包括：成本、上市时间、用户满意度和战略定位。在关注这些常见业务论证时，重要的是考虑业务利益相关者关心什么。仅基于成本节约论证某项决策可能不是正确做法，如果业务利益相关者更关心上市时间而非成本。

### 邮件驱动架构反模式（Email-Driven Architecture Anti-Pattern）

一旦架构师做出决策并充分论证，就会出现第三种架构反模式：邮件驱动架构。邮件驱动架构反模式是指人们丢失、遗忘或根本不知道已做出架构决策，因此无法实施该决策。这种反模式关乎有效传达你的架构决策。邮件是很好的沟通工具，但作为文档仓库系统则很差。

有许多方式可以提高传达架构决策的有效性，从而避免邮件驱动架构反模式。传达架构决策的第一条规则是：**不要在邮件正文中包含架构决策**。在邮件正文中包含架构决策会为该决策创建多个记录系统。很多时候重要细节（包括论证）会从邮件中遗漏，从而再次制造土拨鼠日反模式。此外，如果该架构决策被更改或取代，有多少人会收到修订后的决策？更好的做法是只在邮件正文中提及决策的性质和背景，并提供指向单一记录系统的链接，以获取实际架构决策及相应细节（无论是维基页面链接还是文件系统中的文档）。

有效传达架构决策的第二条规则是：**只通知真正关心该架构决策的人**。一种有效技巧是按以下方式撰写邮件正文：

> 「嗨 Sandra，我针对服务间通信做了一项重要决策，直接影响你。请通过以下链接查看决策……」

注意第一句的措辞：「关于服务间通信的重要决策」。这里提到了决策的背景，但没有提及实际决策本身。第一句的第二部分更重要：「直接影响你」。如果架构决策不直接影响某人，为什么要打扰他们？这是判断哪些利益相关者（包括开发者）应直接收到架构决策通知的良好试金石。第二句提供架构决策位置的链接，因此决策只存在于一处，形成单一记录系统。

## 架构显著性（Architecturally Significant）

许多架构师认为，如果架构决策涉及任何具体技术，那就不是架构决策，而是技术决策。这并不总是对的。如果架构师因某项技术直接支持特定架构特性（如性能或可扩展性）而决定使用该技术，那就是架构决策。

知名软件架构师、《Release It!》（Pragmatic Bookshelf）作者 Michael Nygard 通过创造**架构显著性（architecturally significant）** 一词，解决了架构师应对哪些决策负责（因而什么是架构决策）的问题。根据 Michael 的说法，架构显著性决策是那些影响结构、非功能特性、依赖、接口或构建技术的决策。

- **结构**：指影响所用架构模式或风格的决策。例如，决定在一组微服务之间共享数据。该决策影响微服务的限界上下文，从而影响应用的结构。
- **非功能特性**：指对正在开发或维护的应用或系统重要的架构特性（「-ilities」）。如果技术选择影响性能，而性能是应用的重要方面，则成为架构决策。
- **依赖**：指系统内组件和/或服务之间的耦合点，进而影响整体可扩展性、模块化、敏捷性、可测试性、可靠性等。
- **接口**：指服务和组件的访问与编排方式，通常通过网关、集成中心、服务总线或 API 代理。接口通常涉及定义契约，包括这些契约的版本控制和弃用策略。接口影响使用系统的其他人，因此具有架构显著性。
- **构建技术**：指关于平台、框架、工具甚至流程的决策，虽然本质上是技术性的，但可能影响架构的某些方面。

## 架构决策记录（Architecture Decision Records, ADRs）

记录架构决策最有效的方式之一是**架构决策记录（Architecture Decision Records, ADRs）**。ADR 由 Michael Nygard 在博客文章中首次推广，后来在 ThoughtWorks 技术雷达中被标记为「采用」。ADR 由描述特定架构决策的短文本文件（通常一到两页）组成。ADR 可以用纯文本编写，但通常使用 AsciiDoc 或 Markdown 等文本文档格式。或者，ADR 也可以使用维基页面模板编写。

也有用于管理 ADR 的工具。《Growing Object-Oriented Software Guided by Tests》（Addison-Wesley）合著者 Nat Pryce 编写了名为 ADR-tools 的开源工具。ADR-tools 提供命令行界面来管理 ADR，包括编号方案、位置和取代逻辑。德国软件工程师 Micha Kops 写了一篇关于使用 ADR-tools 的博客文章，提供了如何用其管理架构决策记录的好例子。

### 基本结构（Basic Structure）

ADR 的基本结构由五个主要部分组成：标题、状态、背景、决策和后果。我们通常在基本结构中增加两个部分：合规性和备注。该基本结构（如图 19-1 所示）可扩展以包含任何其他需要的部分，只要模板保持一致且简洁。例如，如需提供所有其他可能替代方案的分析，可添加「替代方案」部分。

```
┌─────────────────────────────────────────────────────────┐
│  ADR 基本结构                                            │
├─────────────────────────────────────────────────────────┤
│  Title（标题）                                            │
│  Status（状态）                                           │
│  Context（背景）                                          │
│  Decision（决策）                                         │
│  Consequences（后果）                                     │
│  Compliance（合规性）                                     │
│  Notes（备注）                                            │
└─────────────────────────────────────────────────────────┘
```

### 标题（Title）

ADR 的标题通常按顺序编号，并包含描述架构决策的短语。例如，订单服务与支付服务之间使用异步消息的决策可能写作：「42. 订单与支付服务间使用异步消息」。标题应足够描述性以消除决策性质和背景的歧义，同时简短扼要。

### 状态（Status）

ADR 的状态可标记为**提议（Proposed）**、**已接受（Accepted）** 或**已取代（Superseded）**。提议状态表示决策必须由更高级别的决策者或某种架构治理机构（如架构评审委员会）批准。已接受状态表示决策已获批准并准备实施。已取代状态表示决策已被更改并被另一 ADR 取代。已取代状态始终假定先前 ADR 状态为已接受；换言之，提议的 ADR 不会被另一 ADR 取代，而是会继续修改直至被接受。

已取代状态是保留历史记录的有力方式，记录做出了哪些决策、当时为何做出、新决策是什么以及为何更改。通常，当 ADR 被取代时，会标记取代它的决策。同样，取代另一 ADR 的决策会标记它取代的 ADR。例如，假设 ADR 42（「订单与支付服务间使用异步消息」）先前已批准，但由于支付服务实现和位置的后续变更，两个服务之间现在必须使用 REST（ADR 68）。状态将如下所示：

```
ADR 42. 订单与支付服务间使用异步消息
状态：被 68 取代

ADR 68. 订单与支付服务间使用 REST
状态：已接受，取代 42
```

ADR 42 和 68 之间的链接和历史轨迹避免了关于 ADR 68 的「为什么不用消息？」这类不可避免的问题。

### ADR 与征求意见稿（RFC）

如果架构师希望发送 ADR 草稿征求意见（当架构师希望与更广泛的利益相关者验证各种假设和主张时，这有时是好主意），我们建议创建名为**征求意见稿（Request for Comments, RFC）** 的新状态，并指定审查完成的截止日期。这种做法避免了决策被永远讨论但从未真正做出的分析瘫痪反模式。一旦到达该日期，架构师可以分析 ADR 上的所有评论，对决策进行必要调整，做出最终决策，并将状态设为提议（除非架构师能够自行批准决策，此时状态将设为已接受）。ADR 的 RFC 状态示例：

```
状态
征求意见稿，截止日期 2010 年 1 月 9 日
```

ADR 状态部分的另一个重要方面是，它迫使架构师与老板或首席架构师就他们可以自行批准架构决策的标准进行必要对话，或是否必须通过更高级别架构师、架构评审委员会或其他架构治理机构批准。

这些对话的良好起点是三个标准：成本、跨团队影响和安全。成本可包括软件采购或许可费、额外硬件成本，以及实施架构决策的总体工作量。工作量成本可通过实施架构决策的预估小时数乘以公司的标准全职等效（FTE）费率来估算。项目负责人或项目经理通常有 FTE 金额。如果架构决策的成本超过一定金额，则必须设为提议状态并由他人批准。如果架构决策影响其他团队或系统或有任何安全影响，则不能由架构师自行批准，必须由更高级别治理机构或首席架构师批准。

一旦标准及相应限制已建立并达成一致（例如「超过 €5,000 的成本必须由架构评审委员会批准」），应妥善记录该标准，以便所有创建 ADR 的架构师知道何时可以、何时不能自行批准架构决策。

### 背景（Context）

ADR 的背景部分说明起作用的力量。换句话说，「什么情况迫使我做出此决策？」ADR 的这一部分允许架构师描述具体情况或问题，并简洁阐述可能的替代方案。如果架构师需要详细记录每个替代方案的分析，可以添加额外的「替代方案」部分，而不是将分析加入背景部分。

背景部分还提供记录架构的方式。通过描述背景，架构师也在描述架构。这是以清晰简洁的方式记录架构特定区域的有效方法。继续前一节的例子，背景可能如下：

> 「订单服务必须将信息传递给支付服务以支付当前正在下的订单。这可以通过 REST 或异步消息完成。」

注意，这一简洁陈述不仅指定了场景，还指定了替代方案。

### 决策（Decision）

ADR 的决策部分包含架构决策及决策的完整论证。Michael Nygard 引入了一种很好的陈述架构决策的方式：使用非常肯定、命令式的语气，而非被动语气。例如，服务间使用异步消息的决策应写作「我们将使用服务间的异步消息」。这比「我认为服务间使用异步消息会是最好的选择」好得多。注意后者不清楚决策是什么，甚至是否已做出决策——只陈述了架构师的意见。

ADR 决策部分最有力的方面之一是，它允许架构师更强调**为什么**而非**如何**。理解为何做出决策远比理解某事如何运作更重要。大多数架构师和开发者可以通过查看上下文图识别事物如何运作，但不知道为何做出决策。了解为何做出决策及相应论证有助于人们更好地理解问题背景，并避免通过重构到可能产生问题的其他解决方案而犯错误。

举例说明：考虑几年前使用 Google 远程过程调用（gRPC）作为两个服务间通信手段的原始架构决策。在不理解为何做出该决策的情况下，几年后另一位架构师选择推翻该决策并改用消息传递以更好地解耦服务。然而，实施此重构突然导致延迟显著增加，进而最终导致上游系统发生超时。若了解最初使用 gRPC 是为了显著降低延迟（以紧密耦合服务为代价），本可以在一开始就阻止该重构。

### 后果（Consequences）

ADR 的后果部分是非常有力的部分。该部分记录架构决策的整体影响。架构师做出的每个架构决策都有某种影响，既有好的也有坏的。必须指定架构决策的影响会迫使架构师思考这些影响是否超过决策的好处。

该部分的另一个好用途是记录与架构决策相关的权衡分析。这些权衡可以是基于成本的，或针对其他架构特性（「-ilities」）的权衡。例如，考虑使用异步（即发即忘）消息在网站上发布评论的决策。该决策的理由是将发布评论请求的响应时间从 3,100 毫秒显著提高到 25 毫秒，因为用户无需等待实际评论发布（只需等待消息发送到队列）。虽然这是好理由，但有人可能争辩这是坏主意，因为异步请求相关的错误处理很复杂（「如果有人发布包含不良词汇的评论会怎样？」）。挑战此决策的人不知道，该问题已与业务利益相关者和其他架构师讨论过，从权衡角度决定提高响应性并处理复杂错误处理比同步向用户提供评论成功发布的反馈等待时间更重要。通过利用 ADR，该权衡分析可以包含在后果部分，提供架构决策背景（和权衡）的完整图景，从而避免这些情况。

### 合规性（Compliance）

ADR 的合规性部分不是 ADR 的标准部分，但我们强烈建议添加。合规性部分迫使架构师思考如何从合规角度衡量和治理架构决策。架构师必须决定该决策的合规检查必须是手动的，还是可以使用适应度函数（fitness function）自动化。如果可以自动化，架构师可以在此部分指定如何编写该适应度函数，以及是否需要代码库的其他更改来衡量此架构决策的合规性。

例如，考虑图 19-2 所示传统 n 层分层架构中的以下架构决策：业务层中业务对象使用的所有共享对象将驻留在共享服务层，以隔离和包含共享功能。

```
┌─────────────────────────────────────────────────────────┐
│  业务层（Business Layer）                                │
│  ┌─────────────┐  ┌─────────────┐                       │
│  │ 业务对象 A  │  │ 业务对象 B  │                       │
│  └──────┬──────┘  └──────┬──────┘                       │
│         │                │                               │
│         └────────┬───────┘                               │
│                  ▼                                       │
│  ┌─────────────────────────────────┐                     │
│  │ 共享服务层（Shared Services）    │                     │
│  └─────────────────────────────────┘                     │
└─────────────────────────────────────────────────────────┘
```

**图 19-2. 架构决策示例**

该架构决策可以使用 Java 的 ArchUnit 或 C# 的 NetArchTest 自动衡量和治理。例如，使用 Java 的 ArchUnit，自动化适应度函数测试可能如下：

```java
@Test
public void shared_services_should_reside_in_services_layer() {
    classes().that().areAnnotatedWith(SharedService.class)
        .should().resideInAPackage("..services..")
        .because("All shared services classes used by business " +
                 "objects in the business layer should reside in the services " +
                 "layer to isolate and contain shared logic")
        .check(myClasses);
}
```

注意，此自动化适应度函数需要编写新故事来创建新的 Java 注解（@SharedService），然后将此注解添加到所有共享类。该部分还指定测试是什么、测试在哪里可以找到，以及测试将如何执行和何时执行。

### 备注（Notes）

另一个非标准 ADR 部分但我们强烈建议添加的是备注部分。该部分包含 ADR 的各种元数据，例如：

- 原始作者
- 批准日期
- 批准人
- 取代日期
- 最后修改日期
- 修改人
- 最后修改内容

即使将 ADR 存储在版本控制系统（如 Git）中，除仓库能支持的内容外，额外的元信息也有用，因此我们建议无论 ADR 如何存储、存储在哪里都添加此部分。

### 存储 ADR（Storing ADRs）

架构师创建 ADR 后，必须存储在某个地方。无论 ADR 存储在哪里，每个架构决策都应有自己的文件或维基页面。有些架构师喜欢将 ADR 与源代码一起保存在 Git 仓库中。将 ADR 保存在 Git 仓库中允许对 ADR 进行版本控制和跟踪。然而，对于大型组织，我们出于几个原因建议谨慎使用此做法。首先，需要查看架构决策的人可能无法访问 Git 仓库。其次，这不是存储应用 Git 仓库之外背景的 ADR 的好地方（如集成架构决策、企业架构决策或每个应用通用的决策）。

因此，我们建议将 ADR 存储在维基（使用维基模板）或共享文件服务器上可被维基或其他文档渲染软件轻松访问的共享目录中。图 19-3 显示了此目录结构（或维基页面导航结构）的示例。

```
ADR 根目录/
├── application/          # 应用相关决策
│   ├── common/           # 所有应用通用
│   ├── ATP/              # ATP 应用
│   └── PSTD/             # PSTD 应用
├── integration/          # 集成相关决策
└── enterprise/           # 企业架构决策
```

**图 19-3. 存储 ADR 的目录结构示例**

应用目录包含特定于某种应用背景的架构决策。该目录进一步细分为子目录。common 子目录用于适用于所有应用的架构决策，例如「所有与框架相关的类将包含标识该类属于底层框架代码的注解（Java 中的 @Framework）或属性（C# 中的 [Framework]）」。应用目录下的子目录对应特定应用或系统背景，包含该应用或系统特定的架构决策（本例中为 ATP 和 PSTD 应用）。集成目录包含涉及应用、系统或服务间通信的 ADR。企业架构 ADR 包含在企业目录中，表示这些是影响所有系统和应用的全局架构决策。企业架构 ADR 的示例是「对系统数据库的所有访问只能来自拥有该数据库的系统」，从而防止跨多个系统共享数据库。

当在维基中存储 ADR 时（我们的建议），先前描述的相同结构适用，每个目录结构代表一个导航着陆页。每个 ADR 将表示为每个导航着陆页（应用、集成或企业）内的单个维基页面。

本节中指示的目录或着陆页名称仅为建议。每家公司可以选择适合其情况的任何名称，只要这些名称在团队间保持一致。

## ADR 作为文档（ADRs as Documentation）

记录软件架构一直是困难的话题。虽然出现了一些架构图示标准（如软件架构师 Simon Brown 的 C4 模型或 The Open Group 的 ArchiMate 标准），但记录软件架构尚无此类标准。这就是 ADR 的用武之地。

架构决策记录可以作为记录软件架构的有效手段。ADR 的背景部分提供了描述需要做出架构决策的系统特定区域的绝佳机会。该部分还提供了描述替代方案的机会。 perhaps 更重要的是，决策部分描述了为何做出特定决策的原因，这是迄今为止最好的架构文档形式。后果部分通过描述特定决策的其他方面（如选择性能而非可扩展性的权衡分析）为架构文档添加最后一块拼图。

## 使用 ADR 制定标准（Using ADRs for Standards）

很少有人喜欢标准。大多数时候，标准似乎更多是为了控制人和他们做事的方式，而不是为了有用。使用 ADR 制定标准可以改变这种不良做法。例如，ADR 的背景部分描述迫使特定标准的情况。ADR 的决策部分不仅可以用来表明标准是什么，更重要的是为什么需要该标准。这是判断特定标准是否应该首先存在的绝佳方式。如果架构师无法论证标准，那或许不是应该制定和执行的好标准。此外，开发者越了解特定标准为何存在，就越可能遵循它（相应地不会挑战它）。

ADR 的后果部分是架构师判断标准是否有效、是否应该制定的另一个好地方。在此部分，架构师必须思考并记录他们制定的特定标准的影响和后果。通过分析后果，架构师可能决定该标准毕竟不应适用。

## 示例（Example）

我们持续进行的「案例研究：Going, Going, Gone」（第 95 页）中存在许多架构决策。事件驱动微服务的使用、竞拍者和拍卖师用户界面的拆分、用于视频捕获的实时传输协议（RTP）的使用、单一 API 层的使用以及发布-订阅消息的使用，只是为该拍卖系统做出的数十个架构决策中的几个。系统中做出的每个架构决策，无论多么明显，都应被记录和论证。
