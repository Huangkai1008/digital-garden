# 第21章 架构图示与演示

新晋架构师经常评论说，他们对这份工作除了促成他们进入架构师角色的技术知识和经验之外的变化之大感到惊讶。特别是，有效沟通对架构师的成功变得至关重要。无论架构师的技术想法多么出色，如果他们无法说服管理者资助、开发者构建，他们的才华就永远不会显现。

架构图示和演示是架构师的两项关键软技能。虽然每个主题都有整本书，但我们会针对每个主题突出一些要点。

这两个主题一起出现是因为它们有一些相似特征：各自形成架构愿景的重要视觉表示，使用不同媒介呈现。然而，**表示一致性（representational consistency）** 是将两者联系在一起的概念。

在视觉描述架构时，创建者通常必须展示架构的不同视图。例如，架构师可能会展示整个架构拓扑的概览，然后深入各个部分以探讨设计细节。然而，如果架构师在未表明其在整体架构中的位置的情况下展示某部分，会使观众困惑。表示一致性是在更改视图之前，始终在图表或演示中展示架构各部分之间关系的实践。

例如，如果架构师想描述 Silicon Sandwiches 解决方案中插件如何相互关联的细节，架构会先展示整个拓扑，然后深入插件结构，向观众展示它们之间的关系；图 21-1 展示了此示例。

```
┌─────────────────────────────────────────────────────────────┐
│  整体架构拓扑                                                  │
│  ┌─────────────────────────────────────────────────────┐     │
│  │  [放大区域] 插件结构细节                              │     │
│  │  Plugin A ←→ Plugin B ←→ Plugin C                    │     │
│  └─────────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────┘
```

**图 21-1. 使用表示一致性在大图中指示上下文**

谨慎使用表示一致性确保观众理解所呈现项目的范围，消除常见的混淆来源。

## 图示（Diagramming）

架构的拓扑始终是架构师和开发者感兴趣的内容，因为它捕捉了结构如何组合在一起，并在团队中形成有价值的共同理解。因此，架构师应将图示技能磨练到极致。

### 工具（Tools）

当前一代的架构图示工具极其强大，架构师应深入学习他们选择的工具。然而，在使用精美工具之前，不要忽视低保真制品，尤其是在设计过程早期。早期构建非常短暂的设计制品可以防止架构师过度依恋所创建的内容，我们称之为**非理性制品依恋（Irrational Artifact Attachment）** 反模式。

**非理性制品依恋（Irrational Artifact Attachment）**……是人对某制品的非理性依恋与生产该制品所花时间之间的比例关系。如果架构师使用 Visio 等工具花两小时创建了精美的图表，他们对该制品有一种与投入时间大致成比例的非理性依恋，这也意味着他们对四小时图表的依恋会比两小时的更强。

敏捷软件开发中使用的低仪式方法的好处之一是围绕创建即时制品，尽可能少的仪式（这有助于解释许多敏捷主义者对索引卡和便利贴的 dedication）。使用低技术工具让团队成员可以扔掉不对的东西，使他们能够进行实验，让制品的真实本质通过修订、协作和讨论 emerge。

架构师对白板手机照片的最爱变体（以及不可避免的「勿擦！」命令）是使用连接到投影仪的平板电脑而非白板。这有几个优势。首先，平板电脑有无限画布，可以容纳团队可能需要的尽可能多的绘图。其次，它允许复制/粘贴「假设」场景，而在白板上完成时会遮蔽原始内容。第三，平板电脑上捕获的图像已经数字化，没有白板手机照片不可避免的反光。

最终，架构师需要在精美工具中创建精美图表，但要确保团队已充分迭代设计，值得花时间捕捉某些东西。

每个平台都有强大的工具可以创建图表。虽然我们不一定要推崇一个胜过另一个（我们很乐意使用 OmniGraffle 制作本书中的所有图表），架构师应至少寻找以下基线功能：

- **图层（Layers）**：绘图工具通常支持图层，架构师应学好。图层允许绘图者将一组项目逻辑上链接在一起，以启用隐藏/显示单个图层。使用图层，架构师可以构建综合图表，但在不需要时隐藏 overwhelming 的细节。使用图层还允许架构师为后续演示增量构建图片（见第 322 页的「增量构建」）。
- **模板/模具（Stencils/templates）**：模具允许架构师构建常见视觉组件的库，通常是其他基本形状的组合。例如，贯穿本书，读者看到了微服务等事物的标准图片，它们作为作者模具中的单个项目存在。为组织内的常见模式和制品构建模具可以在架构图中创建一致性，并允许架构师快速构建新图表。
- **磁吸（Magnets）**：许多绘图工具在形状之间绘制线条时提供辅助。磁吸代表形状上线条自动吸附连接的位置，提供自动对齐和其他视觉细节。有些工具允许架构师添加更多磁吸或创建自己的磁吸，以自定义图中连接的外观。

除了这些具体有用功能外，工具当然还应支持线条、颜色和其他视觉制品，以及以各种格式导出的能力。

### 图示标准：UML、C4 和 ArchiMate（Diagramming Standards）

软件技术图表存在几种正式标准。

**UML（Unified Modeling Language，统一建模语言）** 是统一 1980 年代共存的三种竞争设计哲学的标准。它本应是所有世界中最好的，但像许多委员会设计的东西一样，在强制使用它的组织之外未能产生太大影响。架构师和开发者仍使用 UML 类和序列图来传达结构和工作流，但大多数其他 UML 图表类型已不再使用。

**C4** 是 Simon Brown 开发的图示技术，用于解决 UML 的缺陷并使其方法现代化。C4 中的四个 C 如下：

- **Context（上下文）**：表示系统的整个上下文，包括用户角色和外部依赖。
- **Container（容器）**：架构内的物理（通常也是逻辑）部署边界和容器。此视图形成运维和架构师的良好交汇点。
- **Component（组件）**：系统的组件视图；这与架构师对系统的看法最吻合。
- **Class（类）**：C4 使用与 UML 相同风格的类图，它们很有效，因此无需替换。

如果公司寻求标准化图示技术，C4 提供了良好的替代方案。然而，像所有技术图示工具一样，它无法表达架构可能进行的每种设计。C4 最适合容器和组件关系可能不同的单体架构，不太适合分布式架构，如微服务。

**ArchiMate**（Arch*itecture-Ani*mate 的合并）是一种开源企业架构建模语言，用于支持业务域内和跨业务域的架构描述、分析和可视化。ArchiMate 是 The Open Group 的技术标准，为企业生态系统提供更轻量级的建模语言。ArchiMate 的目标是「尽可能小」，不是覆盖每个边缘情况场景。因此，它已成为许多架构师的热门选择。

### 图示指南（Diagram Guidelines）

无论架构师使用自己的建模语言还是正式语言之一，他们都应在创建图表时建立自己的风格，并应自由借鉴他们认为特别有效的表示。以下是创建技术图表时使用的一些通用指南。

- **标题（Titles）**：确保图表的所有元素都有标题或为观众所熟知。使用旋转和其他效果使标题「粘」在它们关联的事物上，并有效利用空间。
- **线条（Lines）**：线条应足够粗以便看清。如果线条表示信息流，则使用箭头表示单向或双向流量。不同类型的箭头可能暗示不同的语义，但架构师应保持一致。通常，架构图中存在的少数标准之一是实线倾向于表示同步通信，虚线表示异步通信。
- **形状（Shapes）**：虽然描述的所有正式建模语言都有标准形状，但软件开发领域没有普遍的标准形状。因此，每位架构师倾向于制定自己的标准形状集，有时在组织内传播以创建标准语言。我们倾向于使用三维方框表示可部署制品，矩形表示包容关系，除此之外没有特别的图例。
- **标签（Labels）**：架构师应标注图中的每个项目，特别是如果读者有任何歧义可能。
- **颜色（Color）**：架构师通常不够使用颜色——多年来，书籍出于必要以黑白印刷，因此架构师和开发者习惯了单色绘图。虽然我们仍然偏爱单色，但在有助于区分一个制品与另一个时使用颜色。例如，在讨论第 254 页「通信」中的微服务通信策略时，我们使用颜色表示两个不同的微服务参与协调，而不是同一服务的两个实例，如图 21-2 所示。
- **图例（Keys）**：如果形状因任何原因有歧义，在图上包含图例，清楚表明每个形状代表什么。没有什么比导致误解的图表更糟的了，这比没有图表更糟。

## 演示（Presenting）

现代架构师所需的另一项软技能是使用 PowerPoint 和 Keynote 等工具进行有效演示的能力。这些工具是现代组织的通用语言，组织内的人们期望能熟练使用这些工具。不幸的是，与文字处理器和电子表格不同，似乎没有人花太多时间研究如何用好这些工具。

本书合著者之一 Neal 几年前写了一本名为《Presentation Patterns》（Addison-Wesley Professional）的书，关于将软件世界中常见的模式/反模式方法应用于技术演示。

《Presentation Patterns》对创建文档与为某事做演示之间的根本区别做了重要观察——**时间**。在演示中，演示者控制想法展开的速度，而文档的读者控制这一点。因此，架构师可以在他们选择的演示工具中学到的最重要技能之一是如何操纵时间。

### 操纵时间（Manipulating Time）

演示工具提供两种在幻灯片上操纵时间的方式：**过渡（transitions）** 和**动画（animations）**。过渡在幻灯片之间移动，动画允许设计者在幻灯片内创建运动。通常，演示工具每张幻灯片只允许一种过渡，但每个元素有多种动画：构建进入（出现）、构建退出（消失）和动作（如移动、缩放和其他动态行为）。

虽然工具提供各种炫目效果（如掉落铁砧），架构师使用过渡和动画来隐藏幻灯片之间的边界。《Presentation Patterns》中指出的一个常见反模式名为**千篇一律（Cookie-Cutter）**，即想法没有预定的字数，因此设计者不应人为填充内容以使其看起来填满幻灯片。同样，许多想法比单张幻灯片更大。使用溶解等过渡和动画的微妙组合，演示者可以隐藏单个幻灯片边界，将一组幻灯片缝合在一起讲述一个故事。为了表示一个想法的结束，演示者应使用明显不同的过渡（如门或立方体）来提供视觉线索，表明他们正在转向不同主题。

### 增量构建（Incremental Builds）

《Presentation Patterns》一书指出**弹孔尸体（Bullet-Riddled Corpse）** 是企业演示的常见反模式，每张幻灯片本质上是演讲者的笔记，投影给所有人看。大多数读者都有过痛苦的经历：在演示期间观看一张充满文字的幻灯片出现，然后阅读全部内容（因为没有人能在它一出现就忍住不读），然后坐等接下来 10 分钟，演讲者慢慢向观众读要点。难怪这么多企业演示如此乏味！

演示时，演讲者有两个信息渠道：口头和视觉。通过在幻灯片上放置太多文字然后 essentially 说同样的话，演讲者使一个信息渠道过载而使另一个挨饿。此问题的更好解决方案是对幻灯片使用增量构建，根据需要逐步构建（希望是图形的）信息，而不是一次性全部展示。

例如，假设架构师创建演示解释使用特性分支的问题，并想谈论保持分支存活过长的负面后果。考虑图 21-3 所示的图形幻灯片。

**图 21-3. 显示负面反模式的幻灯片的不良版本**

在图 21-3 中，如果演讲者立即展示整张幻灯片，观众可以看到最后发生了不好的事情，但他们必须等待阐述到达那一点。

相反，架构师应使用相同的图像，但在显示幻灯片时遮蔽部分内容（使用无边框白框），并一次展示一部分（通过对覆盖框执行构建退出），如图 21-4 所示。

将动画和过渡与增量构建结合使用，演讲者可以制作更引人入胜、更有趣的演示。

### 信息 deck 与演示（Infodecks Versus Presentations）

有些架构师在 PowerPoint 和 Keynote 等工具中构建幻灯片 deck，但从未真正演示它们。相反，它们像杂志文章一样通过电子邮件发送，每个人以自己的节奏阅读。**信息 deck（Infodecks）** 是并非用于投影而是以图形方式总结信息的幻灯片 deck，essentially 使用演示工具作为桌面出版包。

这两种媒介之间的区别是内容的全面性和过渡与动画的使用。如果有人要像杂志文章一样翻阅 deck，幻灯片作者不需要添加任何时间元素。信息 deck 和演示之间的另一个关键区别是材料数量。因为信息 deck 是独立的，它们包含创建者想要传达的所有信息。在进行演示时，幻灯片（故意）是演示的一半，另一半是站在那里讲话的人！

### 幻灯片是故事的一半（Slides Are Half of the Story）

演讲者常犯的一个错误是将演示的全部内容构建到幻灯片中。然而，如果幻灯片是全面的，演讲者应该省去大家坐着听演示的时间，直接通过电子邮件将 deck 发给每个人！演讲者在可以用更有力的方式表达重要观点时，错误地向幻灯片添加了太多材料。记住，演讲者有两个信息渠道，因此战略性地使用它们可以为信息增添更多冲击力。战略使用**隐形（Invisibility）** 就是一个很好的例子。

### 隐形（Invisibility）

隐形是一种简单的模式，演讲者在演示中插入空白黑幻灯片，以将注意力 solely 重新聚焦到演讲者身上。如果一个人有两个信息渠道（幻灯片和演讲者）并关闭其中一个（幻灯片），它会自动为演讲者增添更多 emphasis。因此，如果演讲者想表达一个观点，插入空白幻灯片——房间里的每个人都会将注意力重新聚焦到演讲者身上，因为他们现在是房间里唯一有趣的东西。

学习演示工具的基础知识和一些使演示更好的技术是架构师技能集的绝佳补充。如果架构师有很好的想法但无法找到有效呈现的方式，他们将永远没有机会实现该愿景。架构需要协作；要获得协作者，架构师必须说服人们签署他们的愿景。现代企业讲坛是演示工具，因此值得学好使用它们。
