# 第7章 架构特性的范围

软件架构世界中一个普遍的公理假设传统上将架构特性的范围置于系统级别。例如，当架构师谈论可扩展性时，他们通常围绕整个系统的可扩展性进行讨论。十年前这是一个安全的假设，当时几乎所有系统都是单体的。随着现代工程技术和它们启用的架构风格（如微服务）的出现，架构特性的范围已经大大缩小。这是公理随着软件开发生态系统持续无情演变而逐渐过时的典型例子。

在编写《构建进化式架构》一书时，作者需要一种技术来衡量特定架构风格的结构可进化性。现有度量都没有提供正确的细节级别。在「结构度量」中，我们讨论了允许架构师分析架构结构方面的各种代码级指标。然而，所有这些指标只揭示关于代码的低级别细节，无法评估代码库外仍影响许多架构特性（尤其是运维特性）的依赖组件（如数据库）。例如，无论架构师在设计高性能或弹性代码库上投入多少努力，如果系统使用的数据库与这些特性不匹配，应用都不会成功。

在评估许多运维架构特性时，架构师必须考虑将影响这些特性的代码库外的依赖组件。因此，架构师需要另一种方法来衡量这些依赖。这导致《构建进化式架构》的作者定义了**架构量子（architecture quantum）**一词。要理解架构量子的定义，我们必须在此预览一个关键指标：共生。

## 耦合与共生

许多代码级耦合指标，如传入和传出耦合（在「结构度量」中描述），在过于细粒度的级别揭示细节，无法进行架构分析。1996 年，Meilir Page-Jones 出版了《每个程序员都应了解的面向对象设计》（Dorset House），其中包含了他命名为**共生（connascence）**的几种新耦合度量，定义如下：

> **共生**：如果改变一个组件需要修改另一个才能维持系统的整体正确性，则两个组件是共生的。

他定义了两类共生：**静态**（可通过静态代码分析发现）和**动态**（涉及运行时行为）。要定义架构量子，我们需要一种衡量组件如何「连接」在一起的方法，这对应于共生概念。例如，如果微服务架构中的两个服务共享某个类（如 address）的相同类定义，我们说它们彼此静态共生——更改共享类需要更改两个服务。

对于动态共生，我们定义两种类型：**同步**和**异步**。两个分布式服务之间的同步调用让调用者等待被调用者的响应。另一方面，异步调用允许事件驱动架构中的即发即弃语义，允许两个不同服务在运维架构特性上有所不同。

## 架构量子与粒度

组件级耦合不是唯一将软件绑定在一起的东西。许多业务概念在语义上将系统的部分绑定在一起，创建功能内聚。要成功设计、分析和演进软件，开发者必须考虑可能破坏的所有耦合点。

许多有科学素养的开发者知道物理学中的量子概念——参与相互作用的任何物理实体的最小量。量子一词源自拉丁语，意为「多大」或「多少」。我们采用这一概念来定义架构量子：

> **架构量子**：具有高功能内聚和同步共生的可独立部署工件

该定义包含几个部分，在此剖析：

**可独立部署**：架构量子包括从架构其他部分独立运行所需的所有必要组件。例如，如果应用使用数据库，它是量子的一部分，因为系统没有它就无法运行。这一要求意味着，几乎所有使用单个数据库部署的遗留系统按定义形成一个量子。然而，在微服务架构风格中，每个服务包括自己的数据库（微服务中驱动哲学的限界上下文的一部分，在第 17 章详细描述），在该架构中创建多个量子。

**高功能内聚**：组件设计中的内聚是指包含的代码在目的上统一的程度。例如，具有全部与 Customer 实体相关的属性和方法的 Customer 组件表现出高内聚；而具有随机杂项方法集合的 Utility 组件则不会。高功能内聚意味着架构量子做有意义的事情。这种区别在具有单个数据库的传统单体应用中意义不大。然而，在微服务架构中，开发者通常将每个服务设计为匹配单个工作流（限界上下文，如「领域驱动设计的限界上下文」所述），从而表现出高功能内聚。

**同步共生**：同步共生意味着应用上下文中或形成此架构量子的分布式服务之间的同步调用。例如，如果微服务架构中的一个服务同步调用另一个，每个服务在运维架构特性上不能表现出极端差异。如果调用者比被调用者更具可扩展性，将发生超时和其他可靠性关注。因此，同步调用在调用期间创建动态共生——如果一个在等待另一个，它们的运维架构特性在调用期间必须相同。

## 领域驱动设计的限界上下文

Eric Evans 的《领域驱动设计》（Addison-Wesley Professional）深深影响了现代架构思维。**领域驱动设计（DDD）**是一种允许对复杂问题域进行有组织分解的建模技术。DDD 定义了**限界上下文（bounded context）**，其中与领域相关的一切在内部可见，但对其他限界上下文不透明。在 DDD 之前，开发者寻求组织内跨通用实体的整体复用。然而，创建通用共享工件会导致一系列问题，如耦合、更困难的协调和增加的复杂性。限界上下文概念认识到每个实体在局部上下文中工作得最好。因此，与其在整个组织中创建统一的 Customer 类，每个问题域可以创建自己的，并在集成点调和差异。

## 案例研究：Going, Going, Gone

架构量子概念为架构特性提供了新的范围。在现代系统中，架构师在量子级别而非系统级别定义架构特性。通过以更窄的范围看待重要运维关注，架构师可以及早识别架构挑战，导致混合架构。为说明架构量子度量提供的范围，考虑另一个架构 kata：Going, Going, Gone。

**描述**：一家拍卖公司希望将其拍卖在线扩展到全国规模。客户选择参与的拍卖，等待拍卖开始，然后出价，就像与拍卖师在房间里一样。

**用户**：每个拍卖最多数百名参与者，可能多达数千名参与者，以及尽可能多的同时拍卖。

**需求**：
- 拍卖必须尽可能实时
- 出价者用信用卡注册；如果出价者获胜，系统自动扣款
- 参与者必须通过声誉指数跟踪
- 出价者可以看到拍卖的实时视频流和所有出价的发生
- 在线和现场出价必须按放置顺序接收

**附加上下文**：
- 拍卖公司通过合并较小竞争对手积极扩张
- 预算不受约束。这是战略方向
- 公司刚结束一场涉嫌欺诈的诉讼

使用架构量子度量，架构师在量子级别确定架构特性的范围。例如，在 Going, Going, Gone 中，架构师会注意到该架构的不同部分需要不同的特性：流式出价、在线出价者和拍卖师是三个明显选择。架构师使用架构量子度量作为思考部署、耦合、数据应驻留何处以及架构内通信风格的方式。在此 kata 中，架构师可以分析每个架构量子的不同架构特性，从而在过程中更早地导致混合架构设计。

因此，对于 Going, Going, Gone，我们确定了以下量子和相应的架构特性：

| 量子 | 架构特性 |
|------|----------|
| **出价者反馈**（出价流和视频流） | 可用性、可扩展性、性能 |
| **拍卖师**（现场拍卖师） | 可用性、可靠性、可扩展性、弹性、性能、安全 |
| **出价者**（在线出价者和出价） | 可靠性、可用性、可扩展性、弹性 |
