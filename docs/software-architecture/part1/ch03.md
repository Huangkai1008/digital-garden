# 第3章 模块化

首先，我们想厘清围绕模块化讨论中常用和滥用的一些术语，并为全书提供定义。

> 关于软件架构的 95% 的篇幅都在颂扬「模块化」的好处，而关于如何实现它却几乎只字未提。
>
> —Glenford J. Myers (1978)

不同平台为代码提供不同的复用机制，但都支持某种将相关代码分组到模块中的方式。虽然这一概念在软件架构中是普遍的，但事实证明它难以定义。随便在网上搜索会得到数十种定义，没有一致性（有些还相互矛盾）。正如 Myers 的引用所示，这不是新问题。然而，由于不存在公认的定义，我们必须加入讨论并为了全书的一致性提供我们自己的定义。

理解模块化及其在所选开发平台中的多种体现对架构师至关重要。我们用于分析架构的许多工具（如指标、适应度函数和可视化）都依赖这些模块化概念。模块化是一种组织原则。如果架构师在设计系统时不注意各部分如何连接，最终会创建一个呈现 myriad 困难的系统。用物理学类比，软件系统模拟复杂系统，而复杂系统倾向于熵（或无序）。必须向物理系统添加能量才能保持秩序。软件系统也是如此：架构师必须不断消耗能量以确保良好的结构健全性，这不会偶然发生。

保持良好模块化体现了我们对隐式架构特性的定义：几乎没有项目会提出要求架构师确保良好模块区分和通信的需求，但可持续的代码库需要秩序和一致性。

## 定义

词典将模块定义为「用于构建更复杂结构的一组标准化部件或独立单元中的每一个」。我们用**模块化（modularity）**来描述相关代码的逻辑分组，可以是面向对象语言中的一组类，或结构化或函数式语言中的函数。大多数语言提供模块化机制（Java 的 `package`、.NET 的 `namespace` 等）。开发者通常使用模块将相关代码分组在一起。例如，Java 中的 `com.mycompany.customer` 包应包含与客户相关的内容。

语言现在具有各种各样的打包机制，使开发者在它们之间选择的琐事变得困难。例如，在许多现代语言中，开发者可以在函数/方法、类或包/命名空间中定义行为，每种都有不同的可见性和作用域规则。其他语言通过添加元对象协议等编程构造进一步复杂化，为开发者提供更多扩展机制。

架构师必须了解开发者如何打包事物，因为这对架构有重要影响。例如，如果几个包紧密耦合在一起，复用其中一个进行相关工作会变得更加困难。

### 类之前的模块复用

早于面向对象语言的开发者可能会困惑为什么通常存在这么多不同的分离方案。大部分原因与向后兼容性有关，不是代码的兼容性，而是开发者思考方式的兼容性。1968 年 3 月，Edsger Dijkstra 在 ACM 通讯上发表了一封题为《Go To 语句被认为有害》的信。他贬低了当时编程语言中常见的 GOTO 语句的普遍使用，该语句允许在代码内进行非线性跳跃，使推理和调试变得困难。

这篇论文帮助迎来了结构化编程语言时代，以 Pascal 和 C 为代表，鼓励更深入地思考事物如何组合在一起。开发者很快意识到，大多数语言没有好的方式将类似事物逻辑地分组在一起。因此，模块语言的短暂时代诞生了，如 Modula（Pascal 创造者 Niklaus Wirth 的下一种语言）和 Ada。这些语言具有模块的编程构造，就像我们今天对包或命名空间的思考一样（但没有类）。

模块编程时代很短暂。面向对象语言变得流行，因为它们提供了封装和复用代码的新方式。尽管如此，语言设计者认识到模块的实用性，以包、命名空间等形式保留了它们。语言中存在许多奇怪的兼容性特性来支持这些不同范式。例如，Java 支持模块（通过包和包级初始化使用静态初始化器）、面向对象和函数式范式，每种编程风格都有自己的作用域规则和怪癖。

对于架构讨论，我们使用模块化作为通用术语来表示代码的相关分组：类、函数或任何其他分组。这并不意味着物理分离，仅仅是逻辑分离；这种区别有时很重要。例如，从便利性角度将大量类集中在一个单体应用中可能说得通。然而，当需要重构架构时，松散分区鼓励的耦合成为拆分单体的障碍。因此，将模块化作为与特定平台强制或暗示的物理分离分开的概念来讨论是有用的。

值得注意的是命名空间的一般概念，与 .NET 平台中的技术实现分开。开发者通常需要软件资产的精确、完全限定名称，以将不同的软件资产（组件、类等）彼此分开。人们每天使用的最明显例子是互联网：与 IP 地址绑定的唯一全局标识符。大多数语言都有某种模块化机制，兼作组织事物的命名空间：变量、函数和/或方法。有时模块结构在物理上反映。例如，Java 要求其包结构必须反映物理类文件的目录结构。

## 度量模块化

鉴于模块化对架构师的重要性，他们需要工具来理解它。幸运的是，研究人员创建了各种与语言无关的指标来帮助架构师理解模块化。我们关注三个关键概念：内聚、耦合和共生。

### 内聚

**内聚（cohesion）**是指模块的各个部分应在多大程度上包含在同一模块内。换句话说，它是各部分彼此相关程度的度量。理想情况下，内聚的模块是所有部分应打包在一起的模块，因为将它们拆成更小的部分需要通过模块间的调用来耦合各部分才能获得有用的结果。

> 试图拆分内聚的模块只会导致耦合增加和可读性降低。
>
> —Larry Constantine

计算机科学家定义了一系列内聚度量，从最好到最差列出如下：

| 类型 | 描述 |
|------|------|
| **功能内聚** | 模块的每个部分都与其他部分相关，模块包含功能所需的一切 |
| **顺序内聚** | 两个模块交互，其中一个输出成为另一个输入的数据 |
| **通信内聚** | 两个模块形成通信链，每个对信息进行操作和/或贡献某些输出 |
| **过程内聚** | 两个模块必须以特定顺序执行代码 |
| **时间内聚** | 模块基于时间依赖关系相关。例如，许多系统有一系列看似无关的事物必须在系统启动时初始化 |
| **逻辑内聚** | 模块内的数据逻辑相关但功能上不相关。例如，考虑一个从文本、序列化对象或流转换信息的模块 |
| **偶然内聚** | 模块中的元素除了在同一源文件中之外没有其他关联；这代表最负面的内聚形式 |

尽管列出了七种变体，内聚是比耦合更不精确的指标。通常，特定模块的内聚程度由特定架构师自行决定。例如，考虑这个模块定义：

**客户维护**
- 添加客户
- 更新客户
- 获取客户
- 通知客户
- 获取客户订单
- 取消客户订单

最后两个条目应该驻留在此模块中，还是开发者应创建两个单独的模块？如往常一样，看情况。这些问题是软件架构师工作核心的权衡分析类型。

Chidamber 和 Kemerer 的**方法缺乏内聚度（LCOM）**指标衡量模块（通常是组件）的结构内聚。LCOM 的简化定义：

> **LCOM**：未通过共享字段共享的方法集合之和

考虑一个具有私有字段 `a` 和 `b` 的类。许多方法只访问 `a`，许多其他方法只访问 `b`。未通过共享字段（a 和 b）共享的方法集合之和很高；因此，该类报告高 LCOM 分数，表示方法缺乏内聚度得分高。

LCOM 指标对分析代码库以从一种架构风格迁移到另一种的架构师有用。迁移架构时的常见头痛之一是共享工具类。使用 LCOM 指标可以帮助架构师找到偶然耦合的类，这些类本不应是单个类。

### 耦合

幸运的是，我们有更好的工具来分析代码库中的耦合，部分基于图论：因为方法调用和返回形成调用图，基于数学的分析成为可能。1979 年，Edward Yourdon 和 Larry Constantine 出版了《结构化设计：计算机程序与系统设计学科基础》（Prentice-Hall），定义了许多核心概念，包括**传入耦合（afferent coupling）**和**传出耦合（efferent coupling）**指标。传入耦合衡量传入代码工件（组件、类、函数等）的连接数。传出耦合衡量传出到其他代码工件的连接数。几乎所有平台都有允许架构师分析代码耦合特性以协助重构、迁移或理解代码库的工具。

### 抽象度、不稳定性和与主序列的距离

虽然组件耦合的原始值对架构师有价值，但几个派生指标允许更深入的评估。这些指标由 Robert Martin 为 C++ 书籍创建，但广泛适用于其他面向对象语言。

**抽象度（abstractness）**是抽象工件（抽象类、接口等）与具体工件（实现）的比率。它表示抽象与实现的度量。抽象度公式：

$$A = \frac{\sum m_a}{\sum m_c}$$

其中 $m_a$ 表示模块内的抽象元素（接口或抽象类），$m_c$ 表示具体元素（非抽象类）。

**不稳定性（instability）**定义为传出耦合与传入和传出耦合之和的比率：

$$I = \frac{C_e}{C_e + C_a}$$

其中 $C_e$ 表示传出耦合，$C_a$ 表示传入耦合。

**与主序列的距离（distance from the main sequence）**是架构师拥有的少数整体指标之一，是基于不稳定性和抽象度的派生指标：

$$D = |A + I - 1|$$

注意抽象度和不稳定性都是分数，其结果将始终落在 0 和 1 之间。因此，在绘图时，我们得到图 3-2 所示的图。距离指标想象了抽象度与不稳定性之间的理想关系；落在这条理想化线附近的类表现出这两种竞争关注点的健康混合。

落在右上角太远的类进入架构师所谓的**无用区（zone of uselessness）**：过于抽象的代码变得难以使用。相反，落在左下角的代码进入**痛苦区（zone of pain）**：实现过多、抽象不足的代码变得脆弱且难以维护。

### 共生

1996 年，Meilir Page-Jones 出版了《每个程序员都应了解的面向对象设计》（Dorset House），完善了传入和传出耦合指标，并用他命名为**共生（connascence）**的概念将其重新塑造为面向对象语言。他的定义：

> 如果改变一个组件需要修改另一个才能维持系统的整体正确性，则两个组件是共生的。
>
> —Meilir Page-Jones

他定义了两类共生：静态和动态。

**静态共生**是指源代码级别的耦合（与执行时耦合相对）：

| 类型 | 描述 |
|------|------|
| **名称共生 (CoN)** | 多个组件必须就实体的名称达成一致 |
| **类型共生 (CoT)** | 多个组件必须就实体的类型达成一致 |
| **意义共生 (CoM) / 约定共生 (CoC)** | 多个组件必须就特定值的意义达成一致 |
| **位置共生 (CoP)** | 多个组件必须就值的顺序达成一致 |
| **算法共生 (CoA)** | 多个组件必须就特定算法达成一致 |

**动态共生**分析运行时调用：

| 类型 | 描述 |
|------|------|
| **执行共生 (CoE)** | 多个组件的执行顺序很重要 |
| **时间共生 (CoT)** | 多个组件的执行时间很重要 |
| **值共生 (CoV)** | 多个值相互关联且必须一起改变 |
| **身份共生 (CoI)** | 多个组件必须引用同一实体 |

**共生属性**：

- **强度**：架构师通过开发者重构该类型耦合的难易程度确定共生强度
- **局部性**：共生的局部性衡量模块在代码库中彼此接近的程度
- **程度**：共生的程度与其影响的大小有关——它影响少数类还是许多类？

Page-Jones 提供了使用共生改善系统模块化的三条指南：

1. 通过将系统拆分为封装元素来最小化整体共生
2. 最小化跨越封装边界的任何剩余共生
3. 最大化封装边界内的共生

Jim Weirich 提供了两条建议：

- **程度规则**：将强形式的共生转换为弱形式的共生
- **局部性规则**：随着软件元素之间的距离增加，使用较弱形式的共生

### 统一耦合与共生指标

耦合和共生代表不同时代、不同目标的度量，但从架构师的角度看，这两种视图重叠。Page-Jones 识别的静态共生代表传入或传出耦合的程度。结构化编程只关心进或出，而共生关心事物如何耦合在一起。

## 从模块到组件

我们在全书中使用**模块（module）**作为相关代码捆绑的通用名称。然而，大多数平台支持某种形式的**组件（component）**，这是软件架构师的关键构建块之一。逻辑或物理分离的概念及相应分析自计算机科学最早的日子就存在。然而，尽管有大量关于组件和分离的写作和思考，开发者和架构师仍在努力实现良好的结果。

我们将在第 8 章讨论从问题域派生组件，但我们必须先讨论软件架构的另一个基本方面：架构特性及其范围。
