# 第5章 识别架构特性

识别驱动架构特性是创建架构或确定现有架构有效性的第一步。为给定问题或应用识别正确的架构特性（「-ilities」）要求架构师不仅理解领域问题，还要与问题域利益相关者协作以确定从领域角度真正重要的内容。

架构师至少通过三种方式发现架构特性：从领域关注提取、从需求提取以及隐式领域知识。我们之前讨论过隐式特性，这里涵盖另外两种。

## 从领域关注提取架构特性

架构师必须能够将领域关注转化为识别正确的架构特性。例如，可扩展性是最重要的关注，还是容错、安全或性能？也许系统需要所有四种特性组合。理解关键领域目标和领域情况使架构师能够将这些领域关注转化为「-ilities」，然后形成正确且合理的架构决策的基础。

与领域利益相关者协作定义驱动架构特性时的一个技巧是努力将最终列表保持尽可能短。架构中的常见反模式是试图设计通用架构——支持所有架构特性的架构。架构支持的每个架构特性都会使整体系统设计复杂化；支持太多架构特性会在架构师和开发者甚至开始解决问题域（编写软件的原始动机）之前导致越来越大的复杂性。不要纠结于特性数量，而要关注保持设计简单的动机。

::: details 案例研究：瓦萨号
过度指定架构特性并最终扼杀项目的原始故事必须是瓦萨号。它是 1626 年至 1628 年间由一位想要创造有史以来最宏伟船只的国王建造的瑞典战舰。在那之前，船只要么是部队运输船，要么是炮舰——瓦萨号两者兼有！大多数船只有一层甲板——瓦萨号有两层！所有大炮的尺寸是类似船只的两倍。尽管专业造船师有些担忧（他们最终无法对阿道夫斯国王说不），造船师完成了建造。在庆祝中，船驶入港口并从一侧鸣炮致敬。不幸的是，由于船头重脚轻，它倾覆沉入瑞典海湾底部。20 世纪初，打捞者救起了这艘船，它现在存放在斯德哥尔摩的博物馆中。
:::

虽然大多数架构师和领域利益相关者希望优先考虑应用或系统必须支持的最终架构特性列表，但在大多数情况下这是徒劳的，不仅会浪费时间，还会在关键利益相关者中产生大量不必要的挫折和分歧。利益相关者很少会同意每个特性的优先级。更好的方法是让领域利益相关者从最终列表中选择最重要的三个特性（任意顺序）。这不仅更容易达成共识，还能促进关于什么最重要的讨论，并帮助架构师在做出重要架构决策时分析权衡。

大多数架构特性来自倾听关键领域利益相关者并与他们协作，以从领域角度确定什么重要。虽然这看似直接的活动，但问题是架构师和领域利益相关者说不同的语言。架构师谈论可扩展性、互操作性、容错、可学习性和可用性。领域利益相关者谈论并购、用户满意度、上市时间和竞争优势。发生的是「翻译丢失」问题，架构师和领域利益相关者彼此不理解。架构师不知道如何创建支持用户满意度的架构，领域利益相关者不理解为什么应用中有如此多的关注和关于可用性、互操作性、可学习性和容错的讨论。幸运的是，通常存在从领域关注到架构特性的翻译。表 5-1 显示了一些更常见的领域关注及对应的「-ilities」。

| 领域关注 | 架构特性 |
|----------|----------|
| 并购 | 互操作性、可扩展性、适应性、可扩展性 |
| 上市时间 | 敏捷性、可测试性、可部署性 |
| 用户满意度 | 性能、可用性、容错、可测试性、可部署性、敏捷性、安全 |
| 竞争优势 | 敏捷性、可测试性、可部署性、可扩展性、可用性、容错 |
| 时间和预算 | 简单性、可行性 |

需要注意的重要一点是，敏捷性不等于上市时间。而是敏捷性 + 可测试性 + 可部署性。这是许多架构师在翻译领域关注时陷入的陷阱。只关注其中一种成分就像忘记在蛋糕面糊中加面粉。例如，领域利益相关者可能会说：「由于监管要求，我们绝对必须在截止时间前完成日终基金定价。」无效的架构师可能只关注性能，因为那似乎是该领域关注的主要焦点。然而，该架构师会因多种原因失败。第一，如果系统在需要时不可用，系统多快都无关紧要。第二，随着领域增长和更多基金创建，系统还必须能够扩展以按时完成日终处理。第三，系统不仅必须可用，还必须可靠，以便在计算日终基金价格时不会崩溃。第四，如果日终基金定价约 85% 完成时系统崩溃怎么办？它必须能够恢复并从定价停止的地方重新开始。最后，系统可能很快，但基金价格计算正确吗？因此，除了性能之外，架构师还必须同样关注可用性、可扩展性、可靠性、可恢复性和可审计性。

## 从需求提取架构特性

一些架构特性来自需求文档中的显式陈述。例如，显式预期的用户数量和规模通常出现在领域或领域关注中。其他来自架构师的固有领域知识——领域知识对架构师始终有益的众多原因之一。例如，假设架构师设计一个处理大学生选课的应用。为简化计算，假设学校有 1,000 名学生和 10 小时选课时间。架构师应该设计一个假设一致规模的系统，隐含假设选课过程中的学生会在时间上均匀分布吗？或者，基于对大学生习惯和倾向的了解，架构师应该设计一个能够处理所有 1,000 名学生在最后 10 分钟尝试选课的系统吗？任何了解学生 stereotypically 拖延程度的人都知道这个问题的答案！像这样的细节很少会出现在需求文档中，但它们确实会影响设计决策。

## 案例研究：硅谷三明治

为说明几个概念，我们使用一个架构 kata（见「架构 Kata 的起源」了解该概念的来源）。为展示架构师如何从需求派生架构特性，我们介绍硅谷三明治 kata。

**描述**：一家全国三明治店希望启用在线订购（除了现有的电话订购服务）。

**用户**：数千，也许有一天数百万

**需求**：
- 用户下单后，将获得取三明治的时间和到店的路线（必须与多个包含交通信息的外部地图服务集成）
- 如果店铺提供配送服务，派遣司机将三明治送给用户
- 移动设备可访问性
- 提供全国每日促销/特价
- 提供本地每日促销/特价
- 接受在线、现场或配送时付款

**附加上下文**：
- 三明治店是特许经营，每家店主不同
- 母公司有近期海外扩张计划
- 公司目标是雇用廉价劳动力以最大化利润

### 显式特性

显式架构特性作为必要设计的一部分出现在需求规格说明中。例如，购物网站可能希望支持特定数量的并发用户，领域分析师在需求中指定。架构师应考虑需求的每个部分，看它是否有助于架构特性。但首先，架构师应考虑关于预期指标的领域级预测，如 kata 的用户部分所示。

首先应引起架构师注意的细节之一是用户数量：目前数千，也许有一天数百万。因此，**可扩展性**——在不严重降低性能的情况下处理大量并发用户的能力——是顶级架构特性之一。注意问题陈述没有明确要求可扩展性，而是将需求表达为预期用户数量。架构师必须经常将领域语言解码为工程等价物。

然而，我们可能还需要**弹性**——处理请求突发的能力。这两种特性经常被混在一起，但它们有不同的约束。可扩展性衡量并发用户的性能。另一方面，弹性衡量流量突发。有些系统可扩展但缺乏弹性。例如，考虑酒店预订系统。在没有特殊销售或活动的情况下，用户数量可能是一致的。相反，考虑音乐会门票预订系统。当新票开售时，狂热粉丝会涌入网站，需要高度弹性。通常，弹性系统也需要可扩展性：处理突发和大量并发用户的能力。

### 隐式特性

许多架构特性未在需求文档中指定，但它们构成了设计的重要方面。系统可能希望支持的一个隐式架构特性是**可用性**：确保用户可以访问三明治网站。与可用性密切相关的是**可靠性**：确保网站在交互期间保持运行——没有人想从不断断开连接、迫使他们重新登录的网站购买。

**安全**作为隐式特性出现在每个系统中：没有人想创建不安全的软件。然而，它可能根据关键性进行优先级排序，这说明了我们定义的相互关联本质。如果安全影响设计的某些结构方面并对应用至关重要或重要，架构师将其视为架构特性。

对于硅谷三明治，架构师可能假设付款应由第三方处理。因此，只要开发者遵循一般安全卫生（不以明文传递信用卡号、不存储过多信息等），架构师不需要任何特殊的结构设计来适应安全；应用中的良好设计就足够了。

### 设计与架构及权衡

在硅谷三明治 kata 中，架构师可能会将可定制性识别为系统的一部分，但问题变成：架构还是设计？架构意味着某种结构组件，而设计驻留在架构内。在硅谷三明治的可定制性案例中，架构师可以选择微内核等架构风格并构建对定制的结构支持。然而，如果架构师由于竞争关注选择了另一种风格，开发者可以使用模板方法设计模式实现定制，该模式允许父类定义可在子类中覆盖的工作流。哪种设计更好？像所有架构一样，这取决于许多因素。

架构师必须记住：架构中没有最佳设计，只有最不坏的权衡集合。架构师还必须优先考虑这些架构特性，努力找到所需的最简单集合。一旦团队完成识别架构特性的第一轮，一个有用的练习是尝试确定最不重要的一个——如果必须消除一个，会是哪个？通常，架构师更可能削减显式架构特性，因为许多隐式特性支持普遍成功。
