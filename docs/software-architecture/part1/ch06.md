# 第6章 度量与治理架构特性

架构师必须处理软件项目各个方面极其广泛的架构特性。性能、弹性和可扩展性等运维方面与模块化和可部署性等结构关注交织在一起。本章专注于具体定义一些更常见的架构特性，并为它们构建治理机制。

## 度量架构特性

组织围绕架构特性定义存在几个常见问题：

**它们不是物理学**：许多常用架构特性的含义模糊。例如，架构师如何为敏捷性或可部署性设计？行业对常用术语有 wildly 不同的视角，有时由合理的不同上下文驱动，有时是偶然的。

**定义差异巨大**：即使在同一组织内，不同部门可能对性能等关键特性的定义存在分歧。在开发者、架构和运维能够就共同定义达成统一之前，很难进行适当的对话。

**过于复合**：许多理想的架构特性由更小规模的许多其他特性组成。例如，开发者可以将敏捷性分解为模块化、可部署性和可测试性等特性。

架构特性的客观定义解决了所有三个问题：通过在组织范围内就架构特性的具体定义达成一致，团队围绕架构创建了通用语言。此外，通过鼓励客观定义，团队可以解包复合特性以发现可以客观定义的可度量特性。

## 运维度量

许多架构特性具有明显的直接度量，如性能或可扩展性。然而，即使这些也提供许多细微的解释，取决于团队的目标。例如，也许团队衡量某些请求的平均响应时间，这是运维架构特性度量的好例子。但如果团队只衡量平均值，如果某些边界条件导致 1% 的请求比其他请求慢 10 倍怎么办？如果网站有足够的流量，异常值可能甚至不会出现。因此，团队可能还想衡量最大响应时间以捕获异常值。

### 性能的多种风味

我们描述的许多架构特性有多种细微的定义。性能是一个很好的例子。许多项目关注一般性能：例如，Web 应用的请求和响应周期需要多长时间。然而，架构师和 DevOps 工程师在建立**性能预算（performance budgets）**方面做了大量工作：应用特定部分的特定预算。例如，许多组织研究了用户行为并确定首次页面渲染（网页在浏览器或移动设备上可见进展的第一个迹象）的最佳时间是 500 毫秒——半秒；大多数应用在这一指标上落在两位数范围。但对于试图捕获尽可能多用户的现代网站，这是一个重要的跟踪指标，其背后的组织建立了极其细微的度量。

其中一些指标对应用的设计有额外影响。许多前瞻性组织为页面下载设置 K 权重预算：特定页面上允许的库和框架的最大字节数。这种结构背后的理由源于物理约束：一次只能有这么多字节通过网络传输，尤其是高延迟地区的移动设备。

## 结构度量

一些客观度量不如性能那么明显。内部结构特性（如定义良好的模块化）呢？不幸的是，内部代码质量的全面指标尚不存在。然而，一些指标和常用工具确实允许架构师解决代码结构的一些关键方面，尽管沿狭窄的维度。

代码的一个明显可度量方面是**复杂度**，由**圈复杂度（cyclomatic complexity, CC）**指标定义。圈复杂度是代码级指标，旨在为代码的复杂度提供客观度量，在函数/方法、类或应用级别，由 Thomas McCabe, Sr. 于 1976 年开发。

它通过将图论应用于代码，特别是导致不同执行路径的决策点来计算。例如，如果函数没有决策语句（如 if 语句），则 CC = 1。如果函数有单个条件，则 CC = 2，因为存在两条可能的执行路径。

计算单个函数或方法的 CC 的公式是 $CC = E - N + 2$，其中 N 表示节点（代码行），E 表示边（可能的决策）。

```java
public void decision(int c1, int c2) {
    if (c1 < 100)
        return 0;
    else if (c1 + c2 > 500)
        return 1;
    else
        return -1;
}
```

示例代码的圈复杂度为 3。架构师和开发者普遍同意过于复杂的代码代表代码坏味道；它损害代码库的几乎所有理想特性：模块化、可测试性、可部署性等。然而，如果团队不关注逐渐增长的复杂度，该复杂度将主导代码库。

行业对 CC 的阈值通常建议 10 以下是可接受的，除非有其他考虑（如复杂领域）。我们认为该阈值非常高，更希望代码落在 5 以下，表示内聚、良好分解的代码。

## 过程度量

一些架构特性与软件开发过程相交。例如，敏捷性经常作为理想特性出现。然而，它是一个复合架构特性，架构师可以将其分解为可测试性、可部署性等特性。

可测试性可以通过几乎所有平台的代码覆盖率工具来度量，评估测试的完整性。与所有软件检查一样，它无法替代思考和意图。例如，代码库可以有 100% 的代码覆盖率，但断言很差，实际上无法提供对代码正确性的信心。然而，可测试性显然是可客观度量的特性。同样，团队可以通过各种指标衡量可部署性：成功与失败部署的百分比、部署所需时间、部署引起的问题/bug 等。

## 治理与适应度函数

一旦架构师建立了架构特性并进行了优先级排序，如何确保开发者会尊重这些优先级？模块化是架构中重要但不紧急的方面的很好的例子；在许多软件项目上，紧迫性占主导，但架构师仍然需要治理机制。

### 治理架构特性

**治理（governance）**源自希腊词 kubernan（掌舵），是架构师角色的重要职责。顾名思义，架构治理的范畴涵盖架构师（包括企业架构师等角色）希望施加影响的软件开发过程的任何方面。例如，确保组织内的软件质量属于架构治理的范畴，因为它属于架构的范畴，疏忽可能导致灾难性的质量问题。

幸运的是，越来越复杂的解决方案存在以减轻架构师的这一问题，这是软件开发生态系统内能力渐进增长的好例子。由极限编程推动的软件项目自动化驱动创建了持续集成，进而导致运维的进一步自动化，我们现在称之为 DevOps，继续延伸到架构治理。《构建进化式架构》（O'Reilly）描述了一系列称为**适应度函数（fitness functions）**的技术，用于自动化架构治理的许多方面。

### 适应度函数

《构建进化式架构》中的「进化式」一词更多来自进化计算而非生物学。作者之一 Rebecca Parsons 博士曾在进化计算领域工作，包括遗传算法等工具。遗传算法执行并产生答案，然后通过进化计算世界中定义的知名技术进行变异。如果开发者尝试设计遗传算法以产生有益结果，他们通常希望指导算法，提供指示结果质量接近目标的客观度量。该指导机制称为适应度函数：用于评估输出与实现目标的接近程度的目标函数。

进化式架构实践借用这一概念创建架构适应度函数：

> **架构适应度函数**：提供对某些架构特性或架构特性组合的客观完整性评估的任何机制

适应度函数不是架构师要下载的新框架，而是对许多现有工具的新视角。注意定义中的短语「任何机制」——架构特性的验证技术与特性本身一样多样。适应度函数与许多现有验证机制重叠，取决于使用方式：作为指标、监控器、单元测试库、混沌工程等。

**循环依赖**：模块化是大多数架构师关心的隐式架构特性，因为维护不善的模块化损害代码库的结构；因此，架构师应将保持良好模块化作为高优先级。然而，许多平台上的力量与架构师的良好意图作对。例如，在任何流行的 Java 或 .NET 开发环境中编码时，一旦开发者引用尚未导入的类，IDE 会 helpful 地呈现一个对话框，询问开发者是否要自动导入引用。这发生得如此频繁，以至于大多数程序员养成了像反射动作一样拍掉自动导入对话框的习惯。然而，任意在彼此之间导入类或组件对模块化来说是灾难。解决方案是编写适应度函数来查找循环。

**与主序列的距离**：架构师也可以使用适应度函数验证与主序列的距离，使用 JDepend 建立可接受值的阈值，如果类落在范围外则测试失败。

架构师必须确保开发者在将适应度函数强加给他们之前理解其目的。适应度函数工具的复杂性在过去几年有所增加，包括一些专用工具。其中一个工具是 **ArchUnit**，一个受 JUnit 生态系统启发并使用其多个部分的 Java 测试框架。ArchUnit 提供各种预定义的治理规则，以单元测试的形式编纂，允许架构师编写解决模块化的特定测试。

.NET 领域的类似工具 **NetArchTest** 允许该平台的类似测试。另一个适应度函数的例子是 Netflix 的 Chaos Monkey 和随附的 Simian Army。特别是，Conformity、Security 和 Janitor Monkeys 体现了这种方法。

几年前，Atul Gawande 的有影响力的著作《清单宣言》（Picador）描述了飞行员和外科医生等职业如何使用清单（有时是法律规定的）。不是因为这些专业人士不了解自己的工作或健忘。相反，当专业人士一遍又一遍地做高度详细的工作时，细节很容易溜走；简洁的清单形成有效的提醒。这是对适应度函数的正确视角——与其说是重量级治理机制，不如说适应度函数为架构师提供了一种表达重要架构原则并自动验证它们的机制。开发者知道他们不应该发布不安全的代码，但该优先级与忙碌开发者的数十或数百个其他优先级竞争。像 Security Monkey 这样的工具，以及适应度函数一般，允许架构师将重要的治理检查编纂到架构的基底中。
