# 第2章 B-Tree 基础

上一章中，我们将存储结构分为可变（mutable）与不可变（immutable）两类，并将不可变性视为影响其设计与实现的核心概念之一。多数可变存储结构采用就地更新（in-place update）机制：在插入、删除或更新时，数据记录直接在目标文件中的位置被修改。

存储引擎通常允许多个版本的同一条数据记录同时存在，例如在使用多版本并发控制（Multiversion Concurrency Control, MVCC）或槽式页（slotted page）组织时。为简化起见，我们暂时假定每个键只关联一条数据记录，且具有唯一位置。

B-Tree 是最流行的存储结构之一。许多开源数据库系统基于 B-Tree，多年来证明其能满足大多数使用场景。

B-Tree 并非新发明：由 Rudolph Bayer 和 Edward M. McCreight 于 1971 年提出，并逐渐普及。到 1979 年已有多种 B-Tree 变体，Douglas Comer 对其进行了收集与系统化。

在深入 B-Tree 之前，我们先讨论为何要考虑传统搜索树（如二叉搜索树、2-3 树、AVL 树）的替代方案，并回顾二叉搜索树的概念。

## 二叉搜索树

二叉搜索树（Binary Search Tree, BST）是一种有序的内存数据结构，用于高效的键值查找。BST 由多个节点组成，每个节点包含一个键、与该键关联的值，以及两个子指针（故称二叉）。

BST 从单一节点开始，称为根节点（root node）。树中只能有一个根。图 2-1 展示了一个二叉搜索树的例子。

::: tip 图 2-1：二叉搜索树
```
         [5]
        /   \
      [3]   [7]
      / \   / \
    [1] [4][6] [9]
```
:::

如图 2-2 所示，每个节点将搜索空间划分为左右子树：节点键大于其左子树中任意键，小于其右子树中任意键。

::: tip 图 2-2：二叉树节点不变量
```
     [K]
    /   \
  < K   >= K
```
:::

从根沿左指针向下到叶层（无子节点的层）可定位树中最小键及其关联值；沿右指针可定位最大键及其关联值。值可存储在树中所有节点。搜索从根开始，若在较高层找到目标键，可在到达底层前结束。

## 树平衡

插入操作不遵循特定模式，元素插入可能导致树不平衡（即某条分支比另一条长）。最坏情况如图 2-3 (b) 所示：形成类似链表的病态树，复杂度从期望的对数变为线性。

::: tip 图 2-3：平衡树 (a) 与不平衡/病态树 (b)
```
(a) 平衡:          (b) 病态:
     [4]                [1]
    /   \                 \
  [2]   [6]                [2]
  / \   / \                  \
[1][3][5][7]                  [3]
                                 \
                                  [4]
```
:::

平衡树的定义：高度为 $\log_2 N$（$N$ 为树中元素总数），且两棵子树高度差不超过 1。若不平衡，将失去二叉搜索树结构的性能优势，插入和删除顺序将决定树的形状。

在平衡树中，沿左或右指针平均可将搜索空间减半，查找复杂度为对数 $O(\log_2 N)$。若树不平衡，最坏复杂度可达 $O(N)$。

树在每次操作后通过重组节点进行平衡，以最小化高度并保持两侧节点数在界内，而非不断向一侧分支添加元素使另一侧为空。

保持平衡的一种方式是在添加或删除节点后执行旋转（rotation）。若插入使某分支不平衡（分支中连续两节点仅有一个子节点），可围绕中间节点旋转。图 2-4 中，旋转时中间节点（3）作为旋转轴（pivot）被提升一层，其父节点成为其右子节点。

::: tip 图 2-4：旋转步骤示例
```
   [5]           [3]
   / \    =>    /   \
 [3]  ...    [1]   [5]
 / \
[1][4]
```
:::

## 面向磁盘的树

如前所述，不平衡树最坏复杂度为 $O(N)$，平衡树平均为 $O(\log_2 N)$。同时，由于扇出（fanout，每节点最大子节点数）较低，需要频繁进行平衡、重定位节点和更新指针。维护成本高使 BST 不适合作为磁盘数据结构。

若在磁盘上维护 BST，会面临若干问题。其一是局部性（locality）：元素以随机顺序添加，无法保证新节点写在其父节点附近，子指针可能跨越多个磁盘页。可通过修改树布局、使用分页二叉树（paged binary tree）在一定程度上改善。

另一个与跟随子指针成本密切相关的是树高。由于二叉树的扇出仅为 2，高度是元素数量的二进制对数，定位元素需要 $O(\log_2 N)$ 次寻道及相应次数的磁盘传输。2-3 树等低扇出树有类似限制：作为内存结构有用，但节点过小不适合外部存储。

朴素的磁盘 BST 实现需要与比较次数一样多的磁盘寻道，因为缺乏内置局部性。这促使我们寻找具有该性质的数据结构。

综合这些因素，更适合磁盘实现的树应具备：
- **高扇出**：改善相邻键的局部性
- **低高度**：减少遍历时的寻道次数

扇出与高度呈反比：扇出越高，高度越低。

## 磁盘型结构

我们已从总体上讨论了内存型与磁盘型存储。对具体数据结构也可做同样区分：有的更适合磁盘，有的更适合内存。

并非所有满足空间与复杂度要求的数据结构都能有效用于磁盘存储。数据库使用的数据结构必须针对持久介质限制进行适配。

当数据量过大、无法或不宜将整个数据集放入内存时，会使用磁盘数据结构。任意时刻只有一部分数据可缓存在内存中，其余必须以支持高效访问的方式存储在磁盘上。

### 硬盘驱动器

多数传统算法在旋转磁盘（spinning disk）作为主流持久存储介质时发展起来，其设计深受影响。后来，闪存等新介质催生了新算法及对现有算法的修改。近年来，出现了针对非易失性按字节寻址存储优化的新数据结构。

在旋转磁盘上，寻道（seek）会增加随机读的成本，因为需要磁盘旋转和磁头移动以定位读写头。但一旦完成寻道，读写连续字节（顺序操作）相对便宜。

旋转磁盘的最小传输单位是扇区（sector），典型大小为 512 字节到 4 KB。磁头定位是 HDD 操作中最昂贵的部分，这也是顺序 I/O 常被提及的原因。

### 固态硬盘

固态硬盘（Solid State Drive, SSD）无机械运动部件。典型 SSD 由存储单元组成：单元组成串（通常每串 32–64 个单元），串组成阵列，阵列组成页，页组成块。

根据技术不同，单元可存储一或多位数据。页大小因设备而异，通常为 2–16 KB。块通常包含 64–512 页。块组织成平面，平面置于晶圆上。SSD 可有一个或多个晶圆。图 2-5 展示了这一层次结构。

::: tip 图 2-5：SSD 组织示意
```
Die -> Plane -> Block -> Page -> Array -> String -> Cell
```
:::

最小可写（编程）或可读单位是页。但只能对空单元（即擦除过的单元）进行修改。最小擦除单位不是页，而是包含多页的块（常称擦除块 erase block）。空块中的页必须顺序写入。

闪存控制器中负责将页 ID 映射到物理位置、跟踪空/已写/已废弃页的部分称为闪存转换层（Flash Translation Layer, FTL）。FTL 还负责垃圾回收：找到可安全擦除的块，将仍含存活页的块中的存活页迁移到新位置并重映射页 ID，然后擦除已无用的块。

由于 HDD 和 SSD 都是按块而非按字节寻址，多数操作系统提供块设备抽象，隐藏内部磁盘结构并在内部缓冲 I/O。从块设备读取单个字时，会读取包含它的整个块。这是设计磁盘驻留数据结构时不可忽视的约束。

在 SSD 上，随机与顺序 I/O 的差异不如 HDD 显著，但预取、连续页读取和内部并行仍会造成一定差异。垃圾回收虽多为后台操作，但可能对写性能产生负面影响，尤其在随机、未对齐的写负载下。

仅写入完整块、合并对同一块的连续写入有助于减少所需 I/O 操作，我们将在后续章节讨论缓冲与不可变性作为实现手段。

### 磁盘上的结构

除磁盘访问成本外，构建高效磁盘结构的主要限制与设计条件是：磁盘操作的最小单位是块。要跟随指向块内某位置的指针，必须读取整个块。既然必须如此，可调整数据结构布局以利用这一点。

本章多次提到指针，但对磁盘结构而言语义略有不同。在磁盘上，多数情况下需手动管理数据布局（除非使用内存映射文件）。这与常规指针操作类似，但需显式计算目标地址并跟随指针。

磁盘偏移通常预先计算（当指针在指向内容之前写入磁盘时）或在刷新到磁盘前缓存在内存中。在磁盘结构中创建长依赖链会大幅增加代码与结构复杂度，因此应尽量减少指针数量及其跨度。

总之，磁盘结构针对目标存储特性设计，通常以减少磁盘访问为目标，可通过改善局部性、优化内部表示、减少跨页指针来实现。

在「二叉搜索树」中我们得出结论：高扇出与低高度是理想磁盘数据结构的性质。我们还讨论了指针带来的额外空间开销，以及平衡导致的指针重映射维护开销。B-Tree 综合这些思路：提高节点扇出，降低树高、节点指针数量和平衡操作频率。

## 分页二叉树

如图 2-6 所示，将二叉树节点分组到页中可改善局部性。要找到下一节点，只需在已取页中跟随指针。但节点与指针之间仍有开销。将结构布局到磁盘并维护并非易事，尤其在键值未预排序、以随机顺序添加时。平衡需要页重组，进而导致指针更新。

## 无处不在的 B-Tree

B-Tree 可类比为图书馆的目录室：先选对柜子，再选对架子，再选对抽屉，最后在抽屉中翻找目标卡片。同样，B-Tree 构建层次结构以快速导航和定位目标。

如「二叉搜索树」所述，B-Tree 建立在平衡搜索树基础上，区别在于扇出更高（子节点更多）、高度更低。

文献中二叉树节点常画成圆，因每节点仅负责一个键并将范围分为两部分。B-Tree 节点常画成矩形，指针块也显式画出以突出子节点与分隔键的关系。图 2-7 并排展示二叉树、2-3 树和 B-Tree 节点。

B-Tree 是有序的：节点内键按序存储，因此可用二分查找等算法定位目标键，查找具有对数复杂度。例如，在 40 亿（$4 \times 10^9$）项中查找约需 32 次比较。若每次比较都需磁盘寻道，会显著变慢；但 B-Tree 节点可存储数十甚至数百项，每层只需一次磁盘寻道。

使用 B-Tree 可高效执行点查询（point query）和范围查询（range query）。点查询由等值谓词（=）表达，定位单项；范围查询由比较谓词（<、>、≤、≥）表达，用于按序查询多项。

### B-Tree 层次

B-Tree 由多个节点组成。每个节点最多持有 $N$ 个键和 $N+1$ 个指向子节点的指针。这些节点逻辑上分为三类：

| 类型 | 说明 |
|------|------|
| **根节点（Root node）** | 无父节点，位于树顶 |
| **叶节点（Leaf nodes）** | 底层节点，无子节点 |
| **内部节点（Internal nodes）** | 连接根与叶的其他节点，通常有多层 |

节点容量与其实际持有的键数之比称为占用率（occupancy）。

B-Tree 以扇出为特征。高扇出有助于分摊保持平衡所需的结构变更成本，并通过在单个或连续块中存储键和子指针减少寻道次数。当节点满或接近空时触发平衡操作（分裂与合并）。

### B+-Tree

我们将 B-Tree 作为具有上述全部或大部分性质的数据结构族的统称。更精确的名称是 B+-Tree。B-Tree 允许在任意层存储值（根、内部、叶）；B+-Tree 仅在叶节点存储值，内部节点仅存储用于引导搜索的分隔键（separator key）。

由于 B+-Tree 的值仅在叶层，所有操作（插入、更新、删除、检索）仅影响叶节点，仅在分裂与合并时向上传播。

B+-Tree 已广泛使用，本书与其他文献一样称其为 B-Tree。

### 分隔键

B-Tree 节点中的键称为索引项（index entry）、分隔键（separator key）或分隔单元（divider cell）。它们将树划分为持有对应键范围的子树。键按序存储以支持二分查找。通过定位键并沿对应指针从高层到低层找到子树。

节点中第一个指针指向小于第一个键的子树，最后一个指针指向大于等于最后一个键的子树。其他指针引用两键之间的子树：$K_{i-1} \le K_s < K_i$，其中 $K$ 为键集，$K_s$ 为子树中的键。

部分 B-Tree 变体在叶层有兄弟节点指针以简化范围扫描，有的实现双向指针形成叶层双向链表以支持反向迭代。

B-Tree 的独特之处在于其自底向上构建（与二叉搜索树自顶向下相反）：叶节点数量增长，内部节点与树高随之增加。

B-Tree 在节点内预留空间供未来插入与更新，存储利用率可低至 50%，但通常更高。高占用率不会对 B-Tree 性能产生负面影响。

### B-Tree 查找复杂度

可从两个角度看待 B-Tree 查找复杂度：块传输次数与比较次数。

就传输次数而言，对数为以 $N$（每节点键数）为底。每层节点数增加 $K$ 倍，跟随子指针将搜索空间缩小 $N$ 倍。查找时最多访问 $\log_K M$ 个页（$M$ 为 B-Tree 中总项数）。根到叶路径上需跟随的子指针数等于层数，即树高 $h$。

就比较次数而言，对数为以 2 为底，因每节点内用二分查找。每次比较将搜索空间减半，复杂度为 $\log_2 M$。

在教科书中，B-Tree 查找复杂度通常记为 $\log M$。复杂度分析中一般不写底数，因为换底只增加常数因子，常数乘法不改变复杂度。

### B-Tree 查找算法

查找算法从根开始，执行一次根到叶的遍历。目标是找到目标键或其前驱。精确匹配用于点查询、更新和删除；前驱用于范围扫描和插入。

算法从根开始，将目标键与根节点中的键进行二分比较，直到找到第一个大于目标值的分隔键，从而定位目标子树。找到子树后，沿对应指针继续相同过程，直到到达目标叶节点，在其中找到目标键或通过定位前驱确认不存在。

每层获得更细的视图：从最粗粒度（根）开始，逐层下降到更精确的键范围，最终到达存放数据记录的叶层。

点查询在找到或确认未找到目标键后结束。范围扫描从最接近的键值对开始，沿兄弟指针迭代直到范围结束或谓词耗尽。

### 键的计数

文献中对键与子节点偏移计数的描述方式不同。[BAYER72] 提到与设备相关的自然数 $k$ 表示最优页大小，页可持有 $k$ 到 $2k$ 个键，子指针数至少 $k+1$、最多 $2k+1$。根页可持有 1 到 $2k$ 个键。本书为清晰起见，统一用 $N$ 表示键数（叶节点为键值对数）。

### B-Tree 节点分裂

插入时，先定位目标叶并找到插入点。若目标节点空间不足，称节点溢出（overflow），需分裂为两个以容纳新数据。更精确地，满足以下条件时分裂：

- **叶节点**：若节点最多可持 $N$ 个键值对，再插入一个将超过容量
- **非叶节点**：若节点最多可持 $N+1$ 个指针，再插入一个将超过容量

分裂步骤：分配新节点，将分裂节点一半元素转移过去，将第一个键和指针加入父节点（该键被提升 promoted）。分裂点（split point）后的元素（非叶节点分裂时含分裂点）转移到新兄弟节点，其余保留在原节点。

若父节点已满无法容纳提升的键和指向新节点的指针，父节点也需分裂，该操作可能递归传播到根。

当分裂传播到根时，需分裂根节点。分裂根时，分配新根持有分裂点键，旧根与新兄弟降级到下一层，树高加一。树高仅在根分裂或两节点合并形成新根时变化；在叶层和内部节点层，树只横向增长。

分裂的四个步骤：
1. 分配新节点
2. 将分裂节点一半元素复制到新节点
3. 将新元素放入对应节点
4. 在分裂节点的父节点添加分隔键和指向新节点的指针

### B-Tree 节点合并

删除时先定位目标叶，然后移除键及其关联值。

若相邻节点元素过少（占用率低于阈值），则合并兄弟节点，称为下溢（underflow）。[BAYER72] 描述两种下溢：若两相邻节点有共同父节点且内容可放入单节点，则合并；若放不下，则在两节点间重新分配键以恢复平衡。

合并的三个步骤（假定元素已移除）：
1. 将右节点所有元素复制到左节点
2. 从父节点移除右节点指针（非叶合并时将其降级）
3. 移除右节点

B-Tree 中常通过重平衡（rebalancing）减少分裂与合并次数。

## 小结

本章从为磁盘存储创建专用结构的动机出发。二叉搜索树虽有类似复杂度特性，但因低扇出及平衡导致的大量重定位与指针更新而不适合磁盘。B-Tree 通过提高每节点存储项数（高扇出）和降低平衡频率解决这两个问题。

随后我们讨论了 B-Tree 内部结构及查找、插入、删除算法的概要。分裂与合并用于在增删元素时保持平衡。我们将树深最小化，在节点仍有空闲时向现有节点添加项。

这些知识可用于创建内存 B-Tree。要实现磁盘版本，还需深入 B-Tree 节点在磁盘上的布局及数据编码格式。
