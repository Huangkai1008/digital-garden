# 第5章 事务处理与恢复

本书采用自底向上的方式介绍数据库概念：先学习存储结构，现在转向负责缓冲管理、锁管理和恢复的高层组件，这些是理解数据库事务的前提。

事务（transaction）是数据库管理系统中不可分割的逻辑工作单元，允许多个操作表示为单一步骤。事务具有原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）和持久性（Durability），合称 ACID。

| 属性 | 说明 |
|------|------|
| **原子性** | 事务步骤不可分割，要么全部成功要么全部不执行；可提交或中止 |
| **一致性** | 事务只将数据库从一个有效状态带到另一个有效状态，保持所有不变量 |
| **隔离性** | 并发事务互不干扰；隔离级别决定变更何时可见 |
| **持久性** | 提交后所有修改必须持久化到磁盘，能经受断电、故障和崩溃 |

实现事务需要事务管理器、锁管理器、页缓存和日志管理器协同工作。

## 缓冲管理

多数数据库采用两级内存层次：较慢的持久存储（磁盘）和较快的主存（RAM）。为减少持久存储访问，页被缓存在内存中。页缓存（page cache，又称 buffer pool）负责缓存从磁盘读取的页。

页缓存的主要功能：在内存中保存缓存页内容；允许对磁盘页的修改在缓存版本上缓冲执行；请求的页不在内存且有空间时将其换入；缓存满时换出并刷新到磁盘。

许多数据库使用 `O_DIRECT` 标志打开文件，绕过内核页缓存直接访问磁盘。缓存语义：所有对缓冲区的修改保留在内存直到写回磁盘；页被修改时标记为脏（dirty）；脏页在换出前必须刷新。

### 缓存淘汰

页缓存容量有限，需淘汰旧页以容纳新内容。脏页在换出前必须刷新。常用淘汰策略：

- **FIFO**：按插入顺序淘汰，不考虑访问模式，不实用
- **LRU**：最近最少使用，访问时移到队列尾部
- **CLOCK**：环形缓冲，用访问位近似 LRU，Linux 使用其变体
- **TinyLFU**：基于使用频率，Caffeine 库实现

### 页固定与预取

高概率被访问的页可固定（pinning）在缓存中。B-Tree 越往上越窄，高层节点对多数读有利，可常驻内存。预取可在访问前提前加载页；大顺序扫描后页可立即换出。

### 页替换

理想策略需预测未来访问顺序。Bélády 异常表明：使用非最优算法时，增大缓存可能增加换出次数。需在减少磁盘访问、提前刷新、最优换出顺序、控制缓存大小、保证持久性之间权衡。

## 恢复

预写日志（Write-Ahead Log, WAL）是仅追加的辅助磁盘结构，用于崩溃和事务恢复。页缓存允许在内存中缓冲修改，在刷新到磁盘前，WAL 是保存操作历史的唯一磁盘副本。

WAL 的主要功能：允许页缓存缓冲更新同时保证持久性；在受影响页的缓存副本同步到磁盘前持久化所有操作；崩溃时从操作日志重建丢失的内存修改。

### 日志语义

WAL 仅追加且内容不可变。WAL 由日志记录组成，每条记录有唯一单调递增的日志序列号（LSN）。除操作记录外，WAL 还持有表示事务完成的记录。事务在提交记录 LSN 被强制写入前不能视为已提交。

### 操作日志与数据日志

状态变更可由前像/后像或 redo/undo 操作表示。物理日志记录前后像；逻辑日志指定需执行的操作。实践中许多系统结合使用：逻辑日志做 undo（并发与性能），物理日志做 redo（恢复时间）。

### Steal 与 Force 策略

- **Steal**：允许在事务提交前刷新其修改的页
- **No-steal**：不允许刷新未提交内容
- **Force**：提交前必须刷新所有修改页
- **No-force**：允许在部分修改页未刷新时提交

### ARIES

ARIES 是 steal/no-force 恢复算法，使用物理 redo 和逻辑 undo。崩溃后恢复分三阶段：

1. **分析阶段**：识别脏页和崩溃时进行中的事务
2. **Redo 阶段**：重放历史到崩溃点，恢复数据库状态
3. **Undo 阶段**：回滚未完成事务，恢复到最后一致状态

## 并发控制

并发控制是处理并发事务交互的技术集合，大致分为：

| 类型 | 说明 |
|------|------|
| **乐观并发控制（OCC）** | 事务自由执行，提交前验证是否可串行化，冲突则中止 |
| **多版本并发控制（MVCC）** | 允许多个带时间戳的版本存在，读可访问旧版本 |
| **悲观并发控制（PCC）** | 运行中检测冲突并阻塞或中止 |

### 可串行性

调度（schedule）是执行一组事务所需操作的列表。串行调度中事务完全独立执行。可串行化调度等价于某串行调度，产生相同结果。

### 事务隔离

隔离级别描述事务间隔离程度及可能出现的异常：

- **脏读（dirty read）**：读取未提交的修改
- **不可重复读（nonrepeatable read）**：同一行两次读取结果不同
- **幻读（phantom read）**：范围查询两次结果不同
- **丢失更新（lost update）**：后提交覆盖先提交
- **写偏斜（write skew）**：各自满足不变量但组合违反

### 隔离级别

| 级别 | 允许的异常 |
|------|------------|
| 读未提交 | 脏读、不可重复读、幻读 |
| 读已提交 | 不可重复读、幻读 |
| 可重复读 | 幻读 |
| 可串行化 | 无 |

快照隔离（snapshot isolation）：事务观察启动时已提交的变更，修改的值在执行期间未变才能提交。可能发生写偏斜。

### 乐观并发控制

分三阶段：读阶段（在私有上下文中执行）、验证阶段（检查冲突）、写阶段（提交写集）。验证可面向已提交事务（后向）或正在验证的事务（前向）。

### 多版本并发控制

允许多个记录版本，使用单调递增的事务 ID 或时间戳。读可继续访问旧值直到新值提交。可用于实现快照隔离。

### 悲观并发控制

时间戳排序：每事务有时间戳，是否允许执行取决于更早时间戳事务是否已提交。Thomas 写规则：时间戳早于 max_write_timestamp 的写可忽略。

### 基于锁的并发控制

两阶段锁（2PL）分增长阶段（获取所有锁）和收缩阶段（释放锁）。一旦释放任一锁就不能再获取新锁。

### 死锁

两事务互相等待对方释放锁即死锁。处理方式：超时中止、保守 2PL（预先获取所有锁）、等待图检测、时间戳优先级（wait-die、wound-wait）避免。

### 锁与闩锁

- **锁（lock）**：保护逻辑数据完整性，在键上获取，由锁管理器管理，事务期间持有
- **闩锁（latch）**：保护物理表示（页内容、树结构），在页级获取，操作期间持有

### 读写锁

读写锁允许多读者并发访问，写者需独占。只有读者可共享锁所有权。

### 闩锁爬行

闩锁爬行（latch crabbing）：尽快释放不再需要的父闩锁。读路径上，定位子节点并获取其闩锁后可释放父闩锁。插入时若子节点未满可释放父闩锁；删除时若子节点足够满、不会导致合并可释放父闩锁。

### Blink-Tree

Blink-Tree 在 B*-Tree 基础上增加高键和兄弟链接。允许半分裂状态：节点已通过兄弟指针引用但尚未通过父节点子指针引用。若搜索键超过节点高键，沿兄弟链接继续搜索。无需在下降时持有父锁，即使子节点将分裂。

## 小结

本章讨论了负责事务处理与恢复的存储引擎组件。实现事务处理面临两问题：提高效率需允许并发执行；保持正确性需确保 ACID。并发执行可能导致读写异常，隔离级别描述并限制这些异常。页缓存减少磁盘访问；WAL 确保未刷新修改不丢失并支持回滚。页缓存与 WAL 通过 force 和 steal 策略协调。
