# 第3章 文件格式

在掌握 B-Tree 的基本语义后，我们开始探讨 B-Tree 及其他结构在磁盘上的具体实现。磁盘访问方式与内存不同：从应用开发者视角看，内存访问大多透明；由于虚拟内存，我们不必手动管理偏移。磁盘通过系统调用访问，通常需指定目标文件内的偏移，并将磁盘表示解释为适合内存的形式。

高效的磁盘结构必须考虑这一区别。为此，需要设计易于构建、修改和解释的文件格式。本章讨论有助于设计各类磁盘结构（不限于 B-Tree）的通用原则与实践。

B-Tree 实现有多种可能，我们讨论若干实用技术。实现细节各异，但通用原则一致。理解 B-Tree 的基本机制（如分裂与合并）是必要的，但不足以完成实际实现，许多环节需协同工作。

磁盘结构中指针管理的语义与内存略有不同。可将磁盘 B-Tree 视为页管理机制：算法需组合和导航页，页及其指针需相应计算与放置。

由于 B-Tree 的复杂性主要来自可变性，我们讨论页布局、分裂、重定位等适用于可变结构的细节。后续讨论 LSM Tree 时，将聚焦排序与维护，那是 LSM 复杂性的主要来源。

## 动机

创建文件格式在许多方面类似于在具有非托管内存模型的语言中创建数据结构：分配数据块并按需切分，使用定长基本类型和结构；引用更大内存块或变长结构时使用指针。

非托管内存模型允许在需要时分配更多内存，而不必关心是否有连续段、是否碎片化或释放后会发生什么。在磁盘上，我们需自行处理垃圾回收与碎片。

数据布局在内存中不如在磁盘上重要。要使磁盘驻留数据结构高效，需以支持快速访问的方式布局数据，考虑持久存储介质的特性，设计二进制数据格式，并找到高效的序列化与反序列化手段。

## 二进制编码

为高效存储，数据需用紧凑且易于序列化/反序列化的格式编码。讨论二进制格式时常见「布局」（layout）一词。由于没有 `malloc` 和 `free` 等原语，只有读写，必须以不同方式思考访问并相应准备数据。

### 基本类型

键和值具有类型（如整数、日期、字符串），可以原始二进制形式表示。多数数值类型为定长。处理多字节数值时，编码与解码必须使用相同字节序（endianness）：

| 字节序 | 说明 |
|--------|------|
| **大端（Big-endian）** | 从最高有效字节（MSB）开始，按重要性递减排列 |
| **小端（Little-endian）** | 从最低有效字节（LSB）开始，按重要性递增排列 |

::: tip 图 3-1：大端与小端字节序
```
0xAABBCCDD:
大端: AA BB CC DD  (地址从左到右增长)
小端: DD CC BB AA
```
:::

记录由数字、字符串、布尔值等基本类型及其组合构成。传输或存储时只能使用字节序列，因此需序列化（转为可解释的字节序列）和反序列化（将字节序列还原为原始记录）。

二进制格式中，基本类型是更复杂结构的构建块。`byte` 为 8 位，`short` 为 2 字节，`int` 为 4 字节，`long` 为 8 字节。浮点数由符号、尾数和指数表示，IEEE 754 描述了广泛采用的表示方式。

### 字符串与变长数据

所有基本数值类型为定长。字符串和变长类型可序列化为：表示长度或数组大小的数字，后跟实际数据。这种表示常称 UCSD String 或 Pascal String：

```
String {
    size  uint_16
    data  byte[size]
}
```

另一种是空终止字符串，读者逐字节消费直到遇到结束符。Pascal 方式的优点：可在常数时间获知长度，可将 `size` 字节切片传入字符串构造函数。

### 位打包数据：布尔、枚举与标志

布尔可用单字节表示，或将 true/false 编码为 1/0。由于布尔只有两种值，用整字节浪费，开发者常将 8 个布尔打包，每位表示一个。1 表示置位，0 表示未置位。

枚举（enum）可表示为整数，常用于二进制格式和通信协议。标志（flags）是打包布尔与枚举的结合，可表示非互斥的命名布尔参数。每位代表一个标志，掩码只能使用 2 的幂：

```c
int IS_LEAF_MASK         = 0x01;  // bit #1
int VARIABLE_SIZE_VALUES = 0x02;  // bit #2
int HAS_OVERFLOW_PAGES   = 0x04;  // bit #3

// 置位
flags |= HAS_OVERFLOW_PAGES;
// 清除
flags &= ~HAS_OVERFLOW_PAGES;
// 测试
is_set = (flags & HAS_OVERFLOW_PAGES) != 0;
```

## 通用原则

设计文件格式时，通常先决定寻址方式：是否将文件划分为同大小页（由单块或多块连续块表示）。多数就地更新存储结构使用同大小页，因可显著简化读写访问。仅追加存储结构也常按页写入：记录依次追加，页在内存中填满后刷新到磁盘。

文件通常以定长头部开始，可能以定长尾部结束，存放需快速访问或解码其余内容所需的辅助信息。其余部分划分为页。

::: tip 图 3-3：文件组织
```
+--------+-------+-------+-------+--------+
| Header | Page1 | Page2 | Page3 | Trailer|
+--------+-------+-------+-------+--------+
```
:::

若设计存储员工姓名、生日、税号、性别的公司目录格式，可将定长字段放在结构头部，变长字段随后。为免多字段计算，可在定长区编码偏移和长度，从而单独定位任意变长字段。

构建更复杂结构通常涉及层次：由基本类型组成字段，由字段组成单元（cell），由单元组成页，由页组成段，由段组成区域等。无严格规则，取决于要格式化的数据类型。

## 页结构

数据库系统将数据记录存储在数据文件和索引文件中。这些文件划分为称为页（page）的定长单元，页大小通常为多个文件系统块，常见为 4–16 KB。

[Bayer72] 描述的定长记录页组织是三元组的简单拼接：k 表示键，v 表示关联值，p 表示子页指针。这种方式简单但有缺点：在非右侧任意位置追加键需重定位元素；无法高效管理或访问变长记录，仅适用于定长数据。

## 槽式页

存储变长记录时，主要问题是空闲空间管理：回收被删除记录占用的空间。若将大小为 n 的记录放入先前大小为 m 的记录所占空间，除非 m=n 或能找到恰好 m-n 大小的另一记录，该空间将无法使用。

为简化变长记录的空间管理，可将页分为定长段，但这也会浪费空间。例如段大小为 64 字节时，除非记录是 64 的倍数，否则会浪费 $64 - (n \mod 64)$ 字节。

空间回收可通过重写页并移动记录实现，但需保留记录偏移，因为页外指针可能使用这些偏移。

槽式页（slotted page）或槽目录（slot directory）可满足以下需求：
- 以最小开销存储变长记录
- 回收被删除记录占用的空间
- 通过 ID 引用页内记录，与具体位置无关

我们将页组织为槽或单元的集合，将指针与单元分置于页的两侧。这样只需重组指向单元的指针以保持顺序，删除记录可通过置空指针或移除实现。

::: tip 图 3-5：槽式页
```
+------------------+------------------+
| Header           | Cell pointers    |
|                  | (offsets)        |
+------------------+------------------+
|                  | Free space       |
|                  |                  |
+------------------+------------------+
| Cell 1 | Cell 2  | Cell 3 | ...     |
+------------------+------------------+
```
:::

## 单元布局

使用标志、枚举和基本值可设计单元布局，再将单元组合成页，由页组成树。在单元层面，键单元（key cell）与键值单元（key-value cell）不同：键单元持有分隔键和指向两相邻指针之间页的指针；键值单元持有键及其关联数据记录。

假定页内单元统一（例如全部仅持键或持键值；全部定长或变长，不混用），因此可在页级存储一次单元元数据，而不必在每个单元中重复。

变长键单元布局示例：
```
0                4               8
+----------------+---------------+-------------+
| [int] key_size | [int] page_id | [bytes] key |
+----------------+---------------+-------------+
```

键值单元以数据记录替代子页 ID，结构类似。偏移与页 ID 的区别：页有定长并由页缓存管理，只需存储页 ID，之后通过查找表转换为文件内实际偏移。单元偏移是页内局部的，相对于页起始偏移。

## 将单元组合成槽式页

单元追加到页的右侧（ toward 页尾），单元偏移/指针保存在页的左侧。键可按任意顺序插入，逻辑排序通过按键序排序单元偏移指针保持。该设计允许以最小代价追加单元，插入、更新或删除时无需重定位单元。

## 管理变长数据

从页中移除项不必删除实际单元并移动其他单元以占用释放空间。可将单元标记为已删除，并更新内存中的可用列表（availability list），记录释放的内存量和指向释放值的指针。插入新单元时，先检查可用列表是否有可容纳的段。

适配策略：
- **首次适配（First fit）**：可能留下过小无法再用的剩余空间
- **最佳适配（Best fit）**：选择插入后剩余最小的段

若碎片字节足够但无足够连续字节，可读取并重写存活单元以整理页并回收空间。若整理后仍无足够空间，需创建溢出页（overflow page）。

## 版本控制

数据库系统不断演进，二进制文件格式可能变化。多数存储引擎需支持多种序列化格式（当前格式及一种或多种旧格式以保持向后兼容）。版本可通过文件名前缀、单独版本文件或索引文件头部存储。部分格式使用魔数（magic number）标识版本。

## 校验和

磁盘上的文件可能因软件缺陷或硬件故障损坏。为主动发现问题并避免将损坏数据传播到其他子系统或节点，可使用校验和（checksum）与循环冗余校验（CRC）。

校验和提供最弱保证，无法检测多位损坏。CRC 可帮助检测突发错误（如多连续位损坏）。非加密哈希和 CRC 不应用于验证数据是否被篡改，应使用强加密哈希。

写入前计算校验和并与数据一起写入。读取时重新计算并与写入值比较。若不匹配，则发生损坏，不应使用读取的数据。通常对页计算校验和并放在页头，这样校验更稳健，且若损坏仅限于单页，不必丢弃整个文件。

## 小结

本章学习了二进制数据组织：如何序列化基本类型、组合成单元、由单元构建槽式页并导航这些结构。学习了如何处理字符串、字节序列、数组等变长类型，以及如何组合持有值大小的特殊单元。讨论了槽式页格式，允许通过单元 ID 从页外引用单元，按插入顺序存储记录，并通过排序单元偏移保持键序。这些原则可用于构建磁盘结构的二进制格式和网络协议。
