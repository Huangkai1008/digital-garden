# 第4章 实现 B-Tree

上一章讨论了二进制格式组合的通用原则，以及如何创建单元、构建层次并用指针连接到页。这些概念适用于就地更新和仅追加存储结构。本章讨论 B-Tree 特有的概念。

本章分为三个逻辑组。首先讨论组织：如何建立键与指针的关系，如何实现页头和页间链接。其次讨论根到叶下降过程中的操作：如何执行二分查找、如何收集面包屑（breadcrumbs）并跟踪父节点以便后续分裂或合并。最后讨论优化技术（重平衡、仅右追加、批量加载）、维护过程和垃圾回收。

## 页头

页头（page header）存放用于导航、维护和优化的信息，通常包含描述页内容与布局的标志、页内单元数、标记空闲区的上下偏移，以及其他元数据。

例如 PostgreSQL 在头部存储页大小和布局版本；MySQL InnoDB 存储堆记录数、层级等；SQLite 存储单元数和最右指针。

### 魔数

文件或页头中常放置魔数（magic number），通常为多字节常量，用于标识块表示页、指定类型或版本。魔数用于验证与健全性检查。例如可将 `50 41 47 45`（PAGE 的十六进制）写入头部，读取时比较以验证页加载与对齐是否正确。

### 兄弟链接

部分实现存储指向前后兄弟页的链接。这些链接可在不回溯父节点的情况下定位相邻节点，但会增加分裂与合并的复杂度，因为需更新兄弟偏移。例如非最右节点分裂时，其右兄弟的后向指针需重新绑定到新节点。

### 最右指针

B-Tree 分隔键有严格不变量：用于将树划分为子树并导航，因此子页指针数总比键数多 1。许多实现中，每个分隔键对应一个子指针，最后一个指针单独存储，因其不与任何键配对。若最右子节点分裂且新单元追加到父节点，最右子指针需重新赋值。

### 节点高键

另一种做法是将最右指针与节点高键（high key）一起存储在单元中。高键表示当前节点子树中可能存在的最大键。PostgreSQL 采用此方式，称为 Blink-Tree。Blink-Tree 为每个节点增加 $K_{N+1}$ 键，指定指针 $P_N$ 所指子树中键的上界。

### 溢出页

节点大小和树扇出是固定的。若存在足够大的变长值，一页只能容纳少量记录；若值很小，又会浪费预留空间。B-Tree 算法要求每节点保持特定数量的项，但定长页可能已无连续空闲空间。

为实现变长节点而不复制到新连续区域，可由多个链接页构建节点。默认页大小为 4K，插入若干值后数据超过 4K 时，可分配 4K 扩展页并从原页链接。这些链接的扩展页称为溢出页（overflow page），原页称为主页（primary page）。

多数 B-Tree 实现只允许在节点中直接存储固定数量的负载字节，其余溢出到溢出页。该值由节点大小除以扇出计算。若插入负载大于 `max_payload_size`，检查是否已有溢出页；若有且空间足够则溢出，否则分配新溢出页。键通常基数高，在主页存储部分键有意义，因多数比较可在主页的截断键上完成。

## 二分查找

B-Tree 查找算法在节点内用二分查找定位目标键。二分查找仅适用于有序数据。二分查找接收有序数组和目标键，返回位置；正数表示找到，负数表示未找到并给出插入点。插入点为第一个大于给定键的元素的索引。

高层搜索多数不产生精确匹配，我们关心搜索方向，需找到第一个大于目标值的键并沿对应子链接进入子树。

### 带间接指针的二分查找

B-Tree 页中单元按插入顺序存储，仅单元偏移保持逻辑顺序。二分查找时取中间单元偏移，跟随指针定位单元，比较键以决定向左或向右，递归直到找到目标或插入点。

## 传播分裂与合并

B-Tree 分裂与合并可向上传播。为此需能从分裂叶或合并叶对回溯到根。部分实现使用父节点指针；由于低层页在被高层引用时会被换入，不必在磁盘上持久化该信息。

部分实现（如 WiredTiger）使用父指针遍历叶节点以避免死锁，而非兄弟指针。

### 面包屑

不存储父指针时，可记录根到目标叶路径上遍历的节点，在插入导致级联分裂或删除导致合并时按逆序回溯。面包屑（breadcrumbs）包含从根跟随的节点引用，用于传播分裂或合并时回溯。最自然的数据结构是栈。例如 PostgreSQL 将面包屑存储在 BTStack 中。

若节点分裂或合并，可用面包屑找到提升到父节点的键的插入点，并在必要时向上传播结构变更。

## 重平衡

部分 B-Tree 实现通过重平衡（rebalancing）推迟分裂与合并以分摊成本：在同层内重新分配元素，或将元素从较满节点移到较空节点，直到最后才执行分裂或合并。这有助于提高节点占用率并可能减少树层数，但可能增加重平衡的维护成本。

负载均衡可在插入和删除时执行。为提高空间利用率，溢出时不立即分裂，可将部分元素转移到兄弟节点以腾出空间。同样，删除时不立即合并，可从相邻节点移动元素使节点至少半满。

B*-Tree 在兄弟节点间持续分配数据直到两兄弟都满，然后将两节点分裂为三个各约三分之二满的节点。SQLite 使用此变体。该方法通过推迟分裂提高平均占用率，但需要额外跟踪与平衡逻辑。

## 仅右追加

许多数据库使用自增单调主键。此时所有插入发生在索引末端（最右叶），多数分裂发生在每层最右节点。PostgreSQL 称此为 fastpath：当插入键严格大于最右页的第一个键且最右页有足够空间时，可直接在缓存的最右叶插入，跳过整个读路径。

SQLite 有类似概念 quickbalance：当在最右端插入且目标节点已满时，不重平衡或分裂，而是分配新的最右节点并将其指针加入父节点。尽管新页几乎为空，但很可能很快被填满。

## 批量加载

若有预排序数据并需批量加载或重建树（如整理碎片），可进一步利用仅右追加的思路。由于数据已排序，批量加载时只需在树的最右位置追加。可完全避免分裂与合并，自底向上逐层构建树。

一种实现方式是按页写入预排序数据到叶层，叶页写完后将其第一个键传播到父节点，用常规算法构建更高层。由于追加键已排序，所有分裂发生在最右节点。叶层可全部写出后再构建高层，构建高层时所有子指针已就绪。该方法的主要好处是不必在磁盘上执行分裂或合并，且构建时只需在内存中保留树的最小部分（当前填充叶节点的所有父节点）。

## 压缩

存储未压缩原始数据会带来显著开销，许多数据库提供压缩以节省空间。权衡在访问速度与压缩比之间：更高压缩比可减少数据量、单次访问获取更多数据，但可能需要更多 RAM 和 CPU 进行压缩与解压。

压缩可在不同粒度进行。整文件压缩可获得更好压缩比，但更新时需重新压缩整个文件，更细粒度压缩通常更适合大数据集。按页压缩与固定大小页的讨论一致，页可独立压缩与解压。多数开源数据库使用可插拔压缩方法，如 Snappy、zLib、lz4 等。

## 真空与维护

除面向用户的操作外，还有与查询并行运行的维护过程：保持存储完整性、回收空间、减少开销、保持页有序。在后台执行这些操作可节省时间，避免在插入、更新、删除时支付清理成本。

槽式页设计需要维护以保持页良好状态。内部节点的分裂与合并、叶层的插入、更新与删除可能导致页有足够逻辑空间但无足够连续空间（碎片化）。删除仅移除单元偏移，不重定位剩余单元或物理删除目标单元，释放的字节可能分散在页中，需整理（defragmentation）。

负责空间回收和页重写的进程称为压缩（compaction）、真空（vacuum）或维护（maintenance）。若页无足够连续物理空间，可在写入时同步重写页；但压缩通常指独立的异步过程：遍历页、执行垃圾回收、重写内容。该过程回收死单元占用的空间，按逻辑顺序重写单元。重写时页可能迁移到文件新位置。未使用的内存页归还页缓存，新可用的磁盘页 ID 加入空闲页列表（freelist）。

## 小结

本章讨论了磁盘 B-Tree 实现特有的概念：页头、最右指针、高键、溢出页；根到叶遍历相关细节（带间接指针的二分查找、父指针或面包屑）；以及优化与维护技术：重平衡、仅右追加、批量加载、垃圾回收。这些概念应能弥合基本 B-Tree 算法与实际实现之间的差距。
