# 第11章 复制与一致性

在深入讨论共识和原子提交算法之前，我们先拼齐理解它们所需的最后一块拼图：**一致性模型（consistency models）**。一致性模型很重要，因为它们解释在存在多个数据副本时系统的可见性语义和行为。

**容错（fault tolerance）** 是系统在组件故障存在下仍能正确运行的属性。使系统容错并非易事，向现有系统添加容错可能很困难。主要目标是消除系统中的单点故障，并确保关键组件有冗余。通常，冗余对用户完全透明。

系统可以通过存储多个数据副本来继续正确运行，这样当一台机器故障时，另一台可以作为故障转移。在具有单一真实源（例如主/从数据库）的系统中，故障转移可以显式完成，通过提升副本成为新主节点。其他系统不需要显式重新配置，通过在读和写查询期间从多个参与者收集响应来确保一致性。

**数据复制（data replication）** 是通过在系统中维护多个数据副本来引入冗余的方式。然而，由于原子更新多个数据副本的问题等价于共识，对数据库中的每个操作执行此操作可能代价高昂。我们可以探索更经济、更灵活的方式，使数据从用户视角看起来一致，同时允许参与者之间存在一定程度的分歧。

复制在多数据中心部署中尤为重要。地理复制服务于多种目的：通过提供冗余增加可用性和承受一个或多个数据中心故障的能力，也可以通过将数据副本物理放置在更靠近客户端的位置来降低延迟。

## 实现可用性

我们已讨论过分布式系统的谬误并识别了许多可能出错的地方。在现实世界中，节点并不总是存活或能够相互通信。然而，间歇性故障不应影响可用性：从用户视角，系统整体必须继续运行，仿佛什么都没发生。

系统可用性是极其重要的属性。为使系统高可用，我们需要以允许优雅处理一个或多个参与者故障或不可用的方式设计它。为此，我们需要引入冗余和复制。然而，一旦添加冗余，我们就面临保持多个数据副本同步的问题，并必须实现恢复机制。

### 臭名昭著的 CAP

**可用性（availability）** 衡量系统成功响应每个请求的能力。理想情况下，我们希望每个操作都是一致的。**一致性（consistency）** 在此定义为原子或线性化一致性。我们希望在容忍网络分区的同时实现一致性和可用性。

网络可能被分成若干部分，进程无法相互通信：分区节点之间发送的某些消息无法到达目的地。可用性要求任何非故障节点交付结果，而一致性要求结果是线性化的。**CAP 猜想**由 Eric Brewer 提出，讨论一致性（Consistency）、可用性（Availability）和分区容错（Partition tolerance）之间的权衡。

在异步系统中不可能满足可用性要求，我们无法在存在网络分区的情况下同时保证可用性和一致性。我们可以构建保证强一致性同时尽力提供可用性的系统，或保证可用性同时尽力提供一致性的系统。

CAP 描述了一个潜在选择的连续体，在光谱的不同端我们有：

| 类型 | 说明 |
|------|------|
| **CP 系统** | 一致且分区容错，宁愿使请求失败也不提供可能不一致的数据 |
| **AP 系统** | 可用且分区容错，放宽一致性要求，允许在请求期间提供可能不一致的值 |

**PACELEC 猜想**是 CAP 的扩展，指出在存在网络分区时，有一致性与可用性之间的选择（PAC）。否则（E），即使系统正常运行，我们仍必须在延迟与一致性之间做出选择。

### 谨慎使用 CAP

重要的是注意 CAP 讨论的是网络分区而非节点崩溃或任何其他类型的故障。分区于集群其余部分的节点可以提供不一致的请求，但崩溃的节点根本不会响应。

### 收获与产出（Harvest and Yield）

CAP 猜想仅讨论最强形式的一致性和可用性。然而，某些应用可以从略微放宽的假设中受益。我们可以定义两个可调指标：**收获（harvest）**（查询的完整程度）和**产出（yield）**（成功完成的请求数与尝试的请求总数之比）。这将权衡焦点从绝对转向相对。

## 共享内存（Shared Memory）

对客户端而言，存储数据的分布式系统表现得好像有共享存储，类似于单节点系统。节点间通信和消息传递被抽象并在幕后发生。这创造了**共享内存（shared memory）**的幻觉。

可通过读或写操作访问的单一存储单元通常称为**寄存器（register）**。我们可以将分布式数据库中的共享内存视为此类寄存器的数组。

我们通过其调用和完成事件识别每个操作。若两个操作的调用和完成事件都发生在另一个操作被调用之前，我们说此操作先于另一个，这两个操作是**顺序的**。否则，我们说它们是**并发的**。

根据寄存器在存在并发操作时的行为，我们区分三种寄存器类型：

| 类型 | 说明 |
|------|------|
| **安全（Safe）** | 并发写期间，对安全寄存器的读可能返回寄存器范围内的任意值 |
| **常规（Regular）** | 读只能返回最近完成的写所写的值，或与当前读重叠的写所写的值 |
| **原子（Atomic）** | 保证线性化：每个写操作有单一时刻，此前每次读返回旧值，此后每次读返回新值 |

## 排序（Ordering）

在分布式系统中，很难知道某事何时确切发生并立即在集群中可用此信息。每个参与者可能有其状态视图，因此我们必须查看每个操作，用其调用和完成事件定义它，并描述操作边界。

思考一致性模型的最简单方式是根据读和写操作及其重叠方式：读操作无副作用，而写改变寄存器状态。

## 一致性模型（Consistency Models）

### 严格一致性（Strict Consistency）

严格一致性等同于完全复制透明：任何进程的任何写对任何进程的后续读立即可用。它涉及全局时钟的概念。不幸的是，这只是理论模型，无法实现。

### 线性化（Linearizability）

**线性化（linearizability）** 是最强的单对象、单操作一致性模型。在此模型下，写的效果在某个时刻（其开始和结束之间）恰好一次对所有读者可见，且没有客户端可以观察到部分或未完成写操作的状态转换或副作用。

**线性化点（linearization point）** 是操作效果变得可见的时刻。一旦操作完成，每个人必须看到它，系统不能「时光倒流」撤销或使其对某些参与者不可见。

### 顺序一致性（Sequential Consistency）

**顺序一致性（sequential consistency）** 允许将操作排序为仿佛以某种顺序执行，同时要求每个进程的操作以进程执行的顺序执行。进程可以以与其自身历史一致的方式观察其他参与者执行的操作，但此视图从全局视角可能是任意陈旧的。

### 因果一致性（Causal Consistency）

在**因果一致性（causal consistency）** 模型下，所有进程必须以相同顺序看到因果相关的操作。无因果关系的并发写可以被不同处理器以不同顺序观察。

**向量时钟（vector clocks）** 用于建立事件之间的偏序，检测和解决事件链之间的分歧。许多数据库（如 Dynamo、Riak）使用向量时钟建立因果顺序。

### 会话模型（Session Models）

**会话模型**从单个客户端视角帮助推理分布式系统的状态。包括：

- **读己之写（read-own-writes）**：每次写后的读必须观察到更新值
- **单调读（monotonic reads）**：若 read(x) 观察到值 V，后续读必须观察到至少与 V 一样新的值
- **单调写（monotonic writes）**：来自同一客户端的写必须以客户端执行的顺序传播
- **写跟随读（writes-follow-reads）**：写必须排在先前读观察到的写之后

### 最终一致性（Eventual Consistency）

在**最终一致性（eventual consistency）** 下，更新异步通过系统传播。形式化地，若对数据项没有额外更新，最终所有访问返回最新写入的值。

### 可调一致性（Tunable Consistency）

最终一致系统通常实现**可调一致性**，使用三个变量复制、读取和写入数据：

- **复制因子 N**：将存储数据副本的节点数
- **写一致性 W**：写成功必须确认的节点数
- **读一致性 R**：读操作必须响应的节点数

选择 $R + W > N$ 时，系统可以保证返回最近写入的值，因为读集和写集总有重叠。

### 法定人数（Quorums）

由 $\lfloor N/2 \rfloor + 1$ 节点组成的一致性级别称为**法定人数（quorum）**。在 $2f + 1$ 节点的系统中，若最多 $f$ 个节点不可用，存活节点可以继续接受写或读。

### 见证副本（Witness Replicas）

**见证副本（witness replicas）** 可以降低存储成本：副本分为复制和见证子集。见证副本在正常操作下仅存储表示写操作发生的记录，在复制副本不足时可临时升级存储记录。

### 强最终一致性与 CRDT

**强最终一致性（strong eventual consistency）** 允许更新延迟或乱序传播，但当所有更新最终传播到目标节点时，可以解决冲突并合并产生相同的有效状态。

**无冲突复制数据类型（CRDT）** 是专门的数据结构，排除冲突存在，允许对这些数据类型的操作以任意顺序应用而不改变结果。最简单的例子是**仅增计数器（grow-only counter）** 和 **LWW 寄存器**。

## 小结

容错系统使用复制提高可用性。我们讨论了多种单操作一致性模型，从保证最多到最少：线性化、顺序一致性、因果一致性、PRAM/FIFO 一致性，以及多种会话模型。我们还讨论了最终一致性和可调一致性的概念。基于法定人数的系统使用多数来提供一致数据。
