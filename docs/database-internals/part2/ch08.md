# 第8章 简介与概览

> 分布式系统是这样一种系统：你甚至不知道其存在的某台计算机的故障，可能导致你自己的计算机无法使用。——Leslie Lamport

分布式系统使电话、转账和远距离信息交换成为可能。我们每天都在使用分布式系统，有时甚至不自知：任何客户端/服务器应用都是分布式系统。

对许多现代软件系统而言，垂直扩展（在更大、更快、更多 CPU、内存或更快磁盘的机器上运行同一软件）并不可行。更大的机器更贵、更难替换，且可能需要专门维护。另一种选择是水平扩展：在多台通过网络连接的机器上运行软件，作为单一逻辑实体工作。

分布式系统在规模和参与者特征上可能差异很大，从少量到数百台机器，从小型手持或传感器设备到高性能计算机。

数据库系统主要运行在单节点上的时代早已过去，大多数现代数据库系统将多节点连接成集群，以增加存储容量、提升性能并增强可用性。

尽管分布式计算中的一些理论突破并不新鲜，但其实际应用大多发生在相对近期。今天，我们看到对该主题的兴趣日益增长，更多研究和新的开发正在进行。

## 基本定义

在分布式系统中，我们有若干参与者（有时称为进程、节点或副本）。每个参与者有自己的本地状态。参与者通过彼此之间的通信链路交换消息进行通信。

进程可以通过时钟访问时间，时钟可以是逻辑的或物理的。逻辑时钟使用某种单调递增的计数器实现。物理时钟（也称挂钟）与物理世界中的时间概念绑定，可通过进程本地手段访问，例如通过操作系统。

分布式系统的固有困难源于其各部分彼此分离。远程进程通过可能缓慢且不可靠的链路通信，这使得了解远程进程的确切状态更加复杂。

分布式系统领域的大多数研究与「没有什么是完全可靠的」这一事实相关：通信信道可能延迟、重排或无法送达消息；进程可能暂停、变慢、崩溃、失控或突然停止响应。

并发编程与分布式编程领域有许多共同主题，因为 CPU 是带有链路、处理器和通信协议的微型分布式系统。你将在「一致性模型」中看到与并发编程的许多相似之处。然而，由于远程方之间通信的成本以及链路和进程的不可靠性，大多数原语无法直接复用。

为克服分布式环境的困难，我们需要使用一类特殊的算法——分布式算法，它们具有本地与远程状态和执行的概念，并在不可靠网络和组件故障下工作。

我们用状态和步骤（或阶段）及其之间的转换来描述算法。每个进程在本地执行算法步骤，本地执行与进程交互的组合构成分布式算法。

分布式算法描述多个独立节点的本地行为和交互。节点通过相互发送消息进行通信。算法定义参与者角色、交换的消息、状态、转换、执行的步骤、传递介质的属性、时序假设、故障模型以及描述进程及其交互的其他特征。

分布式算法服务于多种目的：

- **协调（Coordination）**：监督多个工作进程的行为
- **合作（Cooperation）**：多个参与者相互依赖以完成任务
- **传播（Dissemination）**：进程合作将信息快速可靠地传播给所有相关方
- **共识（Consensus）**：在多个进程间达成一致

本书在算法使用场景下讨论算法，偏好实用方法而非纯学术材料。我们首先覆盖所有必要的抽象、进程及其之间的连接，然后构建更复杂的通信模式。从 UDP（发送方对消息是否到达目的地没有任何保证）开始，最终到共识（多个进程就特定值达成一致）。

## 并发执行

一旦允许两个执行线程访问变量，除非线程间同步步骤，否则并发步骤执行的确切结果是不可预测的。我们不再有单一可能结果，而是得到多种可能，如图 8-1 所示。

a) x = 2  — 两个线程都读到初始值，加法先写，但被乘法结果覆盖
b) x = 3  — 两个线程都读到初始值，乘法先写，但被加法结果覆盖
c) x = 4  — 乘法在加法开始前完成
d) x = 6  — 加法在乘法开始前完成

在跨越单节点边界之前，我们就遇到了分布式系统的第一个问题：**并发**。每个并发程序都具有分布式系统的某些特性。线程访问共享状态，在本地执行操作，并将结果传播回共享变量。

为精确定义执行历史并减少可能结果的数量，我们需要一致性模型。一致性模型描述并发执行，并建立操作可执行和可被参与者可见的顺序。使用不同的一致性模型，我们可以约束或放宽系统可能处于的状态数量。

分布式系统与并发计算领域在术语和研究上有大量重叠，但也有一些差异。在并发系统中，我们可以有共享内存（shared memory），处理器可用来交换信息。在分布式系统中，每个处理器有自己的本地状态，参与者通过传递消息进行通信。

### 并发与并行

我们经常互换使用并发和并行计算这两个术语，但这两个概念略有语义差异。当两个步骤序列并发执行时，两者都在进行中，但任一时刻仅执行其中一个。若两个序列并行执行，其步骤可同时执行。并发操作在时间上重叠，而并行操作由多个处理器执行。

Erlang 语言创造者 Joe Armstrong 举例：并发执行像两台队列共用一台咖啡机，并行执行像两台队列各有一台咖啡机。尽管如此，绝大多数资料用「并发」描述具有多个并行执行线程的系统，「并行」一词很少使用。

## 分布式系统中的共享状态

我们可以尝试在分布式系统中引入某种共享内存概念，例如单一信息源（如数据库）。即使解决了对其的并发访问问题，我们仍无法保证所有进程同步。

要访问该数据库，进程必须通过通信介质发送和接收消息来查询或修改状态。然而，若某个进程长时间未收到数据库的响应会怎样？要回答这个问题，我们首先必须定义「更长」的含义。为此，系统必须用同步性来描述：通信是完全异步的，还是存在某些时序假设。这些时序假设使我们能够引入操作超时和重试。

我们不知道数据库未响应是因为过载、不可用或缓慢，还是因为通往它的网络出现问题。这描述了崩溃的本质：进程可能通过停止参与后续算法步骤而崩溃，可能发生临时故障，或可能遗漏某些消息。在决定如何对待故障之前，我们需要定义故障模型并描述故障发生的方式。

描述系统可靠性以及在故障存在下是否能继续正确运行的属性称为**容错（fault tolerance）**。故障不可避免，因此我们需要用可靠组件构建系统，消除前述单节点数据库形式的单点故障可以是第一步。我们可以通过引入冗余并添加备份数据库来实现。然而，现在我们面临不同的问题：如何保持多个共享状态副本同步？

迄今为止，尝试将共享状态引入我们的简单系统留下了比答案更多的问题。我们现在知道共享状态并不像引入数据库那么简单，必须采用更细粒度的方法，用独立进程和彼此间传递的消息来描述交互。

## 分布式计算的谬误

在理想情况下，两台计算机通过网络通信时一切正常：进程建立连接、发送数据、获得响应，皆大欢喜。假设操作总是成功且不会出错是危险的，因为当某物确实损坏且我们的假设被证明错误时，系统会以难以或无法预测的方式行为。

大多数时候，假设网络可靠是合理的。它必须在某种程度上可靠才有用。我们都遇到过试图建立与远程服务器的连接却得到「网络不可达」错误的情况。但即使可以建立连接，与服务器的成功初始连接也不能保证链路稳定，连接可能随时中断。消息可能已到达对方，但响应可能丢失，或在响应送达前连接已中断。

网络交换机损坏、电缆断开、网络配置可能随时改变。我们应通过优雅处理所有这些场景来构建系统。

连接可以稳定，但我们不能期望远程调用与本地调用一样快。我们应尽可能少地对延迟做假设，永远不要假设**延迟为零**。消息要到达远程服务器，必须经过多个软件层和物理介质（如光纤或电缆）。这些操作都不是瞬时的。

Michael Lewis 在《Flash Boys》一书中讲述了公司花费数百万美元将延迟降低几毫秒以比竞争对手更快访问证券交易所的故事。这是将延迟作为竞争优势的绝佳例子，但值得一提的是，根据其他研究，陈旧报价套利（能够比竞争对手更快了解价格和执行订单而获利的能力）并不一定让快速交易者有能力利用市场。

吸取教训后，我们添加了重试、重连并移除了瞬时执行的假设，但这仍然不够。在增加交换消息的数量、速率和大小，或向现有网络添加新进程时，我们不应假设**带宽是无限的**。

1994 年，Peter Deutsch 发表了一份著名的断言列表，题为「分布式计算的谬误」，描述了分布式计算中容易忽视的方面。除网络可靠性、延迟和带宽假设外，他还描述了其他问题，例如网络安全、敌对方的可能存在、可能破坏我们对特定资源存在和位置的假设的有意和无意的拓扑变化、时间和资源方面的传输成本，以及最后，拥有对整个网络的知识和控制的单一权威的存在。

Deutsch 的分布式计算谬误列表相当详尽，但侧重于通过链路从一个进程向另一个进程发送消息时可能出错的内容。这些关注点有效且描述了最通用和低级的复杂性，但不幸的是，我们在设计和实现分布式系统时做出的许多其他假设在运行时可能导致问题。

### 处理

远程进程在发送对刚收到消息的响应之前，需要在本地执行一些工作，因此我们不能假设**处理是瞬时的**。考虑网络延迟还不够，因为远程进程执行的操作也不是即时的。

此外，无法保证消息一送达处理就开始。消息可能落在远程服务器的待处理队列中，必须等待直到其之前到达的所有消息被处理完毕。

节点可能彼此距离远近不同，拥有不同的 CPU、内存量、不同磁盘，或运行不同版本的软件和配置。我们不能期望它们以相同速率处理请求。若我们必须等待多个并行工作的远程服务器响应才能完成任务，整体执行速度取决于最慢的远程服务器。

与普遍看法相反，队列容量不是无限的，堆积更多请求对系统没有任何好处。**背压（backpressure）** 是一种策略，通过放慢生产者来应对以快于消费者处理速率发布消息的生产者。背压是分布式系统中最不被重视和应用的概念之一，往往事后构建而非系统设计的组成部分。

### 时钟与时间

> 时间是幻觉。午餐时间 doubly so。——Ford Prefect，《银河系漫游指南》

假设远程机器上的时钟同步运行也可能是危险的。与「延迟为零」和「处理瞬时」结合，会导致各种异常，尤其在时间序列和实时数据处理中。例如，在从具有不同时间感知的参与者收集和聚合数据时，应理解它们之间的时间漂移并相应归一化时间，而非依赖源时间戳。除非使用专门的高精度时间源，否则不应依赖时间戳进行同步或排序。当然，这并不意味着我们完全不能或不应依赖时间：毕竟，任何同步系统都使用本地时钟进行超时。

始终考虑进程之间可能的时间差异以及消息传递和处理所需的时间至关重要。例如，Spanner 使用特殊的 time API，返回时间戳和不确定性边界以施加严格的事务顺序。某些故障检测算法依赖共享的时间概念和时钟漂移始终在正确性允许范围内的保证。

除了分布式系统中时钟同步困难之外，当前时间 constantly 在变化：你可以从操作系统请求当前 POSIX 时间戳，执行若干步骤后再请求另一个当前时间戳，两者将不同。这是相当明显的观察，但理解时间源以及时间戳捕获的确切时刻至关重要。

理解时钟源是否单调（即它永远不会倒退）以及计划的时间相关操作可能漂移多少也会有所帮助。

### 状态一致性

大多数前述假设属于「几乎总是错误」类别，但有些更好地描述为「并非总是正确」：当容易走心理捷径并通过以特定方式思考来简化模型时，忽略一些棘手的边缘情况。

分布式算法并不总是保证严格的状态一致性。某些方法具有更宽松的约束，允许副本之间的状态分歧，并依赖冲突解决（在系统内检测和解决分歧状态的能力）和读时数据修复（在副本响应不同结果时在读取期间使副本恢复同步）。假设节点间的状态完全一致可能导致细微错误。

最终一致的分布式数据库系统可能有逻辑通过读取时查询法定节点数来处理副本分歧，但假设数据库模式和集群视图是强一致的。除非我们强制执行此信息的一致性，否则依赖该假设可能产生严重后果。

### 本地与远程执行

将复杂性隐藏在 API 后面可能是危险的。例如，若你对本地数据集有迭代器，即使存储引擎不熟悉，你也可以合理预测幕后发生的事。理解远程数据集迭代的过程则完全是另一个问题：你需要理解一致性和传递语义、数据协调、分页、合并、并发访问影响以及许多其他事项。

简单地将两者隐藏在相同接口后面，无论多么有用，都可能产生误导。可能需要额外的 API 参数用于调试、配置和可观测性。我们应始终牢记本地和远程执行并不相同。

隐藏远程调用的最明显问题是延迟：远程调用比本地调用成本高得多，因为它涉及双向网络传输、序列化/反序列化以及许多其他步骤。交织本地和阻塞远程调用可能导致性能下降和意外副作用。

### 需要处理故障

假设所有节点都正常运行来开始工作是可以的，但认为情况一直如此是危险的。在长期运行的系统中，节点可能因维护而下线（通常涉及优雅关闭）或因各种原因崩溃：软件问题、OOM killer、运行时 bug、硬件问题等。进程确实会故障，你能做的最好的事就是为故障做好准备并理解如何应对。

若远程服务器未响应，我们并不总是知道确切原因。可能是崩溃、网络故障、远程进程或通往它的链路缓慢。某些分布式算法使用心跳协议和故障检测器来形成关于哪些参与者存活且可达的假设。

### 网络分区与部分故障

当两台或更多服务器无法相互通信时，我们称这种情况为**网络分区**。在「CAP 定理的视角」中，Seth Gilbert 和 Nancy Lynch 区分了两种情况：两个参与者无法相互通信，以及若干参与者组彼此隔离、无法交换消息并继续执行算法。

网络的一般不可靠性（丢包、重传、难以预测的延迟）令人烦恼但可容忍，而网络分区可能造成更多麻烦，因为独立组可以继续执行并产生冲突结果。网络链路也可能不对称故障：消息仍可能从一个进程传递到另一个，但反之则不行。

要构建在一个或多个进程故障存在下健壮的系统，我们必须考虑部分故障的情况以及系统在部分不可用或功能异常时如何继续运行。

故障难以检测，且从系统不同部分看并不总是以相同方式可见。在设计高可用系统时，应始终考虑边缘情况：若我们确实复制了数据但未收到确认会怎样？需要重试吗？在已发送确认的节点上，数据是否仍可用于读取？

墨菲定律告诉我们故障确实会发生。编程 folklore 补充说故障会以最糟糕的方式发生，因此我们作为分布式系统工程师的工作是确保减少出错场景的数量，并以能够限制其可能造成损害的方式为故障做好准备。

不可能防止所有故障，但我们仍可以构建在故障存在下正确运行的弹性系统。为故障设计的最佳方式是测试它们。几乎不可能穷尽思考每种可能的故障场景并预测多个进程的行为。设置创建分区、模拟位腐、增加延迟、使时钟分歧、放大相对处理速度的测试框架是最好的方法。

### 级联故障

我们不能总是完全隔离故障：高负载下倾倒的进程会增加集群其余部分的负载，使其他节点更可能故障。级联故障可以从系统的一部分传播到另一部分，扩大问题范围。

有时，级联故障甚至可能由完全良好的意图引发。例如，某节点离线一段时间未收到最新更新。当它重新上线后，热心的对等方会想帮助它赶上最近发生的事，开始向其流式传输缺失的数据，耗尽网络资源或导致节点在启动后不久故障。

为保护系统免受故障传播并优雅处理故障场景，可以使用**断路器（circuit breaker）**。在电气工程中，断路器通过中断电流来保护昂贵且难以更换的部件免受过载或短路。在软件开发中，断路器监控故障并允许回退机制，通过避开故障服务、给它一些恢复时间并优雅处理故障调用来保护系统。

当与某服务器的连接失败或服务器未响应时，客户端启动重连循环。到那时，过载的服务器已经难以跟上新的连接请求，客户端在紧密循环中的重试对情况没有帮助。为避免这种情况，我们可以使用**退避（backoff）**策略。客户端不是立即重试，而是等待一段时间。退避可以帮助我们通过安排重试和增加后续请求之间的时间窗口来避免放大问题。

退避用于增加单个客户端请求之间的时间周期。然而，使用相同退避策略的不同客户端也可能产生大量负载。为防止不同客户端在退避期后同时重试，我们可以引入**抖动（jitter）**。抖动向退避添加小的随机时间周期，降低客户端同时唤醒和重试的概率。

硬件故障、位腐和软件错误可能导致损坏，并通过标准传递机制传播。例如，若未验证，损坏的数据记录可能被复制到其他节点。若没有验证机制，系统可能将损坏的数据传播到其他节点，可能覆盖未损坏的数据记录。为避免这种情况，我们应使用校验和和验证来验证节点之间交换的任何内容的完整性。

可以通过规划和协调执行来避免过载和热点。不让对等方独立执行操作步骤，我们可以使用协调器根据可用资源准备执行计划，并根据其可用的过去执行数据预测负载。

## 分布式系统抽象

讨论编程语言时，我们使用通用术语，用函数、运算符、类、变量和指针来定义程序。拥有共同词汇帮助我们避免每次描述任何事物时发明新词。定义越精确、越少歧义，听众就越容易理解我们。

在进入算法之前，我们首先必须覆盖分布式系统词汇：你在演讲、书籍和论文中经常遇到的定义。

### 链路（Links）

网络不可靠：消息可能丢失、延迟和重排。现在，带着这一想法，我们将尝试构建几种通信协议。

我们从最不可靠和健壮的开始，识别它们可能处于的状态，并找出可以为协议提供更好保证的可能补充。

**公平丢失链路（fair-loss link）**

我们可以从两个通过链路连接的进程开始。进程可以相互发送消息，如图 8-2 所示。任何通信介质都不完美，消息可能丢失或延迟。

```
进程 A ---- M ----> 进程 B
```

::: tip 图 8-2
最简单、不可靠的通信形式
:::

消息 M 发送后，从发送方视角，它可能处于以下状态之一：

- 尚未交付给进程 B（但将在某个时间点交付）
- 在传输中不可恢复地丢失
- 成功交付给远程进程

注意发送方无法得知消息是否已交付。在分布式系统术语中，这种链路称为公平丢失。这种链路的属性是：

- **公平丢失**：若发送方和接收方都正确且发送方无限次重传消息，它最终将被交付
- **有限重复**：发送的消息不会无限次交付
- **无创造**：链路不会凭空产生消息；即不会交付从未发送的消息

公平丢失链路是有用抽象，是构建具有强保证的通信协议的第一块基石。我们可以假设该链路不会系统性地丢失通信方之间的消息，也不会创建新消息。但同时，我们不能完全依赖它。这可能让你想起用户数据报协议（UDP），它允许我们从一个进程向另一个发送消息，但在协议层面没有可靠传递语义。

**消息确认（Message acknowledgments）**

为改善情况并在消息状态方面获得更多清晰度，我们可以引入确认：接收方通知发送方已收到消息的方式。为此，我们需要使用双向通信信道，并添加允许我们区分消息差异的手段，例如序列号——唯一的单调递增消息标识符。

每条消息有唯一标识符即可。序列号只是唯一标识符的特例，我们通过从计数器抽取标识符来实现唯一性。使用哈希算法唯一标识消息时，应考虑可能的碰撞并确保我们仍能消除歧义。

现在，进程 A 可以发送消息 M(n)，其中 n 是单调递增的消息计数器。B 一收到消息，就向 A 发送确认 ACK(n)。图 8-3 显示了这种通信形式。

**消息重传（Message retransmits）**

添加确认仍不足以称此通信协议为可靠的：发送的消息可能仍会丢失，或远程进程可能在确认前故障。为解决此问题并提供传递保证，我们可以尝试重传。重传是发送方重试可能失败操作的方式。我们说「可能失败」，因为发送方并不真正知道是否失败，因为我们将讨论的链路类型不使用确认。

进程 A 发送消息 M 后，等待直到超时 T 触发，然后再次尝试发送相同消息。假设进程之间的链路保持完好、进程之间的网络分区不是无限的、并非所有数据包都丢失，我们可以说，从发送方视角，消息要么尚未交付给进程 B，要么已成功交付给进程 B。由于 A 不断尝试发送消息，我们可以说它在传输中不能不可恢复地丢失。

在分布式系统术语中，这种抽象称为**顽固链路（stubborn link）**。之所以叫顽固，是因为发送方无限次地不断重发消息，但由于这种抽象会非常不实用，我们需要将重试与确认结合。

**重传的问题**

每当我们发送消息时，在收到远程进程的确认之前，我们不知道它是否已被处理、即将被处理、已丢失，或远程进程在接收前已崩溃——这些状态中的任何一种都可能。我们可以重试操作并再次发送消息，但这可能导致消息重复。仅当我们要执行的操作是**幂等的（idempotent）**时，处理重复才是安全的。

幂等操作是可以执行多次、产生相同结果且不产生额外副作用的操作。例如，服务器关闭操作可以是幂等的，第一次调用启动关闭，所有后续调用不产生任何额外效果。

若每个操作都是幂等的，我们可以少考虑传递语义，更多地依赖重传实现容错，并以完全反应式的方式构建系统：响应某些信号触发动作，而不引起意外副作用。然而，操作不一定是幂等的，仅仅假设它们是幂等的可能导致集群范围的副作用。例如，向客户信用卡收费不是幂等的，多次收费绝对不可取。

**消息顺序（Message order）**

不可靠网络给我们带来两个问题：消息可能乱序到达，且由于重传，某些消息可能多次到达。我们已经引入了序列号，可以在接收方使用这些消息标识符来确保先进先出（FIFO）顺序。由于每条消息都有序列号，接收方可以跟踪：

- `nconsecutive`：指定它已看到所有消息的最高序列号。直到此编号的消息可以按序放回。
- `nprocessed`：指定消息已按原始顺序放回并处理的最高序列号。此编号可用于去重。

若收到的消息具有非连续序列号，接收方将其放入重排序缓冲区。例如，它在收到序列号 3 的消息后收到序列号 5 的消息，我们知道 4 仍然缺失，因此需要将 5 搁置直到 4 到来，然后我们可以重建消息顺序。由于我们构建在公平丢失链路之上，我们假设 nconsecutive 和 nmax_seen 之间的消息最终将被交付。

去重通过检查序列号为 n 的消息是否已被处理（由接收方传递到下层）并丢弃已处理的消息来工作。

在分布式系统术语中，这种链路类型称为**完美链路（perfect link）**，提供以下保证：

- **可靠传递**：正确进程 A 向正确进程 B 发送一次的每条消息，最终将被交付
- **无重复**：没有消息被交付超过一次
- **无创造**：与其他链路类型相同，它只能交付实际发送的消息

这可能让你想起 TCP 协议（然而，TCP 中的可靠传递仅在单个会话范围内保证）。

**恰好一次传递（Exactly-once delivery）**

> 分布式系统中只有两个难题：1. 保证消息顺序 2. 恰好一次传递。——Mathias Verraes

关于恰好一次传递是否可能已有许多讨论。在此，语义和精确措辞至关重要。由于可能存在阻止消息首次尝试传递的链路故障，大多数实际系统采用**至少一次传递（at-least-once delivery）**，确保发送方重试直到收到确认，否则消息不被视为已接收。另一种传递语义是**至多一次（at-most-once）**：发送方发送消息且不期望任何传递确认。

TCP 协议通过将消息分解为数据包、逐个传输并在接收端拼接它们来工作。TCP 可能尝试重传某些数据包，多次传输尝试可能成功。由于 TCP 用序列号标记每个数据包，即使某些数据包被传输了多次，它也可以去重数据包并保证接收方将看到消息且仅处理一次。在 TCP 中，此保证仅在单个会话内有效：若消息被确认和处理，但发送方在连接中断前未收到确认，应用程序不知道此传递，根据其逻辑，它可能再次尝试发送消息。

这意味着**恰好一次处理（exactly-once processing）**才是我们感兴趣的，因为重复传递（或数据包传输）没有副作用，仅仅是链路尽力而为的产物。例如，若数据库节点仅收到了记录但尚未持久化，传递已发生，但除非可以检索记录（即除非它既被传递又被处理），否则将毫无用处。

要使恰好一次保证成立，节点应具有共同知识：每个人都知道某个事实，且每个人都知道其他每个人也知道该事实。简而言之，节点必须就记录状态达成一致：两个节点都同意它要么已持久化要么未持久化。正如你将在本章后面看到的，这在理论上是不可能的，但在实践中我们仍通过放宽协调要求来使用这一概念。

## 两将军问题（Two Generals' Problem）

分布式系统中关于一致性的最著名描述之一是一个广为人知的思想实验——两将军问题。

该思想实验表明，若通信是异步的且存在链路故障，则无法在两方之间达成一致。尽管 TCP 表现出完美链路的属性，但重要的是记住完美链路尽管名称如此，并不保证完美传递。它们也不能保证参与者将一直存活，且只关心传输。

想象两支军队，由两位将军率领，准备进攻一座设防城市。军队位于城市两侧，只有同时进攻才能成功围城。

将军们可以通过派遣信使通信，并已制定进攻计划。他们现在唯一需要达成一致的是是否执行该计划。此问题的变体包括：其中一位将军军衔更高但需要确保进攻协调；或将军们需要就确切时间达成一致。这些细节不改变问题定义：将军们必须达成一致。

军队将军们只需就双方都将进行进攻这一事实达成一致。否则，进攻无法成功。将军 A 发送消息 MSG(N)，声明若对方同意则将在指定时间进行进攻的意图。

A 派遣信使后，他不知道信使是否到达：信使可能被俘未能传递消息。当将军 B 收到消息时，他必须发送确认 ACK(MSG(N))。携带此确认的信使也可能被俘或未能传递。B 无法得知信使是否成功传递了确认。

为确定起见，B 必须等待 ACK(ACK(MSG(N)))，即声明 A 收到对确认的确认的二阶确认。

无论将军们相互发送多少进一步确认，他们总是离知道是否可以安全进行进攻差一个 ACK。将军们注定要怀疑携带这最后一个确认的消息是否已到达目的地。

注意我们未做任何时序假设：将军之间的通信是完全异步的。没有设定将军响应所需时间的上限。

## FLP 不可能性（FLP Impossibility）

在 Fisher、Lynch 和 Paterson 的论文中，作者描述了一个著名的问题——FLP 不可能性问题，其中他们讨论了一种共识形式：进程以初始值开始并尝试就新值达成一致。算法完成后，此新值必须对所有非故障进程相同。

若网络完全可靠，就特定值达成一致是直接的；但在现实中，系统容易受到多种故障的影响，如消息丢失、重复、网络分区以及缓慢或崩溃的进程。

共识协议描述这样一个系统：给定多个以其初始状态开始的进程，将所有进程带入决策状态。为使共识协议正确，它必须保持三个属性：

- **一致性（Agreement）**：协议达成的决策必须一致：每个进程决定某个值，这对所有进程必须相同。否则，我们未达成共识。
- **有效性（Validity）**：一致的值必须由参与者之一提出，这意味着系统不应「凭空」产生值。这也意味着值的非平凡性：进程不应总是决定某个预定义的默认值。
- **终止性（Termination）**：仅当没有进程未到达决策状态时，一致才是最终的。

该论文假设处理完全异步；进程之间没有共享的时间概念。此类系统中的算法不能基于超时，进程无法找出其他进程是崩溃了还是运行太慢。论文表明，给定这些假设，不存在可以在有界时间内保证共识的协议。没有完全异步的共识算法能够容忍甚至单个远程进程的未宣布崩溃。

若不考虑进程完成算法步骤的上界时间，进程故障无法可靠检测，且没有确定性算法可以达成共识。

然而，FLP 不可能性并不意味着我们必须收拾东西回家，因为达成共识并非不可能。它仅意味着我们不能总是在有界时间内在异步系统中达成共识。在实践中，系统表现出至少某种程度的同步性，解决此问题需要更精细的模型。

## 系统同步性（System Synchrony）

从 FLP 不可能性可以看出，时序假设是分布式系统的关键特征之一。在异步系统中，我们不知道进程的相对速度，无法保证在有界时间或特定顺序内传递消息。进程可能无限长时间响应，进程故障不能总是可靠检测。

对异步系统的主要批评是这些假设不现实：进程不能有任意不同的处理速度，链路不会无限长时间传递消息。依赖时间既简化推理又有助于提供上界时序保证。

在异步模型中并不总是能解决共识问题。此外，设计高效的同步算法并不总是可实现的，对于某些任务，实际解决方案更可能依赖于时间。

这些假设可以放宽，系统可以被认为是同步的。为此，我们引入时序概念。在同步模型下推理系统要容易得多。它假设进程以可比较的速率进展，传输延迟有界，消息传递不能无限长。

同步系统也可以用同步的进程本地时钟来表示：两个进程本地时间源之间存在上界时间差。

在同步模型下设计系统允许我们使用超时。我们可以在其上构建更复杂的抽象，如领导者选举、共识、故障检测等。这使最佳情况场景更健壮，但若时序假设不成立则导致故障。

异步和同步模型的属性可以结合，我们可以将系统视为**部分同步**的。部分同步系统表现出同步系统的某些属性，但消息传递、时钟漂移和相对处理速度的界限可能不精确，且仅大多数时间成立。

## 故障模型（Failure Models）

我们不断提到故障，但迄今为止它是一个相当宽泛和通用的概念，可能涵盖许多含义。与我们可以做不同的时序假设类似，我们可以假设存在不同类型的故障。故障模型精确描述进程在分布式系统中如何崩溃，算法使用这些假设开发。

我们将讨论分布式系统中存在的多种故障模型，如崩溃、遗漏和任意故障。

### 崩溃故障（Crash Faults）

通常，我们期望进程正确执行算法的所有步骤。进程崩溃的最简单方式是停止执行算法所需的任何进一步步骤，不向其他进程发送任何消息。换句话说，进程崩溃了。大多数时候，我们假设**崩溃-停止（crash-stop）**进程抽象，规定一旦进程崩溃，它保持在此状态。

**崩溃-恢复（crash-recovery）**是不同的进程抽象，进程停止执行算法所需的步骤，但在稍后恢复并尝试执行进一步步骤。恢复的可能性需要在系统中引入持久状态和恢复协议。

### 遗漏故障（Omission Faults）

另一种故障模型是遗漏故障。该模型假设进程跳过某些算法步骤，或无法执行它们，或此执行对其他参与者不可见，或无法向其他参与者发送或接收消息。遗漏故障捕获由故障网络链路、交换机故障或网络拥塞引起的进程之间的网络分区。

### 任意故障（Arbitrary Faults）

最难克服的故障类别是任意或**拜占庭故障（Byzantine faults）**：进程继续执行算法步骤，但以与算法相矛盾的方式（例如，若共识算法中的进程决定了一个其他参与者从未提出过的值）。

你可能听说过航空航天业的拜占庭容错：飞机和航天器系统不会轻信子组件的响应，而是交叉验证其结果。另一个广泛应用是加密货币，其中没有中央权威，不同方控制节点，敌对参与者有物质激励伪造值并试图通过提供错误响应来博弈系统。

## 处理故障（Handling Failures）

我们可以通过形成进程组并在算法中引入冗余来掩盖故障：即使其中一个进程故障，用户也不会注意到此故障。

故障可能有一些性能惩罚：正常执行依赖进程响应，系统必须回退到较慢的执行路径进行错误处理和纠正。许多故障可以在软件层面通过代码审查、广泛测试、通过引入超时和重试确保消息传递、确保步骤在本地按序执行来预防。

我们将在此覆盖的大多数算法假设崩溃故障模型，并通过引入冗余来应对故障。这些假设有助于创建性能更好、更易于理解和实现的算法。

## 小结

本章讨论了分布式系统的一些术语并介绍了基本概念。我们谈到了由系统组件的不可靠性引起的固有困难和复杂性：链路可能无法传递消息，进程可能崩溃，网络可能分区。

这些术语应足以让我们继续讨论。本书其余部分讨论分布式系统中常用的解决方案：我们回顾可能出错的地方，看看我们有哪些可用选项。
