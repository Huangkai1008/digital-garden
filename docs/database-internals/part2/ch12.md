# 第12章 反熵与传播

迄今为止讨论的大多数通信模式要么是点对点的，要么是一对多的（协调器与副本）。要可靠地在整个系统中传播数据记录，传播节点必须可用且能够到达其他节点，但即使如此，吞吐量也限于单台机器。

快速可靠的传播可能对数据记录不太适用，但对集群范围的元数据（如成员信息、节点状态、故障、模式变更等）更重要。包含此信息的消息通常不频繁且较小，但必须尽可能快速可靠地传播。

此类更新通常可以使用三大类方法之一传播到集群中的所有节点（图 12-1）：

- a) 从一个进程到所有其他进程的通知广播
- b) 周期性点对点信息交换。对等方成对连接并交换消息
- c) 协作广播，消息接收者成为广播者并帮助更快更可靠地传播信息

::: tip 图 12-1
广播（a）、反熵（b）和 gossip（c）
:::

```
a) 广播:     C -----> P1, P2, P3, P4
b) 反熵:     P1<->P2, P2<->P3, P3<->P4 ...
c) Gossip:   P1->P2->P3, P1->P4, P2->P4 ...
```

**熵（entropy）** 表示系统中无序程度的属性。在分布式系统中，熵表示节点之间的状态分歧程度。**反熵（anti-entropy）** 通常用于在主传递机制失败时使节点恢复最新。系统可以在协调器某点失败时继续正确运行，因为其他节点将继续传播信息。

反熵触发后台或前台进程来比较和协调缺失或冲突的记录。后台反熵使用 Merkle 树和更新日志等辅助结构识别分歧。前台反熵搭载在读或写请求上：hinted handoff、read repair 等。

## 读修复（Read Repair）

在读取时最容易检测副本之间的分歧，因为此时我们可以联系副本，从每个副本请求查询的状态，并查看它们的响应是否匹配。

**读修复（read repair）** 机制：协调器节点向副本发出请求，等待响应并比较。若某些副本错过了最近更新且响应不同，协调器检测不一致并向副本发送更新。

读修复可以实现为阻塞或异步操作。阻塞读修复确保读单调性，但牺牲可用性。

## 摘要读（Digest Reads）

协调器可以只发出一个完整读请求，向其他副本仅发送摘要请求。摘要请求读取副本本地数据并计算响应的哈希，而非返回完整快照。若所有摘要匹配，协调器可以确信副本同步。若摘要不匹配，协调器必须发出完整读以找出哪些副本落后。

##  hinted handoff

**hinted handoff** 是一种写端修复机制。若目标节点未能确认写，写协调器或副本之一存储称为 **hint** 的特殊记录，在目标节点恢复时重放给目标节点。

在 Apache Cassandra 中，除非使用 ANY 一致性级别，hinted 写不计入复制因子，因为 hint 日志中的数据不可用于读取，仅用于帮助落后参与者赶上。

**松散法定人数（sloppy quorum）** 与 hinted handoff 一起使用：在副本故障情况下，写操作可以使用节点列表中的额外健康节点，这些节点不必是执行操作的目标副本。

## Merkle 树（Merkle Trees）

由于读修复只能修复当前查询数据的不一致，我们应使用不同机制查找和修复未主动查询的数据中的不一致。

**Merkle 树**构成本地数据的紧凑哈希表示，构建哈希树。此哈希树的最底层通过扫描整个保存数据记录的表并计算记录范围的哈希来构建。更高层包含较低层哈希的哈希，构建层次表示，使我们能够通过比较哈希快速检测不一致，沿哈希树节点递归缩小不一致范围。

## 位图版本向量（Bitmap Version Vectors）

**位图版本向量**可用于基于新近度解决数据冲突：每个节点维护本地发生或复制的操作的每对等方日志。反熵期间比较日志，缺失数据复制到目标节点。

每个由节点协调的写由点 (i,n) 表示：由节点 n 协调的具有节点本地序列号 i 的事件。节点逻辑时钟中，由节点本身协调的事件无间隙。若某些写未从其他节点复制，时钟将包含间隙。

## Gossip 传播（Gossip Dissemination）

**Gossip 协议**是基于谣言在人类社会中传播或疾病在人群中传播方式的概率通信过程。主要目标是使用协作传播将信息从一个进程传播到集群的其余部分。

持有需要传播记录的进程称为**感染性（infective）**。尚未收到更新的任何进程称为**易感（susceptible）**。感染性进程在活跃传播一段时间后不愿传播新状态的称为**移除（removed）**。

### Gossip 机制

进程定期随机选择 f 个对等方（f 是称为 **fanout** 的可配置参数）并与它们交换当前「热门」信息。系统达到收敛所需的时间称为**延迟（latency）**。

### 覆盖网络（Overlay Networks）

在 gossip 系统中构建临时固定拓扑可以实现：节点可以对对等方采样，根据邻近度（通常由延迟衡量）选择最佳联系点。节点可以形成**生成树（spanning tree）**：无向、无环、边不同的图，覆盖整个网络。

### 混合 Gossip（Hybrid Gossip）

**Plumtree**（push/lazy-push 多播树）在流行病和基于树的广播原语之间权衡。正常条件下，节点仅向对等采样服务提供的小子集发送完整消息。对其余节点，它懒惰地仅转发消息 ID。若节点收到从未见过的消息标识符，它可以查询对等方获取它。

### 部分视图（Partial Views）

向所有已知对等方广播消息并维护集群的完整视图可能代价高昂且不实用。**对等采样服务**维护集群的**部分视图**，使用 gossip 定期刷新。例如 **HyParView** 协议维护小的活跃视图和较大的被动视图。

## 小结

最终一致系统允许副本状态分歧。可调一致性允许我们权衡一致性与可用性。副本分歧可以使用以下反熵机制之一解决：

| 机制 | 说明 |
|------|------|
| **hinted handoff** | 目标宕机时临时在相邻节点存储写，目标恢复时重放 |
| **read repair** | 读取时通过比较响应协调请求的数据范围，检测缺失记录并发送给落后副本 |
| **Merkle 树** | 通过计算和交换层次哈希树检测需要修复的数据范围 |
| **位图版本向量** | 通过维护包含最近写信息的紧凑记录检测缺失副本写 |

这些反熵方法针对三个参数之一进行优化：范围缩减、新近度或完整性。Gossip 协议可用于在大规模系统中可靠地分发信息。
