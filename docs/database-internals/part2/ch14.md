# 第14章 共识

我们已讨论分布式系统中的诸多概念，从链路和进程、分布式计算问题等基础开始；经过故障模型、故障检测器和领导者选举；讨论了一致性模型；最终准备将所有内容整合到分布式系统研究的顶峰：**分布式共识（distributed consensus）**。

分布式系统中的**共识算法（consensus algorithms）** 允许多个进程就某个值达成一致。FLP 不可能性表明，在完全异步系统中无法在有界时间内保证共识。即使消息传递有保证，一个进程也无法知道另一个是崩溃了还是运行缓慢。

共识算法假设异步模型并保证安全性，而外部故障检测器可以提供关于其他进程的信息，保证活性。进程必须就参与者之一提出的某个值达成一致，即使其中一些碰巧崩溃。**正确（correct）** 的进程是指未崩溃并继续执行算法步骤的进程。

从理论角度，共识算法具有三个属性：

- **一致性（Agreement）**：决策值对所有正确进程相同
- **有效性（Validity）**：决定的值由某个进程提出
- **终止性（Termination）**：所有正确进程最终达到决策

## 广播（Broadcast）

**广播（broadcast）** 是分布式系统中常用的通信抽象。最简单的广播方式是通过**尽力广播（best effort broadcast）**：发送方负责确保消息传递到所有目标。若其失败，其他参与者不尝试重播消息。

为实现**可靠广播（reliable broadcast）**，可以使用故障检测器和回退机制。最简单的回退是允许每个收到消息的进程将其转发给其知道的每个其他进程。当源进程故障时，其他进程检测故障并继续广播消息，有效地用 $N^2$ 条消息淹没网络。

## 原子广播（Atomic Broadcast）

**原子广播（atomic broadcast）**（也称全序多播）保证可靠传递和全序。原子广播必须确保两个基本属性：

- **原子性**：进程必须就收到的消息集达成一致。要么所有非故障进程交付消息，要么都不交付
- **顺序**：所有非故障进程以相同顺序交付消息

## 虚拟同步（Virtual Synchrony）

**虚拟同步** 是使用广播的组通信框架。原子广播向静态进程组交付全序消息，虚拟同步向动态对等组交付全序消息。进程被组织成组，只要组存在，消息就以相同顺序交付给所有成员。

## Zookeeper 原子广播（ZAB）

**ZAB** 是 Apache Zookeeper 使用的原子广播实现。进程可以是 **leader** 或 **follower**。协议时间线分为 **epoch**，每个 epoch 只能有一个领导者。

ZAB 协议分三阶段执行：

1. **发现（Discovery）**：准领导者了解每个其他进程已知的最新 epoch，提出比任何 follower 当前 epoch 更大的新 epoch
2. **同步（Synchronization）**：从先前领导者故障恢复并使落后 follower 赶上。准领导者向 follower 发送消息，提议自己为新 epoch 的领导者并收集确认
3. **广播（Broadcast）**：follower 恢复同步后，leader 接收客户端消息，建立顺序并广播给 follower

## Paxos

**Paxos** 可能是最广为人知的共识算法，由 Leslie Lamport 在「The Part-Time Parliament」论文中首次提出。参与者可以担任三种角色之一：**proposer**、**acceptor** 或 **learner**。

每个提议由客户端提出的值和唯一的单调递增提议号组成。Paxos 算法分两阶段：**投票（propose）** 和 **复制（replication）**。在投票阶段，proposer 竞争建立领导地位。在复制阶段，proposer 将值分发给 acceptor。

### Paxos 中的法定人数

**法定人数（quorum）** 是操作执行所需的最小票数，通常构成参与者多数。一旦足够多的参与者接受提议，该值保证被协议接受，因为任意两个多数至少有一个参与者共同。

Paxos 在任意数量故障存在下保证安全性。活性在 $f$ 个进程故障存在下保证，协议需要总共 $2f + 1$ 个进程。

### 故障场景

Proposer 在第二阶段向所有 acceptor 广播值之前故障时，新 proposer 可以接手并提交该值。两个或多个 proposer 竞争时，可能都无法收集多数，导致活锁。通常通过随机退避解决。

### Multi-Paxos

**Multi-Paxos** 引入领导者概念：** distinguished proposer**。拥有已建立的领导者，可以跳过 propose 阶段直接进行复制。某些 Multi-Paxos 实现使用**租约（lease）** 使领导者能够在不收集法定人数的情况下从活跃领导者读取。

### Fast Paxos

**Fast Paxos** 通过让任何 proposer 直接联系 acceptor 而非通过领导者，可将往返次数减少一。为此需要将法定人数大小增加到 $2f + 1$，总 acceptor 数到 $3f + 1$。Fast Paxos 容易发生碰撞，若两个或多个 proposer 尝试使用快速步骤且 acceptor 收到不同值。

### 平等 Paxos（EPaxos）

**EPaxos** 使用领导者负责特定命令的提交，通过查找和设置依赖关系建立顺序。EPaxos 以 **Pre-Accept** 阶段开始，每个提议必须包含：**依赖（dependencies）** 和**序列号**。有快速路径（依赖匹配时）和慢路径（副本有分歧时）。

### 灵活 Paxos（Flexible Paxos）

**灵活 Paxos** 放宽法定人数要求：仅需第一阶段（选举领导者）的节点组与第二阶段（参与接受提议）的节点组重叠。若参与者总数为 $N$，propose 阶段成功所需节点数为 $Q_1$，accept 阶段成功所需为 $Q_2$，只需确保 $Q_1 + Q_2 > N$。

## Raft

**Raft** 于 2013 年出现，旨在创建易于理解和实现的算法。参与者可以担任三种角色之一：**Candidate**、**Leader** 或 **Follower**。

每个进程开始时作为 follower。时间被分为 **term**（epoch），期间领导者唯一且稳定。领导者选举时，Candidate 发送 RequestVote 消息。协议使用心跳机制确保参与者活性。领导者通过发送 AppendEntries 消息重复向复制日志追加新值。

Raft 保证：给定 term 同时最多只能选出一个领导者；领导者不删除或重排序其日志内容；已提交的日志条目保证存在于后续领导者的日志中。Raft 用于 CockroachDB、Etcd、Consul 等。

## 拜占庭共识（Byzantine Consensus）

迄今为止讨论的所有共识算法假设非拜占庭故障。**拜占庭共识** 算法需要确保系统在部分节点行为异常甚至恶意时仍能正确运行。

**实用拜占庭容错（PBFT）** 假设独立节点故障。为保证安全性和活性，故障副本不能超过 $(n-1)/3$ 个。系统要承受 $f$ 个受损节点，需要至少 $n = 3f + 1$ 个节点。

PBFT 使用 **view** 区分集群配置。每个 view 中，一个副本是 **primary**，其余是 **backup**。协议执行分三阶段：

1. **Pre-prepare**：primary 广播包含 view ID、唯一单调递增标识符、负载和负载摘要的消息
2. **Prepare**：backup 接受 pre-prepare 后，向所有其他副本广播 Prepare 消息
3. **Commit**：backup 向所有其他副本广播 Commit 消息，等待收集 $2f + 1$ 个匹配的 Commit

**恢复与检查点**：副本将接受的消息保存在稳定日志中。每 $N$ 个请求后，primary 制作稳定检查点。

## 小结

共识算法是分布式系统中最有趣 yet 最复杂的主题之一。本章讨论了经典 Paxos 算法及其多种变体：Multi-Paxos、Fast Paxos、EPaxos、Flexible Paxos。Raft 简化了共识的描述术语，使领导成为算法中的一等公民。为保证对抗环境中的共识安全性，应使用拜占庭容错算法，如 PBFT。
