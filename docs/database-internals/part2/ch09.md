# 第9章 故障检测

> 若森林中一棵树倒下而无人听见，它是否发出声音？——佚名

为使系统适当响应故障，故障应及时被检测到。故障进程可能仍被联系，尽管它无法响应，从而增加延迟并降低整体系统可用性。

在异步分布式系统（即不做任何时序假设）中检测故障极其困难，因为无法区分进程是崩溃了，还是运行缓慢并需要无限长时间响应。我们在「FLP 不可能性」中讨论了与此相关的问题。

诸如 dead、failed、crashed 等术语通常用于描述已完全停止执行其步骤的进程。诸如 unresponsive、faulty、slow 等术语用于描述被怀疑的进程，它们可能实际上是死的。

故障可能发生在链路层面（进程之间的消息丢失或缓慢传递），或进程层面（进程崩溃或运行缓慢），且缓慢并不总是可与故障区分。这意味着在错误地将存活进程怀疑为死亡（产生假阳性）与延迟将无响应进程标记为死亡、给予其怀疑的好处并期望其最终响应（产生假阴性）之间，始终存在权衡。

**故障检测器（failure detector）** 是负责识别故障或不可达进程的本地子系统，以将其排除在算法之外，在保持安全性的同时保证活性。

活性和安全性是描述算法解决特定问题的能力及其输出正确性的属性。更正式地，**活性（liveness）** 是保证特定预期事件必须发生的属性。例如，若其中一个进程已故障，故障检测器必须检测到该故障。**安全性（safety）** 保证意外事件不会发生。例如，若故障检测器已将进程标记为死亡，该进程必须实际上已死亡。

从实践的角度，排除故障进程有助于避免不必要的工作并防止错误传播和级联故障，而排除可能被怀疑存活的进程会降低可用性。

故障检测算法应表现出几个基本属性。首先，每个非故障成员最终应注意到进程故障，算法应能够取得进展并最终达到其最终结果。此属性称为**完备性（completeness）**。

我们可以通过算法的效率来判断其质量：故障检测器能多快识别进程故障。另一种方式是看算法的准确性：是否精确检测到进程故障。换句话说，若算法错误地指控存活进程为故障或无法检测到现有故障，则算法不准确。

我们可以将效率与准确性之间的关系视为可调参数：更高效的算法可能更不精确，更准确的算法通常更不高效。可以证明不可能构建既准确又高效的故障检测器。同时，故障检测器允许产生假阳性（即错误地将存活进程识别为故障，反之亦然）。

故障检测器是许多共识和原子广播算法的重要前提和组成部分，我们将在本书后面讨论。

许多分布式系统通过使用心跳实现故障检测器。这种方法因其简单性和强完备性而相当流行。我们在此讨论的算法假设不存在拜占庭故障：进程不会故意谎报其状态或邻居的状态。

## 心跳与 Ping

我们可以通过触发两种周期性进程之一来查询远程进程的状态：

- 我们可以触发 **ping**，向远程进程发送消息，检查它们是否仍然存活，期望在指定时间内收到响应。
- 我们可以触发 **heartbeat（心跳）**，进程主动向对等方通知其仍在运行。

我们在此以 ping 为例，但使用心跳可以解决相同问题并产生类似结果。

每个进程维护其他进程的列表（存活的、死亡的、被怀疑的），并根据每个进程的最后响应时间更新它。若进程在较长时间内未能响应 ping 消息，则被标记为被怀疑。

故障检测的 ping：正常运作，无消息延迟
:::

```
P1  ----Ping---->  P2
P1  <---ACK------- P2
```

相比之下，图 9-2  shows 确认消息被延迟，可能导致将活跃进程标记为宕机。

::: tip 图 9-2

:::

许多故障检测算法基于心跳和超时。例如，Akka（构建分布式系统的流行框架）有 deadline 故障检测器的实现，使用心跳并在固定时间间隔内未注册时报告进程故障。

这种方法有几个潜在缺点：其精度取决于 ping 频率和超时的精心选择，且不捕获从其他进程视角的进程可见性。

## 无超时故障检测器

某些算法避免依赖超时来检测故障。例如，Heartbeat（无超时故障检测器）是一种仅计数心跳并允许应用程序根据心跳计数器向量中的数据检测进程故障的算法。由于该算法无超时，它在异步系统假设下运行。

该算法假设任意两个正确进程通过公平路径彼此连接（公平路径仅包含公平链路，即若通过此链路无限次发送消息，则也无限次接收），且每个进程知道网络中所有其他进程的存在。

每个进程维护邻居列表和与之关联的计数器。进程通过向邻居发送心跳消息开始。每条消息包含心跳迄今为止经过的路径。初始消息包含路径中的第一个发送方和可用于避免多次广播同一消息的唯一标识符。

当进程收到新的心跳消息时，它为路径中存在的所有参与者递增计数器，并向未出现在路径中的参与者发送心跳，将自己追加到路径。一旦进程看到所有已知进程都已收到消息（即进程 ID 出现在路径中），就停止传播消息。

由于消息通过不同进程传播，且心跳路径包含从邻居收到的聚合信息，我们可以在两个进程之间的直接链路故障时（正确地）将不可达进程标记为存活。

心跳计数器表示系统的全局和归一化视图。此视图捕获心跳相对于彼此如何传播，使我们能够比较进程。然而，这种方法的一个缺点是解释心跳计数器可能相当棘手：我们需要选择一个能产生可靠结果的阈值。除非我们能做到这一点，否则算法将错误地将活跃进程标记为被怀疑。

## 外包心跳（Outsourced Heartbeats）

SWIM（可扩展弱一致感染式进程组成员协议）使用的另一种方法是通过使用外包心跳，利用从邻居视角的进程存活信息来提高可靠性。此方法不需要进程知道网络中的所有其他进程，只需要连接的对等方的子集。

如图 9-3 所示，进程 P1 向进程 P2 发送 ping 消息。P2 未响应消息，因此 P1 选择多个随机成员（P3 和 P4）。这些随机成员尝试向 P2 发送心跳消息，若其响应，则向 P1 转发确认。

::: tip 图 9-3
「外包」心跳
:::

```
P1  ----Ping---->  P2
P1  ----Ping---->  P3, P4
P3, P4 ----Ping---->  P2
P3, P4 ----ACK---->  P1
```

这允许同时考虑直接和间接可达性。例如，若有进程 P1、P2、P3，我们可以从 P1 和 P2 的视角检查 P3 的状态。

外包心跳通过将决策责任分散到组成员来允许可靠的故障检测。此方法不需要向广泛的对等方广播消息。由于外包心跳请求可以并行触发，此方法可以快速收集关于被怀疑进程的更多信息，并使我们能够做出更准确的决策。

## Phi 累积故障检测器（Phi-Accural Failure Detector）

与其将节点故障视为二元问题（进程只能处于两种状态：上或下），**phi 累积（φ-accrual）故障检测器**具有连续尺度，捕获被监控进程崩溃的概率。它通过维护滑动窗口、收集来自对等进程的最近心跳的到达时间来实现。此信息用于近似下一次心跳的到达时间，将此近似与实际到达时间比较，并计算怀疑级别 φ：给定当前网络条件，故障检测器对故障的确定程度。

该算法通过收集和采样到达时间，创建可用于对节点健康做出可靠判断的视图。它使用这些样本计算 φ 的值：若此值达到阈值，节点被标记为宕机。此故障检测器通过调整节点可被标记为怀疑者的尺度来动态适应变化的网络条件。

从架构角度，phi 累积故障检测器可被视为三个子系统的组合：

- **监控（Monitoring）**：通过 ping、心跳或请求-响应采样收集存活信息
- **解释（Interpretation）**：决定是否应将进程标记为被怀疑
- **动作（Action）**：每当进程被标记为被怀疑时执行的回调

监控过程在固定大小的心跳到达时间窗口中收集和存储数据样本（假定服从正态分布）。较新的到达被添加到窗口，最旧的心跳数据点被丢弃。通过确定样本的均值和方差从采样窗口估计分布参数。此信息用于计算在先前消息之后 t 时间单位内消息到达的概率。给定此信息，我们计算 φ，描述我们做出关于进程正确决策的可能性。换句话说，我们犯错并收到与计算假设相矛盾的心跳的可能性有多大。

该方法是日本北陆先端科学技术大学院大学的研究人员开发的，现用于许多分布式系统，例如 Cassandra 和 Akka（以及前述的 deadline 故障检测器）。

## Gossip 与故障检测

另一种避免依赖单节点视图来做决策的方法是 **gossip 式故障检测服务**，使用 gossip 传播来收集和分发邻居进程的状态。

每个成员维护其他成员的列表、其心跳计数器和时间戳（指定心跳计数器上次递增的时间）。定期地，每个成员递增其心跳计数器并将其列表分发给随机邻居。收到消息后，邻居节点将列表与自己的合并，更新其他邻居的心跳计数器。

节点还定期检查状态列表和心跳计数器。若任何节点在足够长时间内未更新其计数器，则被视为故障。应谨慎选择此超时周期以最小化假阳性概率。成员彼此通信的频率（即最坏情况带宽）有上限，最多可随系统中进程数量线性增长。

图 9-4 显示了三个通信进程共享其心跳计数器：

- a) 三者都能通信并更新其时间戳
- b) P3 无法与 P1 通信，但其时间戳 t6 仍可通过 P2 传播
- c) P3 崩溃。由于它不再发送更新，被其他进程检测为故障

::: tip 图 9-4
用于故障检测的复制心跳表
:::

这样，我们可以检测崩溃的节点，以及任何其他集群成员都无法访问的节点。此决策是可靠的，因为集群视图是来自多个节点的聚合。若两个主机之间存在链路故障，心跳仍可通过其他进程传播。使用 gossip 传播系统状态增加了系统中的消息数量，但允许信息更可靠地传播。

## 反转故障检测问题陈述

由于传播故障信息并不总是可能，且通过通知每个成员传播可能代价高昂，一种称为 **FUSE（故障通知服务）** 的方法专注于可靠且廉价的故障传播，即使在网络分区情况下也能工作。

为检测进程故障，此方法将所有活跃进程排列成组。若其中一组变得不可用，所有参与者检测到故障。换句话说，每次检测到单个进程故障时，它被转换并传播为组故障。这允许在存在任何断开、分区和节点故障模式的情况下检测故障。

组中的进程定期向其他成员发送 ping 消息，查询它们是否仍然存活。若其中一名成员因崩溃、网络分区或链路故障无法响应此消息，发起此 ping 的成员将反过来停止响应 ping 消息本身。

图 9-5 显示了四个通信进程：

- a) 初始状态：所有进程存活且能通信
- b) P2 崩溃并停止响应 ping 消息
- c) P4 检测到 P2 的故障并停止响应 ping 消息
- d) 最终，P1 和 P3 注意到 P1 和 P2 都不响应，进程故障传播到整个组

::: tip 图 9-5
FUSE 故障检测
:::

所有故障通过系统从故障源传播到所有其他参与者。参与者逐渐停止响应 ping，从单个节点故障转换为组故障。

这里，我们使用通信的缺失作为传播手段。使用此方法的一个优点是每个成员保证了解组故障并适当响应。缺点之一是，将单个进程与其他进程分开的链路故障也可能被转换为组故障，但这取决于用例可能被视为优点。应用程序可以使用其自己的传播故障定义来考虑此场景。

## 小结

故障检测器是任何分布式系统的必要组成部分。如 FLP 不可能性结果所示，没有协议能在异步系统中保证共识。故障检测器通过权衡准确性和完备性来帮助增强模型，使我们能够解决共识问题。该领域的重要发现之一，证明故障检测器的有用性，在论文中描述，表明即使故障检测器犯无限次错误，解决共识也是可能的。

我们覆盖了多种故障检测算法，每种使用不同方法：有些专注于通过直接通信检测故障，有些使用广播或 gossip 传播信息，有些选择使用静默（即通信的缺失）作为传播手段。

我们现在知道可以使用心跳或 ping、硬截止时间或连续尺度。每种方法都有其优点：简单性、准确性或精度。
